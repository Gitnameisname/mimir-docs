1. 작업명

Phase 3 - Task 3-1. 플랫폼 API 아키텍처 원칙 정의

⸻

2. 작업 목적

문서 플랫폼 전반에서 공통적으로 사용할 플랫폼 API 아키텍처 원칙 문서를 작성하라.

이 작업의 목적은 단순한 REST 스타일 가이드 작성이 아니다.
앞으로 구현될 문서 관리, 버전 관리, 권한 제어, 감사 추적, 검색, 이벤트, 웹훅, AI/RAG 연계 기능까지 모두 수용할 수 있도록, 플랫폼 API의 역할, 책임 경계, 설계 우선순위, 확장 원칙을 먼저 확정하는 것이다.

이 문서는 이후 Task 3-2부터 Task 3-12까지의 기준선이 되어야 하므로, 구현 세부사항보다 아키텍처 원칙과 판단 기준을 명확히 정리하는 데 집중하라.

⸻

3. 반드시 반영할 배경

이 프로젝트는 특정 문서 하나를 다루는 서비스가 아니라, 모든 유형의 문서를 다룰 수 있는 범용 문서 플랫폼을 목표로 한다.
또한 이 플랫폼은 단순한 내부 CRUD 앱이 아니라 다음을 동시에 지원해야 한다.
	•	사용자 UI
	•	관리자 UI
	•	외부 애플리케이션 연동
	•	AI 도구 및 RAG 시스템 연계
	•	향후 이벤트 기반 확장
	•	권한/감사 추적/보안 요구사항

따라서 API는 단순한 백엔드 구현 상세가 아니라, 플랫폼 전체의 공식 인터페이스 계층이어야 한다.

이전 Phase에서 이미 다음 방향성이 정해졌다고 가정하고 반영하라.
	•	API-first 원칙
	•	User/Admin UI 분리
	•	범용 문서 도메인 모델 사용
	•	ACL 기반 권한 구조 및 감사 추적 체계 예정
	•	향후 AI/RAG 연동 필요
	•	외부 시스템도 사용할 수 있는 플랫폼 API 필요

5-2. 플랫폼 API의 역할 정의

문서에서 플랫폼 API를 다음 수준으로 설명하라.
	•	단순한 프론트엔드 백엔드가 아님
	•	플랫폼 내부 도메인 기능을 외부에 노출하는 공식 인터페이스 계층
	•	User UI, Admin UI, 외부 서비스, AI agent/tool이 공통적으로 사용할 수 있는 기반
	•	도메인 모델과 소비자 사이를 연결하는 안정된 계약 계층
	•	보안, 추적성, 일관성, 확장성의 기준점

이때 “API = 엔드포인트 모음” 수준이 아니라,
플랫폼의 계약 계층(contract layer) 이라는 관점이 드러나야 한다.

⸻

5-3. API 소비자 유형 구분

다음 소비자를 분리해서 설명하라.
	•	User UI
	•	Admin UI
	•	External Application / Integration
	•	AI / RAG / Agent Consumer
	•	Internal Service Consumer(필요 시)

각 소비자에 대해 아래를 정리하라.
	•	사용하는 이유
	•	기대하는 인터페이스 특성
	•	민감하게 보는 요소
	•	공통 API 위에 올릴 수 있는지 여부
	•	별도 전용 API가 필요한 경우의 기준

중요:
여기서 원칙적으로는 공통 플랫폼 API를 우선하되,
정말 필요한 경우에만 별도 특화 계층을 둘 수 있다는 방향으로 정리하라.

⸻

5-4. 상위 아키텍처 원칙 정의

문서에는 최소한 아래 원칙들을 포함하라.
	•	API-first
	•	Resource-oriented
	•	Consistency over convenience
	•	Secure by default
	•	Extensible by design
	•	Observable by default
	•	AI-integratable
	•	Backward-compatibility aware
	•	Explicit contracts over implicit behavior

각 원칙마다 다음을 써라.
	•	원칙 설명
	•	왜 필요한지
	•	이 원칙이 실제 설계에서 어떤 결정을 유도하는지

예를 들어:
	•	“Consistency over convenience”가 있으면 엔드포인트마다 응답 구조를 제각각 만들지 않음
	•	“Secure by default”가 있으면 권한 정보 없는 요청은 기본 거부 방향으로 설계
	•	“AI-integratable”가 있으면 문서 구조/검색/인용 가능한 식별자 등을 처음부터 고려

⸻

5-5. 계층 경계와 책임 분리

아래 항목의 책임을 구분해서 설명하라.
	•	API Layer
	•	Application/Service Layer
	•	Domain Layer
	•	Authorization Layer
	•	Audit / Logging / Event Layer
	•	Background / Async Processing Layer

반드시 포함할 내용:
	•	API Layer는 입력/출력 계약과 요청 문맥 처리에 집중
	•	비즈니스 규칙은 서비스/도메인 계층에 두는 방향
	•	인증된 주체, 조직 스코프, 권한 맥락은 API에서 받아도 실제 권한 판단은 별도 계층에서 수행
	•	감사 로그나 이벤트 발행이 API와 강하게 연결되지만, API 코드에 직접 흩뿌려지지 않도록 해야 함

즉, “무엇을 API에 두고 무엇을 두지 않을지” 기준을 문서화하라.

⸻

5-6. REST 우선 전략과 예외

다음을 포함해 정리하라.
	•	기본 인터페이스는 REST API로 설계
	•	리소스 중심 구조를 우선
	•	기능 명령형 엔드포인트 남발 금지
	•	다만 아래는 REST 단독으로 다루기 어려울 수 있음
	•	long-running jobs
	•	event delivery
	•	webhook subscription
	•	search/retrieval workflows
	•	AI-oriented composite access patterns

이 경우 “REST를 버린다”가 아니라,
REST 기반 플랫폼 위에 확장 계층을 추가하는 방식으로 정리하라.

⸻

5-7. 보안 및 권한 연계 관점

이번 문서는 인증/인가를 상세 설계하는 Task 3-3 전 단계이므로, 세부 프로토콜보다는 원칙을 정리하라.

반드시 담을 내용:
	•	API는 인증되지 않은 사용자를 기본 신뢰하지 않는다
	•	모든 요청은 주체(subject), 조직/테넌트 문맥, 권한 판단 가능성을 가져야 한다
	•	권한은 단순 UI 숨김이 아니라 API 레벨에서 강제되어야 한다
	•	감사 가능성을 위해 주요 요청은 추적 가능해야 한다
	•	외부 시스템 연계 시에도 동일한 보안 모델에 맞춰야 한다

⸻

5-8. AI/RAG 친화적 인터페이스 관점

이 문서에서는 구체 엔드포인트를 설계하지 말고, 다음 관점을 API 원칙에 포함하라.
	•	AI도 플랫폼의 정식 API 소비자다
	•	문서 본문뿐 아니라 구조, 버전, 노드, metadata, 상태 정보를 접근 가능하게 해야 한다
	•	retrieval / citation / chunk-level access를 고려해야 한다
	•	검색/요약/인덱싱 구현은 나중이라도, API 인터페이스는 이를 수용할 수 있어야 한다
	•	AI 전용 비공식 우회 경로를 만들지 않고 플랫폼 API 체계 안에 편입시키는 방향을 우선한다

⸻

5-9. 운영 및 관측성 원칙

문서에 아래 내용을 반영하라.
	•	API 요청은 추적 가능해야 한다
	•	request_id / trace_id / actor context 같은 운영 메타데이터를 고려해야 한다
	•	오류 응답은 사람이 읽을 수 있어야 하고 동시에 기계 처리 가능해야 한다
	•	감사 로그와 API 로그는 역할이 다르므로 구분이 필요하다
	•	운영 단계에서 디버깅 가능한 API여야 한다
