Claude Code 작업 지시서

Phase 1 - Task 1-2. 핵심 엔티티 식별 및 책임 정의

1. 작업명

범용 문서 플랫폼의 핵심 엔티티 식별 및 책임 정의

2. 작업 목적

Phase 1 - Task 1-1에서 정리한 범용 문서 도메인 요구사항을 바탕으로,
플랫폼의 코어를 구성할 핵심 엔티티들을 식별하고 각 엔티티의 책임과 경계를 명확히 정의한다.

이 작업의 목적은 단순히 엔티티 이름을 나열하는 것이 아니라,
이후 Document / Version / Node / DocumentType / metadata / 상태 모델을 설계할 때 흔들리지 않도록
무엇을 어떤 엔티티가 책임지는지,
무엇은 해당 엔티티의 책임이 아닌지를 분명히 정리하는 것이다.

⸻

3. 작업 배경

범용 문서 플랫폼은 단순 파일 저장 시스템이 아니라,
문서의 정체성, 버전, 내부 구조, 유형, 확장 속성, 상태를 일관된 도메인 모델로 다뤄야 한다.

이때 가장 먼저 정리해야 하는 것은 다음이다.
	•	어떤 엔티티가 플랫폼의 코어인가
	•	각 엔티티는 무엇을 대표하는가
	•	엔티티 간 책임이 어떻게 분리되는가
	•	서로 겹치기 쉬운 개념을 어떻게 구분할 것인가

특히 아래 개념들은 분리되어야 한다.
	•	문서 자체의 정체성
	•	특정 시점의 문서 내용
	•	문서 내부 구조 단위
	•	문서 유형 규칙
	•	문서 확장 속성
	•	문서 상태

이번 작업에서는 이들을 도메인 엔티티 수준에서 정리해야 한다.

⸻

4. 작업 범위

포함 범위
	•	코어 엔티티 후보 식별
	•	각 엔티티의 역할과 책임 정의
	•	엔티티 간 관계의 개략 정의
	•	포함할 엔티티와 제외할 엔티티 구분
	•	부가 엔티티 후보 검토
	•	이후 상세 모델링을 위한 경계 정의

제외 범위
	•	필드 단위 상세 DB 스키마 설계
	•	API 요청/응답 구조 설계
	•	UI 컴포넌트 설계
	•	실제 코드 구현
	•	승인 워크플로 상세 설계
	•	권한 체계 상세 설계

즉, 이번 작업은 도메인 개념 정리와 책임 분리 문서 작성이 목적이다.

⸻

5. Claude Code가 해야 할 일

아래 순서대로 결과를 만들어라.

Step 1. 코어 엔티티 후보 식별

최소 아래 엔티티 후보를 검토 대상으로 포함한다.
	•	Document
	•	Version
	•	Node
	•	DocumentType
	•	Metadata
	•	DocumentStatus

그리고 필요 시 아래 부가 후보도 함께 검토한다.
	•	Attachment
	•	Reference
	•	Relation
	•	Tag
	•	AuditEvent
	•	Approval / Review
	•	Publication
	•	Template

이 단계에서는 각 항목이
	•	코어 엔티티인지
	•	값 객체 또는 보조 개념인지
	•	후속 Phase로 미뤄야 하는지

를 판단해야 한다.

⸻

Step 2. 엔티티별 의미 정의

각 후보에 대해 아래를 정리한다.
	•	이 개념이 무엇을 대표하는가
	•	왜 별도 엔티티 또는 개념 단위로 분리해야 하는가
	•	어떤 책임을 가져야 하는가
	•	무엇은 책임지지 않아야 하는가

예시 관점:

Document
	•	문서의 “정체성”을 대표하는가?
	•	현재 버전 참조를 가지는가?
	•	문서의 모든 내용까지 직접 책임져야 하는가?

Version
	•	특정 시점 문서 상태의 스냅샷인가?
	•	수정 이력과 복원을 위한 단위인가?
	•	문서의 본체인가, 문서의 한 시점인가?

Node
	•	문서 내부 구조를 표현하는 최소 단위인가?
	•	제목/섹션/문단/표 등 다양한 구조를 담는가?
	•	버전과 종속 관계를 가지는가?

DocumentType
	•	단순 분류값인가?
	•	규칙/검증/템플릿과 연결될 여지가 있는가?

Metadata
	•	별도 엔티티인가?
	•	Document/Version/Node에 부속되는 확장 구조인가?

DocumentStatus
	•	독립 엔티티인지, enum/상태 모델인지 판단

⸻

Step 3. 코어 엔티티 확정

위 검토를 바탕으로, 이번 Phase 1의 코어 엔티티를 확정한다.

권장 방향:
	•	Document
	•	Version
	•	Node
	•	DocumentType(개념 또는 독립 관리 대상 여부 판단)
	•	Metadata(독립 엔티티가 아니라 확장 구조일 가능성 검토)
	•	DocumentStatus(상태 모델 개념으로 유지할 가능성 검토)

이 단계에서는 특히 아래를 명확히 해야 한다.
	•	코어 엔티티
	•	보조 개념
	•	후속 확장 엔티티

⸻

Step 4. 책임 분리 표 작성

각 핵심 엔티티에 대해 아래 형식으로 정리한다.
	•	엔티티명
	•	한 줄 정의
	•	핵심 책임
	•	책임하지 않는 것
	•	주요 연관 엔티티
	•	설계 시 주의사항

예시:

엔티티	정의	책임	비책임	연관 대상	비고


이 표는 이후 Task에서 바로 참고할 수 있을 정도로 명확해야 한다.

⸻

Step 5. 엔티티 간 경계와 관계 정리

아래 질문에 답할 수 있도록 정리한다.
	•	Document와 Version은 어떻게 다른가?
	•	Version과 Node는 어떻게 연결되는가?
	•	DocumentType은 Document의 속성인가, 독립 규칙 단위인가?
	•	Metadata는 어디에 귀속되는가?
	•	상태는 Document 수준인가, Version 수준인가?

관계는 ERD 수준의 상세 설계까지 갈 필요는 없지만,
최소한 개념 관계는 정리되어야 한다.

예시:
	•	Document 1 : N Version
	•	Version 1 : N Node
	•	Document N : 1 DocumentType
	•	Metadata는 Document/Version/Node에 부속될 수 있음

⸻

Step 6. 부가 엔티티 후보 검토

아래 후보들에 대해 “지금 코어에 포함할지 / 나중으로 미룰지” 판단한다.
	•	Attachment
	•	Reference
	•	Relation
	•	Tag
	•	AuditEvent
	•	Template
	•	Approval
	•	Publication

각 항목에 대해 짧게 정리한다.
	•	왜 필요한가
	•	지금 코어에 넣어야 하는가
	•	후속 Phase에서 다뤄도 되는가

목적은 모델을 과도하게 비대하게 만들지 않도록 하는 것이다.

⸻

Step 7. 최종 결론 정리

문서 마지막에는 반드시 아래를 정리한다.
	•	이번 단계에서 확정한 코어 엔티티 목록
	•	각 엔티티의 핵심 책임 요약
	•	보조 개념으로 처리하기로 한 항목
	•	후속 Task에서 상세 설계가 필요한 포인트

특히 다음 Task로 넘길 입력값이 분명해야 한다.

예:
	•	Task 1-3에서는 Document 구조를 상세화한다.
	•	Task 1-4에서는 Version 구조를 상세화한다.
	•	Task 1-5에서는 Node 구조를 상세화한다.

⸻

	4.	엔티티별 정의와 책임
	5.	엔티티 간 관계와 경계
	6.	부가 엔티티 후보 검토
	7.	최종 결론 및 다음 Task 입력값

⸻

7. 작성 방식 요구사항

1) 표를 적극 활용

최소 아래 표를 포함하는 것이 좋다.
	•	엔티티 후보 분류표
	•	책임 분리표
	•	부가 엔티티 후보 검토표

2) 책임과 비책임을 분명히 쓸 것

좋은 예:
	•	“Document는 문서의 정체성과 대표 정보를 관리한다.”
	•	“Version은 특정 시점의 문서 내용을 나타낸다.”
	•	“Node는 문서 내부 구조를 표현한다.”
	•	“Metadata는 독립 문서 본체가 아니라 확장 속성 구조로 다룬다.”

좋지 않은 예:
	•	“Document는 문서를 나타낸다.”
	•	“Version은 버전이다.”
	•	“Node는 내용이다.”

3) 구현으로 넘어가지 말 것

이번 문서는 개념 정리 문서다.
테이블 컬럼 설계, ORM 모델 코드, API 스펙 작성으로 넘어가지 마라.

9. 완료 기준

아래를 모두 만족하면 완료로 본다.
	•	Document / Version / Node 등 핵심 개념의 역할이 분리되어 있다.
	•	각 엔티티가 무엇을 책임하고 무엇을 책임하지 않는지 명확하다.
	•	코어 엔티티와 부가 엔티티가 구분되어 있다.
	•	DocumentType / Metadata / Status의 위치가 개념적으로 정리되어 있다.
	•	이후 Task 1-3, 1-4, 1-5로 자연스럽게 이어질 수 있다.
	•	특정 구현 기술이나 DB 설계로 prematurely 내려가지 않았다.

⸻

10. 작업 시 주의사항
	•	특정 DB 구조를 전제로 판단하지 마라.
	•	특정 UI 에디터 구조를 전제로 엔티티를 왜곡하지 마라.
	•	Attachment, Approval 같은 부가 요소를 너무 일찍 코어에 넣지 마라.
	•	Document와 Version의 책임을 섞지 마라.
	•	Node를 단순 텍스트 조각으로 축소하지 마라.
	•	Metadata를 무조건 독립 엔티티로 만들지 말고, 확장 구조로 보는 관점도 함께 검토하라.
	•	상태 모델은 엔티티 그 자체인지, 속성/상태 머신인지 구분해서 다뤄라.

⸻
