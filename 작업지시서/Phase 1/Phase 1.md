Phase 1. 범용 문서 도메인 모델 설계 작업계획서

1. Phase 개요

Phase 명

Phase 1. 범용 문서 도메인 모델 설계

목적

플랫폼이 특정 문서군에 묶이지 않고,
규정집·매뉴얼·보고서·가이드·회의록·지식문서·템플릿 문서 등 모든 문서 유형을 공통된 구조로 수용할 수 있는 코어 도메인 모델을 설계한다.

핵심 목표
	•	문서의 정체성을 나타내는 Document
	•	문서 변경 이력을 나타내는 Version
	•	문서 내부 구조를 표현하는 Node
	•	문서 유형별 차이를 흡수하는 DocumentType
	•	문서별 추가 속성을 담는 metadata 확장 구조
	•	문서의 생명주기를 표현하는 상태 모델 초안

을 설계하여, 이후 API / 저장소 / 편집기 / 검색 / 승인 워크플로의 기준이 되는 핵심 데이터 구조를 확정한다.

결과물
	•	범용 문서 도메인 모델 명세서
	•	엔티티 관계 정의서
	•	Node 트리 구조 설계서
	•	DocumentType 확장 규칙서
	•	metadata 확장 정책서
	•	문서 상태 전이 초안
	•	이후 단계에서 사용할 API/DB 설계 입력 기준

⸻

2. 설계 원칙

Phase 1에서는 구현보다 먼저 아래 원칙을 고정한다.

2-1. 범용성 우선
	•	특정 문서 종류 하나에 최적화하지 않는다.
	•	규정집, 기술문서, 정책문서, 보고서, 매뉴얼, 게시형 문서까지 수용 가능해야 한다.

2-2. 구조와 표현의 분리
	•	데이터 모델은 문서의 의미 구조를 표현한다.
	•	렌더링 방식(UI 표시, 에디터 표시)은 별도 계층에서 해결한다.

2-3. 버전 독립성
	•	문서의 현재 상태와 과거 상태를 분리해서 관리할 수 있어야 한다.
	•	버전 비교, 복원, 승인 흐름으로 확장 가능해야 한다.

2-4. 트리 기반 표준화
	•	문서 본문은 기본적으로 트리 구조로 표현한다.
	•	단순 본문 문서부터 복합 구조 문서까지 확장 가능하도록 한다.

2-5. 유형 확장 가능성
	•	DocumentType은 고정 enum 수준으로 끝내지 않고,
추후 사용자 정의 유형이나 플러그인형 확장을 고려한다.

2-6. 메타데이터 유연성
	•	공통 필드는 최소화하되,
문서 유형별 추가 필드를 수용할 수 있어야 한다.

2-7. 상태 모델의 점진적 고도화
	•	이번 단계에서는 최소 상태 모델을 정의하고,
승인/배포/아카이브 체계는 이후 단계에서 확장한다.

6. 주요 설계 쟁점

Phase 1에서 반드시 판단해야 할 포인트들이다.

쟁점 A. Document와 Version의 책임 분리 수준
	•	Document는 정체성만 가질지
	•	현재 대표 정보 일부를 중복 보관할지

쟁점 B. Version 저장 방식
	•	매 버전 완전 스냅샷
	•	diff 중심
	•	초기에는 스냅샷, 이후 diff 확장 권장

쟁점 C. Node granularity
	•	문단 단위까지 자를지
	•	섹션 중심으로 갈지
	•	너무 잘게 쪼개면 복잡도 증가

쟁점 D. DocumentType의 강제력
	•	단순 분류 태그인지
	•	구조 검증 규칙을 가진 타입인지

쟁점 E. metadata 자유도
	•	완전 자유 JSON
	•	타입별 schema validation 적용
	•	권장 방향은 “공통 코어 + 타입별 검증 가능 구조”

쟁점 F. 상태 모델의 적용 위치
	•	Document 수준만 둘지
	•	Version 수준도 둘지
	•	초기에는 Document 중심, 이후 Version 승인 상태 확장 권장
