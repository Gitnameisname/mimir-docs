

# Task 4-7 작업지시서

## 작업명

렌더링 파이프라인 기초 설계

---

## 1. 작업 목적

Phase 4의 일곱 번째 작업으로, 문서 플랫폼에 저장된 문서 구조를 **사용자가 읽을 수 있는 화면 표현으로 안정적으로 변환하는 렌더링 파이프라인의 기초 구조**를 설계한다.

이번 작업의 목적은 다음과 같다.

- Task 4-3에서 정의한 저장 구조와 Task 4-6에서 정의한 조회 흐름을 실제 표시 가능한 구조로 연결한다.
- 저장 포맷과 렌더링 포맷의 책임을 분리하여 UI 종속성을 낮춘다.
- 현재 문서, 특정 버전, Draft, Published를 동일한 렌더링 규약 위에서 표현할 수 있는 기준선을 마련한다.
- 향후 편집 UI, 읽기 전용 뷰, 비교(diff) 뷰, export(PDF/HTML), AI 요약 뷰로 확장 가능한 변환 구조를 설계한다.
- 렌더링 실패, 미지원 블록, 부분 손상 데이터에 대한 최소 복구 전략을 정의하여 플랫폼의 안정성을 높인다.

이 문서는 이후 구현 단계의 ViewModel 설계, 렌더러 컴포넌트 설계, 조회 응답 구성, 렌더링 테스트 케이스 설계의 기준 문서가 되어야 한다.

---

## 2. 작업 배경

앞선 작업에서 다음이 이미 정리되었다고 가정한다.

- Task 4-1: 문서 생성/개정/발행/복원의 생명주기
- Task 4-2: Version 모델 상세 설계
- Task 4-3: 문서 저장 구조와 content_snapshot 규약
- Task 4-4: 문서 작성/수정 API 설계
- Task 4-6: 버전 조회, 이력 탐색, 복원 흐름 설계

이제 필요한 것은 저장된 Node 트리 또는 content_snapshot을 그대로 화면에 노출하는 것이 아니라, **읽기/탐색/운영 목적에 맞는 렌더링 전용 표현 계층**으로 변환하는 기준을 정하는 일이다.

즉, 이번 작업은 단순히 “문단을 화면에 찍는다”가 아니라, 다음을 정의하는 작업이다.

- 어떤 입력 구조를 렌더링 파이프라인이 받는가?
- 저장 구조와 화면 표시 구조는 어디까지 같고 어디부터 달라지는가?
- 최소 블록 타입에 대해 어떤 렌더링 규칙을 적용하는가?
- 본문, 섹션, 표, 목록, 부록, 참조 영역을 어떤 일관된 방식으로 표현하는가?
- 이상 데이터가 들어왔을 때 렌더링은 어디까지 복구해야 하는가?

## 5. 작업 지시

아래 항목을 순서대로 설계 문서로 정리한다.

### 5-1. 렌더링 파이프라인의 단계 정의

먼저 문서 렌더링이 어떤 단계로 이루어지는지 정의한다.

최소한 아래 단계를 검토한다.

1. 원본 조회 단계
2. 버전 선택 단계
3. content_snapshot 로드 단계
4. 구조 검증 또는 정규화 단계
5. 렌더링 ViewModel 변환 단계
6. 블록별 렌더링 규칙 적용 단계
7. 최종 출력 구성 단계

각 단계에 대해 다음을 정리한다.

- 입력
- 출력
- 책임
- 실패 가능 지점
- 후속 단계와의 관계

중요: 렌더링은 단순 화면 출력이 아니라, **조회 모델을 표시 모델로 변환하는 파이프라인**이라는 점을 문서에 분명히 적는다.

---

### 5-2. 렌더링 입력 모델 정의

렌더링 파이프라인이 어떤 입력 구조를 받는지 정의한다.

다음 후보를 비교 검토한다.

#### 안 A. 저장 구조(content_snapshot)를 직접 입력으로 사용

- 단순함
- 저장 구조와 UI 결합도가 높아질 수 있음

#### 안 B. 조회 서비스가 정리한 중간 렌더링 입력 모델을 사용

- 변환 책임 분리 가능
- 계층이 늘어남

#### 안 C. 저장 구조를 1차 정규화한 후 ViewModel로 변환

- 가장 유연함
- 단계가 다소 많아짐

각 안의 장단점을 정리하고, 이번 Phase 기준 권장안을 제시한다.

반드시 다음 관점도 포함한다.

- 현재 문서 조회와 특정 버전 조회 재사용성
- API 응답 무게
- 오류 복구 가능성
- UI 교체 내성

---

### 5-3. 렌더링 ViewModel 기본 구조 정의

저장 구조와 분리된 렌더링 ViewModel이 필요하다면, 그 최소 구조를 정의한다.

검토 대상 예시:

- render_document_id
- source_document_id
- source_version_id
- render_mode
- title
- subtitle 또는 summary 필요 여부
- blocks
- toc
- metadata_panel 필요 여부
- warnings
- unsupported_blocks

블록 단위 ViewModel 후보 예시:

- block_id
- block_type
- display_role
- content
- children
- order
- heading_level
- table_model
- annotations
- warnings

각 필드에 대해 다음을 정리한다.

- 목적
- 필수 여부
- 생성 시점
- UI 의존 여부
- 확장 가능성

중요: ViewModel은 저장 구조를 그대로 복사한 것이 아니라, **표시 목적에 맞춰 정제된 구조**여야 한다.

---

### 5-4. 최소 렌더링 블록 타입 정의

초기 Phase에서 반드시 지원할 렌더링 블록 타입을 정의한다.

최소 검토 대상:

- document
- section
- heading
- paragraph
- list
- list_item
- table
- quote
- appendix
- reference block 필요 여부
- unsupported block
- error block 또는 fallback block 필요 여부

각 블록 타입에 대해 다음을 정리한다.

- 의미
- 입력 Node와의 대응 관계
- 렌더링 시 기본 표현 방식
- 중첩 가능 여부
- fallback 필요 여부

이번 작업은 디자인이 아니라 구조 설계이므로, 시각 표현보다 **표시 규칙**을 중심으로 적는다.

---

### 5-5. 블록별 렌더링 규칙 정의

주요 블록 타입별로 기본 렌더링 규칙을 정한다.

최소한 아래를 포함한다.

#### Heading
- 레벨 표현 방식
- 목차 반영 여부
- 빈 제목 처리

#### Paragraph
- 텍스트 표시 방식
- 빈 문단 처리
- 줄바꿈/공백 처리 원칙

#### Section
- 섹션 경계 표현 방식
- 하위 블록 순서 규칙
- section title 유무 처리

#### List / List Item
- 순서형/비순서형 구분 방식
- 중첩 리스트 처리

#### Table
- 행/열 구조 표현
- 빈 셀 처리
- 구조 이상 시 fallback 방식

#### Appendix / Reference
- 본문과 분리된 후반부 영역으로 처리할지 여부

각 규칙은 UI 세부 디자인이 아니라, **표시 의미와 정렬/구조 규칙** 중심으로 작성한다.

---

### 5-6. 문서 레이아웃 기본 구성 정의

문서 전체가 어떤 순서와 구획으로 표시되는지 정의한다.

최소 검토 순서 예시:

1. 문서 헤더(제목, 버전 정보, 상태 표시)
2. 부가 메타 정보 영역 필요 여부
3. 목차 영역 필요 여부
4. 본문 블록 영역
5. 부록/참조 영역
6. 경고/복구 정보 표시 영역 필요 여부

다음 질문에도 답한다.

- 읽기 모드와 편집 미리보기 모드가 같은 구조를 공유하는가?
- 상태 정보(Draft / Published)는 어디에 표시되는가?
- 버전 정보는 헤더에 기본 포함되는가?

---

### 5-7. 목차 및 파생 정보 생성 규칙 정의

목차나 섹션 요약처럼 저장 구조에 없더라도 렌더링 시 파생 생성할 수 있는 정보를 정리한다.

최소 검토 대상:

- heading 기반 목차 생성
- section anchor 생성
- 문단 수/섹션 수 등의 간단 통계 필요 여부
- 표/부록 번호화 필요 여부
- 경고 목록 집계

다음 두 방향을 비교 검토한다.

#### 안 A. 목차를 저장 구조에서 직접 관리

- 단순해 보일 수 있음
- 저장 구조 부담 증가

#### 안 B. 렌더링 단계에서 파생 생성

- 저장 구조 단순화
- 렌더링 로직 책임 증가

이번 Phase 기준 권장안을 명시한다.

---

### 5-8. 현재 문서 렌더링과 특정 버전 렌더링 관계 정의

현재 문서와 특정 버전은 렌더링 관점에서 어떤 차이를 가지는지 정리한다.

반드시 다음을 포함한다.

- 공통 파이프라인 재사용 가능 범위
- Draft 렌더링과 Published 렌더링의 차이
- 상태 배지/주의 메시지 차이
- 편집 중 미완성 구조에 대한 허용 범위 차이
- 특정 버전 조회 시 immutable 표시 필요 여부

권장 방향은, **동일한 렌더링 파이프라인을 공유하되 상태/배지/경고 정보만 다르게 주입하는 방식**을 우선 검토하는 것이다.

---

### 5-9. 렌더링 실패 및 fallback 정책 정의

문서 구조가 일부 손상되었거나 미지원 블록이 있을 때 어떻게 표시할지 규칙을 정한다.

최소 검토 시나리오:

- 알 수 없는 block_type
- 필수 payload 누락
- table 구조 손상
- heading level 비정상 값
- 루트 구조 일부 누락
- section 내부 허용되지 않은 자식 배치

각 시나리오에 대해 다음을 작성한다.

- 감지 시점
- 렌더링 중단 여부
- fallback block 생성 여부
- 사용자 경고 표시 여부
- 운영자/로그용 정보 분리 필요 여부

중요: 전체 문서가 일부 문제 때문에 완전히 렌더링 실패하지 않도록, **부분 복구 가능한 전략**을 우선 검토한다.

---

### 5-10. 렌더링 응답 모델과 API 연계 정의

Task 4-4와 연결하여 렌더링 조회 API가 어떤 구조를 반환해야 하는지 정리한다.

반드시 다음을 포함한다.

- 도메인 조회 응답과 렌더링 응답의 차이
- 렌더링 응답 최상위 구조
- source_version 정보 포함 방식
- render_mode 또는 view_type 필요 여부
- warnings / unsupported_blocks / recovery_notes 포함 여부
- UI가 바로 사용할 수 있는 수준까지 가공할지 여부

이 항목은 구현 단계에서 `/render` 계열 API 응답 설계의 기준이 되어야 한다.

### 5-11. 성능 및 확장 고려사항 정의

구현은 하지 않지만, 기초 설계에서 고려해야 할 성능/확장 포인트를 정리한다.

최소 검토 대상:

- 대형 문서 렌더링 시 분할 처리 필요 여부
- 목차/파생 정보 캐싱 가능성
- Published 렌더링 캐싱 가능성
- 현재 문서 vs 특정 버전 렌더링 비용 차이
- diff 뷰 / export / AI summary용 재사용 가능성

이번 항목은 상세 최적화가 아니라, **구조 설계 시 놓치지 말아야 할 고려사항** 수준으로 정리한다.

### 5-12. 최소 예시 변환 흐름 작성

최종적으로 아래 예시에 대해 저장 구조에서 렌더링 구조로 어떻게 변환되는지 간단한 예시를 작성한다.

1. 제목 + 문단 2개 문서
2. section 2개 + heading 구조 문서
3. table 포함 문서
4. unsupported block가 섞인 문서

각 예시에 대해 다음을 포함한다.

- 입력 구조 요약

## 8. 권장 방향

아래 방향을 우선안으로 검토한다.

### 우선안 A

- 저장 구조(content_snapshot)는 그대로 UI에 전달하지 않고, **정규화 + 렌더링 ViewModel 변환 단계**를 둔다.
- 렌더링 파이프라인은 `원본 조회 → 구조 검증/정규화 → ViewModel 변환 → 블록 렌더링` 흐름으로 구성한다.
- 목차, 상태 배지, 일부 경고 정보는 저장 구조가 아니라 렌더링 단계에서 파생 생성한다.
- 현재 문서 렌더링과 특정 버전 렌더링은 동일 파이프라인을 공유하되, source_version / state badge / warning 주입 정보만 다르게 둔다.
- 미지원 블록이나 손상 데이터는 전체 실패보다 **fallback block + warning 수집** 전략을 우선 사용한다.
- 초기 블록 타입은 heading, paragraph, section, list, table, appendix 수준의 최소 집합으로 시작한다.

이 우선안의 장점은 다음과 같다.

- 저장 포맷이 UI 구현에 덜 종속된다.
- 현재/과거 버전 렌더링을 공통 구조로 처리하기 쉽다.
- 장애 복원력과 확장성이 높다.
- 이후 diff/export/AI summary 기능으로 확장하기 좋다.
