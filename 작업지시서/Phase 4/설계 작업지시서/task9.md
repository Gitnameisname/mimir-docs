# Task 4-9 작업지시서

## 작업명

구현 우선순위 및 MVP 범위 확정

---

## 1. 작업 목적

Phase 4의 아홉 번째 작업으로, 지금까지 정의한 문서 작성/개정/버전 관리 코어 설계를 바탕으로 **실제 구현 착수 순서와 MVP 범위를 확정**한다.

이번 작업의 목적은 다음과 같다.

- Phase 4의 설계 산출물을 실제 개발 가능한 작업 묶음으로 재정렬한다.
- 필수 기능과 확장 기능을 구분하여 초기 구현 복잡도를 통제한다.
- backend 우선 구현 범위를 명확히 하여 코어 동작을 먼저 안정화한다.
- 이후 User UI / Admin UI가 어떤 API와 상태 모델 위에서 연결될지 기준선을 만든다.
- 테스트 우선순위를 정리하여 구현 초기에 깨지기 쉬운 핵심 흐름부터 검증할 수 있도록 한다.

이 문서는 구현 단계의 실제 착수 계획, 작업 분할, 테스트 계획, UI 연계 순서의 기준 문서가 되어야 한다.

---

## 2. 작업 배경

앞선 작업에서 다음이 이미 정리되었다고 가정한다.

- Task 4-1: 문서 생성/개정/발행/복원의 생명주기
- Task 4-2: Version 모델 상세 설계
- Task 4-3: 문서 저장 구조와 편집 단위 설계
- Task 4-4: 문서 작성/수정 API 설계
- Task 4-5: Draft / Published 상태 관리 규칙 설계
- Task 4-6: 버전 조회, 이력 탐색, 복원 흐름 설계
- Task 4-7: 렌더링 파이프라인 기초 설계
- Task 4-8: 권한/감사와의 연계 설계

즉, 이제 Phase 4는 설계 자체보다 **무엇부터 구현해야 실패 위험이 낮고, 어떤 기능까지를 MVP로 볼지**를 정리해야 하는 단계다.

이번 작업에서는 다음 질문들에 답할 수 있어야 한다.

- 가장 먼저 구현해야 하는 최소 백엔드 코어는 무엇인가?
- 지금 당장 넣지 않아도 되는 기능은 무엇인가?
- 어떤 시나리오가 MVP를 성립시키는가?
- UI는 언제 어떤 수준까지 붙이면 되는가?
- 테스트는 어떤 흐름부터 검증해야 하는가?

## 4. 핵심 설계 질문

이번 작업에서는 최소한 아래 질문들에 답할 수 있어야 한다.

1. 문서 생성, Draft 저장, Publish, 버전 조회, Restore 중 어떤 순서로 구현해야 하는가?
2. 저장 구조와 Version 모델이 안정되기 전에는 어떤 UI 작업을 미루는 것이 맞는가?
3. MVP에서 반드시 포함해야 하는 최소 사용자 성공 시나리오는 무엇인가?
4. Draft discard, restore conflict handling, fine-grained permissions 같은 기능은 MVP 필수인가?
5. 렌더링 API와 일반 도메인 조회 API 중 무엇을 먼저 구현해야 하는가?
6. backend 우선 구현 시 필요한 최소 데이터 모델과 서비스 계층은 무엇인가?
7. 권한/감사 연동은 MVP에서 어디까지 포함해야 하는가?
8. 테스트는 어떤 흐름부터 우선 자동화해야 하는가?
9. Admin/Operator UI는 MVP에 포함할 것인가, 후속 단계로 둘 것인가?
10. 이후 diff, approval, import/export 같은 기능과 경계를 어떻게 나눌 것인가?

먼저 지금까지 정의된 기능을 구현 관점에서 다시 분류한다.

- 문서 생성
- 문서 메타데이터 수정
- Draft 생성/조회/저장
- Publish
- 버전 목록 조회
- 버전 상세 조회
- Restore

다음 두 그룹으로 기능을 나눈다.

#### 확장 기능

- Draft discard
n- restore conflict 세부 처리 고도화
- 미지원 블록 고급 fallback
- 세분화된 버전 필터링
- 상세 감사 이벤트 확장
- 관리자용 고급 조회 기능
- render warning 집계 고도화

각 기능을 왜 필수/확장으로 분류했는지 근거를 적는다.

### 5-3. MVP 목표 정의

Phase 4 MVP가 어떤 상태를 의미하는지 한 문단과 항목 목록으로 명확히 정의한다.

반드시 다음 질문에 답한다.

- MVP가 끝나면 사용자는 무엇을 할 수 있어야 하는가?
- MVP가 끝나도 아직 못 하는 것은 무엇인가?
- 어떤 수준이면 “워드 기반 문서 관리 일부를 대체할 수 있는 초기 코어”라고 볼 수 있는가?

권장 방향은, MVP를 다음 수준으로 정의하는 것이다.

- 문서를 생성할 수 있다.
- Draft를 저장할 수 있다.
- Published를 만들 수 있다.
- 현재 Published와 Draft를 구분해서 조회할 수 있다.
- 버전 목록과 특정 버전을 조회할 수 있다.
- 과거 Published 또는 허용된 버전을 기반으로 새 Draft를 복원할 수 있다.
- 기본 권한 검사와 기본 감사 이벤트가 동작한다.

실제 채택안은 문서에 명시한다.

---

### 5-4. MVP 사용자 시나리오 정의

최소한 아래 시나리오를 MVP 기준 시나리오로 검토한다.

1. 새 문서 생성 → Draft 저장
2. Draft 편집 반복 → Publish
3. Published 문서 조회
4. Published 이후 새 Draft 생성 및 수정
5. 버전 목록 조회 → 특정 버전 상세 확인
6. 특정 과거 버전 기준 Restore → 새 Draft 생성
7. 권한 없는 Publish 시도 차단
8. Publish / Restore 감사 이벤트 기록

각 시나리오에 대해 다음을 정리한다.

- 시작 상태
- 사용자 행위
- 시스템 결과
- 관련 API
- 관련 테스트 필요 여부
- MVP 필수 여부

이 항목은 테스트 우선순위와 직접 연결되어야 한다.

### 5-5. backend 우선 구현 범위 정의

구현 착수 시 backend에서 먼저 완성해야 하는 최소 범위를 정리한다.

최소 검토 묶음:

#### 1단계 백엔드 코어
- Document / Version 핵심 모델
- content_snapshot 저장 구조
- current_draft / current_published 포인터

#### 2단계 서비스 로직
- 문서 생성
- Draft 저장
- Publish
- 버전 목록/상세 조회
- Restore

#### 3단계 횡단 관심사
- 권한 검사
- 감사 이벤트 적재
- validation
- 예외 처리

#### 4단계 렌더링 계층
- render ViewModel 변환
- `/render` 응답 구성
- fallback / warning 기본 처리

각 단계를 왜 այդ 순서로 구현하는 것이 적절한지 설명한다.

### 5-6. UI 연결 포인트 정의

backend 우선 구현 뒤 어떤 UI를 어떤 순서로 연결할지 정리한다.

최소 검토 대상:

- 사용자용 문서 생성 화면
- Draft 편집 화면
- Published 읽기 화면
- 버전 히스토리 목록 화면
- 버전 상세/복원 진입 화면
- 관리자/운영자용 감사 추적 화면은 MVP 포함 여부 검토

다음 질문에도 답한다.

- UI는 처음부터 전체 편집 UI가 필요한가, 아니면 JSON 기반/단순 편집 화면으로도 MVP 검증이 가능한가?
- 렌더링 읽기 화면과 편집 화면 중 무엇을 먼저 붙여야 하는가?
- Admin/Operator UI는 MVP 이후로 미루는 것이 맞는가?

이번 항목은 UI 상세 설계가 아니라, **어떤 연결 포인트가 구현 순서상 필요한지**를 정리하는 것이다.

## 7. 산출물 작성 원칙

- 구현 코드가 아니라 구현 착수용 계획 문서 작성에 집중한다.
- Task 4-1 ~ 4-8과 모순되지 않도록 설계 결과를 그대로 반영한다.
- “좋으면 다 넣는다”가 아니라 MVP 성립에 필요한 최소 범위를 중심으로 판단한다.
- backend 선행 원칙을 유지하되, UI 검증 포인트가 전혀 빠지지 않도록 한다.
- 테스트 우선순위는 실제 실패 비용이 큰 흐름을 먼저 잡는 방식으로 작성한다.
- 제외 항목은 अस्पष्ट하게 남기지 말고 명시적으로 선을 긋는다.

## 8. 권장 방향

아래 방향을 우선안으로 검토한다.

### 우선안 A

- MVP는 **문서 생성 → Draft 저장 → Publish → 버전 조회 → Restore → 렌더링 조회**까지를 핵심 범위로 본다.
- 구현은 **backend core부터 우선** 진행하고, UI는 최소 검증 가능한 수준으로 후행 연결한다.
- 권한/감사는 고급 운영 기능까지 전부 넣지 말고, **기본 권한 검사와 핵심 감사 이벤트**만 MVP에 포함한다.
- Draft discard는 가능하면 MVP에 포함하되, 고급 충돌 처리까지는 확장 기능으로 미룰 수 있다.
- 다중 Draft, 승인 워크플로, diff, import/export, 고급 운영 UI는 MVP에서 제외한다.
- 테스트는 Publish/Restore/권한 실패/버전 조회 같은 **상태 전이 핵심 흐름**부터 우선 검증한다.

이 우선안의 장점은 다음과 같다.

- 구현 범위가 과도하게 커지지 않는다.
- 문서 플랫폼 코어의 가치가 가장 빨리 검증된다.
- 저장 구조, 상태 정책, API, 렌더링, 권한/감사의 핵심 축이 먼저 안정화된다.
- 이후 UI 확장과 고급 기능 추가가 쉬워진다.
