

# Task 4-5 작업지시서

## 작업명

Draft / Published 상태 관리 규칙 설계

---

## 1. 작업 목적

Phase 4의 다섯 번째 작업으로, 문서 플랫폼에서 **Draft와 Published 상태를 어떻게 운영할지**에 대한 관리 규칙을 상세 설계한다.

이번 작업의 목적은 다음과 같다.

- Task 4-1에서 정의한 문서 생명주기를 운영 정책 수준으로 구체화한다.
- Task 4-2의 Version 모델과 Task 4-4의 API 설계가 흔들리지 않도록 Draft / Published 상태 규칙을 확정한다.
- 문서 작성 중 상태와 공식 배포 상태를 명확히 분리하여 버전 무결성과 책임성을 확보한다.
- 향후 승인 워크플로, 비교(diff), 감사 추적, 권한 제어 확장에 견딜 수 있는 상태 관리 기준선을 마련한다.
- 구현 단계에서 “어떤 상태에서 무엇이 가능한가”를 명확히 하여 서비스 로직과 예외 처리를 단순화한다.

이 문서는 이후 Task 4-6, Task 4-7 및 구현 단계의 상태 전이 로직, 권한 로직, 테스트 케이스 설계의 기준 문서가 되어야 한다.

---

## 2. 작업 배경

앞선 작업에서 다음이 이미 정리되었다고 가정한다.

- Task 4-1: 문서 생성/개정/발행/복원의 생명주기와 상태 전이
- Task 4-2: Version 모델 상세 설계
- Task 4-4: 문서 작성/수정 API 설계
- Phase 2: 역할, 권한, 감사 추적 원칙
- Phase 3: API 공통 규약과 상태 전이 액션 설계 기준

이제 필요한 것은, Draft와 Published를 단순한 상태값으로 두는 수준이 아니라, 실제 운영에서 다음 질문에 답할 수 있는 정책으로 정리하는 일이다.

- 문서당 Draft는 몇 개까지 허용하는가?
- Published는 항상 1개만 유지되는가?
- Published 문서를 수정할 때 새 Draft는 어떻게 생성되는가?
- 기존 Draft가 있을 때 Restore를 허용하는가?
- Draft 폐기와 Published 교체는 어떤 조건에서 가능한가?
- 조회 기준은 최신 Draft인가, 최신 Published인가?

즉, 이번 작업은 **상태값 정의가 아니라 운영 가능한 상태 관리 정책을 설계하는 작업**이다.

## 4. 핵심 설계 질문

이번 작업에서는 최소한 아래 질문들에 답할 수 있어야 한다.

1. 문서당 Draft는 하나만 허용하는가, 여러 개를 허용하는가?
2. Published는 문서당 항상 1개만 공식 상태를 가질 수 있는가?
3. Published가 없는 문서는 어떤 상태로 간주하는가?
4. Draft와 Published가 동시에 존재할 수 있는가?
5. Published 문서를 수정하면 기존 Published를 직접 수정하는가, 새 Draft를 생성하는가?
6. 현재 Draft가 있을 때 새 Draft 생성 요청은 어떻게 처리하는가?
7. 기존 Draft가 있을 때 Restore 요청은 허용하는가?
8. Publish 시 기존 Published는 어떤 상태로 바뀌는가?
9. Draft 폐기(discard)는 어떤 조건에서 가능한가?
10. 일반 사용자, 편집자, 발행자, 관리자 등 권한 주체별로 어떤 상태 전이가 허용되는가?
11. “현재 문서 조회” 기본값은 Draft 기준인가 Published 기준인가?
12. 아직 Published되지 않은 문서의 일반 조회 정책은 무엇인가?

## 5. 작업 지시

아래 항목을 순서대로 설계 문서로 정리한다.

### 5-1. Draft와 Published의 개념적 책임 구분

먼저 Draft와 Published가 각각 무엇을 의미하는지 명확히 정의한다.

반드시 다음을 정리한다.

- Draft의 정의
- Published의 정의
- 두 상태의 책임 차이
- 사용자 관점 의미
- 시스템 관점 의미
- 감사 추적 관점 의미

중요: Draft는 작업 중 버전이고, Published는 공식 참조 버전이라는 점을 정책 수준에서 분명히 적는다.

---

### 5-2. 문서당 Draft 보유 정책 정의

문서 하나에 대해 Draft를 몇 개까지 허용할지 정책을 정한다.

다음 두 안을 비교 검토한다.

#### 안 A. 문서당 단일 Draft 허용

- 운영 단순성 높음
- 충돌과 혼선이 적음
- 이후 다중 브랜치형 편집에는 제약

#### 안 B. 문서당 다중 Draft 허용

- 유연성 높음
- 승인/분기 시나리오 확장 가능
- 초기 구현/운영 복잡도 증가

각 안의 장단점을 적고, 이번 Phase 기준 권장안을 제시한다.

반드시 아래 상황도 함께 정리한다.

- Draft가 이미 있을 때 새 Draft 생성 요청 처리
- Draft 생성 주체와 소유 개념 필요 여부
- Draft 잠금(lock) 또는 점유 개념을 이번 Phase에 둘지 여부

---

### 5-3. Published 보유 정책 정의

Published는 공식 참조 상태이므로, 문서당 몇 개를 허용할지 명확히 정리한다.

반드시 다음을 포함한다.

- Published는 문서당 1개인지 여부
- 새 Published 생성 시 기존 Published 처리 방식
- 과거 Published 버전의 상태값 변화 필요 여부
- “현재 공식 버전” 포인터 필요 여부
- 일반 조회가 기본적으로 참조할 대상

권장 방향은, 문서당 현재 공식 Published는 1개만 유지하고, 과거 Published는 이력으로 남기되 현재 공식 상태에서 제외하는 방식이다.

---

### 5-4. Draft와 Published 공존 정책 정의

Draft와 Published가 동시에 존재할 수 있는지, 존재한다면 어떤 의미를 가지는지 정의한다.

다음 상황을 기준으로 정리한다.

- 이미 Published가 있는 문서에서 수정 시작
- Published 이후 Draft 저장 반복
- Draft가 존재하는 동안 일반 조회
- Draft가 존재하는 동안 편집자 조회
- Draft가 존재하는 동안 재발행 시도

반드시 아래 질문에 답한다.

- Draft + Published 공존 상태를 공식 상태로 인정하는가?
- 공존 시 `current_published_version_id`와 `current_draft_version_id`를 동시에 유지하는가?
- 어떤 사용자에게 어떤 상태가 기본 노출되는가?

---

### 5-5. Publish 전환 시 상태 처리 규칙 정의

Draft가 Published로 전환될 때 어떤 상태 변화가 일어나는지 명확히 정리한다.

반드시 다음을 포함한다.

- Publish 전 선행 조건
- Publish 성공 시 Draft 처리 방식
- 기존 Published 처리 방식
- 현재 포인터 갱신 방식
- Publish 실패 시 롤백 관점에서 고려할 포인트
- Publish 직후 감사 이벤트 포인트

다음 두 방향도 비교 검토한다.

#### 안 A. Draft를 그대로 Published 상태로 승격

- 단순함
- 상태 전이 추적이 직관적일 수 있음

#### 안 B. Publish 시 별도 Published Version 생성 또는 명시적 전환 기록 유지

- 이력 표현이 더 명확할 수 있음
- 모델 복잡도 증가 가능

Task 4-2 Version 모델과 충돌하지 않도록 이번 Phase 기준 권장안을 명시한다.

---

### 5-6. Draft 폐기(discard) 정책 정의

Draft는 항상 발행되는 것이 아니므로, 폐기 정책이 필요하다.

반드시 다음을 정리한다.

- Draft 폐기의 정의
- Draft 폐기 가능 조건
- Published가 없는 문서에서 Draft 폐기 시 문서 전체 처리 방식
- Published가 있는 문서에서 Draft 폐기 시 결과 상태
- 폐기 후 복구 가능성 검토
- 감사 이벤트 필요 여부

중요: Draft 폐기는 Published 삭제와 다른 개념이라는 점을 명확히 적는다.

---

### 5-7. Restore와 Draft 충돌 정책 정의

복원은 보통 새 Draft를 만드는 흐름이므로, 기존 Draft와 충돌할 수 있다.

다음 상황을 반드시 검토한다.

- 기존 Draft 없음 + Restore 요청
- 기존 Draft 있음 + Restore 요청
- 기존 Draft가 있는데 Restore로 대체 허용 여부
- Restore 전 기존 Draft 폐기 요구 여부
- Restore 결과 Draft와 Published 공존 상태 처리

다음 두 안을 비교 검토한다.

#### 안 A. 기존 Draft가 있으면 Restore 금지

- 단순하고 안전함
- 사용자 편의성은 낮을 수 있음

#### 안 B. Restore 시 기존 Draft를 교체하거나 별도 정책 허용

- 유연성 높음
- 상태 혼선 가능성 증가

이번 Phase 기준 권장안을 명시한다.

---

### 5-8. 현재 조회 기준 정책 정의

문서 조회 시 무엇을 기본으로 보여줄지 정책을 정의한다.

반드시 다음을 구분한다.

- 일반 사용자 기본 조회
- 편집 권한 사용자 기본 조회
- 운영자/관리자 조회
- API 기본 조회
- 렌더링 조회

검토 대상:

- 일반 조회는 current_published 기준
- 편집 조회는 필요 시 current_draft 우선 또는 명시적 조회 필요
- 아직 Published가 없는 문서의 조회 정책
- Draft만 있는 문서의 접근 제한 정책

이번 항목은 Task 4-4 API 설계와 직접 연결되므로, URL/응답 관점까지 고려한 정책 문장으로 정리한다.

---

### 5-9. 상태별 허용/금지 행위 표 작성

상태별로 어떤 행위가 가능한지 표로 정리한다.

최소 검토 상태:

- Draft only
- Published only
- Draft + Published
- Archived 필요 여부
- Deleted 또는 Soft deleted 필요 여부

최소 검토 행위:

- Draft 수정
- Draft 폐기
- Publish
- Restore
- 일반 조회
- 편집 조회
- 메타데이터 수정
- 삭제/보관

표에는 최소한 다음 열이 포함되어야 한다.

- 현재 상태
- 수행 행위
- 허용 여부
- 수행 주체 조건
- 결과 상태
- 비고

---

### 5-10. 권한별 상태 전이 제한 정리

상태 전이는 권한과 결합되므로, 역할별 제한을 명시한다.

최소 검토 역할:

- viewer
- editor
- publisher
- admin

각 역할에 대해 다음을 정리한다.

- 조회 가능 범위
- Draft 생성 가능 여부
- Draft 수정 가능 여부
- Publish 가능 여부
- Restore 가능 여부
- Draft 폐기 가능 여부
- Published 삭제/보관 가능 여부

중요: 이번 작업은 상세 RBAC 구현이 아니라, **상태 전이와 직접 연결되는 권한 포인트**만 정리한다.

---

### 5-11. 예외 및 충돌 시나리오 정리

운영상 자주 문제가 될 수 있는 예외 상황을 정리한다.

최소 검토 대상:

- Draft가 없는데 Publish 요청
- Published가 없는데 일반 공개 조회 요청
- 기존 Draft가 있는데 새 Draft 생성 요청
- 기존 Draft가 있는데 Restore 요청
- 폐기된 Draft에 대한 수정 요청
- Archived 또는 삭제 처리된 문서에 대한 Publish 요청
- 권한 없는 사용자의 상태 전이 요청

각 시나리오에 대해 다음을 작성한다.

- 상황 설명
- 허용/금지 여부
- 권장 처리 정책
- API 에러로 연결될 포인트
- 감사 이벤트 필요 여부

---

### 5-12. 권장 운영 정책안 도출

최종적으로 이번 Phase에서 채택할 운영 정책안을 한 번에 정리한다.

최소한 아래 항목을 포함한다.

- Draft 개수 정책
- Published 개수 정책
- Publish 시 기존 Published 처리
- Restore와 기존 Draft 충돌 정책
- 기본 조회 기준
- Draft 폐기 기준
- 권한별 핵심 전이 제한

이 항목은 후속 구현자가 한눈에 이해할 수 있는 **요약 정책 표 또는 결정사항 목록**으로 정리한다.

## 7. 산출물 작성 원칙

- 구현 코드가 아니라 정책/설계 문서 작성에 집중한다.
- Task 4-1, 4-2, 4-4와 용어를 일관되게 유지한다.
- 초기 구현 단순성을 우선하되, 향후 승인/분기 편집 확장을 막지 않는 방향으로 작성한다.
- 상태값 정의와 운영 규칙을 혼동하지 않도록 구분해서 서술한다.
- API 관점, 사용자 관점, 운영 관점을 섞지 말고 필요한 곳에서 구분해서 설명한다.
- “권장안”과 “대안”을 비교한 뒤 최종 결정을 명시한다.

## 8. 권장 방향

아래 방향을 우선안으로 검토한다.

### 우선안 A

- 문서당 **활성 Draft는 1개만 허용**한다.
- 문서당 **현재 공식 Published는 1개만 유지**한다.
- Published 문서를 수정할 때는 기존 Published를 직접 바꾸지 않고 **새 Draft를 생성**한다.
- Draft와 Published는 동시에 공존할 수 있다.
- 일반 조회 기본값은 `current_published`다.
- 편집 작업은 `current_draft`를 명시적으로 조회/저장하는 방식으로 운영한다.
- 기존 Draft가 있는 상태에서는 추가 Draft 생성 또는 Restore를 제한하거나, 명시적 정리 절차를 선행하도록 한다.
- Draft 폐기는 명시적 액션으로 처리하며, Published 삭제와 분리한다.
- Publish는 기존 Published를 현재 공식 상태에서 교체하고, 과거 Published는 이력으로 유지한다.

이 우선안의 장점은 다음과 같다.

- 상태 관리가 단순하고 예측 가능하다.
- UI와 API 동작이 명확해진다.
- 권한과 감사 추적 포인트가 분명하다.
- 이후 승인 워크플로나 다중 Draft 확장을 별도 Phase로 분리하기 쉽다.
