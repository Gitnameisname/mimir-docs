# Task 4-8 작업지시서

## 작업명

권한/감사와의 연계 설계

---

## 1. 작업 목적

Phase 4의 여덟 번째 작업으로, 문서 작성/수정/발행/복원/조회 행위를 **기존 권한 체계와 감사 추적 체계에 안정적으로 연결하는 정책과 구조**를 설계한다.

이번 작업의 목적은 다음과 같다.

- Phase 2에서 정의한 역할/권한/감사 원칙을 Phase 4의 문서 생명주기와 직접 연결한다.
- 작성, 발행, 복원, 버전 조회 같은 핵심 행위에 대해 누가 무엇을 할 수 있는지 명확히 한다.
- Draft / Published / Restore 흐름에서 보안상 민감한 상태 전이 포인트를 분리하고 통제 기준을 정한다.
- 이후 구현 단계에서 API 권한 검사, 서비스 레벨 권한 enforcement, audit event 기록 구조가 흔들리지 않도록 기준선을 마련한다.
- 문서 변경 이력에 대해 “누가, 언제, 무엇을, 어떤 근거로” 변경했는지 추적 가능한 최소 기록 모델을 정리한다.

이 문서는 구현 단계의 권한 체크 로직, 감사 이벤트 적재 로직, 운영자 추적 화면, 테스트 케이스 설계의 기준 문서가 되어야 한다.

---

## 2. 작업 배경

앞선 작업에서 다음이 이미 정리되었다고 가정한다.

- Phase 2: 사용자/조직/역할 모델, ACL 기반 권한 구조, 감사 로그 체계
- Phase 3: API 버전 관리, 공통 응답 포맷, 인증/인가 구조, 상태 전이 액션 API 원칙
- Task 4-1: 문서 생성/개정/발행/복원의 생명주기
- Task 4-2: Version 모델 상세 설계
- Task 4-4: 문서 작성/수정 API 설계
- Task 4-5: Draft / Published 상태 관리 규칙
- Task 4-6: 버전 조회, 이력 탐색, 복원 흐름 설계

이제 필요한 것은 문서 플랫폼에서 다음 질문들에 답할 수 있는 구체적인 연계 규칙이다.

- 문서를 볼 수 있는 사람과 수정할 수 있는 사람은 같은가?
- 작성 권한과 발행 권한을 분리할 것인가?
- 과거 버전 조회는 일반 조회보다 더 넓거나 더 좁은 권한을 가져야 하는가?
- 복원은 누가 할 수 있는가?
- 어떤 행위가 감사 로그에 반드시 남아야 하는가?
- 감사 로그는 행위 결과만 남기면 되는가, 아니면 상태 전이 전후 맥락도 포함해야 하는가?

즉, 이번 작업은 단순히 권한 이름을 나열하는 것이 아니라, **문서 생명주기 행위와 보안/감사 체계를 결합하는 설계 작업**이다.

### 포함 범위

- 문서 작성/수정/발행/복원/조회 행위별 권한 포인트 정의
- 작성 권한과 발행 권한 분리 여부 검토
- 버전 목록/상세/렌더링 조회 권한 범위 정의
- 복원 권한 정의
- Draft 폐기 권한 정의
- 상태 전이별 감사 이벤트 정의
- 감사 이벤트 최소 필드 구조 정의
- 누가/언제/무엇을/어떤 상태에서 변경했는지 기록 기준 정의
- 권한 실패와 상태 전이 실패 시 감사 기록 필요 여부 정의
- 운영자 추적/검수 관점에서 필요한 감사 정보 범위 정리

### 제외 범위

- 실제 RBAC/ACL 코드 구현
- 실제 감사 로그 DB 스키마 구현

- 상세 개인정보 마스킹 정책 구현
- 승인 워크플로 상세 권한 단계 설계
- 실시간 알림/이벤트 브로커 구현

이번 작업은 **문서 행위에 대한 권한/감사 연계 규칙 설계**에 집중한다.

## 4. 핵심 설계 질문

이번 작업에서는 최소한 아래 질문들에 답할 수 있어야 한다.

1. 문서 조회 권한과 Draft 조회 권한은 동일한가?
2. 문서 수정 권한과 발행 권한은 분리해야 하는가?
3. 버전 목록 조회 권한과 버전 상세 content 조회 권한은 같아야 하는가?
4. 복원 권한은 편집 권한만 있으면 충분한가, 발행자 수준 이상이 필요한가?
5. Draft 폐기(discard)는 작성자만 가능한가, 관리자도 가능한가?
6. Published 문서의 일반 조회 권한과 내부 편집용 현재 Draft 조회 권한은 어떻게 분리되는가?
7. 감사 로그는 성공한 상태 전이만 기록하는가, 실패한 시도도 기록하는가?
8. 감사 로그에는 before/after 상태가 모두 필요할까?
9. change_summary와 감사 이벤트는 같은 것인가, 별도로 관리해야 하는가?
10. 누가/언제/무엇을 변경했는지 외에, 어떤 문맥(document_id, version_id, prior_state 등)을 남겨야 사후 추적이 가능한가?
11. 운영자 화면에서 볼 사용자용 이력과 내부 감사 로그는 같은 데이터를 재사용할 수 있는가?
12. 이후 승인 워크플로가 붙더라도 현재 권한/감사 구조가 확장 가능한가?

## 5. 작업 지시

아래 항목을 순서대로 설계 문서로 정리한다.

### 5-1. 문서 행위별 권한 포인트 분해

먼저 문서 플랫폼에서 실제 권한 검사가 필요한 행위를 분해한다.

최소 검토 대상:

- 문서 생성
- 문서 메타데이터 조회
- 문서 메타데이터 수정
- 현재 Published 조회
- 현재 Draft 조회
- Draft 생성
- Draft 수정
- Draft 폐기
- Publish
- 버전 목록 조회
- 버전 상세 조회
- 특정 버전 렌더링 조회
- Restore
- 문서 보관/삭제 필요 여부

각 행위에 대해 다음을 정리한다.

- 행위명
- 대상 리소스
- 민감도 수준
- 권한 검사 필요 위치(API / 서비스 / 둘 다)
- 감사 이벤트 필요 여부

중요: 이 단계에서 “CRUD” 수준이 아니라 **실제 상태 전이와 운영 행위 중심으로 분해**해야 한다.

---

### 5-2. 작성 권한과 발행 권한 분리 여부 검토

다음 두 방향을 비교 검토한다.

#### 안 A. 작성 권한과 발행 권한 분리

- editor는 Draft 생성/수정 가능
- publisher 또는 admin만 Publish 가능
- 책임 분리가 명확함
- 운영 복잡도 증가 가능

#### 안 B. 작성 권한과 발행 권한 통합

- 초기 운영 단순성 높음
- 작은 팀에 적합할 수 있음
- 통제력과 책임 추적은 약해질 수 있음

각 안의 장단점을 다음 관점으로 정리한다.

- 보안 통제 수준
- 운영 단순성
- 향후 승인 워크플로 확장성
- 감사 추적 명확성
- 조직 규모 확장 대응성

이번 Phase 기준 권장안을 명시한다.

---

### 5-3. 버전 조회 권한 범위 정의

버전 이력은 민감할 수 있으므로 조회 범위를 세분화해서 정리한다.

반드시 다음을 구분한다.

- 현재 Published 조회 권한
- 현재 Draft 조회 권한
- 버전 목록 조회 권한
- 버전 상세 메타정보 조회 권한
- 버전 상세 본문(content_snapshot) 조회 권한
- 렌더링 조회 권한
- 복원 가능 여부 정보 조회 권한

다음 질문에도 답한다.

- 일반 viewer는 과거 Published 버전 목록을 볼 수 있는가?
- editor는 Draft 포함 이력을 볼 수 있는가?
- 내부 운영자(admin)는 폐기된 Draft/복원 이력까지 볼 수 있는가?
- 동일 문서라도 공개 범위에 따라 권한이 달라질 수 있는가?

이번 항목은 Task 4-6과 직접 연결되므로, 버전 탐색 정책과 충돌하지 않도록 정리한다.

---

### 5-4. 복원 권한 정의

Restore는 민감한 행위이므로 별도 권한 규칙이 필요하다.

반드시 다음을 검토한다.

- editor에게 restore 허용 여부
- publisher만 restore 허용할지 여부
- admin override 허용 여부
- 복원 대상 버전에 따른 권한 차등 필요 여부
- 현재 Draft 존재 시 restore 권한 외 추가 제약 필요 여부
- restore 후 새 Draft 접근 권한과의 관계

다음 두 안을 비교 검토한다.

#### 안 A. 복원은 publisher 이상만 허용

- 운영 안정성 높음
- 권한 모델이 명확함
- 편집자 자율성은 줄어들 수 있음

#### 안 B. 편집자도 복원 가능하되 Publish는 별도 권한

- 작업 유연성 높음
- 복원 후 검수/발행 절차와 결합 필요 가능성 있음

이번 Phase 기준 권장안을 선택 근거와 함께 정리한다.

### 5-5. 권한 연계 표 작성

최종적으로 역할별 허용 행위를 표로 정리한다.

최소 검토 역할:

- viewer
- editor
- publisher
- admin

최소 검토 행위:

- 문서 생성
- Draft 조회
- Draft 수정
- Draft 폐기
- Publish
- 현재 Published 조회
- 버전 목록 조회
- 버전 상세 조회
- Restore
- 보관/삭제 필요 여부

표에는 최소한 다음 열을 포함한다.

- 역할
- 행위
- 허용 여부
- 조건
- 비고

가능하면 추가 열로 다음도 포함한다.

- 감사 이벤트 필수 여부
- 상태 제약 연계 여부

이 표는 구현 단계 권한 체크와 테스트 케이스의 기준이 되어야 한다.

---

### 5-6. 감사 이벤트 분류 체계 정의

문서 플랫폼에서 어떤 이벤트를 감사 로그로 남길지 분류한다.

최소 검토 이벤트 타입:

- document_created
- document_metadata_updated
- draft_created
- draft_updated
- draft_discarded
- publish_requested 필요 여부
- document_published
- version_viewed 필요 여부
- version_restored
- access_denied
- state_transition_denied
- document_archived 또는 deleted 필요 여부

각 이벤트 타입에 대해 다음을 정리한다.

- 이벤트명
- 의미
- 발생 조건
- 성공/실패 이벤트 구분 필요 여부
- 사용자 표시 이력과 내부 감사 로그 공용 여부

중요: 너무 많은 이벤트를 만들지 말고, 사후 추적과 운영 통제에 필요한 핵심 이벤트를 우선 정의한다.

---

### 5-7. 감사 이벤트 최소 필드 구조 정의

각 감사 이벤트에 공통으로 포함되어야 할 최소 필드를 정의한다.

최소 검토 필드:

- event_id
- event_type
- occurred_at
- actor_user_id
- actor_role 또는 authority_context
- tenant_id / organization_id 필요 여부
- document_id
- version_id 필요 여부
- target_version_id 필요 여부
- previous_state
- new_state
- action_result
- reason 또는 note
- change_summary 참조 여부
- request_id / correlation_id
- source_ip / client_info 필요 여부

각 필드에 대해 다음을 정리한다.

- 목적
- 필수 여부
- 생성 시점
- 개인정보/민감정보 고려 필요 여부

권장 방향은, 최소한 **행위 주체, 시각, 대상, 상태 전후, 결과**를 재구성할 수 있어야 한다는 점이다.

---

### 5-8. 변경 기록 구조 정의

“누가/언제/무엇을 변경했는가”를 어떤 방식으로 남길지 정리한다.

반드시 다음을 구분해서 작성한다.

- 사용자에게 보이는 문서 이력(change log)
- 내부 운영/감사 목적의 audit log
- Version 자체가 가지는 change_summary
- 상태 전이 이벤트가 가지는 before/after 정보

다음 질문에도 답한다.

- change_summary는 사용자가 작성한 설명이고, audit log는 시스템 사실 기록으로 분리해야 하는가?
- 본문 전체 diff가 없더라도 최소 어떤 변경 설명을 남길 수 있어야 하는가?
- 문서 메타데이터 수정과 본문 수정은 같은 수준으로 기록해야 하는가?

이 항목은 사용자용 히스토리와 내부 감사 로그를 혼동하지 않도록 명확히 구분한다.

---

### 5-9. 성공/실패 행위의 감사 기록 정책 정의

감사 로그는 성공한 행위만 남길지, 실패 시도도 남길지 정의한다.

최소 검토 시나리오:

- 권한 없이 Publish 시도
- 권한 없이 Restore 시도
- Draft 없음 상태에서 Publish 시도
- 이미 Draft가 있는 상태에서 중복 Restore 시도
- 존재하지 않는 버전 조회 시도

다음 두 안을 비교 검토한다.

#### 안 A. 성공 이벤트만 기록

- 단순함
- 보안 추적성은 약해질 수 있음

#### 안 B. 민감 행위의 실패 시도도 기록

- 보안/운영 추적성 향상
- 로그량 증가

이번 Phase 기준 권장안을 정리하고, 어떤 실패 시도만 기록할지 범위를 명시한다.

---

### 5-10. 권한 실패와 상태 실패의 구분 정의

시스템은 “권한이 없어서 실패”와 “상태가 맞지 않아 실패”를 구분해야 한다.

최소 검토 시나리오:

- editor가 publish 요청 → 권한 실패
- publisher가 draft 없는 문서에 publish 요청 → 상태 실패
- restore 가능한 역할이지만 현재 Draft 존재로 restore 실패 → 상태/정책 실패

각 경우에 대해 다음을 정리한다.

- 분류 기준
- API 에러 코드 구분 필요 여부
- 감사 이벤트 구분 필요 여부
- 운영자 추적 시 표시 방식 차이

이 항목은 보안 추적성과 디버깅 가시성을 높이기 위한 핵심 규칙이다.

---

### 5-11. 운영자/감사 관점 조회 요구사항 정리

운영자나 감사 담당자가 나중에 사건을 추적할 때 필요한 최소 정보 범위를 정리한다.

최소 검토 질문:

- 누가 어떤 문서에 어떤 행위를 했는지 한눈에 볼 수 있어야 하는가?
- 특정 version_id 기준으로 publish/restore 연쇄 이력을 따라갈 수 있어야 하는가?
- access_denied와 state_transition_denied를 별도로 검색 가능해야 하는가?
- 문서 단위 타임라인과 사용자 단위 타임라인이 모두 필요할 가능성이 있는가?

이번 항목은 상세 UI 설계가 아니라, **나중에 운영자 화면이 요구할 데이터 수준**을 정리하는 것이다.

## 8. 권장 방향

아래 방향을 우선안으로 검토한다.

### 우선안 A

- **작성 권한과 발행 권한은 분리**한다.
- `editor`는 Draft 생성/수정/폐기까지 가능하되, Publish는 `publisher` 이상으로 제한한다.
- 버전 조회는 Published 중심 조회와 Draft/내부 이력 조회를 구분한다.
- Restore는 `publisher` 이상 또는 `admin`에게 우선 허용하는 방향을 검토한다.
- 감사 이벤트는 문서 생성, Draft 수정, Draft 폐기, Publish, Restore, 권한 거부, 상태 전이 거부를 핵심 이벤트로 우선 정의한다.
- 감사 필드는 최소한 `actor`, `target(document/version)`, `occurred_at`, `previous_state`, `new_state`, `action_result`, `request_id`를 포함한다.
- 사용자용 change summary와 내부 audit log는 **분리**한다.
- 모든 실패를 기록하지는 않고, Publish/Restore/권한 거부 같은 **민감 행위 실패**를 우선 기록 대상으로 삼는다.

이 우선안의 장점은 다음과 같다.

- 문서 공식화(Publish) 권한이 명확히 통제된다.
- 사후 추적과 운영 감사가 가능해진다.
- 실패 원인이 권한 문제인지 상태 문제인지 분리 추적할 수 있다.
- 이후 승인 워크플로, 컴플라이언스, 운영자 콘솔로 확장하기 쉽다.

## 9. 완료 기준

다음 조건을 만족하면 이번 작업이 완료된 것으로 본다.

- 작성/발행/복원/조회 행위별 권한 포인트가 문서화되어 있다.
- 작성 권한과 발행 권한 분리 여부가 선택 근거와 함께 정리되어 있다.
- 버전 조회 권한 범위와 복원 권한 기준이 정의되어 있다.
- 역할별 권한 연계 표가 포함되어 있다.
- 필수 감사 이벤트 목록과 공통 필드 구조가 정리되어 있다.
- 사용자용 변경 이력과 내부 감사 로그의 차이가 설명되어 있다.
- 민감 행위 실패 기록 정책이 포함되어 있다.
- 구현 단계에서 권한 검사와 감사 이벤트 적재 로직 설계를 진행할 수 있을 정도로 충분히 구체적이다.

## 10. 금지사항

이번 작업에서는 아래를 하지 않는다.

- 실제 RBAC 미들웨어 구현
- 감사 로그 DB 스키마/마이그레이션 작성
- 외부 감사 시스템 연동 설계로 과도하게 확장
- 승인 워크플로 상세 단계까지 포함
- 운영자 UI 상세 화면 설계
- 개인정보 마스킹/보존기간 정책까지 과도하게 확장

범위를 넘기지 말고, 문서 행위에 대한 권한/감사 연계 규칙 설계에 집중한다.
