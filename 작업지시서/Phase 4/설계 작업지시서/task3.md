

# Task 4-3 작업지시서

## 작업명

문서 저장 구조와 편집 단위 설계

---

## 1. 작업 목적

Phase 4의 세 번째 작업으로, 문서 플랫폼에서 문서 본문을 어떤 구조로 저장하고, 어떤 단위로 수정하며, 어떤 형태로 Version 스냅샷에 포함할지를 상세 설계한다.

이번 작업의 목적은 다음과 같다.

- Node 기반 문서 표현을 실제 저장 구조 수준으로 구체화한다.
- 문서 전체 저장 방식과 부분 편집 가능성 사이의 기준선을 정한다.
- 렌더링, 버전 관리, 복원, 향후 diff 기능과 충돌하지 않는 저장 포맷을 정의한다.
- 특정 UI 에디터 구현에 과도하게 종속되지 않는 도메인 중심 저장 구조를 확정한다.

이 문서는 이후 Task 4-4, 4-6, 4-7 및 구현 단계의 문서 저장/조회 로직 설계의 기준 문서가 되어야 한다.

---

## 2. 작업 배경

Phase 1에서 Document / Version / Node 중심의 범용 문서 모델이 정의되었고, Task 4-1에서 문서 생명주기, Task 4-2에서 Version 모델 상세 설계가 정리되었다고 가정한다.

이제 실제로 결정해야 할 것은 다음이다.

- 문서 본문은 어떤 자료구조로 저장할 것인가?
- Node는 어떤 최소 공통 속성을 가져야 하는가?
- 문서 전체를 한 번에 저장할 것인가, 일부 블록만 수정 가능하게 설계할 것인가?
- 저장 구조와 렌더링 구조는 얼마나 동일해야 하는가?
- Version 스냅샷 안에 어떤 본문 구조가 포함되어야 복원 가능성을 보장하는가?

즉, 이번 작업은 에디터 UI 설계가 아니라, **문서 본문 데이터의 저장 규약과 편집 단위를 정의하는 작업**이다.

## 4. 핵심 설계 질문

이번 작업에서는 최소한 아래 질문들에 답할 수 있어야 한다.

1. 문서 본문은 트리 구조인가, 블록 리스트 구조인가, 혹은 혼합 구조인가?
2. Node는 Document 소속인가, Version 스냅샷 소속인가?
3. 모든 문서 타입이 공유하는 최소 공통 Node 스키마는 무엇인가?
4. 블록 타입별 세부 데이터는 공통 필드 + payload 방식으로 관리하는가?
5. 문서 저장은 항상 전체 문서 스냅샷 단위로 하는가?
6. 부분 수정 API를 고려하더라도 내부 저장 기준은 전체 저장이어야 하는가?
7. 제목, 본문, 표, 리스트, 부록, 인용문 같은 구조는 어떻게 표현하는가?
8. 렌더링용 정보와 저장용 정보는 어디까지 분리해야 하는가?
9. 유효성 검증은 Node 단위와 문서 전체 단위에서 각각 무엇을 확인해야 하는가?
10. 향후 diff, 템플릿, AI 개정 제안 기능을 고려할 때 어떤 구조가 가장 안정적인가?

## 5. 작업 지시

아래 항목을 순서대로 설계 문서로 정리한다.

### 5-1. 저장 구조의 기본 모델 정의

문서 본문을 어떤 기본 구조로 저장할지 먼저 정리한다.

다음 후보를 비교 검토한다.

1. **순수 트리 구조**
   - 문서 전체가 루트 Node 아래에 계층적으로 매달리는 구조
2. **블록 리스트 중심 구조**
   - 본문은 순차 블록 배열로 저장하고 일부 블록만 하위 자식을 가짐
3. **혼합 구조**
   - 문서 상위는 트리, 본문 일부는 블록 배열 또는 payload 내부 구조로 표현

각 안의 장단점을 작성하고, 이번 프로젝트에 적합한 권장안을 제시한다.

반드시 다음 관점을 포함한다.

- 범용 문서 확장성
- 구현 단순성
- 버전 스냅샷 적합성
- 렌더링 용이성
- diff/복원 확장성

---

### 5-2. Node 공통 스키마 정의

모든 Node가 공통적으로 가져야 할 최소 필드를 정의한다.

검토 대상 예시:

- node_id
- node_type
- parent_node_id 필요 여부
- order 또는 position
- children
- payload
- metadata
- annotations 필요 여부
- is_deleted 또는 soft-delete 표시 필요 여부

각 필드에 대해 다음을 정리한다.

- 필드명
- 목적
- 필수 여부
- 변경 가능 여부
- Version 스냅샷 포함 여부

중요: Node 스키마는 특정 에디터 라이브러리의 내부 포맷을 그대로 답습하지 말고, 플랫폼 도메인 기준으로 정의한다.

---

### 5-3. 기본 블록 타입 범위 정의

초기 Phase에서 지원할 최소 블록 타입을 정의한다.

최소 검토 대상은 아래를 포함한다.

- document_root
- section
- heading
- paragraph
- list
- list_item
- table
- table_row
- table_cell
- quote
- code_block 필요 여부
- image / attachment placeholder 필요 여부
- appendix 또는 reference block 필요 여부

각 블록 타입에 대해 다음을 정리한다.

- 의미
- 허용 부모 타입
- 허용 자식 타입
- 필수 payload 요소
- 렌더링 시 기본 표현 기대치

단, 초기 범위에서는 너무 많은 타입을 도입하지 말고, 워드 대체에 필요한 최소 집합을 우선 정의한다.

---

### 5-4. payload 구조 설계 원칙 정의

Node별 상세 데이터는 공통 필드 외에 payload에 담는 방향을 우선 검토한다.

반드시 다음 내용을 다룬다.

- payload와 metadata의 구분 기준
- 텍스트 본문 저장 방식
- 표 데이터 표현 방식
- 리스트/중첩 리스트 표현 방식
- 향후 확장 필드 추가 방식
- 빈 payload 허용 여부

예를 들어 paragraph는 text payload를, table은 row/cell 구조 payload 또는 child node 구조를 가질 수 있다. 이 차이를 어떻게 가져갈지 정리한다.

---

### 5-5. 전체 저장 vs 부분 수정 전략 비교

편집 단위를 정의하기 위해 아래 두 방향을 비교한다.

#### 안 A. 전체 문서 저장 중심

- 저장 시 문서 전체 구조를 Version 스냅샷으로 저장
- 수정 API가 부분 편집처럼 보여도 내부적으로는 전체 구조를 갱신
- 복원과 버전 관리가 단순함

#### 안 B. 부분 블록 수정 중심

- Node 단위 패치 저장
- 변경 추적이 세밀함
- 구현 복잡도와 정합성 관리 비용이 큼

이번 Phase 기준에서는 어떤 방향이 적절한지 선택하고, 선택 이유를 정리한다.

중요: 사용자 편집 경험과 내부 저장 전략은 반드시 동일할 필요가 없다는 점을 명시한다.

---

### 5-6. 편집 단위 기준 정의

문서 플랫폼이 어떤 단위를 “편집의 최소 단위”로 볼지 정의한다.

반드시 아래를 구분해서 검토한다.

- 사용자 편집 단위
- API 요청 단위
- 저장 반영 단위
- Version 스냅샷 단위

예를 들어 사용자는 문단 하나만 수정할 수 있지만, 저장 시점에는 전체 content tree가 스냅샷으로 기록될 수 있다. 이 관계를 명확히 설명한다.

### 5-7. Version content_snapshot 구조 정의

Task 4-2의 Version 모델과 연결하여, `content_snapshot`에 어떤 구조를 담을지 정의한다.

반드시 다음을 포함한다.

- 루트 노드 포함 여부
- children 중첩 구조 방식
- node_id 유지 정책
- 순서 정보 표현 방식
- 빈 문서 표현 방식
- 복원 시 필요한 최소 정보

권장 방향은, Version이 독립적으로 복원 가능하도록 `content_snapshot` 자체만으로 본문 구조를 재구성할 수 있어야 한다는 점이다.

### 5-8. 구조 유효성 검증 규칙 정의

문서 저장 전에 어떤 검증이 필요한지 정의한다.

다음 수준으로 나누어 정리한다.

#### Node 단위 검증

- node_type 유효성
- payload 필수 필드 존재 여부
- 허용되지 않은 필드 존재 여부

#### 트리 구조 검증

- 루트 노드 1개 여부
- 부모-자식 관계 유효성
- 순환 참조 금지
- 허용되지 않은 중첩 구조 금지
- order 중복 여부 또는 정렬 규칙

#### 문서 단위 검증

- 최소 1개 이상의 본문 블록 필요 여부
- 제목/최상위 섹션 필수 여부
- 특정 문서 타입별 추가 제약 가능성

초기 Phase에서는 너무 복잡한 validation rule보다, 무결성을 지키는 핵심 규칙을 우선 정의한다.

### 5-9. 저장 구조와 렌더링 구조 관계 정의

저장 포맷과 렌더링 입력 포맷이 동일해야 하는지 검토하고, 권장안을 제시한다.

다음 두 방향을 비교한다.

#### 안 A. 저장 구조 = 렌더링 구조

- 단순함
- 빠른 구현 가능
- UI 종속 위험 존재

#### 안 B. 저장 구조와 렌더링 ViewModel 분리

- 변환 계층 필요
- 유연성 높음
- 장기 확장성 우수

이번 작업에서는 어떤 방향을 택할지 정하고, Task 4-7 렌더링 설계와 어떻게 연결되는지 적는다.

## 8. 권장 방향

아래 방향을 우선안으로 검토한다.

### 우선안 A

- 문서 본문은 **루트 Node를 가진 트리 기반 구조**로 정의한다.
- 각 Node는 공통 필드와 타입별 `payload`를 가진다.
- Version의 `content_snapshot`에는 문서 전체 트리를 저장한다.
- 사용자는 부분 편집을 하더라도, 초기 저장 전략은 **전체 문서 스냅샷 저장**을 기본으로 한다.
- 저장 구조와 렌더링 구조는 완전히 동일시하지 않고, 렌더링 시 ViewModel 변환 계층을 둘 수 있도록 설계한다.
- 초기 블록 타입은 paragraph, heading, section, list, table 정도의 최소 집합으로 시작한다.

이 우선안의 장점은 다음과 같다.

- 버전 관리와 복원이 단순하다.
- 문서 전체 일관성을 유지하기 쉽다.
- 이후 diff, 템플릿, AI 초안 생성과 연결하기 좋다.
- UI 교체 시 저장 포맷이 덜 흔들린다.
