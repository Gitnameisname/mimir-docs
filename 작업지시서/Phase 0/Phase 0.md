Phase 0 작업 계획서

프로젝트 정의 및 플랫폼 원칙 수립

1. Phase 개요

Phase 명

Phase 0. 프로젝트 정의 및 플랫폼 원칙 수립

목적

전체 시스템의 방향, 설계 철학, 기술적 기준선을 먼저 확정하여 이후 모든 설계와 구현의 판단 기준 문서를 만든다.

핵심 목표
	•	플랫폼의 최종 지향점을 문서 시스템이 아닌 지식 플랫폼으로 명확히 정의
	•	API-first, User/Admin UI 분리, 저장 구조 전략 등 주요 아키텍처 원칙 확정
	•	시스템이 어디까지 책임지고 어디부터 외부 시스템과 연동하는지 시스템 경계 명확화
	•	Phase 1 이후의 도메인 모델 설계가 흔들리지 않도록 기준 문서 작성

최종 결과물
	•	Project Charter / Platform Principles 문서
	•	시스템 범위 및 경계 정의서
	•	기술 스택 방향 문서
	•	저장 구조 전략서
	•	User/Admin 분리 원칙 문서
	•	Phase 1 설계 입력 조건 정리서

⸻

2. Phase 0의 역할

Phase 0은 기능 개발 단계가 아니다.
이 단계의 목적은 “무엇을 만들 것인가”보다 먼저, 아래를 고정하는 것이다.
	1.	우리는 정확히 어떤 플랫폼을 만들고 있는가
	2.	무엇을 코어 책임으로 두고 무엇은 외부 시스템과 연계할 것인가
	3.	어떤 원칙을 지키며 개발할 것인가
	4.	이후 설계에서 절대 흔들리면 안 되는 축은 무엇인가

즉, Phase 0의 산출물은 코드보다 설계 판단 기준이다.

⸻

3. 세부 작업 구성

Task 0-1. 플랫폼 비전 및 목표 정의

목적

프로젝트를 “문서 저장/조회 시스템”이 아니라 “지식 플랫폼”으로 재정의하고, 플랫폼이 제공해야 할 본질적 가치를 정리한다.

주요 작업
	•	프로젝트의 상위 목표 정의
	•	기존 문서 시스템 관점과 지식 플랫폼 관점의 차이 정리
	•	플랫폼이 다루는 핵심 자산 정의
	•	문서
	•	구조화된 지식
	•	메타데이터
	•	버전
	•	관계
	•	검색/탐색 정보
	•	API를 통한 외부 활용 가능 자원
	•	주요 사용자군 정의
	•	일반 사용자
	•	관리자
	•	외부 시스템/에이전트/API 클라이언트
	•	플랫폼이 해결해야 할 핵심 문제 정의
	•	하지 않을 것(Out of Scope)도 함께 정의

산출물
	•	플랫폼 비전 문서
	•	In Scope / Out of Scope 목록
	•	핵심 사용자 및 사용 시나리오 요약

완료 기준
	•	“왜 이 플랫폼을 만드는가”를 1페이지 내로 설명 가능
	•	문서 시스템과 지식 플랫폼의 차이가 문서로 명확히 구분됨
	•	이후 기능 추가 시 범위 판별 기준으로 활용 가능

⸻

Task 0-2. 플랫폼 원칙 선언

목적

향후 설계와 구현에서 반복적으로 참조할 공통 원칙을 확정한다.

확정해야 할 핵심 원칙
	•	API-first
	•	UI와 도메인 로직 분리
	•	User UI / Admin UI 분리
	•	문서 유형 확장 가능성 우선
	•	버전과 변경 이력 추적 가능성 확보
	•	구조화 가능한 것은 구조화하고, 비정형도 수용
	•	검색/탐색/연계 가능성 우선
	•	권한과 운영 기능은 일반 사용자 기능과 분리
	•	AI/외부 도구 연계를 고려한 인터페이스 설계
	•	스토리지와 도메인 모델의 결합 최소화

주요 작업
	•	각 원칙에 대한 정의 작성
	•	원칙이 필요한 이유 작성
	•	원칙 위반 사례 예시 작성
	•	원칙 충돌 시 우선순위 규칙 작성

산출물
	•	Platform Principles 문서

완료 기준
	•	핵심 원칙이 8~12개 수준으로 정리됨
	•	각 원칙마다 “정의 / 이유 / 적용 예시 / 금지 예시”가 존재함
	•	이후 설계 리뷰 체크리스트의 기반이 될 수 있음

⸻

Task 0-3. API-first 원칙 구체화

목적

이 플랫폼이 UI 중심 제품이 아니라, API를 중심으로 서비스와 기능을 조직하는 플랫폼임을 명확히 한다.

주요 작업
	•	API-first의 의미를 프로젝트 맥락에서 정의
	•	모든 핵심 기능은 API를 통해 노출된다는 원칙 명시
	•	UI는 API의 소비자 중 하나라는 구조 명시
	•	외부 앱, 내부 시스템, AI 에이전트, 자동화 툴이 사용할 수 있는 형태를 전제로 설계
	•	향후 예상 API 범주 초안 정리
	•	문서 CRUD
	•	버전 관리
	•	문서 구조 조회
	•	검색/필터링
	•	메타데이터 관리
	•	권한/승인/운영 API
	•	이벤트/웹훅/연동 포인트
	•	API 안정성, 버전 정책, 에러 포맷, 인증 방향에 대한 상위 원칙 정리

산출물
	•	API-first 원칙 정의서
	•	초기 API 도메인 맵

완료 기준
	•	“UI 먼저 만들고 나중에 API 붙인다” 방식이 금지 원칙으로 명시됨
	•	향후 설계가 API를 중심으로 분해될 수 있는 수준의 기능 맵이 존재함

⸻

Task 0-4. User UI / Admin UI 분리 원칙 확정

목적

일반 사용자 경험과 운영/관리 기능을 혼합하지 않도록 초기부터 구조적 분리를 확정한다.

주요 작업
	•	User UI의 목적 정의
	•	문서/지식 탐색
	•	조회
	•	작성/편집
	•	협업
	•	개인화된 작업
	•	Admin UI의 목적 정의
	•	문서 유형 관리
	•	권한 및 정책 관리
	•	상태/승인/운영
	•	시스템 설정
	•	데이터 관리/감사/모니터링
	•	공통 기능과 분리 기능 구분
	•	라우팅/프론트엔드 구조 분리 방향 정의
	•	API 권한 영역도 함께 구분
	•	운영자 기능이 User UI 내부에 흩어지지 않도록 금지 규칙 정의

산출물
	•	User/Admin UI 분리 원칙 문서
	•	UI 책임 범위표

완료 기준
	•	사용자용 화면과 운영용 화면의 목적 차이가 명확히 문서화됨
	•	Phase 2 이후 UI 설계 시 정보구조 기준으로 재사용 가능

	•	설정값

	•	관계형 저장

	•	오브젝트 스토리지
	•	검색 인덱스 분리

	•	내부 시스템 책임 정의

6. 완료 기준

Phase 0는 아래 조건을 만족하면 완료로 본다.
	•	플랫폼이 문서 시스템이 아니라 지식 플랫폼이라는 점이 문서로 명확히 정의됨
	•	API-first 원칙이 선언 수준이 아니라 실제 설계 기준으로 정리됨
	•	User UI와 Admin UI 분리 원칙이 확정됨
	•	기술 스택과 저장 전략이 상위 원칙 수준에서 정리됨
	•	시스템 경계가 정의되어 내부 책임과 외부 연계 책임이 구분됨
	•	Phase 1 도메인 모델 설계에 바로 들어갈 수 있는 입력 조건이 정리됨

7. Phase 1과의 연결 포인트

Phase 0가 끝나면, Phase 1에서는 아래 질문을 바로 다룰 수 있어야 한다.
	•	Document는 어떤 단위의 지식 자산인가
	•	Version은 어떤 수준으로 관리되는가
	•	Node는 문서 구조만 표현하는가, 아니면 범용 지식 단위인가
	•	DocumentType은 얼마나 유연해야 하는가
