Phase 0-3 작업 지시서

API-first 원칙 구체화

1. 작업명

Phase 0-3. API-first 원칙 구체화

⸻

2. 작업 목적

Phase 0-2에서 선언한 API-first 원칙을 실제 설계 기준으로 사용할 수 있도록 구체화한다.
이 작업의 목적은 “API를 잘 만들자” 수준의 선언이 아니라, 이 플랫폼이 UI 중심 시스템이 아니라 API 중심 플랫폼이라는 점을 구조적으로 확정하는 것이다.

즉, 이 문서는 앞으로 아래 질문에 답할 수 있어야 한다.
	•	왜 모든 핵심 기능은 API를 기준으로 설계되어야 하는가
	•	UI는 플랫폼의 주체가 아니라 어떤 위치에 있어야 하는가
	•	외부 애플리케이션, AI 도구, 자동화 워크플로가 어떻게 이 플랫폼과 연결될 수 있어야 하는가
	•	어떤 기능은 반드시 API로 먼저 정의되어야 하는가
	•	API 설계에서 어떤 공통 규칙을 가져야 하는가

⸻

3. 작업 배경

현재 프로젝트는 장기적으로 단순한 웹 화면 제공 시스템이 아니라 다음을 수용해야 한다.
	•	일반 사용자 UI
	•	관리자 UI
	•	외부 애플리케이션 연동
	•	내부 업무 시스템 연동
	•	AI agent / workflow tool / automation consumer
	•	향후 이벤트 기반 연계와 배치 처리

이런 구조에서는 화면을 먼저 만들고 뒤에서 API를 맞추는 방식이 아니라,
플랫폼 기능을 먼저 API 계약으로 정의하고, UI는 그 API를 소비하는 하나의 클라이언트로 두어야 한다.

API-first를 명확히 하지 않으면 쉽게 다음 문제가 생긴다.
	•	UI 내부 로직이 사실상의 서비스 로직이 됨
	•	User UI와 Admin UI가 서로 다른 비공식 로직을 가짐
	•	외부 연동이 필요할 때 내부 기능을 다시 뜯어내야 함
	•	특정 화면 플로우가 데이터 구조를 지배하게 됨
	•	자동화/AI 연계가 어려워짐

⸻

4. 작업 목표

아래 내용을 포함하는 기준 문서를 만든다.
	1.	이 프로젝트에서 API-first가 의미하는 바를 정의
	2.	UI, Admin, 외부 시스템, AI 도구가 API와 어떤 관계를 가져야 하는지 정리
	3.	핵심 기능 영역을 API 도메인 단위로 분해
	4.	API 설계 상위 규칙을 정리
	5.	API 버전, 인증, 권한, 에러 포맷, 이벤트 연계에 대한 상위 방향을 제시
	6.	향후 Phase 1~2 설계에서 API 계약 중심 분해가 가능하도록 입력 조건을 제공

⸻

5. 반드시 반영할 핵심 방향

5-1. UI는 API의 소비자다

User UI와 Admin UI는 모두 API를 소비하는 클라이언트이며,
핵심 비즈니스 로직이 UI 내부에 고정되면 안 된다.

5-2. API는 외부 연계를 전제한다

API는 내부 프론트엔드만을 위한 비공식 엔드포인트 집합이 아니라,
향후 외부 앱·사내 시스템·AI tool·automation consumer가 사용할 수 있는 플랫폼 인터페이스여야 한다.

5-3. API는 도메인 중심으로 나뉘어야 한다

화면 단위가 아니라, 문서/버전/구조/메타데이터/검색/운영 등
도메인 단위로 기능을 분해하는 방향을 잡아야 한다.

5-4. API 규칙은 일관되어야 한다

리소스 명명, 상태 코드, 오류 응답, 인증 방식, 권한 범위, 페이징, 필터링 같은 규칙은
초기부터 상위 원칙으로 정의해야 한다.

5-5. 이벤트/비동기 연계 가능성을 남겨야 한다

모든 것을 동기형 CRUD로만 가정하지 말고,
대용량 처리·색인·동기화·후처리 같은 비동기 작업과 이벤트 연계를 고려할 것.

⸻

6. 세부 수행 항목

아래 순서대로 작업하라.

A. 프로젝트 맥락에서 API-first 정의

다음 질문에 답하는 형태로 정의하라.
	•	이 프로젝트에서 API-first란 무엇인가
	•	단순한 REST 제공과 무엇이 다른가
	•	왜 UI-first 방식이 아니라 API-first여야 하는가
	•	이 원칙이 향후 어떤 비용을 줄여주는가

최소 포함 내용:
	•	기능 정의의 기준점은 API 계약이다
	•	UI는 API의 한 소비자다
	•	외부 연계와 자동화 가능성을 API 설계에 반영한다
	•	내부 구현보다 계약과 책임 분리가 우선된다

⸻

B. 시스템 참여자와 API 관계 정리

아래 참여자들이 API와 어떤 관계를 갖는지 정리하라.

최소 포함:
	•	User UI
	•	Admin UI
	•	External Application
	•	Internal Enterprise System
	•	AI Agent / Workflow Tool / Automation Consumer

각 참여자마다 아래를 작성:
	•	API를 왜 사용하는가
	•	어떤 종류의 기능에 접근할 가능성이 있는가
	•	어떤 수준의 권한/제약이 필요한가

⸻

C. API 도메인 맵 초안 작성

플랫폼 기능을 화면이 아니라 도메인 기준으로 API 영역으로 나누어라.

최소 포함 권장 도메인:
	•	Document API
	•	Version API
	•	Node/Structure API
	•	Metadata API
	•	DocumentType API
	•	Status/Lifecycle API
	•	Search/Query API
	•	Attachment/Resource API
	•	Auth/Permission API
	•	Admin/Governance API
	•	Audit/History API
	•	Integration/Event API

각 도메인마다 아래를 짧게 정리:
	•	책임
	•	대표 리소스
	•	주요 동작 유형
	•	User/Admin/External 중 누가 주로 소비하는지

⸻

D. API 설계 상위 규칙 정의

세부 구현 전 단계에서 반드시 지켜야 할 상위 규칙을 정리하라.

최소 검토 항목:
	•	리소스 중심 명명 규칙
	•	CRUD와 액션형 API 구분 원칙
	•	일관된 응답 구조
	•	오류 응답 포맷 원칙
	•	필터링/정렬/페이징 원칙
	•	id와 외부 참조키 사용 원칙
	•	버전 필드와 동시성 고려 필요성
	•	장기 작업에 대한 비동기 처리 방식 방향
	•	감사/추적과 연관된 식별자 전달 원칙

⸻

E. 인증 / 권한 / 책임 범위 상위 방향 정리

API-first는 공개 개방형 API만 의미하지 않는다.
권한 경계가 명확한 API 플랫폼이어야 한다.

최소 포함:
	•	사용자 인증이 필요한 API
	•	관리자 전용 API
	•	시스템 간 연동용 API 가능성
	•	권한 범위(scope) 또는 역할(role) 개념 필요성
	•	같은 리소스라도 User/Admin/External에서 보이는 범위가 달라질 수 있다는 점
	•	멀티 클라이언트 환경에서 감사 추적이 가능해야 한다는 점

⸻

F. API 버전 정책 방향 정의

아직 세부 버전 전략을 확정하지 않더라도 상위 원칙은 정하라.

최소 포함:
	•	왜 버전 정책이 필요한가
	•	변경 호환성 원칙
	•	깨지는 변경을 어떻게 다룰지
	•	내부용/외부용 API 경계가 필요할지
	•	API 계약 안정성에 대한 기본 원칙

⸻

G. 이벤트 및 비동기 연계 방향 정의

CRUD API만으로 끝나지 않는 영역을 식별하라.

예시:
	•	대량 문서 등록
	•	인덱싱
	•	버전 생성 후 후처리
	•	외부 시스템 동기화
	•	상태 전이 이벤트
	•	감사 로그 적재
	•	AI 연계 파이프라인 트리거

정리 항목:
	•	어떤 작업이 비동기일 가능성이 높은가
	•	어떤 이벤트가 외부 통합 포인트가 될 수 있는가
	•	“동기 API + 비동기 후처리” 구조를 고려해야 하는 지점은 어디인가

⸻

H. UI/화면 설계에 주는 제약 정리

이 API-first 원칙이 이후 UI 설계에 어떤 제약을 주는지 정리하라.

최소 포함:
	•	화면 전용 비즈니스 규칙 금지
	•	User UI와 Admin UI가 별도 API를 쓰더라도 코어 도메인 규칙은 공유해야 함
	•	UI는 조합/표현에 집중하고 도메인 행위 정의는 API 계층에서 수행
	•	임시 편의를 위해 비공식 백도어 엔드포인트를 만들지 않음

⸻

I. Phase 1/2 입력 요약 도출

Phase 1 도메인 모델과 Phase 2 API/서비스 설계에 전달할 핵심 요약을 작성하라.

최소 포함:
	•	도메인 모델은 API 리소스로 자연스럽게 드러날 수 있어야 함
	•	Document / Version / Node / Metadata / Status는 API 계약 단위로도 생각해야 함
	•	User/Admin/External의 접근 차이를 고려한 모델링 필요
	•	감사/이력/버전 추적은 API 계층에서도 드러나야 함

8. 문서 작성 원칙
	•	프레임워크 종속적 표현보다 플랫폼 원칙 중심으로 작성할 것
	•	너무 구현 세부로 내려가지 말 것
	•	그러나 실제 API 설계 시 바로 사용할 수 있을 정도로 구조화할 것
	•	REST, GraphQL, RPC 같은 기술 선택 논쟁으로 문서를 흐리지 말 것
	•	핵심은 “계약 중심 사고”와 “도메인 중심 API 분해”에 둘 것
	•	User/Admin/External/AI 소비자 관점을 반드시 반영할 것

⸻

9. 금지 사항

아래는 하지 마라.
	•	화면 목록을 API 목록처럼 재작성하는 것
	•	특정 프론트엔드 요구에만 맞춘 엔드포인트 설계
	•	“나중에 외부 연계 가능” 정도의 추상 문장으로 끝내는 것
	•	인증/권한을 후속 구현 이슈로만 넘겨버리는 것
	•	모든 작업을 동기 CRUD만으로 가정하는 것
	•	내부 구현 편의를 이유로 API 계약 원칙을 약화시키는 것
