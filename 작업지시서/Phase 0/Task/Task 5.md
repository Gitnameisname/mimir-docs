Phase 0-5 작업 지시서

기술 스택 방향 설정

1. 작업명

Phase 0-5. 기술 스택 방향 설정

⸻

2. 작업 목적

플랫폼 비전, API-first 원칙, User/Admin UI 분리 원칙을 바탕으로 기술 스택의 방향성과 선택 기준을 정의한다.
이 작업의 목적은 특정 프레임워크를 성급히 확정하는 것이 아니라, 이후 설계와 구현에서 흔들리지 않도록 왜 이 계열의 기술이 필요한지를 먼저 고정하는 것이다.

즉, 이 문서는 다음 질문에 답해야 한다.
	•	이 플랫폼은 어떤 기술적 성격을 가진 시스템인가
	•	프론트엔드, 백엔드, 저장소, 검색, 비동기 처리 계층은 어떤 방향이 적합한가
	•	기술 선택 시 무엇을 우선 기준으로 삼아야 하는가
	•	무엇은 지금 확정하고, 무엇은 후속 단계에서 세부 결정할 것인가
	•	도메인 모델과 API 중심 구조를 기술적으로 어떻게 뒷받침할 것인가

⸻

3. 작업 배경

현재 프로젝트는 단순한 웹사이트가 아니라 다음 특성을 가진다.
	•	범용 문서/지식 자산 관리
	•	API-first 플랫폼 구조
	•	User/Admin UI 분리
	•	버전/상태/메타데이터/구조 관리
	•	검색/탐색 중심 활용
	•	향후 외부 시스템 및 AI 도구 연계 가능성
	•	장기적으로 비동기 처리, 이벤트, 대용량 데이터, 인덱싱 확장 가능성 필요

따라서 기술 스택도 단순 CRUD 웹앱 관점이 아니라,
플랫폼형 서비스 아키텍처를 지탱할 수 있는 방향으로 잡아야 한다.

이 단계에서 잘못된 선택을 하면 다음 문제가 생긴다.
	•	UI 구현 편의가 도메인 구조를 지배함
	•	검색/버전/메타데이터 같은 핵심 요구가 뒤늦게 누락됨
	•	Admin 기능 확장 시 구조가 꼬임
	•	외부 연동 시 API 계층이 약함
	•	저장 구조가 특정 구현 방식에 잠식됨
	•	비동기 처리나 대량 처리 도입이 어려워짐

5. 반드시 반영할 핵심 방향

5-1. 기술은 비전을 지원해야 한다

기술 선택은 익숙함이나 단기 속도보다,
범용 지식 플랫폼이라는 목표와 API-first 구조를 얼마나 잘 지지하는가를 기준으로 해야 한다.

5-2. 프론트엔드는 소비자 계층이다

User/Admin UI는 중요하지만, 도메인 규칙의 중심이 되어서는 안 된다.
프론트엔드는 API와 권한 구조 위에서 동작하는 소비자 계층으로 설계되어야 한다.

5-3. 백엔드는 도메인/API 중심이어야 한다

핵심 비즈니스 규칙, 권한, 상태, 버전, 감사, 연계 포인트는
백엔드 서비스 계층과 API 계층에서 일관되게 관리되어야 한다.

5-4. 저장은 다층 구조를 전제로 해야 한다

모든 데이터를 단일 저장소로 해결하려 하지 말고,
	•	관계형 데이터
	•	유연한 메타데이터
	•	파일/오브젝트 저장
	•	검색 인덱스
를 역할별로 나누는 전략을 고려해야 한다.

5-5. 지금은 방향을 정하고, 세부 제품 선택은 후속 검토 가능하다

이 단계는 구체적 제품명 고정이 아니라,
기술 계층별 요구사항과 선택 기준을 명확히 하는 것이 핵심이다.

⸻

6. 세부 수행 항목

아래 순서대로 작업하라.

A. 플랫폼의 기술적 성격 정의

먼저 이 플랫폼이 기술적으로 어떤 유형의 시스템인지 정의하라.

최소 포함 관점:
	•	단순 웹앱인가, 플랫폼 서비스인가
	•	단순 문서 뷰어인가, 구조화된 지식 관리 시스템인가
	•	CRUD 중심인가, 버전/상태/검색/연계가 중요한가
	•	단일 UI 애플리케이션인가, 다중 소비자 API 플랫폼인가

이 항목의 목적은 이후 기술 스택 논의의 전제를 정하는 것이다.

⸻

B. 프론트엔드 방향 설정

프론트엔드 기술 방향을 정의하라.

최소 포함:
	•	User UI와 Admin UI를 분리 가능한 구조로 가져갈 수 있어야 함
	•	API 소비 중심 구조여야 함
	•	복잡한 문서/구조/메타데이터 화면을 다룰 수 있어야 함
	•	장기적으로 상태 관리, 권한 처리, 라우팅 분리, 확장 가능한 컴포넌트 구조를 지원해야 함
	•	디자인 시스템/공통 컴포넌트 재사용은 가능하되, UX 목적은 분리되어야 함

정리 항목:
	•	권장 프론트엔드 아키텍처 방향
	•	상태 관리 원칙
	•	User/Admin 앱 구조 방향
	•	SSR/CSR/Hybrid 고려 필요성
	•	프론트엔드 선택 기준

⸻

C. 백엔드/API 계층 방향 설정

백엔드 기술 방향을 정의하라.

최소 포함:
	•	도메인 중심 서비스 계층 필요성
	•	API-first 구조를 지지할 수 있는 프레임워크/아키텍처 성격
	•	인증/권한/감사/상태 관리가 가능한 구조
	•	문서/버전/노드/메타데이터/검색을 분리된 도메인으로 다룰 수 있어야 함
	•	향후 비동기 작업, 이벤트, 외부 연계를 수용할 수 있어야 함

정리 항목:
	•	백엔드 구조 원칙
	•	API 계층과 도메인 계층 분리 필요성
	•	서비스/유스케이스 계층 필요성
	•	동기 요청 처리와 비동기 작업 처리의 분리 방향
	•	백엔드 기술 선택 기준

⸻

D. 데이터 저장 계층 방향 설정

플랫폼의 데이터 저장 전략을 기술 관점에서 요약하라.

최소 포함:
	•	관계형 DB가 적합한 영역
	•	JSON/유연 구조가 필요한 영역
	•	첨부 파일/원본 리소스 저장 전략
	•	검색 인덱스의 필요성
	•	버전 이력 저장 고려
	•	감사/로그/운영 데이터 저장 고려

정리 항목:
	•	다중 저장소 전략이 필요한 이유
	•	어떤 데이터가 어떤 저장 계층에 적합한지
	•	도메인 모델과 물리 저장을 분리해야 하는 이유
	•	기술 선택 시 고려할 성능/운영성 요소

⸻

E. 검색 및 탐색 계층 방향 설정

이 플랫폼은 검색/탐색이 중요한 만큼, 이를 별도 기술 요구로 정의하라.

최소 포함:
	•	문서 본문 검색
	•	구조 단위 검색 가능성
	•	메타데이터 필터링
	•	버전/상태 기반 탐색
	•	향후 지식 탐색/관계 탐색 확장 가능성

정리 항목:
	•	관계형 DB만으로 충분한지, 별도 검색 계층이 필요한지
	•	검색 인덱스 동기화 문제
	•	검색 정확도와 운영 복잡도의 균형
	•	초기 MVP와 장기 확장 방향 분리

⸻

F. 비동기 처리 / 이벤트 / 배치 방향 설정

비동기성 요구를 기술 스택 관점에서 정리하라.

최소 포함 가능 대상:
	•	대량 문서 등록
	•	인덱싱
	•	버전 생성 후 후처리
	•	외부 시스템 동기화
	•	상태 변경 후 후속 작업
	•	감사 로그 적재
	•	알림/후처리 워크플로

정리 항목:
	•	어떤 작업은 즉시 응답형이 적합한가
	•	어떤 작업은 큐/잡/이벤트 기반이 적합한가
	•	작업 추적/재시도/운영 가시성이 왜 필요한가
	•	초기에는 단순하게 가되 장기적으로 어떤 확장 여지를 남겨야 하는가

⸻

G. 운영 및 배포 관점 요구사항 정리

플랫폼 운영을 고려한 기술 방향도 함께 정리하라.

최소 포함:
	•	환경 분리(dev/stage/prod) 필요성
	•	설정 관리 방식 필요성
	•	로깅/모니터링/감사 가능성
	•	장애 대응 및 운영 가시성
	•	API 서비스로서의 안정성 고려
	•	Admin 기능 확장을 위한 운영 데이터 확보 필요성

⸻

H. 기술 선택 기준표 작성

구체적 제품 결정보다 앞서, 기술 선택 기준을 명시하라.

최소 기준 후보:
	•	API 친화성
	•	타입 안정성
	•	유지보수성
	•	확장성
	•	검색/버전/메타데이터 모델 수용 가능성
	•	운영 용이성
	•	보안/권한 구조 수용 가능성
	•	팀 생산성
	•	테스트 용이성
	•	커뮤니티/생태계 성숙도

각 기준에 대해:
	•	왜 중요한지
	•	어떤 계층에서 특히 중요한지
를 정리하라.

⸻

I. 확정 사항 vs 후속 결정 사항 구분

이 단계에서 무엇을 확정하고, 무엇은 후속 Phase에서 더 상세히 정할지 구분하라.

예시:
	•	지금 확정: 계층별 기술 방향, 다중 저장소 필요성, API-first에 맞는 백엔드 구조 원칙
	•	후속 결정: 구체적 검색 엔진 제품, 메시지 브로커 제품, 세부 배포 토폴로지, 세부 프론트 프레임워크 옵션 비교 등

⸻

J. Phase 1/2 입력 요약 정리

다음 단계에 전달할 핵심 요약을 작성하라.

최소 포함:
	•	도메인 모델이 특정 저장 방식에 묶이지 않도록 해야 함
	•	API 계층이 도메인 모델을 안정적으로 노출할 수 있어야 함
	•	검색, 버전, 상태, 메타데이터는 기술 계층에서도 1급 고려 대상이어야 함
	•	User/Admin 분리 요구가 프론트/백/API 권한 구조에 반영되어야 함
	•	비동기 확장 가능성을 지금부터 배제하지 않아야 함
