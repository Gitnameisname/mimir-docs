Phase 0-6 작업 지시서

저장 구조 전략 정의

1. 작업명

Phase 0-6. 저장 구조 전략 정의

⸻

2. 작업 목적

플랫폼이 다루는 핵심 자산을 어떤 저장 전략으로 관리할지 상위 원칙을 정의한다.
이 작업의 목적은 DB 스키마를 바로 설계하는 것이 아니라, 도메인 모델과 물리 저장 구조를 분리한 상태에서, 어떤 데이터가 어떤 성격의 저장소에 적합한지 기준을 세우는 것이다.

즉, 이 문서는 앞으로 아래 질문에 답해야 한다.
	•	문서 본문, 구조, 버전, 메타데이터, 첨부, 검색 인덱스, 감사 로그는 각각 어떻게 저장되어야 하는가
	•	무엇을 관계형 구조로 두고, 무엇을 유연 구조로 둘 것인가
	•	물리 저장 편의 때문에 도메인 모델이 왜곡되지 않으려면 어떤 원칙이 필요한가
	•	검색, 버전 관리, 외부 연계, 감사 추적을 고려한 저장 전략은 무엇인가
	•	Phase 1의 Document / Version / Node / Metadata 모델이 저장 방식에 끌려가지 않으려면 어떤 기준이 필요한가

⸻

3. 작업 배경

이 플랫폼은 단순 파일 저장소가 아니라 다음과 같은 서로 다른 성격의 데이터를 다룬다.
	•	문서의 식별 정보와 기본 속성
	•	문서 구조(Node/Tree)
	•	버전별 콘텐츠 및 변경 이력
	•	확장 가능한 메타데이터
	•	첨부 파일 및 외부 리소스 참조
	•	검색용 인덱스 데이터
	•	상태 전이 기록
	•	감사 로그 및 운영 로그
	•	권한/정책/설정 정보

이런 자산을 하나의 저장 방식으로만 다루면 쉽게 문제가 생긴다.
	•	관계형 스키마가 과도하게 비대해짐
	•	유연성이 필요한 메타데이터가 경직됨
	•	반대로 모든 것을 JSON으로 밀어넣어 질의성과 무결성이 약해짐
	•	첨부/원본 자산과 메타 정보가 뒤섞임
	•	검색을 위해 필요한 구조가 본 저장 모델을 오염시킴
	•	버전과 현재 상태가 혼동됨
	•	감사와 운영 기록이 도메인 데이터와 섞여 관리성이 떨어짐

따라서 이 단계에서 저장 자산 분류와 저장소 역할 분담 원칙을 먼저 정해야 한다.

⸻

4. 작업 목표

아래를 포함하는 저장 전략 문서를 만든다.
	1.	플랫폼 데이터 자산을 유형별로 분류
	2.	각 자산의 저장 요구 특성을 정의
	3.	관계형 저장 / 유연 구조 / 오브젝트 저장 / 검색 인덱스 / 로그 저장의 역할을 구분
	4.	도메인 모델과 물리 저장의 분리 원칙 정리
	5.	버전 저장과 현재 상태 표현의 관계를 정리
	6.	향후 검색, 감사, AI/외부 연계 확장을 고려한 저장 전략 방향을 제시

⸻

5. 반드시 반영할 핵심 방향

5-1. 도메인 모델이 저장 방식을 먼저 따라가면 안 된다

Phase 1의 Document / Version / Node / Metadata 모델은 저장소 편의에 의해 먼저 결정되면 안 된다.
먼저 도메인 의미를 정의하고, 저장은 그것을 지지하는 방식으로 선택해야 한다.

5-2. 모든 데이터를 하나의 방식으로 저장하려 하지 않는다

관계형 저장, 유연 필드, 오브젝트 저장, 검색 인덱스, 로그 저장은 역할이 다르다.
플랫폼은 다층 저장 전략을 전제로 해야 한다.

5-3. 현재 상태와 이력은 구분되어야 한다

현재 사용자에게 보이는 최신 상태와, 버전/변경/상태 전이/감사 이력은 분리된 관점으로 다뤄야 한다.

5-4. 메타데이터는 확장 가능해야 하지만 무질서하면 안 된다

메타데이터는 유연성을 수용해야 하지만, 질의 가능성·검증 가능성·일관성도 유지해야 한다.

5-5. 검색 인덱스는 저장 원본이 아니다

검색 성능과 탐색 경험을 위해 별도 인덱스 계층이 필요할 수 있지만,
검색 인덱스를 도메인 원본 저장소처럼 취급해서는 안 된다.

⸻

6. 세부 수행 항목

아래 순서대로 작업하라.

A. 데이터 자산 분류표 작성

플랫폼이 다루는 데이터 자산을 먼저 분류하라.

최소 포함 자산:
	•	Document 기본 정보
	•	DocumentType 정의
	•	Version 정보
	•	Node/Structure 정보
	•	본문 콘텐츠
	•	Metadata
	•	Status/Lifecycle 정보
	•	Relationship 정보
	•	Attachment / File / External Resource Reference
	•	Search Index Data
	•	Audit Log
	•	Operational Log / Job Status
	•	Permission / Policy / Config 정보

각 자산마다 아래를 정리:
	•	자산 설명
	•	변경 빈도
	•	조회 패턴
	•	무결성 중요도
	•	검색 필요성
	•	이력 관리 필요성
	•	저장 유연성 필요성

⸻

B. 저장 요구 특성 분석

각 자산이 어떤 저장 특성을 가지는지 분석하라.

예시 관점:
	•	강한 정합성이 필요한가
	•	관계 탐색이 중요한가
	•	부분 필드 질의가 자주 필요한가
	•	구조가 고정적인가, 확장 가능한가
	•	대용량 바이너리인가
	•	전체 텍스트 검색 대상인가
	•	감사 추적이 중요한가
	•	현재 상태만 중요하지 않고 이력도 중요한가

이 분석을 통해 각 자산이 어떤 저장 계층에 더 적합한지 방향을 도출하라.

⸻

C. 저장 계층 역할 정의

최소 아래 저장 계층을 기준으로 역할을 정리하라.

1) 관계형 저장 계층
적합 예:
	•	Document 기본 식별자/소유 정보
	•	Version 메타 정보
	•	Node 구조 참조
	•	상태 값
	•	권한/정책
	•	감사 참조 키

정리할 것:
	•	왜 관계형 저장이 필요한가
	•	어떤 영역은 관계형이 적합한가
	•	무결성과 질의성이 왜 중요한가

2) 유연 구조(JSON 등) 저장
적합 예:
	•	확장 메타데이터
	•	문서 유형별 추가 속성
	•	일부 설정/보조 속성

정리할 것:
	•	왜 유연 구조가 필요한가
	•	어디까지 허용할 것인가
	•	완전한 무질서가 되지 않기 위한 통제 원칙은 무엇인가

3) 오브젝트/파일 저장 계층
적합 예:
	•	첨부 파일
	•	원본 문서 파일
	•	렌더링용 산출물
	•	외부 자원 캐시 또는 참조

정리할 것:
	•	왜 DB 본문과 분리해야 하는가
	•	메타 정보와 실제 파일의 분리 원칙
	•	참조 무결성을 어떻게 생각해야 하는가

4) 검색 인덱스 계층
적합 예:
	•	본문 검색용 토큰화/색인 데이터
	•	Node 단위 검색용 데이터
	•	메타데이터 필터링용 색인 보조 구조

정리할 것:
	•	왜 원본 저장과 검색 인덱스를 분리해야 하는가
	•	색인 지연/동기화 이슈를 어떻게 다뤄야 하는가
	•	초기 MVP와 장기 확장 방향 차이

5) 로그/감사/운영 저장 계층
적합 예:
	•	감사 이벤트
	•	상태 전이 기록
	•	운영 작업 결과
	•	동기화/인덱싱/배치 잡 상태

정리할 것:
	•	도메인 데이터와 로그 데이터를 왜 분리해야 하는가
	•	감사 가능성과 운영 가시성을 위해 필요한 최소 구조는 무엇인가

⸻

D. 현재 상태 vs 버전 vs 이력 관계 정리

플랫폼 저장 전략에서 가장 중요한 축 중 하나인
현재 상태 / 버전 / 이력의 관계를 정리하라.

최소 포함:
	•	Document 현재 상태는 무엇을 의미하는가
	•	Version은 어떤 수준의 스냅샷 또는 단위인가
	•	상태 전이 기록은 별도 이력으로 남겨야 하는가
	•	감사 로그와 버전 이력은 어떻게 다른가
	•	사용자가 보는 “최신 문서”와 시스템이 보관하는 “버전 기록”은 어떻게 구분되는가

이 항목은 Phase 1 Version 모델 설계에 직접 영향을 주도록 작성하라.

⸻

E. Node/Structure 저장 전략 방향 정리

트리 기반 문서 구조를 다루기 위해 Node/Structure 저장의 상위 전략을 정리하라.

최소 포함:
	•	Node는 문서 내부의 구조 단위로 다뤄질 수 있음
	•	트리/계층/순서 정보를 저장해야 할 수 있음
	•	검색, 렌더링, 부분 조회, 버전 관리가 구조 저장 방식에 영향을 줌
	•	구조 표현과 본문 표현을 분리할지 여부 검토 필요
	•	특정 트리 저장 기법에 종속되지 않더라도 요구사항은 먼저 명확히 해야 함

⸻

F. Metadata 저장 전략 방향 정리

Metadata는 범용 플랫폼에서 특히 중요하므로 별도로 정리하라.

최소 포함:
	•	공통 메타데이터와 문서 유형별 확장 메타데이터 구분
	•	정규 필드와 확장 필드의 분리 필요성
	•	필터링과 검색 가능한 메타데이터의 중요성
	•	검증 규칙과 스키마 유연성의 균형
	•	Admin 정책과도 연결될 수 있다는 점

⸻

G. 첨부/외부 자원 참조 전략 정리

첨부 파일과 외부 자원 참조를 어떻게 다룰지 상위 원칙을 정리하라.

최소 포함:
	•	첨부 파일 자체와 첨부 메타정보 분리
	•	원본 파일과 가공 산출물 분리 가능성
	•	외부 URL/외부 시스템 리소스 참조를 수용할 가능성
	•	보안/권한/수명주기 측면 고려
	•	본문 안에 내장할 것과 참조만 둘 것의 구분 필요성

⸻

H. 검색 및 AI 확장을 고려한 저장 전략 정리

장기적으로 검색 고도화나 AI/RAG 확장 가능성을 염두에 두고 저장 전략의 여지를 정리하라.

최소 포함:
	•	본문과 구조가 검색/추출 가능해야 한다는 점
	•	metadata가 필터/검색에 활용될 수 있어야 한다는 점
	•	chunk/node 단위 분해 가능성
	•	embeddings 등 부가 인덱스가 생길 수 있지만 원본 저장과는 분리해야 한다는 점
	•	AI용 보조 데이터는 도메인 원본과 구분해야 한다는 점

⸻

I. 저장 전략의 원칙/안티패턴 정리

좋은 저장 전략 원칙과 피해야 할 안티패턴을 정리하라.

예시 안티패턴:
	•	모든 것을 단일 JSON blob으로 저장
	•	모든 메타데이터를 컬럼으로만 강제
	•	첨부 파일을 본문/DB와 강하게 결합
	•	검색 인덱스를 원본 데이터처럼 사용
	•	버전과 최신 상태를 구분하지 않음
	•	감사 로그를 일반 비즈니스 테이블에 흩뿌림

각 안티패턴마다:
	•	왜 문제가 되는지
	•	어떤 원칙을 위반하는지
	•	대체 방향은 무엇인지
를 작성하라.

⸻

J. Phase 1 입력 요약 정리

Phase 1 범용 문서 도메인 모델 설계에 전달할 핵심 저장 전략 입력을 정리하라.

최소 포함:
	•	Document / Version / Node / Metadata / Status는 저장 방식보다 의미가 먼저 정의되어야 함
	•	그러나 각 모델은 저장 계층 분해 가능성을 고려해 설계되어야 함
	•	현재 상태와 이력의 분리 원칙을 모델 설계에 반영해야 함
	•	메타데이터 확장성과 질의 가능성의 균형이 필요함
	•	첨부/검색/AI 보조 데이터는 코어 도메인과 구분되어야 함

	•	그러나 실제 데이터 설계 회의에 쓸 수 있을 정도로 구체성은 확보할 것
	•	“유연성”만 강조하지 말고, 무결성과 질의 가능성도 함께 다룰 것
	•	검색/감사/버전/첨부/AI 확장을 모두 장기 고려 요소로 반영할 것
	•	특정 DB 제품 또는 특정 트리 저장 기법에 과도하게 종속되지 말 것

⸻

9. 금지 사항

아래는 하지 마라.
	•	곧바로 상세 테이블 설계로 내려가는 것
	•	모든 것을 단일 저장소/단일 형식으로 해결한다고 가정하는 것
	•	현재 상태와 이력을 혼동하는 것
	•	검색 인덱스를 원본 저장처럼 취급하는 것
	•	메타데이터를 무제한 자유 필드로만 방치하는 것
	•	첨부/외부 자원 참조 전략을 생략하는 것

⸻

10. 완료 기준

다음 조건을 만족하면 완료다.
	•	데이터 자산 분류가 정리됨
	•	저장 계층별 역할이 정의됨
	•	현재 상태 / 버전 / 이력 관계가 설명됨
	•	Node / Metadata / Attachment / Search 관련 상위 전략이 문서화됨
	•	안티패턴과 원칙이 정리됨
	•	Phase 1 설계자가 저장 제약과 자유도를 함께 이해할 수 있음

⸻

11. 실행 방식
	•	먼저 데이터 자산 분류
	•	다음으로 저장 계층 역할 정의
	•	그 다음 현재 상태/버전/이력 관계 정리
	•	이후 Node/Metadata/Attachment/Search 전략 추가
	•	마지막으로 안티패턴과 Phase 1 입력 요약으로 정리
