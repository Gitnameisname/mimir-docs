플랫폼 비전 및 목표 정의

1. 작업명

Phase 0-1. 플랫폼 비전 및 목표 정의

⸻

2. 작업 목적

현재 프로젝트의 성격을 단순한 “문서 시스템”이 아니라 범용 지식 플랫폼으로 재정의하고, 이후 모든 설계와 구현의 기준이 되는 상위 목표를 문서화한다.

이 작업의 핵심은 기능 목록을 나열하는 것이 아니라 아래를 명확히 하는 것이다.
	•	이 플랫폼이 본질적으로 무엇인지
	•	어떤 문제를 해결하려는지
	•	누구를 위한 시스템인지
	•	무엇을 범위에 포함하고 무엇을 제외할지
	•	왜 이후 설계가 이 방향을 따라야 하는지

⸻

3. 작업 배경

기존에는 특정 문서(예: 규정집/문서 시스템) 중심 관점이 있었지만, 현재 프로젝트 목표는 더 넓다.

이 플랫폼은 장기적으로 다음을 수용할 수 있어야 한다.
	•	다양한 문서 유형
	•	구조화된 지식
	•	버전과 변경 이력
	•	메타데이터 확장
	•	검색/탐색
	•	외부 시스템/API/AI 도구 연계

따라서 지금 단계에서 “우리가 만드는 것이 정확히 무엇인가”를 다시 고정하지 않으면, 이후 데이터 모델과 API 설계가 좁아지거나 흔들릴 수 있다.

⸻

4. 작업 목표

아래 질문에 답하는 기준 문서를 작성한다.
	1.	이 프로젝트의 최종 지향점은 무엇인가?
	2.	왜 문서 시스템이 아니라 지식 플랫폼이어야 하는가?
	3.	플랫폼이 다루는 핵심 자산은 무엇인가?
	4.	주요 사용자 집단은 누구인가?
	5.	이 플랫폼이 해결하려는 핵심 문제는 무엇인가?
	6.	초기에 하지 않을 것은 무엇인가?
	7.	이후 Phase 1 설계에 어떤 방향성을 제공해야 하는가?

⸻

5. 반드시 반영할 핵심 방향

다음 사항은 결과 문서에 반드시 반영해야 한다.

5-1. 플랫폼 정체성

플랫폼은 단순한 파일 저장소나 문서 뷰어가 아니라,
문서·구조·메타데이터·버전·관계·탐색 기능을 통합한 지식 플랫폼이라는 점을 명확히 할 것.

5-2. 확장성

초기에는 문서 중심으로 출발하더라도, 특정 문서 유형에 종속되지 않는 방향으로 정의할 것.

5-3. API 연계성

이 플랫폼은 UI 전용 시스템이 아니라,
향후 외부 애플리케이션·AI 도구·워크플로 엔진과 연결될 수 있는 플랫폼으로 정의할 것.

5-4. 운영 분리 가능성

일반 사용자 경험과 운영/관리 기능이 분리될 수 있도록 상위 목표 수준에서도 이를 암시하거나 반영할 것.

5-5. 범위 통제

무엇이 플랫폼의 코어 책임인지, 무엇은 후속 단계 또는 외부 연계 영역인지 초안 수준으로라도 구분할 것.

⸻

6. 세부 수행 항목

아래 항목을 순서대로 수행하라.

A. 현재 프로젝트 목표 재정의
	•	기존의 “문서 시스템” 관점과 새로 정의할 “지식 플랫폼” 관점을 비교 정리
	•	왜 방향 전환이 필요한지 설명
	•	한 문단 요약과 확장 설명 둘 다 작성

B. 플랫폼 비전 정의

아래 내용을 포함해 비전 문장을 작성
	•	플랫폼이 제공하려는 핵심 가치
	•	장기적으로 어떤 역할을 수행하는 시스템이 될지
	•	사용자가 이 플랫폼을 통해 무엇을 할 수 있어야 하는지

C. 핵심 문제 정의

이 플랫폼이 해결하려는 문제를 정리
예시 관점:
	•	문서가 유형별로 파편화됨
	•	구조와 메타데이터가 일관되지 않음
	•	버전 추적과 변경 이력 관리가 어려움
	•	외부 시스템과 연계하기 어려움
	•	검색과 재사용이 비효율적임

D. 핵심 자산 정의

플랫폼이 다루는 핵심 자산을 정의
최소 포함:
	•	Document
	•	Version
	•	Node/Structure
	•	Metadata
	•	Relationship
	•	Searchable Knowledge
	•	Attachment or External Resource Reference

E. 주요 사용자군 정의

최소 아래 사용자군을 정의
	•	일반 사용자
	•	관리자/운영자
	•	외부 시스템/API 클라이언트
	•	향후 AI agent/automation consumer

각 사용자군마다
	•	주요 목적
	•	기대 행동
	•	필요 기능 수준
을 짧게 정리

F. In Scope / Out of Scope 정의

초기 단계에서 플랫폼 범위에 포함되는 것과 제외되는 것을 나눠서 정리

예시:
	•	포함: 문서/버전/구조/메타데이터/탐색/API 기반 접근
	•	제외: 고도화된 AI 추론 자체, 복잡한 워크플로 엔진 전체 구현, 특정 외부 시스템 종속 기능 등

G. 성공 기준 초안 정의

이 작업의 결과로 “비전이 잘 정의되었다”고 판단할 수 있는 조건 제시

예시:
	•	플랫폼의 정체성을 3~5문장으로 설명 가능
	•	문서 시스템과 지식 플랫폼의 차이가 분명함
	•	Phase 1 설계에 필요한 방향성이 도출됨

8. 문서 작성 원칙

문서를 작성할 때 아래 원칙을 지켜라.
	•	추상적인 슬로건만 쓰지 말고, 설계에 영향을 줄 수 있는 문장으로 작성할 것
	•	너무 기술 구현 중심으로 내려가지 말 것
	•	그러나 모호한 표현만 남기지 말고, 실제 설계 판단에 도움이 되게 작성할 것
	•	“왜 이런 방향을 택하는가”를 반드시 설명할 것
	•	문장 톤은 기획/아키텍처 기준 문서 스타일로 작성할 것
	•	과한 마케팅 문구 금지
	•	특정 문서 유형에 종속된 표현은 피할 것
	•	이후 API 설계와 데이터 모델 설계로 연결될 수 있게 작성할 것

⸻

9. 금지 사항

아래 내용은 피하라.
	•	특정 UI 화면 설계 수준으로 깊게 들어가는 것
	•	DB 테이블 상세 설계까지 내려가는 것
	•	특정 프레임워크 구현 코드 작성
	•	규정집/매뉴얼/보고서 등 특정 문서군 중심으로 비전을 제한하는 것
	•	플랫폼의 역할과 외부 연계 역할을 혼동하는 것

⸻

10. 기대 결과 예시 성격

최종 결과는 다음 느낌이어야 한다.
	•	“무엇을 만드는가”가 분명하다
	•	“왜 그렇게 만드는가”가 설명된다
	•	“무엇은 아직 안 하는가”도 보인다
	•	Phase 1 설계자가 이 문서를 읽고 데이터 모델 방향을 잡을 수 있다
