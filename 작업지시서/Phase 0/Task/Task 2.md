Phase 0-2 작업 지시서

플랫폼 원칙 선언

1. 작업명

Phase 0-2. 플랫폼 원칙 선언

⸻

2. 작업 목적

프로젝트 전반에서 반복적으로 참조할 플랫폼 설계 원칙을 선언하고 문서화한다.
이 작업의 목적은 좋은 말 모음집을 만드는 것이 아니라, 이후 API 설계, 데이터 모델 설계, UI 설계, 운영 설계에서 판단 기준으로 직접 사용할 수 있는 설계 헌장을 만드는 것이다.

이 문서는 앞으로 다음 상황에서 기준이 되어야 한다.
	•	기능 추가 여부 판단
	•	설계안 비교
	•	기술 선택 논의
	•	User/Admin UI 구조 분리 판단
	•	저장 구조와 도메인 구조 분리 판단
	•	API 우선 설계 여부 검토
	•	특정 요구사항이 플랫폼 철학에 맞는지 검토

⸻

3. 작업 배경

Phase 0-1에서 플랫폼의 비전과 목표를 정의했다면, 이제는 그 비전을 실제 설계 판단으로 연결할 원칙 집합이 필요하다.

프로젝트가 커질수록 아래 문제가 쉽게 발생한다.
	•	화면 중심으로 먼저 구현하고 API가 뒤따라옴
	•	특정 문서 유형에 맞춘 임시 구조가 코어 모델이 됨
	•	운영 기능이 사용자 화면에 섞임
	•	저장소 구조가 곧 도메인 구조가 되어버림
	•	확장성보다 당장 편한 방식이 누적됨
	•	외부 연계가 필요한 시점에 구조가 막힘

이 문제를 방지하기 위해, 플랫폼 수준의 명시적 원칙을 먼저 확정해야 한다.

⸻

4. 작업 목표

아래 질문에 답하는 원칙 문서를 만든다.
	1.	이 플랫폼은 어떤 설계 철학을 우선하는가?
	2.	어떤 종류의 구현 방식은 지양해야 하는가?
	3.	기능 설계 시 무엇을 우선 판단 기준으로 둘 것인가?
	4.	확장성, 운영성, 연계성을 어떻게 보장할 것인가?
	5.	원칙 간 충돌이 생기면 어떤 우선순위로 판단할 것인가?

⸻

5. 반드시 포함해야 할 핵심 원칙 후보

아래 항목들은 반드시 검토 대상으로 포함하고, 최종 문서에는 최소 8개 이상 원칙을 확정하라.

필수 후보
	•	API-first
	•	User UI / Admin UI 분리
	•	도메인 모델과 저장 구현 분리
	•	특정 문서 유형 비종속 코어 모델
	•	버전 및 변경 추적 가능성 우선
	•	구조화 우선 + 비정형 수용
	•	메타데이터 확장 가능성
	•	외부 시스템/AI 도구 연계 가능성
	•	권한/운영 정책의 명시적 분리
	•	검색/탐색 가능성 우선
	•	감사 가능성/운영 추적 가능성
	•	단계적 확장 가능성

최종 원칙 수는 8~12개 정도로 정리하는 것을 권장한다.

⸻

6. 세부 수행 항목

아래 순서대로 작업하라.

A. 플랫폼 원칙 후보 수집

Phase 0-1 결과를 바탕으로, 이후 설계에 지속적으로 영향을 주는 원칙 후보를 먼저 수집하라.

형식 예시:
	•	원칙명
	•	한 줄 설명
	•	왜 필요한가
	•	어느 설계 영역에 영향을 주는가

영향 영역 예시:
	•	API
	•	Domain Model
	•	UI/UX
	•	Admin
	•	Storage
	•	Search
	•	Integration
	•	Governance

⸻

B. 원칙 선별 및 정제

후보 중 중복되거나 지나치게 추상적인 항목은 정리하고, 실제 설계에서 쓸 수 있는 수준으로 다듬어라.

정제 기준:
	•	선언문이 짧고 분명한가
	•	적용 범위가 보이는가
	•	위반 사례를 식별할 수 있는가
	•	설계 리뷰 때 체크 가능할 정도로 구체적인가

⸻

C. 각 원칙을 표준 템플릿으로 문서화

최종 선정된 각 원칙은 아래 형식으로 작성하라.

원칙 템플릿
	1.	원칙명
	2.	선언문
	•	한두 문장으로 핵심을 명확히 표현
	3.	의도
	•	왜 이 원칙이 필요한가
	4.	적용 범위
	•	어느 영역에 영향을 주는가
	5.	적용 예시
	•	이 원칙을 따르는 설계/구현 예
	6.	위반 예시
	•	이 원칙을 어기는 대표 사례
	7.	설계 시사점
	•	이 원칙이 Phase 1 이후에 어떤 영향을 주는가

⸻

D. 원칙 간 우선순위 정의

현실적으로 원칙끼리 충돌할 수 있으므로, 우선순위를 초안으로 정의하라.

예시 충돌:
	•	빠른 구현 vs 확장성
	•	저장 효율 vs 도메인 명확성
	•	단순 UI 구성 vs User/Admin 분리
	•	특정 요구사항 최적화 vs 범용 모델 유지

최소한 아래는 정리할 것:
	•	어떤 원칙이 상위 원칙인가
	•	충돌 시 예외를 허용하려면 어떤 조건이 필요한가
	•	임시 예외를 도입할 때 기록해야 할 항목은 무엇인가

⸻

E. 설계 리뷰 체크리스트 도출

원칙 선언이 선언으로 끝나지 않도록, 설계 리뷰 때 사용할 수 있는 체크리스트를 작성하라.

예시 질문:
	•	이 기능은 API 없이 UI 내부 로직으로만 구현되고 있지 않은가?
	•	이 구조는 특정 문서 유형에 종속적인가?
	•	운영 기능이 일반 사용자 경험에 섞이고 있지 않은가?
	•	저장 편의 때문에 도메인 구조가 왜곡되고 있지 않은가?
	•	버전/변경 이력 추적 가능성이 사라지지 않았는가?

⸻

8. 문서 작성 원칙

다음 기준을 지켜라.
	•	추상 구호보다 설계 판단 기준이 되도록 쓸 것
	•	“좋아 보이는 말”이 아니라 “실제로 위반 여부를 판별할 수 있는 문장”으로 작성할 것
	•	한 원칙이 너무 많은 뜻을 품지 않게 쪼갤 것
	•	반대로 너무 잘게 쪼개어 의미가 약해지지 않게 할 것
	•	원칙 하나하나가 이후 Phase에 어떤 영향을 주는지 드러나야 함
	•	향후 팀 내 리뷰/PR 리뷰/아키텍처 리뷰에도 재사용 가능해야 함

⸻

9. 권장 최종 원칙 예시 방향

최종 문서는 아래와 비슷한 결을 가져야 한다.
	1.	API-first
	2.	범용 도메인 모델 우선
	3.	User/Admin 경험 분리
	4.	도메인 모델과 저장 구조 분리
	5.	버전 및 변경 추적 가능성 확보
	6.	구조화 우선, 비정형 수용
	7.	메타데이터와 문서 유형의 확장 가능성
	8.	검색/탐색 및 재사용 가능성 우선
	9.	외부 연계 가능성 확보
	10.	운영/감사/권한 정책의 명시적 관리

이 목록을 그대로 고정할 필요는 없지만, 이 수준의 구조감은 유지하라.

⸻

10. 금지 사항

아래는 하지 마라.
	•	구현 기술 이름 위주로 원칙을 쓰는 것
	•	너무 추상적이라 해석이 무한히 가능한 표현만 남기는 것
	•	“유연하게 한다”, “확장 가능하게 한다”처럼 근거 없는 모호 문장만 쓰는 것
	•	특정 기능 하나를 위한 로컬 룰을 플랫폼 원칙으로 과대 포장하는 것
	•	User/Admin 분리 원칙을 단순 메뉴 분리 수준으로 축소하는 것

⸻

11. 완료 기준

다음 조건을 만족하면 완료다.
	•	최소 8개 이상의 플랫폼 원칙이 확정됨
	•	각 원칙마다 의도, 적용 범위, 적용 예시, 위반 예시가 존재함
	•	원칙 간 우선순위와 예외 처리 기준이 정의됨
	•	설계 리뷰용 체크리스트가 별도 문서로 정리됨
	•	이후 Phase 1, Phase 2에서 실제 판단 기준으로 재사용 가능한 수준임

⸻

12. 실행 방식
	•	먼저 원칙 후보를 넓게 수집
	•	중복/모호 항목 제거
	•	최종 원칙 8~12개로 정제
	•	각 원칙 상세화
	•	리뷰 체크리스트 도출
	•	마지막으로 “이 문서가 실제 설계 논쟁을 줄일 수 있는가” 기준으로 다듬을 것
