Phase 0-4 작업 지시서

User/Admin UI 분리 원칙 확정

1. 작업명

Phase 0-4. User/Admin UI 분리 원칙 확정

⸻

2. 작업 목적

일반 사용자용 UI와 관리자/운영자용 UI를 초기 설계 단계부터 명확히 분리하는 원칙을 확정한다.
이 작업의 목적은 단순히 메뉴를 나누는 것이 아니라, 목적·권한·정보 구조·행동 흐름이 서로 다른 두 개의 인터페이스 계층을 분리하여 이후 설계 혼선을 방지하는 것이다.

이 문서는 앞으로 아래 판단의 기준이 되어야 한다.
	•	어떤 기능이 User UI에 있어야 하는가
	•	어떤 기능이 Admin UI에 있어야 하는가
	•	같은 리소스라도 사용자와 관리자에게 어떻게 다르게 보여야 하는가
	•	운영 기능이 사용자 경험에 침투하지 않게 하려면 무엇을 지켜야 하는가
	•	API와 권한 설계가 UI 분리 원칙과 어떻게 연결되는가

⸻

3. 작업 배경

지식 플랫폼은 일반적으로 두 종류의 요구를 동시에 가진다.
	1.	일반 사용자 요구
	•	문서/지식 탐색
	•	조회
	•	작성/편집
	•	개인/팀 단위 활용
	•	검색과 재사용
	2.	관리자/운영자 요구
	•	문서 유형 관리
	•	권한/정책 관리
	•	상태/승인/배포 관리
	•	감사/이력 확인
	•	시스템 설정 및 운영 모니터링

이 둘을 한 화면 체계 안에 섞기 시작하면 다음 문제가 발생한다.
	•	사용자 경험이 운영 기능 때문에 복잡해짐
	•	운영 기능이 일반 사용자에게 노출되며 개념 혼란이 생김
	•	권한 모델이 UI에서 우회적으로 꼬임
	•	관리자만 필요한 메타 정보가 사용자 화면에 새어 나옴
	•	Admin 기능이 User UI 내부의 숨겨진 옵션처럼 커지며 유지보수가 어려워짐

따라서 이 단계에서 User UI와 Admin UI를 구조적으로 분리하는 원칙을 먼저 명확히 해야 한다.

⸻

4. 작업 목표

아래를 포함하는 기준 문서를 만든다.
	1.	User UI와 Admin UI의 목적 차이를 정의
	2.	두 UI가 다루는 책임 범위를 구분
	3.	공통 기능과 분리 기능을 판별하는 기준을 만든다
	4.	권한/정보 노출/API 접근 차이를 정리
	5.	향후 IA, 라우팅, 프론트엔드 구조, API 권한 설계에 반영될 원칙을 제시한다

⸻

5. 반드시 반영할 핵심 방향

5-1. User UI와 Admin UI는 목적이 다르다

User UI는 지식 활용과 생산성에 초점을 두고,
Admin UI는 운영, 통제, 정책, 감사, 시스템 관리에 초점을 둔다.

5-2. 분리는 화면 장식이 아니라 책임 분리다

단순히 메뉴만 나누는 것이 아니라,
정보 구조, 행동 흐름, 노출 정보, 진입 경로, 권한 전제가 달라야 한다.

5-3. 같은 리소스라도 사용자와 관리자 관점은 다르다

예를 들어 Document를 보더라도:
	•	User는 내용, 구조, 편집 가능성, 검색 활용을 본다
	•	Admin은 상태, 권한, 승인 이력, 소유권, 정책 위반 여부를 본다

5-4. 운영 기능은 User UI에 스며들지 않아야 한다

운영자용 기능은 예외적으로만 사용자 화면에 겹쳐야 하며,
기본 원칙은 별도 Admin UI에서 수행하게 해야 한다.

5-5. UI 분리는 API/권한 분리와 연결된다

User/Admin UI 분리는 단지 화면 문제만이 아니라,
	•	권한 모델
	•	API 접근 범위
	•	감사 로그
	•	데이터 노출 정책
과 직접 연결되어야 한다.

⸻

6. 세부 수행 항목

아래 순서대로 작업하라.

A. User UI와 Admin UI의 목적 정의

각 UI의 존재 목적을 먼저 정의하라.

User UI 관점에서 최소 포함
	•	문서/지식 탐색
	•	문서 조회
	•	작성/편집
	•	개인 또는 협업 단위 활용
	•	검색/필터링/재사용
	•	사용자 입장에서 필요한 상태 정보만 확인

Admin UI 관점에서 최소 포함
	•	플랫폼 운영
	•	권한/정책 통제
	•	문서 유형 및 메타 정책 관리
	•	승인/배포/상태 변경 관리
	•	감사 로그 확인
	•	장애/오류/운영 상태 파악
	•	시스템 설정 및 통합 관리

각각에 대해 아래를 작성:
	•	핵심 목적
	•	대표 사용자
	•	기대 행동 패턴
	•	정보 복잡도 수준

⸻

B. UI 책임 범위 구분

아래 관점으로 User와 Admin의 책임 범위를 구분하라.

최소 분류 항목:
	•	조회
	•	작성/편집
	•	버전 확인
	•	상태 확인
	•	메타데이터 편집
	•	문서 유형 관리
	•	권한 설정
	•	승인/배포
	•	감사 로그 확인
	•	시스템 설정
	•	외부 연계 설정
	•	운영 모니터링

각 항목마다 다음 중 하나로 분류:
	•	User 중심
	•	Admin 중심
	•	공통이지만 관점 분리 필요
	•	User에 노출 금지

⸻

C. 공통 기능과 관점 차이 정리

동일 기능/동일 리소스라도 User/Admin의 관점 차이를 문서화하라.

예시:
	•	Document 상세 보기
	•	Version 목록
	•	Status 정보
	•	Metadata 표시
	•	Search 결과
	•	변경 이력

각 항목별로 아래를 작성:
	•	User가 보는 관점
	•	Admin이 보는 관점
	•	표시 정보 차이
	•	허용 동작 차이
	•	접근 제어 차이

⸻

D. 정보 구조(IA) 분리 원칙 정의

향후 UI 설계에 반영할 정보 구조 분리 원칙을 정리하라.

최소 포함:
	•	User UI는 작업/콘텐츠 중심
	•	Admin UI는 자원/정책/운영 중심
	•	User UI 네비게이션은 사용자의 목적 기반
	•	Admin UI 네비게이션은 관리 대상 기반
	•	운영 항목을 User UI 깊숙한 곳의 숨은 메뉴로 두지 않음
	•	Admin UI는 감사와 운영성을 고려한 구조를 가져야 함

또한 아래도 검토:
	•	별도 라우팅/별도 앱/별도 셸 구조 필요성
	•	공용 컴포넌트는 재사용하되, 화면 책임은 분리하는 방향
	•	동일 디자인 시스템을 써도 UX 목표는 달라야 한다는 점

⸻

E. 권한 및 노출 정책과의 연결

UI 분리가 권한 정책과 어떻게 연결되는지 정리하라.

최소 포함:
	•	User UI에 보이는 정보는 일반 사용자 권한에 맞아야 함
	•	Admin UI는 운영 권한이 있는 주체만 접근 가능해야 함
	•	같은 API를 사용하더라도 응답 범위가 달라질 수 있음
	•	UI 숨김만으로 보안을 대체하면 안 됨
	•	UI 분리는 반드시 API 권한 통제와 함께 가야 함
	•	운영자 액션은 감사 로그와 연결되어야 함

⸻

F. 잘못된 통합 사례 / 위반 사례 정리

아래와 같은 위반 사례를 구체적으로 정리하라.

예시:
	•	일반 문서 화면 안에 관리자용 승인/권한 변경 기능이 섞여 있음
	•	관리자 기능이 단순 “숨김 버튼” 수준으로만 보호됨
	•	사용자 화면에서 시스템 전역 설정에 접근 가능함
	•	운영자용 상태 정보가 일반 사용자 검색 결과에 노출됨
	•	User/Admin이 서로 다른 비공식 로직을 사용함

각 사례마다:
	•	왜 문제인지
	•	어떤 원칙을 위반하는지
	•	어떤 구조로 바꿔야 하는지
를 함께 적어라.

⸻

G. UI 분리가 API 설계에 주는 요구사항 정리

다음 내용을 정리하라.
	•	User 전용/Admin 전용 API 경계 필요성
	•	공통 도메인 API + 역할별 액션 API 조합 가능성
	•	같은 리소스라도 응답 필드가 달라질 수 있는지
	•	Admin 액션은 더 강한 감사/추적이 필요하다는 점
	•	UI가 달라도 코어 도메인 규칙은 공유해야 한다는 점

최소 포함:

	•	상태, 권한, 감사 정보는 Admin 관점에서 특히 중요함
	•	문서/버전/메타데이터/상태 모델은 표시 관점 차이를 감안해야 함
	•	UI 분리는 나중의 화면 정리 작업이 아니라 초기 아키텍처 기준이라는 점

8. 문서 작성 원칙
	•	“관리자 메뉴 따로 둔다” 수준에서 끝내지 말 것
	•	목적, 정보 구조, 권한, 행동 흐름의 차이를 함께 설명할 것
	•	구체적이되 특정 화면 목업 수준으로는 내려가지 말 것
	•	User 경험 단순화와 Admin 운영성 강화를 동시에 반영할 것
	•	UI 분리가 왜 플랫폼 확장성과 보안/운영성에 중요한지 드러나야 함

⸻

9. 금지 사항

아래는 하지 마라.
	•	User/Admin 분리를 단순 라우팅 이름 차이로 축소하는 것
	•	권한 문제를 프론트 화면 숨김만으로 해결하려는 표현
	•	운영 기능을 “필요하면 사용자 화면에도 넣을 수 있다” 식으로 느슨하게 쓰는 것
	•	Admin UI를 단순 옵션 패널 정도로 취급하는 것
	•	User와 Admin의 목적 차이를 무시하고 동일 UX를 강요하는 것

⸻

10. 완료 기준

다음 조건을 만족하면 완료다.
	•	User UI와 Admin UI의 목적 차이가 명확히 정의됨
	•	기능 책임 범위가 문서화됨
	•	공통 리소스에 대한 관점 차이가 정리됨
	•	UI 분리와 API/권한/감사 관계가 연결됨
	•	위반 사례와 설계 시사점이 정리됨
	•	이후 IA/프론트엔드/API 설계에 직접 활용 가능한 수준임
