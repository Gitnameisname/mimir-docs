# 핵심 엔티티

---

## 1. 엔티티 관계 개요

```
Organization ──< UserOrgRole >── User
                                  │
                              (created_by)
                                  │
                              Document ──────────────────────────┐
                                  │                              │
                                  │ 1:N                          │
                              Version ──────────────────────────┐│
                                  │                             ││
                                  │ 1:N                         ││
                               Node                             ││
                              (트리 구조)                         ││
                                                                ││
                              DocumentChunk ◄──────────────────┘│
                              (벡터 청크)                         │
                                                                 │
                              AuditEvent ◄────────────────────── ┘
                              WorkflowHistory
                              ReviewAction
```

## 2. Document (문서)

문서 플랫폼의 최상위 엔티티. 문서의 정체성(제목, 유형, 상태, 메타데이터)을 관리하며 실제 내용은 Version에 저장된다.

```python
@dataclass
class Document:
    id: str                    # UUID (기본 키)
    title: str                 # 문서 제목 (최대 500자)
    document_type: str         # 문서 유형 코드 (예: POLICY, MANUAL)
    status: str                # 문서 상태: draft | published | archived | deprecated
    metadata: dict             # 확장 key-value (JSONB) — 타입별 커스텀 필드
    summary: Optional[str]     # 문서 요약
    created_by: Optional[str]  # 생성자 actor_id
    updated_by: Optional[str]  # 최종 수정자 actor_id
    created_at: datetime
    updated_at: datetime
    current_draft_version_id: Optional[str]     # 현재 활성 Draft 버전 포인터
    current_published_version_id: Optional[str] # 현재 Published 버전 포인터
```

**Document.status 값:**

| 값 | 의미 |
|----|------|
| `draft` | 작성/수정 중인 문서 |
| `published` | 공식 게시된 문서 (최신 Published 버전 보유) |
| `archived` | 보관 처리된 문서 (조회만 가능) |
| `deprecated` | 더 이상 사용하지 않는 문서 |

---

## 3. Version (버전)

문서의 특정 시점 스냅샷. 문서는 여러 버전을 가질 수 있으며, 한 번에 하나의 Published 버전과 하나의 Draft 버전이 존재할 수 있다.

```python
@dataclass
class Version:
    id: str
    document_id: str           # 소속 문서 ID
    version_number: int        # 순차 번호 (문서 내 고유)
    label: Optional[str]       # 버전 레이블 (예: "v2.1")
    status: str                # draft | published | archived
    workflow_status: str       # 워크플로 상태 (별도 §4 참조)
    change_summary: Optional[str]  # 변경 요약
    source: str                # manual | system | import
    metadata: dict             # JSONB 확장
    created_by: Optional[str]
    created_at: datetime
    # 스냅샷 필드 (Published 시 문서 메타 복사본)
    title_snapshot: Optional[str]
    summary_snapshot: Optional[str]
    metadata_snapshot: Optional[dict]
    content_snapshot: Optional[dict]  # 전체 노드 트리 스냅샷
    published_by: Optional[str]
    published_at: Optional[datetime]
    # 버전 계보
    parent_version_id: Optional[str]     # 기반이 된 이전 버전
    restored_from_version_id: Optional[str]  # 복원 출처 버전
```

## 4. Node (노드)

문서 내용의 최소 구조 단위. 계층형 트리 구조로 문서 본문을 구성하며, 섹션·단락·제목·목록 등 모든 콘텐츠를 표현한다.

```python
@dataclass
class Node:
    id: str
    version_id: str            # 소속 버전 ID
    parent_id: Optional[str]   # 부모 노드 ID (NULL = 루트)
    node_type: str             # paragraph | heading | list_item | table | image | ...
    order_index: int           # 같은 부모 내 순서
    title: Optional[str]       # 노드 제목 (주로 heading 타입)
    content: Optional[str]     # 텍스트 내용
    metadata: dict             # JSONB 확장 (스타일, 속성 등)
    created_at: datetime
```

**node_type 값:**

| 타입 | 용도 |
|------|------|
| `paragraph` | 일반 단락 텍스트 |
| `heading` | 제목 (h1~h6, metadata.level로 구분) |
| `list_item` | 목록 항목 |
| `table` | 표 (metadata에 셀 데이터) |
| `image` | 이미지 (metadata.url) |
| `code_block` | 코드 블록 (metadata.language) |
| `section` | 논리적 섹션 구분 |

**트리 구조 예시:**

```
Version
└── Node (section, "1장 목적")
    ├── Node (heading, "1.1 배경")
    │   └── Node (paragraph, "본 정책은...")
    └── Node (heading, "1.2 적용 범위")
        ├── Node (paragraph, "모든 임직원에게...")
        └── Node (list_item, "개발팀")
```

## 5. DocumentChunk (벡터 청크)

RAG 파이프라인을 위해 Version의 Node 내용을 청크 단위로 분할하고 임베딩을 저장한 엔티티.

```
document_chunks
├── id, document_id, version_id, node_id
├── chunk_index          ← 버전 내 청크 순번
├── source_text          ← 원본 텍스트
├── embedding            ← vector(1536) — OpenAI text-embedding-3-small
├── embedding_model      ← 사용된 임베딩 모델명
├── token_count
├── node_path            ← 노드 계층 경로 배열 (인용용)
├── document_type        ← 빠른 필터링용 중복 저장
├── document_status
├── accessible_roles     ← 접근 가능 역할 목록 (권한 필터)
├── accessible_user_ids  ← 접근 가능 사용자 목록
├── accessible_org_ids   ← 접근 가능 조직 목록
├── is_public            ← 공개 여부
└── is_current           ← 최신 버전 청크 여부 (검색 필터용)
```

## 6. DocumentType (문서 유형)

문서의 동작(청킹 전략, 에디터, 렌더러, 워크플로, 검색 부스트)을 정의하는 설정 엔티티.

```
document_types
├── type_code       ← PK (예: POLICY, MANUAL, REPORT, FAQ)
├── display_name    ← UI 표시 이름
├── description
├── status          ← ACTIVE | INACTIVE
├── schema_fields   ← JSONB — 타입별 metadata 스키마 정의
│   [{"name":"effective_date","type":"date","required":true}, ...]
└── plugin_config   ← JSONB — 플러그인 설정 오버라이드
    {"editor":"richtext-editor-v2","chunking":"section-based", ...}
```

**내장 문서 유형:**

| type_code | 용도 | 청킹 전략 | 특이사항 |
|-----------|------|-----------|---------|
| `POLICY` | 정책/규정 문서 | 조항 단위 (max 512 tokens) | 승인 필수 워크플로 |
| `MANUAL` | 업무 매뉴얼 | 절차 단위 (max 400 tokens) | 단계별 RAG 응답 |
| `REPORT` | 보고서 | 섹션 단위 (max 600 tokens) | 데이터 기반 RAG |
| `FAQ` | FAQ | Q&A 쌍 (max 256 tokens) | 승인 불필요, 간단 워크플로 |

## 7. User / Organization / Role

```
organizations
├── id, name, description
└── status: ACTIVE | INACTIVE

users
├── id (UUID)
├── email (UNIQUE)
├── display_name
├── status: ACTIVE | INACTIVE | SUSPENDED
├── role_name          ← 플랫폼 기본 역할
├── last_login_at
└── created_at, updated_at

roles (시스템 정의)
├── SUPER_ADMIN  ← 플랫폼 전체 관리
├── ORG_ADMIN    ← 조직 범위 관리
├── AUTHOR       ← 문서 작성
├── REVIEWER     ← 문서 검토
├── APPROVER     ← 문서 승인
└── VIEWER       ← 문서 조회만

user_org_roles (N:M 연결)
├── user_id, org_id, role_name
└── 사용자는 조직별로 다른 역할 보유 가능
```

## 8. 감사 및 이력 엔티티

| 엔티티 | 용도 | 보존 기간 |
|--------|------|-----------|
| `audit_events` | API 쓰기 작업 감사 로그 | 90일 |
| `workflow_history` | 워크플로 상태 전이 이력 | Version과 동일 |
| `review_actions` | 검토/승인/반려 액션 상세 | Version과 동일 |
| `change_logs` | 문서 변경 사유 기록 | 문서와 동일 |
| `background_jobs` | 비동기 작업 상태 | 30일 |
