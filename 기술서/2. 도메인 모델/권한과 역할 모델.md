# 권한과 역할 모델

---

## 1. 역할 계층

Mimir는 사용자(USER) 와 에이전트(AGENT) 두 종류의 Principal 을 지원한다.

### 1.1 USER Principal — 6단계 역할 계층

상위 역할은 하위 역할의 모든 권한을 포함한다.

```
SUPER_ADMIN  ← 플랫폼 전체 관리 (사용자/조직/설정)
     │
ORG_ADMIN    ← 조직 범위 관리 (문서/사용자/워크플로)
     │
APPROVER     ← 문서 승인 및 게시
     │
REVIEWER     ← 문서 검토 및 피드백
     │
AUTHOR       ← 문서 작성 및 버전 생성
     │
VIEWER       ← 문서 조회만 허용
```

### 1.2 AGENT Principal — Scope Profile 기반 ACL (S3 Phase 4)

외부 AI 에이전트는 사용자 역할 계층 대신 **Scope Profile 기반 ACL** 을 사용한다.

```
api_keys.principal_kind = "AGENT"
       │
       ▼
agent_principal (api_keys.id 와 1:1)
       │
       ▼
ScopeProfile (scope_profiles 테이블)
  - allowed_tools: JSONB array
  - rate_limit_per_minute
  - read_filter: {published_only, accessible_roles, ...}
       │
       ▼
[MCP 도구 호출 시] AgentPrincipal.can_call_tool(tool_name)
  = manifest.exposure_policy ≠ "never"   (R1)
   AND tool_name ∈ allowed_tools         (R2)
```

자세한 내용은 [Tool ACL 모델](../MCP%20기술서/Tool%20ACL%20모델.md) 및 [보안 모델 §11](../6.%20보안%20모델/보안%20모델.md) 참조.

## 3. 권한 적용 계층

권한은 요청 처리의 4개 계층에서 모두 적용된다.

### 3.1 API 계층

`require_authenticated` / `require_admin_access` dependency를 통해 엔드포인트 진입 시 검사.

```python
# 사용자 인증 필요
actor: ActorContext = Depends(resolve_current_actor)
require_authenticated(actor)

# Admin 권한 필요
require_admin_access(actor)  # ORG_ADMIN 이상
```

### 3.2 서비스 계층

`authorization_service`를 통해 리소스별 작업 권한 검사.

```python
# 문서 편집 권한 확인
authorization_service.check(
    actor=actor,
    resource=ResourceRef(type="document", id=document_id),
    action="write",
)
```

### 3.3 검색 계층

쿼리 실행 시 역할 기반 필터 자동 적용.

```sql
-- 검색 쿼리에 권한 조건 추가
AND (
    d.status = 'published'
    OR (d.created_by = %(actor_id)s)  -- 본인 Draft
)
```

### 3.4 벡터/RAG 계층

`document_chunks` 테이블의 `accessible_roles`, `accessible_user_ids` 컬럼으로 필터.

```sql
WHERE is_current = TRUE
  AND (
      is_public = TRUE
      OR %(role)s = ANY(accessible_roles)
      OR %(actor_id)s = ANY(accessible_user_ids)
  )
```

## 4. 조직(Organization) 기반 권한

사용자는 플랫폼 기본 역할 외에 **조직별 역할**을 별도로 부여받을 수 있다.

```
User A: 플랫폼 기본 역할 = VIEWER
  + 조직 "개발팀"에서 역할 = AUTHOR
  + 조직 "법무팀"에서 역할 = REVIEWER
```

`user_org_roles` 테이블에서 `(user_id, org_id, role_name)` 조합으로 관리.

실제 권한 결정 시: `MAX(플랫폼 기본 역할, 해당 문서의 조직 역할)`을 적용.

## 5. 인증 방법 (Authentication Methods)

| 방법 | 헤더 | 우선순위 | 용도 |
|------|------|---------|------|
| X-Service-Token | `X-Service-Token: <HMAC>` | 1 | 내부 서비스 간 호출 |
| Bearer JWT | `Authorization: Bearer <token>` | 2 | 사용자 인증 (기본) |
| API Key | `X-API-Key: <key>` | 3 | 외부 시스템 연동 |
| Session Cookie | `Cookie: session=<token>` | 4 | 브라우저 세션 |
| Anonymous | (없음) | 5 | 공개 엔드포인트만 허용 |

#### X-Service-Token 검증 방식
```
expected = HMAC-SHA256(INTERNAL_SERVICE_SECRET, "mimir-internal")
검증: hmac.compare_digest(provided, expected)
```

#### JWT 클레임 구조
```json
{
  "sub": "user-uuid",
  "role": "AUTHOR",
  "iat": 1712800000,
  "exp": 1712807200
}
```

## 6. API Key 관리

API Key는 `api_keys` 테이블에서 관리된다.

| 필드 | 설명 |
|------|------|
| `key_prefix` | Key 앞 8자리 (DB 인덱스 조회용) |
| `key_hash` | SHA-256 해시 (원문 저장 안 함) |
| `scope` | READ_ONLY \| READ_WRITE |
| `principal_kind` | `USER` \| `AGENT` (S3 P4) — `AGENT` 면 ScopeProfile 기반 ACL 적용 |
| `scope_profile_id` | `AGENT` 인 경우 연결된 ScopeProfile UUID (S3 P4) |
| `expires_at` | 만료 시각 (NULL = 무기한) |
| `status` | ACTIVE \| REVOKED |

검증 흐름: `prefix로 후보 조회 → SHA-256 해시 비교 → hmac.compare_digest()로 타이밍 공격 방지`

## 7. MCP 도구 등급 (L0~L4) — S3 Phase 4

AGENT Principal 이 호출 가능한 도구는 위험도에 따라 L0~L4 5 등급으로 분류된다.

| 등급 | 의미 | MCP 노출 | 예시 |
|---|---|---|---|
| **L0** | 순수 읽기 (조회) | ✅ 가능 | `search_documents`, `fetch_node`, `read_annotations`, `read_document_render`, `resolve_document_reference`, `search_nodes`, `mimir.vectorization.status` |
| **L1** | 검증 (시간 후 hash 검증 등 read-only) | ✅ 가능 | `verify_citation` |
| **L2** | 가역 쓰기 (propose, draft) — 4 사전 조건 강제 | ✅ 가능 | `save_draft` |
| **L3** | 워크플로 (review/approve/publish) | ❌ 영구 미노출 (R5) | REST 전용 |
| **L4** | 파괴적 (delete/archive/purge) | ❌ **R1 영구 차단** | REST 전용 |

**L2 4 사전 조건**:
1. **Idempotency** — `(agent_id, idempotency_key)` UNIQUE INDEX, 재호출 시 같은 proposal 반환
2. **`requires_human_approval=True`** 강제 (Literal — 자동 머지 영구 금지)
3. **Impact Preview** — `compute_draft_impact()` 가 (chars_added, chars_removed, nodes_added, overwrites_existing_draft) 사전 노출
4. **4 종 Audit Chain** — `agent.proposal.created` / `.updated` / `.replayed` / `.preview_computed`

자세한 내용은 [도구별 명세](../MCP%20기술서/도구별%20명세.md) 및 [매니페스트 정책](../MCP%20기술서/매니페스트%20정책.md) 참조.
