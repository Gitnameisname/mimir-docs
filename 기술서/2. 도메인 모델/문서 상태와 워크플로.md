# 문서 상태와 워크플로

---

## 1. 상태 개요

Mimir의 Version은 두 가지 독립적인 상태를 관리한다:

- **`status`** — 버전의 생명주기 상태 (`draft` / `published` / `archived`)
- **`workflow_status`** — 승인 프로세스 진행 상태 (`draft` / `in_review` / `approved` / `published` / `rejected` / `archived`)

---

## 2. workflow_status 전이 다이어그램

```
                     [submit]
  draft ─────────────────────────► in_review
    ▲                                  │
    │ [reject]                         │ [approve]
    │                                  ▼
    └────────────────────────── approved
                                       │
                                       │ [publish]
                                       ▼
                                  published ────[archive]──► archived
                                       │
                                       │ (새 버전 생성 시 자동)
                                       ▼
                                  (이전 버전은 archived로)
```

### 전이 규칙 및 권한

| 액션 | 전이 | 허용 역할 | 설명 |
|------|------|-----------|------|
| `submit` | `draft` → `in_review` | AUTHOR, ORG_ADMIN, SUPER_ADMIN | 검토 요청 |
| `approve` | `in_review` → `approved` | APPROVER, ORG_ADMIN, SUPER_ADMIN | 승인 |
| `reject` | `in_review` → `draft` | REVIEWER, APPROVER, ORG_ADMIN, SUPER_ADMIN | 반려 (사유 필수) |
| `publish` | `approved` → `published` | APPROVER, ORG_ADMIN, SUPER_ADMIN | 게시 |
| `archive` | `published` → `archived` | ORG_ADMIN, SUPER_ADMIN | 보관 |
| `withdraw` | `in_review` → `draft` | AUTHOR (본인), ORG_ADMIN, SUPER_ADMIN | 검토 취소 |

## 3. 문서 상태와 버전 상태의 관계

```
Document.status               Version.workflow_status
─────────────────             ───────────────────────────────
draft              ◄──────── 활성 Draft 버전이 있음
published          ◄──────── current_published_version_id가 존재
archived           ◄──────── 수동 보관 처리
deprecated         ◄──────── 더 이상 사용하지 않는 문서
```

- 문서에 Published 버전이 없으면 `Document.status = draft`
- 새 버전이 Publish되면 이전 Published 버전은 자동으로 `archived`
- `Document.status`는 `documents_service`에서 버전 상태 변경 시 동기 업데이트

---

## 4. 버전 생성 및 복원 흐름

### 신규 버전 작성

```
POST /documents/{id}/versions
         │
         ▼
    새 Version 생성 (draft)
    version_number = MAX + 1
    parent_version_id = 이전 draft 또는 published 버전
         │
         ▼
    Node 편집
    (POST /versions/{vid}/nodes, PUT, DELETE)
         │
         ▼
    submit → approve → publish
```

### 버전 복원

```
POST /documents/{id}/versions/{vid}/restore
         │
         ▼
    지정 버전의 content_snapshot 복사
    새 Version 생성 (draft)
    restored_from_version_id = 원본 버전 ID
         │
         ▼
    워크플로 재진행
```

## 5. 문서 유형별 워크플로 기본값

| DocumentType | 기본 워크플로 | 승인 필수 여부 |
|-------------|-------------|---------------|
| POLICY | policy-approval-flow | ✅ APPROVER 필수 |
| MANUAL | standard-approval-flow | ✅ APPROVER 필수 |
| REPORT | report-review-flow | REVIEWER 검토만 |
| FAQ | simple-flow | ❌ 직접 publish 가능 |

---

## 6. 워크플로 API 엔드포인트

| 엔드포인트 | 액션 |
|-----------|------|
| `POST /api/v1/documents/{did}/versions/{vid}/workflow/submit` | 검토 제출 |
| `POST /api/v1/documents/{did}/versions/{vid}/workflow/approve` | 승인 |
| `POST /api/v1/documents/{did}/versions/{vid}/workflow/reject` | 반려 |
| `POST /api/v1/documents/{did}/versions/{vid}/workflow/publish` | 게시 |
| `POST /api/v1/documents/{did}/versions/{vid}/workflow/archive` | 보관 |
| `GET  /api/v1/documents/{did}/versions/{vid}/workflow/history` | 워크플로 이력 |

---

## 7. 이력 추적

모든 워크플로 전이는 두 가지 방식으로 기록된다:

1. **`workflow_history`** — 전이 자체 기록 (from_status, to_status, action, actor, comment)
2. **`audit_events`** — 감사 추적 (event_type: `version.workflow.submitted` 등)

검토자/승인자의 상세 코멘트는 `review_actions` 테이블에 별도 저장된다.
