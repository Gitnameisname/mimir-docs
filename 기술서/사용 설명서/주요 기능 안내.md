# 주요 기능 안내

---

## 1. 문서 관리

### 문서 생성

문서는 **Document (메타데이터 컨테이너)** 와 **Version (실제 내용)** 으로 분리된다.

1. `POST /api/v1/documents` — 문서 생성 (Draft 상태)
2. `POST /api/v1/documents/{id}/versions` — 첫 번째 버전 생성
3. `POST /api/v1/versions/{vid}/nodes` — 노드(단락, 제목 등) 추가

### 문서 구조 (트리)

```
문서
└── 버전 (v1.0)
    ├── Node: heading "1장 목적"
    │   └── Node: paragraph "본 정책은..."
    ├── Node: heading "2장 적용 범위"
    │   ├── Node: paragraph "모든 임직원에게..."
    │   └── Node: list_item "개발팀"
    └── Node: heading "3장 비밀번호 정책"
        └── Node: paragraph "최소 8자 이상..."
```

### 문서 유형별 메타데이터

각 문서 유형은 `metadata` 필드에 다른 구조를 요구한다.

**POLICY (정책 문서):**
```json
{
  "effective_date": "2026-06-01",
  "department": "IT보안팀",
  "author": "홍길동",
  "version": "1.0",
  "tags": ["보안", "정책"]
}
```

**MANUAL (업무 매뉴얼):**
```json
{
  "author": "김철수",
  "version": "2.0",
  "category": "IT운영"
}
```

**REPORT (보고서):**
```json
{
  "author": "이영희",
  "period": "2026 Q1"
}
```

## 2. 버전 관리

### 버전 생성 원칙

- 문서당 하나의 **활성 Draft** 버전과 하나의 **게시(Published)** 버전
- 새 버전은 이전 버전을 기반으로 생성
- 버전 번호는 자동 증가 (1, 2, 3...)

### 버전 상태 확인

```bash
GET /api/v1/documents/{id}/versions?filter[status]=draft
```

### 버전 비교 (Diff)

두 버전 간 변경 사항을 노드 단위로 확인:

```bash
GET /api/v1/documents/{id}/versions/{v1id}/diff/{v2id}
```

응답 예시:
```json
{
  "added": [{"node_id": "...", "content": "새로 추가된 단락"}],
  "deleted": [{"node_id": "...", "content": "삭제된 단락"}],
  "modified": [
    {
      "node_id": "...",
      "before": "원본 내용",
      "after": "수정된 내용"
    }
  ]
}
```

### 버전 복원

이전 버전의 내용으로 새 Draft 생성:

```bash
POST /api/v1/documents/{id}/versions/{vid}/restore
{"change_summary": "v2.1 기준으로 롤백"}
```

## 3. 워크플로 (승인 프로세스)

### 전형적인 문서 게시 흐름

```
작성자 (AUTHOR)
  → POST /workflow/submit      # 검토 제출
검토자 (REVIEWER)
  → 코멘트 작성 (선택)
승인자 (APPROVER)
  → POST /workflow/approve     # 승인
  → POST /workflow/publish     # 게시
```

### 반려 흐름

```
승인자
  → POST /workflow/reject      # 반려 (reason 필수)
작성자
  → 내용 수정 후 재제출
```

### 검토 대기 목록

```bash
# 내가 검토해야 할 문서 목록 (REVIEWER/APPROVER)
GET /api/v1/documents?filter[workflow_status]=in_review
```

## 4. 검색

### 전문 검색 (FTS)

키워드로 문서 제목, 내용, 버전 요약을 검색:

```bash
GET /api/v1/search?q=비밀번호&type=document
GET /api/v1/search?q=IT보안&type=all&filter[document_type]=POLICY
```

검색 결과에 하이라이팅 포함 (기본):
```json
"headline": "...최소 <b>비밀번호</b> 길이는 8자 이상..."
```

### 의미 검색 (벡터)

자연어 쿼리로 의미적으로 유사한 청크 검색:

```bash
POST /api/v1/retrieval/search
{"query": "비밀번호 복잡성 요구사항", "document_type": "POLICY"}
```

## 5. RAG 질의응답

### 기본 질의

```bash
POST /api/v1/rag/query
{
  "query": "비밀번호는 몇 자 이상이어야 하나요?",
  "document_type": "POLICY",
  "stream": false
}
```

답변과 함께 **근거(citation)** 가 반환된다:
```json
{
  "answer": "정보보안 정책에 따르면 비밀번호는 최소 8자 이상이어야 하며...",
  "citations": [
    {
      "document_title": "정보보안 정책 v3",
      "node_path": ["3장 비밀번호 정책"],
      "source_text": "비밀번호는 최소 8자 이상이어야 하며..."
    }
  ]
}
```

### 스트리밍 질의 (SSE)

```bash
POST /api/v1/rag/query
{"query": "...", "stream": true}

# 응답 형식 (text/event-stream)
data: {"type": "token", "content": "비밀번"}
data: {"type": "token", "content": "호"}
data: {"type": "citation", "citations": [...]}
data: {"type": "done"}
```

### 멀티턴 대화

```bash
# 첫 번째 질의
POST /api/v1/rag/query
{
  "query": "비밀번호 정책 기준은?",
  "session_id": "my-session-001"
}

# 두 번째 질의 (이전 대화 이력 자동 포함)
POST /api/v1/rag/query
{
  "query": "그럼 분기마다 변경해야 하나요?",
  "session_id": "my-session-001"
}
```

## 6. 관리자 기능

### 사용자 관리

```
Admin UI → 사용자 관리
  - 사용자 목록, 검색, 생성, 역할 변경
  - 비활성화 (물리 삭제 없음)
```

### 문서 유형 관리

```
Admin UI → 문서 유형 관리
  - 새 유형 생성, 플러그인 설정 조회/수정
  - 청킹 전략, RAG 프롬프트, 검색 부스트 조정
```

### 벡터화 상태 모니터링

```
Admin UI → 벡터화 관리
  - 문서별 벡터화 상태 확인
  - 수동 벡터화 요청
  - 토큰 사용량 통계
```

### 감사 로그 조회

```
Admin UI → 감사 로그
  - 이벤트 유형, 사용자, 문서별 필터링
  - 날짜 범위 검색
  - 90일 보존
```

## 8. 외부 AI 에이전트 통합 (MCP) — S3 Phase 4

외부 AI 에이전트는 REST 가 아닌 **MCP (Mimir Control Protocol)** 표면을 통해 정본을 사용한다.

### 8.1 사용 시나리오

| 시나리오 | 사용 도구 | 설명 |
|---|---|---|
| 정본 검색 + 인용 답변 | `search_documents` → `fetch_node` → `verify_citation` | RAG 답변 생성 + pinned citation 보장 |
| 렌더 텍스트 인용 | `read_document_render` → `verify_citation` (citation_basis="rendered_text") | 청크 단위가 아닌 렌더된 본문 인용 |
| 노드 단위 정밀 검색 | `search_nodes` | 청크가 아닌 트리 노드 단위 검색 |
| 문서명/날짜로 lookup | `resolve_document_reference` | "2024년 12월 비밀번호 정책" 같은 자연어 참조 해석 (R3 — `latest_published` default) |
| 주석 조회 | `read_annotations` | 노드의 주석 / mention 조회 |
| 벡터화 상태 | `mimir.vectorization.status` | 문서가 벡터화되었는지 검사 |
| 초안 propose | `save_draft` (L2 — 4 사전 조건 강제) | agent 가 자연어 요청 → 새 draft 제안 (자동 머지 영구 금지) |

### 8.2 도구 사용 권한

각 ScopeProfile 의 `allowed_tools` 화이트리스트에 따라 호출 가능. 운영자가 Admin UI 또는 `PUT /api/v1/admin/scope-profiles/{id}` 로 토글.

| 도구 | 등급 | MCP 노출 |
|---|---|---|
| search_documents / search_nodes / fetch_node / read_annotations / read_document_render / resolve_document_reference / mimir.vectorization.status | L0 | ✅ |
| verify_citation | L1 | ✅ |
| save_draft | L2 | ✅ (4 사전 조건 강제) |
| 워크플로 / 삭제 / archive / publish | L3 / L4 | ❌ R1 영구 차단 — REST 만 허용 |

### 8.3 인용 5중 검증 — 시간 후 재검증

agent 가 답변에 첨부한 인용은 시간이 지나도 **`verify_citation`** 으로 무결성 검증 가능.

```
저장 시점: (document_id, version_id, node_id, content_hash, citation_basis) JSONB 임베드
   ↓ (시간 흐름 — 문서 변경, 새 vN 발행)
재검증: verify_citation 5중 검사
  1. exists       — 좌표 존재
  2. pinned       — vN 이 published/archived (latest 거부)
  3. hash_matches — citation_basis 별 정본 hash 와 비교
  4. quoted_text_in_content — 인용 텍스트가 정본에 포함
  5. span_valid   — span_offset/length 정합
```

자세한 내용은 [Citation 모델](../MCP%20기술서/Citation%20모델.md) 참조.

## 9. 웹훅 연동

문서 이벤트를 외부 시스템에 전달:

```bash
POST /api/v1/webhooks
{
  "url": "https://your-system.example.com/webhook",
  "events": ["document.published", "version.rejected"],
  "secret": "webhook-secret"
}
```

이벤트 발생 시 `POST {url}`로 페이로드 전달:

```json
{
  "event": "document.published",
  "document_id": "550e8400-...",
  "version_id": "660e8400-...",
  "actor_id": "user-002",
  "timestamp": "2026-04-10T14:30:00+00:00"
}
```
