## 1. 구조화 로그

### 형식

모든 로그는 JSON 형식으로 표준 출력에 출력된다.

```json
{
  "timestamp": "2026-04-10T10:00:00.000000+00:00",
  "level": "INFO",
  "logger": "app.api.context.middleware",
  "message": "request completed",
  "request_id": "550e8400-...",
  "trace_id": null,
  "http_method": "POST",
  "path": "/api/v1/documents",
  "status_code": 201,
  "duration_ms": 45.3,
  "actor_id": "user-001",
  "actor_type": "USER",
  "result": "success"
}
```

### 로그 레벨 정책

| 환경 | 앱 레벨 | 외부 라이브러리 |
|------|---------|----------------|
| development | `DEBUG` | `WARNING` |
| staging / production | `INFO` | `WARNING` |

`INFO` 이상 모든 감사 이벤트(인증, 문서 변경, 워크플로 전이)가 로그에 기록됨.  
`WARNING`으로 설정하면 감사 로그가 유실되므로 `INFO` 유지 필수.

### 민감 정보 마스킹

`JsonFormatter._sanitize_record()`가 자동 마스킹:

| 필드명 패턴 | 처리 |
|------------|------|
| `password`, `secret`, `token`, `api_key`, `authorization`, `jwt`, `private_key` | `"***"` 로 교체 |
| 이메일 형식 값 | `us**@example.com` 부분 마스킹 |

## 2. HTTP 요청 로그 (Baseline)

`RequestContextMiddleware`가 모든 요청/응답에 대해 로그를 기록한다.

```
요청 완료 시 → INFO log_request_completion()
  request_id, http_method, path, status_code, duration_ms, actor_id, result
```

**result 분류:**

| result | 조건 |
|--------|------|
| `success` | status < 400 |
| `denied` | status 401/403 |
| `conflict` | status 409 |
| `validation_error` | status 4xx (401/403/409 제외) |
| `failure` | status 5xx |

## 3. Prometheus 메트릭

`GET /api/v1/system/metrics` — Prometheus scrape 엔드포인트 (내부 IP만 접근 가능).

### 수집 메트릭

| 메트릭 | 타입 | 레이블 | 설명 |
|--------|------|--------|------|
| `http_requests_total` | Counter | method, path, status | HTTP 요청 누적 수 |
| `http_errors_total` | Counter | method, path, status | 4xx/5xx 오류 누적 수 |
| `http_active_connections` | Gauge | - | 현재 처리 중인 요청 수 |
| `http_request_duration_seconds` | Histogram (분위수) | method, path | 응답 시간 분포 |

**path 정규화:** UUID를 `{id}`로 치환하여 카디널리티 폭발 방지  
예: `/api/v1/documents/550e8400-.../versions` → `/api/v1/documents/{id}/versions`

## 4. Prometheus + Grafana 설정

### Prometheus 설정 (`deploy/monitoring/prometheus.yml`)

```yaml
scrape_configs:
  - job_name: mimir-backend
    scrape_interval: 15s
    static_configs:
      - targets: ["backend:8000"]
    metrics_path: /api/v1/system/metrics
```

### 알림 규칙 (`deploy/monitoring/alert_rules.yml`)

| 알림 이름 | 조건 | 심각도 |
|----------|------|--------|
| `HighErrorRate` | 오류율 > 1% (5분) | critical |
| `SlowResponseTime` | P95 응답 > 1초 (10분) | warning |
| `ServiceDown` | up == 0 (1분) | critical |

### Grafana 대시보드

```bash
docker compose --profile monitoring up -d
```

- Grafana: `http://localhost:3001`
- 기본 계정: `admin` / `admin` (변경 필요)
- 대시보드: `deploy/monitoring/grafana_dashboard.json` 자동 로드

**패널 구성:**
1. HTTP 요청 수 (requests/s)
2. 오류율 (%)
3. 응답 시간 P50/P95/P99
4. 활성 연결 수

## 5. 로그 수집 파이프라인 (권장)

표준 출력 JSON → 수집 에이전트 → 저장소 → 시각화

```
Docker 컨테이너 표준 출력 (JSON)
     │
     ▼
Fluentd / Fluent Bit / Vector   ← 로그 수집 에이전트
     │
     ├─► Elasticsearch + Kibana   (자체 호스팅)
     └─► CloudWatch Logs          (AWS)
         CloudWatch Insights       (쿼리)
```

### 유용한 CloudWatch / Kibana 쿼리

```
# 오류 요청 조회
result: "failure" OR result: "denied"

# 특정 문서 관련 이벤트
document_id: "550e8400-..."

# 느린 요청 (1초 이상)
duration_ms: [1000 TO *]

# 특정 사용자의 쓰기 작업
actor_id: "user-001" AND http_method: ("POST" OR "PUT" OR "DELETE")
```

## 7. 분산 추적

`X-Trace-ID` 헤더로 분산 추적 ID를 전파한다.

```
외부 시스템 → X-Trace-ID: trace-001 → Mimir API
Mimir 내부 로그: {"trace_id": "trace-001", ...}
```

OpenTelemetry 연동 시 trace_id를 OTel trace context로 매핑 가능.

## 9. MCP 표면 감사 체인 (S3 Phase 4)

외부 AI 에이전트(MCP 클라이언트)의 모든 도구 호출은 별도 audit 체인으로 기록된다.

### 9.1 공통 컨텍스트 필드

모든 MCP audit 이벤트에 다음 필드가 포함됨:

| 필드 | 의미 |
|---|---|
| `actor_kind` | `"AGENT"` (USER 와 구분) |
| `agent_id` | `agent_principal.id` (api_keys.id) |
| `scope_profile_id` | 호출 시점 ScopeProfile UUID |
| `tool_name` | 호출된 MCP 도구명 |
| `request_id` | 동일 요청 묶음 |
| `audited_at` | ISO 8601 timestamp |

### 9.2 도구별 이벤트 종류

| 카테고리 | 이벤트 | 발생 조건 |
|---|---|---|
| ACL 거부 | `mcp.tool.access_denied` | manifest 노출 정책 위반 / ScopeProfile 화이트리스트 미통과 |
| 호출 성공 | `mcp.tool.invoked` | 도구 정상 종료 + envelope 응답 |
| 호출 실패 | `mcp.tool.failed` | 도구 내부 에러 / DB 거부 / 검증 실패 |
| `save_draft` (L2) | `agent.proposal.created` | 신규 proposal 생성 |
| `save_draft` (L2) | `agent.proposal.updated` | 기존 proposal 갱신 (idempotency_key 동일 + content 변경) |
| `save_draft` (L2) | `agent.proposal.replayed` | 동일 (agent_id, idempotency_key) 재호출 — 기존 proposal 반환 |
| `save_draft` (L2) | `agent.proposal.preview_computed` | `compute_draft_impact()` 호출 시 |
| `verify_citation` (L1) | `mcp.citation.verified` | 5중 검사 통과 시 |
| `verify_citation` (L1) | `mcp.citation.broken` | 5중 검사 중 하나라도 실패 — checks 딕셔너리 포함 |

### 9.3 Prometheus 메트릭 (S3 P4)

| 메트릭 | 타입 | 레이블 | 설명 |
|---|---|---|---|
| `mcp_tool_calls_total` | Counter | `tool_name`, `outcome` | MCP 도구 호출 수 (`outcome` ∈ `success`/`denied`/`failed`) |
| `mcp_tool_duration_seconds` | Histogram | `tool_name` | 도구 실행 시간 분포 |
| `mcp_acl_denials_total` | Counter | `tool_name`, `reason` | ACL 거부 횟수 (`reason` ∈ `manifest_exposure`/`scope_profile_disallow`) |
| `mcp_proposals_total` | Counter | `outcome` | save_draft 호출 결과 (`outcome` ∈ `created`/`updated`/`replayed`) |

### 9.4 SLO 추가

| 지표 | 목표 | 측정 |
|---|---|---|
| MCP 도구 응답 P95 | < 800ms (search/fetch) / < 200ms (verify_citation) | `mcp_tool_duration_seconds` |
| ACL 거부율 | < 1% (정상 운영) — 1% 초과 시 ScopeProfile 정합 점검 | `mcp_acl_denials_total` / `mcp_tool_calls_total` |
| Idempotent Replay 정확성 | = 1.0 (절대) | 운영자 골든셋 | 

자세한 내용은 [Tool ACL 모델 §6 audit](../MCP%20기술서/Tool%20ACL%20모델.md) 참조.
