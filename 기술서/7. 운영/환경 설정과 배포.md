## 1. 환경 변수

`.env.example`을 복사하여 `.env` 파일을 생성한다.

```bash
cp .env.example .env
```

### 필수 설정

| 변수 | 설명 | 기본값 |
|------|------|--------|
| `POSTGRES_PASSWORD` | PostgreSQL 비밀번호 | `devpassword` (개발) |
| `JWT_SECRET` | JWT 서명 비밀키 | `dev-jwt-secret-...` (개발) |
| `INTERNAL_SERVICE_SECRET` | 내부 서비스 토큰 | `dev-internal-secret` (개발) |
| `OPENAI_API_KEY` | OpenAI API 키 (임베딩 + RAG) | (비어 있으면 벡터화/RAG 비활성) |

### 선택 설정

| 변수 | 설명 | 기본값 |
|------|------|--------|
| `ENVIRONMENT` | 실행 환경 | `development` |
| `DEBUG` | 디버그 모드 | `false` |
| `ANTHROPIC_API_KEY` | Anthropic API 키 (RAG 대안) | - |
| `LLM_PROVIDER` | LLM 제공자 | `openai` |
| `LLM_MODEL` | LLM 모델명 | `gpt-4o` |
| `CORS_ALLOW_ORIGINS` | 허용 CORS 오리진 (쉼표 구분) | `http://localhost:3000` |

| `RAG_TOP_K` | Retriever Top-K | `20` |
| `RAG_TOP_N` | Reranker Top-N | `5` |
| `GRAFANA_PASSWORD` | Grafana 관리자 비밀번호 | `admin` (개발) |

### DB 연결 설정

| 변수 | 기본값 |
|------|--------|

## 2. 로컬 개발 환경

### 요구사항

- Docker Desktop 4.x 이상
- Python 3.11 (로컬 개발 시)
- Node.js 20 (프론트엔드 로컬 개발 시)

### Docker Compose 실행

```bash
# 전체 서비스 실행 (백엔드 + 프론트엔드 + DB + Valkey)
docker compose up -d

# 로그 확인
docker compose logs -f backend

# 특정 서비스만 재시작
docker compose restart backend
```

### 모니터링 스택 포함 실행

```bash
docker compose --profile monitoring up -d
# Prometheus: http://localhost:9090
# Grafana: http://localhost:3001 (admin/admin)
```

### 개발 서버 직접 실행 (hot reload)

```bash
# Backend
cd backend
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8000

# Frontend
cd frontend
npm install
npm run dev
```

## 3. 배포 파이프라인 (GitHub Actions)

`.github/workflows/ci.yml`

### CI 단계

```
PR/push 이벤트
     │
     ├─► backend-lint
     │   └─ Ruff 린팅 + Bandit 보안 스캔 (HIGH 이상 실패)
     │
     ├─► backend-test (단위 테스트)
     │   └─ pytest + coverage ≥ 80%
     │
     ├─► backend-integration-test (main 브랜치만)
     │   └─ PostgreSQL + Valkey 서비스 컨테이너 포함
     │   └─ 전체 통합 테스트
     │
     ├─► frontend
     │   └─ ESLint + TypeScript 빌드
     │
     └─► ci-gate
         └─ 모든 이전 단계 통과 시에만 성공
```

### CD 단계 (`.github/workflows/deploy.yml`)

```
main 브랜치 push 또는 수동 트리거
     │
     ├─► Docker 이미지 빌드 (multi-stage)
     │   backend:  python:3.11-slim → 비루트 사용자
     │   frontend: node:20-alpine → standalone 출력
     │
     ├─► GHCR (GitHub Container Registry) 푸시
     │   ghcr.io/{owner}/mimir-backend:{sha}
     │   ghcr.io/{owner}/mimir-frontend:{sha}
     │
     ├─► 스테이징 배포
     │   docker compose -f docker-compose.prod.yml pull && up -d
     │
     └─► 운영 배포 (수동 승인)
         rolling update → health check → 실패 시 rollback
```

## 4. Production 배포 설정

`docker-compose.prod.yml` 오버라이드:

```yaml
services:
  db:
    ports: []        # 외부 포트 노출 안 함
    volumes:
      - pg_data_prod:/var/lib/postgresql/data

  valkey:
    ports: []        # 외부 포트 노출 안 함

  backend:
    environment:
      ENVIRONMENT: production
      DEBUG: "false"
      # JWT_SECRET, POSTGRES_PASSWORD 등은 .env 또는 시크릿 관리자에서 주입

  frontend:
    ports:
      - "3000:3000"
```

## 6. DB 마이그레이션 (이중 채널)

Mimir 는 **boot DDL** 과 **Alembic** 두 채널을 함께 사용한다 (S3 Phase 4 기준):

### 6.1 boot DDL (멱등 DDL — startup hook)

`init_db()` 가 startup 에서 멱등 DDL 실행.

```python
# app 시작 시 자동 실행
init_db()  # CREATE TABLE IF NOT EXISTS, ADD COLUMN IF NOT EXISTS
```

용도: 신규 환경 부트스트랩, 작은 컬럼 추가, 기존 환경의 backward-safe 변경.

### 6.2 Alembic (S3 신설)

복잡한 변경 (UNIQUE INDEX, CHECK constraint, JSONB 마이그레이션 등) 은 Alembic 으로 적용.

```bash
cd backend
source .venv/bin/activate
alembic upgrade head
```

| 영역 | 채널 |
|---|---|
| 신규 테이블 (단순) | boot DDL — `CREATE TABLE IF NOT EXISTS` |
| 신규 컬럼 (NULL 허용) | boot DDL — `ADD COLUMN IF NOT EXISTS` |
| 신규 컬럼 (NOT NULL + default) | Alembic |
| UNIQUE INDEX 추가 | Alembic |
| CHECK constraint 추가 | Alembic |
| JSONB 데이터 마이그레이션 | Alembic |
| ENUM 값 추가 | Alembic |

**주의 (S3 P4)**: Alembic revision id 는 PostgreSQL `alembic_version.version_num` 의 `VARCHAR(32)` 제한을 따른다. 32 자 초과 시 적용 실패.

자세한 채널 정책은 [DB 기술서/스키마 정의 §DDL 적용 채널](../DB%20기술서/스키마%20정의.md) 참조.
