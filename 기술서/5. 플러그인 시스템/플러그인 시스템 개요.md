# 플러그인 시스템 개요

---

## 1. 설계 원칙

> **"문서 타입은 하드코딩 금지 — 구조는 generic + config 기반"**  
> (CLAUDE.md 핵심 원칙)

`ChunkingService`, `RAGService`, `SearchService` 코드에 특정 DocumentType을 비교하는 분기(`if type == "POLICY"`)가 없다. 모든 타입별 동작은 **플러그인**이 정의하고, 서비스는 **레지스트리**를 통해 플러그인을 조회한다.

---

## 2. 아키텍처

```
                    DocumentTypeRegistry (싱글턴)
                           │
              ┌────────────┴──────────────┐
              │                           │
         코드 등록 플러그인          DB 오버라이드 플러그인
         (builtin/*.py)            (document_types.plugin_config)
              │
    ┌─────────┼──────────┬──────────┐
    │         │          │          │
 POLICY    MANUAL    REPORT     FAQ
 Plugin    Plugin    Plugin    Plugin
```

### 플러그인 조회 우선순위

```
1. 코드로 등록된 플러그인 (builtin/ 또는 커스텀 등록)
2. DB의 plugin_config로 오버라이드 (Admin UI에서 설정)
3. DefaultDocumentTypePlugin (위 둘 다 없을 때)
```

---

## 3. 플러그인 서브시스템 7개

각 DocumentType 플러그인은 7개 서브플러그인을 통해 동작을 정의한다.

| 서브플러그인 | 클래스 | 역할 |
|-------------|--------|------|
| `MetadataSchemaPlugin` | schema 검증 | metadata 필드 유효성 검사 |
| `EditorPlugin` | 에디터 설정 | 사용할 에디터 컴포넌트 지정 |
| `RendererPlugin` | 렌더러 설정 | 문서 렌더링 방식 지정 |
| `ChunkingPlugin` | 청킹 전략 | 벡터화 청크 분할 방식 |
| `RAGPlugin` | RAG 설정 | 시스템 프롬프트, top_n, max_context_tokens |
| `SearchPlugin` | 검색 설정 | 필드별 검색 부스트 가중치 |
| `WorkflowPlugin` | 워크플로 정책 | 승인 필요 여부, 기본 워크플로 이름 |

## 4. 내장 플러그인 4종

`backend/app/plugins/builtin/` 에 구현됨. 앱 시작 시 자동 등록.

```python
    type_code = "POLICY"

    def get_chunking_plugin(self) -> ChunkingPlugin:
        # 조항 단위 청킹 (max_tokens=512, overlap=50)

    def get_rag_plugin(self) -> RAGPlugin:
        # 정책/조항 인용 시스템 프롬프트
        # top_n=5, max_context_tokens=6000

    def get_search_plugin(self) -> SearchPlugin:
        # policy_number 필드 부스트 5.0

    def get_workflow_plugin(self) -> WorkflowPlugin:
        # requires_approval=True
        # default_workflow="policy-approval-flow"
```

### MANUAL

- 절차 단위 청킹 (max_tokens=400, min_tokens=30, overlap=30)
- 단계별 절차 안내 시스템 프롬프트
- 승인 필요

### REPORT

- 섹션 단위 청킹 (max_tokens=600, overlap=100)
- 데이터 분석 기반 RAG 프롬프트
- REVIEWER 검토만 필요

### FAQ

- Q&A 쌍 청킹 (max_tokens=256, overlap=0)
- 간단 워크플로 (승인 불필요)
- `faq_question`, `faq_answer` 필드 검색 부스트

## 5. DocumentTypeRegistry

플러그인의 중앙 저장소. 싱글턴 패턴.

```python
from app.plugins.base import DocumentTypeRegistry

registry = DocumentTypeRegistry.instance()

# 플러그인 조회
plugin = registry.get("POLICY")

# 플러그인 등록
registry.register(MyCustomPlugin())

# 모든 등록된 타입 조회
all_types = registry.list_type_codes()
```

## 6. ConfigurableDocumentTypePlugin

Admin UI에서 DB 오버라이드를 반영하는 래퍼 플러그인.

```
코드 플러그인 (기본값)
        ↓
ConfigurableDocumentTypePlugin
        ↓ (merge)
DB의 plugin_config (오버라이드)
        ↓
실제 적용 설정
```

`plugin_config JSONB` 서브키:
- `chunking` — max_tokens, min_tokens, overlap
- `rag` — top_n, max_context_tokens, system_prompt
- `search` — boost_fields: {field: weight}
- `workflow` — requires_approval, default_workflow

## 7. 서비스 계층 연동

서비스는 타입별 분기 없이 레지스트리를 통해 플러그인을 호출한다.

```python
# ChunkingService
plugin = registry.get(document_type)
config = plugin.get_chunking_plugin().get_config()
chunks = split_by_config(nodes, config)

# RAGService
plugin = registry.get(document_type)
rag_plugin = plugin.get_rag_plugin()
system_prompt = rag_plugin.build_system_prompt(context)
top_n = rag_plugin.top_n

# SearchService
plugin = registry.get(document_type)
search_plugin = plugin.get_search_plugin()
boost = search_plugin.get_boost_config()
```
