### VULN-001 — X-Service-Token 미검증: 전체 API 무제한 접근 가능 ✅ **패치 완료**

- **파일**: `backend/app/api/auth/dependencies.py:91–134`
- **심각도**: Critical
- **코드**:
  ```python
  def _extract_service_actor(token: str) -> ActorContext:
      # TODO: X-Service-Token 실제 검증 로직 연결 예정 (Phase 8)
      return ActorContext(
          actor_type=ActorType.SERVICE,
          is_authenticated=True,   # 토큰 값 검증 없이 인증됨
          ...
      )
  ```
  ```python
  # authorization.py — SERVICE actor는 모든 액션 허용
  if actor.actor_type == ActorType.SERVICE and actor.is_authenticated:
      return  # RBAC 완전 우회
  ```
- **설명**: `X-Service-Token` 헤더에 임의의 문자열을 넣으면 SERVICE actor로 인증되며, RBAC 전체가 우회된다. 외부 공격자가 이 헤더 하나만으로 전체 API를 무제한 호출할 수 있다.
- **수정 방안**: HMAC 서명 또는 공유 시크릿과의 `secrets.compare_digest()` 비교 추가. Phase 8 전까지는 해당 경로를 내부 IP 범위로만 허용하거나 비활성화.

### VULN-002 — X-Actor 개발 헤더: non-production 환경에서 임의 역할 주입 가능 ✅ **패치 완료**

- **파일**: `backend/app/api/auth/dependencies.py:113–117, 229–243`
- **심각도**: Critical
- **코드**:
  ```python
  if settings.environment != "production":
      actor_id = request.headers.get("X-Actor-Id")
      actor_role = request.headers.get("X-Actor-Role")
      if actor_id and actor_role:
          return _extract_dev_header_actor(actor_id, actor_role)
  ```

### VULN-004 — Bearer 토큰 미검증: X-Actor-Id 헤더로 임의 사용자 사칭

- **파일**: `backend/app/api/auth/dependencies.py:141–168`
- **심각도**: High
- **코드**:
  ```python
  def _extract_bearer_actor(token: str, request: Request) -> ActorContext:
      actor_id = request.headers.get("X-Actor-Id") or None
      # ...
      return ActorContext(
          actor_id=actor_id,
          is_authenticated=bool(actor_id),  # actor_id 존재만으로 인증 성공
          ...
      )
  ```
- **설명**: Bearer 토큰 자체를 검증하지 않는다. `Authorization: Bearer any-value` + `X-Actor-Id: target-uuid`를 보내면 production 환경에서도 임의 사용자로 인증된다.
- **수정 방안**: Phase 8 JWT 연동 시 `token`에서 `actor_id`를 추출하고 `X-Actor-Id` 헤더를 무시하는 방식으로 전환. 연동 전까지 Bearer 경로는 `is_authenticated=False` 반환 유지(실제 토큰 없이는 인증 불가하게).

### VULN-005 — API Key 검증: prefix 8자만 비교, hash 검증 없음

- **파일**: `backend/app/api/auth/dependencies.py:178–211`
- **심각도**: High
- **코드**:
  ```python
  # key_prefix로 후보 조회 후 hash 검증 (현재는 prefix 매칭만)
  prefix = api_key[:8] if len(api_key) >= 8 else api_key
  cur.execute(
      "SELECT ak.issuer_id, u.role_name FROM api_keys ak ... "
      "WHERE ak.key_prefix = %s AND ak.status = 'ACTIVE'",
      (prefix,),
  )
  ```
- **설명**: 키의 앞 8자리 prefix만 비교하고 `key_hash` 컬럼은 검증하지 않는다. prefix를 아는 공격자는 임의 suffix를 붙여 해당 사용자로 인증할 수 있다.
- **수정 방안**: `secrets.compare_digest(hashlib.sha256(api_key.encode()).hexdigest(), stored_hash)` 방식으로 전체 키의 hash 검증 추가.

### VULN-007 — Rate Limiting 미구현: 브루트포스 / DDoS 무방비

- **파일**: 백엔드 전체 API
- **심각도**: High
- **설명**: slowapi, fastapi-limiter 등 rate limiting 미들웨어가 없다. `backend/app/api/errors/handlers.py`에 429 에러 핸들러가 정의되어 있으나 실제로 발동하는 코드가 없다. API 키 brute-force, 반복 인증 시도, 대량 문서 조회가 무제한 가능하다.

### VULN-009 — localStorage에 actorId + role 저장: XSS 발생 시 자격증명 탈취

- **파일**: `frontend/src/hooks/useAuthz.ts:59–68`, `frontend/src/lib/api/admin.ts:27–43`
- **심각도**: High
- **코드**:

### VULN-010 — CSP 및 보안 응답 헤더 미설정 ✅ **패치 완료**

- **파일**: `frontend/next.config.ts`, `backend/app/main.py`

  ```typescript
  ];
  ```

### VULN-011 — 동적 SQL f-string: 구조적 SQL Injection 위험

- **파일**: `backend/app/repositories/users_repository.py:119–120`, `backend/app/api/v1/admin.py:244, 535` 등 다수
- **심각도**: Medium
- **코드**:
  ```python
  where = ("WHERE " + " AND ".join(conditions)) if conditions else ""
  cur.execute(f"SELECT COUNT(*) AS total FROM users {where}", params)
  cur.execute(f"SELECT * FROM users {where} ORDER BY ...", params + [limit, offset])
  
  cur.execute(f"UPDATE users SET {', '.join(fields)} WHERE id = %s RETURNING *", params)
  ```
- **설명**: `conditions`/`fields`의 항목들은 현재 하드코딩된 SQL 조각이지만, f-string으로 쿼리를 조합하는 패턴 자체가 안전하지 않다. 코드 수정 중 외부 입력이 유입되면 SQL Injection으로 직결된다.
- **수정 방안**: 동적 필드명에는 whitelist 검증 후 `psycopg2.sql` 모듈의 `sql.Identifier()` 사용. 패턴을 정책으로 문서화.

### VULN-012 — CORS: allow_credentials + 와일드카드 메서드/헤더 ✅ **패치 완료**

- **파일**: `backend/app/main.py:27–30`
- **심각도**: Medium
- **코드**:
  ```python
  allow_credentials=True,
  allow_methods=["*"],    # DELETE, PATCH 등 모든 메서드
  allow_headers=["*"],    # 모든 커스텀 헤더 포함
  ```
- **설명**: `allow_credentials=True`와 와일드카드를 함께 사용하면 credentialed cross-origin 요청으로 모든 메서드 및 헤더가 허용된다. 악성 사이트가 쿠키/인증 정보를 포함한 DELETE/PATCH 요청을 보낼 수 있다.
- **수정 방안**:
  ```python
  allow_methods=["GET", "POST", "PATCH", "DELETE", "OPTIONS"],
  allow_headers=["Content-Type", "Authorization", "X-Request-Id", "X-Idempotency-Key"],
  ```

### VULN-013 — CORS origins에 내부 서비스(MinIO) 주소 포함

- **내용**:

- **설명**: MinIO 웹 UI 포트(9000, 9091)가 CORS origin으로 허용된다. MinIO UI가 Mimir API에 credentialed 요청을 보낼 수 있는 벡터가 열린다.

## 7. 참고: 안전하게 구현된 항목

- **SQL Injection 방어**: 모든 DB 쿼리에 psycopg2 파라미터화(`%s`) 일관 적용 — 현재 구조에서 직접 SQL Injection은 불가
- **RBAC 실제 동작**: Phase 2 작업으로 `authorization.py`가 인증된 모든 actor에 RBAC 강제 적용
- **XSS 방어 (렌더링)**: `NodeRenderer.tsx`가 React JSX 자동 이스케이핑 사용, `dangerouslySetInnerHTML` 미사용
- **API 에러 코드 구조화**: `ApiError(status, message, code)` 구조로 백엔드 에러 정보를 구조화하여 내부 스택트레이스 미노출
- **audit_events 테이블**: 쓰기 API 성공/실패 시 감사 로그 기록 구현
- **HTTPS 강제**: nginx 설정에서 TLS 종단 예정 (Phase 13)
- **psycopg2 RealDictCursor**: DB row를 객체로 변환 시 타입 보장
