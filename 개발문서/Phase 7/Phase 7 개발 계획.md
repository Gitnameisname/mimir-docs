# Phase 7 개발 계획
## 관리자 UI 구축

---

## 1. 단계 개요

### 단계명
Phase 7. 관리자 UI 구축

### 목적
시스템 운영 및 통제를 위한 관리자 인터페이스를 구축한다.  
이 단계의 목표는 단순 조회 화면을 만드는 것이 아니라, **운영자가 플랫폼 상태를 파악하고**, **권한/정책을 관리하며**, **문서 유형과 외부 연계를 통제하고**, **감사 및 백그라운드 작업을 추적할 수 있는 운영 콘솔**을 확보하는 것이다.

### 기대 결과
- 운영 가능한 플랫폼 완성
- User UI와 분리된 Admin UI 체계 확보
- 권한/감사/인덱싱/외부연계 상태를 관리할 수 있는 운영 화면 제공
- 향후 검색, 벡터화, RAG, 운영 자동화 단계와 연결 가능한 관리 기반 확보

---

## 2. Phase 7의 역할

Phase 7은 이미 구축된 다음 요소들을 **운영 가능한 형태로 시각화하고 제어하는 단계**다.

- Phase 2에서 정의한 권한/역할/감사 모델
- Phase 3에서 구축한 플랫폼 API
- Phase 4~5에서 구축한 문서/버전/워크플로 기능
- 이후 Phase 8~11에서 확장될 검색/벡터/RAG 기능의 운영 제어 기반

즉, Phase 7은 새로운 코어 도메인을 만드는 단계라기보다,  
**기존 플랫폼 기능을 운영자 관점에서 통합 관리할 수 있도록 UI와 관리 흐름을 설계/구현하는 단계**이다.

## 3. 개발 목표

### 3.1 운영자 전용 UI 체계 확립

- API 키 및 외부 연계 설정 관리

### 3.3 감사 및 추적 가능성 확보
- 감사 로그 조회 UI 제공
- 행위 이력 탐색, 필터링, 상세 조회 제공

- 향후 검색/벡터/RAG 운영 화면으로 자연스럽게 확장 가능한 구조 확보
- 설정형 관리 화면과 작업형 관리 화면의 공통 패턴 정립

## 4. 핵심 개발 범위

### 4.1 관리자 UI 공통 프레임 구축
- Admin 레이아웃 설계
- 좌측 메뉴 / 상단 바 / 브레드크럼 구조
- 관리자 홈 대시보드 진입 구조
- 공통 테이블 / 검색바 / 필터 / 상세 패널 / 확인 다이얼로그 패턴 설계
- 관리자 권한 기반 메뉴 노출 제어

### 4.2 사용자 / 조직 / 역할 관리
- 사용자 목록 조회
- 사용자 상세 정보 조회
- 사용자 상태 변경(활성/비활성 등)
- 조직 목록/상세 조회
- 역할(Role) 관리
- 사용자-조직-역할 매핑 관리

### 4.3 권한 정책 관리
- 권한 정책 목록 및 상세 조회
- 역할별 권한 매핑 확인
- 정책 편집 UI
- 정책 변경 시 검증/확인 흐름
- 향후 ACL/리소스 단위 정책 확장 가능 구조 반영

### 4.4 DocumentType 관리
- DocumentType 목록 조회
- 타입별 메타데이터 스키마 확인
- 타입 활성/비활성
- 타입별 editor / renderer / workflow / chunking 관련 설정 확장 포인트 정의
- 신규 타입 추가를 위한 운영 UI 기반 마련

### 4.5 감사 로그 조회
- 감사 이벤트 목록 조회
- 기간/행위자/리소스/이벤트 타입 기준 필터링
- 상세 이벤트 보기
- 변경 전후 값 비교 가능성 고려
- 보안 및 운영상 중요한 이벤트 우선 가시화

### 4.6 시스템 상태 대시보드

  - 총 사용자 수
  - 활성 문서 수
  - 진행 중 작업 수
  - 실패 작업 수
  - 최근 감사 이벤트 수
- 시스템 컴포넌트 상태 표시
- 최근 오류 / 경고 / 작업 현황 요약
- 운영자가 가장 먼저 봐야 하는 정보 중심 구성

### 4.7 벡터화 / 인덱싱 상태 관리
- 인덱싱 대상 및 상태 목록
- 대기 / 진행중 / 완료 / 실패 상태 조회
- 실패 사유 표시
- 재시도 / 재색인 트리거 연계 포인트
- 권한 반영 여부 및 버전 기준 인덱싱 상태 확인 가능성 검토

### 4.8 API 키 및 외부 연계 관리
- API 키 목록 조회 및 발급/폐기 흐름
- 외부 시스템 연계 설정 관리
- 키 노출 최소화 및 보안 경고 처리
- 사용 목적, 발급 주체, 만료 정보 관리
- 향후 AI 도구/외부 앱 연계 정책 반영 가능 구조 확보

### 4.9 백그라운드 작업 상태 관리
- 작업 큐 / 비동기 작업 목록 UI
- 작업 종류, 시작 시간, 상태, 대상 리소스 표시
- 실패 작업 상세 보기

## 6. 구현 원칙

### 6.1 User UI와 완전 분리
- 관리자 화면은 일반 사용자 화면과 메뉴/권한/레이아웃을 분리한다.
- 단순 숨김이 아니라 구조적으로 분리된 라우트와 권한 체계를 가진다.

### 6.2 운영자 중심 UX
- 보기 좋은 화면보다 운영자가 빠르게 판단하고 조치할 수 있는 UI를 우선한다.
- 조회 → 상세 확인 → 조치의 흐름이 짧아야 한다.

### 6.3 권한과 감사 우선
- 관리자 UI의 모든 액션은 적절한 권한 검증과 감사 이벤트 기록을 전제로 한다.
- 민감한 변경은 확인 단계와 변경 사유 입력을 고려한다.

### 6.4 상태 가시성 강화
- 실패, 지연, 충돌, 비정상 상태를 숨기지 않고 명확히 표현한다.
- 정상 상태보다 이상 상태를 더 쉽게 발견할 수 있어야 한다.

### 6.5 후속 Phase 연계성 확보
- 검색, 벡터, RAG, 배포/운영 관리 기능이 뒤에서 확장될 수 있도록 설계한다.
- 현재 화면은 최소 운영 기준을 만족하면서도 확장성을 가져야 한다.

## 11. 최종 정리

Phase 7은 관리자용 화면을 만드는 단계이지만, 본질적으로는  
**Mimir를 실제 운영 가능한 플랫폼으로 전환하는 단계**다.

이 단계에서 중요한 것은 화면 수를 늘리는 것이 아니라,
- 누가 무엇을 관리하는지,
- 어떤 상태를 운영자가 반드시 알아야 하는지,
- 어떤 변경이 추적 가능해야 하는지,
- 어떤 기능이 후속 검색/벡터/RAG 운영으로 이어지는지를

명확하게 구조화하는 것이다.

즉, Phase 7의 산출물은 단순한 UI가 아니라  
**운영 통제 인터페이스의 기반**이어야 한다.
