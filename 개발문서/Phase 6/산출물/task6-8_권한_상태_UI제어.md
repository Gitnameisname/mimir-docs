## 1. 역할별 화면 접근 정책 문서

### 1-1. 역할 정의

| 역할 | 설명 |
|------|------|
| Viewer | 문서 열람만 가능 |
| Author | 문서 작성, 편집, 검토 요청 가능 |
| Reviewer | 검토 및 반려 가능 |
| Approver | 검토, 승인, 발행, 보관 가능 |

> 역할 중복 시 가장 높은 권한 역할이 적용됨

### 1-2. 역할별 화면 접근 매트릭스

| 화면 | Viewer | Author | Reviewer | Approver |
|------|--------|--------|----------|----------|
| `/documents` (목록) | ✓ | ✓ | ✓ | ✓ |
| `/documents/new` (생성) | ✗ | ✓ | ✗ | ✓ |
| `/documents/[id]` (상세) | ✓ (권한 있는 문서) | ✓ | ✓ | ✓ |
| `/documents/[id]/edit` (편집) | ✗ | ✓ (DRAFT/REJECTED만) | ✗ | ✓ |
| `/documents/[id]/versions` (버전 목록) | ✓ | ✓ | ✓ | ✓ |
| `/documents/[id]/versions/[vid]` (버전 상세) | ✓ | ✓ | ✓ | ✓ |
| `/reviews` (검토 대기 목록) | ✗ | ✗ | ✓ | ✓ |

### 1-3. 접근 불가 시 처리

| 상황 | 처리 방식 |
|------|----------|
| 역할 부족으로 화면 접근 불가 | `PermissionDenied` 컴포넌트 (페이지 전체) |
| 라우트 가드 실패 | `/documents`로 리다이렉트 + Toast "접근 권한이 없습니다" |
| 문서 자체 접근 권한 없음 (403) | `PermissionDenied`: "이 문서에 대한 접근 권한이 없습니다" |
| 문서 존재하지 않음 (404) | `ErrorState` (404): "문서를 찾을 수 없습니다" |

## 4. UI Guard 패턴 정의서

### 4-1. Route Guard

```
적용 시점: 페이지 컴포넌트 마운트 전 (Next.js middleware 또는 컴포넌트 레벨)

체크 순서:
  1. 인증 여부 확인 (미인증 → /login)
  2. 역할 확인 (역할 부족 → PermissionDenied 화면)
  3. 문서 상태 확인 (상태 불일치 → 적절한 리다이렉트 또는 안내)
```

### 4-2. Page Action Guard

```
적용 위치: 각 페이지 내 액션 버튼 렌더링 시

예시 로직 (의사코드):
  function canEdit(role, status):
    return (role === 'AUTHOR' or role === 'APPROVER')
      and (status === 'DRAFT' or status === 'REJECTED')

  → true: 버튼 렌더링
  → false: 버튼 미렌더링 (Hidden)
```

### 4-3. Button Guard

```
적용 위치: 개별 버튼 컴포넌트

처리 방식:
  - 조건 미충족 → disabled 속성 + tooltip "현재 상태에서 불가합니다"
  - 역할 자체 없음 → 렌더링 생략 (Hidden)

예시:
  <Button
    disabled={!canApprove(userRole, docStatus)}
    tooltip={disabled ? "검토 중인 문서에서만 승인 가능합니다" : undefined}
  >
    승인
  </Button>
```

### 4-4. Section Guard

```
적용 위치: 페이지 내 특정 섹션 (예: 편집 영역, 승인 패널)

예시:
  - 편집 불가 상태: 편집 영역 전체를 읽기 전용 오버레이로 덮음
  - 승인 패널: Reviewer/Approver 역할만 렌더링
```

## 5. 예외 처리 UX 가이드

### 5-1. Permission Denied

```
화면: PermissionDenied 컴포넌트 (전체 화면 또는 섹션 내)
구성:
  - 🔒 아이콘
  - 제목: "접근 권한이 없습니다"
  - 설명: 구체적 안내 (예: "이 화면은 검토자 이상만 접근 가능합니다")
  - 버튼: [문서 목록으로] 또는 [홈으로]
```

### 5-2. Invalid State (상태 불일치)

```
발생 상황: URL로 직접 접근 시 상태 불일치 (예: PUBLISHED 문서 /edit 접근)
처리:
  - 해당 화면 미렌더링
  - Toast: "현재 상태에서는 접근할 수 없습니다"
  - 자동 리다이렉트: /documents/{id} 상세 화면
```

### 5-3. Stale Data (오래된 상태)

```
발생 상황: 다른 사용자/세션이 문서 상태를 변경한 후, 현재 사용자의 화면이 갱신되지 않은 경우

처리:
  - API 응답에서 예상 상태와 다른 상태 감지 시
  - 배너 표시: "문서 상태가 변경되었습니다. [새로고침]"
  - 워크플로 액션 버튼: 비활성화 처리
```

### 5-4. Server Rejection (서버 검증 실패)

```
발생 상황: 클라이언트에서 허용된 것처럼 보이지만 서버 최종 검증에서 거부

HTTP 상태 → UI 처리:
  403 (권한 없음)  → Toast: "권한이 없습니다" + 상태 재조회
  409 (상태 충돌)  → Toast: "처리할 수 없는 상태입니다. 페이지를 새로고침해 주세요" + 재조회
  422 (검증 실패)  → 해당 폼 필드 오류 표시
  500 (서버 오류)  → Toast: "일시적인 오류가 발생했습니다. 잠시 후 다시 시도해주세요"
```

## 6. UI 제어 원칙 요약

1. **서버 우선**: UI 가드는 보조 장치 — 서버 최종 권한 검증이 항상 존재함
2. **숨김 우선**: 역할 자체가 없는 기능은 숨김. 혼란 최소화
3. **비활성화는 이유 제공**: 상태/조건 불충족으로 불가한 경우 tooltip으로 이유 안내
4. **일관된 패턴**: 동일 조건은 모든 화면에서 동일하게 처리
5. **낙관적 UI 허용**: 단, 서버 실패 시 반드시 복구 UX 제공
