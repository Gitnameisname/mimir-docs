#### [Medium] SEC3-BE-001: SSRF — IPv6 루프백/사설 주소 미차단 ✅ 수정완료

- **파일**: `backend/app/services/alert_notifier.py:26-43`, `backend/app/api/v1/webhooks.py`
- **OWASP**: A10:2021 – Server-Side Request Forgery
- **설명**: `_is_safe_webhook_url()` 함수가 `_BLOCKED_HOST_PREFIXES`로 `127.`, `10.`, `192.168.` 등 IPv4 사설 대역은 차단하지만 IPv6 주소를 고려하지 않음. IPv6 루프백(`::1`) 및 사설 범위(`fc00::/7`, `fe80::/10`) 입력 시 차단 로직을 우회함.

```python
# 현재 코드 (취약)
_BLOCKED_HOST_PREFIXES = ("localhost", "127.", "0.", "169.254.", "10.", "172.", "192.168.")

def _is_safe_webhook_url(url: str) -> bool:
    host = (parsed.hostname or "").lower()
    if any(host.startswith(p) for p in _BLOCKED_HOST_PREFIXES):  # IPv6 우회 가능
        return False
```

- **재현 시나리오**: 공격자가 Webhook URL을 `https://[::1]/admin` 또는 `https://[fc00::1]/internal`로 설정 → `_is_safe_webhook_url` 통과 → 내부 서비스에 HTTP 요청 발생.
- **권장 수정**: `ipaddress` 모듈로 IP 파싱 후 `is_loopback`, `is_private` 체크 추가.
- **수정**: `ipaddress` 모듈 추가. `_is_safe_webhook_url()`에서 `ipaddress.ip_address(host)`로 파싱 후 `.is_loopback`, `.is_private`, `.is_link_local`, `.is_multicast` 체크. IPv6 `[::1]`, `[fc00::1]` 등 차단. `webhooks.py`에도 동일 로직 적용.

#### [Medium] SEC3-BE-002: JWT 토큰 블랙리스트(취소) 미구현 ✅ 수정완료

- **파일**: `backend/app/api/auth/tokens.py`
- **OWASP**: A07:2021 – Identification and Authentication Failures
- **설명**: 액세스 토큰은 15분, 리프레시 토큰은 7일 만료. 로그아웃 또는 계정 탈취 시 발급된 액세스 토큰을 무효화할 수 없음. Refresh Token Family Rotation으로 리프레시 토큰은 탈취 탐지 가능하나 액세스 토큰은 만료까지 유효.
- **재현 시나리오**: 사용자 로그아웃 후 탈취된 액세스 토큰이 최대 15분간 유효 → 탈취 공격자가 API 접근 지속.
- **권장 수정**: 로그아웃 시 JWT `jti` 클레임을 Valkey에 블랙리스트 등록, 인증 미들웨어에서 블랙리스트 조회 추가.
- **수정**: `tokens.py`에 `blacklist_access_token(valkey, jti, ttl)` · `is_access_token_blacklisted(valkey, jti)` 추가 (키: `at_blacklist:{jti}`). 로그아웃 핸들러에서 AT Bearer 헤더를 파싱하여 잔여 TTL+60초로 블랙리스트 등록. `dependencies.py`의 `_extract_bearer_actor`에서 JWT 디코드 후 블랙리스트 체크 → blacklisted 시 anonymous 반환. Valkey 장애 시 fail-open(허용 후 경고 로그)으로 가용성 보장.

#### [Medium] SEC3-BE-003: 문서 읽기 엔드포인트 `require_authenticated=False` ✅ 수정완료

- **파일**: `backend/app/api/v1/documents.py:98, 223`
- **OWASP**: A01:2021 – Broken Access Control
- **설명**: 문서 목록 조회(`GET /documents`) 및 단일 문서 조회(`GET /documents/{id}`) 엔드포인트가 `require_authenticated=False`로 설정되어 비인증 사용자도 접근 가능. TODO 주석으로 의도적 설정임을 표시하고 있으나 기본값이 공개 접근이어서 잘못된 배포 시 위험.

```python
authorization_service.authorize(
    actor=actor,
    action="document.list",
    resource=ResourceRef(resource_type="document"),
    require_authenticated=False,  # TODO: enforcement 활성화 시 True로 전환
)
```

- **재현 시나리오**: 비인증 사용자가 `/api/v1/documents` 반복 호출 → 전체 문서 목록 열거 가능.
- **위험도 평가**: 설계 의도가 공개 문서 플랫폼이라면 허용 범위일 수 있음. 내부 전용이면 위험.
- **권장 수정**: 의도적 공개 접근 정책이라면 문서화 강화. 내부 전용이면 `require_authenticated=True`로 전환.
- **수정**: Mimir는 내부 문서 관리 플랫폼으로 공개 접근이 불필요하다고 판단. `documents.py`의 `list_documents`(line 98) · `get_document`(line 223) 두 곳 모두 `require_authenticated=True`로 변경.

#### [Low] SEC3-BE-004: OAuth 상태(state) 사용 후 명시적 무효화 미구현 ✅ 이미 구현됨

- **파일**: `backend/app/api/auth/oauth_service.py:136-145`
- **OWASP**: A07:2021 – Identification and Authentication Failures
- **설명**: OAuth CSRF 방어용 `state` 파라미터를 Valkey에 10분 TTL로 저장. 토큰 교환 성공 후 state를 명시적으로 삭제하지 않아 TTL 내에 동일 state로 교환을 재시도할 수 있음.
- **재현 시나리오**: 네트워크 오류 등으로 콜백이 두 번 호출될 경우 동일 state로 두 번 처리 시도 가능.
- **위험도 평가**: GitLab이 authorization code를 1회만 허용하므로 실제 악용 가능성은 낮음.
- **확인 결과**: `oauth_service.py:200`에 `valkey.delete(state_key)`가 이미 구현되어 있음. 보고서 기술 오류.

#### [Low] SEC3-BE-005: 속도 제한 프록시 신뢰 설정 미검증 ✅ 수정완료

- **파일**: `backend/app/api/v1/search.py`, `backend/app/main.py`
- **OWASP**: A04:2021 – Insecure Design
- **설명**: slowapi가 `get_remote_address`로 IP 기반 속도 제한 적용. 리버스 프록시(Nginx, ALB 등) 뒤에서 운영 시 실제 클라이언트 IP 대신 프록시 IP로 제한이 적용될 수 있음. 또한 `X-Forwarded-For` 헤더가 악의적으로 스푸핑될 경우 속도 제한 우회 가능.
- **재현 시나리오**: `X-Forwarded-For: 1.2.3.4` 헤더를 임의 값으로 변조하여 속도 제한 우회.
- **수정**: `config.py`에 `trusted_proxy_count: int = 0` 환경변수 추가. `rate_limit.py`에 `_get_client_ip()` 함수 구현 — `trusted_proxy_count=0`이면 REMOTE_ADDR 사용(XFF 무시), N>0이면 XFF 리스트 우측 N개를 프록시로 간주하고 N+1번째(실제 클라이언트) IP만 신뢰. `Limiter(key_func=_get_client_ip)`로 교체.

#### [Medium] SEC3-FE-001: OAuth 리다이렉트 URL 미검증 ✅ 수정완료

- **파일**: `frontend/src/app/account/security/page.tsx:101`
- **OWASP**: A07:2021 – Identification and Authentication Failures
- **설명**: GitLab 계정 연결 시 백엔드 API가 반환하는 `url` 필드를 검증 없이 `window.location.href`에 직접 할당. 백엔드 응답이 변조되거나 MITM 상황에서 악의적 URL로 리다이렉트 가능.

```typescript
// 현재 코드 (취약)
const res = await accountApi.linkGitLab();
window.location.href = res.data.url;  // URL 검증 없음
```

- **재현 시나리오**: MITM 또는 백엔드 취약점으로 `res.data.url`이 `javascript:` 또는 피싱 도메인 URL로 변조 → 사용자가 악의적 사이트로 리다이렉트.
- **수정**: `account/security/page.tsx`의 `handleLinkGitLab()`에서 `new URL(redirectUrl)`로 파싱 후 `protocol === "https:"` 검증 및 `NEXT_PUBLIC_GITLAB_HOST` 환경변수 기반 호스트 검증. 미설정 시 `hostname.includes("gitlab")` 폴백.

#### [Medium] SEC3-FE-002: `useAuthzStore` localStorage 역할 위변조 ✅ 수정완료

- **파일**: `frontend/src/hooks/useAuthz.ts:59-69`
- **OWASP**: A01:2021 – Broken Access Control
- **설명**: `useAuthzStore`가 Zustand `persist` 미들웨어로 `mimir-authz` 키에 `role`, `actorId`를 localStorage에 저장. 브라우저 DevTools 또는 XSS로 localStorage 값을 `SUPER_ADMIN`으로 변조하면 클라이언트 측 `can()` 권한 검사 우회 가능.

```typescript
export const useAuthzStore = create<AuthzState>()(
  persist(
    (set) => ({
      role: "AUTHOR" as UserRole,
      actorId: "00000000-0000-0000-0000-000000000001",
      ...
    }),
    { name: "mimir-authz" }  // localStorage에 role 저장 → 위변조 가능
  )
);
```

- **재현 시나리오**: DevTools → Application → localStorage → `mimir-authz` → `"role":"SUPER_ADMIN"` 수정 → 관리자 UI 렌더링, 관리자 API 호출 시도.
- **위험도 평가**: 백엔드가 모든 요청을 JWT 기반으로 재검증하므로 실제 권한 우회는 불가. 그러나 클라이언트 UI가 의도치 않게 관리자 기능을 노출하고 사용자 혼동 유발 가능.
- **수정**: `useAuthz.ts`에서 `persist` 미들웨어 및 `import { persist }` 제거. 기본 role을 `"AUTHOR"` → `"VIEWER"`(최소 권한)로 변경, actorId 기본값 `""` 변경. Zustand 세션 메모리만 사용.

#### [Low] SEC3-FE-003: 개발 인증 헤더 빌드 타임 의존 ✅ 수정완료

- **파일**: `frontend/src/lib/api/client.ts:57-62`
- **OWASP**: A05:2021 – Security Misconfiguration
- **설명**: 개발 환경 전용 `X-Actor-Id`/`X-Actor-Role` 헤더가 `IS_DEV = process.env.NODE_ENV === "development"` 조건으로 보호됨. 프로덕션 빌드 시 자동 제거되나, 빌드 파이프라인 오설정(NODE_ENV 미지정)으로 프로덕션에 노출될 위험.

```typescript
if (IS_DEV && !at) {
  const { role, actorId } = useAuthzStore.getState();
  if (actorId) headers["X-Actor-Id"] = actorId;
  if (role) headers["X-Actor-Role"] = role;
}
```

- **위험도 평가**: 백엔드가 `debug=True + environment=development` 환경에서만 이 헤더를 수락하므로 프로덕션 백엔드는 무시함. 실질적 위험 낮음.
- **수정**: `client.ts` 조건에 `typeof window !== "undefined"` 추가. SSR 컨텍스트에서 dev 헤더가 전송되지 않도록 차단. TODO(S3-Phase1) 마커 유지.

#### [Low] SEC3-FE-004: 감사 로그 누락 — 민감 작업 미기록 ✅ 이미 구현됨

- **파일**: `backend/app/observability/logging.py`, `backend/app/audit/emitter.py`
- **OWASP**: A09:2021 – Security Logging and Monitoring Failures
- **설명**: `audit_emitter`가 문서 작업에 적용되어 있으나 다음 민감 작업에는 감사 이벤트 미기록:

  - 역할/권한 변경 (Admin 작업)
  - OAuth 토큰 발급/취소
  - API 키 생성/삭제
- **재현 시나리오**: 계정 탈취 사고 발생 시 비밀번호 변경 이력 추적 불가.
- **확인 결과**: `account_router.py`에 비밀번호 변경(`account.change_password`), GitLab 연결(`user.oauth_link_started`), 해제(`user.oauth_unlinked`) 감사 이벤트가 이미 구현됨. `admin.py`에 API 키 생성(`API_KEY_ISSUED`) 및 폐기 감사 이벤트 구현됨. 보고서 기술 오류.
