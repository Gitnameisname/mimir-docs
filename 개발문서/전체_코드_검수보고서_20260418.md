**검수 대상**: Backend (FastAPI/Python) + Frontend (Next.js/TypeScript)

## 1. 개요

Season 2 개발 완료 후 전체 코드베이스에 대한 종합 품질·보안·원칙 준수 검수를 수행한다.  
S1/S2 원칙 준수, 보안 취약점, 코드 품질, 아키텍처 일관성을 포괄적으로 점검한다.

## 3. 백엔드 검수 결과

### 3.1 Critical 이슈

#### [BE-C01] 낮은 테스트 커버리지
- **파일**: `backend/tests/` 전체
- **문제**: 전체 코드의 **4.77%만 커버됨** (816/17,092 라인)
- **미커버 주요 영역**: extraction 서비스, evaluation 서비스, vectorization 서비스, 플러그인 시스템
- **권장조치**: 단위 테스트 + 통합 테스트 확대, 최소 70% 커버리지 목표
- **우선순위**: 즉시

#### [BE-C02] Type Hint 누락
- **파일**: `backend/app/services/extraction/batch_extraction_service.py` (Line 328)
- **문제**: `_fetch_doc_and_schema(conn, document_id, schema_id, schema_version)` 파라미터 타입 힌트 전체 누락
- **권장조치**: `def _fetch_doc_and_schema(conn: Any, document_id: UUID, schema_id: str, schema_version: int) -> tuple[str, dict]:` 형식으로 수정
- **우선순위**: 높음

#### [BE-C03] Unsafe 동적 임포트 패턴
- **파일**: `backend/app/services/extraction/batch_extraction_service.py` (Line 348)
- **문제**: `__import__("psycopg2.extras", fromlist=["RealDictCursor"]).RealDictCursor` 동적 임포트
- **위험성**: 정적 분석 불가, 보안 감사 우회 가능성
- **권장조치**: 모듈 상단에서 `from psycopg2.extras import RealDictCursor` 직접 임포트
- **우선순위**: 높음

### 3.2 High 이슈

#### [BE-H01] DocumentTypeRegistry 적용 불일관
- **파일**: `backend/app/services/chunking_service.py`, `backend/app/services/rag_service.py`
- **문제**: DocumentTypePlugin 플러그인 시스템 설계는 우수하나, 일부 서비스 레이어에서 document_type 문자열 직접 분기 존재 (S1 원칙 ① 위반 가능)
- **권장조치**: 모든 document_type 분기를 `DocumentTypeRegistry` 기반으로 통일
- **우선순위**: 높음

#### [BE-H02] actor_type 로깅 불완전
- **파일**: `backend/app/observability/logging.py` (Line 28-94)
- **문제**: 로깅 프레임워크는 actor_type 지원하나, 배경 작업(background jobs) 및 일부 repository 계층에서 미기록 (S2 원칙 ⑥ 부분 위반)
- **권장조치**: 모든 감사 로그 기록 지점에서 actor_type 필수 기록
- **우선순위**: 높음

#### [BE-H03] 에러 처리 패턴 불일관
- **파일**: `backend/app/repositories/documents_repository.py`, `backend/app/services/extraction/batch_extraction_service.py` 외 다수
- **문제**: try-except 후 None 반환 / ApiError 발생 / 로그만 기록 등 패턴 혼재
- **권장조치**: 에러 처리 정책 표준화 (Critical vs Recoverable 구분, 사내 가이드 문서화)
- **우선순위**: 높음

#### [BE-H04] Dynamic SQL 생성 복잡성
- **파일**: `backend/app/repositories/documents_repository.py` (Line 68-122)
- **문제**: whitelist 기반 SQL 생성은 우수하나, 중첩 JOIN 시 복잡성 증가로 안전성 검증 어려움
- **권장조치**: 해당 SQL 생성 로직에 대한 통합 테스트 강화
- **우선순위**: 중간

### 3.3 Medium 이슈

#### [BE-M01] 개발 전용 인증 헤더 프로덕션 노출 위험
- **파일**: `backend/app/api/auth/dependencies.py` (Line 106-110)
- **문제**: `X-Actor-Id` + `X-Actor-Role` 헤더가 `debug=True` 전용이나, 프로덕션 빌드에서 실수로 활성화될 위험
- **권장조치**: CI/CD 파이프라인에서 `debug=True` 자동 검출 및 배포 차단

#### [BE-M02] JWT Secret 미설정 시 불안전한 폴백
- **파일**: `backend/app/api/auth/dependencies.py` (Line 174-193)
- **문제**: jwt_secret 미설정 시 debug 모드에서 개발 헤더로 폴백 → 프로덕션에서 시크릿 미설정 시 보안 취약점
- **권장조치**: 프로덕션 환경에서 jwt_secret 미설정 시 애플리케이션 기동 중단 (fail-fast)

#### [BE-M03] API Key 추적 실패 무시
- **파일**: `backend/app/api/auth/dependencies.py` (Line 232-305)
- **문제**: `last_used_at` 업데이트 실패 시 오류 무시 → API Key 사용 이력 불완전
- **권장조치**: 추적 실패 시 경고 로그 + 관리자 알림

#### [BE-M04] Pydantic 스키마 적용 불균일
- **파일**: `backend/app/schemas/documents.py` vs extraction/evaluation 도메인
- **문제**: documents 도메인은 우수하나, extraction·evaluation 입력 스키마 검증 강도 미흡
- **권장조치**: 모든 API 입력에 Pydantic 스키마 + 필드 validator 강제

#### [BE-M05] 미사용 SMTP 설정
- **파일**: `backend/app/config.py` (Line 51-57)
- **문제**: SMTP 설정 정의되었으나 실제 메일 발송 로직 미구현 → Dead Code
- **권장조치**: 구현 예정 없으면 제거, 예정 있으면 TODO 명시

## 4. 프론트엔드 검수 결과

### 4.1 High 이슈

#### [FE-H01] dangerouslySetInnerHTML 사용
- **파일**: `frontend/src/features/search/SearchPage.tsx` (Line 53)
- **문제**: `escapeHtmlExceptHighlight()` 기반의 부분 이스케이프 후 innerHTML 사용. 백엔드 `ts_headline()` 출력이 `<b>/<\/b>`만 포함한다는 **가정에 의존**
- **위험성**: 백엔드 응답 변경 시 XSS 가능성
- **권장조치**: DOMPurify 도입 또는 출력 whitelist를 정규표현식으로 엄격 검증

#### [FE-H02] API 응답 런타임 타입 검증 부재
- **파일**: `frontend/src/lib/api/workflow.ts` (Line 32, 39), `frontend/src/lib/api/nodes.ts`
- **문제**: `as unknown[]` → `as WorkflowHistoryItem[]` 직접 캐스팅. 배열 여부만 확인, 각 요소 구조 미검증
- **권장조치**: Zod 스키마 도입으로 런타임 검증 추가

### 4.2 Medium 이슈

#### [FE-M01] localStorage 민감 정보 저장 (임시 구현)
- **파일**: `frontend/src/lib/api/admin.ts` (Line 56-61), `frontend/src/lib/api/s2admin.ts` (Line 36-46)
- **문제**: actor_id, role이 localStorage에 저장 → XSS 취약점
- **현황**: 주석에 JWT 마이그레이션 계획 명시됨 (임시 구현)
- **권장조치**: JWT 마이그레이션 시 localStorage 의존성 완전 제거, HttpOnly Cookie 전환

#### [FE-M02] window 동적 속성 타입 캐스팅
- **파일**: `frontend/src/contexts/AuthContext.tsx` (Line 90-92)
- **문제**: `(window as unknown as Record<string, string>).__mimir_at` 이중 캐스팅 패턴
- **현황**: 레거시 호환성 목적 (주석 명시됨)
- **권장조치**: 전용 Window 인터페이스 선언으로 타입 안전 마이그레이션

#### [FE-M03] 컴포넌트 레벨 Error Boundary 미적용
- **파일**: 주로 `frontend/src/components/chat/` 하위
- **문제**: Next.js route segment error.tsx는 적용되어 있으나, ChatWindow 내 MessageBubble/CitationBlock 레벨 Error Boundary 없음
- **위험**: 깊은 트리 한 컴포넌트 오류 시 전체 ChatWindow 렌더링 실패
- **권장조치**: react-error-boundary 도입, 위험 컴포넌트에 세분화된 ErrorBoundary 적용

#### [FE-M04] 에러 모니터링 미통합
- **파일**: `frontend/src/app/error.tsx`, `frontend/src/components/chat/ChatWindow.tsx`
- **문제**: `console.error` 로깅만 존재, 외부 모니터링(Sentry 등) 미통합
- **권장조치**: 프로덕션 환경에서 Sentry 또는 동등한 에러 추적 서비스 연동

### 4.3 Low 이슈

#### [FE-L01] 개발 환경 인증 헤더 코드 잔존
- **파일**: `frontend/src/lib/api/client.ts` (Line 55-59)
- **문제**: `IS_DEV && !at` 조건부 헤더 — 프로덕션 무해하나 코드 복잡성 증가

#### [FE-L02] 에러 메시지 일관성
- 일부 Toast 메시지가 영어/한국어 혼용

#### [FE-L03] 번들 사이즈 최적화 기회

## 8. 종합 의견

Mimir S2 코드베이스는 **설계 수준에서 S1/S2 원칙을 잘 반영**하고 있다. 플러그인 시스템, Scope Profile 추상화, 다층 인증, 폴백 경로 등 핵심 아키텍처는 견고하다.

주요 개선 영역은:
1. **테스트 커버리지** (4.77% → 70%+): 가장 시급한 개선 사항
2. **타입 안전성**: 백엔드 type hint, 프론트엔드 런타임 검증 강화
3. **일관성**: 에러 처리 패턴, 로깅 패턴의 코드베이스 전체 통일

Critical 이슈 3건은 모두 코드 품질 관련이며, 즉각적인 보안 위협은 없다. High 이슈 6건 중 보안 관련은 BE-H04(SQL 복잡성)로 실질적 SQL Injection 위험은 낮다. 전반적으로 **운영 가능 수준**이나 테스트 커버리지 확대가 장기 안정성을 위한 핵심 과제다.
