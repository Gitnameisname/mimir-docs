## 3. 영향 분석 (제7조 4축)

### 3.1 State (저장 모델)

- 신규 컬럼: `scope_profiles.allowed_tools` (`JSONB NOT NULL DEFAULT '[]'::jsonb`)
- backfill: 기존 행 빈 배열 (default-deny) — 운영자 후속 등록 필수
- 신규 인덱스: `idx_scope_profiles_allowed_tools_nonempty` (운영 진단용)
- Alembic revision id: `s3_p4_scope_profile_allow_tools` (down_revision: `s3_p2_tags`). 컬럼명은 `allowed_tools` 유지, revision id 만 `VARCHAR(32)` 한도로 단축 (31 char)

### 3.2 Intent (명세·계약)

- TOOL_SCHEMAS 의 5 도구에 manifest 4 필드 부착 — 외부 클라이언트의 schema validation 영향 0 (기존 키 모두 유지, 추가형 변경)
- `tools/list` 외부 응답이 `risk_tier`/`status`/`exposure_policy` 비노출로 변경 — 도구등급화_매핑.md §4 옵션 A
- per-actor 필터 — 에이전트의 `tools/list` 응답이 `allowed_tools` 로 좁혀짐
- ScopeProfile API 응답에 `allowed_tools: string[]` 신규 필드

### 3.3 Policy (권한·정책)

- **MCP 표면 전체에 새 진입점 게이트 도입** — 5 도구 모두 ScopeProfile.allowed_tools 검사
- 기존 ACL (ScopeProfile.scopes / settings_json) 와 별 차원 — 중복 의미 없음
- default-deny 정책: 마이그레이션 적용 직후 모든 에이전트가 모든 도구 거부 — **운영자 등록 작업과 같은 운영 창에 배치 필수** (Phase 4 개발계획서 §8 P1 게이트)

### 3.4 Execution (실행 / 가역성)

| 변경 | 가역성 |
|---|---|
| Alembic revision | 가역 — `downgrade()` 가 컬럼 제거 |
| TOOL_SCHEMAS manifest 필드 | 가역 — git revert 로 충분 |
| `_CURATED_TOOLS` / `tools/list` 필터 변경 | 가역 |
| ScopeProfile.allowed_tools 운영자 등록 | 가역 — UI 토글로 즉시 변경 |
| 5 `tool_*` 진입점 게이트 | 가역 — `_check_tool_allowed` 호출 제거 시 본 게이트 무력화 |
