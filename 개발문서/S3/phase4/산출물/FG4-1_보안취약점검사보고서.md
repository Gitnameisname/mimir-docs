# FG 4-1 보안취약점 검사 보고서 — read 응답 envelope 표준

**작성일**: 2026-04-28
**FG**: S3 Phase 4 / FG 4-1
**작성자**: Claude (Design + Implementation Actor)
**검수 대상 파일**: §2 인벤토리

본 FG 의 핵심 보안 가치는 **prompt injection 방어의 응답 메타 표면화**:
read 응답이 명령으로 오해될 수 없음을 envelope 으로 명시 + 위험 메타를 클라이언트에
전달해 **차단 결정을 클라이언트가 내림** (R6 — 차단 아닌 경고).

---

## 1. 위협 모델 (T1~T12)

| # | 위협 | 영향 | 본 FG 의 대응 |
|---|---|---|---|
| T1 | 외부 Chatbot 이 응답 본문을 system prompt 로 오해 → prompt injection 실행 | LLM 명령 탈취 | R4: envelope.instruction_authority 가 Literal["none"] 단일값 — 컴파일 타임 + Pydantic 422 강제 |
| T2 | 검색 결과 본문에 "Ignore previous instructions" 류 페이로드 삽입 → LLM 우회 | jailbreak | `_run_injection_detection` 가 search/fetch/read_annotations 본문 모두 스캔 + risk_mapper 가 directive_pattern (high) 로 매핑하여 envelope.detected_risks 부착 |
| T3 | XSS 류 (`<script>`) 가 응답에 그대로 흘러 클라이언트 렌더링 시점에 실행 | 브라우저 XSS | 본 FG: detected_risks 에 url_obfuscation (medium) 매핑 — 클라이언트 결정. 추가 escape 는 클라이언트 책임 (응답 본문 raw 보존) |
| T4 | `latest` URI 가 응답에 출현 → AI 가 동일 인용을 다른 시점에 다른 본문으로 받음 | 인용 변질 | R3: URI 빌더가 `version_id="latest"` 거부 (ValueError). resolve_latest_version 으로 구체 vN 으로 변환 후에만 URI 출현 |
| T5 | envelope 의 source.uri 가 다른 문서를 가리킴 (citation 변조) | 인용 무결성 깨짐 | _build_envelope 가 raw 데이터의 document_id/version_id/node_id 만 사용 — 외부 입력 직접 사용 안 함. URI 빌더는 빈 문자열 거부 |
| T6 | detected_risks 의 false positive 로 정상 문서 표시 깨짐 | UX | R6: 응답 차단 안 함 — 본문 그대로 반환 + risk 메타만. 클라이언트가 차단 정책 결정 |
| T7 | risk_mapper 가 외부 LLM 의존 → 폐쇄망 (S2 ⑦) 깨짐 | 폐쇄망 호환 깨짐 | risk_mapper 는 `InjectionDetectionResult` 만 의존 — 외부 호출 0. 룰 기반 탐지는 prompt_injection 모듈에서 이미 외부 의존 없음. 회귀 테스트로 확인 |
| T8 | envelope 추가가 외부 클라이언트의 strict schema 거부 | 통합 회귀 | envelope / `_envelope` 필드 모두 Optional + 추가형 — 기존 키 변경 0. 호환성 회귀 테스트 (TestBackwardCompat) 통과 |
| T9 | search_documents 항목별 `_envelope` 부착 시 도큐먼트 ID 없는 결과에 source 빌드 실패 | 노출 정보 누락 (보안 영향 X) | source 빌드 불가 시 envelope 자체 미부착 (`_enrich_search_items_with_envelope::test_no_source_no_envelope`) — 잘못된 source 노출 방지 |
| T10 | `_run_injection_detection` 가 read_annotations 의 본문을 스캔 안 함 (FG 3-3 결정) | annotation 본문 인젝션 노출 | 본 FG 에서 `read_annotations` 의 `annotations[*].content` 도 스캔 추가 — 모든 read 응답 본문 일관 적용 |
| T11 | envelope 빌더 자체에 `system`/`user` 하드코딩 가능성 | R4 우회 | Literal["none"] 컴파일 타임 차단 + 런타임 Pydantic 422. 코드 어디서도 `instruction_authority=` 명시 설정 불가 (모델 default 만) |
| T12 | URI 빌더 / parser 의 정규식 우회 (예: `mimir://documents//versions/v1/nodes/n1`) | 잘못된 URI 노출 | 빌더가 빈 문자열 거부 (ValueError). parser 는 `[^/]+` (1+) 강제 — 빈 segment 거부 |

## 4. 발견 사항 / 결정

| # | 분류 | 발견 | 결정 |
|---|---|---|---|
| 1 | 응답 호환성 | search_documents 의 `_envelope` 가 `SearchResultItem` Pydantic 스키마에 정의되지 않음 (현재는 Any 타입으로 통과) | **의도** — 추가형 변경. 외부 클라이언트가 strict schema 검증 시 unknown key. 필요 시 별 라운드에서 `SearchResultItem` 에 `Optional[dict]` 추가 |
| 2 | LLM 의존 | risk_mapper 는 룰 기반만. LLM 정밀화 미구현 | **본 FG 범위 외** — task4-1 §2.2 이월. R6 (경고만) 자체는 룰 기반으로 충족 |
| 3 | 인젝션 탐지 누락 | verify_citation 의 `content_snapshot` 에 인젝션 탐지 미적용 | **의도** — task4-1 §2.1.4 매핑 표대로. verify_citation 은 tool_metadata 의도이므로 본문 스캔 부적합 |
| 4 | False positive | 정상 한국어 문서가 directive_pattern 으로 잘못 분류될 가능성 | **의도** — R6 보장. 응답 차단 안 하므로 false positive 영향은 메타 노이즈만. LLM 정밀화 (별 라운드) 에서 개선 |
| 5 | URI 정규식 | `[^/]+` 가 segment 안의 `..` / 인코딩 우회를 막지 않음 | **현재 영향 0** — URI 빌더가 외부 입력을 직접 받지 않고 도구 결과의 doc_id/ver_id/node_id 만 사용. 입력 검증은 도구 진입점의 UUID validation 에서 이미 적용 |

## 5. 정합 규칙 (R1~R6 준수 확인)

본 FG 는 R3 / R4 / R6 세 규칙에 직접 영향 — 검수보고서 §1 에 상세. 요약:

- **R1 (L4 차단)**: 본 FG 영향 0 (FG 4-0 에서 종결, 본 FG 는 노출 도구만 다룸)
- **R2 (Scope Profile 단일 ACL)**: 본 FG 영향 0 (FG 4-0 §2.1.6 에서 종결)
- **R3 (pinned)**: ✅ URI 빌더가 `latest` 거부 + resolve_latest_version 헬퍼 + 회귀 테스트
- **R4 (instruction_authority=none)**: ✅ Literal["none"] 컴파일 타임 강제 + Pydantic 422 + 회귀 테스트
- **R5 (REST 1:1 복제 금지)**: 본 FG 영향 0 (응답 직렬화만, 도메인 코어 호출 그래프 변경 없음)
- **R6 (detected_risks 차단 아닌 경고)**: ✅ 응답 차단 분기 0 + 메타 부착만 + 5 페이로드 회귀 (응답 본문 보존 확인)
