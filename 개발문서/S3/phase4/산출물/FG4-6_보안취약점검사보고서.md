## 1. 위협 모델 (T1~T15)

| # | 위협 | 영향 | 본 FG 의 대응 |
|---|---|---|---|
| T1 | agent 가 자신의 proposal 을 self-approve → human review 우회 | 권한 escalation | `agent_proposal_service.approve_draft` 가 actor.role 검사 (S2 자산 그대로). agent (actor_type=AGENT) 는 approve 호출 자체가 거부됨. 본 FG 변경 없음 |
| T2 | idempotency_key 위조 — 다른 agent 의 key 재사용 | 권한 escalation | UNIQUE INDEX (agent_id, idempotency_key) — agent_id 가 검색 키에 포함. 다른 agent 가 같은 key 로 호출하면 새 INSERT (중복 차단 효과 0 — 의도) |
| T3 | tool_save_draft 가 다른 scope 의 document_id 에 draft propose | ACL 우회 | `_check_tool_allowed("save_draft")` + `_ensure_document_allowed(conn, document_id, acl_extra)` 두 단계 게이트. document_id 가 ScopeProfile 외 → ApiNotFoundError → MCPError(NOT_FOUND, 404) |
| T4 | ScopeProfile.allowed_tools 에 save_draft 미등록 시 호출 → 거부 | unauthorized 사용 | `_check_tool_allowed` 가 `can_call_tool` 위임 — `allowed_tools` 미포함 시 MCPError(UNAUTHORIZED, 403). 회귀: TestToolSaveDraft::test_acl_denied_when_tool_not_allowed |
| T5 | agent_id 없는 actor (예: anonymous) 가 save_draft 호출 | 부적절한 audit attribution | 도구 함수 내부에서 `actor.agent_id or actor.actor_id` 가 None 시 MCPError(UNAUTHORIZED, 403). 회귀: TestToolSaveDraft::test_no_agent_id_rejected |
| T6 | 자동 머지 로직 도입으로 reviewer 우회 | human approval 절대 의무 위반 | tool_save_draft 응답 `requires_human_approval=True` 강제 + `MCPWriteEnvelope.requires_human_approval` default True (모델 강제). 자동 머지 코드 경로 0 |
| T7 | impact preview 가 실제 적용과 차이 (preview underestimate / overestimate) | 사용자 의사결정 오도 | 본 FG 는 단순 차이 산출. 운영 시 audit metadata 에 preview + actual 양쪽 기록 — drift 발견 시 본 도구 status="disabled" 즉시 차단 (별 라운드 운영 모니터링) |
| T8 | impact preview 계산 중 race condition (다른 draft 와 동시 변경) | 잘못된 preview | `compute_draft_impact` 가 단일 SQL 트랜잭션에서 SELECT — 격리 수준 READ COMMITTED 전제. 동시 draft 는 propose 시점에 UNIQUE 제약으로 차단 가능 (별 라운드) |
| T9 | content_snapshot 에 prompt injection 페이로드 포함 → reviewer LLM 조작 | LLM 공격 | 본 FG 의 도구는 본문 검색 결과를 반환하지 않음 — content_role=mutation_proposed. 도구 응답 자체에는 인젝션 영향 0. reviewer UI 측 안전성은 별 작업 |
| T10 | save_draft 응답을 외부 클라이언트가 read envelope 로 오해 | 호환 깨짐 | `MCPResponse.envelope` (read) vs `write_envelope` 분리 — 클라이언트가 어느 응답인지 schema 로 구분 가능 |
| T11 | L3/L4 도구가 본 FG 작업 중 실수로 노출 | R1 위반 | TestL3L4PermanentExclusion 4 케이스 + FG 4-4 contract drift 16 케이스 — 머지 차단 |
| T12 | event_types.md 4 신규 events 가 emit 되지만 처리 누락 (오케스트레이터 미구현) | audit chain 누락 | 본 FG 가 `agent_proposal.requested` 만 emit (tool_save_draft 시점). 나머지 3 events 는 reviewer / merge / rollback 시점에 별 코드가 emit — 본 FG 범위 외 (event_types.md 등재만) |
| T13 | Alembic revision 적용 후 운영 중 idempotency_key 가 NULL 이라 UNIQUE INDEX 효과 없음 | idempotent 보장 깨짐 | UNIQUE INDEX `WHERE idempotency_key IS NOT NULL` — NULL 행은 제약 외. 명시 입력 시에만 idempotent. 호출자 (tool_save_draft) 가 SaveDraftRequest 의 필수 필드로 강제 — UI 클라이언트가 실수로 NULL 보낼 수 없음 |
| T14 | content_snapshot dict 가 매우 큰 트리 (1MB+) → DB 성능 저하 | DoS / 자원 남용 | 현재 입력 길이 제약 없음. 운영 모니터링 후 별 라운드 (예: max_content_size 환경변수). agent_proposal_service 의 기존 동작 — propose_draft 의 INSERT 자체가 문제 시 트랜잭션 롤백 |
| T15 | tool_save_draft 가 외부 클라이언트에 의해 자동화 폭주 (rate-limit 부재) | DoS | mcp_router 의 기존 rate limit (`_MCP_TOOL_LIMIT`=20/min, agent별 valkey rate limit) 그대로 적용. 본 FG 추가 보호 없음 — 운영 모니터링 후 별 라운드 |

## 5. 정합 규칙 (R1~R6 준수 확인)

본 FG 는 R1 (강화) / R2 (진입점 게이트) / R4 (write_envelope) 에 직접 영향. 검수보고서 §1 에 상세.

- **R1 (L4 차단)**: ✅ 강화 — TestL3L4PermanentExclusion 4 케이스 + FG 4-4 contract drift 회귀
- **R2 (Scope Profile 단일 ACL)**: ✅ tool_save_draft 진입점 게이트 + `_ensure_document_allowed`
- **R3 (pinned)**: 영향 0 — 본 FG 는 인용 미생성
- **R4 (instruction_authority=none)**: ✅ MCPWriteEnvelope Literal 강제
- **R5 (REST 1:1 복제 금지)**: ✅ agent_proposal_service.propose_draft (도메인 코어) 위임
- **R6 (detected_risks 경고)**: 영향 0 (본 FG 는 본문 검색 결과 미반환). 대신 human approval 패턴이 R6 의 정신 (자동 차단 0) 을 구현
