## 1. 위협 모델 (T1~T11)

| # | 위협 | 영향 | 본 FG 의 대응 |
|---|---|---|---|
| T1 | 외부가 `version_id="latest"` 를 인용으로 사용 → 시간 후 다른 vN 본문이 나와 인용 변질 | 인용 신뢰성 붕괴 | **R3 강제** — `tool_verify_citation` 진입점에서 `latest` 즉시 `MCPError(INVALID_REQUEST, 400)` |
| T2 | content_hash 위조 (외부가 임의 hash 를 보내 verified=True 받기 시도) | 인용 위조 | hash_matches 가 실 정본 (`document_chunks.source_text` 또는 rendered_text) SHA-256 과 비교 — 위조 hash 는 자동 mismatch |
| T3 | citation_basis 입력을 잘못 명시 (실제 node 인용을 rendered 라고 주장) | mismatch 감지 안 됨 | citation_basis 별 분기 — 클라이언트가 명시한 basis 의 정본 hash 와 비교. 잘못 명시 시 hash 자동 mismatch (안전한 fail) |
| T4 | quoted_text 가 본문에 포함되지 않은 가짜 인용 | 거짓 인용 | 검사 4 (quoted_text_in_content) — 미포함 시 verified=False |
| T5 | span_offset / span_length 가 본문 범위 밖 (음수, overflow) | 잘못된 좌표 | 검사 5 (span_valid) — 음수 / overflow 거부 |
| T6 | draft 버전을 인용으로 저장 → 작성 중 본문이 변할 때마다 인용 깨짐 | 인용 불안정 | 검사 2 (pinned) — published / archived 만 통과. draft 거부 |
| T7 | 다른 scope 의 노드 검증 시도 (ACL 우회) | 정보 누설 | `_ensure_document_allowed(conn, document_id, acl_extra)` 호출 — fetch_node 와 동일 게이트. ApiNotFoundError → MCPError |
| T8 | rendered_text 모드의 hash 가 비결정 (render 정책 변동) → 항상 mismatch | 가용성 | `_walk_blocks_for_text` 단일 정본 + plain_text rstrip 정규화. 결정성 회귀 (FG 4-2 와 공유) |
| T9 | render_service 호출 중 예외 → 빈 문자열로 fall through → hash 계산 불가 | 가용성 | `try/except` 에서 `rendered=""` fallback. text_for_check 가 빈 문자열이면 hash mismatch 자연 처리 (안전한 fail-closed) |
| T10 | quoted_text 가 매우 긴 페이로드로 메모리 폭증 | DoS | Pydantic 의 default 제약 + Python `in` 연산자 자체가 효율적. quoted_text 별도 길이 제약 없음 — 운영 모니터링 시 추가 (별 라운드) |
| T11 | citation_basis Literal 외 값 입력 | 422 | Pydantic Literal 자동 거부 (TestSchemaContracts::test_citation_basis_literal_validation) |
