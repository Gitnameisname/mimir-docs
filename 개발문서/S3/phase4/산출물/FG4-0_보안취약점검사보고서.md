## 1. 위협 모델 (T1~T15)

| # | 위협 | 영향 | 본 FG 의 대응 |
|---|---|---|---|
| T1 | 에이전트가 ScopeProfile 외 도구 호출 시도 | tool 자체로 권한 escalation | `_check_tool_allowed` 5 진입점 게이트 (R2) — 빈 allowed_tools = default-deny |
| T2 | L4 도구 (publish/reindex/change_schema/delete_document) 가 코드 변경 중 실수로 MCP 노출 | 비가역 데이터 손실 | (1) TOOL_SCHEMAS 에 L4 항목 등재 0 검증 (`test_mcp_l4_blocked.py::TestL4DefinitionSafetyNet`) (2) `is_tool_mcp_exposed` 가 risk_tier=L4 / maturity=forbidden / status=not_exposed 차단 (3) `_CURATED_TOOLS` 가 manifest 필터 결과만 포함 |
| T3 | 외부 응답에 운영 정보 (risk_tier / status) 노출 → 우회 시도 정찰 | 정찰 정보 제공 | `mcp_exposed_public_view` 가 외부 view 에서 risk_tier / status / exposure_policy 제외 (옵션 A 보수) |
| T4 | ScopeProfile.allowed_tools 에 임의 도구 이름 등록 → manifest drift | 미정의 도구 호출 시도 | `_allowed_tools_validate` 가 `known_tool_names()` 외 거부 (Repository CRUD 전수 적용) |
| T5 | Repository 조회 실패 시 게이트 무력화 (fail-open) | 정의되지 않은 동작 | `can_call_tool` 의 catch-all `except Exception: return False` — fail-closed |
| T6 | scope_profile_id 가 None 인 에이전트 → 게이트 우회 | bypass | `can_call_tool` 첫 분기 — `not self.scope_profile_id: return False` |
| T7 | actor_type 위변조 (anonymous → user 자동 승격 등) | 권한 escalation | actor_type 결정은 `resolve_current_actor` 가 이미 검증 — 본 FG 는 actor 검증 외부에서 게이트만 적용 |
| T8 | 마이그레이션 적용 후 운영자 등록 전 → 모든 에이전트 호출 거부 (서비스 영향) | 운영 가용성 | 의도된 default-deny — 운영 가이드 (검수보고서 §8) 에 적용 창 명시 |
| T9 | Admin UI 가 `KNOWN_MCP_TOOLS` 와 백엔드 TOOL_SCHEMAS drift | 운영자가 잘못된 토글 표시 | 본 FG 의 코드/문서 정합은 검수 시 수동 확인. FG 4-5 (이월) 동적 fetch 로 자동 정합 |
| T10 | per-actor 필터의 cache stale → 변경된 allowed_tools 반영 지연 | 권한 변경 지연 | 캐시 미적용 — 매 `tools/list` 호출마다 ScopeProfile lookup. 트레이드오프: 성능 vs 일관성 — 일관성 우선 |
| T11 | ScopeProfile.allowed_tools 변경 audit 누락 → 누가 정책 변경했는지 추적 불가 | 보안 사고 시 책임 추적 불가 | 신규 event_type `scope_profile.allowed_tools.changed` 신설. metadata: `{before, after, affected_agents}` |
| T12 | Alembic upgrade 시 `IF NOT EXISTS` 미적용 → 재실행 실패 | 운영자 적용 어려움 | 본 revision 은 `ADD COLUMN IF NOT EXISTS` + `CREATE INDEX IF NOT EXISTS` — 재실행 안전 |
| T13 | manifest extract 스크립트가 staging / prod 에서 다른 결과 | 운영 정보 일관성 깨짐 | 추출은 코드 상수 (`TOOL_SCHEMAS`) 만 의존 — 환경 무관. CI drift 검사 (`--check`) 로 detect |
| T14 | 정의되지 않은 MCP 도구 이름이 `tools/call` 로 들어오면 | INVALID_REQUEST 정상 거부 | `_dispatch_tool` 의 `if tool_name not in _CURATED_TOOLS` 에서 거부 (변경 없음) |
| T15 | `tools/list` 응답에 `allowed_tools` 외 도구가 노출됨 (per-actor 필터 우회) | 에이전트가 비허용 도구 인지 후 호출 시도 | `mcp_list_tools` 에서 `actor.is_authenticated and actor.is_agent` 일 때 `_agent_allowed_tools` 적용. 비허용 도구는 응답 자체에서 제외 |

## 3. "뷰 ≠ 권한 준수" 시나리오 검증 (S3 P3 패턴 적용)

| 시나리오 | 결과 |
|---|---|
| 빈 allowed_tools 인 에이전트가 search_documents 호출 | 403 거부 (T1) |
| allowed_tools=["read_annotations"] 인 에이전트가 fetch_node 호출 | 403 거부 (T1) |
| allowed_tools=["search_documents"] 인 에이전트가 search_documents 호출 | 200 정상 |
| 에이전트의 `tools/list` 응답에 비허용 도구 표시 | 표시 안 됨 (T15) |
| user actor 가 search_documents 호출 (scope_profile_id 없음) | 통과 (T7 — 사람은 별 게이트) |
| anonymous 가 어느 도구든 호출 | mcp_router 의 `is_authenticated` 검사에서 401 차단 (변경 없음) |
| L4 도구 이름을 ScopeProfile.allowed_tools 에 등록 시도 | 422 거부 (T4 — `_allowed_tools_validate` 가 known_tool_names() 검사) |
| Repository 조회 실패 (DB down) 상태에서 도구 호출 | 403 거부 (T5 — fail-closed) |
| ScopeProfile.allowed_tools 변경 후 다음 호출 즉시 반영 | 즉시 반영 (T10 — 캐시 미적용, 매 호출 lookup) |
| Alembic 적용 후 운영자 등록 전 도구 호출 | 모든 거부 (T8 의도) |

## 4. 발견 사항 / 결정

| # | 분류 | 발견 | 결정 |
|---|---|---|---|
| 1 | 운영 영향 | default-deny 적용 후 운영자 등록 전 모든 에이전트 호출 거부 | **의도** — 운영 가이드 (검수보고서 §8) 에 같은 운영 창 배치 명시. 옵션 B (`--bootstrap-with-l0`) 로 즉시 운영 재개 가능하나 보안 트레이드오프 명시 |
| 2 | 캐시 부재 | per-actor 필터 가 매 `tools/list` 호출 시 ScopeProfile lookup | **의도** — 일관성 우선. 모니터링 후 별 라운드에서 필요 시 캐시 + invalidation |
| 3 | UI drift 위험 | Admin UI `KNOWN_MCP_TOOLS` 가 백엔드 TOOL_SCHEMAS 와 정합 수동 유지 | 별 라운드 — FG 4-5 (이월) 의 `/api/v1/admin/mcp/manifest` endpoint 신설 시 동적 fetch 로 교체 |
| 4 | 두 ACL 패턴 공존 | scope_filter (S2 search/fetch/verify) ↔ authorization_service (S3 vectorization/annotations) | **본 FG 범위 외** — 별 ADR. 본 FG 의 진입점 게이트는 두 패턴 모두를 감싸므로 보안 영향 없음 |
| 5 | INVALID_PARAMS enum 부재 | tools.py 의 `MCPErrorCode.INVALID_PARAMS` 사용 — `errors.py` 에 정의되지 않음 (pre-existing) | 별 라운드 — FG 4-1 envelope 표준화 시 점검 |
| 6 | `/mcp/resources` 리스트 ACL 우회 | 직접 SQL JOIN — Scope Profile ACL 거치지 않음 (Pre-flight §2 끝) | 별 라운드 — 별 보안 라운드 |
| 7 | 인젝션 탐지 누락 | verify_citation / read_annotations 응답 본문에 `_run_injection_detection` 미적용 | FG 4-1 envelope 표준화 시 통일 (Pre-flight §7) |
