### 1.3 배포 후 확인

#### 1.3.1 도구 호출 동작

```bash
# verify_citation 호출 — 신규 checks 필드가 응답에 포함됨을 확인
curl -X POST https://your-mimir/mcp/tools/call \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "tool_name": "verify_citation",
    "params": {
      "document_id": "<test-doc>",
      "version_id": "<vN>",
      "node_id": "<node>",
      "content_hash": "<hash>"
    }
  }' | jq '.data.checks'
```

기대 응답:
```json
{
  "exists": true,
  "pinned": true,
  "hash_matches": true,
  "quoted_text_in_content": null,
  "span_valid": null
}
```

#### 1.3.2 R3 거부 동작

```bash
# version_id="latest" 입력 → 400 INVALID_REQUEST
curl -X POST https://your-mimir/mcp/tools/call \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "tool_name": "verify_citation",
    "params": {
      "document_id": "<test-doc>",
      "version_id": "latest",
      "node_id": "<node>",
      "content_hash": "<hash>"
    }
  }'
# expect: HTTP 400, error.code = "INVALID_REQUEST", message contains "pinned"
```

## 3. 호환성 영향

### 3.1 백워드 호환 (✅ 보존)

| 항목 | 동작 |
|---|---|
| `VerifyCitationRequest` 의 신규 필드 (`citation_basis` / `quoted_text` / `span_length`) | 모두 Optional/default — 기존 클라이언트 호환 |
| `VerifyCitationData` 의 기존 필드 (`content_snapshot` / `hash_matches` / `version_valid`) | 유지 — 기존 클라이언트 호환 |
| `citation_basis` default = `node_content` | 기존 호출 패턴 유지 |
| 기존 5-tuple 인용 검증 동작 | ✅ — 5중 검증 중 1~3 만 적용 (4/5 는 None 통과) |

### 3.2 비호환 변경 (⚠️ 운영 영향)

| 항목 | 영향 | 대응 |
|---|---|---|
| `version_id="latest"` 입력 → 400 거부 (R3 강제) | 기존 클라이언트가 latest 로 verify 호출 시 거부 | 클라이언트가 인용 시점에 vN resolve 후 저장하도록 마이그레이션 필요 |

### 3.3 모니터링 항목

- `MCPError(INVALID_REQUEST, 400)` 발생 빈도 — verify_citation 거부 사례 추적
- `verify_citation` 응답 시간 — rendered_text 모드는 render_service 호출이 추가됨
- `Citation-present rate` / `Faithfulness` 변동 (FG4-3_AI품질평가보고서 참조)
