## 1. 검수 범위

본 FG 는 **현재 커버리지 35.44% → 80%** 복구의 **측정·게이트 인프라** 를 담당한다. 사용자 승인(2026-04-23) 으로 "기반 구축 + CI 게이트 중심 한 세션" 모드를 채택했다. **테스트 추가(수십~백여 건)는 후속 세션 인수 범위**로 분리된다.

### 1.1 스코프 분할 근거

- 백엔드 서비스 계층만 ~9,000줄, 리포지토리 ~2,100줄. 80% 도달에 1,500~3,000줄 서비스 유닛 테스트 + 5,000~7,000줄 리포지토리 테스트 필요 (작업지시서 §1 "3~5일" 명시).
- 샌드박스 환경에 Docker/psycopg2 부재 → pytest 실행 불가 → 테스트 작성 시 실 검증 불가.
- 사용자와 협의 후 "인프라 + 게이트 + 후속 인수 문서" 로 스코프 확정.

### 1.2 검수 대상 산출물

| 파일 | 역할 |
|------|------|
| `backend/scripts/coverage_baseline.py` | 베이스라인 측정·검증·markdown 생성 스크립트 |
| `backend/pyproject.toml` (수정) | coverage 설정 보강 (omit/exclude_lines/fail_under/precision/html/xml) |
| `backend/.gitlab-ci.yml` (신규 운영, backend repo 루트) | CI 에 coverage 합산·게이트·artifact 업로드 스텝 추가 |

### 3.1 정적 검증

### 3.2 스크립트 기능 검증 (실제 실행)

샌드박스에서 다음을 실제로 수행했다 (기존 stale `coverage.xml` 대상):

| 명령 | 결과 |
|------|------|
| `python3 scripts/coverage_baseline.py --xml coverage.xml` | 32개 패키지 요약 + 작업 우선순위 Top 20 출력 |
| `python3 scripts/coverage_baseline.py --xml coverage.xml --check` | exit 1 + stderr 에 3건 미달 항목 (overall 6.45, services 8.94, repos 3.21) |
| `python3 scripts/coverage_baseline.py --xml coverage.xml --markdown` | `FG0-3_베이스라인.md` 본문 128줄 생성 |

스크립트는 의도대로 동작한다.

### 3.3 CI 워크플로 논리 흐름

```
1. checkout + Python 3.11 + pip install (testcontainers 포함)
2. Wait for services (PG + Valkey)
3. CREATE EXTENSION vector + alembic upgrade head
4. Run integration tests (--cov=app --cov-append --cov-report=)
5. Run unit tests (--cov=app --cov-append --cov-report=)
6. Generate combined coverage report (coverage xml + coverage html)
7. Coverage gate (scripts/coverage_baseline.py --check)   ← 게이트
8. Emit FG0-3 baseline markdown (artifact용)
9. Upload test & coverage results (artifact)
```

`--cov-append` 로 통합·단위 테스트 커버리지가 합산되고, 최종 합산된 `coverage.xml` 에 임계값을 강제한다.

### 3.4 근거 데이터 첨부 (Phase 0 산출물 규약 §3.1)

- **스크립트 자체 동작 로그** — §3.2
- **스냅샷 기반 베이스라인 리포트** — `FG0-3_베이스라인.md` (auto-generated 섹션 §1~§4)
- **pyproject / workflow diff** — §2.4 / §2.3

**아직 미첨부 (후속 인수 조건)**:

- 실 DB + 통합 테스트 합산 후 coverage.xml → 실제 ≥ 35.44% 확인

## 4. 리스크와 완화

| 리스크 | 심각도 | 완화 |
|--------|-------|------|
| "기반만 있고 실제 80% 미달성" 상태가 장기화 | **중** | `FG0-3_진행가이드.md` 에 후속 세션 절차 명문화 + 본 검수보고서 결론에 "실질적 완료" 조건 명시 |
| `--cov-append` 가 test data file(`.coverage`) 을 공유해 충돌 | 낮 | `parallel = false` 로 명시 + 단일 러너 내 순차 실행이라 충돌 없음 |
| coverage_baseline.py 가 stdlib 만 쓰는지 확인 | 낮 | `import argparse / sys / xml.etree / dataclasses / pathlib` 만 사용 — 실 실행으로 증명 |
| CI 에서 임계값 미달 시 artifact 업로드 누락 | 낮 | Upload 스텝에 `if: always()` 명시 — gate 실패해도 진단용 artifact 확보 |
| 의미 없는 "행 통과용" 테스트 양산 (task0-3 §6) | **중** | `FG0-3_진행가이드.md` §3.4 에 "테스트 리뷰 체크리스트" 명시 (assert 의미성, 에러 분기 커버, Flaky 방지) |
| Flaky 테스트 유입 | 낮 | 작업지시서 §6 원칙 준수 — `@pytest.mark.flaky` 임시 사용 금지, 발견 시 격리 후 재설계 |

## 5. 결론 — "선언적 완료 vs 실질적 완료"

Phase 0 산출물 규약 §3.2 기준 재평가:

| 기준 | 충족 여부 | 근거 |
|------|----------|------|
| 전체 백엔드 커버리지 ≥ 75% | ⏸ **후속 인수** | 본 세션은 게이트 구축. 실 도달은 후속 세션에서 `services/` + `repositories/*` 유닛 테스트 추가 후 달성 |
| `app/services/` ≥ 80% | ⏸ **후속 인수** | 동일 |
| `app/repositories/` ≥ 80% | ⏸ **후속 인수** | 동일 |
| CI thresholds 미달 시 실패 재현 | ✅ 설계 검증 | 샌드박스 실행으로 `--check` exit 1 확인 (6.45% 스냅샷 케이스). 실 재현 스크린샷은 운영자 작업 |
| 검수·보안보고서 제출 | ✅ | 본 문서 + `FG0-3_보안취약점검사보고서.md` |

본 FG 는 **"측정·게이트 인프라 선언적 완료"** 상태다. **실질적 완료**는:

1. 후속 세션에서 `FG0-3_진행가이드.md` §3 절차 수행
2. `coverage_baseline.py --check` 가 CI 에서 exit 0 을 3회 연속 반환
3. `FG0-3_베이스라인.md` 재생성본에서 §3 임계값 테이블이 3개 모두 ✅
4. 일부러 한 줄 주석 → CI 실패 스크린샷 첨부

의 4단계로 전환된다. Phase 0 게이트 (§5) 중 3번째 조건 "커버리지 ≥ 80% (서비스/리포지토리)" 와 직접 대응한다.
