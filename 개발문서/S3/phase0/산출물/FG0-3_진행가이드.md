- `[tool.pytest.ini_options] addopts` 의 `--cov-fail-under=80` 을 `0` 으로 변경 (per-step 재정의 허용)

## 3. 후속 세션 인수 시 절차

### 3.1 현재 커버리지 실측

```bash
cd backend

# (옵션 A) 로컬 testcontainers 경유
pytest tests/ --cov=app --cov-append --cov-report= --cov-fail-under=0

# (옵션 B) CI artifact 다운로드 — backend/.gitlab-ci.yml 최신 런에서
#   `backend-integration-results` artifact 를 받아 coverage.xml 사용

# 요약 확인
python scripts/coverage_baseline.py --xml coverage.xml

# 임계값 검증
python scripts/coverage_baseline.py --xml coverage.xml --check
```

### 3.2 우선순위 모듈 선택

`FG0-3_베이스라인.md` §4 를 참고하되, 현재 최신 데이터로 재생성한 뒤 본인의 숫자를 기준으로:

- **gap_lines 가 가장 큰 모듈** + **rate 가 극도로 낮은 모듈** 순서
- `services.*` > `repositories.*` > 기타

### 3.3 테스트 작성 패턴 (반복 가능한 템플릿)

#### 3.3.1 서비스 유닛 테스트

```python
# backend/tests/unit/services/test_<service>_unit.py
from unittest.mock import MagicMock
import pytest

from app.services.<service> import <service_instance_or_class>


class Test<ServiceName>:
    """FG 0-3 (2026-04-23): <모듈 목적> 유닛 커버."""

    def test_<분기_이름>(self):
        # arrange
        conn = MagicMock()
        conn.cursor.return_value.__enter__.return_value.fetchone.return_value = ...

        # act
        result = <service>.<method>(conn, <args>)

        # assert
        assert result.<field> == <expected>
```

#### 3.3.2 리포지토리 유닛 테스트 (실 DB 필요)

```python
# backend/tests/integration/test_<repo>_integration.py
import pytest

from app.repositories.<repo> import <repo_instance>

pytestmark = pytest.mark.integration


def test_<crud_분기>(db_conn):
    # arrange
    with db_conn.cursor() as cur:
        cur.execute("INSERT INTO <table> (...) VALUES (...)")
    db_conn.commit()

    # act
    row = <repo>.<method>(db_conn, <arg>)

    # assert
    assert row is not None
    assert row.<field> == <expected>
```

### 3.4 테스트 리뷰 체크리스트 (PR 머지 전)

- [ ] 의미 있는 `assert` 가 최소 1개 (단순 `None` 체크 금지)
- [ ] 에러 분기도 커버 — `with pytest.raises(<ExpectedError>):` 반드시 포함
- [ ] Flaky 요소 없음 — 시간/random/외부 네트워크 없음 (있으면 `freezegun` / seed 고정)
- [ ] Mock 범위 최소화 — 실 DB 로 잡을 수 있는 버그는 통합 테스트로
- [ ] 테스트 이름이 "무엇을 검증하는지" 직관적 (`test_rejects_when_<조건>`)

### 3.5 임계값 통과 확인

## 4. 자주 묻는 문제

### 4.1 CI 에서 `coverage.xml not found` 실패

`pytest --cov-report=` (빈 값) 로 실행 후 `coverage xml` 을 명시적으로 실행해야 XML 이 생성된다. `--cov-report=xml:coverage.xml` 을 pytest 옵션에 직접 넣는 것과는 다른 경로 — append 모드에서는 후자가 동작하지 않아 현재 워크플로는 전자를 사용.

### 4.2 `coverage combine` 오류

`parallel = false` 상태에서는 combine 이 필요 없다. 만약 pytest-xdist 를 도입한다면 pyproject 의 `parallel = true` 로 바꾸고 `coverage combine` 을 XML 생성 전에 추가.

### 4.3 로컬에서 `--check` 통과는 하는데 CI 에서 실패

CI 는 실 DB 통합 테스트를 합산하므로 일부 모듈이 **추가로** 커버된다. 로컬 단위 테스트만 돌렸을 때의 숫자와 차이 날 수 있음. CI artifact 의 `coverage.xml` 을 로컬에서 파싱해 비교 권장.

### 4.4 `services.retrieval` 같은 하위 패키지가 0% 인데 services 전체 rate 가 크다

`sum_prefix("services")` 는 `services`, `services.llm`, `services.retrieval`, `services.extraction` 모두 합산. 하위 패키지가 0% 라도 `services` 자체가 커버되면 전체 합은 올라간다. **문제는 gap_lines** — 숫자가 높은 하위 패키지부터 개선.

### 4.5 `omit` 에 추가했는데도 커버리지에 계속 잡힘

`[tool.coverage.run] omit` 은 "측정에서 제외" 지 "리포트에서만 숨김" 이 아니다. 이미 임포트되어 바이트코드가 실행됐으면 omit 전에 측정됐을 수 있음 — 이 경우 `[tool.coverage.report] omit` 과 함께 지정. pyproject 의 report 섹션에도 같은 패턴 넣는 것이 안전.
