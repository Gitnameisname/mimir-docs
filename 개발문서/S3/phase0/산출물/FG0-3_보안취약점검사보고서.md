## 1. 위협 모델

본 FG 는 **CI 에서 커버리지를 측정·검증** 하는 보조 인프라다. 애플리케이션 코드 자체를 수정하지 않으며, 외부에서 입력을 받지 않는다. 공격 표면은 **CI 러너 내부** 에 국한된다.

| 경계 | 주체 | 위협 | 범위 |
|------|------|------|------|
| `coverage_baseline.py` | CI 잡 | 악의적 coverage.xml 로 임의 코드 실행 / 경로 조작 | §2.1 |
| `pyproject.toml` | 로컬/CI | omit 패턴 남용으로 실제 미커버 모듈 은폐 | §2.2 |
| CI 워크플로 | GH Actions 러너 | 시크릿 재노출 / artifact 민감 정보 포함 | §2.3 |

## 2. 체크리스트

### 2.1 `coverage_baseline.py` 정적 안전성 (OWASP A03, A08)

| 항목 | 결과 | 상세 |
|------|------|------|
| XML 파서 choice | ✅ | `xml.etree.ElementTree` stdlib — `defusedxml` 미사용이나 coverage.xml 은 **CI 내부에서 생성된 파일** 로 신뢰 경계 내부. 외부 네트워크에서 받은 XML 이 아님 |
| XXE / billion-laughs 가능성 | ✅ | ET 는 기본값으로 외부 엔티티 해석 비활성 — stdlib 동작상 안전. 다만 `etree.parse` 로 로컬 DTD 해석 가능성은 남음. CI 신뢰 경계 내 입력이므로 수용 |
| 경로 traversal | ✅ | `args.xml: Path` 는 사용자 지정. `Path.exists()` 후 read only. 쓰기 경로는 stdout 만 |
| 임의 코드 실행 | ✅ | `eval` / `exec` / `subprocess` 사용 없음 |
| 의존성 | ✅ | stdlib only — `argparse`, `sys`, `xml.etree`, `dataclasses`, `pathlib`, `typing` |
| Int overflow / 분모 0 | ✅ | `rate = (covered / valid * 100) if valid else 0.0` 로 zero-division 방어 |

**권고**: 장기적으로 `defusedxml.ElementTree` 로 전환 — 현 시점 수용.

### 2.2 pyproject.toml 변경분 (OWASP A04, A05)

| 항목 | 결과 | 상세 |
|------|------|------|
| omit 패턴 남용 가능성 | ⚠️ 제한적 | `scripts/*` 가 넓은 패턴이지만 `scripts/` 디렉터리 자체는 CI 도구 전용 (coverage_baseline / rag_smoke 등). 프로덕션 경로 아님 — 수용 |
| fail_under 하향 (80 → 75) | ✅ | 작업지시서 §4.5 요구사항. 전체 75 + per-path 80 조합이 task0-3 §5 완료 기준과 동등 |
| exclude_lines `logger.debug` | ⚠️ 제한적 | debug 로깅은 방어적 경로라 coverage 제외 타당. 다만 debug 라인에 실제 비즈니스 로직이 섞이면 은폐됨 — PR 리뷰 시 debug 호출 안에 `if` 분기 금지 원칙 |
| exclude_lines `except ImportError` | ✅ | optional 의존성 fallback 경로. 표준 패턴 |

### 2.3 CI 워크플로 변경분 (OWASP A05, A07)

| 항목 | 결과 | 상세 |
|------|------|------|
| 시크릿 직접 참조 | ✅ | 신규 추가 스텝에 secret 변수 직접 참조 없음. 기존 env 블록만 활용 |
| `coverage report` stdout 이 secret 출력 | ✅ | coverage 출력은 파일명/라인번호/% 만 — env var 값 포함되지 않음 |
| 임계값 gate 가 `if: always()` 인지 | ⚠️ | Coverage gate 스텝은 `if: always()` 로 설정 — 이전 스텝 실패 시에도 실행됨. 기능상 의도된 설계이나, integration 실패 시에도 coverage gate 메시지가 나와 혼란 가능. **권고**: 검수보고서 §FAQ 에 명시 (진행가이드 §4 에 포함) |
| artifact 업로드 권한 | ✅ | `actions/upload-artifact@v4` — GITHUB_TOKEN 의 기본 `contents: read` 만 필요 |
| artifact 에 민감 정보 포함 여부 | §2.4 에서 별도 | |

### 2.4 artifact 노출 (OWASP A01)

| artifact | 내용 | 위험 | 판정 |
|----------|------|------|------|
| `test-results/*.xml` | pytest-junit | 테스트 이름 + 실행 시간 | 정상 (공개 가능) |
| `coverage.xml` | Cobertura XML | 파일명 + 라인번호 + 커버리지 % | 정상 |
| `htmlcov/` | HTML 리포트 | **파일 내용의 실제 소스 코드 단편** 포함 | ⚠️ |
| `test-results/FG0-3_베이스라인_*.md` | 패키지 요약 | 패키지/파일명 | 정상 |

**`htmlcov/` 고려사항**:

- HTML 리포트는 코드 각 줄을 표시하고 "covered/not covered" 색상을 입힌다. 즉, artifact 를 받는 사람은 **소스 코드 전체를 열람할 수 있다**.
- GitHub Actions artifact 는 조직 구성원(private repo 기준)에게만 보인다. public repo 인 경우 artifact 도 공개.
- Mimir 은 폐쇄망 대응 플랫폼 (S2 ⑦) 이라 소스 일부 노출이 영업 비밀 수준 문제는 아님 — **수용**. 그러나 민감 상수/하드코드 토큰이 들어가면 그대로 노출됨 → 정적 스캔(Bandit/secret-scanner) 병행 권고.

**권고** (잔존): CI 에서 artifact 업로드 전에 `grep -r "ghp_\|sk-\|xoxb-"` 등 토큰 패턴 체크 — 별도 보안 트랙.
