# FG 0-2 운영가이드 — `EMBEDDING_DIM` 차원 변경 절차

| 항목 | 값 |
|------|----|
| 작성일 | 2026-04-23 |
| 대상 | DB 운영자 / DevOps / 백엔드 오너 |
| 관련 | `backend/app/db/migrations/versions/20260423_1500_s3_p0_embedding_dim_check.py`, `/api/v1/system/health` |

본 가이드는 Mimir 의 임베딩 차원(`EMBEDDING_DIM`)을 변경해야 할 때 **데이터 정합성을 유지하며 배포하는 절차** 를 정리한다. 차원 변경은 임베딩 벡터 스토어 전체 재구축을 수반하므로, 모든 단계를 **동일 배포 창** 에 수행해야 한다.

---

## 1. 배경

### 1.1 BUG-04 요약

S2 종결 시점에 `EMBEDDING_DIM=768` 설정과 `document_chunks.embedding VECTOR(1536)` 컬럼 차원이 불일치하는 상태가 관찰되었다. 이 상태에서는:

- 임베딩 생성은 정상 완료 (768차원 벡터 반환)
- INSERT 시점에 PostgreSQL 이 `vector(1536)` 컬럼에 길이 768 벡터를 거부 → `vectorization_service` 예외
- `publish` 는 이미 성공 반환 → 사용자에게는 "성공" 인데 RAG 는 찾을 수 없는 **조용한 실패** 상태

### 1.2 FG 0-2 이후 가드

- **Alembic revision `s3_p0_embedding_dim_check`** 가 매 `alembic upgrade head` 시점에 config↔DB 차원 일치를 검증한다 (불일치 시 revision 실패).
- **`/api/v1/system/health` 서브체크** 가 운영 중에도 불일치를 감지하면 `healthy: false` + `degraded: true` 로 알려준다.

### 1.3 현재 아키텍처 주의

**임베딩의 1차 저장소는 Milvus** (`app/db/milvus.py`) 다. PostgreSQL `document_chunks.embedding` 컬럼은 **현재 존재하지 않으며**, 존재하지 않는 상태는 정상 (pass) 으로 처리된다. 본 가드는 "향후 pgvector 경로를 활성화할 경우의 재발 방지" + "Milvus collection 차원 일치(best-effort)" 를 함께 다룬다.

## 3. 차원 변경 7단계 절차

> 본 절차는 `EMBEDDING_DIM` 을 기존 값 X 에서 새 값 Y 로 변경하는 표준 플로우다.

### Step 1. `.env` 수정 (staging 먼저)

```bash
# backend/.env
- EMBEDDING_DIM=768
+ EMBEDDING_DIM=1024
- EMBEDDING_DIMENSIONS=768
+ EMBEDDING_DIMENSIONS=1024
```

> 아직 서비스 재기동 금지. pg column / Milvus collection 이 기존 값이라 config 만 바꾸면 불일치가 곧바로 healthcheck 에 노출된다.

### Step 2. pgvector 컬럼 alter migration 작성 (pgvector 경로 사용 중일 때만)

`backend/app/db/migrations/versions/YYYYMMDD_HHMM_embedding_dim_alter.py` 신규 revision 생성:

```python
revision = "embedding_dim_alter_YYYYMMDD"
down_revision = "s3_p0_embedding_dim_check"

def upgrade() -> None:
    op.execute("ALTER TABLE document_chunks DROP COLUMN IF EXISTS embedding")
    op.execute(f"ALTER TABLE document_chunks ADD COLUMN embedding vector({NEW_DIM})")
    # HNSW 인덱스도 함께 재생성
    op.execute("CREATE INDEX IF NOT EXISTS idx_chunks_embedding_hnsw "
               "ON document_chunks USING hnsw (embedding vector_cosine_ops)")
```

> `DROP + ADD` 는 모든 기존 벡터를 삭제한다. 재벡터화 배치 전까지 RAG 의미 검색 결과는 빈 배열.

### Step 3. 검증 revision 이 통과함을 로컬 확인

```bash
cd backend
EMBEDDING_DIM=1024 alembic upgrade head
# 성공 메시지 예:
# INFO [alembic.runtime.migration.s3_p0_embedding_dim_check]
#     [FG0-2] EMBEDDING_DIM 검증 pass — config=1024 / db=1024 (type=vector(1024))
```

실패 시 로그 메시지에 `config` / `db` 값이 노출된다. 불일치 원인을 확인 후 Step 2 revision 수정.

### Step 4. Milvus collection 재생성 (Milvus 사용 시)

```bash
cd backend
python3 -c "
from app.db.milvus import get_milvus
client = get_milvus()
client.drop_collection('document_chunks')  # 기존 삭제
# 새 collection 은 서비스 기동 시 settings.embedding_dim 으로 자동 생성됨
"
```

또는 수동 재생성:

```python
from pymilvus import CollectionSchema, FieldSchema, DataType
schema = CollectionSchema([
    FieldSchema("chunk_id", DataType.VARCHAR, is_primary=True, max_length=36),
    FieldSchema("embedding", DataType.FLOAT_VECTOR, dim=NEW_DIM),
])
client.create_collection(schema)
```

### Step 5. 배포 창에서 변경 적용

```bash
# 배포 순서:
# 5-1. 앱 서비스 정지 (또는 blue-green 비활성화)
# 5-2. alembic upgrade head (Step 2 의 alter revision 적용)
# 5-3. .env 업데이트 배포
# 5-4. 앱 서비스 기동
```

### Step 6. `/system/health` 에서 match: true 확인

```bash
curl -s https://prod.example.com/api/v1/system/health | jq '.data.embedding_dim'
# 기대:
# {
#   "config": 1024,
#   "db": 1024 (pgvector 사용 시) 또는 null (Milvus only),
#   "match": true (또는 null),
#   "column_present": true (또는 false),
#   "milvus": { "dim": 1024, "match": true, "reason": "" }
# }
```

`match: false` 가 보이면 **즉시 롤백**. `.env` 와 migration 적용 순서가 뒤바뀐 상태일 가능성이 높다.

### Step 7. 재벡터화 실행

```bash
# published 문서 전체 재벡터화
curl -X POST https://prod.example.com/api/v1/vectorization/documents/all \
     -H "X-Actor-Role: SUPER_ADMIN" \
     -H "X-Actor-Id: migration-ops-001"

# 또는 CLI (존재 시)
cd backend
python3 scripts/reindex.py --type=all
```

> 재벡터화 중 RAG 는 degrade 동작한다 (FTS 폴백). 완료 시점까지 품질 저하 공지 필요.

## 4. 롤백 절차

### 4.1 Step 3 실패 (revision 실패)

`alembic upgrade head` 가 BUG-04 감지 메시지와 함께 실패. DB 는 원상태. `.env` 만 원복하면 된다.

```bash
# .env 되돌리기
EMBEDDING_DIM=768  # 원래 값
```

### 4.2 Step 5 이후 실패

- pgvector 컬럼을 원래 차원으로 돌리는 alter revision 필요 (downgrade 지원 revision 이면 `alembic downgrade -1`).
- Milvus collection 은 백업에서 복구 (Step 4 에서 만든 새 collection 이 비어 있으므로, 원래 데이터를 가진 백업 스냅샷 필요).
- 앱 서비스는 원래 `.env` 로 재기동.

**중요**: Step 4 에서 `drop_collection` 직전 Milvus collection 을 백업(`backup`/`restore` API 또는 데이터 덤프)해 둘 것.

## 5. 자주 묻는 문제

### 5.1 `match: false` 가 떴는데 재기동으로 해결하려 함

재기동은 config 재로드일 뿐 DB 차원을 바꾸지 않는다. 반드시 Step 2 의 alter migration 이 적용돼야 한다.

### 5.2 Milvus 에 `document_chunks` collection 이 여러 개 보임

pymilvus 버전 차이 또는 수동 생성 흔적. `list_collections()` 로 확인 후 불필요한 것 정리.

### 5.3 CI 에서 `alembic upgrade head` 가 불일치로 실패

CI 의 `.env` (services secret) 와 실제 DB schema 가 일시적으로 어긋남. Phase 0 FG 0-1 의 `backend-integration.yml` 에서는 EMBEDDING_DIM 을 명시적으로 `768` 로 고정했으므로 변경 시 동반 업데이트 필요:

```yaml
env:
  EMBEDDING_DIM: "1024"        # ← 새 차원으로
  EMBEDDING_DIMENSIONS: "1024" # ← 동일 값으로
```

### 5.4 임베딩 provider 를 함께 교체 중

provider 교체(OpenAI → 사내 SBERT 등)는 차원도 달라지는 경우가 많다. **provider 변경 → 차원 변경 → 재벡터화** 순서로 한 번에 수행. 중간 상태에서는 RAG 가 무응답에 가깝다.
