## 1. 위협 모델

본 FG 는 **기존 Admin-only 재벡터화 API 의 권한을 문서 작성자까지 확장** 한다. 이는 편의성 이득이 크지만 권한 확장 자체가 보안 영향을 동반한다. 또한 **상태 API 가 새로 공개** 되고 **MCP Tool 이 외부 에이전트에 노출** 된다.

| 경계 | 주체 | 위협 | 범위 |
|------|------|------|------|
| `GET /status` | 인증 사용자 | 비승인 정보 노출 (`last_error` 스택) | §2.1 |
| `POST /documents/{id}` 권한 확장 | Admin + 작성자 | 작성자가 반복 재벡터화로 자원 소모 | §2.2 |
| Valkey 쿨다운 | 내부 | 우회 (in-memory 폴백의 프로세스 로컬 한계) | §2.3 |
| MCP Tool `mimir.vectorization.status` | 외부 에이전트 | IDOR / Scope ACL 우회 | §2.4 |
| 감사 로그 `vectorization.failed` | audit_events | PII · 민감 error 메시지 저장 | §2.5 |
| Frontend `ConfirmDialog` | 브라우저 | CSRF / 재전송 | §2.6 |

## 2. 체크리스트

### 2.1 상태 API 정보 노출 (OWASP A01, A05)

| 항목 | 결과 | 상세 |
|------|------|------|
| 인증 요구 | ✅ | `authorization_service.authorize(action="document.read", ...)` + `is_authenticated` 체크 |
| Scope Profile ACL | ✅ | `document.read` 정책을 기존 authorize 경로에 위임 — Scope Profile 필터링 자동 적용 |
| `last_error` 스택 노출 수준 | ⚠️ 제한적 | `reason` 을 300자 truncate 후 반환. 민감 내부 호스트(예: milvus.internal:19530) 가 문자열로 포함될 가능성. **권고**: Admin 전용 필드로 분리 (`last_error_detail`) — 본 FG 에선 수용, Phase 1+ 개선 |
| `indexed_version_id` 노출 | ✅ | UUID 이라 추가 위험 낮음 |
| `can_reindex` 판정이 서버 신뢰 | ✅ | 프런트는 표시에만 사용, 실제 권한 검증은 POST 호출 시 서버에서 재확인 (`_require_admin_or_creator`) |

### 2.2 권한 확장 (OWASP A01, A07, CWE-285)

| 항목 | 결과 | 상세 |
|------|------|------|
| `created_by == actor.user_id` 엄격 비교 | ✅ | 문자열 직접 비교 (`str(actor_user_id) == str(document_created_by)`) |
| Admin 역할 세트 고정 | ✅ | `_ADMIN_ROLES = frozenset({"ADMIN", "SUPER_ADMIN", "ORG_ADMIN"})` — 외부 입력 영향 없음 |
| 미인증 → 401 | ✅ | `actor.is_authenticated` 선행 체크 |
| 문서 부재 → 404 | ✅ | |
| 권한 없음 → 403 | ✅ | |
| 에이전트(actor_type=agent) 가 재벡터화 실행 가능? | ✅ (방어) | MCP Tool 에 재벡터화 미노출. REST API 는 에이전트도 OAuth2 client_credentials 경유로 호출 가능하나, actor_role 이 `AGENT` 여야 하고 documents.created_by 는 user_id 형식이므로 매칭 실패 → 거부. Admin 에이전트만 통과 (의도된 동작) |

### 2.3 쿨다운 우회 (OWASP A04)

| 항목 | 결과 | 상세 |
|------|------|------|
| Valkey 기반 원자성 | ✅ | `SET key 1 NX EX ttl` — atomic |
| in-memory 폴백의 한계 | ⚠️ | 다중 프로세스/컨테이너에서는 각 프로세스 로컬 — 실제로 Valkey 를 쓰지 않는다면 **프로세스 수 × N 회** 까지 쿨다운 뚫림. 운영 환경 권고: Valkey 필수. 코드는 경고 로그 1회 + 계속 동작 (폐쇄망 호환 유지) |
| actor_id 당 별도 키 | ✅ | 한 사용자가 다른 문서에 영향 주지 않음 (`vec_reindex:{doc}:{actor}`) |
| 쿨다운 키 흔적이 보안 이슈? | ✅ | Valkey key 에는 문서 UUID + actor_id 만. 민감 데이터 없음. TTL 10초 |
| 쿨다운 Bypass (예: 재로그인으로 actor_id 변경) | ⚠️ | actor_id 재발급은 OAuth refresh 로 불가 (동일 subject). 별도 계정 생성 시 다른 actor_id 지만 Admin 이 아니면 작성자도 아니므로 거부 |

### 2.4 MCP Tool 보안 (OWASP A01, LLM 시리즈)

| 항목 | 결과 | 상세 |
|------|------|------|
| 읽기 전용 (재벡터화 미노출) | ✅ | `tools.py::tool_vectorization_status` 만 추가. dispatcher 에 write 분기 없음 |
| 에이전트 킬스위치 체크 | ✅ | `_check_agent_write_blocked(actor)` 재사용 (기존 search/fetch 와 동일 수준) |
| IDOR (임의 document_id 조회) | ⚠️ 제한적 | UUID 형식 검증 + `scope_profile_required=True` 인증. 그러나 Scope Profile 에 의한 document-level ACL 은 `get_vectorization_status` 내부에서 추가 적용되지 **않음**. 에이전트가 UUID 를 알면 any 문서 상태 조회 가능. **권고 (잔존)**: 라우트 레벨 authorize 를 MCP Tool 경로에도 적용 — Phase 1+ |
| 감사 로그 | ✅ | `_emit_audit("mcp.tool.vectorization_status", ...)` actor_type=agent 기록 |
| Tool 응답의 `last_error` 노출 | ⚠️ | §2.1 과 동일 위험. 에이전트가 내부 호스트 정보 획득 가능 — 권고 Phase 1+ |

### 2.5 감사 로그 (OWASP A09, LLM06)

| 항목 | 결과 | 상세 |
|------|------|------|
| `vectorization.failed` 이벤트에 actor_type="system" | ✅ | `_trigger_vectorization_async` 의 자동 트리거는 system 주체. emitter 의 Literal 검증 통과 |
| `reason` 필드 400자 truncate | ✅ | `str(exc)[:400]` — 과도한 stack 방지 |
| `vectorization.reindex_requested` 이벤트 actor_type | ✅ | 호출자 actor_type 을 사용자/에이전트로 정규화 |
| audit_events 에 PII 저장 | ✅ | `document_id` UUID 만. 사용자 식별자는 `actor_user_id`(기존 컬럼) — 정책 내 |

### 2.6 프런트엔드 보안 (OWASP A01, A05)

| 항목 | 결과 | 상세 |
|------|------|------|
| CSRF 방어 | ✅ | 기존 `api.post` 가 Bearer Token 전달 (SameSite 쿠키 아닌 헤더 기반) — CSRF 면역 |
| 확인 모달 (재벡터화 실행 전) | ✅ | `ConfirmDialog` 로 2단계 요구 — 실수 클릭 방지 |
| 에러 메시지 XSS | ✅ | React 가 기본 escape. `<pre>` 안 `last_error` 도 text content 로 렌더 — 주입 불가 |
| 폴링으로 인한 스파이크 | ✅ | 2초 간격 + 60초 상한. 이후에 수동 새로고침만 |
| 버튼 데드락 (inFlight + cooldown 이 동시에 true) | ✅ | 두 조건 OR 로 disabled — 확실히 막힘. 다만 UI 에서 구분 표시 |
