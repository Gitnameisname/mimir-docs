## 3. 구현 품질 검증

### 3.1 정적 검증

| 대상 | 방법 | 결과 |
|------|------|------|
| `judge_prompts.py` / `judge.py` / `rag_quality.py` | `ast.parse` | ✅ 모두 OK |

| `judge.py` standalone | `python3 judge.py` (DummyJudge 빌드) | ✅ `judge ready` + `dummy sanity: score=0.85` |
| `rag_quality.py --dry-run` | 실행 | ✅ 51건 처리 + 리포트 저장 + GATE 통과(mock) |

### 3.2 스크립트 기능 동작 (샌드박스 실행)

다음을 실제 수행했다:

```
$ python3 rag_quality.py --dry-run --out /tmp/fg04_dryrun.json
→ judge: provider=dummy model=dummy-v1
→ [dry-run] RAG 호출 생략 — placeholder QueryResult 생성
→ 판정 단계 시작 — 51 쿼리
→ report saved
→ [GATE] 목표 달성 — Faithfulness=0.850 / Citation=1.0
```

리포트 JSON 의 필수 키 (`run_at` / `judge.provider` / `summary.faithfulness_macro` / `summary.citation_present_rate` / `by_category` / `results[]`) 모두 생성 확인.

### 3.3 어댑터 설계

| 어댑터 | Base URL 기본 | 타임아웃 | Response format | Retry |
|--------|--------------|---------|----------------|-------|
| OpenAI | api.openai.com | 30s | `response_format: {"type":"json_object"}` | 1회 |
| vLLM | localhost:8000 | 30s | 동일 (OpenAI 호환) | 1회 |
| Ollama | localhost:11434 | 60s | `format: "json"` (native) | 1회 |
| Dummy | — | — | 하드코드 | — |

공통: `_extract_json_blob()` 에서 fenced ```json``` 블록 / plain JSON / 중괄호 범위 3단계 파싱.

### 3.4 적합성 — adversarial·domain-shift 검증

adversarial 쿼리(Q47~Q51) 설계 의도 대비:
- Q47: 법무실 author 중복(G1/G2/G3) 에서 "7년" 답은 G1 에만 있음 → G1 우선 retrieval 요구
- Q48: G3 의 `effective_date: 2025-06-01` + `next_revision_planned: 2026-Q2` 로 시기 모호성 유도
- Q49: G4 의 단편 회람 혼합 구조에서 "20만원" 을 정확히 찾기
- Q50: G1 의 긴 본문(10 섹션) 에서 §4 GDPR 조항 정확히 추출
- Q51: G2(법무실) vs C4(SRE) 의 "분기" 주제 혼동에서 "17명" 정답 선택

domain-shift 쿼리(Q42~Q46) 는 완전히 새 도메인(해외사업/브랜딩)에 대한 retrieval 안정성 확인.

## 5. 결론 — "선언적 완료 vs 실질적 완료"

Phase 0 산출물 규약 §3.2 기준 재평가:

| 기준 | 충족 여부 | 근거 |
|------|----------|------|
| 골든셋 ≥ 50건 | ✅ | 51건 |
| 판정 어댑터 2종 이상 (로컬 ≥1, 외부 ≥1) | ✅ | Ollama + OpenAI + vLLM (+ Dummy) |
| **실측** Faithfulness ≥ 0.80 (폐쇄망) | ⏸ **후속 인수** | mock 기준 0.850. 실 Ollama 연결 실측 필요 |
| **실측** Citation-present ≥ 0.90 | ⏸ **후속 인수** | mock 기준 1.0. 실 RAG 응답 필요 |
| AI 품질 평가 보고서 | ✅ (템플릿) | 실행 절차 + 갱신 가이드 포함. 실 수치는 운영자 갱신 |
| 검수 · 보안 보고서 | ✅ | 본 문서 + `FG0-4_보안취약점검사보고서.md` |

본 FG 는 **"AI 품질 측정 인프라 선언적 완료"** 상태다. **실질적 완료**는:

1. 운영자가 `FG0-4_AI품질평가보고서.md` §6.1 절차로 Mimir 서버 기동
2. `FG0-4_AI품질평가보고서.md` §6.2 절차로 Ollama + llama3.1:8b 준비
3. `FG0-4_AI품질평가보고서.md` §6.3 절차로 폐쇄망 + 외부 2회 실측
4. `FG0-4_AI품질평가보고서.md` §3~§5 를 실 수치로 갱신
5. Faithfulness ≥ 0.80 + Citation-present ≥ 0.90 확인

의 5단계로 전환된다. Phase 0 게이트 중 "FG 0-4 AI 품질 평가 보고서 제출 (Faithfulness ≥ 0.80, Citation-present ≥ 0.90 실측 포함)" 에 직접 대응.
