## 1. 위협 모델

본 FG 는 **CI 파이프라인에서 실 PostgreSQL(+pgvector) + Valkey 컨테이너를 기동하고 pytest 통합 테스트를 실행**하는 인프라다. 배포되는 코드는 아니지만 **CI 러너에서 시크릿과 의존성 이미지를 다룬다**는 점에서 공급망(supply chain) / 시크릿 / 테스트 격리 관점의 검토가 필요하다.

| 경계 | 주체 | 위협 | 범위 |
|------|------|------|------|
| GH Actions 러너 | Anthropic-owned ubuntu-22.04 | 러너 탈취로 시크릿 유출 | 외부 컨트롤 (운영자 secret 회전 정책) |
| Docker 이미지 | `pgvector/pgvector:pg16`, `valkey/valkey:8` | 이미지 공급망 공격 (malicious push) | 태그 pin + 주기적 스캔 |
| 테스트 코드 | backend/tests/integration/ | 악성 테스트가 실 DB에 prod 데이터 주입 | 코드 리뷰 + 스키마 격리 |
| 시크릿 | `CI_POSTGRES_PASSWORD` 등 | 로그/artifact 유출 | secrets context + 본 보고서 §2.1 체크 |
| 환경 변수 | `JWT_SECRET`, `INTERNAL_SERVICE_SECRET` | 약한 기본값 사용 시 프로덕션 오용 | `-not-for-production` 라벨 + 길이 검증 |

## 2. 체크리스트

### 2.1 시크릿 / 자격증명 (OWASP A02, A07, CWE-798)

| 항목 | 결과 | 상세 |
|------|------|------|
| 워크플로 YAML 내 평문 시크릿 금지 | ✅ | 모든 민감값이 `${{ secrets.* }}` 참조. `secrets` 미설정 시 폴백값은 `ci-only-*-not-for-production-*` 문자열로 운영 무관 표기. |
| 폴백값 운영 혼동 방지 | ✅ | 이름에 `not-for-production` 문자열을 포함해 grep 으로 즉시 식별 가능. |
| JWT_SECRET 길이 | ✅ | CI 폴백 `ci-only-jwt-secret-not-for-production-xxxxxxxx` = 49자 (≥32 기준 충족). 실 secret 은 운영자가 주입. |
| POSTGRES_PASSWORD 이 stdout/log 로 새는 경로 | ✅ | psql step 에서 `PGPASSWORD` env 로만 전달. `pg_isready` / `psql -c` 명령 인자에 노출 없음. |
| 테스트 코드가 시크릿을 artifact 에 쓰는 경로 | ✅ | `test-results/*.xml` 과 `coverage-integration.xml` 은 pytest-junit / coverage 포맷 — 환경 변수를 직접 기록하지 않음. |
| testcontainers 자동 생성 암호 | ✅ | `secrets.token_urlsafe(16)` 로 128-bit 랜덤. 컨테이너 수명 동안만 유효. |
| `conftest.py` 내 하드코드된 기본 암호 | ✅ | 없음 (모든 민감값은 env 또는 testcontainers 동적 생성). 테스트용 기본값 `mimir_test/*****` 는 컨테이너 내부 전용. |
| `tests/integration/` 가 `.env.production` 등 로드 시도 | ✅ | 없음 (conftest.py 가 `os.environ.setdefault("ENVIRONMENT", "test")` 로 test 환경을 강제). |

**권고**: 운영자는 3종 secret 을 등록할 때 최소 32자 난수(`openssl rand -base64 48 | head -c 32`)로 생성. `CI_POSTGRES_PASSWORD` 는 영숫자만 — URL-safe 문자 제약은 없으나 일부 특수문자(`$`, `@`, `:`, `/`)는 psql 접속 URL 의 quoting 복잡성을 유발하므로 회피 권장.

### 2.2 공급망 / Docker 이미지 (OWASP A06, CWE-1357)

| 항목 | 결과 | 상세 |
|------|------|------|
| 이미지 태그 pin | ✅ | `pgvector/pgvector:pg16`, `valkey/valkey:8`, `postgres:16` 모두 태그 고정. `latest` 미사용. |
| 이미지 digest pin | ⏸ (잔존) | 보안 최상위 기준(`@sha256:...`)은 적용 안됨. pgvector 이미지는 자체 릴리스 주기상 빈번히 재빌드되므로 digest pin 시 유지보수 부담 증가. **권고**: Dependabot + Trivy 정기 스캔으로 대체. |
| testcontainers 이미지 override env 제공 | ✅ | `INTEGRATION_PG_IMAGE` / `INTEGRATION_VALKEY_IMAGE` — 사내 레지스트리 경로로 치환 가능 (폐쇄망 S2 ⑦ 호환). |
| 불필요 네트워크 노출 | ✅ | CI 러너 환경에서 services 포트는 `127.0.0.1:5432/6379` 로만 바인딩되며 외부 노출 없음. |
| Dockerfile 없음 (본 FG 는 이미지 소비만) | ✅ | 자체 이미지 빌드 단계 없음 — 공격 표면 최소. |

**권고**:

1. `.github/dependabot.yml` 에 `github-actions` 및 `docker` ecosystem 를 추가해 이미지 태그 자동 업데이트 알림 수령.
2. 별도 보안 트랙으로 `aquasec/trivy-action` 을 워크플로에 추가 — 본 FG 에서는 시간 제약상 미포함.

### 2.3 파일 시스템 / 경로 안전성 (OWASP A01, A08)

| 항목 | 결과 | 상세 |
|------|------|------|
| 테스트가 absolute path 로 임의 파일 생성 | ✅ | conftest.py 내 `Path` 사용은 모두 `_BACKEND_ROOT` / `_REPO_ROOT` 상대 경로. 파일 쓰기 없음. |
| pytest 가 artifact 를 `backend/test-results/` 에 쓸 때 경로 traversal 가능성 | ✅ | junit-xml / coverage-xml 모두 pytest-known 고정 경로. |
| alembic.ini 가 실행 중 DB URL 노출 | ⚠️ 제한적 | env.py 가 `sqlalchemy.url` 을 placeholder 로 초기화 후 런타임 주입. 주입된 URL 은 `engine_from_config` 로만 전달되며 로그 레벨 `WARN` 이상에서만 표시. |

### 2.4 CI 파이프라인 제어 흐름 (OWASP A04, A05)

| 항목 | 결과 | 상세 |
|------|------|------|
| `pull_request_target` 같은 위험 트리거 사용 여부 | ✅ | `on: [push, pull_request]` 만 사용. fork PR 이 secret 에 접근하지 않음 (GH 기본 동작). |
| 악성 PR 이 테스트 코드를 수정해 secret 을 stdout 으로 덤프 | ⚠️ (일반 CI 공통 리스크) | CODEOWNERS / branch protection 으로 완화되어야 함. **권고**: `.github/CODEOWNERS` 에 `.github/workflows/*` 와 `backend/tests/integration/conftest.py` 를 명시적으로 추가. |
| `concurrency` 그룹 설정 | ✅ | `concurrency: group: backend-integration-${{ github.ref }}` + `cancel-in-progress: true` — 중복 실행 시 앞선 잡 중단으로 시크릿 반복 노출 최소화. |
| `permissions:` 최소 권한 | ✅ | `permissions: contents: read` — GITHUB_TOKEN 에 쓰기 권한 없음. |
| step-level secret 재노출 | ✅ | workflow env 에서만 사용. step 개별 `env:` override 없음. |
| 긴 log 에서 secret masking 우회 | ✅ | `echo "$CI_POSTGRES_PASSWORD"` 등 값을 직접 에코하는 명령 없음. `PGPASSWORD` env 도 psql 프로세스 환경에만 주입. |

### 2.5 테스트 격리 / 부작용 (OWASP A01, A04)

| 항목 | 결과 | 상세 |
|------|------|------|
| 운영 DB 에 연결 가능성 | ✅ | CI 는 `POSTGRES_HOST=localhost` + 컨테이너 전용. 로컬은 testcontainers 가 새 컨테이너에 할당. 실수로 운영 DB 를 대상으로 실행되려면 `USE_EXISTING_DB=1` + 운영 자격증명을 동시 주입해야 하므로 2단계 방어 존재. |
| 테스트가 운영 데이터 reference | ✅ | 모든 테스트가 UUID 기반 신규 객체 생성. 기존 레코드 조회는 `_apply_schema` 가 만든 빈 테이블만 대상. |
| IT-04 chunk 테스트 정리 | ✅ | `try/finally` 블록으로 테스트 종료 시 chunk/version/documents 모두 DELETE. |
| pytest fixtures 의 `session` scope 가 테스트 파일 간 데이터 누수 | ⚠️ 제한적 | `fastapi_app`/`client` 가 session scope 라, 한 테스트가 만든 문서는 다음 테스트에서도 조회 가능. 의도된 설계이며, ACL 테스트는 UUID 기반 유니크 마커로 음/양성 판정. |
| slowapi 429 rate-limit 버킷이 파일 간 공유 | ⚠️ 알려진 이슈 | IT-10 이 버스트로 429 를 유도한 뒤, 후속 RAG 테스트가 영향받을 수 있음. 검수보고서 §4 에 기재. 운영 배포에 영향 없음 (CI only). |

## 5. 결론

본 FG 0-1 의 보안 수준은 **CI 인프라 표준에 부합**한다.

- **Critical / High 위험 없음**
- 시크릿 관리 / 이미지 pin / 네트워크 노출 / 권한 최소화 모든 항목 정상
- 잔존 권고 5건 모두 운영 절차 수준 (코드 변경 불요)
- S1 원칙 ②(CI secret 평문 금지) / S2 원칙 ⑦(폐쇄망 호환 — 사내 레지스트리 이미지 override 가능) 준수

"선언적 완료 vs 실질적 완료" 관점에서 **실질적 완료**를 위해서는 §4 의 P1 조치 2건이 운영자 작업으로 남아있으며, 이후 CI 녹색 3회 연속 검증까지 완료되면 S3 Phase 1 착수 가능 상태가 된다.
