## 1. 위협 모델

본 FG 는 **DB 스키마 차원과 설정값의 일치를 검증**하는 수동적(verification-only) 모듈이다. 새로운 데이터를 쓰거나 외부에서 입력을 받지 않으며, 공격 표면이 매우 제한적이다. 그럼에도 다음 관점을 점검했다:

| 경계 | 주체 | 위협 | 범위 |
|------|------|------|------|
| Alembic revision | OWNER DB 유저 | DDL 권한 오용 | 검증만 수행, DDL 없음 |
| `/system/health` 응답 | 공개 (Tier 1) | 내부 구성 정보(차원) 노출 | §2.3 에서 검토 |
| `pg_attribute` 쿼리 | read-only | SQL injection | §2.2 에서 검토 |
| Milvus client probe | 외부 서비스 | SSRF / 타임아웃 | §2.4 에서 검토 |
| EMBEDDING_DIM env | 로컬 환경변수 | 잘못된 값 주입 | §2.5 에서 검토 |

## 2. 체크리스트

### 2.1 Alembic revision 안전성 (OWASP A01, A04)

| 항목 | 결과 | 상세 |
|------|------|------|
| DDL 실행 없음 | ✅ | `upgrade()` 는 `check_embedding_dim()` 호출만 — 테이블/컬럼 변경 없음 |
| 예외 전파 안전성 | ✅ | `EmbeddingDimMismatchError(RuntimeError)` — Alembic 이 트랜잭션 롤백 + 종료 코드 반환 |
| sys.path 조작 범위 | ✅ | `backend/` 한 경로만 prepend — `env.py` 와 동일 패턴, 외부 경로 미포함 |
| 재실행 안전성 | ✅ | 검증만 수행하므로 idempotent. 여러 번 `upgrade head` 돌려도 결과 동일 |
| downgrade 부작용 | ✅ | `return None` — 아무것도 하지 않음 |
| revision 체인 조작 가능성 | ✅ | `down_revision = "p7_2_c_uppercase_doc_type"` 하드코드 |

### 2.2 SQL 안전성 (OWASP A03)

| 항목 | 결과 | 상세 |
|------|------|------|
| `pg_attribute` 조회의 SQL injection | ✅ | 하드코딩된 스키마/테이블/컬럼 문자열만 사용. 사용자 입력 없음 |
| `format_type()` 출력 파싱 | ✅ | 정규식 `r"vector\s*\(\s*(\d+)\s*\)"` — `\d+` 만 매칭. 임의 문자열 실행 위험 없음 |
| 쿼리 파라미터 | ✅ | 파라미터 자체 없음 — 리터럴만 포함된 SELECT |
| CTE / 서브쿼리 복잡도 | ✅ | 단일 JOIN 레벨, 복잡도 낮음 |

### 2.3 Healthcheck 응답 정보 노출 (OWASP A01, A05)

**중요한 고려사항**: `/system/health` 는 Tier 1 (인증 불필요) 공개 엔드포인트다. 본 FG 에서 추가된 `embedding_dim` 블록은 다음 정보를 노출한다:

| 필드 | 정보 유형 | 위험도 | 판정 |
|------|----------|-------|------|
| `config` (예: 768) | 임베딩 차원 | 낮 | 공개 info. 대부분의 오픈 임베딩 모델 규격으로 추론 가능한 수준 |
| `db` (예: 768 or null) | DB 컬럼 차원 | 낮 | 공격자가 얻어도 직접적 공격 유도 어려움 |
| `match` (bool/null) | 운영 상태 | 중 | "DB 가 불일치 상태" 라는 운영 정보 노출 — degrade 유도 공격에 활용 가능성 |
| `reason` (string) | 사람이 읽는 요약 | 중 | 내부 아키텍처 표현 ("Milvus 중심") 노출 |
| `column_type` (예: vector(768)) | pgvector 타입 문자열 | 낮 | 이미 `match` 에서 추론 가능 |
| `milvus.dim` | Milvus 차원 | 낮 | 공개 임베딩 모델과 일치하는 값 |
| `milvus.reason` | Milvus 에러 메시지 | 중 | "Milvus 조회 실패: ConnectionError..." 등 스택 힌트 노출 가능 |

**권고 (잔존)**: `/system/health` 의 Tier 1 정책이 유지되는 한 본 FG 의 확장이 정책 상 위반은 아니다. 운영상 민감하면 Tier 2 로 이동 검토:

- `reason` / `milvus.reason` 의 문자열은 서버 내부 구조를 드러낼 수 있음 → 장기적으로 error-code 기반 요약으로 대체 권고 (예: `"COLUMN_ABSENT"`, `"MILVUS_UNREACHABLE"`)
- 다만 Phase 0 는 **방어적 구간** 이므로 디버깅 가능성을 위해 사람이 읽는 문자열을 유지 — 본 FG 완료 시점에는 수용 가능

### 2.4 Milvus probe 의 SSRF / 타임아웃 (OWASP A10)

| 항목 | 결과 | 상세 |
|------|------|------|
| 임의 호스트 접속 가능성 | ✅ | `settings.milvus_host` 는 `.env` / `docker-compose` 설정 — 런타임 주입 불가 |
| 타임아웃 보장 | ⚠️ | `_probe_milvus_dim` 이 직접 타임아웃을 설정하지 않음. pymilvus 기본값에 의존. `get_milvus()` 는 모듈 싱글턴 — 매 healthcheck 마다 새 연결은 아니므로 영향 제한적. **권고**: pymilvus 클라이언트 생성 시 `timeout` 파라미터 명시 (별도 트랙) |
| 예외 전파 방지 | ✅ | `_probe_milvus_dim` 이 모든 예외를 `(None, "Milvus 조회 실패: ...")` 로 포장 |
| 폐쇄망 (milvus_host 빈 문자열) | ✅ | 즉시 `(None, "milvus_host 미설정 (비활성)")` 반환, 네트워크 콜 없음 (S2 ⑦) |

### 2.5 환경변수 / 설정 안전성 (OWASP A05)

| 항목 | 결과 | 상세 |
|------|------|------|
| `EMBEDDING_DIM` 타입 검증 | ✅ | `int(env.strip())` — 실패 시 settings 폴백. NoValueError 로 revision 깨지지 않음 |
| `EMBEDDING_DIM` 음수/0 방어 | ⚠️ | `_get_configured_dim()` 이 음수/0 을 그대로 반환 → DB 차원과 자연스럽게 불일치 → revision 실패로 탐지. 기능상 문제 없으나 명시 에러 메시지가 더 친절했을 것. **권고 (잔존)**: `if configured <= 0: raise` 추가 — 현재는 검수 로깅으로 충분 |
| settings 객체 import 시점 | ✅ | `_get_configured_dim` 내부에서 lazy import — 테스트 monkeypatch 편의 |

### 2.6 테스트 코드 안전성

| 항목 | 결과 | 상세 |
|------|------|------|
| `monkeypatch.setenv` 누수 | ✅ | pytest monkeypatch 픽스처는 테스트 종료 시 자동 원복 |
| `importlib.util.spec_from_file_location` 동적 로딩 | ✅ | 경로가 repo 상대(`Path(__file__).parents[2]`) — 임의 경로 로딩 없음 |
| `MagicMock` 으로 DB fake | ✅ | 실 DB 건드리지 않아 격리됨 |

## 5. 결론

본 FG 0-2 의 보안 수준은 **검증-only 모듈 표준에 부합**한다.

- **Critical / High 위험 없음**
- SQL 안전성 / 예외 전파 / 폐쇄망 호환 (S2 ⑦) 모두 정상
- 발견된 3건 모두 Info/Low (Phase 1+ 에서 개선 여지)
- S1 원칙 ③ (보안 취약점 검사 필수) 준수 — 본 보고서로 증명

"선언적 완료 vs 실질적 완료" 관점에서 본 FG 는 **실질적 완료**에 가깝다 (보안 측면). 남은 조건은 §4 의 P3 3건 모두 **권고 수준** 이며 릴리스를 막지 않는다.
