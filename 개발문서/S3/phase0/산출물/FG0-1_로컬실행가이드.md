# FG 0-1 로컬 실행 가이드 — 실 DB 통합 테스트

| 항목 | 값 |
|------|----|
| 작성일 | 2026-04-23 |
| 대상 | 백엔드 개발자 / QA / CI 운영자 |
| 관련 산출물 | `backend/.gitlab-ci.yml` (운영, backend repo 루트), `backend/tests/integration/` |

본 가이드는 **로컬 개발 환경** 에서 `backend/tests/integration/` 의 통합 테스트를 실행하는 방법을 정리한다. CI 에서는 별도 action 이 자동 실행하므로 이 문서를 볼 필요가 없다.

---

## 1. 사전 요구사항

### 1.1 공통

- Python 3.11 (venv 권장)
- `backend/requirements.txt` 의존성 설치 완료

### 1.2 실행 경로 선택

두 가지 중 **하나** 만 준비하면 된다:

| 경로 | 장점 | 단점 | 적합 시나리오 |
|------|------|------|--------------|
| **A. testcontainers (기본)** | 명령어 한 줄. DB/Valkey 자동 기동·정리. | Docker 필요. 첫 실행 시 이미지 풀(≈ 1-2분). | 개인 개발자, 샌드박스 |
| **B. 기존 docker-compose 서비스 재사용** | Docker 컨테이너 재활용으로 속도 빠름. | 사전에 수동 기동 필요. | CI 시뮬레이션, 반복 실행 |

샌드박스에 Docker 가 없는 경우 두 경로 모두 사용할 수 없다. `SKIP_INTEGRATION_TESTS=1` 로 명시적 스킵 가능.

## 2. 경로 A — testcontainers (권장)

### 2.1 설치

```bash
cd backend
pip install -r requirements.txt
# requirements.txt 에 이미 testcontainers[postgres] 포함됨
```

### 2.2 실행

```bash
cd backend
pytest tests/integration/ -m integration -v
```

첫 실행 시 아래 이미지를 자동 풀 한다:

- `pgvector/pgvector:pg16`
- `valkey/valkey:8` (IT-02 등 Valkey 의존 테스트가 있을 때)

이미지 교체가 필요하면 (예: 사내 레지스트리):

```bash
INTEGRATION_PG_IMAGE=registry.example.com/pgvector/pgvector:pg16 \
INTEGRATION_VALKEY_IMAGE=registry.example.com/valkey/valkey:8 \
pytest tests/integration/ -m integration -v
```

### 2.3 디버깅

```bash
# 첫 실패에서 중단하고 pdb 진입
pytest tests/integration/test_it01_document_lifecycle.py --pdb

# 직전 실패한 테스트만 재실행
pytest tests/integration/ --lf

# 특정 테스트 id 만 실행
pytest tests/integration/test_it02_rag_after_publish.py::test_it02_publish_triggers_vectorization_and_chunks_are_indexed -v
```

## 3. 경로 B — docker-compose 서비스 재사용

### 3.1 사전 기동

프로젝트 루트에 `.env` 가 있어야 한다 (최초 1회):

```bash
cp .env.example .env
# POSTGRES_PASSWORD / JWT_SECRET / INTERNAL_SERVICE_SECRET 값을 수정
```

서비스 기동 (pgvector 이미지로 교체된 `docker-compose.yml` 기준):

```bash
# docker-compose.yml 의 `image: postgres:16` 을 `image: pgvector/pgvector:pg16` 로 수정 후
docker compose up -d db valkey
```

> **주의**: 프로젝트의 기본 `docker-compose.yml` 은 `postgres:16` 이미지라 pgvector 확장이 없다.
> 경로 B 를 쓰려면 compose override 파일(`docker-compose.integration.yml`)을 별도로 두고 거기서 이미지를 교체하거나,
> 수동으로 `CREATE EXTENSION vector` 를 매번 실행해야 한다.

### 3.2 환경 변수 + 실행

```bash
cd backend

export USE_EXISTING_DB=1
export POSTGRES_HOST=localhost
export POSTGRES_PORT=5432
export POSTGRES_USER=master
export POSTGRES_PASSWORD=<your-.env-value>
export POSTGRES_DB=mimir
export VALKEY_HOST=localhost
export VALKEY_PORT=6379
export ENVIRONMENT=test
export JWT_SECRET=<any-32+-char-string>
export INTERNAL_SERVICE_SECRET=<any-32+-char-string>
export OPENAI_API_KEY=""    # 폐쇄망 경로 검증용 (IT-05)

pytest tests/integration/ -m integration -v
```

### 3.3 정리

```bash
# 서비스 중지 (볼륨 유지)
docker compose down

# 데이터까지 완전 삭제
docker compose down -v
```

## 4. 개별 테스트 실행 팁

### 4.1 IT-00 스모크만 빠르게

인프라가 정상인지 확인만 하고 싶을 때:

```bash
pytest tests/integration/test_it00_infrastructure_baseline.py -v
```

### 4.2 IT-10 은 독립 실행 권장

IT-10 은 `/rag/answer` 를 빠르게 80회 호출하여 slowapi 429 를 유도한다. 이때 사용하는 rate-limit 버킷은 `testclient` 원격 주소 기반이므로 **같은 세션에서 실행되는 후속 RAG 테스트가 영향을 받을 수 있다**. 권장:

```bash
# 나머지 모두 먼저 실행
pytest tests/integration/ -m integration --ignore=tests/integration/test_it10_rag_rate_limit_retry.py

# IT-10 만 별도 세션에서 실행
pytest tests/integration/test_it10_rag_rate_limit_retry.py -v
```

### 4.3 IT-07 은 FG 0-2 완료 전까지 skip

`@pytest.mark.skip(reason="pending FG 0-2 ...")` 로 명시됨. 통상 수집 시 자동 skip.

### 4.4 IT-02 임베딩 미설정 환경 우회

임베딩 서비스가 붙어있지 않으면 IT-02 는 `pytest.skip` 으로 자동 우회된다. 명시 스킵을 원하면:

```bash
IT02_SKIP=1 pytest tests/integration/
```

## 6. 환경변수 요약

| 변수 | 기본값 | 용도 |
|------|-------|------|
| `USE_EXISTING_DB` | 미설정 | `=1` 이면 환경변수의 POSTGRES_*/VALKEY_* 를 직접 사용 (testcontainers 건너뜀) |
| `SKIP_INTEGRATION_TESTS` | 미설정 | `=1` 이면 `pytest.skip` 으로 전체 스킵 |
| `POSTGRES_HOST/PORT/USER/PASSWORD/DB` | `localhost/5432/master/""/mimir` | CI 모드 DB 접속 |
| `VALKEY_HOST/PORT` | `localhost/6379` | CI 모드 Valkey 접속 |
| `JWT_SECRET` | `"test-secret-for-testing-only"` | FastAPI startup 검증용. 로컬 기본값 OK |
| `INTERNAL_SERVICE_SECRET` | `"test-internal-secret"` | 내부 서비스 인증용. 로컬 기본값 OK |
| `OPENAI_API_KEY` | `""` | IT-05 폐쇄망 검증 — 비워두면 IT-05 가 활성 |
| `INTEGRATION_PG_IMAGE` | `pgvector/pgvector:pg16` | testcontainers 용 이미지 override (사내 레지스트리 등) |
| `INTEGRATION_VALKEY_IMAGE` | `valkey/valkey:8` | 동일 |
| `IT02_SKIP` | 미설정 | `=1` 이면 IT-02 스킵 |
| `IT02_VECTORIZE_WAIT_SEC` | `60` | IT-02 의 청크 인덱싱 폴링 상한 |

## 7. 자주 묻는 문제

### 7.1 `pgvector 확장을 만들 수 없습니다` 로 skip

**원인**: PostgreSQL 이미지가 pgvector 미포함 (`postgres:16` 등).

**해결**: 경로 A 는 conftest 가 `pgvector/pgvector:pg16` 을 자동 사용하므로 발생하지 않음. 경로 B 이면 compose 에서 `pgvector/pgvector:pg16` 이미지를 사용해야 한다.

### 7.2 `testcontainers 도 사용 불가` 로 skip

**원인**: Docker daemon 에 접근 불가 (WSL / 권한 / colima off 등).

**해결**: `docker ps` 로 Docker 가 동작 중인지 먼저 확인. macOS Colima 사용자는 `colima start`. Linux 는 `sudo systemctl start docker`.

### 7.3 `alembic.command.upgrade` 가 "already at head" 로 실패

**원인**: 동일 컨테이너를 반복 사용. upgrade 는 idempotent 여야 하는데 서로 다른 revision 계열이 섞인 경우.

**해결**: 경로 A 에서는 매 세션 새 컨테이너이므로 발생 안함. 경로 B 에서는 `docker compose down -v` 후 재기동.

### 7.4 IT-01 이 `FK 위반` 으로 실패

**원인**: Alembic head 가 최신이 아님. `20260422_1200_uppercase_doc_type_codes.py` 까지 적용되지 않음.

**해결**:

```bash
cd backend && alembic upgrade head
```

### 7.5 IT-10 이 "429 미발생" 으로 skip

**원인**: 테스트 환경에서 slowapi limiter 가 사실상 무제한 동작 중 (Valkey 미연결 등으로 in-memory fallback).

**해결**: CI 환경에서는 Valkey 컨테이너가 함께 올라가 자연스럽게 429 가 발생한다. 로컬에서 발생시키려면 `VALKEY_HOST` / `VALKEY_PORT` 가 정상 응답하는지 `redis-cli -h localhost -p 6379 ping` 으로 확인.
