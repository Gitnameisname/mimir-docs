## 1. 요약

FG 1-1 은 **저장 포맷 단일화 + 서버 파생 동기화** 가 주된 변경이다. 신규 외부 의존성 0건이며, 주요 리스크는 `content_snapshot` 이라는 사용자/에이전트 입력이 DB 의 유일 정본이 됨에 따라 발생 가능한 **악성 입력 / 권한 우회 / 감사 누락 / 데이터 유실** 네 가지이다. 각각에 대해 다음과 같이 방어했다:

| # | 리스크 | 방어 수단 | 결과 |
|---|-------|---------|-----|
| R1 | 악성 content_snapshot (구조 공격) | `schemas/versions.py::validate_content_snapshot` 에서 `type=="doc"` + `content: list` + 10MB 상한 강제 | ✅ 선언적 통과 |
| R2 | nodes 파생 동기화 중 SQL/JSONB 우회 | `snapshot_sync_service` 는 파라미터 바인딩 유지 (`nodes_repository.replace_for_version` 의 %s 바인딩) | ✅ 코드 확인 |

| R4 | backfill 중 데이터 손실 | dry-run 모드 + 원본을 `metadata_snapshot.content_snapshot_before_backfill` 로 보존 + published/superseded 불변 | ✅ 코드 확인 |

## 3. 입력 검증

### 3.1 content_snapshot 강화 (핵심)

`app/schemas/versions.py::validate_content_snapshot` 에 3중 가드:
1. `isinstance(v, dict)` — dict 아니면 거부
2. `len(json.dumps(v)) <= 10MB` — 크기 상한 (기존)
3. `v.get("type") == "doc"` — 루트 타입 강제 (신규)
4. `isinstance(v.get("content"), list)` — 자식 구조 강제 (신규)

검증된 거부 케이스 (`test_draft_service_fg11.py::TestContentSnapshotValidator`):
- `{type:"text", content:"x"}` (과거 agent_proposal_service 포맷)
- `{type:"document", children:[]}` (초기 S1 에서 사용된 추정 레거시)
- `{type:"doc"}` (content 누락)
- 문자열/리스트/None

**공격 표면 분석**:
- XSS: ProseMirror JSON 은 수동 렌더링 시에만 위험. Phase 1 의 render_service 는 JSON 을 그대로 소비하고 실제 HTML 렌더는 프런트 TipTap(FG 1-2) 이 담당. TipTap 은 schema 외 노드/마크를 자동 drop → HTML sanitize 보다 안전.
- JSON 폭탄: 크기 상한 10MB 로 방어. 깊이 제한은 명시적으로 두지 않았으나 `prosemirror_from_nodes` 는 재귀 깊이가 parent_id 트리 깊이로 제한됨 (DB nodes 수 ≤ 메모리 제약)
- Prototype pollution: Python dict 는 JS 와 달리 `__proto__` 경로 없음 → N/A

### 3.2 nodes 파생 동기화 바인딩

`nodes_repository.replace_for_version` 은 기존 psycopg2 parameterized query (`%s` 바인딩) 를 그대로 사용. `snapshot_sync_service.rebuild_nodes_from_snapshot` 은 flat dict 만 전달하며 raw SQL 조립은 없음.

```python
# rebuild_nodes_from_snapshot → replace_for_version → 기존 parameterized INSERT
```

SQL injection 경로 신규 없음.

### 3.3 agent_proposal_service

`create_proposal` 의 INSERT 는 기존과 동일한 parameterized query. 바뀐 것은 `content_snapshot` 의 구조뿐이며, 이는 `prosemirror_from_text` 로 서버 측 생성 → 사용자 제어 content 는 `text` 필드에만 들어가고 ProseMirror 트리의 다른 attrs 는 서버가 UUID 부여.

## 7. 데이터 보존 / 롤백 가능성

Alembic `s3_p1_content_snapshot_backfill.upgrade()` 는 비가역 작업이지만 다음과 같이 복구 가능성을 확보:

1. **dry-run 모드** — `SCHEMA_MIGRATION_DRY_RUN=1` 시 UPDATE 미실행, 대상/건수만 로깅
2. **원본 보존** — 기존 비표준 content_snapshot 이 존재하던 경우 `metadata_snapshot.content_snapshot_before_backfill` 키로 JSONB 이관
3. **published 불변** — `published/superseded` 상태 버전은 WARNING 만 출력하고 UPDATE 건너뜀
4. **schema_version** — 마이그레이션 후 문서는 `schema_version: 1` 이 부여되어 이후 마이그레이션 경로에서 식별 가능
5. **downgrade no-op** — Alembic downgrade 자체는 no-op. 개별 복원은 `metadata_snapshot.content_snapshot_before_backfill` 로 수동 가능
