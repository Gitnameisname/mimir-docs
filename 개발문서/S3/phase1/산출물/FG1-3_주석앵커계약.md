## 1. 목적

Phase 3 에서 도입될 인라인 주석 기능을 위한 **앵커 식별자 규약** 을 Phase 1 종료 시점에 확정한다. Phase 1 에서 쌓은 다음 요소들이 이 규약의 물리적 근거:

- **content_snapshot 단일 정본** (FG 1-1) — ProseMirror doc JSONB
- **NodeId global attribute** (FG 1-2) — 모든 block 레벨 노드에 UUID `node_id`
- **node_id stable 불변식** (FG 1-3) — 토글·편집·분할 전후 기존 id 보존

이 세 가지가 성립하는 동안 아래 §2 의 앵커 키는 **동일 노드를 가리키는 안정 식별자** 로 작동한다.

---

## 2. 앵커 키 구조

```
AnnotationAnchor {
  document_id : UUID
  version_id  : UUID          # 주석 생성 당시 버전
  node_id     : UUID          # content_snapshot attrs.node_id
  span_offset : int?          # (옵션) 해당 노드 내 텍스트 시작 위치
  span_length : int?          # (옵션) 선택 길이 — 하이라이트 주석용
}
```

- `document_id` + `version_id` + `node_id` 만으로도 "블록 전체" 주석이 가능 (기본 형태)
- `span_offset` / `span_length` 가 있으면 블록 내부의 일부 텍스트에 앵커 (하이라이트)
- `span_offset` 단위는 **ProseMirror doc 상의 노드 내부 text 좌표** (UTF-16 code unit; TipTap 기본)

## 3. 불변식 (Phase 1 에서 보장)

### 3.1 node_id 안정성
편집 전후로 **기존 블록의 `node_id` 가 변하지 않는다**.
- 텍스트 수정 → id 유지
- 다른 블록 추가/삭제 → 영향받지 않은 블록의 id 유지
- 블록 위치 이동 → id 유지, `order_index` 만 변경
- 뷰 토글(block ↔ flow) → id 유지, 동일 doc

### 3.2 생성 규칙
신규 블록이 생길 때만 새 UUID 부여:
- 사용자 입력(Enter 로 새 블록) — NodeId extension 의 `appendTransaction` 이 새 id 생성
- paste / setContent 로 id 없는 블록 유입 — 마찬가지로 자동 부여
- `keepOnSplit: false` — 블록 분할 시 **원본 id 는 위쪽 블록에 유지**, 아래쪽 블록에 새 id

### 3.3 저장 / 로드 왕복
`content_snapshot` JSONB 저장 + `nodes` 테이블 파생 동기화 (FG 1-1 `snapshot_sync_service`) 로 DB 의 두 표현이 항상 정합. 로드 시 `content_snapshot` 이 정본이며, nodes 테이블은 FTS/vectorization 소비자 용.

## 4. 블록 분할 / 합병 정책

### 4.1 분할 (Split — Enter 키 등)
- **위쪽(원본 id 유지)** + **아래쪽(새 id 생성)**
- 위쪽 블록에 걸린 기존 주석 앵커는 **그대로 유효**
- 아래쪽으로 옮겨간 텍스트 범위에 걸린 span 주석은:
  - `span_offset` 이 분할 지점보다 뒤라면 → "블록 내 좌표 깨짐" 상태 → Phase 3 에서 **stale 마킹**
  - Phase 3 에서 offset 재매핑 heuristic (예: text 일치 기반) 여부는 별도 결정

### 4.2 합병 (Merge — Backspace 로 블록 합치기)
- **앞 블록 id 가 살아남는다** (TipTap 기본 동작과 일치 예상 — Phase 3 에서 실측 확정)
- 뒷 블록에 걸린 주석은:
  - span 이 있으면 → 합병된 블록의 새 범위에 offset 조정 가능
  - 전체 블록 주석이면 → **해당 앵커는 stale** 처리 (블록이 사라짐)
- Stale 마킹은 주석의 `status` 필드(`active | stale | resolved`) 로 관리 예정 (Phase 3)

### 4.3 삭제 (블록 제거)
- 해당 `node_id` 를 가진 주석은 **고아 상태**
- 정책: 고아 주석은 기본 `stale` 처리 + "원본 블록" 스냅샷(메타)을 주석 본문에 보존 → 사용자가 재배치 또는 닫기

---

## 5. Version 간 주석 보존

Phase 3 의 인라인 주석은 `version_id` 를 포함하므로 **특정 버전에 붙는다**. Draft 가 publish 되거나 새 Draft 가 만들어질 때의 보존 정책:

- **동일 node_id 가 새 버전에도 존재** → 주석 자동 carry-over (UI 에서 "이전 버전에서 가져온 주석" 으로 표시)
- **node_id 가 사라짐** → stale
- **node_id 는 있으나 content_hash 불일치** → "내용이 변경됐습니다" 배너 (`CitationService.verify` 의 `modified=True` 와 동일 패턴)

## 6. 구현 체크리스트 (Phase 3 가 참고)

Phase 3 FG 3-3 구현 시 다음 순서 권장:

1. `annotations` 테이블 신설 — `{id, document_id, version_id, node_id, span_offset, span_length, status, body, author_id, created_at, resolved_at, resolved_by}`
2. 인덱스: `(document_id, node_id)` + `(version_id, status)`
3. 서비스: `annotations_service.create / list / resolve / migrate_to_version`
4. `migrate_to_version` — 새 Draft 생성 시 node_id 기반 자동 carry + content_hash 비교로 `modified` 플래그
5. API: `GET /documents/{id}/versions/{vid}/annotations`, `POST .../annotations`, `PATCH .../annotations/{aid}`
6. 프런트: TipTap decoration 으로 하이라이트 렌더 + 사이드 패널 UI
7. 권한: Scope Profile 상속 (문서 읽기 권한 있으면 주석도 읽기, 쓰기는 editor 이상)

## 7. 의존 / 전제

### 7.1 Phase 1 결과물 (이미 확보)
- `content_snapshot` 단일 정본 + schemas/versions.py validator (`type=="doc"`)
- NodeId extension (`keepOnSplit: false` + `appendTransaction` UUID 자동 부여)
- `snapshot_sync_service` 왕복 변환 + 파생 동기화
- `CitationService.verify` 의 content_hash 재계산 로직

### 7.2 Phase 3 에서 확정할 것
- Annotation `status` 전이 규약 (`active → stale → resolved` 등)
- Split 시 span offset 재매핑 heuristic 채택 여부
- Publish 시 주석 자동 마이그레이션 on/off (Scope Profile 설정)
- 주석 기반 알림 (author / 언급 `@user`) — 별도 FG 고려
