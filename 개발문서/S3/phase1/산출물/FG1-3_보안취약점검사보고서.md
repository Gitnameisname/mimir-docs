## 3. 권한 / ACL

### 3.1 `/account/preferences` 자기자신 한정

```python
@router.get("/preferences")
def get_preferences(request, actor):
    user_id = _require_auth(actor)  # actor.resolved_id, 401 if 미인증
    with get_db() as conn:
        prefs = users_repository.get_preferences(conn, user_id)
```

- path-less endpoint (`/preferences` 경로에 user_id 없음) → 타인 접근 불가
- SELECT / UPDATE 쿼리 WHERE 절은 `user_id = %s` 로 parameterized
- `_require_auth` 는 `actor.is_authenticated and actor.resolved_id` 실패 시 401 `HTTPException`

### 3.2 에이전트 접근

- S2 ⑤ 원칙상 에이전트도 자기 자신의 preferences 는 읽기/쓰기 가능
- 감사 로그에 `actor_type="agent"` 로 구분 기록. 필요 시 Phase 3 에서 에이전트 전용 preferences 스코프 도입

### 3.3 Scope Profile 영향

본 FG 는 Scope Profile 어휘 / ACL 로직을 건드리지 않음. users 테이블에 컬럼만 추가.

## 4. JSONB / SQL 안전성

### 4.1 users_repository.update_preferences

```python
cur.execute(
    "UPDATE users SET preferences = %s::jsonb, updated_at = NOW() WHERE id = %s RETURNING preferences",
    (json.dumps(merged, ensure_ascii=False), user_id),
)
```

- psycopg2 parameterized query — SQL injection 경로 없음
- `json.dumps` 후 `::jsonb` 캐스트 — 악성 문자열도 JSONB 파싱이 거부하면 DB 에러 (에러는 ApiError envelope 으로 랩핑됨)

### 4.2 get_preferences 의 예외 처리

컬럼 부재 시 `except Exception: conn.rollback(); return {}` — Alembic `s3_p1_users_preferences` 이전 DB 에서도 기능 degrade (S2 ⑦ 폐쇄망 동등성 변형). 쿼리 실패를 500 으로 흘리지 않음.

### 4.3 Alembic migration 안전성

- `ADD COLUMN IF NOT EXISTS` 가드 — 재실행 안전
- `NOT NULL DEFAULT '{}'` — 기존 row 자동 backfill (PG 11+ metadata-only)
- downgrade `DROP COLUMN IF EXISTS` — 재실행 안전. 단, **데이터 소실** 경고는 문서화 (검수보고서 §6)
