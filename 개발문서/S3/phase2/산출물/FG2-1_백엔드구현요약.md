## 2. 설계 판단

### 2.1 materialized path 선택 (ltree 미사용)

- DB extension 의존을 피하기 위해 PostgreSQL `ltree` 대신 **VARCHAR(2048) path** + `text_pattern_ops` 인덱스 채택
- path 포맷 규약: 항상 `/` 로 시작/종료. 루트 = `/<name>/`, 자식 = `<parent.path><name>/`
- 이름에 `/` 가 들어오면 `_` 로 치환 (구분자 혼동 방지). API 레이어에서는 rename 전에 사용자에게 거부 (`ApiValidationError`)

### 2.2 순환 참조 3중 방어

1. **DB CHECK**: `chk_folders_no_self_parent` — `parent_id <> id` (부분적 방어)
2. **Service `move_folder`**: `is_descendant` 검사로 "자기 자신 또는 하위를 부모로" 차단
3. **Service `move_folder`**: 루트 이동 시 `new_parent_id = folder_id` 도 별도 체크

DB CHECK 만으로는 간접 사이클(A→B→A 3 노드 이상)을 방어하지 못하므로 서비스 레이어가 정본.

### 2.3 Scope 밖 문서 추가 차단

`add_documents` 는 viewer Scope 밖 문서를 **조용히 제외** (response 에 `rejected` 카운트만 반환, 구체 id 는 유출하지 않음). 이는 "뷰 ≠ 권한" 과 "존재 유출 방지" 를 동시에 만족.

### 2.4 `?include_subfolders=true` 처리

`folders.path` 가 prefix 매칭되도록 `LIKE` 서브쿼리 구성:

```sql
id IN (
  SELECT df.document_id
  FROM document_folder df
  JOIN folders f ON f.id = df.folder_id
  WHERE f.path LIKE (SELECT path || '%' FROM folders WHERE id = %s)
)
```

`idx_folders_path_pattern (owner_id, path text_pattern_ops)` 인덱스가 빠른 prefix 스캔을 보장.

### 2.5 폴더 삭제는 CASCADE 금지

`folders.parent_id` 는 `ON DELETE RESTRICT`, `document_folder.folder_id` 도 `RESTRICT`. 실수로 상위를 지우면 하위 전부가 CASCADE 로 함께 사라지는 것을 방지. 서비스 `delete_if_empty` 가 선결.
