## 1. 배경

옵시디언 수준의 태그 동적 그룹을 도입하려면, 사용자가 태그를 **어떻게 입력/편집** 하고 **어디에 저장** 할지 규약이 먼저 확정되어야 한다. 옵시디언은 두 경로를 모두 지원하며 Mimir 의 FG 2-6 (Vault Import) 도 두 경로 모두에서 유입되는 자산을 받게 된다.

---

## 2. 선택지 비교

### A. 인라인 `#tag` 만

**장점**
- 본문과 태그가 한 곳 → 사용자 입력이 자연스러움
- 옵시디언 / Notion / Roam 등에서 친숙한 패턴

**단점**
- Markdown heading (`# 제목`) 와 혼동 가능 — 파서 복잡도 증가
- Heading 토큰은 블록 시작, inline hashtag 는 텍스트 중간이므로 구분은 가능하지만 정규식/mark 설계 주의 필요
- frontmatter 가 없으므로 **Vault Import 에서 옵시디언의 `tags:` frontmatter 태그가 누락** 된다 → 필수 병행 요구

### B. frontmatter `tags: [...]` 만

**장점**
- 본문과 분리 → 파서 단순
- 태그 편집 UI (칩 입력) 를 별도 위젯으로 구성 가능, 본문 편집과 독립

**단점**
- 사용자가 태그를 붙이려고 **매번 frontmatter 편집기를 열어야 함** → UX 손실
- 옵시디언 vault 중 인라인 `#tag` 만 쓰는 자산 대량 유입 시 태그가 소실됨

### C. 인라인 `#tag` + frontmatter `tags: [...]` 병행 ✅

**장점**
- 본문 집중 입력 ↔ 메타 중심 입력 두 스타일 모두 수용
- Vault Import 가 자연스러움 (옵시디언 원본 보존)
- 서버측 파생 테이블로 정규화하면 소비자 관점에서는 단일 집합

**단점**
- 두 소스의 중복 / 충돌 관리 필요 → 서버 파서가 정본이 되는 규약 필요
- 정규식 복잡도는 A 와 동일 (frontmatter 는 단순 JSON 파싱)

## 3. 최종 결정

### 3.1 입력 경로

두 경로 모두 지원:

| 경로 | 저장 위치 | 주 사용 시나리오 |
|------|----------|---------------|
| 인라인 `#tag` | `content_snapshot` 안의 text run — TipTap `HashtagMark` 가 시각적 강조 | 본문 작성 중 태그 즉시 부여 |
| frontmatter `tags` | `documents.metadata.tags: ["a", "b"]` (JSONB) | 칩 입력 위젯 · Vault Import 수신 · 관리자 일괄 태깅 |

### 3.2 정본 및 통합 규약

- **서버의 `extract_tags_from_snapshot(snapshot, metadata) -> set[str]` 가 정본** — PM doc 순회(인라인) + `metadata.tags` 읽기(frontmatter) 를 하나의 집합으로 반환
- `rebuild_tags_for_version(conn, version_id, snapshot, metadata)` 가 `tags` / `document_tags` 테이블을 replace
- `snapshot_sync_service` 의 `rebuild_nodes_from_snapshot` 와 **같은 트랜잭션에서 연쇄 호출** → Phase 1 의 "단일 쓰기 경로 + 파생 동기화" 패턴 그대로 재사용

### 3.3 정규화 규칙

```python
def normalize_tag(raw: str) -> Optional[str]:
    s = raw.strip().lower()
    if not s:
        return None
    # 앞의 '#' 제거 (인라인에서 오는 경우)
    if s.startswith("#"):
        s = s[1:]
    # 허용 문자: 영숫자 / _ / - / / (nested tag)
    # NFKC 정규화 (한글 호환)
    import unicodedata
    s = unicodedata.normalize("NFKC", s)
    import re
    if not re.fullmatch(r"[\w/-]{1,64}", s, flags=re.UNICODE):
        return None
    return s
```

- 대소문자 무시 (lowercase 저장)
- 길이 상한 64 (DB `tags.name_normalized VARCHAR(64) UNIQUE`)
- 중첩 태그 허용: `ai/ml`, `team/security`
- 한글 / 유니코드 허용 (NFKC 후 `\w` 매칭)

### 3.4 인라인 파서 정규식

```python
_INLINE_HASHTAG_RE = re.compile(
    r"(?:^|(?<=\s)|(?<=[^\w#]))"    # 단어 경계 또는 공백 뒤
    r"#([\w][\w/-]{0,63})"          # '#' + 허용 문자 1..64
    r"(?=\s|$|[^\w/-])",            # 뒤는 공백/끝/허용 외 문자
    flags=re.UNICODE,
)
```

- **`##` 또는 `###` 은 매칭 안 됨** (Markdown heading 과 충돌 방지는 heading 이 블록 시작이므로 text run 순회에서 자연 회피되지만, inline 에 `##foo` 같이 나타나도 위 정규식은 hash 가 2 개 이상인 경우 `(?<=[^\w#])` 가 실패해 매칭 안 됨)
- 코드 블록 / codeBlock 노드는 순회에서 **건너뜀** (메타: hashtag 는 문서 구조 태그이지 코드가 아님)
- link text 안의 hashtag 도 수집 대상 제외 (PM `link` mark 가 있는 text run 스킵)

### 3.5 TipTap HashtagMark (인라인 하이라이트)

- 정규식 기반 InputRule + PasteRule 로 `#tag` 를 자동 mark 처리
- Mark 는 **시각 강조용** 이며 **태그 인덱스의 정본은 서버 파서**. 즉 mark 가 빠져도 서버는 정규식으로 수집.
- 클릭 시 `/documents?tag=<name>` 로 이동

### 3.6 frontmatter 위젯

- 문서 상세 페이지 상단에 **태그 칩 입력**. `Tab` / `,` / `Enter` 로 확정. 백스페이스로 마지막 칩 제거.
- 저장 시 `PATCH /documents/{id}` 가 `metadata.tags` 만 업데이트
- 자동 완성: 기존 `tags` 테이블 전수 fetch → 입력 prefix 매칭

### 3.7 충돌 / 중복 처리

- 최종 집합은 `inline_tags ∪ frontmatter_tags` (union)
- 태그 **삭제** 는 "자신이 생성된 소스에서 제거" 해야 함 — 인라인 태그 칩을 UI 에서 X 로 지울 때는 본문 수정 모드로 이동하는 안내 표시. frontmatter 태그는 칩 위젯에서 바로 제거.
- `document_tags` 테이블에 `source: ENUM('inline', 'frontmatter', 'both')` 컬럼 보관 → UI 에서 지울 수 있는 경로 안내

## 5. 검수 / 보안 체크리스트 (Phase 2 각 FG 검수에서 확인)

- [ ] 인라인 파서가 코드블록 / 링크 텍스트 안의 `#foo` 를 태그로 오인하지 않는가
- [ ] `##heading` 과 혼동하지 않는가 (heading 은 block 시작 토큰이므로 구조상 안전하지만 테스트로 재확인)
- [ ] 악성 입력: 초장문 `#` 반복 (DoS), 유니코드 변종 (`#𝐚𝐛𝐜`), 호모글립 공격 → 정규화로 흡수되는지
- [ ] frontmatter 의 `tags` 가 list of str 이 아닐 때(dict, None, 중첩 list) 안전하게 무시하는가
- [ ] tag 삭제 시 `document_tags` 고아 row 가 남지 않는가
