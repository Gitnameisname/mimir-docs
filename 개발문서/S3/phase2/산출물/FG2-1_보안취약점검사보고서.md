## 2. SQL Injection

| 공격 벡터 | 방어 | 확인 |
|----------|------|------|
| `query.filters` 필드 이름 | `_FILTER_FIELD_MAP` **whitelist** — 없는 키는 skip. 컬럼명이 파라미터화되지 않고 직접 삽입되지만 **whitelist 만** 허용 | `documents_repository.py` L37-46 |
| `query.filters` 값 | `%s` 바인딩. 모든 사용자 입력은 params 배열로 전달 | L80-85 |
| `q` 파라미터 | ILIKE 패턴 구성 시 `%` / `_` / `\` **사전 이스케이프** + `ESCAPE '\\'` 절 명시 | L91-105 |
| `collection_id` / `folder_id` | UUID 타입이지만 Pydantic validator `type="uuid"` 로 검증. repo 에서도 `%s` 바인딩 | L121-146 |
| `scope_profile_id` IN 목록 | 목록 길이에 맞춰 `%s, %s, ...` placeholder 동적 생성. 값은 params 로 | L112-118 |
| `folders.path` 이스케이프 | 폴더 이름에 `/` 가 들어오면 `_` 로 치환 후 path 구성. 이름 자체는 `%s` 바인딩 | `folders_repository.py::compute_child_path` |
| search_service 서브쿼리 | documents_repository 와 동일 구조 | `search_service.py::_build_document_query` |

**결과: SQL injection 벡터 없음.** 모든 사용자 입력은 psycopg2 파라미터 바인딩 또는 `path`/패턴 이스케이프를 거친다. 회귀 테스트 `test_documents_list_q_filter.py::test_percent_is_escaped` / `test_underscore_is_escaped` / `test_backslash_is_escaped` 로 문서화됨.

| 시나리오 | 방어 | 테스트 |
|---------|------|-------|
| 타 Scope 문서 단건 조회 | `documents_service.get_document` viewer_scope 필터 → None → **404** | `test_documents_scope_profile_filter::test_get_with_mismatched_scope_raises_404` |

## 6. 존재 유출 방지 (403 → 404)

Scope 밖 문서는 **403 이 아닌 404** 로 응답 — 사용자에게 "권한 없음" 이 아니라 "존재하지 않음" 으로 보이도록. 공격자가 id 를 반복 호출해 존재 여부 매핑하는 공격을 차단.

```python
# documents_service.get_document
viewer_ids = _resolve_viewer_scope_profile_ids(actor)
doc = documents_repository.get_by_id(conn, document_id, viewer_scope_profile_ids=viewer_ids)
if doc is None:
    raise ApiNotFoundError(f"Document '{document_id}' not found")
```

`collections_service.add_documents` 도 **거부 건수만** 반환하고 구체 id 는 제외 → 공격자가 특정 id 가 컬렉션에 담을 수 있는지(=존재 + Scope 맞는지) 식별 불가.

## 8. 입력 검증

| 항목 | 제한 | 검증 위치 |
|------|------|----------|

## 11. Rate limit / DoS

| 경로 | 보호 |
|------|------|
| `/api/v1/collections/*`, `/api/v1/folders/*` | 플랫폼 기본 인증 경로 (라우터 레벨 custom rate limit 없음 — 기존 컬렉션·폴더는 리소스 생성 빈도가 낮아 기본값 충분) |
| `/api/v1/search/*` | `@limiter.limit("30/minute")` / `"120/minute"` (기존 플랫폼 설정) — FG 2-1 변경 시 영향 없음 |
| SearchInput debounce | FE 300ms debounce 로 자동 요청 완화 |
| 폴더 재귀 이동 | DB CHECK depth ≤ 10 + Service `list_by_owner` + 하위 path prefix 매칭 — 최악의 경우 O(N) scan but 인덱스 있음 |
| q 검색어 길이 | 현재 명시 제한 없음 (title ILIKE) — 악의적으로 긴 문자열 공격 시 PostgreSQL 자체 한계로 자연 거부되나 **서버에서 최대 길이 강제** 는 §12 이월 |

## 16. 후속 라운드에서 재확인 (low-severity)

1. **q 검색어 최대 길이 제한** (예: 300자) — 현재 `/documents?q=` 와 `/search?q=` 에 명시 제한 없음. 서버측 길이 가드 추가 권장
2. **collection/folder API rate limit** — 생성 빈도가 낮아 영향 적으나, 대량 생성 스크립트 공격 대비 per-user throttle 검토

위 2건은 **현재 취약점이 아닌 강화 포인트**. 본 FG 공식 종결과 무관하며 FG 2-2 ~ 이후에서 함께 정리.
