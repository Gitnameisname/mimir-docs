## 2. SQL Injection

| 공격 벡터 | 방어 | 확인 |
|----------|------|------|
| `tags_repository.search_prefix` | `WHERE t.name_normalized LIKE %s` + params `[f"{q}%"]`. `q` 는 서비스 레이어에서 `normalize_tag` 통과 후만 전달. 정규화 실패 시 None → `WHERE` 절 자체 생략 | `tags_repository.py:108-113` |
| `tags_repository.upsert_many` | `INSERT INTO tags (name_normalized) VALUES (%s) ON CONFLICT (name_normalized)` — 단일 파라미터 바인딩 | `tags_repository.py:65-74` |
| `tags_repository.replace_for_document` | DELETE `%s`, INSERT mogrify `(%s, %s, %s)` 로 미리 파라미터 이스케이프 후 조립 | `tags_repository.py:181-193` |
| `tags_repository.document_ids_for_tag` | `WHERE t.name_normalized = %s` + `d.scope_profile_id IN (%s, %s, ...)` 동적 placeholder. 값은 params | `tags_repository.py:226-247` |
| `tags_repository.delete` | `DELETE FROM tags WHERE id = %s` — UUID 파라미터 바인딩. id 자체는 Pydantic path param (FastAPI 기본 str) 이지만 psycopg2 바인딩으로 안전 | `tags_repository.py:160` |
| `_build_list_query` tag special key | 프런트 `?tag=<name>` → 서비스 `normalize_tag(name)` → repository `document_ids_for_tag(name_normalized=...)` → 결과 id 목록을 `documents.id IN (%s,...)` 로 재사용 | 전 경로 파라미터 바인딩 |

**결과: SQL injection 벡터 없음.** `LIKE` 의 `%` / `_` / `\` 메타문자는 `normalize_tag` 가 애초에 허용하지 않는 문자라 자연 차단된다 (허용문자: `[\w/-]{1,64}`). 회귀: `test_tag_rules.py` 26건이 메타문자 거부를 전수 검증.

## 6. 존재 유출 방지 (403 vs 404)

| 공격 | 방어 |
|------|------|
| 존재하지 않는 tag_id 로 DELETE | `tags_repository.delete` 가 rowcount 0 → admin 이어도 204 반환 (정책). 실존 여부 유출 없음 |
| Scope 밖 문서의 document_tags 조회 | `documents_service.get_document` 가 404 먼저 반환. document_tags 는 도달 불가 |
| 다른 유저 문서 id 로 `?tag=X` 조회 | `document_ids_for_tag` 가 scope 필터로 **id 자체를 빼고** 반환. 리스트에 나타나지 않음 → 404 와 동일 효과 |
| 특정 태그가 "존재" 하는지 확인 | `/tags?q=<prefix>` 는 org-global. 보안 경계는 태그 레이블이 아니라 **해당 태그를 가진 문서** 에 있음 (§5 참조) — 블로커1 결정서 §3.5 원칙 |

## 8. 입력 검증

| 항목 | 제한 | 검증 위치 |
|------|------|----------|
| tag 이름 | **정규화 후** `[\w/-]{1,64}` 만 허용. 1..64자. `##` 거부 | `services/tag_rules.py::normalize_tag` + `_NORMALIZED_PATTERN.fullmatch` |
| 태그 집합 길이 (metadata.tags) | Pydantic schema 명시 제한 없음 (list[str]). 파서가 개별 항목을 normalize_tag 로 거부 가능 | 이론상 수천 개 list 제출 가능 → §16 이월 (low) |
| `/tags?q=` | 길이 상한 없음. 서비스 레이어에서 normalize_tag 통과 못하면 None 으로 떨어짐 | `api/v1/tags.py::list_tags` |
| `/tags?limit=` | Query(ge=1, le=100) | FastAPI Query |
| `/tags/popular?limit=` | Query(ge=1, le=200) | 동일 |
| `/tags/popular?min_usage=` | Query(ge=1, le=1000) | 동일 |
| `document_ids_for_tag(tag_name_normalized=...)` | 서비스 레이어에서 이미 정규화된 문자열 전달 규약 | `_build_list_query` 에서만 호출 |

## 10. 사용자 입력 렌더링 (XSS) — 본 FG 의 우선 순위

HashtagMark 가 본문에 `.tag-pill` span 을 삽입하므로 특별히 정밀 점검.

| 렌더 지점 | 데이터 | 방어 |
|----------|--------|------|
| `HashtagMark.renderHTML` | `{ "data-tag": attrs.tag, class: "tag-pill" }` + `0` placeholder → TipTap 이 **텍스트 노드를 자식으로** 삽입. attrs.tag 는 `parseHTML` 시 `normalize_tag` 허용 문자만 도달 (파서 측 방어). renderHTML 도 TipTap ProseMirror toDOM 이 속성값을 HTML escape | `HashtagMark.ts:59-69` |
| `.tag-pill` 내부 텍스트 | 사용자가 친 원본 `#tag` 가 text node. React/ProseMirror 모두 escape | 검증: `parseHTML` 이 `span.tag-pill[data-tag]` selector 로만 매칭 → 임의 HTML span 이 들어와도 `data-tag` 속성값이 없거나 normalize 통과 못하면 mark 가 붙지 않음 |
| 붙여넣기 (paste) | `PASTE_REGEX = /(?:^|[^\w#])(#([\p{L}\p{N}_/-]{1,64}))/gu` 로 문자 집합 제한. 악성 스크립트 태그 파편이 매칭될 수 없음 | `HashtagMark.ts:29` |
| `TagChipsEditor` chip 이름 | `<span>#{t.name}</span>` — JSX interpolation 으로 React 자동 escape. `t.name` 은 서버에서 정규화된 문자열 | `TagChipsEditor.tsx:224` |
| `TagChipsEditor` input | `<input value={draft}>` — React controlled input. 서버 전송 전 `normalizeClient` 허용 외 문자 거부 | 동일 |
| `TagsSection` chip | `<Link href={/documents?tag=${encodeURIComponent(t.name)}}>#${t.name}</Link>` — `encodeURIComponent` 로 URL injection 방지 + JSX interpolation 으로 XSS 방지 | `TagsSection.tsx:101, 112` |
| Document list 필터 뱃지 | `label={태그: #${urlTag}}` — FilterChip 이 JSX interpolation 으로 렌더 | `DocumentListPage.tsx` |
| `attachHashtagClickHandler` | MouseEvent 에서 `pill.getAttribute("data-tag")` 로 name 추출 → `onClick(name)` 콜백. 콜백이 `router.push` 하더라도 `encodeURIComponent` 통과 | `HashtagMark.ts:93-107` |
| 자동완성 드롭다운 | `<span>#{s.name}</span>` — JSX interpolation | `TagChipsEditor.tsx:283` |

**핵심 방어선**: *정규화가 모든 입력 경계에서 일관되게 실행되며 허용 외 문자가 DB/DOM 에 진입하지 않는다.* 서버 `normalize_tag` + 프런트 `normalizeClient` + HashtagMark 정규식 = 같은 규약 `[\w/-]{1,64}` 의 3중 확인. 회귀 테스트 26+7+4 = 37 건.

**결과: XSS 벡터 없음.**

## 11. Rate limit / DoS — 자동완성 prefix 공격

| 공격 | 방어 |
|------|------|
| `/tags?q=` 반복 호출 (prefix brute force) | 서버: FastAPI 기본 rate limit (기존 플랫폼). 프런트: `useTagAutocomplete` 에 **q.trim().length > 0** 에서만 enabled + 150ms debounce in TagChipsEditor |
| 긴 q 문자열 | `normalize_tag` 가 64자 초과 시 None → 결과 빈 쿼리로 떨어지며 bruteforce 와 등가 (추가 피해 없음). 단 **서버에서 raw q 길이 가드** 는 없음 → §16 이월 |
| `/tags/popular` 반복 호출 | 인증 필요 + staleTime 60s (프런트 React Query). 태그 수가 수백 개 이상 쌓여도 LIMIT 200 clamp |
| DELETE spam | admin only + audit 이벤트 발생 → 추적 가능 |
| 대량 태그 생성 공격 | `upsert_many` 가 ON CONFLICT DO UPDATE. `tag_rules` 가 허용하는 문자만 들어가므로 `/tags` 개수가 급증해도 DB 인덱스로 소화 가능 |
| 단일 문서에 수천 태그 제출 | 현재 프런트는 UI 로 자연 제한 (칩 하나씩 수동 추가). API 직접 호출 시 metadata.tags 길이 명시 제한 없음 → §16 이월 |

## 12. PATCH metadata replace 간섭

**잠재 위험**: FG 2-2 는 `metadata.tags` 에만 관심이 있지만, `PATCH /documents/{id}` 는 서버에서 metadata 전체 replace 정책 (`schemas/documents.py::DocumentUpdateRequest` 주석 참조). 클라이언트가 `tags` 만 바꾼다고 전달하면 기타 metadata 키 유실 위험.

| 방어 | 확인 |
|------|------|
| `useSetDocumentTags` 가 호출부에서 `baseMetadata` 를 spread 후 `tags` 만 덮어씀 | `features/tags/hooks/useTags.ts:64-69` |
| `DocumentDetailPage` 가 `baseMetadata={doc.metadata ?? {}}` 로 항상 최신 metadata 전달 | `DocumentDetailPage.tsx` |
| 서버에서도 동일: `update_document` 는 받은 metadata dict 를 그대로 저장 → 클라이언트가 반드시 full merge 결과를 보내야 함 | 규약상 |
| 테스트로 검증 | `frontend/tests/TagChipsEditorNormalizeFg22.test.tsx` 7건 (normalize) + `DocumentsGetAdapterTagsFg22.test.tsx` 4건 (source 유지) — 간접 검증. **직접 merge 회귀 테스트는 §16 이월 (low)** |

**Race condition 위험**: 사용자가 같은 시각에 다른 브라우저 탭에서 metadata 를 바꾸면 baseMetadata 가 stale 이 되어 다른 변경을 덮어쓸 수 있음. 현재 FG 2-2 범위 밖. optimistic concurrency (ETag / updated_at 검증) 은 플랫폼 전역 과제.

## 13. CASCADE 연쇄 삭제 / 부수효과

| 관계 | 동작 | 확인 |
|------|------|------|
| `documents` 삭제 → `document_tags` | FK ON DELETE CASCADE (Alembic `s3_p2_tags` 에 선언) | `20260424_1900_s3_p2_tags.py` |
| `tags` 삭제 → `document_tags` | FK ON DELETE CASCADE. 대상 태그 행도 즉시 제거 | 동일 |
| `documents` 삭제가 `tags` 전역 풀에 영향 | 아니오. 개별 태그의 사용 횟수(usage_count) 는 live COUNT 이므로 자동 0 으로 수렴 | `tags_repository.popular/search_prefix` |
| 태그 1개 DELETE 시 다른 사용자 문서의 `document_tags` 행 | **조용히 전역 제거됨** — admin 전용이고 audit 으로 추적 가능하지만, **복구 UI 는 없음** | §16 이월 (low) |

### 16.3 후속 라운드에서 재확인 (low-severity)

1. **`/tags?q=` raw 길이 가드** — 현재 normalize_tag 가 64자 초과를 None 으로 떨어뜨리지만, 요청 단계에서 예: 512자 상한을 더하면 PostgreSQL 부하 완전 차단. low severity
