## 1. 위협 모델 (Threat Model)

### 1.1 자산

- `audit_events` — 누가 무엇을 언제 했는지 (개인정보 / 활동 추적)
- `workflow_history` — 누가 승인 했는지 (조직 정책)
- `documents.created_by` — 작성자 식별 (개인정보)
- `users.display_name` — 표시명 (PII)

### 1.2 행위자

- 인증 사용자 (User scope 안)
- 인증 사용자 (다른 scope)
- 비인증 (anonymous)
- 에이전트 (Agent actor)
- 관리자 (Admin)
- 외부 공격자 (네트워크 침투 후 API 호출)

### 1.3 위협

| ID | 위협 | 본 FG 대응 |
|----|-----|----------|
| T1 | scope 밖 사용자가 contributor 정보로 다른 부서 활동 추적 | ACL — `documents_service.get_document` 가 404 (존재 유출 방지) |
| T2 | 비인증 사용자가 API 직접 호출 | `authorization_service.authorize(require_authenticated=True)` 가 차단 |
| T3 | 인증 사용자가 enumerate 로 audit_events 탐색 | API 가 documents.scope_profile_id 필터 통과 후에만 동작. 직접 audit_events 노출 없음 |
| T4 | viewer 정보로 누가 무엇을 봤는지 추적 → 사생활 침해 | **본 FG 의 default 는 include_viewers=false**. FG 3-2 정책 게이트가 결합되면 강제 false 가능 |
| T5 | actor_user_id (raw UUID) 노출 → 외부 SSO ID 유추 | 응답에 actor_id 포함되지만 표시명은 `users.display_name` 사용. raw_id 는 admin 전용 별 라운드 |
| T6 | XSS via display_name | display_name 은 React 가 자동 escape (dangerouslySetInnerHTML 사용 안 함) |
| T7 | SQL injection via document_id | psycopg2 params 바인딩만 사용. 직접 문자열 보간 0 |
| T8 | rate-limit 우회로 audit emit 폭주 | `viewed_throttle` per (actor, doc) 5분 dedup. 다중 워커 환경 누수 분석은 §3 |
| T9 | path traversal via document_id | FastAPI Path validator + UUID 미검증 (현 documents 라우터 패턴과 일관) — `documents_service.get_document` 가 not-found 시 404 |
| T10 | 정보 노출 헤더 (`X-Viewers-Hidden-Reason`) 가 admin 만 의도된 정보 누출 | `scope-policy` 문자열만 노출 — Scope 명 / 사용자 ID 등 식별자 없음. 비공격적 |

## 2. 정적 검사

### 2.1 입력 검증

| 입력 | 검증 |
|-----|-----|
| `document_id` (Path) | FastAPI 가 받는 string. service 가 UUID 형식 미검증 (FG 2-0 패턴 일관). 잘못된 형식 → DB 쿼리에서 cast 실패 → 500 (별 라운드 — 표준 not_found 매핑) |
| `since` (Query, ISO 8601) | FastAPI Pydantic 자동 검증. 잘못된 형식 → 422 |
| `include_viewers` (Query, bool) | FastAPI Pydantic 자동 검증 |
| `limit_per_section` (Query, int) | `Query(..., ge=1, le=200)` 로 강제 |

### 2.2 ACL 게이트

```python
authorization_service.authorize(
    actor=actor,
    action="document.read",
    resource=ResourceRef(resource_type="document", resource_id=document_id),
    require_authenticated=True,
)
```

→ 인증 누락 / 권한 부재 시 401 / 403. 통과해도 service 레이어의 `documents_service.get_document(actor=actor)` 가 viewer 의 scope_profile_id 필터를 한 번 더 적용. 이중 게이트.

### 2.3 SQL injection

- `contributors_repository.list_editors` / `list_approvers` / `list_viewers` 모두 `cur.execute(sql, tuple(params))` 형태로 params 바인딩만 사용
- SQL 안에 `f-string` / `%` 보간 없음. since_clause 는 `"AND occurred_at >= %s"` literal 만 추가

→ **SQL injection 리스크 0**

### 2.4 raw audit_events 노출

- `ContributorsResponse` Pydantic schema 가 화이트리스트로 동작 (extra fields 응답 X)
- audit_events 의 `previous_state` / `new_state` / `request_id` / `trace_id` 등 민감 가능 컬럼은 응답에 포함 안 됨
- `actor_user_id` 만 (display_name 머지 후) 노출

### 2.5 PII / Information Disclosure

- `users.display_name` 노출 — 의도된 표시 (이미 다른 화면에서도 노출)
- `actor_id` 노출 — UUID 문자열. 외부 SSO ID 형식 (e.g. email, oauth_id) 일 수 있음. 향후 admin 전용 raw_id 분리 검토 (별 라운드)
- 누락 user → `(알 수 없는 사용자)` placeholder. raw actor_id 는 응답 schema 에 actor_id 필드로 노출됨. 이는 의도 (편집자 추적이 본 패널의 핵심) — 최소화 대안은 admin 만 raw_id 노출, 일반은 표시명만 (별 라운드)

## 3. 회귀 / 동적 검증

### 3.1 ACL 차단 회귀 (단위 테스트)

- `test_acl_failure_propagates` — `documents_service.get_document` 가 raise 하면 contributors_service 도 그대로 전파. 통과.

### 3.2 viewers default off 회귀

- `test_include_viewers_false_skips_repo_call` — repository.list_viewers 호출 자체가 안 일어남. 통과.
- `test_viewers_marked_included_when_requested` — 명시적으로 `include_viewers=true` 일 때만 viewers 응답에 포함. 통과.

### 3.3 throttle 회귀

- `test_anonymous_actor_returns_false` — anonymous (`actor_id=None`) 는 emit 차단. 통과.
- `test_concurrent_access_thread_safe` — 20 thread 동시 호출에서 정확히 1건만 emit 허용. 통과.

### 3.4 actor_type 안전성

- `test_alien_actor_type_defaults_to_user` — DB 의 자유 문자열이 alien 값이어도 응답에서 'user' 로 좁아짐. 권한 격상 없음.

## 4. "뷰 ≠ 권한 준수 확인" (CLAUDE.md §4.1 S2 원칙)

| 시나리오 | 결과 |
|---------|-----|
| user A 가 자기 scope 의 doc1 contributors 조회 | 200, 정상 응답 |
| user A 가 user B 의 scope 의 doc2 contributors 조회 | 404 (존재 유출 X) — `documents_service.get_document(actor=actor)` 가 차단 |
| anonymous 가 doc 의 contributors 조회 | 401 — `require_authenticated=True` |
| viewer scope 와 무관한 admin 이 모든 문서의 contributors 조회 | 200 (admin 권한이 scope 우회) — 의도된 동작 |
| API 가 viewer scope 무관 다른 문서의 audit_events 행을 응답에 섞는 가능성 | 0 — repository SQL 이 `WHERE document_id = %s` 단일 문서로 강제 |
| `include_viewers=true` 로 강제 호출하면 다른 사용자의 view 흔적 조회 가능 | 본 FG 에서는 가능 (UX). FG 3-2 가 결합되면 정책에 의해 false 강제 — **본 FG 의 응답 schema 는 이 결합을 위한 설계 (선택 키 + 헤더)** |

## 5. 외부 공격 surface

### 5.1 Rate Limit

- 본 라우터에 명시적 rate-limit 미적용. 작업지시서 §2.1.4 의 "per-user 60 req/min" 은 FastAPI dependency 로 별도 라운드 (이미 다른 라우터에 적용된 패턴 재사용). **본 라운드 미적용**.
- 영향: 단일 사용자가 같은 문서를 60+/min 로 호출 → 응답 비용 (audit_events 4 종 집계 SQL) 발생. 다만 응답 캐시 (`Cache-Control: private, max-age=15`) 가 클라이언트 캐시로 대부분 흡수.
- 권장: 별 라운드에서 rate-limit 적용.

### 5.2 Caching

- `Cache-Control: private, max-age=15` — 사용자별 사적 캐시 15초. CDN 등 공유 캐시에는 저장 안 됨 (private). 적정.
