## 5. 재검수 (1회차)

자기 점검 추가 발견:

1. ✅ `update_scope_profile` 의 settings_patch 처리에서 `existing` 가 변경 전 값을 한 번 가져오고 그 후 `repo.update` 가 다시 한 번 raw 머지를 위해 SELECT — 총 2회 SELECT. 성능 영향 미미 (admin write 경로). 별 라운드에서 `update` 의 raw 인자를 받게 리팩토링 검토.
2. ✅ `_settings_to_raw` 가 known key 만 dump → `update` 의 raw 머지가 미지의 키를 보존. 즉 round-trip 으로 미지의 키 손실 X.
3. ⚠️ 캐시 invalidation 은 같은 워커 내 process-local 만. 다중 워커 환경에서 다른 워커는 TTL (30s) 까지 stale. UI 안내로 mitigate. 별 라운드 cluster-wide invalidate (Valkey pub/sub 등).
4. ✅ 정책 게이트 fail-closed 가 모든 분기에서 일관 (None actor / 미인증 / scope_profile_id 누락 / repo 실패).
5. ✅ `effective_include_viewers` 가 sql 호출 자체를 막으므로 (성능 + 보안) viewer 정보가 응답 직전에서만 잘리는 게 아니라 fetch 자체가 일어나지 않음.
