## 1. 위협 모델

### 1.1 자산

- `annotations` — 주석 본문 (PII / 조직 의견)
- `annotation_mentions` — 누가 누구를 멘션했는지
- `notifications` — 알림 (수신자 ID + 본문 snippet)
- 주석 작성자 식별 (author_id)

### 1.2 위협

| ID | 위협 | 본 FG 대응 |
|----|-----|----------|
| T1 | 다른 scope 의 사용자가 주석 조회 | service `list_for_document` / `get` 가 `documents_service.get_document(actor)` 로 ACL 통과 검증 (404) |
| T2 | 비인증 사용자가 주석 작성 | service `create_annotation` 가 `actor.is_authenticated` 검증 |
| T3 | 다른 사용자 주석 본문 수정 | service `update_content` 가 `actor.actor_id == annotation.author_id` 강제 (admin 도 본문 수정 X) |
| T4 | 다른 사용자 주석 삭제 | service `delete` 가 owner 또는 admin 만 허용. 답글은 cascade 자동 삭제 |
| T5 | XSS via content | 서버 raw 저장. 프런트 React 자동 escape (dangerouslySetInnerHTML 사용 안 함) |
| T6 | 멘션 폭주 (DDoS / 도배) | per (author, recipient) 분당 5건 rate-limit (`notifications_service.enqueue_mention`) |
| T7 | 자기 자신 멘션 알림 | service 가 author == recipient 시 silent skip |
| T8 | 다른 사용자 알림 조회 / 읽음 처리 | repository SQL 안 `user_id` 필터 강제 + service 가 `actor.actor_id` 사용 |
| T9 | 알림 payload 의 snippet 으로 XSS | snippet 200 자 truncate + React 자동 escape |
| T10 | mention 으로 존재하지 않는 user_id 유출 | username → user_id resolve 후 valid 만 채택. 미존재 silent skip (응답에 영향 없음) |
| T11 | MCP read_annotations 우회 | (1) `_check_agent_write_blocked` (kill switch), (2) `authorization_service.authorize`, (3) `annotations_service.list_for_document` ACL — **3 단계 게이트** |
| T12 | MCP read_annotations 결과로 다른 scope 의 작성자 정보 누출 | annotations_service 가 ACL 차단된 문서면 ApiNotFoundError → MCPError NOT_FOUND. 결과 자체가 안 나옴 |
| T13 | annotation 생성으로 ACL 외 문서에 쓰기 | service 가 documents_service.get_document 로 ACL 검증 후만 INSERT. 못 보면 404 |
| T14 | parent_id 로 다른 문서의 답글 시도 | service 가 parent.document_id != document_id 면 ApiValidationError |
| T15 | SQL injection via document_id / annotation_id | psycopg2 params 바인딩만 사용. 직접 보간 0 |
| T16 | 알림 무한 폴링으로 DB 부담 | 30s polling + unread-count COUNT(*) 인덱스 (`idx_notifications_user_unread`) — light query |
| T17 | content 길이 공격 (10MB+ 본문) | DDL CHECK length(content) BETWEEN 1 AND 10000 + service `_validate_content` 이중 검증 |
| T18 | 답글 깊이 공격 (재귀) | service 가 답글의 답글을 root 로 평탄화 — 깊이 1 고정 |
| T19 | content 안 ANSI control / null byte | TEXT 컬럼은 모든 char 허용. 별 라운드에서 sanitize 검토 |
| T20 | 알림 metadata 로 민감정보 누출 | payload = `{author_id, annotation_id, document_id, snippet}` 만 — IP / 세션 / 비밀값 0 |

## 2. 정적 검사

### 2.1 입력 검증

| 입력 | 검증 |
|-----|-----|
| document_id (Path) | FastAPI string. service 가 documents_service 로 검증 (404 시 차단) |
| annotation_id (Path) | service `get_by_id` → not-found 시 404 |
| content (body) | Pydantic min_length=1 max_length=10_000 + service `_validate_content` (whitespace-only 차단) |
| node_id (body) | service 가 형식 검증 안 함 (Phase 1 의 다른 FG 와 일관) |
| span_start / span_end (body) | Pydantic ge=0 / ge=1. DDL CHECK span_start < span_end (둘 다 NOT NULL 시) |
| parent_id (body) | service 가 부모 존재 + 같은 문서 검증 |
| limit (query) | Pydantic ge=1 le=500 |

### 2.2 권한 매트릭스

| 액션 | 권한 |
|-----|-----|
| create | 인증 사용자 (ACL 통과 후) |
| update content | 작성자 본인만 (admin X) |
| resolve / reopen | 작성자 또는 admin |
| delete | 작성자 또는 admin (cascade 답글) |
| list / get | 인증 사용자 (ACL 통과 후 — viewer 가 못 보면 404) |
| MCP read_annotations | agent kill switch off + Scope Profile + document.read |

### 2.3 SQL injection

- 모든 SQL 이 psycopg2 params 바인딩
- ANY(%s::uuid[]) 패턴도 params 바인딩 (notifications mark_read 등)
- 직접 string concat 0

### 2.4 raw audit_events / 알림 노출

- ContributorsResponse / AnnotationResponse / NotificationResponse 모두 Pydantic schema (extra fields 응답 X)
- annotation_mentions 의 `mentioned_user_id` 는 응답에 포함 (다른 사용자가 누구를 멘션했는지 노출) — 의도된 동작 (스레드 협업)

### 2.5 ACL 강제

```
documents_service.get_document(conn, document_id, actor=actor)
  → scope 밖이면 ApiNotFoundError (404)
```

모든 service 진입점에서 호출. 우회 경로 0.

### 2.6 MCP 권한 다중 게이트 (실측 — Codex 정적 리뷰 2026-04-27 반영)

```python
def tool_read_annotations(request, actor, conn):
    _check_agent_write_blocked(actor)              # 0) no-op (tools.py:327–331 주석 — 읽기 도구는 인증 단계에서 차단)
    if not _UUID_RE.match(...): raise              # 1) input validation
    authorization_service.authorize(...)            # 2) document.read scope
    items = annotations_service.list_for_document(  # 3) service ACL (get_document)
        conn, actor=actor, document_id=...
    )
```

→ **실효 3 게이트** (UUID + document.read + service ACL). 단일 우회는 어렵지만 task3-3.md §[129,223–225,318] 가 요구한 **tool-level allow 체크 (`ScopeProfile.allowed_tools["read_annotations"]` + `AgentPrincipal.can_read("annotations")`) 가 미구현**. agent 가 document.read 권한만 있으면 본 도구가 ScopeProfile 에서 비활성이어도 호출 가능. 본 갭은 read_annotations 단일이 아니라 MCP 표면 전체 모델 누락이며, FG 3-3 §5 #9 / 종결회고 §4.1 #11 잔존부채 등재. Phase 4 FG 4-1 회귀 적용 범위에서 일괄 처리.

## 3. "뷰 ≠ 권한 준수 확인"

| 시나리오 | 결과 |
|---------|-----|
| user A 가 자기 scope doc1 의 주석 list | 200 정상 |
| user A 가 user B scope doc2 의 주석 list | 404 (존재 유출 X) |
| user A 가 user B 의 scope 외 주석 id 직접 GET | 404 (annotations_service.get 의 ACL 통과 실패) |
| user A 가 다른 사용자 주석을 update | 403 ApiPermissionDeniedError |
| admin 이 다른 사용자 주석 update | 403 (본문 수정은 본인만) |
| admin 이 다른 사용자 주석 resolve / delete | 200 (admin 권한 인정) |
| anonymous user 가 list | 401 (resolve_current_actor + service 자체 차단) |
| 에이전트가 MCP read_annotations 로 ACL 외 문서 조회 | 404 NOT_FOUND |
| 에이전트가 kill switch on 인 상태 | _check_agent_write_blocked 가 차단 |
| user A 가 user B 멘션 6번째 → 알림 5건만 enqueue | rate-limit 동작 (5건 후 silent skip) |
