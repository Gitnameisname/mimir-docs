## 4. 종결 테스트 세션 회고 (2026-04-18)

Phase 9 완료 선언 후 실제 인프라(Docker PostgreSQL, Valkey, Milvus, SBERT) 기반으로 블랙박스 테스트를 수행한 결과 **5건의 실제 버그**와 **2건의 보안 취약점**을 추가 발견·수정했다.

### 4.1 발견된 버그 요약

| ID | 위치 | 내용 | 심각도 |
|----|------|------|--------|
| BUG-01 | `workflow_service.py` | publish 시 `versions.status`, `documents.current_published_version_id` 미갱신 | High |
| BUG-02 | `vectorization_service.py` | `content_snapshot` 경로로 저장한 문서는 `nodes` 테이블이 비어 벡터화 불가 | Medium |
| BUG-03 | `vectorization_service.py` | `_record_token_usage` FK 오류 후 트랜잭션 aborted → 청크 전체 롤백 | High |
| BUG-04 | `document_chunks` 스키마, `config.py` | pgvector 컬럼 1536차원 vs SBERT 768차원 불일치 | Medium |
| BUG-05 | `embedding_service.py`, `config.py` | 로컬 SBERT 임베딩 프로바이더 미구현, 환경변수 매핑 누락 | Medium |

### 4.2 교훈

**1. 트랜잭션 상태 관리 — SAVEPOINT 패턴 필수화**

BUG-03은 psycopg2에서 예외 발생 후 트랜잭션이 aborted 상태로 남아 이후 `commit()`이 아무것도 저장하지 않는 현상이다. 내부에서 예외를 `except`로 잡더라도 DB 트랜잭션은 여전히 오염된 상태다. **선택적으로 실패해도 되는 DB 쓰기는 반드시 `SAVEPOINT` / `ROLLBACK TO SAVEPOINT`로 격리해야 한다.**

**2. 임베딩 차원은 스키마 생성 시점에 고정**

`vector(1536)`으로 컬럼을 만들었다가 768로 바꾸려면 인덱스 재생성이 필요하다. 모델 선택과 DB 스키마를 함께 결정해야 하며, `EMBEDDING_DIM` 환경변수를 config에 명시적으로 매핑하는 것을 배포 체크리스트에 포함시켜야 한다.

**3. content_snapshot 경로와 nodes 경로의 이원화**

`PUT /draft`(content_snapshot)와 `PUT /draft/nodes`(nodes 테이블 직접) 두 경로가 있는데, 벡터화 서비스는 nodes 테이블만 보고 있었다. **두 저장 경로가 있을 때 하위 서비스가 어느 경로를 보는지 명시적으로 문서화**해야 한다.

**4. 단위 테스트가 통과해도 인프라 통합 버그는 나온다**

479개 단위 테스트가 모두 통과했음에도 실제 DB 연동 시 FK 제약, 차원 불일치, 트랜잭션 오염 같은 버그가 드러났다. **S3에서는 Docker Compose 기반 실 DB 통합 테스트를 CI에 포함**시키는 것이 필수다.

## 5. Keep / Problem / Try

### Keep

- **FG별 검수 게이트**: Phase마다 검수보고서·보안보고서를 의무화한 것이 최종 품질 보증에 결정적이었다.
- **S2 원칙의 일관된 적용**: 접근범위 하드코딩 0건, 폐쇄망 환경 전 기능 동작, AI 에이전트 1급 소비자 처리가 코드 전체에 일관되게 반영되었다.
- **폴백 설계(graceful degrade)**: RAG 인프라 없이도 서비스가 정상 동작하고, 인프라 추가 시 코드 변경 없이 기능이 활성화되는 구조가 테스트에서 검증되었다.
- **Citation 계약의 일관성**: Phase 2부터 Phase 9까지 5-tuple Citation 계약이 한 번도 깨지지 않았다.

### Problem

- **실 DB 통합 테스트 부재**: 단위 테스트만으로는 트랜잭션 오류, FK 제약 위반, 차원 불일치를 잡지 못한다.
- **두 저장 경로의 하위 호환성**: `content_snapshot` vs `nodes` 이원화로 벡터화 파이프라인에 갭이 생겼다. 신규 저장 경로 추가 시 연관 서비스 영향 분석이 필요하다.
- **임베딩 서비스 안정성**: 개발 환경에서 `--reload` 플래그로 실행된 임베딩 서버가 재시작 중 요청을 받으면 실패한다. 서비스 프로세스 관리 정책이 필요하다.
- **환경변수 명명 규칙 불일치**: `EMBEDDING_SVC_URL`이 config 필드 `embedding_service_url`과 이름이 달라 매핑이 누락됐다. 환경변수 명명 규칙을 통일해야 한다.

### Try

- **S3에서 실 DB CI 도입**: `testcontainers` 또는 Docker Compose 기반으로 PostgreSQL + Valkey + pgvector를 CI에 포함한다.
- **저장 경로 단일화 또는 명시적 문서화**: `content_snapshot`과 `nodes` 중 하나를 primary로 정하거나, 두 경로가 공존하는 경우 연관 서비스 목록을 ADR로 관리한다.
- **환경변수 명명 규칙 통일**: `COMPONENT_SUB_KEY` 패턴으로 통일하고 config 필드와 1:1 매핑되도록 레거시를 정리한다.
- **SAVEPOINT 패턴 가이드라인 문서화**: "선택적 DB 쓰기는 SAVEPOINT로 격리" 규칙을 CLAUDE.md 코딩 원칙에 추가한다.
