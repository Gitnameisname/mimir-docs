## 1. Phase 8 — 승인된 추출 결과의 저장 방식 변경

### 문제
승인된 추출 결과를 `Document.metadata`에 직접 통합하는 설계는 S1 원칙 ② "DB 모델은 최대한 generic 유지"와 충돌하며, 추출 스키마 버전/모델 버전/인간 수정 이력의 독립 추적을 약화시킴.

### 수정 내용
FG8.2 "추출 결과 저장" 항목(기존 220행 부근)을 다음과 같이 변경:

- `Document.metadata` 직접 통합 제거
- 별도 artifact(`ApprovedExtraction`) 테이블 도입: `document_id + version_id + extraction_schema_version` 복합 키
- 필요 시 projection/materialized view 경유로만 metadata 반영
- artifact에 `extraction_schema_version`, `extraction_model`, `human_edits`, `approved_at/by` 필드 포함
- 검수 기준도 변경된 저장 구조에 맞춰 업데이트

### 근거
Document 도메인 모델의 범용성을 보존하면서, 추출 결과의 버전 관리와 감사 추적을 독립적으로 수행할 수 있음.

## 2. Phase 1 — 폐쇄망 모델 반입 정책 추가

### 문제
BGE-M3, E5 로컬 Provider가 Hugging Face 자동 다운로드에 의존하는 설계는 폐쇄망에서 성립하지 않으며, S2 원칙 ⑦ "폐쇄망 환경에서 전체 기능 동작 — degrade하되 실패하지 않음"과 충돌.

### 수정 내용
FG1.2 Embedding Provider 구현 항목(기존 160~180행 부근)을 다음과 같이 변경:

- BGE-M3: `BGE_MODEL_PATH`를 필수 환경변수로 격상, 자동 다운로드 기본값 제거
- 폐쇄망 지원 정책 3요소 명시: 사전 반입(pre-provisioning) 경로, SHA256 무결성 검증(`BGE_MODEL_CHECKSUM`), 미설치 시 FTS fallback
- E5: 동일 정책 적용
- `EmbeddingProviderFactory`: `MODEL_AUTO_DOWNLOAD` 환경변수 도입 (기본값 `false`), 온라인 환경에서만 `true` 설정 시 자동 다운로드 허용
- 모델 미존재 시 해당 provider를 `unavailable`로 등록, FTS 전용 모드로 degrade (서비스 중단 없음)

### 근거
폐쇄망 환경에서 네트워크 의존 없이 서비스가 시작되며, 임베딩 모델 없이도 FTS 기반 검색으로 graceful degradation 보장.

## 3. Phase 0 — capabilities 엔드포인트 2-tier 분리

### 문제
`/api/v1/system/capabilities`를 인증 없이 공개하면 `supported_providers`, `pgvector_enabled` 등 내부 구성 정보가 공격자에게 공격 표면 분석 자료를 제공함.

### 수정 내용
FG0.1 항목 4(기존 135행 부근)를 3-tier 엔드포인트로 재설계:

- **4a. Public Health** (`/api/v1/system/health`): 인증 불필요. `status`와 `version`만 노출
- **4b. Capabilities** (`/api/v1/system/capabilities`): 인증 필요 (최소 authenticated user). `rag_available`, `chunking_enabled` 노출
- **4c. Admin Capabilities** (`/api/v1/admin/system/capabilities`): Admin 권한 필수. `supported_providers`, `pgvector_enabled`, 내부 기능 스위치 상세 노출

### 근거
정보 노출 최소화 원칙(Least Information Disclosure) 준수. 비인증 사용자에게는 서비스 가용성 정보만 제공하고, 내부 구성 상세는 권한에 따라 단계적으로 공개.

## 4. Phase 9 — Prompt Injection 대응책 교체

### 문제
"체인-오브-스로트(CoT) 프롬프트로 논리 강화"는 prompt injection 완화의 핵심 제어가 아니며, 오히려 민감 추론 과정을 노출하여 공격 표면을 확대할 수 있음.

### 수정 내용
FG9.2 LLM01 Prompt Injection 대응책(기존 69~73행 부근)에서 CoT 항목을 제거하고 다음으로 대체:

- **Structured Prompt Slots**: system / context / query 구조 분리, 슬롯 경계 침범 방지
- **Tool Allowlist**: LLM 호출 가능 도구를 사전 정의된 allowlist로 제한, 미허용 호출 즉시 차단 및 로깅
- **Schema Validation**: LLM 응답의 JSON schema 검증, 예상치 못한 필드/구조 변형 탐지
- **Post-check 정책**: rule-based 후처리 검사 (금지 패턴, citation 무결성, 지시문 재삽입 여부)
- CoT 제외 사유를 취소선 주석으로 명시

### 근거
OWASP Top 10 for LLM Applications 권고에 부합하는 실질적 방어 계층(instruction/data separation, output validation, allowlisting)으로 교체.

## 5. Phase 1, 2 — "Phase 10" 의존 표기 명확화

### 문제
Phase 1 FG1.2 항목 6과 Phase 2 선행 조건에서 "Phase 10 벡터화"를 참조하는데, S2 문맥에서 존재하지 않는 Phase로 읽힐 수 있어 일정 관리상 혼선 유발.

### 수정 내용

- Phase 1 FG1.2 항목 6: "Phase 10 벡터화" → "S1 Phase 10 벡터화"
- Phase 2 선행 조건: "Phase 10 벡터화 파이프라인 기초" → "S1 Phase 10 벡터화 파이프라인 기초"

### 근거
S1/S2 Phase 번호가 겹치므로, S1 자산 참조 시 반드시 "S1" 접두어를 붙여 혼선 방지.
