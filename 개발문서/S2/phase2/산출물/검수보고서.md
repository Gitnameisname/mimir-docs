### [검수-001] FTSRetriever/VectorRetriever — 빈 source_text로 CitationBuilder 호출 시 ValueError 미처리

| 항목 | 내용 |
|------|------|
| **심각도** | Major (운영 중 검색 결과 누락 또는 500 오류) |
| **파일** | `app/services/retrieval/fts_retriever.py:94-114`, `vector_retriever.py:110-130` |
| **원인** | `_row_to_result()`에서 `source_text=row["source_text"] or ""`를 사용할 때, DB에 빈 문자열이 저장된 청크에 대해 `CitationBuilder.build("")` 호출 → `ValueError("source_text must not be empty")` 발생. 해당 예외는 `retrieve()` 내의 list comprehension에서 처리되지 않아 `[]` 대신 예외가 상위로 전파됨 |
| **수정 내용** | `_row_to_result()`의 반환 타입을 `Optional[RetrievalResult]`로 변경. 빈 `source_text`가 있으면 `DEBUG` 로그 후 `None` 반환. 호출 측에서 `None` 필터링 처리 |
| **수정 파일** | `fts_retriever.py`, `vector_retriever.py` |

**수정 전:**
```python
return [self._row_to_result(row) for row in rows]  # 예외 전파 가능

def _row_to_result(row: dict) -> RetrievalResult:
    citation = CitationBuilder.build(..., source_text=row["source_text"] or "")  # "" 가능 → ValueError
```

**수정 후:**
```python
results = []
for row in rows:
    r = self._row_to_result(row)
    if r is not None:
        results.append(r)
return results

def _row_to_result(row: dict) -> Optional[RetrievalResult]:
    source_text = row.get("source_text") or ""
    if not source_text:
        logger.debug("skipping chunk %s — empty source_text", ...)
        return None
    # ... CitationBuilder.build(source_text=source_text)
```

### [검수-002] search_documents — 동기 핸들러에서 asyncio.run() 사용

| 항목 | 내용 |
|------|------|
| **심각도** | Major (특정 환경/미들웨어에서 RuntimeError 발생 가능) |
| **파일** | `app/api/v1/search.py:213-227` |
| **원인** | `search_documents`는 `def`(동기)로 정의되어 있으나 `search_with_plugins()`는 `async` 코루틴. 이를 호출하기 위해 `asyncio.run()`을 사용했는데, FastAPI는 sync handler를 threadpool에서 실행하며 일부 환경(nest_asyncio 미적용, 특정 ASGI 미들웨어)에서 `RuntimeError: This event loop is already running` 발생 가능 |
| **수정 내용** | `search_documents`를 `async def`로 변경하고 `asyncio.run()` 제거, `await` 직접 사용 |
| **수정 파일** | `app/api/v1/search.py` |

**수정 전:**
```python
def search_documents(...):
    ...
    asyncio.run(search_service.search_with_plugins(...))
```

**수정 후:**
```python
async def search_documents(...):
    ...
    plugin_results = await search_service.search_with_plugins(...)
```

### [검수-003] MultiturnRAGService — 대화 압축 결과 미사용

| 항목 | 내용 |
|------|------|
| **심각도** | Major (기능 결함: 압축 활성화되었다고 표시하나 실제 동작 무효) |
| **파일** | `app/services/multiturn_rag_service.py:103-105` |
| **원인** | `await self._compressor.compress()` 호출 후 반환된 압축된 컨텍스트 문자열을 버리고 `context_compressed = True`만 설정. 이후 `rewritten_query`는 여전히 원본 `history`(압축 전)를 기준으로 생성됨 |
| **수정 내용** | 압축된 요약을 `SYSTEM` 역할 메시지로 변환하여 `rewriter.rewrite_query()`에 전달. 압축 후 재작성도 새 컨텍스트 기반으로 수행 |
| **수정 파일** | `app/services/multiturn_rag_service.py` |

**수정 전:**
```python
if len(history) > _COMPRESS_THRESHOLD:
    await self._compressor.compress(history, strategy="summarize")  # 결과 버림
    context_compressed = True
search_query = rewritten_query or query  # 여전히 원본 history 기반
```

**수정 후:**
```python
if len(history) > _COMPRESS_THRESHOLD:
    compressed_context = await self._compressor.compress(history, strategy="summarize")
    context_compressed = True
    history = [ConversationMessage(role=MessageRole.SYSTEM, content=compressed_context, ...)]
    rewritten_query = await self._rewriter.rewrite_query(query, history)  # 압축된 컨텍스트로 재작성
```

### [검수-005] POST /rag/answer — DB 연결을 LLM 호출 시간 동안 점유

| 항목 | 내용 |
|------|------|
| **심각도** | Major (고부하 시 DB 연결 풀 고갈 위험) |
| **파일** | `app/api/v1/rag.py`, `app/services/multiturn_rag_service.py` |
| **원인** | `with get_db() as conn:` 블록 내에서 `MultiturnRAGService.answer()`를 전체 실행함. 이 메서드는 DB 조회(prepare_context) 후 LLM API 호출(수 초 소요)까지 conn을 점유함. S1 RAG-002 패턴(prepare 완료 후 conn 해제, LLM 호출)을 위반 |
| **수정 내용** | Phase 3에서 `MultiturnRAGService`를 `prepare()` + `generate()` 2단계로 분리하도록 주석 추가. 현재는 `prepare_context()` 완료 후 LLM 호출이 이루어지므로 S1 내부 패턴(prepare_context 이후 conn 불사용)이 적용되어 있음을 코드 주석으로 명시 |
| **상태** | **기술 부채 (Phase 3에서 서비스 분리 필요)** |

> **Note**: `rag_service.prepare_context()`는 이미 DB 작업을 완료하고 반환하므로 이후 LLM 호출에서 conn을 실제로 참조하지 않는다. 그러나 Python `with` 블록 scope상 conn이 해제되지 않은 상태로 LLM 호출이 이루어져 연결 풀 관점에서는 점유 중으로 계산된다. Phase 3에서 구조적 분리 필요.

## 6. 보안 검토

| 항목 | 결과 | 비고 |

| Retriever SQL — parameterized query 사용 | ✅ | f-string 사용 부분은 enum 값으로 제한 |
