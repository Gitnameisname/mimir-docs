## 발견 항목 상세

### SEC-01 · Critical · A01 접근 제어 실패

**위치**: `backend/app/api/v1/extractions.py` — `list_pending_extractions`  
**설명**: `_scope_profile_id(actor)` 반환값이 `None`인 경우(scope가 설정되지 않은 사용자) `ExtractionCandidateRepository.list_pending(scope_profile_id=None)`이 WHERE 절 없이 전체 레코드를 반환한다. 이는 조직 간 데이터 격리 위반이다.

**루트 원인**:
```python
# extraction_candidate_repository.py
if scope_profile_id is not None:
    scope_filter = "AND scope_profile_id = %s"
# → None이면 scope_filter = "" → 전체 조회
```

**수정 방법**: `scope_id is None` AND 관리자 역할 아님 → HTTP 403 반환

```python
# 수정 후
scope_id = _scope_profile_id(actor)
role = getattr(actor, "role", None)
if scope_id is None and role not in {"ORG_ADMIN", "SUPER_ADMIN"}:
    raise HTTPException(status_code=403, detail="scope_profile_id가 없는 사용자는 목록을 조회할 수 없습니다.")
```

**상태**: ✅ 수정 완료

### SEC-02 · High · A07 인증 실패

**위치**: `backend/app/api/auth/` — JWT 페이로드  
**설명**: `actor_type` 클레임이 JWT에 없을 경우 `actor.actor_type`이 `None`이 되고, 모든 감사 로그에 `actor_type or "user"`로 기본값이 설정된다. 에이전트가 `actor_type="user"`로 위장할 수 있다.

**상태**: ⛔ 제외 — Phase 4 인증 레이어 소관. Phase 4 후속 작업으로 추적 중.

### SEC-03 · High · A04 안전하지 않은 설계

**위치**: `backend/app/models/batch_extraction.py` — `BatchRetryRequest`  
**설명**: `date_from`/`date_to` 범위에 상한이 없어 수년치 문서를 한 번에 재추출 요청할 수 있다. 이는 DB 과부하 및 비용 폭발로 이어질 수 있다.

**수정 방법**: `_validate_date_range` 검증에 365일 상한 추가

```python
if (self.date_to - self.date_from) > timedelta(days=365):
    raise ValueError("date range must not exceed 365 days")
```

**상태**: ✅ 수정 완료

### SEC-04 · High · A04 안전하지 않은 설계

**위치**: `backend/app/services/extraction/diff_calculator.py` — `DiffCalculator._compare_values`  
**설명**: dict 값에 대해 `self.compare(a, b)` 재귀 호출 시 깊이 제한이 없다. 악의적으로 깊게 중첩된 JSON 추출 결과를 입력하면 Python 스택 오버플로우를 유발할 수 있다.

**수정 방법**: `_MAX_RECURSION_DEPTH = 10` 상수 추가, `compare()` 메서드에 `_depth` 파라미터 추가, 초과 시 `MISMATCH` 반환

```python
_MAX_RECURSION_DEPTH = 10

def compare(self, original, new, fields_filter=None, _depth=0):
    if _depth > _MAX_RECURSION_DEPTH:
        return MatchStatus.MISMATCH, []
    ...
    # dict 재귀 시
    _, sub_diffs = self.compare(a, b, _depth=_depth + 1)
```

**상태**: ✅ 수정 완료

### SEC-05 · High · A03 인젝션 (SSTI)

**위치**: `backend/app/services/extraction/extraction_evaluator.py` — `EvaluationReportGenerator`  
**설명**: `jinja2.Environment()`에 `autoescape=True`가 없다. `field_details`에 `{{ 7*7 }}`과 같은 문자열이 포함된 경우 템플릿 인젝션이 발생할 수 있다.

**수정 방법**: `select_autoescape` 적용

```python
from jinja2 import Environment, select_autoescape
self._env = Environment(
    autoescape=select_autoescape(default=True, default_for_string=True)
)
```

**상태**: ✅ 수정 완료

### SEC-06 · High · A03 인젝션

**위치**: `backend/app/services/extraction/extraction_evaluator.py` — `ExtractionEvaluator.evaluate_extraction`  
**설명**: `actual_fields` dict의 값 타입을 LLM 응답에서 그대로 사용한다. `_check_type`은 검증만 하고 반환값은 그대로 `field_details`에 저장된다. 타입 변환 없이 downstream에서 문자열로 재직렬화될 경우 데이터 오염 가능성이 있다.

**상태**: ⛔ 제외 — 추출 스키마 타입 강제화는 Phase 8 FG8.1 스키마 검증 레이어의 소관. 추후 FG8.1 개선 시 반영 예정.

### SEC-07 · Medium · A09 로깅 실패

**위치**: `backend/app/api/v1/extraction_schemas.py`, `extractions.py` — 500 에러 핸들러  
**설명**: `raise HTTPException(status_code=500, detail=f"서버 오류: {exc}")` 패턴이 다수 사용되어 내부 예외 메시지가 API 응답에 그대로 노출된다. DB 스키마, 파일 경로, 설정 정보 등이 누출될 수 있다.

**수정 방법**: 모든 500 핸들러의 `detail`을 정적 문자열로 교체 (실제 오류는 `logger.exception`으로 서버 사이드에만 기록)

```python
# 수정 전
raise HTTPException(status_code=500, detail=f"서버 오류: {exc}")
# 수정 후
raise HTTPException(status_code=500, detail="내부 서버 오류가 발생했습니다.")
```

대상 파일: `extraction_schemas.py` 3건, `extractions.py` 4건 — 총 7건 수정

**상태**: ✅ 수정 완료

### SEC-08 · Medium · A01 접근 제어 실패

**위치**: `backend/app/api/v1/batch_extractions.py` — `cancel_batch_job`  
**설명**: 취소 권한 검증이 `scope_profile_id` 일치 여부만 확인하고 `created_by` 검증이 없다. 동일 scope 내 다른 사용자의 배치 작업을 취소할 수 있다.

**수정 방법**: `job.created_by != actor_id` 검증 추가

```python
if role not in {"ORG_ADMIN", "SUPER_ADMIN"}:
    if job.scope_profile_id != scope_id or job.created_by != actor_id:
        raise HTTPException(403, "이 배치 작업에 접근할 권한이 없습니다.")
```

**상태**: ✅ 수정 완료

### SEC-10 · Medium · A09 로깅 실패 (PII)

**위치**: `backend/app/services/extraction/extraction_verification_service.py` — `build_audit_trail`  
**설명**: 감사 이력(audit trail)에 `"extracted_result": record.extracted_result` 전체가 포함된다. 개인정보(PII)가 포함된 추출 결과가 감사 로그에 그대로 저장되는 개인정보 침해 위험이 있다.

**수정 방법**: 전체 결과 대신 필드 수와 결과 해시만 저장

```python
"extracted_field_count": len(record.extracted_result) if record.extracted_result else 0,
"extracted_result_hash": self.compute_document_hash(
    str(sorted(record.extracted_result.items()) if record.extracted_result else "")
),
```

**상태**: ✅ 수정 완료

### SEC-12 · Low · A01 접근 제어 실패

**위치**: `backend/app/api/v1/extraction_evaluations.py` — `create_golden_set`  
**설명**: 요청 본문의 `scope_profile_id`를 그대로 수용하여 `gset.scope_profile_id = gset.scope_profile_id or _scope_id(actor)` 패턴을 사용한다. 클라이언트가 임의의 scope에 Golden Set을 생성할 수 있다.

**수정 방법**: 클라이언트 입력을 무시하고 항상 actor의 scope를 사용

```python
gset.scope_profile_id = _scope_id(actor)  # 클라이언트 입력 무시
```
