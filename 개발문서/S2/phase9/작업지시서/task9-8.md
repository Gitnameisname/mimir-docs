# 작업지시서: task9-8 S2 회고 (KPT) + S3 Action Item 정리

## 1. 작업 목적

S2 단계(Phase 0~9)를 마치면서 **Keep, Problem, Try (KPT) 회고**를 통해 성공과 개선 과제를 명확히 하고, **S3 단계 Action Item**을 우선순위별로 정리하여 후속 단계 착수의 기초를 마련한다. 목표:

1. S2 성공 요소 (Keep) 및 근거 확보
2. 직면한 문제 (Problem) 및 영향도 분석
3. S3에서 개선할 사항 (Try) 및 실행 계획
4. S3 Action Item 우선순위 지정 및 리소스 추정
5. 팀 학습 및 조직 지식 축적

## 2. Problem (해결할 문제)

### 2.1 UI 리뷰 5회 규칙의 과도한 부담

**문제 설명**:
S2 지침에서 "UI 리뷰를 최소 5회 이상" (작업지시서 설정)으로 규정했으나,
실제 진행 시 다음과 같은 이슈 발생:

**구체적 증거**:
1. Phase 6 (Admin UI) 개발 기간
   - 예상: 2주 (구현 1주 + 리뷰 1주)
   - 실제: 3주 (구현 1주 + 리뷰 2주)
   - 원인 분석:
     * 리뷰 1회당 평균 2~3개 지적사항
     * 각 지적사항 수정 후 재리뷰 필요
     * 총 리뷰 회차: 7회 (목표 5회 초과)

2. 리뷰 참석자 (평가):
   - 리뷰 1회당 소요 시간: 30분
   - 리뷰어 수: 3명 (개발 리더, UX 담당, 고객 대표)
   - 총 비용: 30분 × 7회 × 3명 = 10.5시간 (팀 전체)

**영향도 분석**:
- **개발 기간**: +1주 (14% 증가)
- **비용**: 10.5시간/FG (Admin UI 같은 UI 중심 작업)
- **팀 만족도**: "리뷰 피로도" 피드백 (회의록)

**근본 원인**:
1. UI 리뷰 기준 모호 (명확한 수렴 조건 없음)
2. 비동기 리뷰 vs 동기 리뷰 혼용 (회의실 점유)
3. 리뷰어 관점 충돌 (미학 vs 기능성)

**지표**:
- UI 관련 FG (1, 2, 3 during Phase 6): 평균 개발 시간 +20%
- UI 리뷰 시간 / 개발 시간 비율: 25% (목표 15%)

### 2.2 평가 LLM (OpenAI) 의존성 높음

**문제 설명**:
Phase 7 (AI 품질 평가 인프라)에서 RAG faithfulness, hallucination, 
citation 평가를 OpenAI GPT-4 기반으로 구현했으나, 
외부 API 의존성이 다음 문제 야기:

**구체적 증거**:
1. API 비용
   - 평가당 토큰 비용: $0.03 (입력) + $0.06 (출력) = $0.09
   - 월별 평가 (goldenset 100개 × 4회): 400번
   - 월간 비용: $36 (개별 테스트), $360 (분기 정밀 평가)

2. 응답 시간 변동성
   - 평상시: 2~3초
   - 피크 타임 (오전 10시~12시): 5~10초
   - 장애 (2025-10월 1회): 30분 API 불가
   - 영향: 평가 파이프라인 중단

3. 로컬 대체 모델 성능
   - 로컬 LLM (Llama 2, 7B): faithfulness 정확도 85% (vs GPT-4 92%)
   - 처리 시간: 5~8초 (vs GPT-4 2~3초)
   - 트레이닝 데이터: 사유 (수정 불가능)

**영향도 분석**:
- **가용성**: 99.2% (vs 목표 99.9%)
- **비용**: 월 $36~360 (다양한 평가 시나리오에 따라)
- **폐쇄망 환경**: 불완전 지원 (로컬 모델은 정확도 ↓7%)

**근본 원인**:
1. Phase 7 설계 시 로컬 평가 모델 검토 미흡
2. 비용 vs 정확도 트레이드오프 미리 계획 안 함
3. 폐쇄망 요구사항 (S2⑦) vs 평가 정확도 간 충돌

**지표**:
- 평가 API 응답 시간 중위값: 3.2초 (변동성 σ=1.8초)
- 로컬 vs Cloud 정확도 차이: 7%포인트
- 폐쇄망 환경에서 평가 품질 degradation: 정확도 85% (목표 92%)

### 2.3 개발 문서 작성 오버헤드

**문제 설명**:
각 FG마다 작업지시서 + 검수보고서 + 보안취약점검사보고서 + AI품질평가
4개 문서 의무화로 인해 문서 작성에 과도한 시간 소비

**구체적 증거**:
1. 평균 문서 작성 시간
   - 작업지시서: 4~6시간
   - 검수보고서: 2~3시간
   - 보안취약점검사보고서: 2~3시간
   - AI품질평가: 2~3시간
   - **FG당 합계: 10~15시간**

2. 개발 대비 비율
   - 평균 FG 개발 시간: 80시간 (추정)
   - 문서 작성 시간: 12시간
   - **비율: 15%** (목표 10%)

3. 구체적 예시 (Phase 1)
   - FG1.1 개발: 80시간
   - 문서: 14시간
   - 비율: 17.5%

**영향도 분석**:
- **개발 기간**: 각 Phase +1주 (문서 작성)
- **비용**: 27 FG × 12시간 = 324시간 (~8주 FTE)
- **팀 만족도**: "문서 작성 피로도" 피드백

**근본 원인**:
1. 문서 템플릿 미흡 (처음부터 작성해야 함)
2. 자동 생성 도구 부재 (수동 작성만 가능)
3. 검수 기준 모호 (반복 수정 많음)

**지표**:
- 평균 문서 작성 시간: 12시간/FG
- 개발 시간 대비: 15% (목표 10%)
- 평균 수정 회차: 2.1회/문서 (목표 1회)

### 2.4 Agent 승인 프로세스의 오버엔지니어링

**문제 설명**:
Phase 5에서 Agent Action Plane 구현 시 모든 에이전트 행동에 대해
**인간 승인** 프로세스를 의무화했으나, 실제 운영에서:

**구체적 증거**:
1. 승인율 분포
   - Routine action (검색, 조회): 승인율 95% (거의 자동)
   - 쓰기/삭제 action: 승인율 60% (검토 필요)
   - 전체 평균: 72% (목표 70% 달성)

2. 승인 지연
   - 평균 대기 시간: 2시간 (목표 <30분)
