## 4. 주요 작업 항목

### 4-1. 평가 항목 7개 확정 및 정의

#### 4-1-1. Faithfulness (충실성)
- **정의**: 생성된 답변이 제공된 근거 문서와 모순하지 않는가?
- **측정 방법**: LLM 판정 (질문-근거-답변 3개 입력, 0~1 점수 출력)
- **계산 공식**: 평가 쿼리별 Faithfulness 점수의 평균
- **LLM 의존도**: 높음 (판정 LLM 필수)
- **폐쇄망 지원**: ✅ (로컬 LLM 사용 가능)
- **기준값**: ≥ 0.80 (S2 Phase 1~9 기본)

#### 4-1-2. Answer Relevance (답변 관련성)
- **정의**: 생성된 답변이 사용자의 질문에 의미적으로 관련이 있는가?
- **측정 방법**: embedding 기반 코사인 유사도 (질문-답변 embedding 비교)
- **계산 공식**: 평가 쿼리별 코사인 유사도의 평균
- **LLM 의존도**: 없음 (embedding 모델만 필요)
- **폐쇄망 지원**: ✅ (로컬 embedding 모델 사용)
- **기준값**: ≥ 0.75

#### 4-1-3. Context Precision (컨텍스트 정밀도)
- **정의**: 검색 결과 중 실제로 질문 답변에 기여하는 청크의 비율?
- **측정 방법**: LLM이 검색 결과 각 청크에 대해 "답변에 필요한가?" 판정 후 집계
- **계산 공식**: (필요 청크 수 / 전체 검색 청크 수)
- **LLM 의존도**: 높음
- **폐쇄망 지원**: ✅ (로컬 LLM 판정)
- **기준값**: ≥ 0.75

#### 4-1-4. Context Recall (컨텍스트 재현율)
- **정의**: 질문 답변에 필요한 모든 청크를 검색했는가?
- **측정 방법**: 골든셋 방식 (예상 근거 문서 vs 실제 검색 결과 비교)
- **계산 공식**: (검색된 필요 청크 수 / 골든셋의 필요 청크 수)
- **LLM 의존도**: 낮음 (골든셋 준비만 필요)
- **폐쇄망 지원**: ✅
- **기준값**: ≥ 0.80

#### 4-1-5. Citation-present rate (근거 포함 비율)
- **정의**: 응답의 각 문장이 검증 가능한 근거(citation with content_hash)를 가지는 비율?
- **측정 방법**: 규칙 기반 (응답 문장 수 vs citation 있는 문장 수)
- **계산 공식**: (citation 있는 문장 수 / 전체 답변 문장 수)
- **LLM 의존도**: 없음 (규칙 기반)
- **폐쇄망 지원**: ✅ (항상 가능)
- **기준값**: ≥ 0.90

#### 4-1-6. 헛소리율 (Hallucination rate)
- **정의**: 근거 없이 생성된 답변의 비율? (Citation-present rate의 반대)
- **측정 방법**: 규칙 기반 (citation 없는 문장 / 전체 문장)
- **계산 공식**: 1 - Citation-present rate, 또는 직접 계산
- **LLM 의존도**: 없음 (규칙 기반)
- **폐쇄망 지원**: ✅
- **기준값**: ≤ 0.10 (10% 이하)

#### 4-1-7. 지연·토큰·비용 (Performance Metrics)
- **정의**: 모델별/프롬프트별 실행 성능과 비용
- **측정 항목**:
  - 평균 응답 지연 (ms)
  - 입력 토큰 수 (평균)
  - 출력 토큰 수 (평균)
  - 예상 비용 (쿼리당, 달러 기준)
- **LLM 의존도**: 없음 (메타데이터 수집)
- **폐쇄망 지원**: ✅
- **기준값**: 없음 (추이 관찰, Phase 6 최적화 기준 재설정)

#### 포맷 예시

```markdown
# AI품질평가 — Phase 2 FG2.1 Citation Contract 확정

## 1. 평가 목표
Citation 5-tuple 계약의 구현이 완료되었으므로, RAG 검색 응답의 citation 정확도 및 
완전성(Citation-present rate)을 측정한다.

## 2. 평가 환경
- 평가 일시: 2026-05-15
- 실행 환경: closed-network (OpenAI API 비활성)
- 모델: vLLM (Meta-Llama-3-8B-Instruct)
- Embedding: BGE-M3-Large (로컬)
- 문서 베이스: 테스트 코퍼스 250개 문서

## 3. 평가 데이터셋
- 골든셋 규모: 50 Q&A pairs (팀 내부 작성)
- 쿼리 분포:
  - 단순 사실 조회: 20개 (40%)
  - 다중 근거 필요: 20개 (40%)
  - 모호한 질의: 10개 (20%)

## 4. 평가 결과

| 지표 | 실측값 | 기준값 | 통과 |
|------|------|------|------|
| Faithfulness | 0.78 | ≥0.75 | ✅ |
| Answer Relevance | 0.72 | ≥0.70 | ✅ |
| Citation-present | 0.88 | ≥0.90 | ❌ (재실행 필요) |
| 헛소리율 | 0.12 | ≤0.15 | ✅ |
| 평균 지연 | 245ms | - | - |

## 5. 기준값 대비 평가
- Faithfulness: 0.78 (기준 0.75) — 통과, 마진 +0.03
- Citation-present: 0.88 (기준 0.90) — **미통과**, 마진 -0.02
- 헛소리율: 0.12 (기준 ≤0.15) — 통과

**최종 판정**: 미통과 (Citation-present 기준 미충족)

## 6. 개선 항목
### 미통과 지표: Citation-present 0.88 → 0.90+
- 원인: "수정된 내용" 부분에서 citation 누락 1건 (Q17)
- 개선안: RAG 파이프라인의 "근거 문장 추출" 로직 검토, 임계값 조정
- 예상 재실행: 2026-05-20

## 7. 부록: 오류 케이스

**Query 17 (미통과)**
- 사용자 질문: "프로젝트 예산 수정 프로세스는?"
- 예상 근거: doc-123 v2, node-45
- 실제 검색: doc-123 v2, node-45 ✅ (검색됨)
- 생성 답변: "... 수정은 PM 승인 이후 ... (citation 누락)"
- 원인: 검색된 node-45에서 "수정 프로세스" 부분이 별도 span인데 
        RAG 파이프라인이 이를 통합 문장으로 생성함

## 8. 샘플 쿼리/답변

**최고 점수 (Faithfulness 0.95, Citation-present 1.0)**
- Query 5: "2026년 Q1 매출은?"
- 답변: "2026년 Q1 매출은 $500M입니다. [Citation: doc-456 v1, node-12]"

**최저 점수 (Faithfulness 0.62, Citation-present 0.75)**
- Query 18: "앞으로의 경영 방향은?"
- 원인: 질문이 모호하고, 문서 기반 근거가 부족함 → 평가 쿼리 개선 필요
```

### 4-5. Phase 0 시점의 제약 및 산출물

**Phase 0에서는 실제 평가를 수행할 수 없는 이유**:
1. 평가 대상 AI 기능이 아직 구현되지 않음 (Phase 1~9에서 구현)
2. 평가 인프라(평가 러너, 지표 계산 엔진)가 Phase 7 FG7.2에서 구축됨
3. 골든셋(평가용 Q&A)도 Phase 7 FG7.1에서 준비됨

**Phase 0 산출물 (실측값 없음)**:
- ✅ 평가 항목 정의 (7개 지표)
- ✅ 기준값 설정 (기본값, 폐쇄망 완화값)
- ✅ AI품질평가.md 보고서 템플릿
- ✅ AI 기능 포함 FG 목록
- ✅ Phase 1~9별 평가 계획
- ❌ 실제 측정값 (Phase 1 이후에야 가능)

**Phase 1부터 적용**:
- Phase 1 종료 후, 이 템플릿을 사용하여 AI품질평가.md 작성 시작
- 초기 Phase에서는 기준값을 완화하여 "측정 가능함"에 집중

## 7. 작업 지침

### 지침 7-1. 지표 정의의 구체성

- "얼마나 좋은가"를 정성적으로 설명하지 말 것
- "무엇을 측정하는가"를 구체적 계산식으로 제시
- 예: ❌ "답변이 좋다" → ✅ "코사인 유사도(질문-답변 embedding) ≥ 0.75"

### 지침 7-2. 폐쇄망 환경 고려

- 모든 지표에 대해 "폐쇄망(LLM/외부API 없음)에서 측정 가능한가?" 확인
- 불가능한 지표 없음을 목표 (모두 로컬 자원으로 측정 가능하게)

### 지침 7-3. 템플릿의 재사용성

- 각 Phase FG의 담당자가 이 템플릿을 복사하여 값만 채우면 완료되도록 설계
- 변수화 가능한 부분은 `{{변수_이름}}` 형태로 마킹

### 지침 7-4. Phase 1 핸드오프 준비

- Phase 1 FG1.1 담당자와 협력하여 "Phase 1에서 이 템플릿을 사용할 수 있는가?" 사전 검증
- 필요시 미리 조정
