# Task 0-4. Feature Group 소분 운영 규약 확정

## 1. 작업 목적

S2부터 모든 Phase를 2~3개 Feature Group(FG)으로 소분하여 독립적인 검수 게이트를 운영한다. 이 Task는 FG 단위의 독립 검수 규칙, FG별 의무 산출물, FG 완료 기준, Phase 종결 보고서 양식 등을 공식 규약으로 확정함으로써 S2 전체 운영 일관성을 확보한다.

특히 S1 회고 §4.4의 Action Item을 구체화한다.

---

## 2. 작업 범위

### 포함 범위

1. **FG 단위 독립 검수 게이트 규칙**
   - FG별 검수 흐름 (개발 → 검수 승인 → Phase 진행)
   - FG 간 의존성 관리 (선행 FG 없이 후행 FG 시작 가능한가?)
   - 블로킹 기준 (어떤 조건에서 FG 승인을 미루는가?)
   - 병렬 처리 가능성

2. **FG별 의무 산출물 목록**
   - 기존 S1 산출물: 작업지시서, 검수보고서, 보안취약점검사보고서
   - S2 신규 산출물: AI품질평가보고서 (AI 기능 포함 FG에 한함)
   - 각 산출물의 양식, 작성 책임자, 제출 기한

3. **FG 완료 기준**
   - 모든 의무 산출물이 승인되었는가?
   - 회귀 테스트 통과 여부
   - 블로킹 이슈 해결 여부
   - "FG 완료" 판정을 위한 체크리스트

4. **Phase 종결 보고서 작성 규칙**
   - 구성 요소: 계획 대비 실적, FG별 산출물 목록, 리스크·이슈 정리, 다음 Phase Action Item
   - 작성 시점, 책임자, 서명 규칙

5. **Phase별 의무 산출물 전체 목록**
   - Phase 레벨: 개발계획서, 종결 보고서
   - FG 레벨: 작업지시서, 검수보고서, 보안취약점검사보고서, AI품질평가보고서(해당 시)
   - 전체 산출물 마스터 목록 (Phase 0~9)

6. **S1 회고 §4.4 Action Item과의 연결**
   - FG 소분 방식이 S1 회고의 개선 사항을 반영했는가?
   - "검수 효율성", "배포 자동화" 등의 Action Item이 구현되었는가?

### 제외 범위

- 실제 FG 개발 진행
- 개별 FG의 상세 검수 기준 (각 FG 작업지시서에서 담당)
- Phase별 일정 계획 (별도의 프로젝트 관리 도구에서 담당)

---

## 3. 선행 조건

- Phase 0 개발계획서 검토 (§3 FG 상세, §6~11 전체 구조)
- S1 회고 검토 (§4.4 Action Item, §7 제안 사항)
- S1에서의 검수, 보안 검사, 배포 프로세스 경험
- Task 0-1, 0-2, 0-3 완료 (기초 규약이 확정된 후 FG 운영 규칙 수립)

## 4. 주요 작업 항목

### 4-1. FG 단위 독립 검수 게이트 규칙 정의

#### 4-1-1. FG 검수 흐름

```
[1] FG 개발 완료
    ├─ 모든 작업 항목 구현
    └─ 기본 테스트 통과
         ↓
[2] 검수 요청 (PR 또는 MR)
    ├─ 작업지시서 vs 구현 비교
    ├─ 기술 리드 리뷰
    └─ 보안 담당자 리뷰
         ↓
[3] 검수 보고서 작성
    ├─ 검수 기준 항목별 통과/미통과
    ├─ 이슈 등급 분류 (Critical / High / Medium / Low)
    └─ 블로킹 이슈 식별
         ↓
[4] 보안취약점검사 수행
    ├─ 일반 OWASP Top 10
    ├─ LLM Top 10 (Phase 5 이후)
    └─ 보고서 작성
         ↓
[5] AI품질평가 (AI 기능 포함 FG에 한함)
    ├─ 7개 지표 측정
    ├─ 기준값 대비 판정
    └─ 개선 항목 정의
         ↓
[6] 의무 산출물 모두 승인
    ├─ Critical 이슈 0건 해결
    ├─ High 이슈 모두 해결 (또는 명시적 승인)
    └─ 회귀 테스트 통과
         ↓
[7] FG 완료 선언
    └─ Phase 다음 FG 또는 Phase 진행 가능
```

#### 4-1-2. FG 간 의존성 관리

**의존성 정의 규칙**:
- FG는 명시적으로 "다른 FG를 선행 조건으로" 지정할 수 있다
- 선행 조건이 없으면 병렬 처리 가능 (동시 개발, 병렬 검수)
- 선행 FG 완료 후 후행 FG 착수

**Phase 0 예시**:
- FG0.1 (원칙 및 규약) — 선행 조건 없음 (동시 시작 가능)
- FG0.2 (기술 부채) — FG0.1 완료 후 착수 (설계 기준이 확정된 후)
  - FG0.2 Critical (네비게이션) ↔ High (unwrapEnvelope 등) 병렬 가능

**Phase 1 예시**:
- FG1.1 (LLM Provider) — 선행 없음
- FG1.2 (Embedding) — 선행 없음 (독립적)
- FG1.3 (Prompt Registry) — FG1.1, FG1.2 완료 후 (LLM/Embedding 인터페이스 기반)

#### 4-1-3. 블로킹 기준

**FG 승인을 미루는 경우** (즉, "FG 완료 불가"):

| 조건 | 미루기 규칙 |
|------|-----------|
| Critical 보안 이슈 | 반드시 해결하고 승인 (미루기 불가) |
| High 기능 이슈 | 해결 또는 명시적 Risk Acceptance 서명 필요 |
| AI 기능 기준값 미달 | AI품질평가 미통과 → Phase 종결 이전에 개선 필수 (다음 Phase로 진행 가능하나, Phase 종결 불가) |
| 회귀 테스트 실패 | 모든 회귀 테스트 통과 필수 |
| 의무 산출물 누락 | 모두 제출 필수 |

**미루기 가능한 경우** (즉, "FG 완료, 다음 단계 진행 가능"):
- Medium/Low 기능 이슈 → 다음 Phase에서 개선
- AI 기능 초기 Phase 완화 기준 (Phase 1~3)

### 4-2. FG별 의무 산출물 목록 확정

#### 표: FG별 의무 산출물

| 산출물 | 형식 | 작성자 | 제출 기한 | 용도 |
|--------|------|--------|---------|------|
| **작업지시서** | Markdown | 기술 리드 | FG 착수 전 | FG 범위/완료 기준 정의 |
| **검수보고서** | Markdown | 검수팀 (기술 리드 외) | FG 완료 후 | 구현 검증 |
| **보안취약점검사보고서** | Markdown | 보안 담당자 | FG 완료 후 | 보안 검증 |
| **AI품질평가보고서** | Markdown | AI/ML 담당자 | FG 완료 후 (AI 기능 FG에 한함) | 품질 검증 |

#### 각 산출물의 상세 규칙

**작업지시서 (Task N-M.md)**
- 양식: Task 0-1~0-4 참조
- 섹션: 목적, 범위, 선행 조건, 주요 항목, 산출물, 완료 기준, 지침
- 줄 수: 100~180 줄 (너무 상세하지 않으면서 명확함)
- 승인: 팀 회의에서 구두 합의로 충분 (문서 서명 불필요)

**검수보고서**
- 양식: `검수보고서_FG[X]-[Y].md` (예: `검수보고서_FG2-1.md`)
- 섹션: 검수 일시, 검수자, 작업지시서 기준 항목별 검수 결과, 이슈 목록(등급별), 개선 사항, 최종 판정
- 이슈 분류: Critical (차단) / High (해결 또는 Risk 승인) / Medium (다음 Phase) / Low (문서화)
- 최종 판정: "승인" 또는 "조건부 승인" (이슈 목록 명시)
- 승인: 검수팀 및 PO 서명

**보안취약점검사보고서**
- 양식: `보안취약점검사보고서_FG[X]-[Y].md`
- 섹션: 검사 일시, 검사자, OWASP 항목별 검사 결과, 발견 취약점 목록, 개선안, 최종 판정
- 검사 범위:
  - Phase 0~4: OWASP Top 10
  - Phase 5~9: OWASP Top 10 + OWASP Top 10 for LLM Applications
- 최종 판정: "승인" 또는 "조건부 승인"
- 승인: 보안 담당자 및 CTO 서명

**AI품질평가보고서** (AI 기능 포함 FG에 한함)
- 양식: Task 0-3의 템플릿 참조
- 섹션: 평가 목표, 평가 환경, 평가 데이터셋, 결과(7개 지표), 기준값 대비, 개선 항목, 오류 케이스
- 평가 범위: Task 0-3의 FG 목록 참조 (Phase 1 FG1.1~1.3 중 일부만)
- 최종 판정: "승인" (기준값 통과) 또는 "조건부 승인" (개선 계획 포함)
- 승인: AI/ML 담당자 및 기술 리드 서명

### 4-3. FG 완료 기준 정의

FG가 "완료"로 판정되는 조건 (모두 만족 필요):

```
☑ 1. 의무 산출물 모두 제출됨
     └─ 작업지시서, 검수보고서, 보안취약점검사보고서
     └─ AI품질평가보고서 (AI 기능 FG에 한함)

☑ 2. 검수보고서에서 "승인" 또는 "조건부 승인" 판정
     └─ Critical 이슈 0건
     └─ High 이슈는 해결되거나 명시적 Risk Acceptance 서명됨

☑ 3. 보안취약점검사보고서에서 "승인" 또는 "조건부 승인" 판정
     └─ Critical 취약점 0건
     └─ High 취약점은 패치되거나 mitigation 계획 수립됨

☑ 4. AI품질평가보고서 승인 (해당 시)
     └─ 기준값 통과 또는 개선 계획 명시

☑ 5. 회귀 테스트 100% 통과
     └─ 기존 기능 레그레션 없음
     └─ FG의 새 기능 테스트 통과

☑ 6. 의존하는 선행 FG 모두 완료됨
     └─ 선행 FG가 없으면 이 항목 생략

☑ 7. PO 또는 스티어링 위원회 최종 승인
     └─ 모든 산출물 검토 완료
```

**예외 규칙**:
- AI품질평가 초기 Phase(1~3)는 완화된 기준값 적용 가능 (Task 0-3 규약 참조)
- Medium/Low 이슈가 있어도 조건부 승인 가능 (단, 다음 Phase Action Item으로 등재)
