# Task 0-10. [High] UI 5회 리뷰의 여정 기반 체크리스트 표준화

## 1. 작업 목적

**"여정 기반"(journey-based) 사용자 시나리오**에 따른 UI 리뷰 체크리스트를 정의하고, 이를 **Phase 0~9 전체**에서 **재사용 가능한 표준 템플릿**으로 제공한다.

현재 상태:
- UI 5회 리뷰는 Phase 0 규칙이나, 구체적 항목과 시나리오 정의 부재
- 각 개발자가 임의로 리뷰 수행 가능
- 리뷰 기록이 불일관적
- Phase 1 이후 일관성 유지 불가

이 작업은 **S2 절대 규칙**("UI 리뷰를 최소 5회 이상 하여 개선할 것")과 **CLAUDE.md UI 규칙**("데스크톱 어플리케이션, 웹 애플리케이션을 모두 호환 가능하게")을 **구체화**한다.

---

## 2. 작업 범위

### 포함 범위

1. 여정 기반 사용자 시나리오 정의
   - 시나리오 1: 일반 사용자 흐름 (로그인 → 문서 목록 → 문서 열기 → 검색 → 로그아웃)
   - 시나리오 2: 관리자 흐름 (로그인 → Admin 전환 → 대시보드 → 사용자 관리 → 설정 → User UI 복귀)
   - 시나리오 3: RAG 질의 흐름 (검색 입력 → 질의 → citation 확인 → 원문 참조)

2. UI 리뷰 항목 카테고리 정의
   - 접근성 (a11y): 키보드 네비게이션, ARIA 라벨, 포커스 관리
   - 반응형 (responsive): 모바일(320px), 태블릿(768px), 데스크톱(1200px+)
   - 오류 메시지: 명확성, 작업 지침 포함, 복구 경로 안내
   - 네비게이션 흐름: 경로 명확성, 뒤로가기 동작, 메뉴 위계
   - 데이터 로딩 상태: 로딩 표시, 로딩 중 인터랙션 제어, 타임아웃 처리
   - 빈 상태(empty state): 초기 상태 가이드, 다음 단계 제안

3. 5회 리뷰 기록 양식 정의
   - 회차, 일시, 리뷰어, 발견 사항, 개선 조치, 상태

4. Desktop + Web 호환성 확인 항목
   - 데스크톱 앱(Electron 등) UI 테스트 항목
   - 웹 브라우저(Chrome, Safari, Firefox) 호환성 항목

5. Markdown 템플릿 파일 작성
   - Phase별로 복제 가능한 템플릿
   - 리뷰 항목 자동 체크리스트

### 제외 범위

- UI 디자인 가이드라인 정의 (별도 design-system.md 참고)
- A/B 테스팅 항목 (후속 Phase)
- 성능 최적화 항목 (별도 performance-review.md 참고)

## 4. 주요 작업 항목

### 4-1. 여정 기반 시나리오 정의

**문서:** `/backend/docs/UI-REVIEW-SCENARIOS.md` (신규) 또는 프로젝트 문서

**시나리오 상세:**

```markdown
# UI 리뷰 여정 기반 시나리오

## 시나리오 1: 일반 사용자 기본 흐름

**목적:** 일반 사용자의 일상적 사용 경로를 검증

**흐름:**
1. 로그인 페이지 접근 (`/login`)
   - 이메일/비밀번호 입력 필드 표시
   - 로그인 버튼 활성 여부
   
2. 로그인 완료 → 문서 목록 페이지 (`/app/documents`)
   - 사용자 메뉴(헤더 우측) 표시
   - 문서 리스트 로딩 상태 표시
   - 빈 상태 메시지(문서 없을 시)
   
3. 문서 열기 (`/app/documents/{id}`)
   - 문서 제목, 내용 표시
   - 로딩 상태 표시
   
4. 검색 기능 (`/app/search`)
   - 검색 입력창 포커스 자동 설정
   - 검색 결과 표시
   - 빈 검색 결과 메시지
   
5. 로그아웃
   - 헤더 메뉴 → "로그아웃" 클릭
   - 로그인 페이지로 리다이렉트

**예상 문제:**
- 로그인 페이지에서 엔터키로 제출 불가
- 사용자 메뉴 드롭다운이 모바일에서 잘려감
- 로딩 중 다중 클릭 방지 미흡

---

## 시나리오 2: 관리자 설정 흐름

**목적:** 관리자만 접근 가능한 경로를 검증

**흐름:**
1. 로그인 (admin 계정) → User UI 진입 (`/`)
2. 헤더 메뉴 → "관리자 설정" 클릭 (`/admin`)
   - "관리자 모드" 배지 표시
   - Admin UI 헤더 로드
   
3. Admin 대시보드 (`/admin`)
   - 주요 통계 표시
   - 빠른 링크 표시
   
4. 사용자 관리 (`/admin/users`)
   - 사용자 리스트 표시
   - 필터/검색 기능
   - 수정/삭제 버튼
   
5. 역할 관리 (`/admin/roles`)
   - 역할 리스트 표시
   - 권한 설정
   
6. 설정 (`/admin/settings`)
   - 시스템 설정 폼
   
7. User UI로 복귀 (`/`)
   - Admin 헤더 → "← User UI로 돌아가기" 클릭
   - User UI로 정상 전환

**예상 문제:**
- Admin 메뉴가 일반 사용자에게 표시됨
- Admin UI에서 User UI로 복귀 링크 누락
- Admin 페이지 접근 제어 미흡

---

## 시나리오 3: RAG 질의 흐름

**목적:** AI 기능과 citation 표시를 검증

**흐름:**
1. 검색/질의 페이지 접근
2. 질의 입력 및 제출
   - 입력 필드 포커스
   - 제출 버튼 활성/비활성
   
3. 응답 로딩
   - "응답 생성 중..." 표시
   - 타임아웃 처리
   
4. 응답 표시
   - 텍스트 응답
   - **Citation 표시** (중요!)
     - 각 citation마다 링크 표시
     - [Source 1] [Source 2] 형태로 표시
   
5. Citation 클릭 → 원문 참조
   - 원문 문서 열기
   - Highlight 표시 (citation 해당 부분)

**예상 문제:**
- Citation이 응답에 표시되지 않음
- Citation 링크가 깨짐
- 원문 참조 시 해당 섹션으로 스크롤 미흡
- RAG 응답 로딩이 길어질 경우 타임아웃 처리 부재
```

### 4-2. UI 리뷰 항목 카테고리 정의

**문서:** `/backend/docs/UI-REVIEW-CATEGORIES.md` (신규)

```markdown
# UI 리뷰 항목 카테고리

## 카테고리 1: 접근성 (Accessibility, a11y)

### 항목
- [ ] 키보드 네비게이션 (Tab, Shift+Tab, Enter, Escape)
- [ ] ARIA 라벨 (aria-label, aria-describedby)
- [ ] 포커스 관리 (포커스 표시, 순환 참조)
- [ ] 색상 대비 (WCAG AA 기준, 4.5:1 이상)
- [ ] 스크린 리더 호환성 (VoiceOver, NVDA 테스트)

### 체크 방법
- 키보드만으로 모든 기능 사용 가능한가?
- 포커스 표시가 명확한가?
- 마우스/터치 없이 접근 가능한가?

---

## 카테고리 2: 반응형 (Responsive Design)

### 항목
- [ ] 모바일 (320px): 기본 레이아웃, 메뉴 동작
- [ ] 타블릿 (768px): 2단 레이아웃 적용
- [ ] 데스크톱 (1200px+): 전체 레이아웃
- [ ] 폰트 크기 (최소 16px, 줌 가능)
- [ ] 버튼/입력 크기 (최소 44px × 44px)
- [ ] 이미지 반응형 (srcset, picture 태그)

### 테스트 방법
- 브라우저 DevTools 반응형 모드 테스트
- 실제 기기 테스트 (아이폰, 안드로이드)
- 가로/세로 모드 전환 테스트

---

## 카테고리 3: 오류 메시지

### 항목
- [ ] 명확한 오류 설명 (무엇이 잘못되었는가)
- [ ] 작업 지침 포함 (어떻게 해결하는가)
- [ ] 복구 경로 안내 (다음 단계)
- [ ] 시각적 강조 (빨강 배경, 아이콘)
- [ ] 폼 검증 (실시간 vs. 제출 시)

### 예시
- ❌ 나쁜 예: "오류 발생"
- ✓ 좋은 예: "이메일이 이미 등록되었습니다. [비밀번호 재설정] 링크를 사용하세요."

---

## 카테고리 4: 네비게이션 흐름

### 항목
- [ ] 경로 명확성 (현재 위치 표시, breadcrumb)
- [ ] 뒤로가기 동작 (browser back 버튼 호환)
- [ ] 메뉴 위계 (주 메뉴, 서브 메뉴, 현재 활성 표시)
- [ ] 링크 표시 (밑줄, 다른 색상)
- [ ] 진행 상황 표시 (Step 1/3 형태)

---

## 카테고리 5: 데이터 로딩 상태

### 항목
- [ ] 로딩 표시 (spinner, progress bar)
- [ ] 로딩 중 인터랙션 제어 (버튼 비활성화, 다중 제출 방지)
- [ ] 로딩 중 취소 기능 (선택)
- [ ] 타임아웃 처리 (30초 이상 → 재시도 제안)
- [ ] 에러 상태 표시 (로딩 실패 메시지)
- [ ] 스켈레톤 화면 (선택, 더 좋은 UX)

---

## 카테고리 6: 빈 상태 (Empty State)

### 항목
- [ ] 초기 상태 메시지 (무엇을 해야 하나)
- [ ] 다음 단계 제안 (예: "새 문서 생성" 버튼)
- [ ] 시각적 요소 (아이콘, 삽화)
- [ ] 검색 후 결과 없음 (명확한 "결과 없음" 메시지)
```

## 7. 작업 지침

### 지침 7-1. 시나리오 선택 기준

각 작업에서 모든 3가지 시나리오를 검증하되:
- 기능 범위에 맞는 시나리오만 깊게 검증
- 예: RAG 기능 미실장 Phase에서는 시나리오 3 부분 검증 생략 가능

### 지침 7-2. 리뷰어 구성

5회 리뷰에 참여할 리뷰어:
- 최소 2명 이상 (개발자 + PM/UX)
- 회차마다 다른 관점 강조

### 지침 7-3. 우선순위 판단

발견 사항의 우선순위:
- **Critical:** 기능 작동 불가 (로그아웃 실패 등)
- **High:** 중요 사용성 (로딩 표시 부재, 포커스 표시 부재)
- **Mid:** 부분 호환성 (특정 브라우저에서만 오류)
- **Low:** 사소한 개선 (버튼 간격 조정)

### 지침 7-4. 타임박싱

5회 리뷰 일정 예시 (1주일 스프린트 기준):
- 1회: 월 오전 (초안, 1시간)
- 2회: 월 오후 (a11y, 1시간)
- 3회: 화 오전 (responsive, 1시간)
- 4회: 수 오전 (사용성, 1시간)
- 5회: 목 오전 (최종, 1시간)
- 금: 개선 사항 반영
