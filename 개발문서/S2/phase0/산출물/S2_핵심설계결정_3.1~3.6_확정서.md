## 결정 3.1 — 에이전트 인증: OAuth client-credentials + access_context

### 정의

챗봇 서버 ↔ Mimir 서버 간 인증은 OAuth 2.0 client-credentials 플로우를 사용한다.  
**Mimir는 access_context를 그대로 신뢰하지 않는다** — `user_id`로 DB를 조회하여 실제 권한을 재검증한다.

### 인증 흐름

```
[1] 챗봇 서버
    POST /oauth/token
    {grant_type: client_credentials, client_id, client_secret}
    ↓
[2] Mimir
    유효한 API Key인가? → 토큰 발급 (TTL: 1시간)
    ↓
[3] 챗봇 서버
    POST /api/v1/search
    Headers: Authorization: Bearer <token>
    Body: {query, access_context: {user_id, org_id, permissions}}
    ↓
[4] Mimir
    ① 토큰 유효성 확인 + delegate:search 권한 확인
    ② access_context.user_id → DB 조회 → 실제 ACL 적용
    ③ 권한 불일치 시 403 반환 + 감사 로그 기록
```

### 핵심 설계 결정

| 항목 | 결정 | 이유 |
|------|------|------|
| 인증 방식 | OAuth 2.0 client-credentials | 서버-투-서버 인증 표준. API Key만으로는 대행 권한 제어 어려움 |
| access_context 신뢰 | 재검증 필수 (trust but verify) | 챗봇이 잘못된 user_id를 전달해도 Mimir가 방어 |
| Scope 표현 | `delegate:search`, `delegate:write` | 최소 권한 원칙. 에이전트별 허용 행동 명시적 제한 |
| 토큰 TTL | 1시간 (재발급 가능) | 탈취 시 피해 최소화 |

### 대안 비교

| 옵션 | 장점 | 단점 | 선택 |
|------|------|------|------|
| OAuth client-credentials | 표준, 확장성 | 구현 복잡도 ↑ | ✅ |
| API Key만 | 간단 | 대행 권한 제어 불가 | ✗ |
| mTLS | 보안 매우 강함 | 인증서 관리 복잡 | S3 고려 |

### Phase 구현 기준

- **Phase 4 FG4.1**: MCP Server OAuth 핸드셰이크 구현
- **Phase 4 FG4.2**: API Key에 `delegate:search`, `delegate:write` scope 추가
- **검수 기준**: access_context 재검증 로직이 모든 쓰기·읽기 API에 적용되었는가?

### 폐쇄망 동작 (원칙 ⑦)

OAuth 토큰 발급·검증은 Mimir 내부에서 처리한다. 외부 IdP 의존 없음. 폐쇄망에서 완전 동작.

## 결정 3.2 — Scope Profile + FilterExpression + $ctx.* 변수 모델

### 정의

외부 소비자의 scope 어휘(`team`, `org`, `confidential` 등)는 Mimir 코드에 등장하지 않는다.  
관리자가 설정하는 **Scope Profile**이 scope name ↔ ACL filter 매핑을 담당한다.

### 데이터 모델

```
ScopeProfile {
  id: UUID
  name: "기본 챗봇용"
  description: string
  ScopeDefinition[] {
    scope_name: "team"           ← 외부 소비자 요청 어휘
    acl_filter: FilterExpression ← 관리자 정의 ACL 규칙
  }
}
```

### FilterExpression 규칙 (S2 수위)

```
허용 필드 (whitelist):
  organization_id, team_id, visibility, classification, owner_id
  + DocumentType별 metadata 필드 (Phase 1에서 확장)

허용 연산자:
  eq, neq, in, not_in, starts_with, contains, gte, lte (숫자)

구조: AND, OR 조합 (최대 깊이 3단계)

$ctx.* 바인딩: access_context 필드명만 참조 가능
  예: $ctx.organization_id → access_context.organization_id로 치환
```

**예시:**
```json
{
  "scope_name": "team",
  "acl_filter": {
    "and": [
      {"field": "organization_id", "op": "eq", "value": "$ctx.organization_id"},
      {"field": "team_id", "op": "eq", "value": "$ctx.team_id"}
    ]
  }
}
```

### 핵심 설계 결정

| 항목 | 결정 | 이유 |
|------|------|------|
| Scope 관리 계층 | Scope Profile (관리자 설정) | 코드 수정 없이 정책 변경 가능 |
| FilterExpression 수위 | 화이트리스트 필드 + 제한 연산자 | 주입 공격 방어. SQL-like 표현식 금지 |
| $ctx.* 치환 | 선언된 필드명만 | 코드 실행 불가, 주입 방어 |
| S3 확장 | 임의 SQL-like 표현식 | S2에서는 제외. 보안 검토 후 S3에서 확장 |

### 대안 비교

| 옵션 | 장점 | 단점 | 선택 |
|------|------|------|------|
| Scope Profile + FilterExpression | 유연, 코드 수정 불필요 | 구현 복잡 | ✅ |
| 하드코딩된 scope 문자열 | 간단 | 정책 변경 시 배포 필요 | ✗ |
| 임의 SQL 표현식 | 완전 유연 | 주입 공격 위험 | S3 |

### Phase 구현 기준

- **Phase 4 FG4.2**: Scope Profile CRUD API 구현
- **Phase 6 FG6.2**: Admin UI에서 Scope Profile 편집 가능
- **검수 기준**: 코드에 `if scope == "..."` 형태의 하드코딩 scope 문자열 0건

### 폐쇄망 동작 (원칙 ⑦)

Scope Profile은 Mimir DB에 저장. 외부 의존 없음. 폐쇄망에서 완전 동작.

## 결정 3.4 — Citation Contract: 5-tuple

### 정의

검색 응답의 citation을 아래 5-tuple로 고정한다:

```
Citation {
  document_id: UUID        # 어떤 문서인가
  version_id: UUID         # 몇 번째 개정본인가 (개정 후에도 유효성 검증 가능)
  node_id: UUID            # 문서 내 어떤 노드인가
  span_offset: int | null  # [선택] 노드 내 위치 (100단어 미만 청크에서는 생략)
  content_hash: string     # SHA-256(노드 내용) — 내용 변조 여부 검증
}
```

### 각 필드의 검증 가능성

| 필드 | 검증 가능 내용 |
|------|-------------|
| `document_id + version_id` | 개정 후에도 어떤 버전인지 명확. 현재 버전과 비교 가능 |
| `content_hash` | 청크 내용이 원문과 동일한지 즉시 검증 (`SHA-256` 비교) |
| `span_offset` | 긴 노드에서 정확한 위치 지정. 단락 단위 청크에서는 생략 가능 |

### 핵심 설계 결정

| 항목 | 결정 | 이유 |
|------|------|------|
| content_hash 알고리즘 | SHA-256 | 충돌 위험 없음. Blake3는 S3에서 성능 최적화 시 검토 |
| span_offset 생략 규칙 | 100단어 미만 청크: 생략 가능 / 이상: 필수 | 긴 노드에서 부분 참조 정확도 확보 |
| 하위 호환 마이그레이션 | S1 RAG 응답 포맷 → 5-tuple로 점진 전환 | Phase 2에서 수행 |

### 대안 비교

| 옵션 | 장점 | 단점 | 선택 |
|------|------|------|------|
| 5-tuple (document+version+node+span+hash) | 검증 가능, 안정적 | 응답 크기 ↑ | ✅ |
| document_id + node_id만 | 간단 | 개정 후 유효성 확인 불가 | ✗ |
| URL 기반 참조 | 직관적 | URL 변경 시 citation 무효 | ✗ |

### Phase 구현 기준

- **Phase 2 FG2.1**: 검색 응답에 5-tuple 필수 포함, span → 원문 역참조 API
- **검수 기준**: 모든 검색 응답에 `{document_id, version_id, node_id, content_hash}` 필드가 있는가?

### 폐쇄망 동작 (원칙 ⑦)

Citation 생성·검증은 Mimir 내부(DB + SHA-256). 외부 의존 없음. 폐쇄망 완전 동작.

## 결정 3.5 — 에이전트 Principal 모델

### 정의

에이전트는 사용자와 별개의 **독립 주체(Principal)**로 모델링한다.

```
Principal {
  type: "user" | "agent" | "system"
  id: UUID
  acting_on_behalf_of: UUID | null   # agent 또는 system인 경우만
}
```

### 별개 주체로 모델링하는 이유

1. **추적 가능성**: 에이전트별 행동 이력을 명확히 기록
2. **킬스위치**: 특정 에이전트만 즉시 차단 가능
3. **위임 명시화**: 에이전트가 사용자를 대행하는 관계를 구조로 표현

### Delegation 정의

```
에이전트 권한 = 에이전트 자신의 API Key scope ∩ 대행 사용자의 권한
               (교집합 — 항상 더 제한적)
```

- `acting_on_behalf_of: <user_id>` → 감사 로그에 필수 기록
- 에이전트가 사용자보다 넓은 권한을 가질 수 없다

### 감사 로그 형식

```json
{
  "timestamp": "2026-04-16T12:00:00Z",
  "actor_type": "agent",
  "actor_id": "uuid-agent-1",
  "acting_on_behalf_of": "uuid-user-1",
  "action": "create_draft",
  "resource_id": "uuid-document-1",
  "result": "success",
  "reason": null
}
```

### Kill Switch 정의

| 항목 | 규칙 |
|------|------|
| **트리거** | 관리자 수동 또는 자동 규칙 (반려율 >80%, 제안 >100/시간) |
| **동작** | 특정 에이전트의 쓰기(draft, 전이 제안) 즉시 차단 |
| **차단 지연** | < 5초 (설정 반영 + 캐시 무효화) |
| **해제** | 관리자 수동만 가능 (자동 해제 없음) |

### Phase 구현 기준

- **Phase 4 FG4.2**: Agent principal type 추가, Delegation ACL 적용
- **Phase 5 FG5.3**: 킬스위치 API, 에이전트 활동 감사 뷰
- **Phase 6 FG6.2**: Admin UI 킬스위치 조작, 에이전트 활동 대시보드
- **검수 기준**: 에이전트의 모든 쓰기가 감사 로그에 `actor_type=agent` 기록되는가?

### 폐쇄망 동작 (원칙 ⑦)

Principal 모델·감사 로그·킬스위치는 Mimir DB 기반. 외부 의존 없음. 폐쇄망 완전 동작.

## 결정 3.6 — 평가용 판정 전략

### 정의

폐쇄망에서 "판정 LLM 품질 = 평가 지표 품질"이 된다.  
S2에서는 **LLM 의존도를 낮춘 판정부터 시작**하고 점진 확장한다.

### LLM 의존도별 판정 전략

| 판정 유형 | LLM 필요 | 측정 방법 | S2 도입 시점 |
|----------|---------|---------|-----------|
| Citation-present check | 불필요 | 규칙 기반 (citation 있는 문장 / 전체 문장) | Phase 2 이후 |
| 근거 문장 ↔ 원문 매칭 | 불필요 | content_hash 비교 / 코사인 유사도 | Phase 2 이후 |
| Answer relevance | embedding만 | 질문-답변 코사인 유사도 | Phase 1 이후 |
| Faithfulness | LLM 판정 | 질문-근거-답변 3입력, 0~1 점수 | Phase 7 FG7.2 |
| Context precision | LLM 판정 | 검색 청크별 "필요한가?" 판정 | Phase 7 FG7.2 |
| Context recall | 골든셋 | 골든셋 vs 검색 결과 비교 | Phase 7 FG7.2 |

### 핵심 설계 결정

| 항목 | 결정 | 이유 |
|------|------|------|
| 판정 LLM | Phase 1 모델 추상화 계층 재사용 | 폐쇄망에서 자동으로 로컬 모델로 전환 |
| 폐쇄망 판정 | 규칙 기반 4개 지표 필수 가동 | LLM 없이도 최소한의 품질 측정 가능 |
| Faithfulness 기준값 | ≥0.80 (기본), ≥0.75 (폐쇄망) | Task 0-3 AI품질평가 규약과 일치 |

### Phase 구현 기준

- **Phase 7 FG7.2**: 6개 지표 러너 구현
- **Phase 7 FG7.3**: CI 게이트 — Faithfulness < 0.80 시 빌드 실패
- **검수 기준**: 폐쇄망 환경에서도 citation-present, answer relevance, 근거 매칭 3개 지표 측정 가능한가?

### 폐쇄망 동작 (원칙 ⑦)

| 지표 | 폐쇄망 가능 여부 | 비고 |
|------|--------------|------|
| Citation-present | ✅ 항상 가능 | 규칙 기반 |
| 근거 매칭 | ✅ 항상 가능 | content_hash |
| Answer relevance | ✅ 로컬 embedding | BGE-M3, E5 |
| Faithfulness | ✅ 로컬 LLM | vLLM, Ollama |
| Context precision/recall | ✅ 로컬 LLM | vLLM, Ollama |
