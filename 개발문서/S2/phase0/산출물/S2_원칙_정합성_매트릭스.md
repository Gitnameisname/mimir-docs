## 2. 원칙 간 관계 매트릭스

| S2 원칙 | S1 ① API-first | S1 ② Dual UI | S1 ③ Type=행동 | S1 ④ 권한=모든 계층 |
|--------|---------------|-------------|--------------|-----------------|
| ⑤ AI 1급 | **강화** — AI 소비자도 동일 API 계약 | **중립** — AI는 UI 없음, API만 소비 | **중립** — Type 로직은 서비스 레이어 유지 | **강화** — 감사 로그에 `actor_type` 추가 |
| ⑥ 범위 하드코딩 금지 | **강화** — ACL 정책도 API로 관리, 코드 수정 불필요 | **중립** — Admin UI에서 Scope Profile 설정 (② 유지) | **동형 확장** — ③의 "Type 하드코딩 금지" 논리를 scope로 확장 | **강화** — 모든 계층에서 Scope Profile 기반 ACL 적용 |
| ⑦ 폐쇄망 동등성 | **강화** — 폴백 경로에서도 API 계약 유지 | **중립** — UI는 degrade 상태를 표시하면 충분 | **중립** — Type별 동작은 폐쇄망에서도 동작 | **강화** — FTS 폴백에서도 동일 ACL 적용 필수 |

**범례:**
- **강화**: S1 원칙의 적용 범위를 넓히거나 요건을 높임
- **중립**: 독립적인 관심사, 충돌 없음
- **동형 확장**: S1 원칙과 동일한 논리를 다른 도메인에 적용

## 3. 모순 검토

S1과 S2 원칙 간 모순 가능성을 검토하고, 해소 방안을 명시한다.

### 3-1. ⑤ AI 1급 vs ② Dual UI

**잠재적 모순**: AI는 UI 없이 API만 소비하는데, Dual UI(User UI / Admin UI)는 사람을 전제한다.  
**해소**: 모순 없음. ②는 사람 소비자의 UI 경험을 정의하고, ⑤는 AI 소비자의 API 계약을 정의한다. 대상이 다르다. AI는 User UI / Admin UI를 우회하여 API를 직접 호출한다.

### 3-2. ⑥ 범위 하드코딩 금지 vs ③ Type = 행동

**잠재적 모순**: ③에서 DocumentType별 행동이 정의되는데, ⑥에서 "코드에 특정 문자열 금지"라고 하면 DocumentType도 금지가 아닌가?  
**해소**: ③의 DocumentType 분기는 *서비스 레이어에서만*, 설정(config)으로 정의된 타입에 따라 동작한다. ③은 이미 "하드코딩 금지" 원칙을 내포한다. ⑥은 이를 scope 어휘로 확장한 것이다. ③과 ⑥은 동일한 철학("설정 기반, 코드 수정 불필요")을 서로 다른 도메인에 적용한다.

### 3-3. ⑦ 폐쇄망 동등성 vs ④ 권한은 모든 계층에 적용

**잠재적 모순**: 폐쇄망에서 FTS 폴백 시 벡터 검색 ACL이 적용되지 않을 수 있다.  
**해소**: FTS 폴백에서도 ④의 ACL이 동일하게 적용되어야 한다. 이는 ⑦의 구현 기준에 명시되어 있다. 폴백 경로도 ACL을 회피하는 경로가 되어서는 안 된다.

## 5. 원칙 우선순위 규칙

원칙 간 충돌이 발생할 경우 다음 순서로 우선한다:

```
S1 원칙 ①~④ > S2 원칙 ⑤~⑦
```

**근거**: S2 원칙은 S1 원칙을 무효화하지 않는다. S2 원칙은 S1 원칙 위에서 AI 소비자·폐쇄망·접근 범위 설정이라는 추가 요건을 강화하는 것이다. S1 원칙이 S2 원칙보다 더 기초적인 설계 기반이다.

**예시:**
- ⑤(AI 1급)가 ①(API-first)과 충돌하는 것처럼 보이는 경우 → ①(모든 기능은 API로 먼저 정의)이 우선. ⑤는 그 API를 AI가 소비할 수 있도록 추가 요건을 부여할 뿐이다.
- ⑦(폐쇄망)이 ④(권한 모든 계층)와 충돌하는 경우 → ④(ACL 적용)이 우선. ⑦의 폴백 경로에서도 ACL을 반드시 유지해야 한다.
