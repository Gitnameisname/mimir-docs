# Feature Group (FG) 운영 규약

**확정일**: 2026-04-16
**근거**: Season 2 개발계획.md §4 / S1 회고 §4.4
**상태**: 확정 — Phase 0부터 S2 전체에 적용

---

## 1. 개요

S2의 모든 Phase는 2~3개 **Feature Group(FG)**으로 분할된다.  
각 FG는 독립적인 검수 게이트를 가지며, FG가 완료되어야 다음 FG 또는 Phase로 진행한다.

---

## 2. FG 검수 흐름

```
[1] FG 개발 완료
    ├─ 모든 작업지시서 항목 구현
    └─ 기본 단위 테스트 통과

[2] 검수 요청 (PR 또는 MR)
    ├─ 작업지시서 vs 구현 체크리스트 비교
    ├─ 기술 리드 코드 리뷰
    └─ 보안 담당자 리뷰 (병렬 가능)

[3] 검수보고서 작성
    ├─ 검수 기준 항목별 통과/미통과
    ├─ 이슈 등급 분류 (Critical / High / Medium / Low)
    └─ 블로킹 이슈 식별

[4] 보안취약점검사 수행 (병렬 가능)
    ├─ Phase 0~4: OWASP Top 10
    ├─ Phase 5~9: OWASP Top 10 + OWASP Top 10 for LLM Applications
    └─ 보안취약점검사보고서 작성

[5] AI품질평가 (AI 기능 포함 FG에 한함)
    ├─ 측정 가능한 지표 측정 (AI품질평가_규약.md 기준)
    ├─ 기준값 대비 판정
    └─ AI품질평가보고서 작성

[6] 의무 산출물 모두 승인
    ├─ Critical 이슈 0건
    ├─ High 이슈 해결 또는 Risk Acceptance 서명
    └─ 회귀 테스트 100% 통과

[7] FG 완료 선언
    └─ 다음 FG 또는 Phase 진행 가능
```

## 3. FG 간 의존성 관리

### 의존성 정의 규칙

- FG는 명시적으로 "선행 FG 완료"를 조건으로 지정할 수 있다
- 선행 조건이 없으면 **병렬 처리 가능** (동시 개발, 병렬 검수)
- 선행 FG 완료 전 후행 FG 코드 착수 불가

### Phase별 의존성 예시

**Phase 0:**
- FG0.1 (원칙·규약) — 선행 없음, 즉시 착수
- FG0.2 (기술 부채) — FG0.1 완료 후 착수 (설계 기준이 확정된 후)
  - FG0.2 Critical 항목 ↔ High 항목: 병렬 가능

**Phase 1:**
- FG1.1 (LLM Provider) — 선행 없음
- FG1.2 (Embedding) — 선행 없음 (독립적)
- FG1.3 (Prompt Registry) — FG1.1 + FG1.2 완료 후 착수

## 4. FG별 의무 산출물

### 산출물 타입 정의

| 산출물 | 파일명 규칙 | 작성자 | 제출 기한 | 필수 여부 |
|--------|-----------|-------|---------|---------|
| **작업지시서** | `task{N}-{M}.md` | 기술 리드 | FG 착수 전 | 항상 필수 |
| **검수보고서** | `검수보고서_FG{N}-{M}.md` | 검수팀 (기술 리드 외) | FG 완료 후 | 항상 필수 |
| **보안취약점검사보고서** | `보안취약점검사보고서_FG{N}-{M}.md` | 보안 담당자 | FG 완료 후 | 항상 필수 |
| **AI품질평가보고서** | `AI품질평가보고서_FG{N}-{M}.md` | AI/ML 담당자 | FG 완료 후 | AI 기능 FG만 |

### 산출물별 세부 규칙

#### 작업지시서
- **양식**: Task 0-1~0-4 구조 참조
- **필수 섹션**: 작업 목적, 작업 범위, 선행 조건, 주요 작업 항목, 산출물, 완료 기준, 작업 지침
- **길이**: 100~200줄 (명확하되 과도하게 상세하지 않음)

#### 검수보고서
- **필수 섹션**: 검수 일시, 검수자, 작업지시서 기준 항목별 결과, 이슈 목록, 개선 사항, 최종 판정
- **이슈 등급**: Critical(차단) / High(해결 필수) / Medium(다음 Phase) / Low(문서화)
- **최종 판정**: "승인" 또는 "조건부 승인" (이슈 목록 명시)
- **서명**: 검수자 + PO

#### 보안취약점검사보고서
- **필수 섹션**: 검사 일시, 검사자, OWASP 항목별 결과, 취약점 목록, 개선안, 최종 판정
- **검사 범위**:
  - Phase 0~4: OWASP Top 10 (A01~A10)
  - Phase 5~9: OWASP Top 10 + OWASP Top 10 for LLM Applications
- **서명**: 보안 담당자 + CTO

#### AI품질평가보고서
- **양식**: `AI품질평가_템플릿.md` 사용
- **적용 대상**: `AI품질평가_규약.md §4` FG 목록 참조
- **서명**: AI/ML 담당자 + 기술 리드

## 5. FG 완료 기준 체크리스트

FG가 "완료"로 판정되려면 아래 7개 조건을 모두 만족해야 한다:

```
☑ 1. 의무 산출물 모두 제출됨
      └─ 작업지시서, 검수보고서, 보안취약점검사보고서
      └─ AI품질평가보고서 (AI 기능 FG에 한함)

☑ 2. 검수보고서 "승인" 또는 "조건부 승인"
      └─ Critical 이슈 0건
      └─ High 이슈: 해결 또는 Risk Acceptance 서명

☑ 3. 보안취약점검사보고서 "승인" 또는 "조건부 승인"
      └─ Critical 취약점 0건
      └─ High 취약점: 패치 또는 mitigation 계획 수립

☑ 4. AI품질평가보고서 "승인" (해당 FG에 한함)
      └─ 기준값 통과 또는 개선 계획 명시

☑ 5. 회귀 테스트 100% 통과
      └─ 기존 기능 레그레션 없음
      └─ 신규 기능 테스트 통과

☑ 6. 선행 FG 모두 완료됨
      └─ 선행 FG가 없으면 이 항목 생략

☑ 7. PO 또는 기술 리드 최종 승인
      └─ 모든 산출물 검토 완료
```

### 블로킹 기준

| 조건 | 처리 |
|------|------|
| Critical 보안 이슈 | 반드시 해결 후 FG 완료 가능 |
| High 기능 이슈 | 해결 또는 Risk Acceptance 서명 필요 |
| AI 기능 기준값 미달 | 개선 계획 명시 시 조건부 완료 가능. Phase 종결 전 재측정 필요 |
| 회귀 테스트 실패 | 통과 필수 |
| 의무 산출물 누락 | 모두 제출 필수 |
| Medium/Low 이슈 | 다음 Phase Action Item으로 등재 후 조건부 완료 가능 |

## 8. 예외 규칙

1. **초기 Phase AI 기준값 완화**: Phase 1~3은 기준값 -0.05 완화 가능 (AI품질평가 규약 §3 참조)
2. **문서화만인 FG**: 코드 없는 순수 문서 FG(예: FG0.1)는 회귀 테스트 항목 생략 가능
3. **긴급 보안 패치**: 보안 Critical 이슈 발견 시 현재 Phase를 일시 정지하고 즉시 처리. 별도 핫픽스 보고서 작성
