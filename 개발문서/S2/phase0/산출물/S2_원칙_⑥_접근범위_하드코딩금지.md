# S2 원칙 ⑥ — 접근 범위도 하드코딩 금지

**확정일**: 2026-04-16
**근거**: Season 2 개발계획.md §1, §3.2

---

## 1. 정의

외부 소비자의 scope 어휘는 **Scope Profile**(관리자 설정)으로 관리하며,  
Mimir 코드에 특정 scope 문자열이 등장하지 않는다.

S1 원칙 ③("문서 타입은 하드코딩 금지")의 논리를 접근 제어 계층까지 확장한 것이다.

---

## 2. 배경

S1에서 RBAC는 `admin`, `editor`, `viewer` 역할을 하드코딩하여 구현했다.  
S2에서 챗봇은 `team`, `org`, `confidential` 같은 자체 scope 어휘로 Mimir에 접근한다.  
이 어휘가 변경되거나 새로운 소비자가 다른 어휘를 요구할 때, 코드 수정 없이 관리자 설정만으로 대응해야 한다.

---

## 3. Scope Profile 구조

```
ScopeProfile
  ├─ id: UUID
  ├─ name: "기본 챗봇용"            ← 관리자가 명명
  └─ ScopeDefinition[]
       ├─ scope_name: "team"        ← 외부 소비자가 요청하는 어휘
       └─ acl_filter: FilterExpression  ← 관리자가 정의하는 ACL 규칙
```

**작동 흐름:**
1. 관리자가 Admin UI에서 Scope Profile을 생성하고 scope name ↔ ACL filter를 정의
2. API Key 발급 시 Scope Profile을 바인딩
3. 외부 소비자가 `scope: "team"` 요청 → Mimir가 해당 API Key의 Scope Profile에서 `"team"` → ACL filter 해석
4. 해석된 ACL filter를 기존 권한 엔진에 AND로 합성하여 검색/조회 실행

---

## 4. FilterExpression 규칙 (S2 수위)

| 항목 | 규칙 |
|------|------|
| **허용 필드** | whitelist: `organization_id`, `team_id`, `visibility`, `classification`, `owner_id` (DocumentType별 metadata 필드 추가 가능) |
| **허용 연산자** | 비교: `eq`, `neq`, `in`, `not_in` / 문자열: `starts_with`, `contains` / 범위: `gte`, `lte` (숫자) |
| **구조** | AND, OR 조합 가능 (최대 깊이 3단계) |
| **동적 바인딩** | `$ctx.*` — 요청의 `access_context` 필드로 동적 치환 |

**예시:**
```json
{
  "and": [
    {"field": "organization_id", "op": "eq", "value": "$ctx.organization_id"},
    {"field": "team_id", "op": "eq", "value": "$ctx.team_id"}
  ]
}
```

---

## 5. $ctx.* 주입 공격 방어

- `$ctx.*` 변수는 `access_context`의 선언된 필드명만 참조 가능 (코드 실행 불가)
- FilterExpression 파싱: 정해진 문법만 인정 (SQL-like 표현식 거부)
- 필드명 검증: whitelist 확인 후 필터 적용
- 잘못된 FilterExpression → 요청 거부 (400), 감사 로그 기록

---

## 6. 코드 금지 목록

```python
# ❌ 금지: scope 문자열 하드코딩
if scope == "team":
    filter_by(team_id=user.team_id)

# ❌ 금지: 조직별 하드코딩
if user.org_id == "acme":
    apply_special_filter()

# ❌ 금지: 환경변수에 scope 고정
ALLOWED_SCOPES = os.environ.get("SCOPES", "team,org")

# ✅ 올바른 패턴: Scope Profile 로드 후 FilterExpression 적용
scope_profile = load_scope_profile(api_key.scope_profile_id)
acl_filter = scope_profile.resolve(requested_scope, access_context)
query = query.apply_filter(acl_filter)
```

## 7. 의무 적용 계층

| 계층 | 적용 방식 |
|------|---------|
| API 인증 레이어 | client-credentials 검증 후 Scope Profile 로드 |
| ACL 필터링 | 모든 검색·조회·쓰기 API에 AND 합성 |
| RAG 파이프라인 | 벡터 검색 + FTS 모두 ACL filter 적용 |
| MCP Tool | Tool 호출 시에도 동일 ACL 적용 |

## 8. FAQ — 현장에서의 의문

**Q. S3에서 임의 SQL-like 표현식을 지원할 예정이라면, S2 FilterExpression 설계가 S3와 충돌하지 않나?**  
A. Scope Profile 인터페이스(scope_name ↔ acl_filter 매핑)는 유지. S3에서는 FilterExpression 수위를 높이는 것이지 인터페이스를 바꾸는 것이 아니다.

**Q. `organization_id`나 `team_id`가 없는 문서 타입에서도 이 원칙이 적용되나?**  
A. 예. whitelist 필드는 DocumentType별 metadata 필드도 포함할 수 있다. Scope Profile 정의 시 해당 DocumentType의 필드를 명시하면 된다.

**Q. 관리자가 Scope Profile을 잘못 설정하면 데이터가 노출될 수 있지 않나?**  
A. Scope Profile은 기존 RBAC 권한 엔진에 AND 합성이다. 기존 권한이 없으면 Scope Profile이 추가 접근을 부여할 수 없다. 최소 권한 원칙은 RBAC 계층이 보장한다.

## 10. S1 원칙 ③(문서 타입 하드코딩 금지)과의 관계

| 측면 | S1 원칙 ③ | S2 원칙 ⑥ |
|------|----------|----------|
| 하드코딩 금지 대상 | DocumentType 분기 로직 | scope 문자열, ACL 정책 |
| 동적 설정 계층 | DocumentType config | Scope Profile |
| 변경 시 배포 불필요 | DocumentType 추가 | scope 어휘 변경 |

S2 원칙 ⑥은 S1 원칙 ③의 논리를 접근 제어로 확장한다. 두 원칙은 동일한 철학("운영자 설정으로 코드 수정 없이 행동 변경")을 서로 다른 계층에 적용한다.
