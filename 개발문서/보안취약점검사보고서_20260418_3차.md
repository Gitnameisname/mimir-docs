### SEC4-BE-001 · Medium · [완료] avatar_url SSRF 방어 우회 가능성

| 항목 | 내용 |
|------|------|
| **위치** | `backend/app/api/v1/account_router.py` — `_validate_avatar_url()` |
| **취약점** | 이전 구현은 `http`/`https` 스킴 확인만 수행, IPv6 SSRF 우회 가능 |
| **OWASP** | A10:2021 – Server-Side Request Forgery |
| **심각도** | Medium |
| **조치** | `urlparse` + `ipaddress.ip_address()` 기반 SSRF 방어 로직 추가 (is_loopback, is_private, is_link_local, is_multicast 차단) |
| **상태** | ✅ 완료 |

### SEC4-BE-002 · 정보성 · [허용] 동적 SQL f-string 패턴

| 항목 | 내용 |
|------|------|
| **위치** | `backend/app/repositories/scope_profile_repository.py`, `agent_repository.py`, `workflow_repository.py`, `users_repository.py` |
| **취약점 유형** | 동적 SQL 구성 시 f-string 사용 — 코드 품질 우려 |
| **OWASP** | A03:2021 – Injection |
| **심각도** | 정보성 (실제 취약점 없음) |
| **분석** | 모든 f-string 삽입 부분은 코드 내 하드코딩된 컬럼 명세 문자열(`"name = %s"`, `"description = %s"` 등)이며, 사용자 입력 값은 항상 `%s` 파라미터 바인딩으로 처리됨. SQL 인젝션 경로 없음 |
| **조치** | 코드 수정 불필요. 신규 쿼리 작성 시 동일 패턴(하드코딩 컬럼명 + `%s` 바인딩) 유지하도록 코드 리뷰 체크리스트 반영 |
| **상태** | ✅ 허용(Accepted Risk) — 실제 취약점 없음 확인 |
