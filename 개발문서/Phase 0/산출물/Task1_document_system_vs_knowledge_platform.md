# Phase 0 — 문서 시스템 vs 지식 플랫폼

## 1. 두 개념의 정의

### 문서 시스템 (Document System)
파일 또는 문서를 저장하고 조회할 수 있는 시스템. 핵심 관심사는 **파일의 저장과 열람**이다. 문서 유형은 고정되어 있거나 사전에 정해진 형식에 따라 처리된다. 사용자는 주로 UI를 통해 접근하며, 구조·메타데이터·버전은 부가 기능으로 다뤄지는 경우가 많다.

**특징 요약:**
- 파일/문서 저장 및 다운로드 중심
- 특정 문서 유형(규정집, 보고서 등)에 최적화된 구조
- 검색은 파일명 또는 전문 검색 수준
- 버전은 파일 교체 또는 이름 기반 관리
- 외부 시스템은 파일을 직접 가져가는 방식으로 연계
- UI 없이는 접근이 어렵거나 불가능

### 지식 플랫폼 (Knowledge Platform)
지식 자산을 구조화된 형태로 저장하고, 추적하며, 다양한 소비자(UI, API, AI 도구)가 일관된 방식으로 접근할 수 있게 하는 시스템. 핵심 관심사는 **지식 자산의 구조·버전·관계·탐색 가능성**이다.

**특징 요약:**
- Document·Node·Metadata·Version·Relationship을 독립적 엔티티로 관리
- 문서 유형에 종속되지 않는 범용 구조
- 메타데이터 기반 정밀 탐색과 필터링
- 모든 변경이 버전으로 추적됨
- API를 통해 UI, 외부 시스템, AI 도구 모두 동등하게 접근
- 지식 단위(Node 수준)로 접근 가능

## 2. 차이점 비교

| 관점 | 문서 시스템 | 지식 플랫폼 |
|------|------------|------------|
| **핵심 단위** | 파일 (File) | Document + Node + Metadata + Version |
| **구조 관리** | 파일 내부 (비정형) | 플랫폼 수준에서 Node 트리로 관리 |
| **버전 관리** | 파일 교체 / 이름 기반 | 버전 엔티티로 추적, 이전 버전 접근 가능 |
| **메타데이터** | 파일 속성 또는 태그 수준 | 확장 가능한 정형 스키마, 유형별 정의 |
| **검색** | 파일명, 전문 검색 | 구조·메타데이터·관계 기반 복합 탐색 |
| **접근 방식** | UI 중심, 파일 다운로드 | API-first, Node/Document 단위 접근 |
| **외부 연계** | 파일 export 또는 scraping | 정형 API 응답으로 구조화된 지식 제공 |
| **문서 유형** | 고정되거나 사전 정의된 유형 | DocumentType으로 유형 확장 가능 |
| **관계 표현** | 불가능 또는 링크 수준 | Relationship 엔티티로 문서 간 관계 모델링 |
| **AI 연계** | 파일 파싱 후 처리 | 구조화된 API 응답을 직접 소비 |

## 3. 왜 Mimir는 지식 플랫폼이어야 하는가

### 3-1. 문서 유형의 다양성
조직이 관리해야 하는 문서는 규정집만이 아니다. 매뉴얼, 지침, 보고서, 정책서, 프로세스 정의서 등 다양한 유형이 존재한다. 문서 시스템으로 출발하면 새로운 유형이 추가될 때마다 시스템 전체 구조가 흔들린다. 지식 플랫폼은 DocumentType 메커니즘으로 이를 확장 가능하게 수용한다.

### 3-2. 버전과 이력의 본질적 필요성
규정이나 정책 문서는 "현재 버전"만큼이나 "과거에 어떤 내용이었는가"가 중요하다. 파일 기반 버전 관리는 이 요구를 신뢰성 있게 충족하지 못한다. 버전을 플랫폼의 기본 구조로 내재화해야 한다.

### 3-3. 외부 시스템과 AI 도구의 연계 필요성
향후 이 플랫폼의 지식 자산은 다른 업무 시스템, AI 기반 검색·요약 도구, 자동화 워크플로와 연계될 것이다. 이를 위해 지식이 구조화된 API 응답으로 제공되어야 한다. 파일 다운로드 방식으로는 이 연계를 지속 가능하게 만들 수 없다.

### 3-4. 검색과 탐색의 정밀성 요구
"이 조항이 어느 문서에 있는가", "특정 메타데이터 조건을 만족하는 문서 목록은?", "이 문서와 관련된 다른 문서는?" 같은 탐색은 파일명 검색으로 해결할 수 없다. Node 수준의 구조 관리와 메타데이터 인덱싱이 전제되어야 한다.

### 3-5. 운영과 사용 기능의 분리 가능성
지식 플랫폼은 일반 사용자 접근과 운영/관리 기능을 처음부터 분리하여 설계한다. 이는 문서 시스템에서는 보통 후속 과제로 밀리는 항목이다.

## 4. 이 차이가 이후 설계에 미치는 영향

### 데이터 모델 설계 (Phase 1)
- Document는 파일 레코드가 아니라 독립적 도메인 엔티티로 설계해야 한다.
- Version은 Document의 부가 속성이 아니라 별도의 엔티티로 분리한다.
- Node는 문서 내부 구조를 표현하는 독립 엔티티로, DocumentType별 계층 구조를 가진다.
- Metadata는 고정 컬럼이 아니라 확장 가능한 스키마로 설계한다.

### API 설계 (Phase 2 이후)
- API는 파일 업로드/다운로드 중심이 아니라, Document·Version·Node·Metadata 단위의 CRUD와 탐색 엔드포인트로 구성한다.
- 모든 핵심 기능은 API로 먼저 노출되고, UI는 API의 소비자로 동작한다.

### UI 설계 (Phase 2 이후)
- User UI는 지식 탐색·조회·편집을 위한 API 소비자로 설계한다.
- Admin UI는 별도 영역으로 분리하여 문서 유형, 권한, 운영 기능을 담당한다.

### 저장 구조 설계 (Phase 0-6)
- 문서 본문과 구조·메타데이터·버전 이력은 분리된 저장 전략으로 관리한다.
- 검색 인덱스는 도메인 모델과 독립된 계층으로 구성한다.
