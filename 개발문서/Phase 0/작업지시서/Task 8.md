Phase 0-8 작업 지시서

Phase 1 입력 조건 정리

1. 작업명

Phase 0-8. Phase 1 입력 조건 정리

⸻

2. 작업 목적

Phase 0에서 확정한 비전, 플랫폼 원칙, API-first 원칙, User/Admin UI 분리, 기술 방향, 저장 전략, 시스템 경계를 Phase 1 도메인 모델 설계에 직접 연결되는 입력 조건으로 정리한다.

이 작업의 목적은 새로운 내용을 추가로 만드는 것이 아니라, 지금까지 정한 상위 원칙을 Phase 1이 바로 사용할 수 있는 설계 입력서로 압축하는 것이다.

즉, 이 문서는 앞으로 아래 질문에 답해야 한다.
	•	Phase 1에서 무엇을 반드시 반영해야 하는가
	•	어떤 논의는 이미 끝난 것으로 보고 넘어가야 하는가
	•	Document / Version / Node / DocumentType / Metadata / Status 모델은 어떤 상위 제약 아래 설계되어야 하는가
	•	무엇은 도메인 의미로 먼저 정의하고, 무엇은 후속 상세 설계로 넘겨야 하는가
	•	Phase 1 설계자가 어디서 자유롭게 설계할 수 있고, 어디서는 방향을 벗어나면 안 되는가

⸻

3. 작업 배경

Phase 0의 산출물이 많아질수록, 정작 다음 Phase에서 핵심이 흐려지는 문제가 생길 수 있다.
특히 Phase 1은 플랫폼의 코어 모델을 다루기 때문에, 상위 원칙이 충분히 정리되어 있지 않으면 다음 문제가 발생한다.
	•	도메인 모델이 특정 저장 방식에 끌려감
	•	특정 문서 유형에 맞춘 구조가 코어 모델이 됨
	•	Version과 상태 이력이 혼동됨
	•	Metadata를 무제한 자유 필드로 방치하거나, 반대로 과도하게 경직시킴
	•	API-first 원칙이 모델에 반영되지 않음
	•	User/Admin 관점 차이가 도메인 모델에 반영되지 않음
	•	외부 시스템 형식이 내부 도메인을 지배함

따라서 Phase 1에 들어가기 전에,
지금까지의 상위 결정들을 도메인 모델 설계 입력 조건으로 재구성해야 한다.

⸻

4. 작업 목표

아래를 포함하는 입력 정리 문서를 만든다.
	1.	Phase 1이 다뤄야 할 핵심 설계 대상 재정의
	2.	Phase 0에서 이미 확정된 전제 정리
	3.	Phase 1에서 반드시 반영해야 할 설계 제약 정리
	4.	Phase 1 설계자가 자유롭게 탐색할 수 있는 영역과 아닌 영역 구분
	5.	Document / Version / Node / DocumentType / Metadata / Status 각각에 대한 상위 입력 조건 정리
	6.	Phase 2(API/서비스 설계)로 자연스럽게 이어질 수 있도록 연결 포인트 정리

⸻

5. 반드시 반영할 핵심 방향

5-1. Phase 1은 코어 의미를 정의하는 단계다

Phase 1은 저장 구현, UI 구현, 외부 연동 세부를 먼저 정하는 단계가 아니라,
플랫폼의 핵심 자산이 어떤 의미 구조를 갖는지 정의하는 단계다.

5-2. 범용성은 필수다

Phase 1 모델은 특정 문서 유형, 특정 업무 흐름, 특정 외부 시스템 형식에 종속되면 안 된다.

5-3. 모델은 API와 저장 분해를 견딜 수 있어야 한다

도메인 모델은 하나의 화면이나 하나의 테이블 구조가 아니라,
향후 API 리소스, 저장 계층, 검색 계층, 운영 계층으로 분해 가능한 구조여야 한다.

5-4. 현재 상태와 이력은 구분되어야 한다

Document, Version, Status, Audit, Lifecycle의 관계를 설계할 때
현재 사용자에게 보이는 상태와 시스템이 보관하는 이력을 혼동하면 안 된다.

5-5. 확장성과 질서가 동시에 필요하다

Metadata, DocumentType, 구조 표현은 확장 가능해야 하지만,
질의성, 검증 가능성, 운영 가능성을 잃어서는 안 된다.

⸻

6. 세부 수행 항목

아래 순서대로 작업하라.

A. Phase 1의 설계 대상 재정의

먼저 Phase 1이 정확히 무엇을 설계하는 단계인지 다시 정의하라.

최소 포함:
	•	Document
	•	Version
	•	Node / Structure
	•	DocumentType
	•	Metadata 확장 구조
	•	상태 모델 기초(Status/Lifecycle)

아래도 함께 정리:
	•	이 단계에서 다루는 것
	•	이 단계에서 아직 다루지 않는 것
	•	이 단계의 결과물이 왜 플랫폼 전체에 결정적 영향을 주는가

⸻

B. Phase 0 확정 전제 요약

Phase 0에서 이미 확정된 내용을 Phase 1 관점으로 재정리하라.

최소 포함:
	•	플랫폼은 문서 시스템이 아니라 범용 지식 플랫폼이다
	•	API-first가 기본 원칙이다
	•	User/Admin UI는 책임과 관점이 분리된다
	•	도메인 모델은 저장 구현과 분리되어야 한다
	•	다층 저장 전략을 전제로 한다
	•	외부 시스템 연계는 가능하지만 코어 도메인을 지배하면 안 된다

이 항목은 “Phase 1에서 다시 논쟁하지 않아야 할 것” 목록 역할도 해야 한다.

⸻

C. Phase 1 설계 제약과 자유도 구분

Phase 1 설계자가 어떤 부분은 반드시 지켜야 하고, 어떤 부분은 탐색 가능할지 구분하라.

예시 구분:
	•	고정 전제
	•	특정 문서 유형 비종속
	•	버전 개념 존재
	•	구조 기반 표현 필요
	•	메타데이터 확장 가능해야 함
	•	API 리소스로도 노출 가능해야 함
	•	탐색 가능
	•	Node 세부 구조 표현 방식
	•	DocumentType의 세부 확장 메커니즘
	•	Metadata 스키마 계층화 수준
	•	상태 모델의 세부 단계 수

각 항목마다 왜 고정인지, 왜 탐색 가능한지 설명하라.

⸻

D. Document 모델 입력 조건 정리

Document에 대해 상위 입력 조건을 정리하라.

최소 포함:
	•	Document는 플랫폼의 핵심 지식 자산 단위다
	•	단순 파일이 아니라 식별 가능한 관리 단위여야 한다
	•	Version, Node, Metadata, Status, Relationship과 연결될 수 있어야 한다
	•	특정 문서 유형에 종속되지 않아야 한다
	•	API/검색/권한/운영 관점에서도 의미가 유지되어야 한다

추가로 정리:
	•	Document와 첨부 파일의 관계
	•	Document와 외부 리소스 참조의 관계 가능성
	•	Document와 검색 대상의 관계

⸻

E. Version 모델 입력 조건 정리

Version에 대해 상위 입력 조건을 정리하라.

최소 포함:
	•	Version은 단순 수정 시각이 아니라 의미 있는 변경 단위여야 한다
	•	최신 상태와 버전 기록은 구분되어야 한다
	•	Version은 본문, 구조, 메타데이터 일부 또는 전체와 연결될 수 있어야 한다
	•	Version 이력과 Audit 로그는 동일하지 않다
	•	Version은 API와 운영 관점에서 추적 가능해야 한다

추가로 정리:
	•	Version granularity를 후속 설계에서 결정할 여지
	•	Draft/Published 같은 상태와 Version 관계
	•	구조 변경과 버전 관계

⸻

F. Node / Structure 모델 입력 조건 정리

Node/Structure에 대해 상위 입력 조건을 정리하라.

최소 포함:
	•	문서는 트리/계층 기반 구조를 가질 수 있어야 한다
	•	Node는 부분 조회/부분 렌더링/부분 검색 가능성을 열어야 한다
	•	순서, 부모-자식 관계, 계층 표현이 필요할 수 있다
	•	Node 표현은 특정 문서 포맷에 종속되지 않아야 한다
	•	구조와 본문은 개념적으로 구분 가능해야 한다

추가로 정리:
	•	구조 변경이 버전과 어떻게 연결될 수 있는지
	•	Node 단위 메타데이터 필요성 가능성
	•	향후 chunk/search/AI 활용 가능성

⸻

G. DocumentType 모델 입력 조건 정리

DocumentType에 대해 상위 입력 조건을 정리하라.

최소 포함:
	•	DocumentType은 특정 종류의 문서 행동/메타/구조 정책을 정의할 수 있어야 한다
	•	그러나 코어 도메인 자체를 특정 타입에 종속시키면 안 된다
	•	타입별 확장 규칙은 허용하되, 공통 코어 모델은 유지해야 한다
	•	Admin 관점에서 관리 가능해야 한다
	•	API와 UI 양쪽에서 해석 가능한 수준이어야 한다

추가로 정리:
	•	타입별 validation 정책 가능성
	•	타입별 상태 모델 차이 가능성
	•	타입별 metadata 확장과의 관계

⸻

H. Metadata 모델 입력 조건 정리

Metadata에 대해 상위 입력 조건을 정리하라.

최소 포함:
	•	공통 메타데이터와 확장 메타데이터를 구분할 수 있어야 한다
	•	확장 가능해야 하지만, 질의/필터/검증 가능성도 고려해야 한다
	•	검색 및 탐색에서 활용 가능해야 한다
	•	User/Admin 관점에서 노출 수준이 달라질 수 있다
	•	저장 전략상 정규 필드와 유연 필드의 균형을 고려해야 한다

추가로 정리:
	•	Metadata schema/validation 필요성
	•	타입별 메타데이터와 공통 메타데이터의 관계
	•	정렬/필터/집계에 쓰이는 메타데이터 고려

⸻

I. Status / Lifecycle 모델 입력 조건 정리

상태 모델 기초에 대한 상위 입력 조건을 정리하라.

최소 포함:
	•	문서는 단순 존재 여부가 아니라 상태를 가진다
	•	상태는 작성/검토/승인/게시/보관 등으로 확장될 수 있다
	•	현재 상태와 상태 전이 이력은 구분되어야 한다
	•	상태 정보는 User/Admin에게 다르게 보일 수 있다
	•	Version과 상태는 관련되지만 동일 개념은 아니다

추가로 정리:
	•	상태 전이 규칙의 후속 확장 여지
	•	Admin 정책과 연결될 가능성
	•	API/감사/운영 로그와의 연결성

⸻

J. 외부 연계와 독립성 관련 입력 정리

Phase 1이 외부 시스템 형식에 끌려가지 않도록 입력 조건을 정리하라.

최소 포함:
	•	외부 인증 구조가 사용자/권한 도메인 전체를 지배하면 안 됨
	•	외부 파일 저장 구조가 Document 모델 자체를 규정하면 안 됨
	•	검색 인덱스 구조가 원본 도메인 모델이 되면 안 됨
	•	AI 추출/보조 데이터는 코어 모델과 구분되어야 함
	•	외부 문서 소스 포맷은 Connector 계층에서 흡수해야 함

⸻

K. Phase 2 연결 포인트 정리

Phase 1 결과가 Phase 2 API/서비스 설계로 자연스럽게 연결될 수 있도록 포인트를 정리하라.

최소 포함:
	•	각 코어 모델은 향후 API 리소스로 드러날 수 있어야 함
	•	상태/버전/메타데이터/구조는 서비스 계층에서 조합 가능해야 함
	•	User/Admin/External 소비자 차이를 API 설계에서 반영해야 함
	•	감사/이력/비동기 처리를 위한 확장 여지를 남겨야 함

⸻

L. 설계 체크리스트 도출

Phase 1 설계 리뷰에 바로 쓸 수 있는 체크리스트를 작성하라.

예시 질문:
	•	이 모델은 특정 문서 유형에 종속적인가?
	•	최신 상태와 이력이 분리되어 있는가?
	•	구조와 본문이 개념적으로 구분되는가?
	•	메타데이터 확장성과 질의 가능성의 균형이 잡혀 있는가?
	•	API 리소스로 노출할 때 의미가 유지되는가?
	•	저장 방식이 모델 의미를 지배하고 있지 않은가?
	•	외부 시스템 형식이 코어 모델에 침투하고 있지 않은가?

8. 문서 작성 원칙
	•	새로운 비전을 만드는 문서가 아니라, 기존 결정을 압축하는 문서로 작성할 것
	•	Phase 1 설계자가 바로 사용할 수 있는 수준으로 구체화할 것
	•	구현 상세나 테이블 설계로 내려가지 말 것
	•	상위 원칙과 모델별 입력 조건의 연결이 보여야 함
	•	무엇이 이미 정해졌고 무엇이 아직 열려 있는지 분명히 할 것

⸻

9. 금지 사항

아래는 하지 마라.
	•	Phase 0 문서를 반복 요약만 하고 모델별 입력으로 연결하지 않는 것
	•	저장/검색/API/UI 세부 구현으로 내려가는 것
	•	특정 문서 유형을 기준으로 코어 모델을 해석하는 것
	•	Version과 Audit, Status를 서로 섞어 쓰는 것
	•	탐색 가능한 영역과 고정 전제를 구분하지 않는 것

10. 완료 기준

다음 조건을 만족하면 완료다.
	•	Phase 1 설계 대상이 명확히 재정의됨
	•	Phase 0 확정 전제가 Phase 1 관점으로 정리됨
	•	Document / Version / Node / DocumentType / Metadata / Status별 입력 조건이 정리됨
	•	고정 전제와 탐색 가능 영역이 구분됨
	•	Phase 2 연결 포인트가 정리됨
	•	설계 리뷰 체크리스트가 도출됨

	•	먼저 Phase 0 확정 전제를 정리
