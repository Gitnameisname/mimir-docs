Phase 0-7. 시스템 경계 정의

⸻

2. 작업 목적

플랫폼이 직접 책임지는 영역과 외부 시스템과 연계하는 영역을 구분하여, 이후 설계와 개발 범위가 불필요하게 팽창하거나 혼선이 생기는 것을 방지한다.

이 작업의 목적은 단순히 연동 목록을 만드는 것이 아니라, 아래를 명확히 하는 것이다.
	•	이 플랫폼의 코어 책임은 무엇인가
	•	무엇은 플랫폼 내부 기능으로 가져가야 하는가
	•	무엇은 외부 시스템과의 연계로 다뤄야 하는가
	•	어떤 영역은 어댑터/커넥터/통합 포인트로 처리해야 하는가
	•	시스템 내부 경계와 외부 경계가 Phase 1 이후 설계에 어떤 제약을 주는가

즉, 이 문서는 플랫폼의 책임 범위 헌장 역할을 해야 한다.

⸻

3. 작업 배경

범용 지식 플랫폼은 요구가 커질수록 쉽게 모든 것을 안으로 끌어들이게 된다.
특히 다음 영역에서 경계 혼선이 자주 발생한다.
	•	인증/SSO를 플랫폼이 직접 다룰지 외부 인증 시스템과 연계할지
	•	파일 저장을 플랫폼이 직접 할지 외부 스토리지를 참조할지
	•	검색을 코어 DB에서 해결할지 별도 검색 계층으로 분리할지
	•	승인/워크플로를 코어 기능으로 둘지 외부 BPM/알림 시스템과 연동할지
	•	AI 기능을 플랫폼의 본질로 둘지 보조 연계 기능으로 둘지
	•	외부 문서 소스 동기화를 직접 수용할지 커넥터 계층으로 둘지

이 경계를 먼저 정의하지 않으면 다음 문제가 생긴다.
	•	플랫폼이 과도하게 비대해짐
	•	코어 도메인 모델이 외부 시스템 구조에 잠식됨
	•	API 설계가 내부/외부 책임을 혼동함
	•	Admin 기능이 운영 툴과 통합 툴을 모두 떠안게 됨
	•	일정과 범위 통제가 어려워짐

따라서 이 단계에서 시스템 컨텍스트와 책임 경계를 먼저 명확히 해야 한다.

5. 반드시 반영할 핵심 방향

5-1. 코어 플랫폼은 지식 자산 관리에 집중한다

이 플랫폼의 본질은 문서/버전/구조/메타데이터/상태/검색/권한/운영 통제를 포함한
지식 자산 관리 플랫폼이다.

5-2. 외부 시스템과의 연계는 필요하지만, 코어를 대체하면 안 된다

인증, 외부 저장소, 검색 엔진, 알림, AI, 워크플로, 외부 업무 시스템과 연동할 수 있지만,
그 연계 구조가 코어 도메인을 지배해서는 안 된다.

5-3. 경계는 기능 경계이자 책임 경계다

같은 기능처럼 보여도,
	•	코어 시스템이 책임져야 하는 부분
	•	외부 시스템이 책임지는 부분
	•	어댑터가 변환해야 하는 부분
을 구분해야 한다.

5-4. 내부와 외부를 직접 결합하지 않는다

가능한 한 코어 도메인과 외부 시스템은 API/이벤트/커넥터 계층을 통해 느슨하게 연결해야 한다.

5-5. 경계 정의는 범위 통제 도구다

경계 문서는 “이것도 플랫폼 안에서 다 만들자”라는 요구를 조정하는 기준이 되어야 한다.

6. 세부 수행 항목

아래 순서대로 작업하라.

A. 코어 플랫폼 책임 정의

먼저 이 플랫폼이 반드시 직접 책임져야 하는 코어 기능을 정의하라.

최소 포함:
	•	Document 관리
	•	Version 관리
	•	Node/Structure 관리
	•	Metadata 관리
	•	DocumentType 관리
	•	Status/Lifecycle 관리
	•	Search/Query의 코어 경험 정의
	•	Permission/Access Control의 코어 정책
	•	Audit/History 기록
	•	User/Admin UI를 위한 API 제공

각 항목마다 아래를 정리:
	•	왜 코어 책임인지
	•	외부 시스템에 위임하면 안 되는 이유 또는 범위
	•	어떤 수준까지 플랫폼이 책임져야 하는지

⸻

B. 내부 서브시스템 경계 초안 정의

플랫폼 내부를 다시 몇 개의 서브시스템/영역으로 나누어라.

권장 최소 구분:
	•	Core Domain
	•	API/Application Layer
	•	User UI
	•	Admin UI
	•	Search/Index Integration Layer
	•	File/Resource Handling Layer
	•	Auth/Permission Boundary
	•	Audit/Operations Layer
	•	Integration/Connector Layer
	•	Async Job/Event Handling Layer

각 영역마다 아래를 작성:
	•	역할
	•	코어 여부
	•	직접 보유하는 책임
	•	다른 영역과의 인터페이스 방식

⸻

C. 외부 시스템 유형 분류

플랫폼과 연계될 수 있는 외부 시스템 유형을 분류하라.

최소 포함:
	•	Identity / SSO / IAM
	•	External File Storage / Object Storage
	•	Search Engine / Indexing Engine
	•	Notification / Messaging System
	•	Workflow / Approval / BPM System
	•	External Business Applications
	•	External Document Sources / Connectors
	•	AI / NLP / RAG / Automation Services
	•	Observability / Logging / Monitoring Stack

각 유형마다 아래를 정리:
	•	왜 연계 대상이 되는가
	•	플랫폼이 직접 대체할 필요가 없는 이유
	•	어느 계층을 통해 연결되는 것이 적절한가

⸻

D. 내부 책임 vs 외부 연계 책임 매트릭스 작성

각 기능 영역에 대해 아래 중 하나로 분류하라.
	•	플랫폼 코어 책임
	•	플랫폼 코어 + 외부 연계 필요
	•	외부 시스템 책임, 플랫폼은 참조/통합만 수행
	•	현재 범위 밖(후속 검토)

기능 후보 예시:
	•	문서 원본 저장
	•	문서 구조 표현
	•	버전 관리
	•	권한 정책
	•	사용자 인증
	•	전문 검색
	•	알림 발송
	•	승인 프로세스 엔진
	•	외부 문서 수집
	•	AI 요약/추출/보조 기능
	•	운영 로그 수집
	•	배치 스케줄링

각 항목마다 왜 그렇게 분류했는지 한 줄 이상 설명하라.

⸻

E. API 경계 / 데이터 경계 / 운영 경계 구분

시스템 경계는 하나가 아니라 여러 층위가 있으므로 구분해서 정리하라.

1) API 경계
	•	어떤 API가 플랫폼 코어 API인가
	•	어떤 API는 외부 연계용/관리용/내부 운영용인가
	•	외부 시스템이 어느 수준까지 직접 접근 가능한가

2) 데이터 경계
	•	어떤 데이터는 플랫폼이 원본 소유자인가
	•	어떤 데이터는 외부 시스템의 미러/캐시/참조인가
	•	어떤 데이터는 일시적 통합 데이터인가

3) 운영 경계
	•	어떤 장애/오류는 플랫폼 팀이 직접 책임지는가
	•	어떤 문제는 외부 시스템 의존성으로 분리해야 하는가
	•	운영자 UI가 보여줘야 하는 경계 정보는 무엇인가

⸻

F. 시스템 컨텍스트 다이어그램 초안 정의

문서 수준에서 시스템 컨텍스트를 설명할 수 있도록 다이어그램 초안을 작성하라.

최소 포함 노드:
	•	End User
	•	Admin/Operator
	•	Knowledge Platform Core
	•	User UI
	•	Admin UI
	•	External Identity System
	•	External Storage
	•	Search/Index Service
	•	External Business Systems
	•	AI/Automation Services
	•	Notification/Workflow Services

각 연결에 대해 아래를 짧게 적어라.
	•	연결 목적
	•	연결 방식(API/Event/Sync/Async/Manual/Admin)
	•	데이터 소유권 주체

⸻

G. 커넥터/어댑터 원칙 정리

외부 시스템 연계를 코어 도메인에 직접 박아넣지 않기 위한 원칙을 정리하라.

최소 포함:
	•	외부 시스템별 전용 커넥터/어댑터 계층 필요성
	•	도메인 모델은 외부 API 형식에 종속되지 않아야 함
	•	외부 식별자와 내부 식별자 분리 필요성
	•	외부 장애가 코어 기능 전체를 마비시키지 않도록 격리 필요
	•	특정 벤더 종속 로직을 코어 모델에 넣지 않음

⸻

H. 범위 통제를 위한 In/Out 경계 정리

시스템 경계 문서가 실제 범위 통제에 쓰일 수 있도록, 아래를 정리하라.
	•	플랫폼이 반드시 직접 제공해야 하는 것
	•	플랫폼이 연동만 제공하면 되는 것
	•	후속 Phase로 넘겨야 하는 것
	•	“요구는 많지만 코어 목표와 맞지 않는 것” 예시

이 항목은 프로젝트 관리 측면에서도 쓰일 수 있게 작성하라.

⸻

I. 위반 사례 / 경계 붕괴 사례 정리

아래 같은 사례를 구체적으로 적어라.

예시:
	•	외부 워크플로 엔진의 데이터 구조를 코어 Document 모델에 그대로 반영함
	•	특정 SSO 공급자 요구사항이 사용자 모델 전체를 지배함
	•	검색 엔진 인덱스 구조가 원본 저장 모델이 됨
	•	AI 추출 결과가 검증 없이 원본 도메인 데이터로 저장됨
	•	외부 문서 수집기 로직이 코어 도메인 서비스 내부에 하드코딩됨

각 사례마다:
	•	왜 문제인지
	•	어떤 경계 원칙을 위반하는지
	•	어떤 계층으로 분리해야 하는지
를 정리하라.

J. Phase 1/2 입력 요약 정리

다음 단계에 전달할 핵심 시스템 경계 요약을 작성하라.

최소 포함:

	•	인증, 검색, 알림, AI, 외부 수집 등은 경계 정의를 전제로 통합해야 함
	•	Admin 기능은 코어 운영과 외부 통합 운영을 구분해서 설계해야 함

8. 문서 작성 원칙
	•	범위를 넓히는 문서가 아니라, 범위를 통제하는 문서로 작성할 것
	•	내부 기능과 외부 통합 기능을 혼동하지 말 것
	•	“연동 가능”을 “플랫폼이 직접 구현”과 동일시하지 말 것
	•	실제 아키텍처 회의에서 경계 논쟁을 줄일 수 있을 정도로 구체적으로 쓸 것
	•	특정 벤더/제품 중심이 아니라 책임 중심으로 작성할 것

⸻

9. 금지 사항

아래는 하지 마라.
	•	외부 시스템 목록만 나열하고 책임 구분을 하지 않는 것
	•	코어 도메인과 연동 도메인을 한 덩어리로 쓰는 것
	•	검색, 인증, AI, 워크플로를 모두 플랫폼 내부 기능으로 당연시하는 것
	•	데이터 소유권과 운영 책임을 생략하는 것
	•	경계 정의 없이 “유연하게 추후 결정”만 반복하는 것

	•	마지막으로 위반 사례와 Phase 1/2 입력 요약 작성
