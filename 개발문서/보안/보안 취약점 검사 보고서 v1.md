## 2. Critical 결함

### [S-1] RBAC 매트릭스 `search.*` 액션 미등록 → 인증된 사용자 전체 검색 403

**파일**: `backend/app/api/auth/authorization.py`  
**OWASP**: A01 — Broken Access Control

**현상**  
`_PERMISSION_MATRIX`에 `search.*` 액션이 하나도 등록되지 않았다.  
`authorize()` 메서드는 알 수 없는 action에 대해 **기본 거부**(`raise ApiPermissionDeniedError`)를 적용한다.

```python
# authorize() 내부
allowed_roles = _PERMISSION_MATRIX.get(action)
if allowed_roles is None:
    raise ApiPermissionDeniedError(f"Unknown action '{action}'. Access denied by default.")
```

`search.py` 라우터의 모든 검색 엔드포인트는 `require_authenticated=False`를 지정하고 있으므로:
- **익명 사용자**: `require_authenticated=False` + `ActorType.ANONYMOUS` → 3단계에서 조기 반환, **통과**
- **인증된 사용자 (VIEWER~SUPER_ADMIN)**: 4단계 RBAC 도달 → 알 수 없는 action → **403 거부**

결과적으로 로그인된 모든 사용자가 검색을 사용할 수 없고, 익명 사용자만 검색 가능한 역전 현상이 발생한다.

**수정**: `authorization.py`의 `_PERMISSION_MATRIX`에 6개 `search.*` 액션 추가.  
**상태**: ✅ 수정 완료

---

### [S-2] `POST /search/reindex` 익명 접근 허용 → 인증 없이 전체 재인덱싱 실행 가능

**파일**: `backend/app/api/v1/search.py`  
**OWASP**: A01 — Broken Access Control

**현상**  
```python
# 수정 전
authorization_service.authorize(
    action="search.reindex",
    require_authenticated=False,  # TODO: admin 권한 강제 적용
)
```

[S-1]과 결합하면:
- 익명 사용자: `require_authenticated=False` + anonymous → 조기 통과
- 인증된 SUPER_ADMIN: `search.reindex`가 매트릭스에 없음 → 403

즉 익명 사용자가 재인덱싱을 실행할 수 있고, 인증된 관리자는 실행할 수 없었다.  
재인덱싱은 전체 테이블 UPDATE 쿼리로 DB 부하를 유발하므로 DoS 벡터가 될 수 있다.

동일한 이유로 `GET /search/index-stats`도 `require_authenticated=False`였다.

**수정**:
- `search.reindex` → `SUPER_ADMIN`만 허용, `require_authenticated=True`
- `search.index_stats` → `ORG_ADMIN`, `SUPER_ADMIN`만 허용, `require_authenticated=True`

**상태**: ✅ 수정 완료

### [S-3] Bearer 경로에서 `X-Actor-Role` 헤더를 debug 가드 없이 수락 → 프로덕션 권한 상승

**파일**: `backend/app/api/auth/dependencies.py`  
**OWASP**: A07 — Identification and Authentication Failures

**현상**  
```python
# 수정 전 — _extract_bearer_actor()
actor_role = request.headers.get("X-Actor-Role")
role: str | None = actor_role if actor_role in _VALID_ROLES else None
```

개발 헤더 경로(step 5)는 `settings.debug=True` 가드가 있지만,  
Bearer 경로(step 2)의 `X-Actor-Role` 수락에는 가드가 없었다.

공격자는 임의 Bearer 토큰(`Authorization: Bearer anything`)과 함께  
`X-Actor-Role: SUPER_ADMIN` 헤더를 전송하여 최고 관리자 권한을 획득할 수 있다.

```http
GET /api/v1/search/reindex
Authorization: Bearer fake_token
X-Actor-Role: SUPER_ADMIN
```

→ 프로덕션에서 `settings.debug=False`임에도 SUPER_ADMIN으로 인식됨.

**수정**: `_extract_bearer_actor()`에서 `X-Actor-Role` 헤더 처리를 `settings.debug` 가드로 감쌈.  
```python
# 수정 후
if settings.debug:
    actor_role = request.headers.get("X-Actor-Role")
    if actor_role in _VALID_ROLES:
        role = actor_role
```

**상태**: ✅ 수정 완료

## 3. High 결함

### [S-4] API key prefix 8자리만 비교, 실제 hash 미검증 → key 부분 노출로 사용자 사칭 가능

**파일**: `backend/app/api/auth/dependencies.py` — `_extract_api_key_actor()`  
**OWASP**: A07 — Identification and Authentication Failures

**현상**  
```python
prefix = api_key[:8] if len(api_key) >= 8 else api_key
cur.execute("... WHERE ak.key_prefix = %s ...", (prefix,))
```

전달받은 API key의 앞 8자리를 DB의 `key_prefix`와 단순 비교한다.  
실제 key를 hash하여 `key_hash` 컬럼과 비교하는 로직이 없다.

동일 접두사를 가진 임의 문자열로 해당 사용자 계정을 사칭할 수 있다.

**수정**: `_extract_api_key_actor()`에 `_hash_api_key()` 헬퍼 추가.
1. `key_prefix`로 후보 행 조회 (DB 인덱스 활용)
2. `hashlib.sha256(api_key).hexdigest()`로 hash 계산
3. `hmac.compare_digest(provided_hash, db_hash)`로 비교 (타이밍 공격 방지)
4. 성공 시 `last_used_at`, `use_count` 갱신  

**상태**: ✅ 수정 완료

### [S-5] 검색어 `q` 길이 제한 없음 → FTS 계산 과부하로 DoS 가능

**파일**: `backend/app/api/v1/search.py`  
**OWASP**: A05 — Security Misconfiguration (DoS 범주)

**현상**  
검색어 `q`에 길이 제한이 없어, 수만 자의 검색어로 `to_tsquery()`를 반복 실행할 수 있다.  
`_safe_ts_query()`는 공백 split + 토큰 순회이므로 긴 입력일수록 처리 시간이 선형 증가한다.

**수정**: `search.py`에 `_validate_q()` 헬퍼 추가, 300자 초과 시 HTTP 400 반환.  
동일 헬퍼에서 `document_id` UUID 형식 및 `from_date`/`to_date` 날짜 형식도 검증.  
**상태**: ✅ 수정 완료

## 4. Medium 결함

### [S-6] `DocumentSearchResult.metadata` JSONB 필드를 검색 결과에 그대로 노출

**파일**: `backend/app/services/search_service.py` — `_map_document_row()`  
**OWASP**: A02 — Cryptographic Failures (정보 노출)

**현상**  
```python
metadata=row.get("metadata") or {},
```

`documents.metadata`는 임의 구조의 JSONB 필드다. 내부 시스템 메타데이터, 개인 식별 정보, 미공개 태그 등이 포함될 수 있으며, 검색 결과 응답에 전체가 노출된다.

**수정**: `search_service.py`에 `_filter_metadata(metadata, actor_role)` 헬퍼 추가.

| 역할 | 노출 범위 |
|------|-----------|
| `SUPER_ADMIN`, `ORG_ADMIN` | metadata 전체 |
| `VIEWER`, `AUTHOR`, `REVIEWER`, `APPROVER`, 익명 | `_`로 시작하지 않는 키만 (public_metadata) |

**키 규약**: metadata 내 `_`로 시작하는 키(`_internal_ref`, `_system_tag` 등)는 비공개.  
`_map_document_row(row, actor_role)`에 역할을 전달하여 필터 적용.

**상태**: ✅ 수정 완료

### [S-7] `from_date` / `to_date` 형식 미검증 → PostgreSQL cast 오류로 500 반환

**파일**: `backend/app/api/v1/search.py`  
**OWASP**: A03 — Injection (에러 정보 노출)

**현상**  
```python
# 서비스 레이어
where_clauses.append("d.created_at >= %s::timestamptz")
params.append(from_date)  # 형식 검증 없음
```

유효하지 않은 날짜 문자열(`2026-13-99`, `'; DROP TABLE--` 등)이 전달되면  
PostgreSQL이 cast 오류를 발생시키고, FastAPI가 이를 500 Internal Server Error로 반환한다.  
SQL 인젝션은 파라미터 바인딩으로 방어되나, DB 에러 메시지가 응답에 포함될 가능성이 있다.

**수정**: `_validate_date()` 헬퍼로 `YYYY-MM-DD` 정규식 사전 검증.  
**상태**: ✅ 수정 완료

### [S-8] 세션 쿠키 무음 무시 (silent ignore) → 인증 상태 혼동 가능

**파일**: `backend/app/api/auth/dependencies.py` — `_extract_session_actor()`  
**OWASP**: A07 — Identification and Authentication Failures

**현상**  
```python
def _extract_session_actor(session_token: str) -> ActorContext:
    # TODO: session resolver 연결 예정
    return ActorContext(
        actor_type=ActorType.USER,
        is_authenticated=False,  # 항상 미인증
        ...
    )
```

유효한 세션 쿠키를 가진 사용자가 익명으로 처리된다.  
특히 `actor_type=USER`이면서 `is_authenticated=False`인 불일치 상태가 반환된다.  
클라이언트는 쿠키가 있으므로 인증된 상태로 간주하고 요청하지만, 서버는 anonymous로 처리한다.

**수정**: `app/api/auth/session.py` 신규 추가 + `_extract_session_actor()` Valkey 조회로 교체.
- `session:{token}` 키로 Valkey에 JSON 저장, TTL = `JWT_EXPIRE_MINUTES * 60`
- `create_session()` / `resolve_session()` / `delete_session()` / `refresh_session()` 제공
- Valkey 오류 / 세션 없음 / 만료 → anonymous 반환 (fail-closed)

**상태**: ✅ 수정 완료

## 5. Low 결함

### [S-9] 검색 엔드포인트에 Rate Limiting 없음

**파일**: `backend/app/api/v1/search.py`  
**OWASP**: A05 — Security Misconfiguration

FTS 검색은 DB 집약적 연산이다. Rate limiting 없이 익명 공개 접근이 허용되므로,  
반복 쿼리로 DB 리소스를 소진할 수 있다.

**수정**: `slowapi` + Valkey `RedisStorage` 백엔드로 분산 rate limiting 구현.
- `app/api/rate_limit.py` — `Limiter` 싱글톤 (Valkey 미연결 시 메모리 폴백)
- 검색 4개 엔드포인트에 `@limiter.limit("30/minute")` 적용 (IP 기준)
- 초과 시 HTTP 429 반환
- `app.state.limiter` 주입 및 `RateLimitExceeded` 핸들러 등록

**상태**: ✅ 수정 완료

### [S-10] Bearer 토큰 자체 미검증 (`X-Actor-Id`로만 인증)

**파일**: `backend/app/api/auth/dependencies.py` — `_extract_bearer_actor()`  
**OWASP**: A07 — Identification and Authentication Failures

~~현재 Bearer 토큰 문자열 자체는 검증하지 않는다. `X-Actor-Id` 헤더가 있으면 해당 ID로 인증된다.~~

**수정**: `_extract_bearer_actor()`를 JWT 서명 검증 방식으로 전면 교체.
- `PyJWT==2.10.1` 의존성 추가 (`requirements.txt`)

- `sub` → actor_id, `role` → 역할 추출
- `jwt_secret` 미설정 시: debug 모드에서만 개발 헤더 폴백, 그 외 anonymous 반환
- `app/api/auth/tokens.py` 추가: `create_access_token()` / `decode_access_token()` 발급·검증 유틸

**JWT 클레임 규약**:
```json
{ "sub": "<actor_id>", "role": "AUTHOR", "iat": 1712600000, "exp": 1712607200 }
```

**상태**: ✅ 수정 완료
