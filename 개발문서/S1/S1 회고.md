## 2. Keep — 유지할 것

### 2.1 "설계 원칙 4개"의 일관된 관철

Phase 0에서 확정한 API-first / Dual UI / Document+Type / 모든 계층 권한 4개 원칙이 15개 Phase 전 구간에서 흔들리지 않았다. 특히:

- **문서 타입 하드코딩 금지** 규칙이 Phase 12의 `DocumentTypeRegistry` 플러그인 체계로 구조화되면서, Phase 14 인증/관리 기능을 추가할 때도 분기 없이 자연스럽게 흡수됐다.
- **API-first**가 지켜졌기 때문에 Phase 14 에서 프런트엔드 14개 페이지를 동시 착수할 수 있었다(API 계약이 선행 확정).

### 2.2 "검수 + 보안 보고서" 이중 게이트

CLAUDE.md의 "개발 후 검수 / 보안 검사 보고서 작성" 규칙이 **품질 바닥선**을 만들었다. Phase별로 평균 4~11건의 결함·취약점이 드러났고, 전부 해당 Phase 내에서 봉합됐다. Phase 13의 11건, Phase 14의 4건 모두 "다음 Phase로 넘기지 않음"이라는 규율을 지켰다.

### 2.3 Phase 14의 회귀 게이트 (`test_integration_security_phase14_16.py`)

10 카테고리 41 항목의 통합 검증 스크립트가 **회귀 방어선**으로 자리잡았다. 검수 4건 수정 후 재실행으로 무회귀를 즉시 입증했다. S2 이후 리팩토링·기능 추가 시 그대로 사용 가능한 자산이다.

### 2.4 방어적 심층 보안 (Defense-in-Depth)

- RBAC `_PERMISSION_MATRIX` 화이트리스트 + API Key scope + 프런트 `AuthGuard(requiredRole)` 삼중 방어.
- RT Rotation + family 폐기 + 블랙리스트로 탈취 복원력 확보.
- OAuth 콜백에서 이메일 자동 연결을 과감히 포기(VULN-P14-01 수정). 편의보다 안전성 선택.

### 2.5 "하위호환 + 새 경로"로 흡수한 후기 변경

Phase 14 마감 직후 아이디 로그인 요구가 들어왔을 때, `LoginRequest`에 `@model_validator(mode="before")`로 `email → identifier` 별칭 변환을 넣어 기존 클라이언트/테스트를 깨지 않고 확장했다. "스프린트 후기에 요구가 바뀌어도 부술 필요 없다"는 좋은 선례.

## 3. Problem — 아팠던 지점

### 3.1 공식 사인오프 이후에야 드러난 환경 의존 이슈 5건

Phase 14 검수/보안 사인오프 완료 후에도 아래 5건이 실사용 부팅·UI 조작에서 터졌다:

1. `seed_users.py` 단독 실행 시 `users` 테이블 부재 → `UndefinedTable`
2. pgvector 미설치 환경에서 `init_db()` 전체 롤백
3. `gpt-4o` 기본 모델 권한 부재 → RAG 채팅 전원 실패
4. 로그인에 아이디 경로 없음, 회원가입 진입 부재 (기능 범위 확장)
5. `documentsApi.create/update` 가 envelope를 언래핑하지 않아 `doc.id = undefined` → 후속 호출 500

**공통 원인**:

- 단위 테스트와 통합 검증은 "깨끗하게 조립된 상태"를 가정한다. "처음 boot" / "확장 없음" / "권한 없음" / "envelope 경계" 같은 경계 조건은 게이트를 통과한다.
- 프런트-백엔드 응답 envelope 계약(`{data, meta}`)이 `adapt*` 함수마다 제각기 언래핑되고 있어 **통일된 클라이언트 계층**이 없다.
- 환경 의존(pgvector, OpenAI 모델 권한, SMTP)에 대한 graceful degrade 정책이 Phase별로 다르다.

### 3.2 API envelope 언래핑 산발

`documentsApi.get` 은 언래핑, `create/update` 는 패스스루 — 한 파일 안에서도 제각각이었다. `versions.ts`에도 동일 패턴이 잠복해 있다. 각 `*Api` 파일마다 수작업 언래핑을 반복하는 구조 자체가 리스크다.

### 3.3 Phase 단위의 긴 작업 지시서 블록이 누적되며 "맥락 길이"와 싸움

Phase 14만 16 task에 검수보고서 10개 · 보안보고서 10개 · 산출물 5개로, 후반 검수 단계에서 컨텍스트가 압축되는 구간이 있었다. 맥락이 요약되면서 "어느 결함이 어느 Task에 속하는지" 추적에 여러 번 cross-read가 필요했다.

### 3.4 개발 PG · 운영 PG 간 확장(extension) 격차

pgvector는 RAG 핵심이지만 개발 환경에서는 선택적이다. Phase 10부터 의존이 생겼음에도 "미설치 시 나머지 DDL 전체가 죽는다"가 Phase 14 말기에야 드러났다. 의존이 생긴 시점에 degrade 전략을 설계했어야 한다.

### 3.5 "시드 계정 기본 비밀번호 노출" 운영 리스크

`seed_users.py`가 기본값 `Admin!2345` / `User!2345`를 CLI로 찍어준다. 개발 편의상 유지했지만, 실수로 production에서 실행되는 것을 코드 레벨에서 막진 않는다.

### 3.6 UI 5회 리뷰의 품질 편차

CLAUDE.md 규칙대로 Phase별 UI 5회 리뷰는 수행했으나, 실제 사용자 플로우(회원가입 진입, 아이디 로그인 등)가 Phase 14 마감 후에 요구됐다. "5회 리뷰"가 **체크박스화**되지 않았는지 돌아볼 필요가 있다.

## 4. Try — S2에서 시도할 것

### 4.1 "첫 부팅 스모크 테스트" 도입 [우선순위 High]

CI에 `fresh-boot` 잡을 추가한다. 절차:

1. 빈 Postgres 컨테이너 기동 (pgvector 있음/없음 2개 매트릭스)
2. `python -m scripts.seed_users` 실행 (환경변수 없이)
3. `uvicorn app.main:app` 10초 헬스체크
4. `curl /api/v1/auth/login` + `curl /api/v1/documents (POST)` + `GET` 로 문서 UUID 라운드트립 검증

§3.1의 5건 중 4건은 이 게이트로 사전 차단 가능하다.

### 4.2 API 클라이언트 계층 통일 — `unwrapEnvelope<T>()` 단일화

`frontend/src/lib/api/client.ts`에 제네릭 `unwrapEnvelope<T>(raw): T` 헬퍼를 추가하고, `request<T>` 단계에서 자동 언래핑 옵션(`{ unwrap?: boolean }`)을 지원한다. 모든 `*Api` 파일이 수작업 `r.data ?? raw`를 버린다.

병행으로 backend `success_response` 계약을 `docs/기술서/`에 명문화하여 "envelope은 벗기는 것이 기본"을 규약화한다.

### 4.3 외부 의존(pgvector / OpenAI 모델 / SMTP)의 graceful-degrade 패턴 표준화

각 의존마다:

- SAVEPOINT 또는 선행 availability check로 "없으면 degrade, 있으면 활성"을 1개 지점에서 결정.
- 런타임에 활성 상태를 반환하는 `/api/v1/system/capabilities` 엔드포인트 신설. 프런트가 RAG 버튼 활성/비활성을 이 신호로 제어.
- `/health`와 별개로 "degrade 상태"를 관측 가능한 메트릭으로 노출.

### 4.4 Phase 단위 → Feature 단위 소분 구조

Phase 14처럼 16 task가 1개 Phase에 묶이면 검수 후반 맥락 압축이 심하다. S2에서는 한 Phase를 2~3개 "Feature Group"으로 분할하여 각 그룹마다 독립적인 검수 게이트를 돌리는 것을 시도한다. 맥락 길이를 물리적으로 자르는 효과.

### 4.5 Production-safe 시드 스크립트

`seed_users.py`:

- `ENVIRONMENT=production` 이면 **기본 비밀번호 차단**, `SEED_*_PASSWORD` 환경변수가 없으면 즉시 실패.
- 출력에서 비밀번호 전체 대신 마지막 4자만 표시.
- 감사 로그(`audit_events`)에 `SEED_USER_CREATED` 이벤트 남김.

### 4.6 UI "리뷰 체크리스트"의 유저 여정 기반화

"5회 리뷰"의 각 회차에 구체 목표를 부여한다:

1. 첫 방문 / 미로그인 상태 여정
2. 로그인 경로 (이메일 / 아이디 / OAuth 각각)
3. 에러 상태 (권한 부족, 세션 만료, 네트워크 오류)
4. 키보드 전용 접근성 / 스크린리더
5. 모바일·데스크탑·태블릿 레이아웃

Phase 14 후반에 나온 "아이디 로그인 / 회원가입 진입" 요구는 (1)(2)가 체계화됐다면 사전에 포착 가능했다.

### 4.7 회귀 게이트의 지속 확장

`test_integration_security_phase14_16.py`가 잘 작동하는 만큼, S2에서도 **각 Feature Group별 통합 검증 스크립트**를 의무화한다. 게이트가 작을수록 실패를 정확히 국소화할 수 있다.

## 6. 개인 회고 (작업자 시선)

S1은 "**원칙을 끝까지 지켰더니 확장이 싸게 먹히더라**"를 실증한 스프린트였다. Phase 0의 4개 원칙과 CLAUDE.md의 6개 코드/UI 규칙이 Phase가 거듭될수록 "설계 질문을 미리 끝내준 문서"로 작동했다.

반면 아픈 지점은 일관되게 **"시스템 바깥의 경계"** 였다: 설치되지 않은 확장, 권한 없는 모델, 초기 부팅, 프론트-백 계약의 미세한 어긋남. 코드 내부의 결함은 게이트가 잘 잡아냈지만, 경계는 "사용해봐야" 드러났다. S2에서는 경계를 더 일찍 건드려보는 쪽으로 엔지니어링 자원을 배분한다.

마지막으로, 실사용 검증 단계에서 5건이 연달아 드러났을 때 "Phase 14 완료 취소"가 아니라 "범위 안에서 봉합"으로 대응한 것은 옳은 선택이었다고 본다. 원칙과 게이트가 건재한 상태에서, 경계 이슈는 S2의 Try 항목으로 흡수하는 것이 스프린트 회계상 정확하다.
