

# Task 4-1 작업지시서

## 작업명

문서 작성/개정 생명주기 정의

---

## 1. 작업 목적

Phase 4의 첫 번째 작업으로, 문서 플랫폼에서 문서가 생성되고 수정되며 Draft와 Published 상태를 거쳐 버전으로 관리되고 복원되는 전체 생명주기를 정의한다.

이 작업의 목적은 다음과 같다.

- 문서 작성 흐름의 표준 상태 전이를 확정한다.
- Document와 Version의 역할 차이를 실제 운영 흐름에 연결한다.
- Draft / Published 관리 기준을 이후 상세 설계와 API 설계의 기준선으로 삼는다.
- 버전 생성, 발행, 복원 시점의 정책을 명확히 하여 이후 구현 혼선을 줄인다.

이 문서는 이후 Task 4-2, 4-4, 4-5, 4-6의 선행 기준 문서 역할을 해야 한다.

---

## 2. 작업 배경

앞선 Phase에서 다음이 이미 정리되었다고 가정한다.

- 플랫폼은 API-first 원칙을 따른다.
- User UI와 Admin UI는 분리된다.
- 문서 도메인은 Document / Version / Node 구조를 중심으로 설계된다.
- 권한, 감사 추적, 공통 API 계층은 별도 Phase에서 정의되었다.

이번 작업에서는 그 위에 실제 문서 생명주기를 얹어, “문서를 어떻게 생성하고, 어떻게 개정하며, 언제 버전이 생기고, 어떤 기준으로 발행되고, 어떻게 복원되는가”를 설계해야 한다.

## 4. 핵심 설계 질문

이번 작업에서는 최소한 아래 질문들에 답할 수 있어야 한다.

1. 문서가 처음 생성될 때 Document와 Draft Version은 어떻게 만들어지는가?
2. 문서 수정은 항상 Draft를 기준으로 하는가, 아니면 Published에서 바로 새 Draft를 파생시키는가?
3. Draft는 문서당 하나만 허용하는가, 여러 개를 허용하는가?
4. Published 전환 시 기존 Published는 어떻게 처리되는가?
5. 버전은 저장할 때마다 생성되는가, 특정 시점에만 생성되는가?
6. 복원은 과거 버전을 덮어쓰는가, 아니면 새 Draft/새 Version을 생성하는가?
7. “현재 문서”라는 개념은 무엇을 가리키는가? 최신 Draft인가, 최신 Published인가, 둘 다 별도 관리하는가?
8. 문서가 아직 한 번도 Published되지 않은 경우 조회 기준은 무엇인가?
9. 삭제 또는 폐기 가능한 상태와 불가능한 상태는 무엇인가?
10. 어떤 상태 전이가 허용되고 어떤 전이는 금지되는가?

## 5. 작업 지시

아래 항목을 순서대로 설계 문서로 정리한다.

### 5-1. 생명주기 엔티티 관점 정리

다음 개념을 먼저 명확히 구분한다.

- Document
- Draft Version
- Published Version
- Restored Version
- Current Document View

각 개념에 대해 아래를 작성한다.

- 정의
- 책임
- 다른 개념과의 관계
- 변경 가능 여부
- 조회 기준에서의 역할

---

### 5-2. 문서 상태 흐름 정의

문서 단위에서 어떤 상태를 가질지 정의한다.

예시로 아래 같은 상태를 검토하되, 필요 시 조정 가능하다.

- Draft only
- Published
- Draft + Published 공존
- Archived
- Deleted 또는 Soft deleted

각 상태에 대해 다음을 정리한다.

- 상태 의미
- 진입 조건
- 이탈 조건
- 허용 행위
- 금지 행위

단, 상태 수를 불필요하게 늘리지 말고 운영상 의미가 분명한 상태만 남긴다.

---

### 5-3. 버전 생성 규칙 정의

버전이 언제 생성되는지 정책을 정의한다.

반드시 다음 관점을 다룬다.

- 문서 최초 생성 시 버전 생성 여부
- Draft 저장 시 버전 번호 증가 여부
- Published 전환 시 별도 버전 생성 여부
- 복원 시 새 버전 생성 여부
- 단순 메타데이터 수정과 본문 수정의 버전 처리 차이 여부

여기서는 구현 편의보다 이력 무결성과 운영 명확성을 우선한다.

---

### 5-4. Draft / Published 관계 정의

다음 정책을 명시한다.

- Draft는 문서당 하나인지, 여러 개인지
- Published는 항상 하나인지
- Published 이후 수정이 발생하면 어떤 흐름으로 새 Draft가 생기는지
- Draft와 Published가 동시에 존재할 수 있는지
- 사용자가 보는 기본 조회 대상은 무엇인지

여기서는 향후 승인/배포 체계 확장을 고려하여, 지나치게 단순하지만 확장 불가능한 구조는 피한다.

---

### 5-5. 복원 흐름 정의

버전 복원은 반드시 별도 흐름으로 정리한다.

다음 내용을 포함한다.

- 복원의 정의
- 복원의 입력과 결과
- 복원 대상 제약
- 복원 후 생성되는 객체
- 복원 이후 Draft / Published 상태 변화
- 감사 추적 필요 포인트

중요: 복원은 과거 버전을 직접 변경하거나, 과거 시점으로 시스템 전체를 되돌리는 개념이 아니라, **과거 버전을 기반으로 현재 시점의 새 작업 상태를 만드는 행위**로 정리하는 방향을 우선 검토한다.

---

### 5-6. 대표 시나리오 작성

최소한 아래 시나리오를 작성한다.

1. 새 문서 작성 후 Draft 저장
2. Draft 문서를 첫 Published로 전환
3. Published 문서를 수정하여 새 Draft 생성
4. 새 Draft를 다시 Published로 전환
5. 과거 Published 버전을 기반으로 복원 시작
6. Draft가 있는 상태에서 수정 반복
7. 아직 Published되지 않은 문서 조회
8. Published된 문서의 일반 조회

각 시나리오는 다음 형식으로 작성한다.

- 시작 조건
- 사용자 행위
- 시스템 반응
- 상태 변화
- 생성/갱신되는 엔티티
- 감사 이벤트 포인트

---

### 5-7. 상태 전이표 작성

최종적으로 상태 전이표를 만든다.

표에는 최소한 다음 열이 포함되어야 한다.

- 현재 상태
- 수행 액션
- 전이 가능 여부
- 다음 상태
- 비고

예:

- Draft only → Publish → 가능 → Draft + Published 또는 Published
- Published → Edit → 가능 → Draft + Published
- Published → Delete permanently → 원칙적으로 제한 또는 금지

전이표는 이후 API 권한 제어와 테스트 케이스 설계의 기반이 되어야 한다.

## 7. 산출물 작성 원칙

- 구현 코드 작성이 아니라 설계 문서 작성에 집중한다.
- 과도한 추측을 피하고, 필요한 가정은 명시적으로 적는다.
- 이후 API 설계와 DB 설계로 자연스럽게 이어질 수 있도록 용어를 일관되게 사용한다.
- “현재는 단순화하되, 이후 확장 가능해야 한다”는 원칙을 유지한다.
- 사용자 관점 서술과 시스템 관점 서술을 혼합하지 말고 구분해서 작성한다.

## 8. 권장 방향

아래 방향을 우선안으로 검토한다.

### 우선안 A

- Document는 문서 식별자와 현재 활성 포인터를 가진다.
- Version은 스냅샷 단위로 관리한다.
- Published는 문서당 1개만 공식 활성 상태를 가진다.
- 수정은 기존 Published를 직접 바꾸지 않고 새 Draft를 통해 진행한다.
- 복원은 과거 Version을 기반으로 새 Draft 또는 새 Version을 생성한다.

이 우선안은 향후 승인 워크플로, diff, 감사 추적, AI 작성 보조와 연결하기 가장 안정적이다.
