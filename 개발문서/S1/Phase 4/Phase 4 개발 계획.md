# Phase 4. 문서 작성, 개정, 버전 관리 코어 구축

## 1. Phase 목적

문서 플랫폼의 핵심 사용 흐름인 다음 기능을 구현한다.

- 새 문서를 생성한다.
- 기존 문서를 수정한다.
- 수정 결과를 버전으로 남긴다.
- Draft와 Published 상태를 구분한다.
- 과거 버전을 조회하고 필요 시 복원한다.
- 저장된 문서 구조를 안정적으로 렌더링한다.

즉, 이 단계의 목적은  
**"문서를 단순 저장하는 시스템"이 아니라, "정식 문서 생명주기를 관리하는 시스템"의 중심부를 완성하는 것**이다.

## 2. Phase 목표

### 기능 목표

- 문서 생성/수정 API 및 내부 서비스 구조 확립
- 문서 버전 생성 규칙 확립
- Draft / Published 전환 흐름 정의
- 버전 목록/상세/복원 기능 정의
- 문서 구조 저장 포맷과 렌더링 입력 포맷 정리

### 제품 목표

- 사용자가 워드 파일 없이도 플랫폼 안에서 문서를 작성·개정·관리할 수 있는 기반 확보
- 외부 UI, 관리자 UI, 향후 AI 보조 기능이 모두 동일한 버전 구조를 사용하도록 표준화
- 문서 변경 이력 추적과 책임성을 확보

### 기술 목표

- Document / Version / Node 모델을 실제 쓰기 흐름에 연결
- API-first 방식으로 작성/개정 흐름을 서비스화
- 이후 협업 편집, 승인, 비교(diff), AI 초안 생성 기능이 얹히기 쉬운 구조 확보

## 4. 핵심 설계 원칙

### 4-1. Document와 Version은 분리한다

- Document는 "문서의 식별자와 현재 상태를 대표하는 엔티티"
- Version은 "특정 시점의 확정된 문서 스냅샷"

즉, 문서를 고칠 때마다 Document 자체를 덮어쓰는 것이 아니라,  
**Version을 중심으로 변경 이력을 축적**해야 한다.

### 4-2. Draft와 Published를 명확히 구분한다

- Draft는 작업 중인 상태
- Published는 공식 조회 기준 상태

이 구분이 있어야 문서 작성과 공표가 분리되고,  
향후 승인/배포 체계를 자연스럽게 얹을 수 있다.

### 4-3. 복원은 "과거 버전을 현재로 되돌리는 새 버전 생성"으로 본다

- 과거 버전을 직접 수정하는 방식은 피한다.
- 복원은 과거 스냅샷을 기반으로 새 Draft 또는 새 Version을 만드는 방식이어야 한다.

이렇게 해야 감사 추적과 이력 일관성이 유지된다.

### 4-4. 렌더링은 저장 포맷과 느슨하게 결합한다

- 내부 저장 구조와 UI 표시 구조를 완전히 동일시하지 않는다.
- 렌더링용 DTO / view model 계층을 둔다.

이 원칙이 있어야 나중에 웹 UI, 모바일 UI, PDF export, AI 요약 뷰를 쉽게 붙일 수 있다.

### 4-5. API는 UI 전용이 아니라 플랫폼 전용이어야 한다

- 사용자 UI
- 관리자 UI
- 외부 시스템
- AI / RAG 서비스

모두 동일한 문서 작성/버전 API를 사용하도록 설계해야 한다.

## 5. 하위 작업 구조

## Task 4-1. 문서 작성/개정 생명주기 정의

**목적**  
문서 생성부터 수정, 발행, 복원까지의 전체 상태 흐름을 정의한다.

**주요 작업**

- 문서 작성 시나리오 정리
- 문서 개정 시나리오 정리
- Draft → Published 전환 규칙 정의
- 수정 가능한 대상과 불변 대상 구분
- 버전 생성 시점 규칙 정의
- 복원 시나리오 정의

**산출물**

- 문서 생명주기 정의서
- 상태 전이 규칙
- 핵심 시퀀스 다이어그램

---

## Task 4-2. 버전 모델 상세 설계

**목적**  
Version 엔티티와 관련 규칙을 실제 구현 가능한 수준으로 구체화한다.

**주요 작업**

- Version 필드 정의
- version_number / revision_id 전략 정의
- current draft / current published 연결 방식 정의
- version lineage(부모 버전 참조 여부) 검토
- 스냅샷 범위 정의
- 메타데이터 포함 범위 정의

**산출물**

- Version 상세 모델 명세
- 버전 식별 및 증가 규칙
- 스냅샷 규칙 문서

---

## Task 4-3. 문서 저장 구조와 편집 단위 설계

- 문서 구조 직렬화 포맷 정의
- 헤더/본문/부록/표/리스트 등 기본 블록 범위 정의
- 유효성 검증 규칙 정의

**산출물**

- 문서 본문 저장 포맷 명세

## 8. 완료 기준

다음 조건을 만족하면 Phase 4 완료로 볼 수 있다.

### 설계 완료 기준

- 문서 생성, 수정, 발행, 복원 흐름이 끊김 없이 정의되어 있다.
- Draft와 Published 상태 전환 규칙이 명확하다.
- Version 모델과 스냅샷 범위가 확정되어 있다.
- 저장 구조와 렌더링 구조의 관계가 정리되어 있다.
- API 명세가 UI/외부 시스템/AI 연계를 고려해 작성되어 있다.

### 제품 완료 기준

- 플랫폼이 워드 파일 없이도 문서 작성과 개정을 다룰 수 있는 구조를 갖춘다.
- 버전 조회와 복원이 공식 기능으로 성립한다.
- 향후 승인, 비교(diff), 협업 편집, AI 보조 작성 기능의 기반이 마련된다.

## 9. 리스크와 주의점

### 리스크 1. Draft와 Published 개념이 섞일 위험

이게 섞이면 나중에 승인/배포 체계가 꼬인다.  
처음부터 둘을 엄격히 분리해야 한다.

### 리스크 2. 저장 포맷이 UI에 과도하게 종속될 위험

특정 에디터 UI 구조를 그대로 저장하면 나중에 교체가 어렵다.  
저장 포맷은 도메인 중심이어야 한다.

### 리스크 3. 복원 기능이 이력 무결성을 깨뜨릴 위험

"과거 버전 덮어쓰기" 방식은 피해야 한다.  
반드시 새 버전 생성 방식으로 가야 한다.

### 리스크 4. 버전 수 증가에 따른 조회 성능 문제

초기부터 목록 조회, 최신 버전 참조, published 조회 경로를 분리해두는 것이 좋다.

### 리스크 5. 작성 권한과 발행 권한이 혼동될 위험

보안/컴플라이언스 관점에서 작성과 공표는 다를 수 있다.  
Phase 2, 3과 반드시 연결해서 봐야 한다.
