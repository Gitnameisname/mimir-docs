

# Task 6-7. 승인 / 검토 UI 설계

## 1. 작업 목적

검토자와 승인자가 문서 워크플로에 참여할 수 있도록 **승인 / 검토 UI**를 설계한다.

이 작업의 핵심 목표는 다음과 같다.

- Review 단계에 있는 문서를 효율적으로 검토할 수 있게 한다
- 승인 / 반려 / 의견 입력 흐름을 명확하게 제공한다
- 문서 상태 전이와 역할 기반 액션을 UI에서 안전하게 제어한다

---

## 2. 작업 범위

### 포함 범위

- 검토 대상 문서 목록 UI 설계
- 승인 / 반려 / 의견 입력 UI 설계
- 상태 전이 액션 UI 설계
- 워크플로 이력 표시 UI 초안 정의

### 제외 범위

- 실제 승인 로직 구현
- 감사 로그 상세 화면 구현

---

## 4. 주요 설계 대상

### 4-1. 검토 대상 목록 화면

검토자 또는 승인자가 현재 처리해야 할 문서를 볼 수 있는 화면을 설계한다.

#### 구성 요소

- Page Header (예: My Tasks / Reviews)
- 검토 대상 문서 리스트
- 상태 / 우선순위 / 요청일 기준 정보 표시
- 검색 / 필터 / 정렬 기능

#### 표시 항목

- 문서 제목
- 현재 상태
- 검토 요청자
- 요청일
- 문서 유형

---

### 4-2. 검토 상세 진입 흐름

검토 대상 목록에서 특정 문서를 선택하면 검토용 상세 화면으로 진입한다.

#### 요구사항

- 문서 본문 열람 가능
- 메타데이터 및 현재 상태 확인 가능
- 버전 정보 확인 가능
- 승인 / 반려 / 의견 입력 액션과 자연스럽게 연결

---

### 4-3. 승인 / 반려 액션 UI

문서 상태와 권한에 따라 승인 또는 반려 액션을 제공한다.

#### 주요 액션

- Approve
- Reject / Return for Revision
- Add Comment / Review Note

#### UX 요구사항

- 상태 전이에 따라 가능한 액션만 노출
- 위험한 액션은 Confirm Dialog 제공
- 반려 시 사유 입력을 요구할 수 있도록 설계

---

### 4-4. 검토 의견 입력 UI

검토자는 문서에 대해 의견을 남길 수 있어야 한다.

#### 입력 방식

- 전체 문서 기준 코멘트
- 상태 변경과 함께 의견 저장

#### 요구사항

- 승인과 반려 모두에 의견 첨부 가능
- 의견 없이도 가능한 액션과 필수 의견 액션을 구분

---

### 4-5. 상태 전이 UI 정책

UI는 현재 문서 상태와 사용자 역할에 따라 가능한 상태 전이만 허용해야 한다.

#### 예시

- Draft → Review Request
- Review → Approve
- Review → Reject / Return
- Approved → Published (후속 정책에 따라)

#### 요구사항

- 허용되지 않은 액션은 숨김 또는 비활성화
- 불가능한 경우 이유 안내 가능

---

### 4-6. 워크플로 이력 표시 UI 초안

문서가 어떤 흐름을 거쳐 왔는지 사용자가 이해할 수 있도록 이력 표시 UI를 정의한다.

#### 표시 항목

- 상태 변경 시각
- 변경 주체
- 변경 결과
- 사유 또는 코멘트

#### UI 형태

- Timeline 형태 권장

---

### 4-7. 상태별 UI 처리

#### Loading

- 검토 대상 목록 / 상세 로딩 상태 표시

#### Empty

- 검토할 문서 없음 안내

#### Error

- 목록 로드 실패 또는 액션 실패 메시지

#### Permission Denied

- 승인 권한 없음 안내
