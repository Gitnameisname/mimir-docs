# Phase 6. 사용자 UI 구축 개발 계획

## 1. 단계 목적

Phase 6의 목적은 Mimir의 핵심 기능을 일반 사용자가 실제 업무에서 활용할 수 있도록 **사용자 전용 UI(User UI)** 를 구축하는 것이다.

이 단계에서 구현하는 UI는 단순 화면 구성이 아니라, 다음을 만족해야 한다.

- 문서를 쉽게 찾고
- 문서 구조를 이해하며 읽고
- 권한에 따라 작성/편집하고
- 버전과 변경 이력을 확인하며
- 검토/승인 프로세스에 참여하고
- 후반부에는 RAG 기반 질의 기능까지 연결할 수 있는

업무형 문서 인터페이스를 제공해야 한다.

즉, Phase 6은 “API가 준비된 플랫폼”을 “실제 사용 가능한 제품 UI”로 전환하는 단계다.

---

## 2. 단계 목표

Phase 6에서 달성해야 할 주요 목표는 다음과 같다.

1. **사용자 중심 정보 구조 확립**
   - 일반 사용자가 문서를 탐색하고 작업하기 쉬운 화면 흐름 정의
   - 작성자/검토자/열람자의 주요 시나리오 반영
2. **문서 작업 전 주기 UI 제공**
   - 목록 조회
   - 검색
   - 상세 열람
   - 구조 탐색
   - 편집
   - 버전 확인
   - 검토/승인
3. **권한/상태 기반 UI 제어**
   - 문서 상태와 사용자 역할에 따라 가능한 액션을 제한
   - 버튼 노출, 편집 가능 여부, 승인 액션 범위 반영
4. **API-first 원칙에 맞는 UI-API 연동 구조 수립**
   - UI는 독립 로직이 아니라 플랫폼 API의 클라이언트로 동작
   - 이후 Admin UI, 외부 앱, AI 연계와도 일관된 구조 유지
5. **후반 Phase 확장을 고려한 UI 골격 마련**
   - Search 강화(Phase 8)
   - Diff 가시화(Phase 9)
   - RAG 연계(Phase 11)
   - DocumentType 확장(Phase 12)
   - 를 수용할 수 있도록 설계

---

## 3. 범위

Phase 6의 범위는 아래와 같다.

### 포함 범위

- 사용자용 레이아웃 및 내비게이션
- 문서 목록/검색 화면
- 문서 상세 보기 화면
- 구조 탐색 트리 UI
- 문서 작성/편집 UI
- 버전 관리 UI
- 승인/검토 UI
- 상태/권한 기반 액션 제어
- 공통 사용자 피드백 UI
- 후반 연계 가능한 RAG 질의 UI 초안 또는 연결 포인트

### 제외 또는 후속 연계 범위

- 검색 엔진 고도화 자체 구현은 Phase 8 중심
- 버전 diff 시각화의 본격 기능은 Phase 9 중심
- 벡터화/임베딩 파이프라인은 Phase 10
- 실제 고급 RAG 응답 엔진은 Phase 11
- 관리자용 정책/조직 관리 UI는 Phase 7

즉, Phase 6은 사용자 인터페이스 관점의 골격과 업무 흐름 확보에 집중한다.

## 4. 설계 원칙

Phase 6 UI는 아래 원칙을 따라야 한다.

### 4-1. User UI와 Admin UI의 명확한 분리

- 일반 사용자가 보는 화면만 다룬다
- 운영/정책/시스템 제어 기능은 Admin UI로 넘긴다

### 4-2. 문서 중심 UX

- 파일 중심이 아니라 문서 객체(Document / Version / Node) 중심으로 설계
- 문서 구조, 상태, 버전, 권한이 UI에서 자연스럽게 드러나야 한다

### 4-3. 역할과 상태에 따른 행동 제어

- 작성자, 검토자, 열람자마다 액션이 달라야 한다
- Draft / Review / Approved / Published 상태에 따라 편집 및 승인 가능 범위가 달라져야 한다

### 4-4. API-first 정합성 유지

- UI 편의 로직이 API 계약을 우회하지 않도록 구성
- 모든 핵심 액션은 API와 1:1 대응되는 것이 원칙

### 4-5. 확장 가능한 화면 구조

- 추후 문서 유형별 전용 editor/renderer가 붙을 수 있어야 함
- 검색 고도화, diff, RAG 패널이 나중에 자연스럽게 확장 가능해야 함

## 5. 사용자 시나리오

Phase 6은 다음 사용자 시나리오를 지원해야 한다.

### 시나리오 A. 열람자

- 문서 목록에서 원하는 문서를 찾는다
- 검색 또는 필터로 문서를 좁힌다
- 문서 상세 화면에서 내용을 읽는다
- 구조 트리로 특정 섹션으로 이동한다
- 현재 버전과 상태를 확인한다

### 시나리오 B. 작성자

- 새 문서를 생성한다
- 문서 내용을 구조적으로 편집한다
- Draft 상태에서 저장과 수정 반복
- 검토 요청 전 버전 상태 확인
- 필요 시 이전 버전 참조

### 시나리오 C. 검토자

- Review 대상 문서를 확인한다
- 문서 내용을 검토한다
- 검토 의견, 승인, 반려 액션을 수행한다
- 변경 사유와 버전 흐름을 확인한다

### 시나리오 D. 후반 확장 사용자

- 문서를 열람하면서 관련 질의를 수행한다
- 문서 기반 RAG 질의 결과를 확인한다
- 근거 문서/노드로 다시 이동한다

### Task 6-1. User UI 정보 구조 및 화면 맵 설계

#### 주요 작업

- User UI의 주요 메뉴/라우트 정의
- 문서 목록 → 상세 → 편집 → 버전 → 검토 흐름 설계
- 작성자/검토자/열람자별 주요 진입점 정의
- 공통 레이아웃 구조 정의
- 브레드크럼, 사이드 패널, 액션 바 위치 설계

#### 산출물

- User UI 화면 맵
- 라우팅 구조 초안
- 사용자 흐름도

---

### Task 6-2. 공통 UI 프레임워크 및 레이아웃 컴포넌트 설계

#### 목적

전체 User UI에서 재사용할 공통 구조를 마련한다.

#### 주요 작업

- 기본 레이아웃 설계
- 헤더/사이드바/콘텐츠 영역 구조 정의
- 공통 페이지 패턴 정의
- Empty / Loading / Error / Permission Denied 상태 정의
- 공통 액션 버튼, 상태 배지, 알림 패턴 정의

#### 산출물

- 공통 레이아웃 명세
- 공통 컴포넌트 목록
- 상태 표현 가이드

---

### Task 6-3. 문서 목록 및 검색 UI 설계

#### 목적

사용자가 문서를 빠르게 찾고 접근할 수 있는 기본 화면을 구성한다.

#### 주요 작업

- 문서 목록 테이블/카드 구조 정의
- 제목, 유형, 상태, 작성자, 수정일 등 표시 항목 정의
- 검색 입력/필터/정렬 UI 설계
- 권한 없는 문서 노출 정책 반영
- pagination 또는 infinite scroll 정책 정의

#### 산출물

- 문서 목록 화면 명세
- 검색/필터 UI 명세
- API 연동 포인트 정의

---

### Task 6-4. 문서 상세 보기 및 구조 탐색 UI 설계

#### 목적

문서를 읽고 구조적으로 탐색하는 핵심 화면을 설계한다.

#### 주요 작업

- 문서 상세 페이지 레이아웃 설계
- 본문 영역, 메타데이터 영역, 상태 표시 구성
- 트리 기반 구조 탐색 패널 설계
- 섹션 이동, 앵커 이동, 접기/펼치기 정책 정의
- 문서 상태/버전/권한 정보 노출 방식 정리

#### 산출물

- 문서 상세 보기 화면 명세
- 구조 트리 UI 명세
- 읽기 모드 UX 정의서

### Task 6-5. 문서 작성/편집 UI 설계

#### 목적

작성자가 문서를 실제로 생성하고 수정할 수 있는 편집 인터페이스를 구축한다.

#### 주요 작업

- 노드 추가/삭제/이동 UX 정의
- 자동 저장/임시 저장 정책 정의
- 저장 충돌 및 편집 잠금 여부 검토
- Draft 상태 중심 편집 정책 반영

#### 산출물

- 문서 편집 화면 명세
- 구조 편집 UX 정의
- 저장 정책 문서

---

### Task 6-6. 버전 관리 UI 설계

#### 목적

사용자가 문서 버전을 조회하고 작업 맥락을 이해할 수 있게 한다.

#### 주요 작업

- 버전 목록 표시 방식 정의
- 버전 메타데이터(번호, 작성자, 생성 시각, 사유 등) 표시
- 현재 버전/발행 버전 구분 방식 정의
- 특정 버전 상세 조회 진입점 정의
- 복원 가능 여부와 권한 노출 방식 정의

#### 산출물

- 버전 관리 화면 명세
- 버전 목록/상세 UI 정의
- 권한 기반 버전 액션 정책

#### 주요 작업

- 워크플로 이력 조회 UI 초안 정의

#### 산출물

- 승인/검토 화면 명세

- 상태 전이 UI 정책 문서

#### 산출물

- 버전 목록 조회
- Review 대상 확인
- 승인/반려 기본 액션
- 공통 오류/로딩/권한 처리

### MVP 제외 또는 후속

- 고급 검색 필터
- 버전 간 시각적 diff
- 고급 코멘트/협업 기능
- 실시간 공동 편집
- 고급 RAG 대화 UI
- 문서 유형별 전용 editor 최적화

## 9. 완료 기준

Phase 6 완료는 아래 기준을 만족해야 한다.

1. 일반 사용자가 문서를 목록에서 찾고 상세에서 읽을 수 있다
2. 구조 트리를 통해 문서를 구조적으로 탐색할 수 있다
3. 권한 있는 사용자가 Draft 문서를 편집하고 저장할 수 있다
4. 사용자가 문서의 버전 목록과 주요 상태를 확인할 수 있다
5. 검토자가 Review 상태 문서에 대해 검토/승인/반려 액션을 수행할 수 있다
6. 역할과 상태에 따라 UI 액션이 올바르게 제한된다
7. 공통 오류/로딩/권한 부족 상태가 일관되게 처리된다
8. 후반 Phase의 Search/Diff/RAG 기능을 붙일 수 있는 UI 확장 포인트가 확보된다

## 10. 기대 결과

Phase 6이 끝나면 Mimir는 더 이상 단순한 API 설계 결과물이 아니라, 실제 사용자들이 문서를 관리하고 검토할 수 있는 실무형 문서 플랫폼 UI를 갖추게 된다.

이 단계의 의미는 다음과 같다.

- 문서 플랫폼의 사용성이 가시화된다
- 작성/검토/열람 흐름이 실제로 검증된다
- 이후 Phase 7 Admin UI와 역할이 명확히 분리된다
- Phase 8~11의 검색, 비교, RAG 기능이 붙을 자리가 확보된다
