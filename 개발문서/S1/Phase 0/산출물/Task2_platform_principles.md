# Phase 0 — 플랫폼 원칙 선언 (Platform Principles)

## 문서 목적

이 문서는 Mimir 플랫폼의 설계·구현·운영 전반에서 판단 기준으로 사용할 원칙 집합이다. 기능 추가, 설계안 비교, API 설계, 데이터 모델 설계, UI 구조 결정, 저장 전략 수립 시 이 원칙을 먼저 확인한다.

이 원칙들은 선언으로 끝나지 않는다. 각 원칙은 위반 사례를 식별할 수 있을 만큼 구체적으로 작성되었으며, 설계 리뷰 체크리스트의 기반이 된다.

## 원칙 상세

---

### P1. API-first

**선언문**
모든 핵심 기능은 API로 먼저 설계하고 노출한다. UI는 API의 소비자 중 하나일 뿐이며, UI 구현이 API 설계를 이끌어서는 안 된다.

**의도**
UI 중심으로 개발하면 핵심 로직이 프론트엔드 코드에 종속되거나, 외부 시스템이 접근할 수 없는 구조가 만들어진다. API를 먼저 설계하면 UI, 외부 시스템, AI 도구가 동일한 인터페이스로 지식 자산에 접근할 수 있다.

**적용 범위**
API, Domain Model, UI/UX, Integration

**적용 예시**
- 문서 생성 기능을 먼저 `POST /documents` API로 설계하고, User UI는 이 API를 호출하는 방식으로 구현한다.
- 검색 기능을 `GET /search?q=...&type=...` API로 먼저 정의하고, UI 검색창은 이를 호출한다.

**위반 예시**
- 프론트엔드 컴포넌트 내부에서 DB 쿼리를 직접 실행한다.
- 특정 화면에서만 동작하는 문서 상태 변경 로직이 API 없이 UI 코드 내부에 있다.
- "일단 화면 먼저 만들고 나중에 API 붙이자"는 방식으로 진행한다.

**설계 시사점**
Phase 1 도메인 모델이 결정되면, 해당 모델에 대한 API 엔드포인트 설계가 모델과 동시에 정의되어야 한다. UI 설계는 그 이후다.

---

### P2. 범용 도메인 모델 우선

**선언문**
플랫폼 코어 모델은 특정 문서 유형에 종속되지 않는다. 새로운 문서 유형이 추가될 때 도메인 모델 전체를 수정하지 않고 수용할 수 있어야 한다.

**의도**
특정 유형(예: 규정집)에 최적화된 구조를 코어 모델로 만들면, 다른 유형이 추가될 때마다 모델이 흔들린다. 범용 구조 위에 유형별 확장을 올리는 방식이 장기적으로 더 안정적이다.

**적용 범위**
Domain Model, Storage, API

**적용 예시**
- `Document` 엔티티는 문서 유형과 무관하게 동일한 핵심 필드를 가지며, 유형별 특성은 `DocumentType`과 `Metadata`를 통해 표현한다.
- 새 문서 유형 "프로세스 정의서"를 추가할 때 `DocumentType` 정의만 추가하고, 도메인 모델 테이블을 변경하지 않는다.

**위반 예시**
- `regulation_document` 테이블, `manual_document` 테이블을 유형별로 별도 생성한다.
- 특정 유형에만 존재하는 필드를 `Document` 엔티티에 직접 추가한다.
- 코어 API 응답 구조가 특정 문서 유형의 필드 집합과 동일하게 설계된다.

**설계 시사점**
Phase 1에서 `Document`, `Node`, `Metadata`, `DocumentType` 엔티티를 설계할 때, 유형 종속 필드와 범용 필드를 명확히 분리해야 한다.

### P3. User / Admin 경험 명시적 분리

**선언문**
일반 사용자 기능과 운영·관리 기능은 처음부터 명시적으로 분리한다. 운영 기능이 사용자 화면에 조건부로 노출되는 방식은 허용하지 않는다.

**의도**
운영 기능이 사용자 화면 내부에 섞이면, 보안 경계가 불명확해지고 UI 복잡도가 증가하며, 권한 모델이 불필요하게 복잡해진다. 분리된 구조는 책임 범위를 명확히 하고 각 영역의 독립적 진화를 가능하게 한다.

**적용 범위**
UI/User, UI/Admin, API, Governance

**적용 예시**
- User UI는 `/app/*` 경로, Admin UI는 `/admin/*` 경로로 라우팅 영역을 분리한다.
- 문서 유형 관리, 권한 정책 수정, 감사 로그 조회는 Admin UI에서만 접근 가능하다.
- API도 `/api/v1/documents` (사용자용) 와 `/api/v1/admin/document-types` (관리자용)으로 경로를 분리한다.

**위반 예시**
- 관리자인 경우 동일한 문서 화면에 "삭제 정책 설정" 버튼이 조건부로 나타난다.
- User UI 컴포넌트 내부에 `if (isAdmin)` 분기로 운영 기능이 삽입된다.
- 사용자 문서 편집 API와 문서 유형 관리 API가 동일한 권한 범위를 공유한다.

**설계 시사점**
Phase 2 이후 UI 설계 시, User UI와 Admin UI를 별도 애플리케이션 또는 명확히 분리된 라우팅 트리로 구성한다.

### P4. 도메인 모델과 저장 구조 분리

**선언문**
도메인 엔티티는 저장 방식(테이블 구조, JSON 컬럼, 오브젝트 스토리지)에 의해 정의되지 않는다. 저장 구조는 도메인 모델을 구현하기 위한 수단이다.

**의도**
저장 구조를 먼저 설계하면 도메인 모델이 DB 스키마와 동일해지는 문제가 발생한다. 이는 저장소를 교체하거나 최적화할 때 도메인 전체를 변경해야 하는 결합을 만들어낸다.

**적용 범위**
Domain Model, Storage

**적용 예시**
- `Document` 도메인 엔티티는 "제목, 내용, 버전 목록, 메타데이터"를 가진다. 이 내용이 DB의 어느 테이블 컬럼에 저장되는지는 별도의 설계 결정이다.
- 문서 본문이 오브젝트 스토리지에 저장되더라도, 도메인 레이어에서는 `Document.body`로 동일하게 접근한다.

**위반 예시**
- 도메인 모델이 `documents` 테이블 컬럼 목록과 완전히 동일하다.
- JSON 컬럼으로 저장하기 편하다는 이유로 관련 없는 속성들이 하나의 도메인 엔티티에 묶인다.
- 저장 최적화를 위해 도메인 엔티티에 DB 특화 필드(예: `row_version`, `partition_key`)가 추가된다.

**설계 시사점**
Phase 1 도메인 모델 설계와 저장 구조 설계를 별도 작업으로 진행한다. 도메인 모델이 먼저 확정된 후 저장 매핑을 결정한다.

### P5. 버전 및 변경 추적 가능성 확보

**선언문**
모든 문서 변경은 버전으로 기록된다. 버전은 부가 기능이 아니라 플랫폼의 기본 동작이다. 특정 시점의 상태 접근과 변경 이력 조회는 항상 가능해야 한다.

**의도**
규정·정책·지식 문서는 현재 내용만큼이나 과거 상태가 중요하다. 버전 관리를 나중에 추가하려 하면 도메인 모델 전체를 재설계해야 하는 경우가 생긴다. 처음부터 버전을 기본 구조로 내재화해야 한다.

**적용 범위**
Domain Model, API, Storage

**적용 예시**
- 문서 내용을 수정할 때 기존 레코드를 덮어쓰지 않고 새 Version 엔티티를 생성한다.
- `GET /documents/{id}/versions/{versionId}` API로 특정 버전의 상태를 조회할 수 있다.
- 변경 이력은 who, when, what이 항상 기록된다.

**위반 예시**
- 문서 수정 시 `UPDATE documents SET body = ?`로 본문을 직접 덮어쓴다.
- "버전 관리는 나중에 추가하면 되니까 일단 최신 본만 저장"이라는 결정을 한다.
- 특정 필드만 버전 관리 대상에서 제외한다는 예외를 기본값으로 설정한다.

**설계 시사점**
Phase 1에서 `Version` 엔티티는 `Document`와 독립적으로 설계되어야 하며, 어떤 범위(Document 전체 vs Node 단위)로 버전을 관리할지 명시적으로 결정해야 한다.

### P6. 구조화 우선, 비정형 수용

**선언문**
구조화 가능한 정보는 플랫폼이 인식하는 구조로 저장한다. 비정형 정보도 수용하되, 구조화 여지를 남겨두는 방식으로 저장한다.

**의도**
비정형 데이터만 저장하면 검색, 탐색, 외부 연계 품질이 낮아진다. 반대로 과도한 정규화는 유연성을 잃게 한다. 핵심 구조는 명시적으로 관리하고, 나머지는 확장 필드로 수용하는 균형이 필요하다.

**적용 범위**
Domain Model, Storage, Search

**적용 예시**
- 문서 내부 구조(섹션, 조항)는 Node 엔티티로 저장하여 플랫폼이 인식한다.
- 유형별로 다른 속성은 정형 메타데이터 스키마로 정의하고, 예외 속성은 JSON 확장 필드로 수용한다.

**위반 예시**
- 문서 전체를 단일 텍스트 또는 HTML Blob으로 저장하여 플랫폼이 내부 구조를 인식하지 못한다.
- 모든 속성을 완전히 정규화된 컬럼으로 강제하여 예외적 메타데이터를 추가할 수 없다.

**설계 시사점**
Phase 1에서 Node 엔티티와 Metadata 스키마를 설계할 때, 정형 필드와 비정형 확장 필드의 경계를 명시적으로 정의해야 한다.

### P7. 메타데이터와 문서 유형의 확장 가능성

**선언문**
메타데이터는 고정 컬럼이 아니라 확장 가능한 스키마로 관리한다. 새로운 문서 유형과 그에 따른 메타데이터 정의는 코드 변경 없이 추가할 수 있어야 한다.

**의도**
조직의 지식 자산 유형은 계속 늘어난다. 유형별 메타데이터가 코어 모델에 하드코딩되면 유형이 추가될 때마다 스키마 마이그레이션이 필요하다. 확장 가능한 메타데이터 구조가 플랫폼의 장기 안정성을 보장한다.

**적용 범위**
Domain Model, API, Search

**적용 예시**
- `DocumentType` 엔티티가 해당 유형의 메타데이터 스키마를 정의한다.
- 새 문서 유형 "교육 자료"를 추가할 때 `DocumentType` 레코드와 메타데이터 스키마 정의를 추가하고, 코어 모델은 변경하지 않는다.

**위반 예시**
- `documents` 테이블에 `regulation_number`, `manual_code` 같은 유형별 컬럼이 직접 추가된다.
- 메타데이터 추가가 DB 마이그레이션 없이는 불가능한 구조로 설계된다.

**설계 시사점**
Phase 1에서 `DocumentType`과 `Metadata` 설계 시, 메타데이터 스키마 정의와 실제 메타데이터 값을 분리하는 구조(EAV 또는 JSON Schema 기반 등)를 결정해야 한다.

### P8. 검색·탐색·재사용 가능성 우선

**선언문**
지식 자산은 저장이 목적이 아니라 탐색과 재사용이 목적이다. 모든 핵심 엔티티는 검색·필터링·탐색이 가능한 방식으로 설계한다.

**의도**
저장에만 집중하고 검색을 나중에 고려하면, 도메인 모델과 저장 구조가 검색에 부적합한 형태로 굳어진다. 탐색 가능성을 처음부터 설계 기준으로 두어야 한다.

**적용 범위**
Search, API, Domain Model

**적용 예시**
- 메타데이터 필드를 정의할 때 검색 인덱싱 여부를 함께 정의한다.
- Node 단위로 검색하여 문서 내 특정 조항을 직접 찾을 수 있다.
- `GET /documents?type=regulation&status=active&tag=privacy` 형태의 복합 필터를 API로 제공한다.

**위반 예시**
- 검색은 문서 제목 전문 검색만 지원하고, 메타데이터·유형·상태 기반 필터는 없다.
- Node 내부 내용은 검색 불가능하고, 문서 전체 단위로만 검색된다.

**설계 시사점**
Phase 1 도메인 모델 설계 시, 각 엔티티에서 검색 대상이 될 필드를 미리 식별하고, 검색 인덱스 설계를 저장 구조와 별개로 계획해야 한다.

### P9. 외부 연계 가능성 확보

**선언문**
이 플랫폼의 지식 자산은 외부 시스템, 자동화 도구, AI 에이전트가 API를 통해 직접 소비할 수 있어야 한다. 연계는 파일 export가 아니라 구조화된 API 응답으로 제공한다.

**의도**
지식 플랫폼의 가치는 플랫폼 외부로 지식이 흘러나갈 수 있을 때 극대화된다. 외부 연계가 후속 과제로 밀리면 API 구조가 UI 전용으로 굳어져 연계 시점에 대규모 수정이 필요해진다.

**적용 범위**
API, Integration

**적용 예시**
- 외부 시스템이 `GET /documents/{id}` API를 호출하여 구조화된 JSON으로 문서를 수신한다.
- AI 에이전트가 Node 단위로 지식을 조회하고, 메타데이터 필터를 적용하여 원하는 지식 단위만 수집한다.
- Webhook 또는 이벤트 스트림으로 문서 변경을 외부 시스템에 알릴 수 있다.

**위반 예시**
- 외부 시스템 연계 방법이 "파일 공유 폴더에 PDF 내보내기"뿐이다.
- API 응답 구조가 특정 UI 컴포넌트의 렌더링 형식에 종속되어 있다.
- 인증 없이 접근 가능한 API가 없어 모든 자동화가 불가능하다.

**설계 시사점**
Phase 2 이후 API 설계 시, 외부 소비자를 내부 UI와 동등한 1등 API 사용자로 취급하여 응답 구조와 인증 방식을 결정해야 한다.

### P10. 운영·감사·권한 정책의 명시적 관리

**선언문**
권한, 운영 상태, 감사 기록은 도메인 로직과 분리된 명시적 정책으로 관리한다. 이 기능들은 암묵적 코드 분기가 아닌 선언적 정책으로 표현한다.

**의도**
권한 검사와 감사 로그가 비즈니스 로직 코드 내부에 분산되면, 정책 변경 시 여러 곳을 수정해야 하며 감사 누락 위험도 높아진다. 명시적 정책 레이어를 두면 운영·컴플라이언스 요구사항 대응이 용이해진다.

**적용 범위**
Governance, Admin, API

**적용 예시**
- 문서 접근 권한은 도메인 서비스 코드 내 `if (user.role === 'admin')`이 아니라, 권한 정책 엔티티를 통해 선언적으로 정의한다.
- 모든 문서 상태 변경(생성, 수정, 승인, 폐기)은 감사 로그에 자동으로 기록된다.
- 운영 정책(예: 특정 유형 문서는 승인 후에만 공개) 변경은 코드 배포 없이 Admin UI에서 처리한다.

**위반 예시**
- 감사 로그가 일부 기능에만 존재하고, 나머지는 기록되지 않는다.
- 권한 검사 로직이 각 API 핸들러마다 다르게 구현된다.
- 문서 상태 변경 규칙이 코드에 하드코딩되어 있어 정책 변경 시 배포가 필요하다.

**설계 시사점**
Phase 1에서 문서 상태 모델 설계 시, 상태 전이 규칙과 권한 정책을 도메인 모델에 명시적으로 반영해야 한다. 감사 로그 기록 전략도 초기부터 포함해야 한다.

## 원칙 간 우선순위

### 상위 원칙

충돌 상황에서 아래 순서를 기준으로 판단한다.

```
P1 (API-first) > P2 (범용 도메인 모델) > P5 (버전 추적) > P3 (User/Admin 분리) > 나머지
```

**해석:**
- 어떤 기능도 API 없이 UI 전용으로 구현하는 것은 P1 위반이며 허용하지 않는다.
- 단기 편의를 위한 유형 종속 코어 모델 변경은 P2 위반이며 허용하지 않는다.
- 버전 추적을 포기하면 P5 위반이며, 이는 플랫폼 정체성과 직결되므로 허용하지 않는다.

### 충돌 시나리오와 판단 기준

| 충돌 상황 | 판단 기준 |
|----------|----------|
| 빠른 구현 vs API-first | API-first 우선. 빠른 구현이 필요하면 API를 먼저 최소 구현 후 UI 연결. |
| 저장 효율 vs 도메인 명확성 | 도메인 명확성 우선. 저장 최적화는 도메인 모델이 확정된 후 별도 결정. |
| 단순 UI 구성 vs User/Admin 분리 | User/Admin 분리 우선. UI 복잡도가 높아지더라도 혼합은 허용하지 않는다. |
| 특정 요구사항 최적화 vs 범용 모델 유지 | 범용 모델 유지 우선. 특정 요구가 범용 구조로 수용 불가능한 경우 예외 적용 절차를 따른다. |

### 예외 적용 원칙

원칙의 예외를 허용하려면 다음 조건을 충족해야 한다.

1. **예외 사유 명시**: 어떤 원칙을, 왜, 어느 범위에서 예외 적용하는지 문서화한다.
2. **범위 제한**: 예외는 특정 기능·모듈로 범위를 제한한다. 전체 플랫폼 원칙을 뒤집는 예외는 인정하지 않는다.
3. **회귀 계획**: 예외가 임시 조치인 경우, 원칙 복귀 시점과 방법을 함께 기록한다.
4. **기록 위치**: 예외 내용은 해당 설계 문서 또는 PR/ADR에 명시적으로 기록한다.

## 설계 시 활용 방법

1. **기능 추가 요청 시**: In Scope/Out Scope 기준(Phase 0-1)과 P2(범용 모델), P10(권한/운영) 원칙을 먼저 확인한다.
2. **설계안 비교 시**: 각 설계안이 P1, P4, P5를 충족하는지 비교한다.
3. **API 설계 시**: P1, P9를 기준으로 API가 내부 UI가 아닌 외부 소비자도 고려한 설계인지 검토한다.
4. **데이터 모델 설계 시**: P2, P4, P6, P7을 기준으로 유형 종속·저장 종속 여부를 검토한다.
5. **UI 설계 시**: P3을 기준으로 운영 기능이 User UI에 침범하고 있지 않은지 확인한다.
6. **설계 리뷰 시**: [Task2_design_review_checklist.md](./Task2_design_review_checklist.md)를 활용한다.
