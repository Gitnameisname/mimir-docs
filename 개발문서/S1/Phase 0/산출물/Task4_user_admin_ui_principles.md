# Phase 0 — User/Admin UI 분리 원칙

## 문서 목적

이 문서는 Mimir 플랫폼에서 일반 사용자 UI(User UI)와 관리자·운영자 UI(Admin UI)를 구조적으로 분리하는 원칙을 확정한다. 단순히 메뉴를 나누는 것이 아니라, 목적·정보 구조·행동 흐름·권한·데이터 노출 범위가 다른 두 인터페이스 계층의 설계 기준을 정의한다. 이후 IA 설계, 라우팅 구조, 프론트엔드 아키텍처, API 권한 설계에서 이 문서를 기준으로 판단한다.

---

## 1. User UI 정의

### 핵심 목적
일반 사용자가 지식 자산을 탐색·조회·생산·편집·재사용하는 작업 환경을 제공한다.

### 대표 사용자
조직 구성원으로서 문서와 지식을 소비하거나 생산하는 일반 사용자.

### 기대 행동 패턴
- 원하는 문서를 검색하거나 구조 탐색으로 찾는다.
- 문서 내용을 읽고 특정 버전을 조회한다.
- 새 문서를 작성하거나 기존 문서를 편집한다.
- 문서에 첨부 파일이나 관련 리소스를 연결한다.
- 문서의 현재 상태(게시됨, 검토 중 등)를 확인한다.

### 정보 복잡도 수준
**낮음~중간.** 사용자가 필요한 정보에 집중할 수 있도록 운영·관리 정보는 노출하지 않는다. 문서 상태는 사용자 행동에 필요한 수준(예: "게시됨", "수정 필요")으로만 표시한다.

---

## 2. Admin UI 정의

### 핵심 목적
플랫폼 운영자·관리자가 지식 자산의 정책을 정의하고, 권한을 통제하며, 상태를 관리하고, 운영 현황을 감사하는 운영 환경을 제공한다.

### 대표 사용자
플랫폼 관리자, 콘텐츠 운영자, 시스템 담당자.

### 기대 행동 패턴
- 문서 유형과 메타데이터 스키마를 정의·수정한다.
- 사용자 권한과 역할을 설정·변경한다.
- 문서의 게시·승인·폐기 상태를 관리한다.
- 감사 로그를 조회하여 누가 무엇을 했는지 확인한다.
- 외부 연계 설정(Webhook, 동기화)을 관리한다.
- 시스템 설정과 운영 정책을 조정한다.

### 정보 복잡도 수준
**높음.** 운영자는 플랫폼 전체 상태를 파악하고 제어해야 하므로, 복잡한 정보와 세부 설정이 노출된다. 운영성과 감사 가능성이 UX보다 우선된다.

---

## 3. 분리 원칙

### P-UI-1. 목적에 의한 분리
User UI와 Admin UI는 메뉴 구조가 다른 것이 아니라 **목적 자체가 다른 별개의 인터페이스**다. User UI는 지식 활용·생산성에, Admin UI는 운영·통제·감사에 최적화된다.

### P-UI-2. 정보 구조의 분리
- **User UI**: 작업과 콘텐츠 중심. 네비게이션은 사용자의 목적(탐색, 작성, 편집)을 기반으로 구성한다.
- **Admin UI**: 자원·정책·운영 중심. 네비게이션은 관리 대상(문서 유형, 사용자, 권한, 시스템)을 기반으로 구성한다.

### P-UI-3. 진입 경로의 분리
- User UI: `/app/*` 경로
- Admin UI: `/admin/*` 경로
- 두 영역은 라우팅 수준에서 분리되며, User UI에서 Admin UI로 진입하는 경로를 두지 않는다.

### P-UI-4. 운영 기능의 사용자 화면 침투 금지
운영 기능(승인, 권한 변경, 정책 설정, 감사 조회)은 User UI에 조건부 노출로도 두지 않는다. 운영 기능이 필요한 경우 Admin UI에서 수행한다.

### P-UI-5. 화면 숨김이 아닌 API 권한으로 통제
UI에서 버튼을 숨기는 방식만으로 접근을 통제하지 않는다. API 레이어에서 역할·권한 기반 접근 통제가 이루어져야 하며, UI 분리는 이를 시각적으로 반영하는 것이다.

### P-UI-6. 공용 컴포넌트와 화면 책임의 분리
디자인 시스템과 UI 컴포넌트는 User UI와 Admin UI가 공유할 수 있다. 그러나 화면의 목적과 책임은 공유하지 않는다. 동일한 컴포넌트로 구성되더라도 화면이 무엇을 위한 것인지는 분명히 다르다.

### P-UI-7. Admin 액션과 감사 로그의 연결
Admin UI에서 발생하는 모든 변경·운영 액션은 감사 로그와 연결되어야 한다. 운영 행위의 추적 가능성은 Admin UI 설계의 기본 전제다.

## 4. 공통 리소스에 대한 관점 차이

같은 리소스를 보더라도 User와 Admin이 보는 정보와 허용 행동이 다르다.

### Document 상세 보기

| 관점 | User | Admin |
|------|------|-------|
| 주요 관심사 | 내용, 구조, 편집 가능성 | 상태, 소유권, 권한, 정책 적합성 |
| 표시 정보 | 제목, 본문, 버전, 메타데이터(사용자 관련 항목) | 위와 동일 + 승인 이력, 감사 로그, 권한 설정, 정책 위반 여부 |
| 허용 동작 | 조회, 편집, 첨부 추가 | 조회, 상태 변경, 권한 변경, 폐기, 감사 조회 |
| 접근 제어 | 일반 사용자 권한 | 관리자 권한 |

### Version 목록

| 관점 | User | Admin |
|------|------|-------|
| 주요 관심사 | 어떤 내용이 언제 바뀌었는가 | 누가 언제 어떤 이유로 변경했는가, 승인 여부 |
| 표시 정보 | 버전 번호, 변경 날짜, 변경 요약 | 위와 동일 + 변경자 정보, 승인자, 감사 추적 ID |
| 허용 동작 | 이전 버전 조회, 현재 버전 비교 | 위와 동일 + 버전 복원, 감사 기록 다운로드 |

### Status 정보

| 관점 | User | Admin |
|------|------|-------|
| 주요 관심사 | 이 문서를 지금 사용할 수 있는가 | 왜 이 상태인가, 다음 단계는 무엇인가, 누가 처리해야 하는가 |
| 표시 정보 | "게시됨", "검토 중" 등 단순 레이블 | 상태 전이 이력, 전이 담당자, 대기 시간, 정책 적합성 |
| 허용 동작 | 상태 확인 (변경 불가 또는 제한적) | 상태 전이 실행, 정책 예외 처리 |

### Metadata 표시

| 관점 | User | Admin |
|------|------|-------|
| 주요 관심사 | 이 문서는 어떤 속성을 가지는가 | 메타데이터 스키마 준수 여부, 필수값 충족 여부 |
| 표시 정보 | 사용자 활용에 필요한 메타데이터 값 | 위와 동일 + 스키마 정의, 필드별 유효성 상태 |
| 허용 동작 | 메타데이터 값 조회·편집 | 위와 동일 + 스키마 수정, 일괄 재검증 |

### Search 결과

| 관점 | User | Admin |
|------|------|-------|
| 주요 관심사 | 관련성 높은 문서를 찾는 것 | 전체 문서 상태를 파악하는 것 |
| 표시 정보 | 제목, 요약, 메타데이터, 상태(사용자 관점) | 위와 동일 + 운영 상태, 권한 정보, 감사 정보 |
| 필터 범위 | 게시된 문서 중심 | 모든 상태의 문서 포함 |

## 5. 위반 사례

### 사례 1. 일반 문서 화면에 관리자 기능 혼재

**상황**: 문서 상세 화면에서 관리자인 경우 "권한 변경", "승인 처리" 버튼이 조건부로 노출된다.

**왜 문제인가**: 운영 기능이 사용자 경험 흐름에 섞여 UI 복잡도가 높아진다. API 권한 경계가 아닌 UI 조건부 렌더링으로만 통제되어 보안이 취약하다.

**위반 원칙**: P-UI-4, P-UI-5

**개선 방향**: 해당 기능을 Admin UI의 문서 관리 화면으로 분리한다. 사용자 화면에서는 상태 정보만 표시한다.

---

### 사례 2. 관리자 기능을 "숨김 버튼"으로만 보호

**상황**: 관리자 기능 버튼을 프론트엔드에서 `isAdmin ? show : hide` 처리로만 보호한다. API는 역할 검사 없이 열려 있다.

**왜 문제인가**: UI 숨김은 보안 수단이 아니다. 개발자 도구나 API 직접 호출로 우회가 가능하다.

**위반 원칙**: P-UI-5

**개선 방향**: API 레이어에서 역할·권한 기반 접근 제어를 반드시 적용한다. UI 숨김은 UX 편의를 위한 보조 수단이지 보안 수단이 아니다.

---

### 사례 3. 운영자 상태 정보가 사용자 검색 결과에 노출

**상황**: 검색 결과에 "승인 대기", "정책 위반", "소유자 없음" 같은 운영 상태 레이블이 일반 사용자에게도 표시된다.

**왜 문제인가**: 사용자에게 의미 없는 운영 정보가 노출되어 혼란을 야기한다. 내부 운영 정보가 의도치 않게 외부에 노출될 수 있다.

**위반 원칙**: P-UI-1, P-UI-4

**개선 방향**: API 응답에서 역할별로 노출 필드를 분리한다. 사용자용 검색 API는 사용자 관점의 상태 필드만 포함한다.

---

### 사례 4. User UI 깊숙한 곳의 숨은 운영 메뉴

**상황**: 사용자 프로필 설정 화면 하단에 "관리자 패널 바로가기" 링크가 존재하거나, 문서 설정 화면 내에 문서 유형 관리 링크가 포함된다.

**왜 문제인가**: 관리 기능 진입 경로가 User UI 내부에 숨어 있으면, Admin UI와 User UI의 경계가 무너지고 유지보수가 어려워진다.

**위반 원칙**: P-UI-3, P-UI-4

**개선 방향**: Admin UI 진입 경로는 `/admin/*`으로만 존재한다. User UI와 Admin UI는 라우팅 수준에서 완전히 분리된다.

---

### 사례 5. User UI와 Admin UI가 서로 다른 도메인 규칙을 직접 구현

**상황**: 문서 상태 변경 로직이 User UI 코드와 Admin UI 코드에 각각 다르게 구현되어 있다.

**왜 문제인가**: 동일 도메인 규칙의 불일치가 발생한다. 규칙이 바뀔 때 두 곳을 모두 수정해야 하며, 누락이 쉽다.

**위반 원칙**: P1 (API-first), P-UI-1

**개선 방향**: 도메인 규칙은 API 레이어에 집중한다. User UI와 Admin UI 모두 동일한 API를 호출하며, 도메인 규칙을 직접 구현하지 않는다.

## 6. 설계 시사점

### 도메인 모델 (Phase 1)
- 문서·버전·메타데이터·상태 엔티티는 User 관점과 Admin 관점의 필드를 모두 수용할 수 있어야 한다.
- 상태 모델에는 사용자용 단순 상태와 운영자용 상세 상태 전이 정보가 함께 설계되어야 한다.
- 감사 필드(`createdBy`, `approvedBy`, `auditTrail`)는 도메인 엔티티 수준에서 정의한다.

### API 설계 (Phase 2)
- 같은 리소스 API라도 역할별로 응답 필드 범위를 다르게 정의한다.
- Admin 전용 액션 API(`/admin/*`)는 별도 권한 검사와 감사 로그 기록을 요구한다.
- User API와 Admin API가 같은 도메인 서비스를 사용하되, 노출 범위와 허용 동작이 다르다.

### 프론트엔드 구조 (Phase 2)
- User UI와 Admin UI는 별도 라우팅 트리(`/app/*`, `/admin/*`)로 분리한다.
- 공용 UI 컴포넌트 라이브러리는 공유하되, 화면 구성과 데이터 흐름은 독립적으로 유지한다.
- 한 UI에서 다른 UI로 직접 진입하는 경로를 두지 않는다.
