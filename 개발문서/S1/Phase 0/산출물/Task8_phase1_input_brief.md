# Phase 0 → Phase 1 설계 입력서 (Input Brief)

## 문서 목적

이 문서는 Phase 0에서 확정한 모든 원칙과 전략을 Phase 1 도메인 모델 설계에 직접 사용할 수 있는 입력 조건으로 압축한다. Phase 1 설계자는 이 문서를 읽고 어떤 부분이 이미 결정되었는지, 어떤 부분에서 설계 자유도가 있는지 파악하여 방향 논쟁 없이 모델 설계에 진입할 수 있어야 한다.

---

## 1. Phase 1 설계 대상

### Phase 1이 다루는 것

| 설계 대상 | 설계 내용 |
|---------|---------|
| **Document** | 플랫폼 핵심 지식 자산 단위의 속성, 관계, 생명주기 |
| **Version** | 문서 변경 단위의 의미, 구조, 추적 방식 |
| **Node / Structure** | 문서 내부 계층 구조의 표현 방식과 탐색 가능성 |
| **DocumentType** | 문서 유형 확장 메커니즘과 유형별 정책 정의 |
| **Metadata** | 공통·확장 메타데이터 구조, 스키마 정의, 질의 가능성 |
| **Status / Lifecycle** | 문서 상태 모델 기초, 전이 규칙, 현재 상태와 이력 분리 |

### Phase 1이 아직 다루지 않는 것

- DB 테이블 상세 설계 및 ORM 매핑
- API 엔드포인트 상세 설계
- UI 컴포넌트 및 화면 구성
- 검색 엔진 제품 선택 및 인덱스 설계
- 인증 시스템 구현 상세
- 외부 시스템 커넥터 구현

### Phase 1이 전체 플랫폼에 결정적 영향을 주는 이유

Phase 1에서 정의되는 도메인 모델은 이후 모든 API 계약, 저장 구조 매핑, 검색 인덱스 설계, UI 데이터 흐름의 기반이 된다. 이 단계에서 특정 저장 방식이나 외부 시스템 형식에 종속된 모델이 만들어지면, 이후 모든 계층이 그 제약을 떠안게 된다.

## 2. Phase 0 확정 전제 요약

아래는 Phase 1에서 다시 논쟁하지 않아야 할 확정 전제다.

### P-0-1. 이 플랫폼은 범용 지식 플랫폼이다

특정 문서 유형(규정집, 매뉴얼 등)에 맞춰진 구조가 코어 모델이 되어서는 안 된다. 코어 모델은 어떤 문서 유형도 수용할 수 있는 범용 구조여야 한다.

### P-0-2. API-first가 기본 원칙이다

모든 코어 도메인 엔티티는 API 리소스로 노출될 수 있어야 한다. 모델을 설계할 때 "이 엔티티가 API 응답으로 어떻게 드러나는가"를 함께 고려해야 한다.

### P-0-3. User/Admin UI는 관점이 다른 별개의 소비자다

도메인 모델은 일반 사용자가 보는 관점(내용·구조·편집)과 관리자가 보는 관점(상태·권한·감사)을 동시에 수용할 수 있어야 한다. 하나의 관점만을 위한 모델은 완성된 설계가 아니다.

### P-0-4. 도메인 모델은 저장 구현과 분리된다

도메인 엔티티는 어떤 저장 방식으로 구현될지와 무관하게 의미가 명확해야 한다. 저장 방식은 도메인 모델이 확정된 후 결정한다.

### P-0-5. 다층 저장 전략을 전제로 한다

RDB, JSON 확장 필드, 오브젝트 스토리지, 검색 인덱스, 감사 로그 저장소가 역할을 분담한다. 도메인 모델은 특정 저장 방식 하나에만 의존하지 않도록 설계한다.

### P-0-6. 외부 시스템은 연계 대상이지 코어를 지배하는 주체가 아니다

외부 인증 시스템, 파일 스토리지, 검색 엔진, AI 서비스의 구조가 코어 도메인 모델에 침투해서는 안 된다. 외부 시스템과의 경계는 커넥터/어댑터 계층이 담당한다.

## 3. 모델별 입력 조건

### 3-1. Document

**본질**: Document는 플랫폼이 관리하는 최상위 지식 자산 단위다. 파일이 아니라 식별 가능하고 추적 가능한 관리 단위다.

**필수 입력 조건**:
- Document는 특정 문서 유형에 종속되지 않는 범용 엔티티다. 유형별 특성은 DocumentType과 Metadata를 통해 표현한다.
- Document는 Version·Node·Metadata·Status·Relationship과 연결될 수 있어야 한다.
- Document는 API 리소스(`/documents/{id}`)로 노출될 수 있는 구조여야 한다.
- Document는 검색 대상이다. 검색 가능한 필드를 도메인 설계 시 식별해야 한다.
- Document와 첨부 파일의 관계는 참조(Reference)다. 파일 원본은 Document 엔티티 내부에 포함되지 않는다.
- Document와 외부 리소스 참조 가능성을 수용해야 한다. (ExternalResourceRef)

**설계 자유도**: Document의 구체적 속성 집합(필드 이름, 선택적 필드 등), 관계 표현 방식.

---

### 3-2. Version

**본질**: Version은 Document 내용의 특정 시점 스냅샷이다. 단순 수정 시각 컬럼이 아니라 독립적인 의미 있는 변경 단위다.

**필수 입력 조건**:
- Version은 Document와 독립된 엔티티다. Document의 부가 속성이 아니다.
- 모든 Document 수정은 새 Version을 생성한다. 기존 Version의 본문을 덮어쓰지 않는다.
- Document의 "현재 상태"는 가장 최근 유효 Version을 가리키는 포인터다.
- Version 이력과 Audit 로그는 다른 개념이다. Version은 "무엇이 변했는가"를, Audit은 "누가 무엇을 언제 했는가"를 기록한다.
- Version은 API 관점에서 추적 가능해야 한다. (`/documents/{id}/versions`)
- Version은 불변이다. 생성 후 본문 수정은 허용하지 않는다.

**설계 자유도**: Version의 범위(Document 전체 vs Node 수준), diff 저장 방식(전체 스냅샷 vs 변경분), 버전 번호 체계.

---

### 3-3. Node / Structure

**본질**: Node는 Document 내부의 계층 구조 단위다. 플랫폼이 인식하고 관리하는 독립 엔티티다.

**필수 입력 조건**:
- Document는 트리/계층 기반 내부 구조를 가질 수 있어야 한다.
- Node는 부분 조회, 부분 렌더링, 부분 검색이 가능한 단위여야 한다.
- 순서, 부모-자식 관계, 계층 깊이 표현이 필요하다.
- Node 표현은 특정 문서 포맷(Markdown, HTML, DOCX 등)에 종속되지 않아야 한다.
- 구조(Node 트리)와 본문 내용(Node 텍스트)은 개념적으로 구분 가능해야 한다.
- Node 단위 메타데이터 부착 가능성을 수용해야 한다.
- 향후 AI/검색 파이프라인에서 Node 단위 청크(chunk) 처리 가능성을 열어두어야 한다.

**설계 자유도**: 트리 저장 표현 기법(인접 리스트, 경로 열거 등), Node 세부 유형 체계, 구조 변경과 버전 연동 방식.

---

### 3-4. DocumentType

**본질**: DocumentType은 플랫폼의 확장 메커니즘이다. 특정 유형의 문서 행동·메타데이터·구조·상태 정책을 정의한다.

**필수 입력 조건**:
- 코어 도메인(Document, Version, Node, Metadata)은 특정 DocumentType에 종속되지 않는다.
- DocumentType은 코어 위에 올라타는 확장 정의 계층이다.
- 새 DocumentType 추가는 코드 변경 없이 Admin이 수행할 수 있어야 한다.
- DocumentType은 해당 유형의 메타데이터 스키마, 허용 상태 전이, 구조 규칙을 정의할 수 있어야 한다.
- DocumentType은 Admin 관점에서 관리 가능해야 한다.
- DocumentType은 API와 UI 양쪽에서 해석 가능한 구조여야 한다.

**설계 자유도**: 유형별 validation 정책 세부 구현, 유형별 상태 모델 차이 표현 방식, 메타데이터 스키마의 계층화 수준.

---

### 3-5. Metadata

**본질**: Metadata는 Document·Node에 부착되는 정형화 가능한 속성 집합이다. 검색·필터링·외부 연계의 기반이다.

**필수 입력 조건**:
- 공통 메타데이터(모든 유형 공통: 작성자, 작성일, 태그 등)와 유형별 확장 메타데이터를 구분해야 한다.
- 확장 메타데이터는 DocumentType이 정의한 스키마에 따라 검증된다. 완전 자유 필드가 아니다.
- 메타데이터는 검색·필터링·집계에 활용될 수 있어야 한다. 검색 불가능한 메타데이터는 의미가 약하다.
- User/Admin 관점에서 노출되는 메타데이터 범위가 다를 수 있다.
- 메타데이터 스키마 정의(어떤 필드가 허용되는가)와 메타데이터 값(실제 데이터)을 분리하여 설계한다.

**설계 자유도**: Metadata 스키마 계층화 수준(단순 key-value vs 타입 있는 스키마), 정렬·집계 지원 범위.

---

### 3-6. Status / Lifecycle

**본질**: Status는 Document의 현재 운영 상태다. 단순 존재 여부가 아니라 지식 자산의 라이프사이클을 표현한다.

**필수 입력 조건**:
- Document는 상태를 가진다. 최소 초안(DRAFT), 검토(REVIEW), 승인(APPROVED), 게시(PUBLISHED), 보관(ARCHIVED) 개념을 수용해야 한다.
- 현재 상태값(단순 필드)과 상태 전이 이력(별도 이력)을 구분한다.
- 상태 전이 규칙(어떤 상태에서 어떤 상태로 전이 가능한가)은 도메인 서비스 계층에 명시적으로 표현된다.
- Version과 Status는 관련되지만 동일 개념이 아니다. 상태 변경이 반드시 새 Version을 생성하지는 않는다.
- User가 보는 상태 정보(단순 레이블)와 Admin이 보는 상태 정보(전이 이력, 정책 적합성)가 다를 수 있다.
- 상태는 API 접근 제어와 연동될 수 있다. (예: 게시된 문서만 외부 접근 허용)

**설계 자유도**: 상태 단계 수와 세부 명칭, 상태 전이 규칙 표현 방식, DocumentType별 상태 모델 차이 허용 범위.

## 4. 외부 연계 독립성 원칙

Phase 1 모델 설계 시 아래 원칙을 반드시 지킨다.

- **외부 인증 구조가 사용자/권한 도메인을 지배하지 않는다.** User 엔티티는 플랫폼 내부 식별자와 플랫폼 관점 속성으로만 정의한다. 외부 SSO 클레임 구조는 별도 매핑 엔티티로 분리한다.
- **파일 저장 구조가 Document 모델을 규정하지 않는다.** Document는 파일 참조 경로가 아닌 `AttachmentId`로 첨부를 참조한다. 오브젝트 스토리지 공급자 종속 필드는 도메인 엔티티에 포함하지 않는다.
- **검색 인덱스 구조가 원본 도메인 모델이 되지 않는다.** 검색 인덱스는 도메인 모델의 파생물이다. 인덱스 구조에 맞춰 도메인 모델을 조정하지 않는다.
- **AI 추출·보조 데이터는 코어 모델과 구분된다.** AI 생성 키워드·요약·임베딩은 별도 보조 저장소에 관리한다. 검증 없이 원본 Metadata 필드에 직접 저장하지 않는다.
- **외부 문서 소스 포맷은 Connector 계층에서 흡수한다.** 특정 외부 시스템의 문서 구조가 Node·Document 모델에 반영되어서는 안 된다.

---

## 5. Phase 2 연결 포인트

Phase 1 결과가 Phase 2 API·서비스 설계로 자연스럽게 이어지려면 아래 연결 포인트를 확보해야 한다.

| 연결 포인트 | Phase 1 설계 요구 | Phase 2 영향 |
|-----------|----------------|-------------|
| **도메인 엔티티 → API 리소스** | 각 엔티티가 API 리소스로 노출 가능한 구조 | `/documents`, `/versions`, `/nodes` 엔드포인트 설계 |
| **상태·버전·메타데이터 조합** | 서비스 계층에서 복합 조회·조합이 가능한 구조 | 서비스 레이어 유스케이스 설계 |
| **User/Admin 관점 차이** | 동일 엔티티의 관점별 필드 범위 차이 수용 | API 응답 직렬화 계층 설계 |
| **감사·이력 추적 가능성** | 도메인 이벤트(`DocumentCreated`, `StatusChanged`) 정의 | 감사 로그·비동기 이벤트 처리 설계 |
| **비동기 처리 확장 여지** | 검색 인덱싱·이벤트 발행이 도메인 레이어와 분리 가능한 구조 | Async Job·Event Layer 설계 |
