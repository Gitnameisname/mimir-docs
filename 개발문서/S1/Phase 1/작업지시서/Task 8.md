Claude Code 작업 지시서

Phase 1 - Task 1-8. 문서 상태 모델 기초 설계

1. 작업명

범용 문서 플랫폼의 문서 상태 모델 기초 설계

2. 작업 목적

Phase 1의 앞선 작업에서 정리한 요구사항, 핵심 엔티티 책임, Document / Version / Node / DocumentType / metadata 구조 초안을 바탕으로,
범용 문서 플랫폼에서 문서의 생명주기를 표현하는 최소 상태 모델을 설계한다.

이 작업의 목적은 단순히 상태 이름 몇 개를 정하는 것이 아니라,
문서가 생성되고, 수정되고, 검토되고, 배포되고, 보관되는 흐름을
플랫폼 차원에서 일관되게 다룰 수 있도록 상태의 의미, 적용 범위, 전이 규칙의 기초를 확정하는 것이다.

즉, 이번 작업은 다음을 명확히 하는 단계다.
	•	상태는 왜 필요한가
	•	상태는 Document에 붙는가, Version에 붙는가, 둘 다 가능한가
	•	초기 MVP에서 어떤 상태 집합이 적절한가
	•	상태 전이는 어떤 원칙을 가져야 하는가
	•	DocumentType별 상태 차이를 어디까지 허용할 것인가
	•	archived / deleted / obsolete / published 같은 개념을 어떻게 구분할 것인가

⸻

3. 작업 배경

범용 문서 플랫폼은 단순 저장소가 아니라
문서의 작성, 수정, 검토, 공개, 보관을 다루는 시스템으로 확장될 가능성이 크다.

문서에는 보통 다음과 같은 흐름이 존재한다.
	•	작성 중
	•	검토 중
	•	확정/배포됨
	•	더 이상 사용되지 않음
	•	보관됨
	•	삭제됨

하지만 상태 모델을 너무 단순하게 만들면
실제 운영 흐름을 표현하지 못하고,
반대로 너무 복잡하게 만들면 초기 설계가 과도해진다.

또한 상태는 다음과 같은 질문과 연결된다.
	•	현재 이 문서는 사용자에게 보여도 되는가?
	•	이 문서는 수정 가능한가?
	•	이 문서는 공식 배포본인가?
	•	이 문서는 보관 상태인가?
	•	과거 버전은 어떤 위치를 가지는가?

따라서 이번 작업에서는
초기 MVP에 적절한 최소 상태 모델과
후속 확장이 가능한 상태 모델 구조를 함께 정리해야 한다.

⸻

4. 작업 범위

포함 범위
	•	상태 모델의 역할 정의
	•	상태가 적용되는 단위(Document / Version / 혼합) 검토
	•	최소 상태 집합 설계
	•	상태별 의미 정의
	•	상태 전이 규칙 초안 설계
	•	archived / deleted / published / draft 등 개념 구분
	•	DocumentType과의 관계 검토
	•	Version 정책과의 연결 검토
	•	최소 권장 상태 모델 제시
	•	후속 Task 입력값 정리

제외 범위
	•	승인 워크플로 상세 설계
	•	권한 정책 설계
	•	알림/이벤트 시스템 설계
	•	실제 상태 머신 코드 구현
	•	DB enum/컬럼 설계
	•	UI 상태 배지 구현

즉, 이번 작업은 문서 상태 모델 개념 설계 문서 작성이다.

⸻

5. Claude Code가 해야 할 일

아래 순서대로 수행하라.

Step 1. 상태 모델의 역할 정의

먼저 문서 상태 모델이 무엇을 위한 것인지 명확히 써라.

반드시 아래 관점을 포함한다.
	•	상태 모델은 문서의 생명주기 단계를 표현한다.
	•	상태는 단순 표시값이 아니라 운영 규칙의 기준이 될 수 있다.
	•	상태는 문서의 가시성, 수정 가능성, 공식성, 보관 여부와 연결될 수 있다.
	•	상태 모델은 Document 중심으로 시작하되, Version과의 관계도 고려해야 한다.
	•	상태 모델은 초기에는 단순해야 하지만 후속 확장 가능해야 한다.

그리고 아래도 함께 정리한다.
	•	상태 모델이 책임하는 것
	•	상태 모델이 책임하지 않는 것
	•	권한 모델과 상태 모델의 차이
	•	승인 프로세스와 상태 모델의 차이

⸻

Step 2. 상태 적용 단위 검토

이 단계는 매우 중요하다. 반드시 별도 섹션으로 작성하라.

아래 세 관점을 비교 검토하라.

관점 A. Document 상태만 관리
예:
	•	draft
	•	review
	•	published
	•	archived

장점:
	•	단순함
	•	초기 MVP에 적합

한계:
	•	특정 버전 단위 검토/배포 표현이 약함

관점 B. Version 상태만 관리
장점:
	•	배포본/검토본 표현에 유리

한계:
	•	문서 전체의 현재 운영 상태 표현이 약해질 수 있음

관점 C. Document 상태 + Version 상태의 혼합 모델
장점:
	•	확장성 높음

한계:
	•	초기 복잡도 증가

이 세 관점을 비교하고,
Phase 1 기준 권장 방향을 제안하라.

권장 방향 예시:
	•	초기에는 Document 중심 상태 모델
	•	Version에는 후속 확장을 위한 상태/marker 가능성만 열어둠

⸻

Step 3. 최소 상태 집합 후보 설계

최소 아래 상태 후보를 검토하라.
	•	draft
	•	review
	•	published
	•	archived

추가 검토 후보:
	•	deprecated
	•	inactive
	•	deleted
	•	scheduled
	•	superseded
	•	rejected

각 상태에 대해 아래를 정리하라.
	•	의미
	•	왜 필요한가
	•	MVP 포함 여부
	•	다른 상태와 겹치는지 여부

주의:
	•	초기에는 상태 수를 최소화하라.
	•	deleted는 상태인지 삭제 정책인지 구분해서 검토하라.
	•	deprecated / superseded는 후속 확장일 가능성이 크다.

⸻

Step 4. 상태별 의미 정의

이 단계는 핵심이다. 반드시 상태별 정의를 명확히 문장형으로 작성하라.

최소 아래 상태들에 대해 정리하라.

draft
	•	아직 확정되지 않은 작업 중 문서
	•	수정 가능
	•	내부 작성 단계

review
	•	검토/승인 대기 상태
	•	수정 정책은 운영 정책에 따라 달라질 수 있음

published
	•	공식적으로 유효하거나 공개 가능한 상태
	•	사용자 소비 기준의 대표 상태

archived
	•	더 이상 활성 운영 대상은 아니지만 보관되는 상태
	•	검색/열람 정책은 별도 운영 규칙과 연결될 수 있음

그리고 아래 구분도 함께 설명하라.
	•	archived vs deleted
	•	published vs effective
	•	draft vs working copy
	•	archived vs deprecated/superseded

⸻

Step 5. 상태 전이 규칙 초안 설계

아래를 기준으로 최소 상태 전이 규칙을 정리하라.

예시 후보:
	•	draft → review
	•	review → draft
	•	review → published
	•	published → archived
	•	published → draft (개정 작업 시 허용 여부 검토)
	•	archived → draft 또는 archived → published 허용 여부 검토

각 전이에 대해 아래를 정리하라.
	•	허용 여부
	•	이유
	•	일반적 사용 시나리오
	•	주의사항

또한 아래도 정리하라.
	•	모든 상태 간 자유 전이를 허용할 것인가
	•	상태 전이는 제어된 전이만 허용할 것인가
	•	복원/재활성화 시나리오를 둘 것인가

권장 방향:
	•	초기에는 명시적이고 제한된 전이 규칙을 두는 것이 적절함

⸻

Step 6. 상태와 수정 가능성의 관계 검토

아래를 검토하라.
	•	draft 상태 문서는 수정 가능한가
	•	review 상태에서 직접 수정 가능한가
	•	published 상태 문서를 직접 수정할지, 새 Version/새 draft를 만들지
	•	archived 상태 문서를 수정 가능하게 둘지

핵심 질문:
	•	상태 모델이 편집 정책과 어떻게 연결되는가?

권장 방향 예시:
	•	published 문서는 직접 덮어쓰기보다 새 작업 버전 생성으로 연결
	•	archived는 원칙적으로 비활성, 필요 시 복원 후 수정

⸻

Step 7. 상태와 Version 정책의 관계 검토

이 단계도 별도 섹션으로 작성하라.

아래를 검토하라.
	•	published는 Document 상태인가, 특정 Version의 속성인가
	•	현재 문서가 published 상태라는 것은 무엇을 의미하는가
	•	개정 중 문서는 published 문서와 별도로 draft 작업 상태를 가질 수 있는가
	•	Version 단위 published marker가 필요한가
	•	Document.current_version과 published version이 달라질 수 있는가

권장 방향:
	•	초기에는 Document 상태 중심
	•	그러나 후속 Phase를 위해 Version 단위 published marker 가능성은 열어둠

⸻

Step 8. deleted / archived / deprecated 개념 구분

이 단계는 반드시 별도 섹션으로 작성하라.

아래 개념을 구분해서 정리하라.

archived
	•	운영 대상에서는 빠졌지만 보존되는 문서

deleted
	•	논리 삭제 또는 물리 삭제 정책과 연결되는 개념
	•	상태라기보다 삭제 처리 정책일 가능성 검토

deprecated / superseded
	•	여전히 기록으로 남지만 최신/유효 문서가 아님
	•	후속 확장 상태 또는 metadata로 볼 가능성 검토

목적은
나중에 상태 모델과 삭제 정책이 섞이지 않도록 하는 것이다.

⸻

Step 9. DocumentType과 상태 모델의 관계 검토

아래를 검토하라.
	•	모든 DocumentType이 동일 상태 모델을 공유해야 하는가
	•	특정 타입은 별도 상태가 필요한가
	•	report: submitted 가능성
	•	policy: effective / obsolete 가능성
	•	meeting_note: approved 필요성이 낮을 수도 있음
	•	초기에는 공통 상태 모델을 유지할지
	•	타입별 상태 확장은 언제 도입할지

권장 방향:
	•	초기에는 공통 상태 모델
	•	타입별 세부 상태는 후속 확장 포인트로만 남김

⸻

Step 10. 최소 권장 상태 모델 제시

위 검토를 바탕으로,
Phase 1 기준의 최소 권장 상태 모델을 제안하라.

아래를 포함하라.
	•	MVP 상태 집합
	•	상태별 의미
	•	기본 전이 규칙
	•	후속 확장 가능 상태
	•	상태 적용 단위 권장안

구현 코드가 아니라 개념 모델 명세 형태로 작성하라.

⸻

Step 11. 설계 쟁점 및 판단 기록

아래 쟁점에 대해 판단을 남겨라.
	•	상태를 Document에만 둘지
	•	Version 상태를 초기부터 둘지
	•	review 상태를 코어에 포함할지
	•	archived를 상태로 둘지
	•	deleted를 상태로 둘지 삭제 정책으로 분리할지
	•	published 문서를 직접 수정 가능하게 둘지
	•	deprecated/superseded를 초기 상태로 넣을지

각 항목에 대해:
	•	쟁점
	•	선택한 방향
	•	이유
를 기록하라.

7. 작성 방식 요구사항

1) 표를 적극 활용

최소 아래 표를 포함하는 것이 좋다.
	•	상태 후보 비교표
	•	상태 전이 규칙 표
	•	설계 쟁점 판단표

2) 구현 세부사항으로 내려가지 말 것

좋은 예:
	•	“초기 상태 모델은 Document 중심으로 유지하는 것이 적절하다.”
	•	“deleted는 상태보다는 삭제 정책으로 분리하는 것이 바람직하다.”
	•	“published 상태 문서의 수정은 직접 수정이 아니라 새 작업 흐름으로 연결하는 것이 안전하다.”

좋지 않은 예:
	•	상태 머신 코드 작성
	•	enum 코드 작성
	•	DB 컬럼 정의

3) 단순성과 확장성의 균형을 유지할 것

초기 상태 집합은 작게 유지하되,
후속 Phase에서 승인/배포/유효기간 개념으로 확장 가능해야 한다.

	•	상태 적용 단위(Document/Version)가 정리되어 있다.
	•	MVP 상태 집합이 과도하지 않게 정리되어 있다.
	•	상태 전이 규칙이 초안 수준으로 마련되어 있다.
	•	삭제 정책과 상태 모델의 구분이 정리되어 있다.
