Claude Code 작업 지시서

Phase 1 - Task 1-1. 도메인 모델 요구사항 정리

1. 작업명

범용 문서 도메인 모델 요구사항 정리

2. 작업 목적

Mimir 프로젝트의 범용 문서 플랫폼 설계를 위해,
여러 종류의 문서를 공통 구조로 수용할 수 있도록 도메인 모델 요구사항을 체계적으로 정리한다.

이 작업의 목표는 구현이 아니라, 이후 Task에서 사용할 수 있는 요구사항 기준 문서를 만드는 것이다.
즉, Document / Version / Node / DocumentType / metadata / 상태 모델을 설계하기 전에,
무엇을 반드시 지원해야 하는지 먼저 명확히 정의하는 단계다.

⸻

3. 작업 배경

Mimir는 특정 문서 한 종류만 다루는 시스템이 아니라,
향후 다음과 같은 다양한 문서 유형을 수용할 수 있어야 한다.

예시:
	•	규정집 / 규칙 / 정책 문서
	•	매뉴얼 / 가이드
	•	기술 문서
	•	보고서
	•	회의록
	•	일반 지식 문서
	•	템플릿 기반 문서
	•	추후 추가될 사용자 정의 문서

따라서 이번 작업에서는 특정 문서 유형에 끌려가지 않고,
범용 문서 플랫폼의 코어 모델이 만족해야 할 요구사항을 정리해야 한다.

⸻

4. 작업 범위

포함 범위
	•	지원해야 할 문서 유형 후보 정리
	•	문서 유형별 공통점과 차이점 분석
	•	범용 모델에서 필요한 핵심 요구사항 도출
	•	구조적 요구사항 / 이력 요구사항 / 메타데이터 요구사항 / 상태 요구사항 초안 도출
	•	이후 모델 설계에 필요한 제약사항 정리

제외 범위
	•	DB 스키마 설계
	•	API 설계
	•	UI/에디터 설계
	•	상세 권한 체계 설계
	•	승인 워크플로 상세 설계
	•	실제 코드 구현

즉, 이번 작업은 분석 및 문서화 작업이다.

⸻

5. Claude Code가 해야 할 일

아래 순서대로 결과를 만들어라.

Step 1. 지원 대상 문서 유형 분류

최소 아래 문서군을 포함하여 정리한다.
	•	policy / regulation 문서
	•	manual / guide 문서
	•	technical document
	•	report
	•	meeting note
	•	generic knowledge document
	•	template-based document

각 문서 유형에 대해 아래를 간단히 정리한다.
	•	문서 목적
	•	구조적 특징
	•	버전 관리 필요성
	•	메타데이터 특징
	•	상태 관리 필요성

⸻

Step 2. 문서 유형 간 공통점 도출

모든 문서 유형에 공통으로 필요한 요소를 정리한다.

예시 관점:
	•	문서 식별자 필요
	•	제목 필요
	•	본문 구조 필요
	•	생성/수정 이력 필요
	•	현재 버전 개념 필요
	•	문서 유형 분류 필요
	•	메타데이터 확장 필요
	•	상태 개념 필요

이 단계에서는 “공통 코어”가 무엇인지 드러나야 한다.

⸻

Step 3. 문서 유형 간 차이점 도출

문서 유형별로 어떤 점이 달라지는지 정리한다.

예시 관점:
	•	구조 복잡도 차이
	•	필수 metadata 차이
	•	승인/배포 필요 여부 차이
	•	버전 중요도 차이
	•	노드 구성 차이
	•	템플릿 기반 여부 차이

이 단계의 목적은 “공통 모델 + 확장 구조”가 왜 필요한지를 보여주는 것이다.

⸻

Step 4. 범용 도메인 모델 요구사항 목록 작성

다음 범주로 나누어 요구사항을 정리한다.

4-1. 문서 식별 요구사항
예:
	•	문서는 고유 식별자를 가져야 한다.
	•	문서는 문서 유형을 가져야 한다.
	•	문서는 대표 제목/설명을 가질 수 있어야 한다.

4-2. 구조 표현 요구사항
예:
	•	문서 본문은 트리 기반 구조로 표현 가능해야 한다.
	•	섹션/문단/리스트/표 등 다양한 요소를 수용할 수 있어야 한다.
	•	단순 문서와 복합 문서 모두 표현 가능해야 한다.

4-3. 버전 관리 요구사항
예:
	•	문서의 여러 버전을 관리할 수 있어야 한다.
	•	특정 시점의 문서 상태를 복원 가능해야 한다.
	•	현재 버전과 과거 버전을 구분할 수 있어야 한다.

4-4. 메타데이터 요구사항
예:
	•	공통 metadata와 유형별 확장 metadata를 함께 수용할 수 있어야 한다.
	•	일부 metadata는 검색/필터 기준으로 활용 가능해야 한다.

4-5. 상태 관리 요구사항
예:
	•	문서는 최소한 draft / review / published / archived 수준의 상태를 가질 수 있어야 한다.
	•	상태 모델은 문서 유형에 따라 확장 가능해야 한다.

4-6. 확장성 요구사항
예:
	•	새로운 문서 유형을 추가할 수 있어야 한다.
	•	특정 문서군만을 위한 특수 metadata를 수용할 수 있어야 한다.
	•	향후 권한/승인/API 연동 구조와 충돌하지 않아야 한다.

⸻

Step 5. 설계 제약사항 및 판단 기준 정리

아래 관점에서 제약과 원칙을 정리한다.
	•	특정 문서 유형에 종속되지 말 것
	•	렌더링 방식과 저장 구조를 혼동하지 말 것
	•	범용성과 단순성의 균형을 맞출 것
	•	초기 MVP와 장기 확장을 구분할 것
	•	상태/버전/metadata를 과도하게 결합하지 말 것

⸻

Step 6. 최종 문서 작성

최종 결과는 아래 목차를 따르는 Markdown 문서로 작성한다.

권장 문서명:
Task1_domain_model_requirements.md

권장 목차:
	1.	작업 목적
	2.	분석 대상 문서 유형
	3.	문서 유형별 특성 요약
	4.	공통 요구사항
	5.	유형별 차이점
	6.	범용 도메인 모델 요구사항 목록
	7.	설계 제약사항 및 원칙
	8.	다음 Task로 넘길 핵심 결론

⸻

6. 출력 형식 요구사항

반드시 Markdown(.md) 문서로 작성

문서는 사람이 읽고 바로 다음 설계 작업에 사용할 수 있어야 한다.

표를 적극 활용

다음 부분은 표 형태로 정리하는 것이 좋다.
	•	문서 유형별 특성 비교
	•	공통점/차이점 비교
	•	요구사항 목록

요구사항은 문장형으로 명확하게 작성

예:
	•	“문서는 하나 이상의 버전을 가질 수 있어야 한다.”
	•	“문서 본문은 계층 구조로 표현 가능해야 한다.”
	•	“유형별 metadata 확장을 허용해야 한다.”

좋지 않은 예:
	•	“버전 관리”
	•	“트리 구조 필요”
	•	“메타데이터 유연성”

8. 완료 기준

아래를 모두 만족하면 완료로 본다.
	•	범용 문서 플랫폼이 지원해야 할 문서 유형이 정리되어 있다.

	•	특정 문서 유형에 치우치지 않은 범용 관점이 유지되어 있다.
