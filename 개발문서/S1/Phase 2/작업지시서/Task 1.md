Task 2-1 작업 지시서

주제

권한/역할/조직 도메인 모델 정의

작업 목적

Phase 2의 출발점으로, 문서 플랫폼에서 권한 체계를 구성하는 핵심 주체들을 먼저 정리한다.
이 단계에서는 실제 구현을 하지 않고, 권한 판단의 기준이 되는 도메인 모델과 관계 구조를 설계 문서 수준으로 확정하는 것이 목표다.

이 작업의 결과는 이후 Task 2-2의 보호 대상 리소스 정의, Task 2-3의 ACL 구조 설계, Task 2-4의 API 권한 enforcement 설계의 기반이 된다.

⸻

이번 작업의 범위

이번 작업에서는 아래 항목만 다룬다.
	1.	권한 주체 정의
	•	User
	•	Organization
	•	Team 또는 Group 필요 여부
	•	Membership
	•	Role
	2.	역할 부여 단위 정의
	•	사용자에게 직접 부여되는 역할
	•	조직 단위 역할
	•	특정 리소스 범위 역할의 필요성 검토
	3.	전역 역할과 소속 기반 역할 구분
	•	시스템 관리자
	•	조직 관리자
	•	일반 사용자
	•	감사/보안 담당자 등 확장 가능성
	4.	역할과 권한의 관계 원칙
	•	Role은 Permission 집합인가
	•	Role은 단순 명칭인지 정책 단위인지
	•	향후 ACL과 어떻게 연결될지 고려한 구조화
	5.	도메인 관계 정리
	•	한 사용자가 여러 조직에 속할 수 있는지
	•	조직 내 여러 역할을 동시에 가질 수 있는지
	•	팀/그룹을 통해 간접 권한 부여가 필요한지

⸻

이번 작업에서 하지 말 것

아래는 이번 단계에서 다루지 않는다.
	•	실제 DB 스키마 구현
	•	ORM 모델 작성
	•	API 엔드포인트 작성
	•	ACL 엔트리 상세 설계
	•	권한 evaluation 알고리즘 구현
	•	감사 로그 스키마 상세화
	•	UI 설계

즉, 이번 단계는 코딩 단계가 아니라 설계 확정 단계다.

⸻

설계 전제

아래 전제를 반영하여 설계할 것.
	1.	이 플랫폼은 규정집 전용 시스템이 아니라 모든 유형의 문서를 다룰 수 있는 범용 문서 플랫폼이다.
	2.	향후 외부 애플리케이션 및 AI 도구와의 연계를 고려한 API-first 구조를 전제로 한다.
	3.	사용자 UI와 관리자 UI는 분리될 가능성이 높으므로, 관리자 역할과 일반 사용자 역할의 경계가 중요하다.
	4.	향후 문서, 버전, 노드, 첨부파일, 코멘트, 워크플로, 승인 등 다양한 리소스가 생길 수 있으므로 역할 모델은 확장 가능해야 한다.
	5.	향후 ACL, 정책 엔진, 감사 추적, approval workflow와 연결될 수 있어야 한다.
	6.	단순 RBAC만으로 끝내지 말고, RBAC + resource-scoped assignment + ACL 확장 가능성을 염두에 두고 설계할 것.

⸻

Claude Code가 수행해야 할 작업

아래 순서대로 정리할 것.

1. 권한 주체 후보 식별

다음 엔티티를 정의 후보로 검토하고, 각자의 역할을 설명할 것.
	•	User
	•	Organization
	•	Membership
	•	Role
	•	Team 또는 Group
	•	Service Account 또는 API Client 필요 여부

각 엔티티에 대해 아래를 작성할 것.
	•	왜 필요한가
	•	어떤 책임을 가지는가
	•	다른 엔티티와 어떤 관계를 가지는가
	•	MVP에서 반드시 필요한가, 아니면 확장용인가

⸻

2. 핵심 관계 모델 제안

최소한 아래 관계를 설계안으로 제시할 것.
	•	User ↔ Organization
	•	User ↔ Membership
	•	Membership ↔ Role
	•	Organization ↔ Role
	•	User ↔ Team/Group
	•	Team/Group ↔ Role

그리고 아래 질문에 명확히 답할 것.
	•	사용자는 여러 조직에 속할 수 있는가
	•	조직 안에서 사용자에게 복수 역할 부여가 가능한가
	•	역할은 사용자에 직접 붙는가, membership에 붙는가
	•	팀/그룹은 초기 MVP에 필요한가, 후순위인가
	•	서비스 계정은 초기부터 고려해야 하는가

⸻

3. 역할 체계 초안 정의

최소한 아래 역할 계층을 검토할 것.

전역 역할 예시
	•	PlatformOwner
	•	PlatformAdmin
	•	SecurityAuditor
	•	SupportOperator
	•	NormalUser

조직 역할 예시
	•	OrganizationOwner
	•	OrganizationAdmin
	•	WorkspaceManager 또는 SpaceManager
	•	Editor
	•	Commenter
	•	Viewer

각 역할에 대해 다음을 작성할 것.
	•	적용 범위
	•	권한 성격
	•	다른 역할과의 차이
	•	시스템 전역 역할인지 조직/리소스 범위 역할인지

중요:
여기서는 권한 목록을 완전히 세분화할 필요는 없지만,
역할의 성격과 계층 구조는 분명히 설명해야 한다.

⸻

4. 역할 부여 방식 설계

다음 세 가지 방식을 비교 검토하고, 추천안을 제시할 것.
	•	사용자에게 직접 역할 부여
	•	조직 membership에 역할 부여
	•	팀/그룹 membership을 통한 간접 역할 부여

각 방식에 대해 다음을 비교할 것.
	•	단순성
	•	관리 편의성
	•	확장성
	•	향후 ACL 연결 용이성
	•	API enforcement 적합성

그리고 최종 권장안을 제시할 것.

⸻

5. 도메인 모델 구조안 제시

텍스트 기반으로 아래 수준까지 정리할 것.

예시 형식:
	•	User
	•	id
	•	identity fields
	•	status
	•	Organization
	•	id
	•	name
	•	status
	•	Membership
	•	user_id
	•	organization_id
	•	membership_status
	•	Role
	•	id
	•	name
	•	scope_type
	•	system_defined 여부
	•	Team/Group
	•	id
	•	organization_id
	•	name

여기서는 실제 컬럼 확정보다는 개념 필드 수준으로 정리하면 된다.

⸻

6. 권장 도메인 원칙 정리

최종적으로 아래 질문에 대한 설계 원칙을 문장으로 정리할 것.
	•	역할은 사람(User)에 직접 붙기보다 Membership 중심으로 가야 하는가
	•	플랫폼 전역 관리자와 조직 관리자 권한은 분리해야 하는가
	•	팀/그룹은 MVP 필수인가 아니면 후속 단계인가
	•	서비스 계정 또는 API 클라이언트 주체는 언제 도입하는 것이 좋은가
	•	향후 ACL과 결합할 때 현재 모델이 어떤 장점을 가지는가

의사결정 원칙

설계 중 아래 원칙을 반드시 지킬 것.
	1.	단순하지만 확장 가능하게
	•	MVP에서 과도한 복잡성은 피하되, 나중에 깨지지 않는 구조를 선택할 것
	2.	직접 권한보다 소속 기반 구조 우선
	•	사용자 자체보다 조직/멤버십 중심으로 역할이 부여되는 구조를 우선 검토할 것
	3.	전역 권한과 조직 권한 분리
	•	플랫폼 운영 권한과 조직 내부 운영 권한은 명확히 구분할 것
	4.	ACL 확장성 보장
	•	이번 단계는 RBAC 중심이어도, 이후 ACL 및 resource-scoped permission으로 자연스럽게 확장 가능해야 함
	5.	감사 추적 친화성
	•	누가 어떤 자격으로 어떤 행동을 했는지 추적 가능한 구조를 선호할 것
