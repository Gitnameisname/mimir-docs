# Phase 13. 운영, 보안, 품질, 배포 체계 구축

## 1. Phase 목적

Phase 0~12에서 완성한 Mimir 플랫폼을 실제 서비스 환경에서 안전하고 안정적으로 운영할 수 있는 상태로 완성한다.

기능 개발이 아닌 **운영 가능성(Operability)** 에 집중한다. 보안, 테스트, 모니터링, 배포 파이프라인, 장애 대응 체계를 구축하여 Mimir를 신뢰 가능한 엔터프라이즈 플랫폼으로 마무리한다.

---

## 2. Phase 13 범위

### 포함 범위

- 보안 강화 (인증/인가 정책, OWASP Top 10 검토, API 보안)
- 로깅 및 모니터링 체계 구축 (구조화 로그, 메트릭, 알림)
- 테스트 전략 및 커버리지 목표 수립
- 백업 및 복구 체계 구축
- 데이터 보존 정책 구현 (감사 로그, 버전 데이터)
- CI/CD 배포 파이프라인 구축
- 성능 최적화 및 부하 테스트
- 운영 문서화 (Runbook, 장애 대응 절차)
- 최종 출시 준비 체크리스트

### 제외 범위

- 새 기능 개발 (Phase 0~12에서 완성됨)
- 인프라 프로비저닝 자동화 (Terraform 등, 환경별 별도 진행)

## 4. 핵심 기술 결정

### 4-1. 로깅 스택

| 항목 | 결정 |
|------|------|
| 로그 형식 | JSON 구조화 로그 |
| 수집 | 표준 출력 → 수집 에이전트 (Fluentd/Vector) |
| 저장 | Elasticsearch 또는 CloudWatch Logs |
| 시각화 | Kibana 또는 CloudWatch Dashboard |

---

### 4-2. 모니터링 스택

| 항목 | 결정 |
|------|------|
| 메트릭 수집 | Prometheus |
| 시각화 | Grafana |
| APM | OpenTelemetry (선택) |
| 알림 | PagerDuty / Slack 연동 |

---

### 4-3. CI/CD 파이프라인

| 단계 | 도구 |
|------|------|
| 소스 관리 | Git (GitHub/GitLab) |
| CI | GitHub Actions / GitLab CI |
| 컨테이너 | Docker + Docker Compose |
| 배포 | Kubernetes (운영) / Docker Compose (스테이징) |
| 시크릿 관리 | 환경 변수 + Vault (선택) |

---

### 4-4. 테스트 전략

| 단계 | 도구 | 커버리지 목표 |
|------|------|------------|
| 단위 테스트 | pytest (Backend) / Jest (Frontend) | 80% 이상 |
| 통합 테스트 | pytest + 실제 DB | 주요 API 엔드포인트 100% |
| E2E 테스트 | Playwright | 핵심 사용자 시나리오 |
| 부하 테스트 | Locust | P95 응답 목표 달성 확인 |
| 보안 테스트 | OWASP ZAP / Bandit | CI 단계 자동 실행 |

## 6. 핵심 운영 지표 (SLO)

| 지표 | 목표 |
|------|------|
| API 응답 시간 P95 | < 500ms (검색/문서 조회) |
| RAG 응답 TTFT P95 | < 2초 |
| 가용성 | 99.5% (월 기준) |
| 오류율 | < 0.1% (5xx 기준) |
| 백업 복구 목표 시간(RTO) | < 4시간 |
| 백업 복구 목표 시점(RPO) | < 24시간 |

---

## 7. 완료 기준

- OWASP Top 10 취약점이 코드베이스에 없다
- 모든 API 엔드포인트에 인증/인가가 적용된다
- 구조화 로그가 수집되고 검색 가능하다
- 핵심 메트릭 알림이 설정되어 있다
- 단위 테스트 커버리지 80% 이상을 달성한다
- CI/CD 파이프라인이 PR 단계부터 자동 실행된다
- DB 백업이 자동으로 실행되고 복구 테스트를 통과한다
- Runbook이 작성되어 있어 담당자 없이도 장애 대응 가능하다
