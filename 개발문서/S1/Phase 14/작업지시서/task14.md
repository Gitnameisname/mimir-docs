# Task 14-14. 배치 작업 관리 UI 구현

## 1. 작업 목적

관리자가 배치 작업(인덱싱, 벡터화, 정리 등)의 상태를 조회하고 수동 실행·스케줄 관리를 할 수 있는 UI를 구현한다.

이 작업의 목표는 다음과 같다.

- 배치 작업 목록/상태 조회 UI
- 작업 수동 실행 기능
- 작업 스케줄 조회/변경
- 작업 취소 기능
- 배치 작업 관리 API 확장

---

## 2. 작업 범위

### 포함 범위

- `GET /api/v1/admin/jobs/{job_id}` — 작업 상세 조회 API (신규)
- `POST /api/v1/admin/jobs/{job_id}/run` — 수동 실행 API (신규)
- `PATCH /api/v1/admin/jobs/{job_id}/schedule` — 스케줄 변경 API (신규)
- `POST /api/v1/admin/jobs/{job_id}/cancel` — 작업 취소 API (신규)
- `/admin/jobs` UI 페이지

### 제외 범위

- 작업 로직 자체 구현 (기존 Phase에서 완성)
- 새로운 배치 작업 타입 추가

## 4. 주요 구현 대상

### 4-1. 배치 작업 관리 API 확장

```python
# GET /api/v1/admin/jobs/{job_id}
# 작업 상세 (실행 이력 포함)

Response 200:
{
  "id": "reindex_all",
  "name": "전체 검색 인덱스 재구축",
  "status": "idle",             # idle, running, failed
  "schedule": "0 2 * * *",     # cron 표현식 (매일 02:00)
  "last_run_at": "2026-04-13T02:00:00Z",
  "last_run_duration_ms": 45000,
  "last_run_result": "success",
  "next_run_at": "2026-04-14T02:00:00Z",
  "recent_runs": [
    { "started_at": "...", "finished_at": "...", "result": "success", "items_processed": 1847 },
    ...
  ]
}

# POST /api/v1/admin/jobs/{job_id}/run
# 수동 실행 (이미 running이면 409)
Response 202: { "message": "작업이 시작되었습니다", "run_id": "uuid" }

# PATCH /api/v1/admin/jobs/{job_id}/schedule
Request Body: { "schedule": "0 3 * * *" }  # cron 변경
Response 200: { "schedule": "0 3 * * *", "next_run_at": "2026-04-14T03:00:00Z" }

# POST /api/v1/admin/jobs/{job_id}/cancel
# 실행 중인 작업 취소 (running 아니면 400)
Response 200: { "message": "작업 취소가 요청되었습니다" }
```

### 4-3. 상태 표시 규칙

| 상태 | 아이콘 색상 | 설명 |
|------|---------|------|
| idle | 회색 (●) | 대기 중 |
| running | 파랑 (●) + 스피너 | 실행 중 |
| success | 초록 | 마지막 실행 성공 |
| failed | 빨강 (●) | 마지막 실행 실패 |

## 6. 완료 기준

- 배치 작업 목록과 상태가 표시된다
- 수동 실행 버튼으로 작업을 즉시 실행할 수 있다
- 실행 중인 작업을 취소할 수 있다
- 스케줄을 Cron 표현식으로 변경할 수 있다
- 최근 실행 이력이 표시된다
- UI 디자인 리뷰를 최소 5회 수행한다

---

## 7. Codex 작업 지침

- 수동 실행은 확인 다이얼로그를 표시한다 ("이 작업을 실행하시겠습니까?")
- Cron 표현식 미리보기는 `cronstrue` 등 라이브러리를 사용한다
- 실행 중인 작업은 5초 간격으로 상태를 폴링한다
- 작업 취소는 graceful shutdown을 요청하며 즉시 중단을 보장하지 않는다
- 작업 스케줄 변경은 감사 이벤트에 기록한다
