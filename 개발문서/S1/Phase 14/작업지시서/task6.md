# Task 14-6. 로그인/회원가입 UI 구현

## 1. 작업 목적

사용자가 Mimir에 접근하기 위한 로그인, 회원가입, 비밀번호 재설정 UI를 구현한다.

이 작업의 목표는 다음과 같다.

- 로그인 페이지 (이메일/비밀번호 + GitLab 로그인)
- 회원가입 페이지 (이메일/비밀번호)
- 비밀번호 재설정 요청 페이지
- 비밀번호 재설정 실행 페이지
- 이메일 인증 결과 페이지
- 반응형 레이아웃 (데스크탑/모바일 호환)

---

## 2. 작업 범위

### 포함 범위

- `/login` — 로그인 페이지
- `/register` — 회원가입 페이지
- `/forgot-password` — 비밀번호 재설정 요청 페이지
- `/reset-password` — 비밀번호 재설정 실행 페이지 (토큰 포함 URL)
- `/verify-email` — 이메일 인증 결과 페이지
- `/auth/callback` — OAuth 콜백 수신 페이지
- 폼 유효성 검사 (클라이언트 사이드)
- 에러 메시지 표시
- 로딩 상태 표시

### 제외 범위

- 인증 상태 관리 (Task 14-8에서 구현)
- 계정 관리 페이지 (Task 14-7에서 구현)

## 4. 주요 구현 대상

### 4-1. 로그인 페이지 (`/login`)

```
레이아웃:
  - 중앙 정렬 카드 형태
  - Mimir 로고 + 타이틀
  - 이메일 입력 필드
  - 비밀번호 입력 필드 (토글 표시)
  - "로그인" 버튼
  - 구분선 ("또는")
  - "GitLab으로 로그인" 버튼 (GitLab 아이콘 포함)
  - "비밀번호를 잊으셨나요?" 링크
  - "계정이 없으신가요? 회원가입" 링크

동작:
  - 이메일/비밀번호 제출 → POST /api/v1/auth/login
  - GitLab 버튼 클릭 → window.location = /api/v1/auth/oauth/gitlab
  - 로그인 성공 → Access Token 메모리 저장 → 메인 페이지 리다이렉트
  - 에러 표시: "이메일 또는 비밀번호가 올바르지 않습니다"
  - 잠금 상태: "로그인 시도 횟수를 초과했습니다. 15분 후 다시 시도해주세요"
```

---

### 4-2. 회원가입 페이지 (`/register`)

```
레이아웃:
  - 이메일 입력 필드
  - 이름 입력 필드
  - 비밀번호 입력 필드 (강도 표시 바)
  - 비밀번호 확인 필드
  - "회원가입" 버튼
  - "이미 계정이 있으신가요? 로그인" 링크

클라이언트 유효성 검사 (Zod):
  - 이메일 형식 검증
  - 비밀번호 최소 8자
  - 영문+숫자+특수문자 중 2종 이상
  - 비밀번호 확인 일치
  - 이름 1자 이상

비밀번호 강도 표시:
  - 약함 (1종만) → 빨강
  - 보통 (2종) → 주황
  - 강함 (3종) → 초록
```

---

### 4-3. 비밀번호 재설정 요청 (`/forgot-password`)

```
레이아웃:
  - 안내 문구: "가입한 이메일을 입력하면 재설정 링크를 보내드립니다"
  - 이메일 입력 필드
  - "재설정 링크 발송" 버튼
  - "로그인으로 돌아가기" 링크

동작:
  - 제출 → POST /api/v1/auth/forgot-password
  - 성공 → "이메일이 발송되었습니다. 받은편지함을 확인해주세요" 메시지
  - 이메일 존재 여부와 무관하게 동일 메시지 표시
```

---

### 4-4. 비밀번호 재설정 실행 (`/reset-password?token=...`)

```
레이아웃:
  - 새 비밀번호 입력 필드 (강도 표시 바)
  - 비밀번호 확인 필드
  - "비밀번호 변경" 버튼

동작:
  - URL에서 token 파라미터 추출
  - 제출 → POST /api/v1/auth/reset-password { token, new_password }
  - 성공 → "비밀번호가 변경되었습니다" + 로그인 페이지 링크
  - 토큰 만료/무효 → "링크가 만료되었습니다. 재설정을 다시 요청해주세요"
```

---

### 4-5. OAuth 콜백 페이지 (`/auth/callback`)

```
동작:
  - URL fragment에서 access_token, expires_in 추출
  - Access Token을 메모리에 저장 (Task 14-8 AuthContext와 연계)
  - 메인 페이지로 리다이렉트
  - 에러 파라미터 있으면 에러 메시지 표시 + 로그인 페이지 링크

화면:
  - 로딩 스피너 + "로그인 처리 중..." 텍스트
  - 에러 시 에러 메시지 표시
```

---

### 4-6. 반응형 디자인

```
데스크탑 (1024px+):
  - 로그인 카드 최대 너비 420px
  - 좌우 여백 자동 중앙 정렬

태블릿 (768px ~ 1023px):
  - 카드 너비 80%

모바일 (767px 이하):
  - 카드 너비 100% (패딩만 유지)
  - 입력 필드 전체 너비
```

## 6. 완료 기준

- 로그인 페이지에서 이메일/비밀번호와 GitLab 로그인이 모두 가능하다
- 회원가입 폼에서 클라이언트 유효성 검사가 동작한다
- 비밀번호 강도 표시 바가 실시간으로 업데이트된다
- 비밀번호 재설정 전체 플로우(요청→이메일→재설정)가 동작한다
- OAuth 콜백 처리 후 메인 페이지로 정상 리다이렉트된다
- 모든 페이지가 모바일/데스크탑에서 반응형으로 동작한다
- 에러 상태가 사용자 친화적 메시지로 표시된다

## 7. Codex 작업 지침

- 기존 Phase 6/7 UI 컴포넌트 스타일을 일관되게 유지한다
- 비밀번호 필드에 paste를 허용한다 (비밀번호 관리자 호환)
- 에러 메시지에 기술적 상세 정보(stack trace 등)를 노출하지 않는다
- GitLab 로그인 버튼은 GitLab 브랜드 가이드를 참고한다
- 모든 폼은 Enter 키로 제출 가능하도록 한다
- UI 디자인 리뷰를 최소 5회 수행하여 개선한다 (CLAUDE.md 규칙)
