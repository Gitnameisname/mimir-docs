# Task1_api_architecture_principles.md
# 플랫폼 API 아키텍처 원칙 정의

---

## 1. 문서 목적

이 문서는 Mimir 문서 플랫폼 전반에서 사용할 **플랫폼 API의 상위 아키텍처 원칙**을 확정한다.

이 문서는 단순한 REST 스타일 가이드가 아니다. 문서 관리, 버전 관리, 권한 제어, 감사 추적, 검색, 이벤트, 웹훅, AI/RAG 연계 기능까지 모두 수용할 수 있도록, 플랫폼 API의 역할, 책임 경계, 설계 우선순위, 확장 원칙을 먼저 확정하는 기준 문서다.

**이 문서는 Task 3-2부터 Task 3-12까지의 기준선이다.** 이후 하위 Task는 이 문서에서 정의한 원칙에서 벗어나는 결정을 내릴 때 반드시 명시적 이유를 제시해야 한다.

## 2. 플랫폼 API의 역할

### 2-1. API는 플랫폼의 계약 계층(Contract Layer)이다

Mimir의 API는 단순한 프론트엔드-백엔드 연결 수단이 아니다.

| 관점 | 설명 |
|---|---|
| 공식 인터페이스 계층 | 플랫폼 내부 도메인 기능을 외부에 노출하는 유일한 공식 경로 |
| 다중 소비자 기반 | User UI, Admin UI, 외부 서비스, AI 에이전트가 공통으로 사용하는 기반 |
| 안정된 계약 | 도메인 모델과 소비자 사이를 연결하는 명시적 계약(contract)이며, 내부 구현 변경에 독립적이어야 함 |
| 보안/추적성/일관성의 기준점 | 권한 강제, 감사 추적, 요청 추적의 단일 진입점 |
| 확장의 구조적 토대 | 미래의 이벤트 기반 확장, AI 연계, 외부 통합을 처음부터 수용할 수 있는 구조 |

### 2-2. "API = 엔드포인트 모음"이 아니다

엔드포인트는 계약의 표현 수단일 뿐이다. 플랫폼 API는 다음을 보장해야 한다.
- **누가 무엇을 할 수 있는지**에 대한 일관된 규칙
- **어떤 형태로 응답하는지**에 대한 예측 가능한 구조
- **무슨 일이 있었는지**에 대한 추적 가능성
- **앞으로 무엇이 추가되어도**흔들리지 않는 기반 구조

## 3. API의 주요 소비자 유형

플랫폼 API는 다음 다섯 가지 소비자 유형을 공식적으로 지원해야 한다.

### 3-1. User UI

| 항목 | 내용 |
|---|---|
| 사용 이유 | 문서 열람, 작성, 버전 관리, 검색 등 일반 사용자 기능 제공 |
| 기대하는 인터페이스 특성 | 빠른 응답, 페이지네이션, 부분 로딩, 명확한 오류 메시지 |
| 민감한 요소 | UX 일관성, 응답 속도, 권한별 UI 가시성 |
| 공통 API 사용 가능 여부 | 가능. 별도 UI 전용 API 불필요 |
| 별도 계층 필요 기준 | 성능 최적화를 위한 aggregation이 필요한 경우, BFF(Backend for Frontend) 패턴을 제한적으로 고려 가능하되 계약 원칙은 동일 적용 |

### 3-2. Admin UI

| 항목 | 내용 |
|---|---|
| 사용 이유 | 조직/사용자 관리, 문서 타입 구성, 권한 정책 관리, 감사 로그 조회 |
| 기대하는 인터페이스 특성 | 세부 제어 가능성, 관리 전용 엔드포인트 분리, 강화된 감사 |
| 민감한 요소 | 권한 범위 초과 방지, 운영 안정성, 감사 기록 |
| 공통 API 사용 가능 여부 | 기본 리소스 조회/수정은 공통 API 사용. Admin 전용 기능은 `/admin` 네임스페이스로 분리 |
| 별도 계층 필요 기준 | 일반 사용자에게 노출해서는 안 되는 기능(예: 사용자 역할 재설정, 시스템 설정 변경)은 admin 네임스페이스 필요 |

### 3-3. External Application / Integration

| 항목 | 내용 |
|---|---|
| 사용 이유 | 외부 시스템이 문서 데이터를 읽거나 처리 결과를 쓰는 연동 |
| 기대하는 인터페이스 특성 | 안정적 API 계약, 버전 관리, 기계 가독성 높은 응답 구조 |
| 민감한 요소 | API 버전 호환성, 인증 토큰 관리, rate limit |
| 공통 API 사용 가능 여부 | 가능. 동일 플랫폼 API 사용 |
| 별도 계층 필요 기준 | 공개 API와 내부 서비스 API를 구분해야 하는 경우, API 게이트웨이 레벨에서 처리 |

### 3-4. AI / RAG / Agent Consumer

| 항목 | 내용 |
|---|---|
| 사용 이유 | 문서 구조 탐색, 노드 단위 콘텐츠 접근, 검색, 인용 식별자 기반 참조 |
| 기대하는 인터페이스 특성 | 구조화된 데이터 접근, 청크/노드 단위 접근, 버전 명시적 접근, 인덱싱 상태 조회 |
| 민감한 요소 | 접근 가능한 범위 제어, 인용 추적 가능성, 일관된 식별자 |
| 공통 API 사용 가능 여부 | 원칙적으로 공통 API 위에서 동작. AI 전용 우회 경로 생성 금지 |
| 별도 계층 필요 기준 | AI-specific read 최적화가 필요한 경우 공통 API의 쿼리 확장으로 처리. 별도 비공식 API 생성 금지 |

### 3-5. Internal Service Consumer

| 항목 | 내용 |
|---|---|
| 사용 이유 | 동일 플랫폼 내 다른 서비스(예: 검색 인덱서, 이벤트 처리기)가 API를 소비하는 경우 |
| 공통 API 사용 가능 여부 | 가능. 내부 서비스도 공식 API 계약을 따름. 내부 전용 숏컷 생성 금지 |

**원칙: 공통 플랫폼 API를 우선하되, 정말 필요한 경우에만 별도 특화 계층을 둔다.** 모든 소비자는 동일한 보안/계약/관측성 원칙에 따른다.

## 4. 플랫폼 API 아키텍처의 상위 원칙

### 원칙 1. API-first

**설명**: API는 UI보다 먼저 설계되고 독립적으로 완결되어야 한다. UI는 API의 소비자 중 하나일 뿐이다.

**왜 필요한가**: UI와 API를 동시에 만들면 API가 특정 UI에 맞게 변형된다. 플랫폼 API는 UI가 없어도 완전해야 한다.

**설계에서 어떤 결정을 유도하는가**:
- UI 편의를 위해 API 계약을 변형하지 않는다
- API 설계는 UI 요구사항보다 도메인 모델과 리소스 구조를 먼저 반영한다
- API 문서는 구현과 동기화된 공식 산출물이다

---

### 원칙 2. Resource-oriented

**설명**: API는 동작(action)이 아니라 리소스(resource)를 중심으로 구조화된다.

**왜 필요한가**: 기능 중심 엔드포인트는 무한정 늘어나고 일관성이 무너진다. 리소스 중심 구조는 발견 가능성과 일관성을 보장한다.

**설계에서 어떤 결정을 유도하는가**:
- `/documents`, `/versions`, `/nodes` 같은 명사 중심 URI 구조
- HTTP 메서드(GET/POST/PUT/PATCH/DELETE)로 행위를 표현
- 기능 명령형 엔드포인트(예: `/doPublish`, `/runSearch`) 남발 금지
- 단, long-running 작업, 이벤트, 웹훅 같은 확장 패턴은 별도 처리 (원칙 6 참조)

---

### 원칙 3. Consistency over Convenience

**설명**: 소비자 편의를 위한 단기적 변형보다 전체 API의 일관성을 우선한다.

**왜 필요한가**: 제각각인 응답 구조, 오류 형식, 파라미터 이름은 소비자에게 숨겨진 학습 비용을 발생시킨다. 일관성은 장기적으로 더 낮은 비용이다.

**설계에서 어떤 결정을 유도하는가**:
- 모든 엔드포인트가 동일한 응답 envelope 구조를 따름
- 목록 조회 파라미터 이름과 동작 방식을 전 리소스에서 통일
- 오류 응답 형식은 어디서나 동일한 구조

---

### 원칙 4. Secure by Default

**설명**: 인증/인가 정보 없는 요청은 기본으로 거부된다. 보안은 옵션이 아닌 기본 상태다.

**왜 필요한가**: "나중에 보안 추가"는 구조적 결함을 낳는다. 모든 요청은 누가 무엇을 할 수 있는지 판단할 수 있어야 한다.

**설계에서 어떤 결정을 유도하는가**:
- 인증 컨텍스트 없는 요청은 기본 거부
- 공개 접근이 필요한 엔드포인트는 명시적 예외로 선언
- 권한 검증은 API 레벨에서 강제되며, UI 숨김만으로 대체 불가

---

### 원칙 5. Extensible by Design

**설명**: 문서 타입이 늘어나고 기능이 확장되어도 API 구조가 무너지지 않도록 확장 가능성을 처음부터 설계에 반영한다.

**왜 필요한가**: 단기 편의로 만든 구조는 기능 추가 시마다 누더기가 된다.

**설계에서 어떤 결정을 유도하는가**:
- 문서 타입을 API 경로에 하드코딩하지 않음 (예: `/regulations` 대신 `/documents?type=regulation`)
- 확장 포인트(async 작업, 이벤트, 웹훅, AI 연계)를 막는 구조 금지
- 버전 관리 전략을 처음부터 적용

---

### 원칙 6. Observable by Default

**설명**: 모든 API 요청은 추적 가능해야 하며, 운영 중 디버깅과 감사가 가능해야 한다.

**왜 필요한가**: 운영 단계에서 문제 추적이 불가능한 API는 유지보수 비용이 폭증한다.

**설계에서 어떤 결정을 유도하는가**:
- 모든 요청에 `request_id` / `trace_id` 부여
- 오류 응답은 사람이 읽을 수 있고 동시에 기계 처리 가능한 구조
- 감사 로그와 API 로그는 역할이 다르므로 분리 설계

---

### 원칙 7. AI-integratable

**설명**: AI/RAG/에이전트는 플랫폼의 정식 API 소비자다. AI 연계는 부가 기능이 아니라 공식 요구사항이다.

**왜 필요한가**: AI를 위한 비공식 우회 경로를 만들면 보안/일관성/감사 원칙이 무력화된다.

**설계에서 어떤 결정을 유도하는가**:
- 문서 구조, 버전, 노드, 메타데이터를 API로 접근 가능하게 설계
- retrieval/citation/청크 단위 접근을 고려한 리소스 구조
- AI 전용 비공식 우회 API 생성 금지

---

### 원칙 8. Backward-compatibility Aware

**설명**: API 변경은 기존 소비자를 깨뜨리지 않는 방향으로 관리된다.

**왜 필요한가**: 외부 연동, AI 소비자, 다중 클라이언트가 있는 플랫폼에서 하위 호환성 파괴는 직접적인 운영 장애를 유발한다.

**설계에서 어떤 결정을 유도하는가**:
- API 버전 관리 전략 필수 (Task 3-4에서 구체화)
- 필드 추가는 허용, 필드 제거/타입 변경은 버전 업을 통해 처리
- deprecation 정책 명시

---

### 원칙 9. Explicit Contracts over Implicit Behavior

**설명**: API의 동작은 암묵적 가정이 아니라 명시적 계약으로 표현된다.

**왜 필요한가**: 암묵적 동작은 소비자마다 다르게 해석되고 결국 버그와 혼란을 유발한다.

**설계에서 어떤 결정을 유도하는가**:
- 응답 필드의 의미와 조건을 문서화
- 빈 값, null, 생략의 차이를 명시
- 오류 코드와 의미를 명시적으로 정의

## 5. 계층 경계와 책임 분리 원칙

플랫폼을 구성하는 각 계층의 책임을 명확히 구분한다.

### 5-1. 계층 구조

```
┌─────────────────────────────────────────────┐
│              API Layer                       │
│  요청 수신, 입출력 계약, 문맥 추출, 응답 구성  │
├─────────────────────────────────────────────┤
│          Authorization Layer                 │
│  주체/테넌트/리소스 기반 권한 판단            │
├─────────────────────────────────────────────┤
│       Application / Service Layer            │
│  유스케이스 조율, 도메인 서비스 호출          │
├─────────────────────────────────────────────┤
│            Domain Layer                      │
│  핵심 비즈니스 규칙, 도메인 모델             │
├─────────────────────────────────────────────┤
│     Audit / Logging / Event Layer            │
│  감사 기록, 이벤트 발행, 운영 로깅           │
├─────────────────────────────────────────────┤
│    Background / Async Processing Layer       │
│  장기 작업, 비동기 처리, 스케줄링            │
└─────────────────────────────────────────────┘
```

### 5-2. API Layer의 책임

API Layer가 해야 할 것:
- 요청 파싱 및 검증 (입력 형식 검증)
- 인증 컨텍스트 추출 (주체 식별)
- 서비스 계층 호출
- 응답 구성 및 직렬화
- 오류 변환 및 응답

API Layer가 하지 않아야 할 것:
- 직접적인 비즈니스 규칙 판단
- 인가 로직 직접 작성 (API 레이어에 if문으로 흩뿌리기 금지)
- DB 직접 접근
- 감사 로그 직접 기록 (감사 레이어에 위임)

### 5-3. Authorization Layer의 책임

- 인증된 주체(subject), 조직/테넌트 문맥, 요청 리소스를 기반으로 권한 판단
- 권한 판단 로직의 단일 집중점 — API 라우터마다 중복 하드코딩 금지
- 권한 거부 시 명시적 오류 반환

### 5-4. Service/Domain Layer의 책임

- 비즈니스 규칙은 서비스/도메인 계층에 집중
- API가 어떻게 호출되었는지에 무관하게 동작해야 함 (HTTP 의존 금지)

### 5-5. Audit / Event Layer의 책임

- 감사 로그(누가 언제 무엇을 했는지)는 API와 연결되지만 API 코드에 직접 삽입하지 않음
- 이벤트 발행은 도메인 계층 또는 서비스 계층에서 후킹
- API 운영 로그(요청/응답 로그)와 감사 로그는 분리 설계

## 6. REST 우선 전략과 예외 처리 원칙

### 6-1. REST 우선

- 기본 인터페이스는 HTTP 기반 REST API로 설계
- 리소스 중심 URI 구조 우선
- HTTP 메서드로 행위 표현
- 기능 명령형 엔드포인트 남발 금지

### 6-2. REST로 처리하기 어려운 패턴

아래 패턴은 REST 단독으로 다루기 어렵다.

| 패턴 | 설명 | 처리 방향 |
|---|---|---|
| Long-running Jobs | 완료까지 시간이 걸리는 작업 | `operations` 리소스로 비동기 처리 |
| Event Delivery | 실시간 이벤트 스트리밍 | REST 기반 polling 또는 webhook 구독 |
| Webhook Subscription | 외부 시스템으로 이벤트 전달 | `webhooks` 리소스 + delivery 메커니즘 |
| Search / Retrieval | 복합 조건 검색, 시맨틱 검색 | `searches` 리소스 또는 `documents/search` 형태로 편입 |
| AI Composite Access | 구조+콘텐츠 복합 접근 | 기존 리소스 쿼리 확장으로 처리 |

**원칙: "REST를 버린다"가 아니라, REST 기반 플랫폼 위에 확장 계층을 추가하는 방식으로 설계한다.**

## 7. 보안 및 권한 연계 관점의 원칙

*(인증/인가 상세 설계는 Task 3-3에서 구체화. 여기서는 원칙만 확정.)*

| 원칙 | 설명 |
|---|---|
| 기본 불신 원칙 | 인증 정보 없는 요청은 기본 거부. 공개 접근은 명시적 예외로 선언 |
| 주체/문맥 보유 원칙 | 모든 요청은 주체(subject), 조직/테넌트 문맥을 가져야 한다 |
| API 레벨 강제 원칙 | 권한 검증은 API 레벨에서 강제. UI 숨김만으로 대체 불가 |
| 감사 가능성 원칙 | 주요 요청은 추적 가능해야 하며, 감사 기록이 가능해야 한다 |
| 외부 연동 동일 원칙 | 외부 시스템 연계 시에도 동일한 보안 모델 적용 |

## 8. 확장성 원칙

### 8-1. 문서 타입 확장

- 문서 타입을 API 경로나 엔드포인트에 하드코딩 금지
- 타입은 도메인 모델 레벨에서 구성 가능한 속성으로 처리
- API 구조는 타입 무관하게 일관성 유지

### 8-2. 기능 확장

- 새로운 기능 추가가 기존 계약을 파괴하지 않는 구조
- 확장 포인트(async 작업, 이벤트, 웹훅, AI 연계)를 막는 설계 금지
- 확장은 새 리소스 또는 기존 리소스의 쿼리/필드 확장으로 처리

### 8-3. 소비자 확장

- 새로운 소비자 유형이 추가되어도 기존 API 계약 변경 불필요
- 소비자별 특화 요구는 쿼리 파라미터, 필드 선택(sparse fieldsets), projection으로 수용

## 9. AI/RAG 친화적 인터페이스 관점

### 9-1. AI는 플랫폼의 정식 API 소비자다

- AI/RAG/에이전트를 위한 별도 비공식 API 생성 금지
- 동일한 플랫폼 API 체계 안에서 AI 소비자의 요구를 수용

### 9-2. AI가 필요로 하는 데이터 접근 구조

AI 소비자가 필요한 것:
- 문서 전체 본문 뿐만 아니라 **구조(노드 트리)** 접근
- **버전 명시적** 접근 (특정 버전의 특정 노드)
- **메타데이터, 상태 정보** 접근
- **청크/노드 단위** 콘텐츠 접근
- 인용(citation)을 위한 안정적 **식별자**
- 인덱싱 상태, 처리 상태 조회

### 9-3. 현 단계 처리 방향

- 검색 알고리즘, 벡터 DB, 임베딩 모델은 구현 미결정 사항
- 그러나 **API 인터페이스는 이를 수용할 수 있는 구조**여야 함
- retrieval/citation/청크 접근 API는 후속 Task에서 구체화 (Task 3-9)

## 10. 운영 및 관측성 관점의 원칙

### 10-1. 요청 추적 가능성

- 모든 API 요청에 `request_id` 부여 (클라이언트 제공 또는 서버 생성)
- 분산 추적을 위한 `trace_id` / `span_id` 전달 구조 고려
- 요청 메타데이터는 응답 헤더 또는 응답 envelope에 포함

### 10-2. 오류 응답 구조

- 오류 응답은 사람이 읽을 수 있어야 하고 동시에 기계 처리 가능
- 오류 코드(machine-readable), 메시지(human-readable), 세부 정보(details) 분리
- 오류 구조는 전 API에서 일관 (Task 3-10에서 구체화)

### 10-3. 감사 로그 vs API 운영 로그 분리

| 구분 | API 운영 로그 | 감사 로그 |
|---|---|---|
| 목적 | 디버깅, 성능 모니터링 | 누가 무엇을 했는지 기록 |
| 포함 정보 | 요청/응답, 응답 시간, 오류 코드 | 주체, 리소스, 행위, 결과, 시각 |
| 소비자 | 개발자, 운영팀 | 감사자, 보안팀, 컴플라이언스 |
| 보존 정책 | 단기 | 장기 |

### 10-4. 운영 단계 디버깅 가능성

- API 오류는 어디서 발생했는지 추적 가능해야 함
- `request_id`로 로그 추적이 가능한 구조
- 내부 오류 세부사항은 외부 응답에 노출하지 않되, 내부 추적은 가능하게

## 12. 결론

Mimir 플랫폼 API는 단순한 백엔드 구현 수단이 아니라, 플랫폼의 공식 계약 계층(Contract Layer)이다.

이 문서에서 확정된 9개 원칙은 이후 모든 API 설계 결정의 기준선이다:

1. **API-first** — UI보다 API가 먼저
2. **Resource-oriented** — 리소스 중심 구조
3. **Consistency over Convenience** — 편의보다 일관성
4. **Secure by Default** — 기본 불신, 명시적 허가
5. **Extensible by Design** — 확장을 막는 구조 금지
6. **Observable by Default** — 기본적으로 추적 가능
7. **AI-integratable** — AI는 정식 소비자
8. **Backward-compatibility Aware** — 하위 호환성 관리
9. **Explicit Contracts over Implicit Behavior** — 명시적 계약

이 원칙들로부터 벗어나는 결정은 명시적 이유와 함께 별도 기록되어야 한다.
