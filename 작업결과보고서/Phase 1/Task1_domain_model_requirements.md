# Phase 1 - Task 1-1. 범용 문서 도메인 모델 요구사항 정리

## 1. 작업 목적

Mimir는 특정 문서 유형 하나만을 위한 시스템이 아니라,
규정집·매뉴얼·보고서·회의록·기술 문서·지식 문서·템플릿 문서 등
다양한 종류의 문서를 공통된 구조로 수용하는 범용 문서 플랫폼이다.

이 문서는 **구현 이전 단계**로서, 이후 Document / Version / Node / DocumentType / metadata / 상태 모델 설계의 기준이 될 요구사항을 체계적으로 정리한다.

---

## 2. 분석 대상 문서 유형

Mimir가 지원해야 할 최소 문서 유형은 아래 7종이다.

| 유형 ID | 유형명 | 설명 |
|---|---|---|
| `policy` | 규정·정책 문서 | 조직의 규칙·규범·규정집·정책 선언 |
| `manual` | 매뉴얼·가이드 | 절차 안내, 사용 방법, 운영 지침 |
| `technical_doc` | 기술 문서 | 시스템·API·아키텍처 설명 문서 |
| `report` | 보고서 | 특정 기간/사안에 대한 결과 보고 |
| `meeting_note` | 회의록 | 회의 일시·참석자·안건·결정사항 기록 |
| `knowledge_doc` | 지식 문서 | 백과사전형, 노하우, 레퍼런스 정보 |
| `template_doc` | 템플릿 문서 | 반복 사용하는 빈 양식·구조 정의 |

---

## 3. 문서 유형별 특성 요약

| 유형 | 문서 목적 | 구조적 특징 | 버전 관리 필요성 | 메타데이터 특징 | 상태 관리 필요성 |
|---|---|---|---|---|---|
| `policy` | 조직 내 규칙·기준 선언 | 조항 번호 계층 구조, 섹션 체계 강함 | 높음 (개정 이력이 법적 의미를 가짐) | 시행일, 개정일, 관할부서, 승인자 | 높음 (초안→검토→공표→폐기) |
| `manual` | 절차·방법 안내 | 단계별 순서, 목차 구조, 주석 포함 | 중간 (제품/프로세스 변경 시 개정) | 대상 독자, 적용 버전, 작성부서 | 중간 (초안→검토→배포) |
| `technical_doc` | 시스템·구조 설명 | 다이어그램 참조, 코드 블록, 복합 구조 | 높음 (시스템 버전과 연동) | API 버전, 시스템명, 연관 컴포넌트 | 중간 (초안→검토→공개) |
| `report` | 결과·현황 보고 | 표·차트 중심, 서술형 혼합, 결론 강조 | 낮음 (보고 시점 고정, 소급 수정 드묾) | 작성부서, 대상기간, 보고 대상자 | 낮음 (초안→제출 완료) |
| `meeting_note` | 회의 내용 기록 | 항목 나열형, 참석자 목록, 결정사항 | 낮음 (사후 수정은 예외적) | 회의일시, 참석자, 장소, 안건 목록 | 낮음 (초안→확정) |
| `knowledge_doc` | 지식·정보 정리 | 자유 형식, 링크·참조 구조 | 중간 (지속 업데이트) | 카테고리, 태그, 관련 문서 | 낮음 (초안→게시) |
| `template_doc` | 재사용 양식 제공 | 구조가 곧 내용, 플레이스홀더 포함 | 중간 (양식 변경 시 개정) | 양식 용도, 대상 부서, 필수 항목 | 중간 (초안→승인→활성) |

## 4. 공통 요구사항

모든 문서 유형에서 공통으로 요구되는 핵심 요소는 다음과 같다.

| 범주 | 공통 요구사항 |
|---|---|
| 식별 | 문서는 시스템 전체에서 고유한 식별자를 가져야 한다 |
| 식별 | 문서는 사람이 이해할 수 있는 제목을 가져야 한다 |
| 식별 | 문서는 문서 유형 분류를 가져야 한다 |
| 구조 | 문서 본문은 계층 구조로 표현 가능해야 한다 |
| 구조 | 문서는 단순 본문부터 복합 구조까지 동일한 모델로 표현 가능해야 한다 |
| 버전 | 문서는 하나 이상의 버전을 가질 수 있어야 한다 |
| 버전 | 현재 버전(current)과 과거 버전을 구분할 수 있어야 한다 |
| 메타데이터 | 문서는 생성자·생성일시·최종수정일시를 기록해야 한다 |
| 메타데이터 | 문서 유형별 확장 메타데이터를 수용할 수 있어야 한다 |
| 상태 | 문서는 최소한 생명주기를 나타내는 상태를 가져야 한다 |
| 확장 | 새로운 문서 유형이 추가될 때 코어 모델을 변경하지 않아야 한다 |

## 5. 유형별 차이점

문서 유형 간에 달라지는 부분이 공통 코어 + 확장 구조가 필요한 이유다.

### 5-1. 구조 복잡도 차이

| 유형 | 구조 복잡도 | 설명 |
|---|---|---|
| `policy` | 높음 | 조항·항·호의 다단계 계층, 참조 조항 연결 |
| `manual` | 중간~높음 | 목차 → 섹션 → 단계 순서 구조 |
| `technical_doc` | 높음 | 코드 블록, 다이어그램, 참조, 외부 링크 혼합 |
| `report` | 중간 | 표·차트 + 서술형 단락, 구조는 단순 |
| `meeting_note` | 낮음 | 나열형 항목, 단순 계층 |
| `knowledge_doc` | 자유 | 단락 중심, 구조 제약 없음 |
| `template_doc` | 중간 | 구조 자체가 산출물, 플레이스홀더 필요 |

### 5-2. 필수 메타데이터 차이

| 유형 | 유형 전용 필수 메타데이터 예시 |
|---|---|
| `policy` | 시행일, 개정일, 관할부서, 법적 근거, 승인자 |
| `manual` | 적용 대상, 대상 시스템/제품 버전, 작성부서 |
| `technical_doc` | 연관 시스템, API 버전, 아키텍처 구성요소 |
| `report` | 작성부서, 대상기간, 보고 대상자, 작성일 |
| `meeting_note` | 회의일시, 장소, 참석자 목록, 안건 목록 |
| `knowledge_doc` | 카테고리, 태그, 관련 문서 링크 |
| `template_doc` | 양식 용도, 필수 항목 목록, 활성 여부 |

### 5-3. 버전 관리 및 상태 모델 차이

| 유형 | 버전 관리 중요도 | 상태 전이 복잡도 | 비고 |
|---|---|---|---|
| `policy` | 매우 높음 | 높음 | 개정 이력이 법적·조직적 의미 보유 |
| `manual` | 높음 | 중간 | 제품 변경 시 동기화 필요 |
| `technical_doc` | 높음 | 중간 | 시스템 버전과 연동 가능성 |
| `report` | 낮음 | 낮음 | 보고 완료 후 거의 변경 없음 |
| `meeting_note` | 낮음 | 낮음 | 사후 수정은 예외적 상황 |
| `knowledge_doc` | 중간 | 낮음 | 지속 편집, 단순 게시 |
| `template_doc` | 중간 | 중간 | 양식 승인 후 활성화 |

### 5-4. 노드 구성 차이

| 유형 | 주요 노드 유형 |
|---|---|
| `policy` | section, article(조항), paragraph, reference |
| `manual` | section, step, list, warning, image |
| `technical_doc` | section, paragraph, code_block, table, diagram_ref |
| `report` | section, paragraph, table, chart_embed |
| `meeting_note` | agenda_item, participant_list, decision, action_item |
| `knowledge_doc` | paragraph, list, callout, link |
| `template_doc` | placeholder, field_group, instruction, section |

### 5-5. 템플릿 기반 여부

| 유형 | 템플릿 기반 | 설명 |
|---|---|---|
| `template_doc` | 예 (정의 주체) | 구조 자체가 재사용 양식 |
| `policy`, `report`, `meeting_note` | 부분 예 | 기관 표준 양식을 따르는 경우 많음 |
| `manual`, `technical_doc`, `knowledge_doc` | 아니오 | 자유 구조로 작성 |

## 6. 범용 도메인 모델 요구사항 목록

### 6-1. 문서 식별 요구사항

- 문서는 시스템 내에서 전역적으로 고유한 식별자를 가져야 한다.
- 문서는 사람이 이해할 수 있는 대표 제목을 가져야 한다.
- 문서는 문서 유형 분류 값을 가져야 한다.
- 문서는 소속 조직·프로젝트·카테고리 등 컨텍스트 정보를 가질 수 있어야 한다.
- 문서 식별자는 외부에 노출되는 공개 식별자와 내부 기술 식별자를 분리할 수 있어야 한다.
- 문서는 소프트 삭제(soft delete) 또는 아카이브로 비활성화할 수 있어야 한다.

### 6-2. 구조 표현 요구사항

- 문서 본문은 계층적 트리 구조로 표현 가능해야 한다.
- 단일 본문 텍스트 문서부터 다단계 섹션을 가진 복합 구조 문서까지 동일한 모델로 표현 가능해야 한다.
- 트리 노드는 최소한 섹션, 문단, 리스트, 표 등의 유형을 수용해야 한다.
- 노드는 자신의 유형, 순서, 부모-자식 관계를 명확히 표현해야 한다.
- 구조(데이터 모델)와 표현(렌더링)은 분리되어야 한다.
- 미래에 에디터·렌더러 플러그인이 추가되어도 저장 구조는 변경되지 않아야 한다.

### 6-3. 버전 관리 요구사항

- 문서는 하나 이상의 버전을 가질 수 있어야 한다.
- 각 버전은 해당 시점의 문서 내용 전체를 독립적으로 보존해야 한다.
- 현재 유효한 버전(current version)이 명확히 식별되어야 한다.
- 버전은 생성 후 변경되지 않는 불변성(immutability)을 원칙으로 해야 한다.
- 특정 과거 버전 상태를 조회·복원할 수 있어야 한다.
- 버전 간 비교가 가능한 구조여야 한다.
- 버전은 생성자·생성일시·변경 설명을 기록해야 한다.

### 6-4. 메타데이터 요구사항

- 문서는 모든 유형에 공통으로 적용되는 최소 공통 메타데이터를 가져야 한다.
- 문서 유형별로 추가 메타데이터 필드를 수용할 수 있어야 한다.
- 공통 메타데이터와 유형별 확장 메타데이터는 구조적으로 분리되어야 한다.
- 일부 메타데이터 필드는 검색 및 필터링의 기준으로 활용 가능해야 한다.
- 확장 메타데이터는 유형별 스키마로 검증 가능한 구조여야 한다.
- 메타데이터 구조는 코어 모델 변경 없이 유형별로 독립 확장 가능해야 한다.

### 6-5. 상태 관리 요구사항

- 문서는 최소한 `draft`, `review`, `published`, `archived` 상태를 가져야 한다.
- 상태 전이는 정의된 규칙을 따라야 하며, 임의 전이는 허용하지 않아야 한다.
- 상태 모델은 문서 유형에 따라 단순화 또는 확장 가능해야 한다.
- 삭제와 아카이브는 명확히 구분되어야 한다.
- 이후 승인 워크플로 확장 시 상태 모델과 자연스럽게 통합 가능해야 한다.

### 6-6. 확장성 요구사항

- 새로운 문서 유형이 추가될 때 코어 모델(Document, Version, Node)을 변경하지 않아야 한다.
- 유형별 노드 구성 규칙을 코어와 독립적으로 정의할 수 있어야 한다.
- 유형별 메타데이터 스키마를 코어와 독립적으로 확장할 수 있어야 한다.
- 향후 권한 체계, 승인 워크플로, 검색 인덱싱 구조와 충돌하지 않아야 한다.
- 사용자 정의 문서 유형(custom type)을 나중에 수용할 수 있는 구조여야 한다.

## 7. 설계 제약사항 및 원칙

| 제약사항 | 설명 |
|---|---|
| 유형 종속 금지 | 특정 문서 유형 하나를 기준 모델로 설계하지 않는다. 규정집 중심이나 보고서 중심으로 설계하면 나머지 유형을 억지로 맞춰야 한다. |
| 구조와 표현 분리 | 저장 구조는 의미·계층만 표현하고, 렌더링 방식은 별도 계층에서 결정한다. |
| 범용성과 단순성의 균형 | 과도한 범용성은 복잡도를 증가시킨다. "공통 코어 + 필요 시 확장" 패턴을 원칙으로 한다. |
| MVP와 장기 확장 분리 | 초기 구현에 필요한 핵심 구조와 나중에 추가할 확장 요소를 설계 단계에서 구분한다. |
| 상태·버전·메타데이터 독립 | 세 요소는 각각 독립된 책임을 가지며, 상호 강결합하지 않는다. |
| 불변성 원칙 | 확정된 버전의 내용은 변경되지 않는다. 수정 = 새 버전 생성. |
| 검증 책임 위치 | 메타데이터 유효성 검증은 DB 레벨이 아닌 서비스 레이어에서 담당한다. |

## 8. 다음 Task로 넘길 핵심 결론

### Task 1-2 (핵심 엔티티 식별)로 전달할 결론

**코어 엔티티는 3개다:**
- `Document` — 문서의 정체성 (유형, 상태, 대표 정보)
- `Version` — 특정 시점의 문서 내용 스냅샷 (불변)
- `Node` — 문서 내부 구조의 최소 단위 (트리 형태)

**부가 엔티티 검토가 필요한 항목:**
- `DocumentType` — 분류값 + 확장 규칙 정의 역할
- `Attachment` — 파일 첨부 (문서 유형 무관하게 공통 필요)
- `Audit/Event` — 변경 이력 추적 (승인 워크플로 확장 대비)
- `Reference` — 문서 간 참조 관계 (policy 유형에서 특히 필요)

**설계 판단이 필요한 핵심 쟁점:**

| 쟁점 | 권장 방향 |
|---|---|
| Document와 Version의 책임 분리 수준 | Document는 정체성만 보유, 현재 내용은 current_version 참조로 접근 |
| Version 저장 방식 | MVP: 완전 스냅샷 방식, 이후 diff 확장 고려 |

| metadata 자유도 | 공통 코어 필드 고정 + 유형별 확장 필드는 스키마 연결 방식 |
