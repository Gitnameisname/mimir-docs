# Phase 0 — 플랫폼 비전 및 목표 정의

## 1. 프로젝트 개요

Mimir는 조직이 생산하고 관리하는 지식 자산을 구조적으로 저장, 버전 관리, 탐색, 재사용할 수 있도록 지원하는 **범용 지식 플랫폼**이다. 단일 문서 유형에 종속된 파일 관리 시스템이 아니라, 문서·구조·메타데이터·버전·관계·탐색 기능을 하나의 플랫폼으로 통합하여 지식 자산 전체를 일관된 방식으로 다룰 수 있게 한다.

플랫폼은 UI 소비자를 위한 기능뿐 아니라, 외부 애플리케이션·API 클라이언트·AI 도구가 동일한 지식 자산에 접근하고 활용할 수 있는 인터페이스를 제공한다.

---

## 2. 비전

> **모든 지식 자산이 구조화되고, 추적되며, 어디서든 접근 가능한 플랫폼.**

Mimir는 조직 내 지식이 비정형 파일로 파편화되는 문제를 해결하고, 모든 문서와 구조화된 정보가 하나의 일관된 플랫폼 위에서 관리·탐색·재사용될 수 있도록 한다. 장기적으로는 UI 사용자, 외부 시스템, AI 에이전트 모두가 동일한 지식 계층에 접근하는 범용 지식 레이어가 된다.

### 핵심 가치

- **일관성**: 문서 유형에 관계없이 동일한 구조와 메타데이터 원칙이 적용된다.
- **추적 가능성**: 모든 변경은 버전으로 기록되며 이력이 보존된다.
- **접근 가능성**: API를 통해 UI, 외부 시스템, AI 도구 모두가 동등하게 접근할 수 있다.
- **확장 가능성**: 새로운 문서 유형, 메타데이터 스키마, 연계 시스템을 플랫폼 핵심을 변경하지 않고 수용할 수 있다.

---

## 3. 목표

1. 다양한 문서 유형을 수용하는 범용 문서·지식 관리 구조를 설계한다.
2. 버전 관리와 변경 이력을 플랫폼의 기본 기능으로 탑재한다.
3. 메타데이터와 구조 노드를 통해 지식을 정형화하고 탐색 가능하게 한다.
4. API-first 원칙으로 모든 핵심 기능을 외부에서 접근 가능하게 한다.
5. 일반 사용자 경험과 운영/관리 기능을 구조적으로 분리한다.

---

## 4. 핵심 문제 정의

### 4-1. 문서의 파편화
조직 내 문서가 유형별로 서로 다른 저장소, 형식, 관리 방식으로 분산되어 있어 통합적 탐색과 재사용이 불가능하다.

### 4-2. 구조와 메타데이터의 비일관성
같은 종류의 문서도 작성자·시점에 따라 구조와 메타데이터가 달라져 자동화 처리와 외부 연계가 어렵다.

### 4-3. 버전 추적의 부재
변경 이력이 파일명 기반 관리(v1, v2, 최종본...)로 처리되어 실질적인 버전 비교와 복원이 불가능하다.

### 4-4. 외부 연계의 어려움
문서가 특정 UI 시스템에 종속되어 있어, 외부 시스템이나 AI 도구가 지식 자산에 접근하려면 별도의 파싱·수집 작업이 필요하다.

### 4-5. 검색과 재사용의 비효율
문서 전체 파일 단위로만 검색·조회가 가능하고, 구조 단위 또는 메타데이터 기반의 정밀 탐색이 불가능하다.

---

## 5. 핵심 자산 정의

| 자산 | 설명 |
|------|------|
| **Document** | 플랫폼이 관리하는 최상위 지식 단위. 특정 문서 유형에 종속되지 않는 범용 엔티티. |
| **Version** | Document의 변경 단위. 모든 수정은 새 버전으로 기록되며, 이전 상태로의 접근을 보장한다. |
| **Node / Structure** | Document 내부의 구조적 요소. 섹션, 조항, 단락 등 문서 유형에 따라 달라지는 계층 구조를 표현한다. |
| **Metadata** | Document 및 Node에 부착되는 정형 속성. 유형별로 확장 가능하며, 검색·필터링·외부 연계의 기반이 된다. |
| **Relationship** | Document 또는 Node 간의 참조, 상속, 의존 관계. 지식 간 연결 구조를 표현한다. |
| **Searchable Knowledge** | 검색 인덱스로 가공된 지식 자산. 전문 검색, 구조 탐색, 메타데이터 필터링을 지원한다. |
| **Attachment / External Resource Reference** | 문서에 연결된 파일, 외부 URL, 오브젝트 스토리지 참조. 플랫폼이 직접 저장하거나 참조 경로를 관리한다. |

---

## 6. 주요 사용자군

### 일반 사용자 (End User)
- **주요 목적**: 지식 자산 탐색, 조회, 작성, 편집
- **기대 행동**: 문서 검색, 특정 버전 조회, 문서 작성 및 수정, 구조 탐색
- **필요 기능 수준**: 직관적 UI, 검색, 버전 열람, 문서 편집기

### 관리자 / 운영자 (Admin)
- **주요 목적**: 문서 유형 관리, 권한 설정, 운영 정책 적용, 시스템 상태 감사
- **기대 행동**: 문서 유형 정의, 권한 부여/회수, 상태 관리, 감사 로그 조회
- **필요 기능 수준**: Admin UI, 정책 설정, 권한 관리, 데이터 운영 도구

### 외부 시스템 / API 클라이언트
- **주요 목적**: 지식 자산 자동화 처리, 동기화, 데이터 수집
- **기대 행동**: REST API를 통한 문서 CRUD, 버전 조회, 메타데이터 쿼리
- **필요 기능 수준**: 인증된 API, 명확한 에러 포맷, 안정적인 API 버전 정책

### AI 에이전트 / 자동화 소비자
- **주요 목적**: 지식 자산을 학습·추론·생성에 활용
- **기대 행동**: 구조화된 지식 단위 조회, 버전별 diff 접근, 메타데이터 기반 필터링
- **필요 기능 수준**: 구조화된 API 응답, Node/Metadata 단위 접근, 검색 인터페이스

---

## 7. In Scope / Out of Scope

### In Scope (초기 단계 포함)

- 문서(Document) 생성, 조회, 수정, 삭제
- 버전 관리 및 변경 이력 추적
- 문서 내부 구조(Node/Structure) 관리
- 메타데이터 정의 및 부착
- 문서 유형(DocumentType) 정의 및 확장
- 전문 검색 및 메타데이터 기반 필터링
- API를 통한 모든 핵심 기능 접근
- 사용자 권한 및 접근 제어
- 첨부 파일 및 외부 리소스 참조 관리
- User UI / Admin UI 분리된 인터페이스

### Out of Scope (현재 단계 제외, 향후 검토)

- 복잡한 워크플로 엔진 전체 구현 (승인 흐름은 상태 모델 수준에서만 처리)
- 고도화된 AI 추론·생성 기능 자체 (외부 AI 도구 연계 인터페이스만 제공)
- 실시간 협업 편집 (동시 편집 충돌 해결 등)
- 특정 외부 시스템 종속 기능 (특정 ERP, DMS 전용 기능)
- 이메일·메신저 등 알림 발송 시스템 직접 구현 (연동 포인트만 정의)
- 플랫폼 자체 인증 시스템 전체 구현 (외부 인증 시스템 연동 방향)

---

## 8. 성공 기준

- 플랫폼의 정체성을 3~5문장으로 명확히 설명할 수 있다.
- 문서 시스템과 지식 플랫폼의 차이가 설계 판단에 활용 가능한 수준으로 구분된다.
- 핵심 자산 7가지가 정의되어 Phase 1 도메인 모델 설계의 출발점이 된다.
- In Scope / Out of Scope가 명확히 구분되어 이후 기능 요청의 범위 판별 기준으로 사용할 수 있다.
- Phase 1 설계 시 데이터 모델 방향을 이 문서 하나로 정당화할 수 있다.
