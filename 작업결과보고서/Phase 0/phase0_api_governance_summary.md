# Phase 0 — API 거버넌스 요약

## 문서 목적

이 문서는 Mimir API 전반에 적용할 설계 상위 규칙, 인증·권한 방향, 버전 정책, 오류 응답 원칙, 이벤트·비동기 연계 방향을 정리한다. Phase 2 API 설계에 들어가기 전 모든 설계자가 공유해야 할 기준이다.

---

## 1. 설계 상위 규칙

### 1-1. 리소스 중심 명명 규칙

- API 경로는 도메인 리소스를 기준으로 명명한다.
- 리소스 이름은 명사 복수형을 사용한다. (`/documents`, `/versions`, `/nodes`)
- 리소스 하위 자원은 중첩 경로로 표현한다. (`/documents/{id}/versions`, `/documents/{id}/nodes`)
- 화면 이름이나 행위 동사를 경로에 직접 사용하지 않는다. (`/getDocument` 금지)

### 1-2. CRUD와 액션형 API 구분 원칙

- 표준 CRUD는 HTTP 메서드로 표현한다.
  - 생성: `POST /documents`
  - 조회: `GET /documents/{id}`
  - 수정: `PUT /documents/{id}` 또는 `PATCH /documents/{id}`
  - 삭제: `DELETE /documents/{id}`
- 상태 전이나 복잡한 비즈니스 행위는 서브 리소스 또는 액션 경로로 표현한다.
  - `POST /documents/{id}/publish` (게시 처리)
  - `POST /documents/{id}/versions/{versionId}/restore` (버전 복원)
- 액션형 API도 멱등성 여부를 명시한다.

### 1-3. 일관된 응답 구조

모든 API 응답은 아래 구조를 따른다.

```json
// 단건 응답
{
  "data": { ... },
  "meta": { "requestId": "...", "timestamp": "..." }
}

// 목록 응답
{
  "data": [ ... ],
  "pagination": {
    "page": 1,
    "pageSize": 20,
    "totalCount": 142,
    "hasNext": true
  },
  "meta": { "requestId": "...", "timestamp": "..." }
}
```

- `data` 필드는 항상 존재한다. 빈 결과는 `null` 또는 `[]`로 반환한다.
- `meta`에는 추적용 `requestId`가 포함된다. 감사 로그 연계에 사용된다.

### 1-4. 필터링·정렬·페이징 원칙

- 목록 조회 API는 기본적으로 페이지 기반 페이징을 지원한다.
- 필터 파라미터는 쿼리 스트링으로 전달한다. (`?type=regulation&status=active`)
- 정렬은 `sort` 파라미터로 표현한다. (`?sort=createdAt:desc`)
- 커서 기반 페이징이 필요한 대용량 데이터 API는 별도로 명시한다.

### 1-5. 식별자 사용 원칙

- 모든 리소스는 플랫폼 내부 UUID 기반 `id`를 가진다.
- 외부 시스템 연계용 슬러그(slug) 또는 외부 키가 필요한 경우, 코어 `id`와 별도 필드로 관리한다.
- 외부 참조키로 리소스를 조회하는 경우를 위한 lookup API를 별도로 제공한다. (`GET /documents?externalId=xxx`)

### 1-6. 동시성과 낙관적 잠금

- 수정 API에서 충돌 방지가 필요한 리소스는 버전 필드(`version` 또는 `etag`)를 요청에 포함한다.
- 버전 불일치 시 `409 Conflict`를 반환하고, 소비자가 최신 버전을 다시 조회하도록 유도한다.

### 1-7. 장기 작업 처리 방향

- 즉시 완료되지 않는 작업(대량 문서 처리, 검색 인덱싱 등)은 비동기 작업 ID를 반환한다.
  - 요청: `POST /import-jobs`
  - 응답: `202 Accepted` + `{ "jobId": "...", "statusUrl": "..." }`
  - 상태 조회: `GET /import-jobs/{jobId}`
- 비동기 작업 완료 알림은 Webhook을 통해 제공한다.

## 2. 인증·권한 방향

### 2-1. 인증 방식

- 모든 API 요청은 인증을 요구한다. 공개(anonymous) API는 명시적으로 선언된 경우에만 허용한다.
- 인증 방식은 토큰 기반(JWT 또는 OAuth 2.0)을 기본으로 한다.
- 서비스 간 연동(Internal Enterprise System, AI Agent)을 위한 서비스 계정 인증 방식을 별도로 정의한다.

### 2-2. 권한 모델

- 역할(Role) 기반 접근 제어(RBAC)를 기본으로 한다.
- 기본 역할:
  - `user` — 일반 사용자 (문서 탐색·조회·작성·편집)
  - `admin` — 관리자 (문서 유형·정책·권한 관리)
  - `service` — 서비스 계정 (외부 시스템·자동화)
- 리소스 단위의 세분화된 권한(예: 특정 문서 유형에 대한 접근만 허용)이 필요한 경우를 위한 확장 포인트를 설계 초기부터 고려한다.

### 2-3. 소비자별 접근 범위

- **User UI**: 인증된 사용자 권한으로 문서·버전·노드·검색에 접근. 관리 기능 접근 불가.
- **Admin UI**: 관리자 권한으로 DocumentType, 권한 정책, 감사 로그, 시스템 설정에 접근.
- **External Application / AI Agent**: 서비스 계정 토큰으로 읽기 또는 제한된 쓰기 접근. 관리 기능 접근 불가.
- 같은 리소스라도 역할에 따라 응답 필드 범위가 달라질 수 있다. (예: 감사 필드는 Admin에게만 노출)

### 2-4. 멀티 클라이언트 추적

- 모든 API 요청에는 소비자 식별 정보가 포함된다. (`requestId`, 클라이언트 ID)
- 이 정보는 감사 로그에 기록되어 어떤 클라이언트가 어떤 행위를 했는지 추적 가능하게 한다.

## 3. API 버전 정책 방향

### 3-1. 왜 버전 정책이 필요한가

외부 시스템과 AI 도구가 API를 안정적으로 사용하려면, API 계약이 예고 없이 변경되어서는 안 된다. 버전 정책은 소비자에게 변경을 예측 가능하게 제공하기 위한 기반이다.

### 3-2. 버전 표현

- API 버전은 URL 경로에 포함한다. (`/api/v1/documents`, `/api/v2/documents`)
- 메이저 버전만 경로에 표시한다. 하위 호환 변경은 버전을 올리지 않는다.

### 3-3. 변경 호환성 원칙

- **하위 호환 변경** (버전 유지): 응답 필드 추가, 선택적 요청 파라미터 추가, 새 엔드포인트 추가
- **파괴적 변경** (버전 증가 필요): 응답 필드 제거·이름 변경, 필수 요청 파라미터 추가, 엔드포인트 제거·경로 변경

### 3-4. 버전 폐기 정책

- 이전 버전은 새 버전 출시 후 일정 기간(최소 1 릴리즈 사이클) 동안 병행 운영한다.
- 폐기 예정 API는 응답 헤더로 사전 통지한다. (`Deprecation: date`)

### 3-5. 내부/외부 API 경계

- 내부 UI 전용의 비공식 엔드포인트는 허용하지 않는다. 모든 엔드포인트는 문서화된 계약을 가진다.
- 단, 내부 운영 도구용 API가 필요한 경우 `/internal/` 경로로 명시적으로 분리하고, 외부 노출을 차단한다.

## 4. 오류 응답 원칙

### 4-1. 공통 오류 응답 구조

```json
{
  "error": {
    "code": "DOCUMENT_NOT_FOUND",
    "message": "The requested document does not exist.",
    "detail": "Document with id 'abc-123' was not found.",
    "requestId": "req-xyz-456"
  }
}
```

- `code`: 기계가 처리 가능한 오류 코드 (대문자 스네이크 케이스)
- `message`: 사람이 읽을 수 있는 요약
- `detail`: 추가 상세 (선택적)
- `requestId`: 추적을 위한 요청 식별자

### 4-2. HTTP 상태 코드 사용 원칙

| 상황 | 상태 코드 |
|------|----------|
| 성공 (단건 생성) | `201 Created` |
| 성공 (조회·수정) | `200 OK` |
| 비동기 작업 수락 | `202 Accepted` |
| 인증 없음 | `401 Unauthorized` |
| 권한 없음 | `403 Forbidden` |
| 리소스 없음 | `404 Not Found` |
| 충돌 (낙관적 잠금) | `409 Conflict` |
| 유효성 오류 | `422 Unprocessable Entity` |
| 서버 오류 | `500 Internal Server Error` |

## 5. 이벤트·비동기 연계 방향

### 5-1. 비동기 처리가 필요한 작업

| 작업 | 이유 |
|------|------|
| 문서 생성·수정 후 검색 인덱싱 | 즉시 동기화 시 응답 지연 |
| 대량 문서 등록·가져오기 | 처리 시간이 요청 타임아웃을 초과할 수 있음 |
| 버전 생성 후 diff 계산 | 연산 비용이 큼 |
| 상태 전이 이벤트 외부 전달 | 외부 시스템 응답 대기 불필요 |
| 감사 로그 적재 | 메인 처리 흐름에서 분리하여 성능 보장 |
| AI 연계 파이프라인 트리거 | 처리 시간 예측 불가 |
| DocumentType 스키마 변경 후 기존 데이터 재검증 | 배치 처리 필요 |

### 5-2. 이벤트 발생 원천

아래 행위는 외부 통합 포인트로 이벤트를 발행할 수 있다.

| 이벤트 | 트리거 행위 |
|--------|-----------|
| `document.created` | 문서 생성 |
| `document.updated` | 문서 내용·메타데이터 수정 |
| `document.status_changed` | 문서 상태 전이 |
| `document.deleted` | 문서 삭제 |
| `version.created` | 새 버전 생성 |
| `node.updated` | 노드 구조 변경 |

### 5-3. 동기 API + 비동기 후처리 패턴

동기 API는 즉시 응답을 반환하고, 후처리(인덱싱, 이벤트 발행 등)는 비동기로 처리한다.

```
클라이언트 → POST /documents → 201 Created (즉시 응답)
                                      ↓ (비동기)
                              검색 인덱스 갱신
                              이벤트 발행 → 외부 Webhook
                              감사 로그 적재
```

## 6. Phase 1/2 설계 시사점

### Phase 1 (도메인 모델 설계)

- 도메인 엔티티(Document, Version, Node, Metadata, DocumentType)는 API 리소스로 자연스럽게 드러날 수 있어야 한다. 엔티티 간 관계가 API 경로 구조에 반영된다.
- 감사 추적을 위한 식별자(`createdBy`, `updatedBy`, `requestId`)는 도메인 엔티티 수준에서 정의한다.
- 문서 상태 전이 규칙은 도메인 레이어에 명시적으로 표현되어 API에서 상태 전이 API로 노출될 수 있어야 한다.

### Phase 2 (API·서비스 설계)

- API 도메인 맵(`phase0_api_domain_map.md`)을 기반으로 각 도메인 API의 엔드포인트를 상세 설계한다.
- User/Admin/External 소비자별 접근 범위 차이를 API 설계에 반영한다.
- 비동기 처리가 필요한 작업 목록을 API 설계와 함께 결정한다.
- 각 API의 감사 로그 기록 여부와 이벤트 발행 여부를 API 명세에 명시한다.
