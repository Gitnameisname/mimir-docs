# Phase 0 → Phase 1 저장 전략 입력 요약

## 문서 목적

이 문서는 Phase 0-6에서 정의한 저장 구조 전략을 Phase 1 도메인 모델 설계에 직접 연결하는 요약서다. 도메인 모델 설계자가 저장 방식에 끌려가지 않도록, 저장 전략이 도메인 설계에 부여하는 제약과 자유도를 함께 제공한다.

---

## 1. 핵심 원칙 요약: 의미가 먼저, 저장은 나중

**Document / Version / Node / Metadata / Status 모델은 저장 방식보다 도메인 의미가 먼저 정의되어야 한다.**

저장 계층은 도메인 모델을 지지하는 수단이지, 도메인 모델을 결정하는 요인이 아니다. Phase 1에서 각 엔티티를 설계할 때 "이 데이터가 DB 어느 테이블에 들어갈지"보다 "이 엔티티가 도메인에서 무엇을 의미하는지"를 먼저 결정한다.

---

## 2. Document 설계 시 주의점

### 저장 전략이 주는 제약

- Document 본문 내용을 Document 테이블 컬럼에 직접 저장하는 구조로 설계하지 않는다.
  - 본문은 Version 엔티티와 오브젝트 스토리지를 통해 관리한다.
  - Document 테이블에는 식별자, 유형, 소유자, 현재 버전 참조, 상태 값이 위치한다.
- Document는 "최신 Version을 가리키는 컨테이너"로 이해한다. 본문 자체를 가지는 엔티티가 아니다.

### 저장 전략이 주는 자유도

- Document의 도메인 속성(제목, 유형, 상태, 소유자 등)은 관계형 컬럼으로 자유롭게 정의할 수 있다.
- 확장 필요 속성은 JSON 확장 필드를 활용하므로, 모든 속성을 컬럼으로 강제할 필요가 없다.

---

## 3. Version 설계 시 주의점

### 저장 전략이 주는 제약

- Version은 불변 엔티티다. 생성 후 본문을 수정하지 않는다.
- Version 메타 정보(작성자, 생성 시각, 버전 번호, 상태)는 관계형 DB에 저장한다.
- Version 본문 원본은 크기에 따라 관계형 컬럼 또는 오브젝트 스토리지에 저장한다. 도메인 모델에서는 `contentRef` 형태로 추상화하고, 물리 위치는 인프라 계층에서 결정한다.
- diff 기반 저장(전체 스냅샷 vs 변경분만)은 Phase 1 설계에서 결정해야 할 사항이다.

### 저장 전략이 주는 자유도

- Version의 도메인 모델은 "이 시점에 문서가 어떤 상태였는가"를 표현하는 데 집중할 수 있다.
- 물리 저장 효율(diff 압축 등)은 인프라 계층의 문제다.

---

## 4. Node 설계 시 주의점

### 저장 전략이 주는 제약

- Node의 구조 메타 정보(순서, 부모 참조, 유형, 깊이)와 본문 내용을 분리하여 설계한다.
- 트리 저장 기법(인접 리스트, 경로 열거, 중첩 집합)은 Phase 1에서 결정하되, 도메인 모델이 특정 기법에 종속되지 않도록 추상화한다.
- Node는 독립적으로 검색 가능해야 하므로, 검색 대상이 되는 필드를 도메인 모델 설계 시 식별해야 한다.

### 저장 전략이 주는 자유도

- Node의 도메인 개념(섹션, 조항, 단락 등 유형별 구조 단위)은 자유롭게 정의할 수 있다.
- 물리 트리 저장 방식은 도메인 모델과 분리되어 있으므로, 성능 요구에 따라 나중에 변경할 수 있다.

---

## 5. Metadata 설계 시 주의점

### 저장 전략이 주는 제약

- 공통 메타데이터(작성자, 작성일, 태그 등)와 유형별 확장 메타데이터를 도메인 모델에서 명확히 구분한다.
  - 공통 메타: 정형 필드로 설계 (관계형 컬럼)
  - 유형별 확장 메타: 확장 스키마 기반으로 설계 (JSON 필드)
- 메타데이터 스키마(DocumentType이 정의하는 허용 메타데이터 목록)와 메타데이터 값을 별개 엔티티로 분리한다.
- 확장 메타데이터 값은 스키마에 의해 검증된다. 무한 자유 필드가 아니다.

### 저장 전략이 주는 자유도

- 메타데이터 스키마는 코드 변경 없이 Admin이 추가할 수 있도록 설계한다.
- 새 문서 유형의 메타데이터를 추가할 때 도메인 모델이나 DB 스키마를 변경하지 않아도 된다.

---

## 6. Status(상태 모델) 설계 시 주의점

### 저장 전략이 주는 제약

- 현재 상태값과 상태 전이 이력을 별도로 관리한다.
  - 현재 상태: Document 엔티티의 상태 필드 (관계형 컬럼)
  - 상태 전이 이력: 감사/로그 계층 (별도 테이블 또는 로그 시스템)
- 상태 전이 로직(어떤 상태에서 어떤 상태로 전이 가능한가)은 도메인 서비스 계층에 정의한다. 저장 계층에 종속시키지 않는다.

### 저장 전략이 주는 자유도

- 상태 전이 규칙은 코드 또는 정책 테이블로 표현할 수 있으며, 도메인 설계에서 결정한다.

---

## 7. 공통 주의점: 감사 필드와 도메인 분리

- 도메인 엔티티에 감사 기록 컬럼(`createdBy`, `updatedBy`, `deletedBy`)은 포함해도 무방하다.
- 그러나 상세 감사 이벤트(변경 diff, 행위 사유, 클라이언트 IP 등)는 감사 계층에 분리 저장한다.
- 도메인 엔티티가 감사 로그를 직접 들고 있는 구조(내장 감사 히스토리 컬럼)는 지양한다.

---

## 8. 저장 방식에 끌려가지 않기 위한 원칙 요약

| 위험 패턴 | 대응 원칙 |
|----------|----------|
| "이 데이터를 어느 테이블에 넣을지"로 모델 결정 | 도메인 의미를 먼저 정의하고, 테이블 매핑은 이후 |
| 본문을 Document 테이블에 직접 저장 | 본문은 Version + OBJ 계층, Document는 참조만 |
| 메타데이터를 모두 정형 컬럼으로 강제 | 공통은 컬럼, 확장은 JSON 스키마 기반 |
| Version을 Document의 단순 컬럼으로 처리 | Version은 독립 불변 엔티티 |
| 상태 전이 이력을 Document 테이블에 저장 | 상태 전이 이력은 LOG 계층으로 분리 |
| 첨부 파일을 도메인 엔티티와 강하게 결합 | AttachmentId 참조로 분리, 물리 파일은 OBJ |
| 검색 인덱스를 도메인 원본처럼 취급 | 인덱스는 항상 재구성 가능한 파생물 |
