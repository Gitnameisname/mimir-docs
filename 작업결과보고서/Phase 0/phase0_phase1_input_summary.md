# Phase 0 → Phase 1 설계 입력 요약

## 목적

이 문서는 Phase 0-1의 비전 정의 결과를 Phase 1 도메인 모델 설계에 직접 연결하기 위한 요약서다. Phase 1 설계자는 이 문서를 참조하여 데이터 모델의 방향을 결정해야 한다.

---

## 1. 핵심 설계 방향

### Document는 파일이 아니라 지식 엔티티다
Document를 파일 레코드로 취급하지 않는다. Document는 플랫폼이 관리하는 독립적 지식 자산이며, 특정 파일 형식이나 저장 방식에 종속되지 않아야 한다. Document 엔티티는 메타데이터·버전·구조 노드와 관계를 가지며, 이 관계들이 지식의 실체를 이룬다.

### Version은 Document의 기본 구조다
버전 관리는 선택적 기능이 아니라 플랫폼의 기본 동작이다. Document의 모든 변경은 Version으로 기록되어야 하며, 특정 시점의 상태 복원과 버전 간 비교가 가능해야 한다. Version 엔티티는 Document와 독립적으로 설계한다.

### Node는 문서 내부 구조를 표현하는 독립 엔티티다
문서 내부의 구조(섹션, 조항, 단락 등)는 플랫폼이 인식하고 관리하는 Node 엔티티로 표현한다. Node는 DocumentType에 따라 다른 계층 구조를 가질 수 있으며, Node 단위로 메타데이터 부착과 검색이 가능해야 한다.

### Metadata는 확장 가능한 정형 구조로 설계한다
메타데이터는 고정 컬럼으로 처리하지 않는다. DocumentType별로 정의 가능한 스키마 기반 메타데이터 구조가 필요하다. 동시에 비정형 확장 필드도 수용할 수 있어야 한다. 메타데이터는 검색·필터링·외부 연계의 핵심 기반이므로, 인덱싱 전략과 함께 설계해야 한다.

### DocumentType은 플랫폼의 확장 메커니즘이다
특정 문서 유형에 종속된 구조를 하드코딩하지 않는다. DocumentType은 문서의 구조 스키마, 허용 메타데이터, 상태 모델, 권한 정책을 정의하는 확장 포인트다. 새로운 문서 유형이 추가될 때 도메인 모델 전체를 수정하지 않아도 되어야 한다.

## 2. 도메인 모델 설계 체크리스트

| 항목 | 설계 기준 |
|------|----------|
| Document | 파일 종속 없이 독립적 엔티티로 설계되었는가? |
| Version | Document와 분리된 독립 엔티티인가? 이전 버전 접근이 가능한가? |
| Node | DocumentType별 계층 구조를 수용할 수 있는가? |
| Metadata | 고정 컬럼이 아니라 확장 가능한 스키마로 설계되었는가? |
| DocumentType | 새 유형 추가 시 도메인 모델 변경 없이 수용 가능한가? |
| Relationship | Document·Node 간 관계를 모델링할 수 있는가? |
| 상태 모델 | 문서 라이프사이클(초안·검토·승인·폐기 등)을 수용하는가? |
| 권한 | 문서·버전·노드 단위의 접근 제어가 가능한가? |
| 검색 | 메타데이터·구조·전문 검색을 위한 인덱싱 전략이 고려되었는가? |
| API 접근 | 모든 핵심 엔티티가 API를 통해 CRUD 가능한가? |

---

## 3. 설계 시 피해야 할 방향

- Document를 특정 파일 형식(PDF, DOCX 등)에 종속시키는 것
- 버전을 파일 복사본으로 처리하는 것
- 메타데이터를 고정 컬럼으로 처리하여 유형별 확장을 막는 것
- 특정 문서 유형 하나를 위한 구조를 전체 모델로 일반화하는 것
- Node를 Document의 단순 텍스트 블록으로 취급하는 것
