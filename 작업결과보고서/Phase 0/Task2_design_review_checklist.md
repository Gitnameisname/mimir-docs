# Phase 0 — 설계 리뷰 체크리스트

## 사용 목적

이 체크리스트는 설계 리뷰, PR 리뷰, 아키텍처 검토 시 플랫폼 원칙([Task2_platform_principles.md](`Task2_platform_principles.md`)) 준수 여부를 확인하기 위한 도구다. 모든 항목을 통과해야 하는 것은 아니나, 체크하지 못한 항목은 예외 사유를 명시해야 한다.

---

## API

> 관련 원칙: P1 (API-first), P9 (외부 연계 가능성)

- [ ] 이 기능의 핵심 로직이 API 엔드포인트로 노출되어 있는가?
- [ ] UI가 이 기능을 사용할 때 API를 호출하는가, 아니면 내부 로직을 직접 실행하는가?
- [ ] API 없이 UI에서만 동작하는 기능이 존재하는가? (존재하면 위반)
- [ ] API 응답 구조가 특정 UI 컴포넌트의 렌더링 형식에 종속되어 있지 않은가?
- [ ] 외부 시스템 또는 AI 도구가 이 API를 동등하게 사용할 수 있는가?
- [ ] API 에러 응답 포맷이 일관된 구조를 따르는가?
- [ ] API 인증 방식이 정의되어 있는가?

## Domain Model

> 관련 원칙: P2 (범용 도메인 모델), P4 (도메인/저장 분리), P5 (버전 추적), P6 (구조화 우선), P7 (메타데이터 확장)

- [ ] 이 도메인 엔티티가 특정 문서 유형에만 의미 있는 필드를 직접 포함하고 있지 않은가?
- [ ] 새로운 문서 유형이 추가될 때 이 모델을 수정해야 하는가? (수정이 필요하면 재검토)
- [ ] 도메인 엔티티가 DB 테이블 구조와 동일하게 설계되어 있지 않은가?
- [ ] 이 엔티티의 변경 이력을 추적할 수 있는 Version 구조가 있는가?
- [ ] 문서 내부 구조(Node)가 플랫폼이 인식하는 엔티티로 관리되고 있는가?
- [ ] 메타데이터가 고정 컬럼이 아닌 확장 가능한 방식으로 정의되어 있는가?
- [ ] 정형 필드와 비정형 확장 필드의 경계가 명확히 정의되어 있는가?
- [ ] 도메인 엔티티에 저장소 특화 필드(파티션 키, DB 전용 버전 컬럼 등)가 섞여 있지 않은가?

## UI / User

> 관련 원칙: P1 (API-first), P3 (User/Admin 분리), P8 (탐색 가능성)

- [ ] 이 화면은 순수하게 사용자의 지식 탐색·조회·편집을 위한 것인가?
- [ ] 이 화면에 운영·관리 기능이 조건부(isAdmin 등)로 포함되어 있지 않은가?
- [ ] 이 화면의 모든 데이터는 API 호출로 가져오는가?
- [ ] 사용자가 메타데이터 기반으로 필터링하거나 검색할 수 있는가?
- [ ] 사용자가 문서의 특정 버전에 접근하거나 이력을 조회할 수 있는가?
- [ ] User UI의 라우팅 경로가 Admin UI와 명확히 구분되어 있는가? (`/app/*` vs `/admin/*`)

---

## UI / Admin

> 관련 원칙: P3 (User/Admin 분리), P10 (운영·감사·권한 명시적 관리)

- [ ] 이 화면이 Admin 전용 영역(`/admin/*`)에 위치해 있는가?
- [ ] 문서 유형 관리, 권한 정책 수정, 감사 로그 조회가 Admin UI에서만 접근 가능한가?
- [ ] Admin 기능이 User UI에 노출되는 경로가 없는가?
- [ ] 운영 정책 변경(예: 승인 흐름 활성화)이 코드 배포 없이 Admin UI에서 처리 가능한가?
- [ ] Admin API가 User API와 권한 범위를 공유하고 있지 않은가?

---

## Storage

> 관련 원칙: P4 (도메인/저장 분리), P5 (버전 추적), P6 (구조화 우선)

- [ ] 저장 구조(테이블, 컬럼, JSON 스키마)가 도메인 모델을 반영하고 있는가, 아니면 도메인 모델이 저장 편의를 따르고 있는가?
- [ ] 문서 본문과 메타데이터, 버전 이력이 분리된 방식으로 저장되는가?
- [ ] 문서 변경 시 기존 레코드를 덮어쓰는 구조인가? (덮어쓰면 P5 위반)
- [ ] 비정형 데이터를 수용하는 JSON 확장 필드가 검색 가능성을 고려하여 설계되었는가?
- [ ] 저장 구조 변경이 도메인 레이어에 영향을 주는가? (영향을 주면 P4 위반)
- [ ] 추후 AI/RAG/검색 확장을 위한 데이터 접근 여지가 저장 설계에 남아 있는가?

---

## Integration

> 관련 원칙: P1 (API-first), P9 (외부 연계 가능성)

- [ ] 외부 시스템이 이 기능을 파일 export 없이 API로 접근할 수 있는가?
- [ ] 외부 연계를 위한 API 응답이 구조화된 JSON 형식으로 제공되는가?
- [ ] 문서 변경 이벤트를 외부 시스템에 전달할 수 있는 메커니즘(Webhook/Event)이 있거나 계획되어 있는가?
- [ ] 외부 연계 API가 내부 UI API와 동일한 구조적 품질(인증, 에러 포맷, 버전 정책)을 유지하는가?
- [ ] 특정 외부 시스템에 종속된 포맷이나 프로토콜이 코어 플랫폼에 직접 포함되어 있지 않은가?

## Governance

> 관련 원칙: P3 (User/Admin 분리), P10 (운영·감사·권한 명시적 관리)

- [ ] 이 기능에서 발생하는 모든 상태 변경(생성, 수정, 삭제, 승인)이 감사 로그에 기록되는가?
- [ ] 권한 검사가 각 핸들러마다 다르게 구현되지 않고 일관된 정책 레이어를 통해 이루어지는가?
- [ ] 문서 접근 권한이 코드 내 역할 분기(`if isAdmin`)가 아닌 선언적 정책으로 정의되어 있는가?
- [ ] 감사 로그에 who(행위자), when(시간), what(변경 내용), where(대상 엔티티)가 포함되는가?
- [ ] 운영 정책(승인 요건, 공개 조건 등)이 코드에 하드코딩되지 않고 설정 가능한 구조인가?
- [ ] 권한 정책 변경이 코드 배포 없이 가능한 구조인가?
