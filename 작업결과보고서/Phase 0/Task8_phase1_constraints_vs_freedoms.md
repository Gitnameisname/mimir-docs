# Phase 0 → Phase 1 설계 제약 vs 자유도

## 사용 목적

이 표는 Phase 1 도메인 모델 설계에서 어떤 부분이 Phase 0에서 이미 확정된 고정 전제인지, 어떤 부분이 Phase 1 설계자가 탐색하고 결정할 수 있는 자유도 영역인지를 구분한다. 설계 논의에서 이미 결정된 사항을 다시 논쟁하는 데 시간을 쓰지 않도록 한다.

### 분류 기준

| 분류 | 의미 |
|------|------|
| **고정** | Phase 0에서 확정. Phase 1에서 변경·우회 불가. 원칙 위반 시 Phase 0 재검토 필요. |
| **탐색** | Phase 0 방향 안에서 Phase 1 설계자가 선택 가능. 최적 구현을 탐색하는 영역. |

## 플랫폼 수준 전제

| 설계 항목 | 분류 | 설명 | 관련 원칙 |
|---------|:----:|------|---------|
| 플랫폼은 범용 지식 플랫폼이다 | **고정** | 특정 문서 유형 중심 모델 금지 | Phase 0-1 비전 |
| API-first: 모든 핵심 기능은 API로 노출 | **고정** | 도메인 모델은 API 리소스로 드러날 수 있어야 함 | P1 (API-first) |
| User/Admin 관점 분리 | **고정** | 동일 엔티티라도 관점별 접근 범위·표현이 다름 | P3 (User/Admin 분리) |
| 도메인 모델 ≠ 저장 구조 | **고정** | 저장 방식이 먼저 결정되어 모델에 영향을 주면 안 됨 | P4 (도메인/저장 분리) |
| 다층 저장 전략 | **고정** | 단일 RDB로 모든 데이터를 해결하지 않음 | Phase 0-5, 0-6 |
| 외부 시스템이 코어 모델을 지배하면 안 됨 | **고정** | 외부 구조는 커넥터 계층에서 흡수 | Phase 0-7 시스템 경계 |

## Document

| 설계 항목 | 분류 | 설명 | 관련 원칙 |
|---------|:----:|------|---------|
| Document는 독립적 지식 자산 단위 | **고정** | 파일이 아닌 식별·추적 가능한 관리 단위 | Phase 0-1 핵심 자산 |
| Document는 특정 유형에 종속되지 않음 | **고정** | 범용 코어 모델. 유형 특성은 DocumentType으로 분리 | P2 (범용 도메인 모델) |
| Version·Node·Metadata·Status와 연결 가능 | **고정** | 이 관계 구조가 없으면 지식 플랫폼이 아님 | Phase 0-1 핵심 자산 |
| 첨부 파일은 참조(AttachmentId)로만 연결 | **고정** | 파일 원본은 Document 엔티티 외부에 저장 | Phase 0-6 저장 전략 |
| Document 속성 집합 (구체적 필드 이름) | 탐색 | 어떤 필드를 가질지는 Phase 1에서 결정 | — |
| Document와 Relationship 표현 방식 | 탐색 | 문서 간 관계를 어떻게 표현할지 Phase 1에서 결정 | — |

## Version

| 설계 항목 | 분류 | 설명 | 관련 원칙 |
|---------|:----:|------|---------|
| Version은 Document와 독립된 엔티티 | **고정** | Document의 부가 컬럼이 아님 | P5 (버전 추적) |
| 모든 수정은 새 Version 생성 | **고정** | 기존 Version 본문 덮어쓰기 금지 | P5 (버전 추적) |
| Document.currentVersion은 포인터 | **고정** | Document가 본문을 직접 가지지 않음 | Phase 0-6 저장 전략 |
| Version은 불변 | **고정** | 생성 후 본문 수정 불가 (관리자 폐기 제외) | Phase 0-6 저장 전략 |
| Version 이력 ≠ Audit 로그 | **고정** | 다른 개념. 혼용 금지 | Phase 0-6 |
| Version의 범위 (Document 전체 vs Node 수준) | 탐색 | 버전 단위를 어디까지 할지 Phase 1에서 결정 | — |
| diff 저장 방식 (전체 스냅샷 vs 변경분) | 탐색 | 저장 효율 vs 구현 복잡도 트레이드오프 | — |
| 버전 번호 체계 | 탐색 | 단순 증가, 의미적 버전(v1.2.3) 등 | — |
| Draft Version과 Published Version 구분 방식 | 탐색 | 상태 모델과의 연동 방식 | — |

## Node / Structure

| 설계 항목 | 분류 | 설명 | 관련 원칙 |
|---------|:----:|------|---------|
| Document는 내부 계층 구조(Node)를 가질 수 있음 | **고정** | Node가 없으면 구조화 검색·탐색 불가 | Phase 0-1 핵심 자산 |
| Node는 독립 엔티티 | **고정** | Document의 단순 텍스트 블록이 아님 | P6 (구조화 우선) |
| Node는 부분 조회·검색 가능 단위 | **고정** | Node 단위 검색·렌더링·AI 청크 처리 가능성 확보 | Phase 0-6 |
| Node 표현은 특정 포맷(Markdown 등)에 종속 불가 | **고정** | 포맷 독립적 구조 | P2 (범용 도메인 모델) |
| 구조(트리)와 본문 내용은 개념적으로 구분 | **고정** | 구조 변경과 내용 변경을 별도로 처리 가능해야 함 | Phase 0-6 |
| 트리 저장 표현 기법 | 탐색 | 인접 리스트, 경로 열거, 중첩 집합 등 Phase 1에서 결정 | — |
| Node 세부 유형 체계 | 탐색 | 어떤 유형의 노드가 있는지 Phase 1에서 결정 | — |
| 구조 변경과 버전 연동 방식 | 탐색 | 구조 변경 시 Version 생성 여부 Phase 1에서 결정 | — |
| Node 단위 메타데이터 허용 범위 | 탐색 | Node에도 Metadata를 붙일지, 어느 수준까지 허용할지 | — |

## DocumentType

| 설계 항목 | 분류 | 설명 | 관련 원칙 |
|---------|:----:|------|---------|
| 코어 모델은 DocumentType에 종속되지 않음 | **고정** | DocumentType은 코어 위 확장 계층 | P2, P7 |
| 새 DocumentType 추가 시 코드 변경 불가 | **고정** | Admin이 런타임에 추가 가능해야 함 | P7 (확장 가능성) |
| DocumentType은 메타데이터 스키마를 정의 | **고정** | 어떤 확장 메타데이터가 허용되는지 유형별로 정의 | P7 |
| DocumentType은 Admin이 관리 | **고정** | User UI에서 DocumentType 수정 불가 | P3 (User/Admin 분리) |
| 유형별 validation 정책 세부 구현 | 탐색 | 스키마 검증 수준·방식 Phase 1에서 결정 | — |
| 유형별 상태 모델 차이 허용 범위 | 탐색 | 일부 유형에서 특정 상태 단계 제외 가능한지 | — |
| DocumentType 계층화 (부모-자식 유형) 가능성 | 탐색 | 유형 상속 구조 필요성 Phase 1에서 검토 | — |

## Metadata

| 설계 항목 | 분류 | 설명 | 관련 원칙 |
|---------|:----:|------|---------|
| 공통 Metadata와 확장 Metadata 구분 | **고정** | 공통은 정형 컬럼, 확장은 스키마 기반 JSON | Phase 0-6 저장 전략 |
| 확장 Metadata는 스키마 검증 필수 | **고정** | 완전 자유 필드 금지 | Phase 0-6 |
| Metadata는 검색·필터에 활용 가능해야 함 | **고정** | 검색 불가 Metadata는 설계 완성도 낮음 | P8 (검색 가능성) |
| Metadata 스키마 정의 ≠ Metadata 값 | **고정** | 두 개념을 분리 설계 | Phase 0-6 |
| Metadata의 User/Admin 노출 범위 차이 가능 | **고정** | Admin만 보이는 운영 메타데이터 존재 가능 | P3 (User/Admin 분리) |
| Metadata 스키마 계층화 수준 | 탐색 | 단순 key-value vs 타입 있는 스키마 vs JSON Schema | — |
| 정렬·집계 가능한 Metadata 범위 | 탐색 | 어떤 메타데이터를 인덱싱할지 Phase 1에서 결정 | — |
| Node 단위 Metadata 허용 여부 | 탐색 | Node에도 Metadata를 붙일지 Phase 1에서 결정 | — |

## Status / Lifecycle

| 설계 항목 | 분류 | 설명 | 관련 원칙 |
|---------|:----:|------|---------|
| Document는 상태를 가진다 | **고정** | 단순 존재 여부가 아닌 라이프사이클 | Phase 0-1 |
| 현재 상태값과 상태 전이 이력은 분리 | **고정** | 현재 상태: Document 필드 / 이력: 감사 계층 | Phase 0-6 |
| Version ≠ Status | **고정** | 상태 변경이 반드시 버전 생성은 아님 | Phase 0-6 |
| 상태 전이 규칙은 도메인 서비스 계층에 위치 | **고정** | API·저장 계층에 분산 금지 | P10 (명시적 관리) |
| User/Admin 상태 정보 노출 범위 차이 | **고정** | User: 단순 레이블 / Admin: 전이 이력·정책 정보 | P3 (User/Admin 분리) |
| 상태 단계 수와 세부 명칭 | 탐색 | 최소 DRAFT·PUBLISHED는 필요. 세부 단계는 결정 가능 | — |
| DocumentType별 상태 모델 차이 허용 범위 | 탐색 | 유형마다 다른 상태 단계 가능한지 Phase 1에서 결정 | — |
| 상태 전이 규칙 표현 방식 | 탐색 | 코드 기반, 정책 테이블 기반 등 | — |
