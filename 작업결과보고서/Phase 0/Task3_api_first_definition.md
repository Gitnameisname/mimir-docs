# Phase 0 — API-first 원칙 정의서

## 문서 목적

이 문서는 Mimir 플랫폼이 API-first 방식으로 설계되어야 한다는 원칙을 구체화한다. "API를 잘 만들자"는 선언이 아니라, 이 플랫폼이 UI 중심 시스템이 아니라 API 중심 플랫폼이라는 구조적 사실을 확정한다. 이후 도메인 모델 설계, API 설계, UI 설계, 외부 연계 설계에서 이 문서를 판단 기준으로 참조한다.

---

## 1. 이 프로젝트에서 API-first란 무엇인가

### 정의

**API-first**란, 플랫폼의 모든 핵심 기능이 API 계약(contract)으로 먼저 정의되고, UI와 외부 시스템은 모두 그 API를 소비하는 클라이언트로 설계되는 방식이다.

Mimir에서 API-first는 다음 세 가지를 동시에 의미한다.

1. **기능 정의의 기준점은 API 계약이다.** 어떤 기능을 만들기로 결정했다면, 그 기능의 입력·출력·부작용은 API 계약 형태로 먼저 정의된다. 화면 목업이나 DB 스키마가 기능 정의를 대신하지 않는다.

2. **UI는 API의 한 소비자다.** User UI와 Admin UI는 플랫폼의 주체가 아니라 API를 호출하는 클라이언트다. UI가 API 설계를 이끌어서는 안 된다.

3. **외부 연계와 자동화 가능성을 API 설계에 처음부터 반영한다.** API는 내부 프론트엔드만을 위한 임시 엔드포인트 집합이 아니라, 외부 애플리케이션·AI 도구·자동화 시스템이 동등하게 사용할 수 있는 플랫폼 인터페이스다.

### 단순한 REST 제공과의 차이

| 구분 | REST API 제공 | API-first |
|------|--------------|-----------|
| 기능 정의 기준 | UI 화면 또는 DB 스키마 | API 계약 |
| API 위치 | 화면 구현 후 추가 | 기능 설계의 출발점 |
| 소비자 관점 | 내부 프론트엔드 중심 | UI + 외부 시스템 + AI 동등 취급 |
| 계약 안정성 | UI 변경에 따라 변동 | 변경 정책과 버전 관리로 보호 |
| 도메인 분해 | 화면 단위 | 도메인/리소스 단위 |

## 2. UI와 API의 관계

### API가 먼저, UI가 나중

```
플랫폼 도메인 (Document / Version / Node / Metadata / ...)
        ↓
    API 계약 정의 (리소스 / 동작 / 응답 / 권한)
        ↓
┌───────────────────────────────────────────────────────┐
│  User UI  │  Admin UI  │  External App   │  AI Agent  │
└───────────────────────────────────────────────────────┘
```

User UI와 Admin UI는 플랫폼 위에 구축된 첫 번째 API 클라이언트들이다. 이 두 UI가 필요한 기능은 반드시 API를 통해 제공된다. UI 내부에 비즈니스 규칙이 있어서는 안 된다.

### 핵심 경계 규칙

- **API 레이어**: 도메인 규칙, 상태 전이, 권한 검사, 버전 생성, 감사 기록이 이 계층에서 이루어진다.
- **UI 레이어**: API 응답을 조합·표현하는 역할에 집중한다. 도메인 행위를 직접 결정하지 않는다.

## 3. API-first가 필요한 이유

### 3-1. 다수의 소비자를 동등하게 지원해야 한다

Mimir는 단일 UI만 지원하는 시스템이 아니다. User UI, Admin UI, 외부 업무 시스템, AI 에이전트, 자동화 워크플로가 모두 같은 플랫폼의 기능에 접근해야 한다. UI 중심으로 설계하면 UI에 최적화된 구조가 만들어지고, 다른 소비자가 진입할 때 재설계가 필요해진다.

### 3-2. 도메인 로직이 UI에 묶이는 것을 방지한다

UI-first 방식에서는 빠른 개발을 위해 비즈니스 규칙이 프론트엔드 코드에 들어가는 경우가 많다. 이 경우 같은 규칙이 User UI와 Admin UI에 각각 다르게 구현되거나, 외부 연계 시점에 동일한 로직을 다시 구현해야 하는 문제가 생긴다.

### 3-3. 외부 연계와 자동화를 미리 수용한다

규정 문서, 지식 자산이 외부 AI 도구나 업무 시스템과 연계되려면, 지식이 구조화된 API 응답으로 제공되어야 한다. 이를 나중에 추가하려 하면 기존 API 구조를 대규모로 변경해야 한다.

### 3-4. API 계약이 안정성의 기반이 된다

API 계약이 명확하면, 프론트엔드와 백엔드가 독립적으로 개발될 수 있고, 외부 시스템이 안정적으로 연계할 수 있다. 계약이 없으면 변경 시마다 모든 소비자에게 영향이 전파된다.

---

## 4. 기대 효과

| 효과 | 내용 |
|------|------|
| **소비자 독립성** | User UI, Admin UI, 외부 시스템, AI 도구가 API를 통해 독립적으로 개발·연계 가능 |
| **도메인 규칙 단일화** | 비즈니스 규칙이 API 레이어에 집중되어 UI 간 불일치 방지 |
| **외부 연계 비용 감소** | 처음부터 외부 소비자를 고려한 설계이므로 연계 시 재설계 불필요 |
| **테스트 가능성** | UI 없이도 API 단위로 기능 검증 가능 |
| **변경 통제** | API 계약 변경 정책을 통해 소비자에 대한 파급 범위를 통제 가능 |

---

## 5. 하지 말아야 할 방식

| 금지 패턴 | 이유 |
|----------|------|
| "화면 먼저, API 나중에" | UI에 최적화된 API가 만들어져 외부 연계 불가 |
| UI 내부에 도메인 규칙 구현 | 동일 규칙이 여러 곳에 중복되거나 불일치 발생 |
| 특정 화면 플로우에 종속된 API 경로 설계 | 화면이 바뀌면 API도 바뀌어야 함 |
| 임시 편의를 위한 비공식 엔드포인트 | 문서화되지 않은 계약이 누적됨 |
| 내부 UI와 외부 연계 API를 다른 원칙으로 설계 | 외부 소비자 진입 시 별도 API 계층 필요 |
| 인증/권한을 후속 과제로만 처리 | 모든 API가 열린 채로 굳어질 위험 |
| 모든 기능을 동기 CRUD로만 가정 | 비동기 작업, 이벤트 연계 수용 불가 |
