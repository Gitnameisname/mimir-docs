# Phase 0 — 기술 스택 방향 문서

## 문서 목적

이 문서는 Mimir 플랫폼의 기술 스택 방향성과 계층별 요구사항을 정의한다. 특정 라이브러리나 제품을 확정하는 것이 아니라, 각 기술 계층이 어떤 역할을 해야 하는지, 왜 그 방향이어야 하는지를 명확히 한다. 이 문서는 이후 Phase에서 세부 기술 선택을 논의할 때 판단 기준이 된다.

---

## 1. 플랫폼의 기술적 성격

Mimir는 다음과 같은 기술적 성격을 가진다.

| 관점 | 성격 |
|------|------|
| 시스템 유형 | 단순 웹앱이 아닌 **API 중심 플랫폼 서비스** |
| 데이터 성격 | 단순 CRUD를 넘어 **버전·상태·구조·메타데이터·검색이 핵심** |
| UI 역할 | 주체가 아닌 **API 소비자 계층** (User UI + Admin UI 분리) |
| 소비자 다양성 | UI 외 **외부 시스템·AI 도구·자동화 워크플로**도 1급 소비자 |
| 처리 성격 | 동기 요청 처리 + **비동기 후처리·이벤트·인덱싱** 병존 |
| 저장 성격 | 단일 저장소가 아닌 **역할별 다층 저장소** 필요 |
| 확장 방향 | 특정 문서 유형에 종속되지 않는 **범용 지식 플랫폼 구조** |

이 성격 정의는 이후 모든 기술 선택의 전제다. "단순 게시판을 빠르게 만드는" 관점의 기술 스택은 이 플랫폼에 적합하지 않다.

## 2. 프론트엔드 방향

### 2-1. 아키텍처 방향

- **API 소비자 계층**으로 설계한다. 비즈니스 규칙은 API에서 처리하고, 프론트엔드는 API 응답을 조합·표현하는 데 집중한다.
- **User UI와 Admin UI를 라우팅 구조에서 분리**한다. 동일 코드베이스 내 별도 라우팅 트리(`/app/*`, `/admin/*`)로 분리하거나, 필요 시 별도 애플리케이션으로 분리한다.
- **컴포넌트 계층 구조**를 명확히 한다. 공용 UI 컴포넌트(버튼, 폼, 테이블)는 공유하되, 화면 수준의 비즈니스 로직은 User/Admin 영역 내부에서만 관리한다.
- 문서 구조(Node 트리), 버전 비교, 메타데이터 편집 같은 **복잡한 도메인 화면**을 수용할 수 있는 컴포넌트 조합 구조가 필요하다.

### 2-2. 렌더링 전략 (SSR/CSR/Hybrid)

- **User UI**: 검색 최적화(SEO)와 초기 로딩 성능이 중요한 경우 SSR 또는 Hybrid를 고려한다. 그러나 플랫폼의 핵심이 공개 콘텐츠 노출보다 인증 기반 내부 활용이라면 CSR로도 충분하다.
- **Admin UI**: 공개 노출이 필요 없으므로 CSR 기반으로 구성해도 무방하다. 운영 복잡도를 줄이는 것이 우선이다.
- 렌더링 전략의 최종 결정은 Phase 2에서 확정한다.

### 2-3. 상태 관리 원칙

- 서버 상태(API 데이터)와 클라이언트 상태(UI 상호작용 상태)를 명확히 분리한다.
- API 응답 캐싱과 재사용은 서버 상태 관리 계층에서 처리하고, UI 내부 상태는 로컬 컴포넌트 상태로 관리한다.
- 전역 상태는 최소화한다. 도메인 규칙을 프론트엔드 전역 상태로 관리하는 방식은 피한다.

### 2-4. 기술 선택 방향

- **타입 안정성**: TypeScript 기반의 강타입 환경을 전제한다. API 계약과 프론트엔드 코드 간 타입 일관성 확보.
- **컴포넌트 기반 SPA 프레임워크**: 복잡한 문서 편집기, 노드 트리, 메타데이터 폼 같은 인터랙티브 화면을 수용할 수 있는 생태계가 필요하다.
- **API 클라이언트 계층**: API 요청·응답 처리, 에러 핸들링, 인증 토큰 관리가 일관된 방식으로 처리되어야 한다.
- 세부 프레임워크 선택(React/Next.js 등)은 Phase 2에서 확정한다.

## 3. 백엔드 / API 계층 방향

### 3-1. 계층 구조 원칙

백엔드는 아래 계층으로 분리된다.

```
API 계층 (라우팅, 입력 검증, 인증, 직렬화)
       ↓
서비스 / 유스케이스 계층 (도메인 규칙, 상태 전이, 비즈니스 정책)
       ↓
도메인 계층 (엔티티, 도메인 이벤트, 불변 규칙)
       ↓
인프라 계층 (저장소, 외부 API, 이벤트 발행, 검색 인덱싱)
```

- **API 계층**은 도메인 로직을 포함하지 않는다. 요청 검증·인증 확인·응답 직렬화에 집중한다.
- **서비스 계층**이 도메인 규칙의 진입점이다. 상태 전이, 버전 생성, 권한 검사, 감사 로그 기록이 이 계층에서 이루어진다.
- **도메인 계층**은 저장소·프레임워크에 종속되지 않는 순수 비즈니스 모델을 포함한다.
- **인프라 계층**은 저장소 구현, 검색 인덱싱, 이벤트 발행, 외부 API 연동을 담당하며 도메인 레이어와 인터페이스로만 연결한다.

### 3-2. API-first 구현 방향

- 모든 기능은 API 엔드포인트로 먼저 노출되어야 한다.
- API 계약은 버전 관리되고, 변경 정책에 따라 소비자에게 사전 통지된다.
- REST 기반 HTTP API를 기본으로 한다. GraphQL이나 gRPC 도입은 구체적 필요가 확인된 후 검토한다.

### 3-3. 인증·권한 처리

- 인증은 백엔드 미들웨어에서 처리한다. UI 레이어에서 인증 판단을 내리지 않는다.
- 역할 기반 접근 제어(RBAC)를 API 레이어 미들웨어 수준에서 일관되게 적용한다.
- Admin 전용 API 경로(`/admin/*`)는 별도 미들웨어 체인에서 강화된 권한 검사와 감사 로그 기록을 처리한다.

### 3-4. 동기/비동기 처리 분리

- HTTP 요청 처리(동기)와 후처리 작업(검색 인덱싱, 이벤트 발행, 감사 로그 적재)을 분리한다.
- API 응답은 즉시 반환하고, 후처리는 비동기 큐 또는 이벤트 스트림으로 처리한다.

### 3-5. 기술 선택 방향

- **타입 안정성이 높은 언어/프레임워크**: 복잡한 도메인 모델(버전, 상태, 메타데이터 스키마)을 안전하게 표현할 수 있어야 한다.
- **도메인 레이어 분리를 지원하는 구조**: 프레임워크가 도메인 모델을 침식하지 않도록 계층 분리를 자연스럽게 지원해야 한다.
- **미들웨어 확장성**: 인증, 권한, 감사 로그, API 버전 관리를 미들웨어 체인으로 처리할 수 있어야 한다.
- 세부 프레임워크 선택(FastAPI/Django/NestJS 등)은 Phase 2에서 확정한다.

## 4. 데이터 저장 계층 방향

### 4-1. 다층 저장 전략이 필요한 이유

단일 관계형 DB만으로는 다음 요구를 모두 충족하기 어렵다.

| 요구 | 단일 RDB 한계 |
|------|-------------|
| 유연한 메타데이터 스키마 | 유형별 컬럼 추가 시 마이그레이션 필요 |
| 전문 검색 + 메타데이터 필터 복합 쿼리 | 성능과 쿼리 복잡도가 급격히 증가 |
| 대용량 첨부 파일 저장 | DB에 직접 저장은 부적합 |
| 버전 이력 대용량 보관 | 메인 테이블 비대화 |
| 감사/운영 로그 | 주 DB 성능에 영향 |

### 4-2. 저장 계층별 역할

| 저장 계층 | 저장 대상 | 이유 |
|----------|----------|------|
| **관계형 DB** | Document 엔티티, Version, Node, DocumentType, 권한, 상태, 감사 핵심 필드 | 트랜잭션, 정합성, 외래키 관계, 복잡한 조건 쿼리 |
| **JSON 확장 필드 (RDB 내)** | 메타데이터 값, 비정형 확장 속성 | 스키마 변경 없이 유연한 속성 추가. RDB의 JSONB 활용 |
| **오브젝트 스토리지** | 첨부 파일, 문서 원본 파일, 대용량 버전 본문 | 저비용 대용량 저장, CDN 연계 가능 |
| **검색 인덱스** | 전문 검색 대상 텍스트, 메타데이터 필터 인덱스 | 복합 검색 쿼리 성능, 언어별 형태소 분석 |
| **감사/로그 저장소** | 운영 로그, 감사 이벤트, 상태 전이 이력 | 주 DB 성능 분리, 장기 보관 |

### 4-3. 도메인 모델과 물리 저장 분리 원칙

- 도메인 모델은 저장 방식에 독립적으로 설계한다.
- 예: `Document.metadata`는 도메인 모델에서 단순히 `Map<String, Any>`이지만, 물리 저장은 JSONB 컬럼이나 별도 EAV 테이블일 수 있다.
- 저장소 구현은 인프라 계층(Repository 패턴)에서 처리하고, 도메인 레이어는 저장소 인터페이스만 의존한다.

### 4-4. 버전 이력 저장 전략

- 버전 본문이 대용량인 경우, 본문 원본은 오브젝트 스토리지에 저장하고 RDB에는 참조 경로와 메타데이터만 저장한다.
- diff 기반 저장(전체 본문이 아닌 변경분만) 여부는 Phase 1에서 검토한다.

## 5. 검색 및 탐색 계층 방향

### 5-1. 관계형 DB만으로 충분한가

초기 MVP에서는 관계형 DB의 전문 검색 확장(PostgreSQL의 `tsvector`, JSONB 인덱스)으로 기본 검색을 제공할 수 있다. 그러나 다음 요구가 발생하면 별도 검색 엔진이 필요해진다.

- 다국어 형태소 분석이 필요한 경우
- Node 단위의 복합 검색 (문서 수만 건 이상)
- 메타데이터 필터 + 전문 검색 동시 쿼리의 성능 문제
- 향후 AI/RAG 파이프라인과 연계하는 벡터 검색

### 5-2. 검색 인덱스 동기화 전략

- 문서·버전·노드 변경 시 검색 인덱스는 비동기로 갱신한다. 동기 처리 시 API 응답 지연이 발생한다.
- 인덱스 갱신 실패 시 재시도 메커니즘이 필요하며, 인덱스와 주 저장소 간 일시적 불일치를 허용하는 Eventually Consistent 모델을 적용한다.

### 5-3. 초기 MVP vs 장기 확장

| 단계 | 검색 전략 |
|------|---------|
| **초기 MVP** | 관계형 DB 전문 검색 확장 (PostgreSQL tsvector + JSONB) |
| **중기** | 별도 검색 엔진 도입 (Elasticsearch, OpenSearch 등) |
| **장기** | 벡터 검색 확장 (AI/RAG 파이프라인 연계) |

초기에 별도 검색 엔진 없이 시작하더라도, **검색 인터페이스는 도메인 계층에서 추상화**하여 검색 엔진 교체 시 API와 도메인 계층의 변경이 없도록 설계한다.

## 6. 비동기 처리 / 이벤트 / 배치 방향

### 6-1. 즉시 응답형 vs 비동기형 분류

| 작업 | 처리 방식 | 이유 |
|------|----------|------|
| 문서 CRUD | 즉시 응답 | 사용자가 결과를 즉시 확인해야 함 |
| 검색 인덱스 갱신 | 비동기 | 응답 지연 방지, Eventually Consistent 허용 |
| 감사 로그 적재 | 비동기 | 메인 처리 흐름 성능 보호 |
| 버전 diff 계산 | 비동기 | 연산 비용이 큼 |
| 대량 문서 등록/가져오기 | 비동기 잡 | 처리 시간이 요청 타임아웃 초과 가능 |
| 외부 시스템 Webhook 발송 | 비동기 | 외부 응답 대기 불필요 |
| 상태 전이 후속 처리 | 비동기 이벤트 | 도메인 이벤트 기반 처리 |
| AI 연계 파이프라인 트리거 | 비동기 | 처리 시간 예측 불가 |

### 6-2. 초기 구조와 장기 확장 방향

- **초기**: 단순한 비동기 잡 큐(in-process 또는 DB 기반 큐)로 시작한다. 외부 메시지 브로커 없이도 기본 비동기 처리가 가능하다.
- **중기**: 처리량이 증가하면 외부 메시지 브로커(RabbitMQ, Kafka 등)로 교체하거나 병행 운영한다.
- 핵심 원칙: **비동기 작업 추상화 인터페이스를 초기부터 분리**하여, 구현 교체 시 도메인 코드 변경이 없도록 한다.

### 6-3. 작업 추적 및 운영 가시성

- 비동기 작업은 작업 ID와 상태를 추적할 수 있어야 한다.
- 작업 실패 시 재시도 횟수와 실패 이유가 기록되어야 한다.
- 긴 작업의 경우 진행 상태를 API로 조회할 수 있어야 한다 (`GET /jobs/{jobId}`).

## 7. 운영 및 배포 관점 요구사항

| 요구 | 방향 |
|------|------|
| **환경 분리** | dev / stage / prod 환경 분리. 환경별 설정을 코드와 분리하여 관리 |
| **설정 관리** | 환경 변수 또는 외부 설정 서비스로 런타임 설정 주입. 하드코딩 금지 |
| **로깅** | 구조화 로그(JSON 포맷)로 수집. 요청 ID를 모든 로그에 포함 |
| **모니터링** | API 응답 시간, 오류율, 비동기 작업 처리 현황 추적 가능해야 함 |
| **감사 가능성** | 주요 도메인 이벤트가 감사 저장소에 구조화된 형태로 기록됨 |
| **장애 대응** | 비동기 작업 실패 시 재시도 가능해야 하며, Dead Letter Queue 또는 실패 로그로 확인 가능 |
| **API 안정성** | API 버전 관리로 소비자에게 하위 호환성 보장 |
| **Admin 운영 데이터** | Admin UI가 운영 현황을 파악할 수 있도록 감사 로그와 상태 데이터가 쿼리 가능해야 함 |

## 8. 확정 사항 vs 후속 결정 사항

### 지금 확정 (Phase 0)

| 항목 | 확정 내용 |
|------|----------|
| 기술 계층 구조 | API → 서비스 → 도메인 → 인프라 계층 분리 |
| 다층 저장소 필요성 | 관계형 DB + JSON 확장 + 오브젝트 스토리지 + 검색 인덱스 |
| API-first 백엔드 구조 | 모든 기능은 API 계층을 통해 노출 |
| User/Admin 프론트엔드 분리 | 라우팅 구조에서 분리 (`/app/*` vs `/admin/*`) |
| 타입 안정성 | 프론트엔드·백엔드 모두 강타입 환경 |
| 비동기 처리 필요성 | 인덱싱·이벤트·감사 로그는 비동기 처리 |
| 검색 인터페이스 추상화 | 검색 엔진 교체에 대비한 인터페이스 분리 |
