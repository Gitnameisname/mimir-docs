# Phase 0 — 저장 구조 전략서

## 문서 목적

이 문서는 Mimir 플랫폼이 다루는 데이터 자산을 어떤 저장 계층에서 어떤 방식으로 관리할지 상위 전략을 정의한다. DB 스키마를 바로 설계하는 것이 아니라, 도메인 모델과 물리 저장 구조를 분리한 상태에서 각 자산의 저장 철학과 계층 역할을 확정한다. Phase 1 도메인 모델 설계가 저장 편의에 끌려가지 않도록 기준을 제공한다.

---

## 1. 저장 계층 역할 정의

Mimir는 단일 저장소가 아닌 다층 저장 전략을 채택한다. 각 계층은 명확한 역할을 가지며, 서로 다른 성격의 데이터 요구를 분담한다.

### 1-1. 관계형 저장 계층 (Relational DB)

**역할**: 플랫폼 핵심 엔티티의 식별자·관계·상태·권한·정책을 정합성 있게 관리한다.

**적합한 자산**:
- Document 기본 정보 (식별자, 유형, 소유자, 상태, 생성일시 등)
- DocumentType 정의
- Version 메타 정보 (버전 번호, 작성자, 생성 시각, 상태)
- Node/Structure 참조 정보 (트리 구조 표현)
- Status/Lifecycle 현재 상태값
- Permission / Policy / Config 정보
- Relationship 정보 (문서 간 참조 관계)
- Attachment 메타 정보 (파일명, 크기, 참조 경로)

**왜 관계형이어야 하는가**:
- 외래키 관계와 트랜잭션으로 데이터 정합성을 보장해야 한다.
- 상태·권한·버전 등 핵심 엔티티 간 관계를 JOIN으로 탐색해야 한다.
- Document·Version·Node·Metadata가 서로 참조하는 복잡한 관계를 안정적으로 표현해야 한다.

**주의사항**:
- 관계형 계층은 무결성과 질의성을 위한 계층이다. 대용량 본문 텍스트나 비정형 확장 속성을 모두 컬럼으로 수용하려 해서는 안 된다.

---

### 1-2. 유연 구조(JSON) 저장 계층

**역할**: 고정 스키마로 표현하기 어려운 확장 가능한 속성을 수용한다. 관계형 계층 내 JSONB 컬럼 또는 별도 EAV 구조로 구현한다.

**적합한 자산**:
- 문서 유형별 확장 메타데이터 값
- DocumentType별 메타데이터 스키마 정의 자체
- 설정/운영 보조 속성
- Node의 유형별 부가 속성

**왜 유연 구조가 필요한가**:
- 문서 유형이 늘어날 때마다 관계형 테이블에 컬럼을 추가하면 스키마 마이그레이션이 반복된다.
- 유형별로 다른 속성 집합을 하나의 고정 스키마로 표현하는 것은 불가능하다.

**무질서 방지 원칙**:
- JSON 확장 필드는 DocumentType이 정의한 메타데이터 스키마에 따라 검증된다. 완전 자유 필드가 아니다.
- 검색·필터링이 필요한 메타데이터 값은 검색 인덱스에 별도로 반영한다.
- 핵심 식별자, 상태, 관계는 반드시 정형 컬럼으로 유지하고 JSON 확장 필드에 중복 보관하지 않는다.

---

### 1-3. 오브젝트/파일 저장 계층

**역할**: 대용량 바이너리 파일과 문서 본문 원본을 저장한다. DB에 직접 저장하기 부적합한 자산을 관리한다.

**적합한 자산**:
- 첨부 파일 원본 (이미지, PDF, Office 문서 등)
- 문서 본문 원본 (대용량 마크다운, HTML, XML 등)
- 버전별 본문 스냅샷 (버전 수가 많을 경우)
- 렌더링 산출물 (PDF 변환본, 프리뷰 이미지 등)

**왜 DB와 분리해야 하는가**:
- 대용량 바이너리를 DB에 저장하면 DB 크기와 백업·복구 비용이 급격히 증가한다.
- 오브젝트 스토리지는 저비용 대용량 저장, CDN 연계, 접근 정책 관리에 최적화되어 있다.

**메타 정보와 파일 분리 원칙**:
- 파일 자체는 오브젝트 스토리지에 저장하고, 파일명·크기·MIME 타입·저장 경로·접근 권한은 관계형 DB에 저장한다.
- 도메인 모델에서는 파일 저장 경로가 아닌 `AttachmentId` 또는 `ResourceRef`로 참조한다.
- 오브젝트 스토리지의 객체 키는 도메인 식별자와 매핑되며, 직접 URL 노출 여부는 보안 정책에 따라 서명된 URL로 제어한다.

---

### 1-4. 검색 인덱스 계층

**역할**: 전문 검색, 메타데이터 필터링, 구조 단위 탐색을 위한 인덱스를 관리한다. 원본 저장소의 복사본이 아니라 검색 목적에 최적화된 별도 표현이다.

**적합한 자산**:
- 문서 본문 전문 검색용 토큰화 데이터
- Node 단위 검색용 구조화 데이터
- 메타데이터 필터링용 색인 보조 구조
- 향후 벡터 임베딩 (AI/RAG 파이프라인)

**왜 원본 저장과 분리해야 하는가**:
- 검색 성능을 위해 역색인, 형태소 분석, 벡터화 등 별도 처리가 필요하다.
- 검색 인덱스는 삭제·재구성이 가능해야 하며, 원본 데이터와 독립적으로 관리된다.
- 인덱스가 손상되더라도 원본에서 재인덱싱으로 복구 가능해야 한다.

**색인 동기화 원칙**:
- 문서·버전·노드 변경 시 검색 인덱스는 비동기로 갱신한다.
- 일시적인 인덱스 지연(Eventually Consistent)은 허용하되, 지연 범위를 모니터링한다.
- 인덱스 갱신 실패는 재시도 큐로 처리하고 실패 이력을 기록한다.

**초기 MVP vs 장기 확장**:
- 초기: PostgreSQL tsvector + JSONB 인덱스로 기본 검색 제공
- 중기: 별도 검색 엔진(Elasticsearch, OpenSearch) 도입
- 장기: 벡터 검색 확장 (AI/RAG 연계)

---

### 1-5. 로그·감사·운영 저장 계층

**역할**: 플랫폼 운영 이벤트, 감사 기록, 비동기 잡 상태를 도메인 데이터와 분리하여 저장한다.

**적합한 자산**:
- 감사 이벤트 (누가, 언제, 무엇을, 어떤 대상에 했는가)
- 상태 전이 이력
- 비동기 잡 상태 및 결과
- 인덱싱/동기화 작업 이력
- 운영 오류·재시도 기록

**왜 도메인 데이터와 분리해야 하는가**:
- 감사 로그는 대용량이며 쓰기가 잦아 주 DB 성능에 영향을 준다.
- 감사 데이터는 도메인 데이터와 다른 보존 정책(장기 보관, 불변 기록)을 가진다.
- 로그 계층은 분석과 감사 목적의 쿼리 패턴에 최적화되어야 한다.

## 2. 현재 상태 / 버전 / 이력 관계 정리

이 세 개념은 구분이 흐려지면 도메인 모델 전체가 뒤엉키므로 명확히 정의한다.

### 개념 정의

| 개념 | 의미 | 저장 계층 |
|------|------|----------|
| **현재 상태 (Current State)** | 사용자에게 보이는 Document의 최신 유효 상태. 특정 Version을 가리키는 포인터 또는 파생 뷰. | 관계형 DB (Document 테이블의 현재 버전 참조) |
| **Version** | Document 내용의 특정 시점 스냅샷. 모든 수정은 새 Version을 생성한다. | 관계형 DB (Version 테이블) + 오브젝트 스토리지 (본문) |
| **상태 전이 이력 (Status History)** | Document의 상태(초안→검토→승인→게시→폐기)가 어떻게 변화했는지의 기록. | 로그·감사 계층 |
| **감사 로그 (Audit Log)** | 누가 언제 어떤 행위를 했는지의 추적 기록. Version 생성, 상태 전이, 권한 변경 등 포함. | 로그·감사 계층 |

### 관계 다이어그램

```
Document (현재 상태)
    └── currentVersionId ──→ Version N (최신)
    
Version 이력
    ├── Version 1 (초안, 2024-01)
    ├── Version 2 (수정, 2024-02)
    └── Version N (현재, 2024-03)
    
상태 전이 이력
    ├── DRAFT → REVIEW (2024-01-10, 작성자 A)
    ├── REVIEW → APPROVED (2024-01-15, 승인자 B)
    └── APPROVED → PUBLISHED (2024-01-16, 운영자 C)
    
감사 로그
    ├── [2024-01-10] User A: created Version 1
    ├── [2024-01-12] User A: updated Version 2
    ├── [2024-01-15] User B: approved Document
    └── [2024-01-16] User C: published Document
```

### 핵심 원칙

- **Document 수정은 항상 새 Version을 생성한다.** 기존 Version 본문을 덮어쓰지 않는다.
- **Document의 현재 상태는 최신 유효 Version을 가리키는 포인터다.** Document 자체에 본문을 저장하지 않는다.
- **상태 전이 이력과 Version 이력은 별개다.** 상태가 바뀐다고 Version이 생성되지는 않으며, Version이 생성된다고 상태가 바뀌지도 않는다.
- **감사 로그는 도메인 데이터가 아니다.** 감사 로그를 도메인 테이블에 직접 쌓지 않는다.

## 3. Node/Structure 저장 전략 방향

### 요구사항

- 문서 내부 계층 구조(섹션 > 조항 > 항목 등)를 플랫폼이 인식하고 관리해야 한다.
- Node는 독립적으로 검색되고, 메타데이터를 부착받을 수 있어야 한다.
- 트리 순서, 깊이, 부모-자식 관계를 저장해야 한다.
- 버전 관리가 Document 전체 수준인지 Node 수준인지를 Phase 1에서 결정해야 한다.

### 상위 전략 방향

- Node의 구조 메타 정보(순서, 부모 참조, 유형)는 관계형 DB에 저장한다.
- Node의 본문 내용은 크기에 따라 관계형 컬럼 또는 오브젝트 스토리지에 저장한다.
- 트리 표현 방식(인접 리스트, 경로 열거, 중첩 집합 등)은 Phase 1 상세 설계에서 결정한다. 이 단계에서는 특정 기법에 종속되지 않는다.
- 구조 표현(Node 트리)과 본문 표현(Node 내용)은 분리하여 설계한다.

---

## 4. Metadata 저장 전략 방향

### 요구사항

- 공통 메타데이터(모든 문서 유형 공통): 작성자, 작성일, 태그, 언어 등
- 유형별 확장 메타데이터: DocumentType이 정의한 스키마에 따라 다름
- 메타데이터는 검색·필터링에 사용되어야 한다.
- 메타데이터 스키마는 관리자가 정의하고 변경할 수 있어야 한다.

### 상위 전략 방향

| 구분 | 저장 방식 | 이유 |
|------|----------|------|
| 공통 메타데이터 | 관계형 컬럼 | 질의, 정렬, 필터에 직접 사용 |
| 유형별 확장 메타데이터 값 | JSONB 컬럼 | 유형별 스키마가 다르므로 유연성 필요 |
| 메타데이터 스키마 정의 | 관계형 테이블 (JSON 포함) | 구조 자체는 정형, 내부 스키마 표현은 JSON |

### 검증과 유연성의 균형

- 확장 메타데이터 값은 DocumentType이 정의한 스키마에 따라 저장 전 검증한다.
- 스키마 검증 실패 시 저장을 거부한다. 완전 자유 필드가 아니다.
- 필터링·검색이 필요한 메타데이터 필드는 검색 인덱스에도 반영한다.

## 5. 첨부·외부 자원 참조 전략 방향

### 분리 원칙

- **파일 자체**: 오브젝트 스토리지에 저장
- **파일 메타 정보** (파일명, 크기, MIME 타입, 저장 경로, 업로드 시각, 업로더): 관계형 DB에 저장
- **도메인 참조**: 도메인 모델은 파일 경로가 아닌 `AttachmentId`로 참조

### 외부 리소스 참조

- 외부 URL 참조(외부 문서 링크, 외부 시스템 리소스)는 `ExternalResourceRef` 엔티티로 관리한다.
- 외부 URL의 가용성은 플랫폼이 보장하지 않는다. 참조 경로와 접근 시각만 기록한다.

### 수명주기와 권한

- 첨부 파일 접근은 Document 접근 권한과 연동한다.
- 파일 직접 URL 노출 대신 서명된 임시 URL을 발급한다.
- Document 삭제 시 첨부 파일의 물리 삭제 여부는 보존 정책에 따라 결정한다. (즉시 삭제 vs 소프트 삭제)

---

## 6. 검색·AI 확장을 고려한 저장 전략

### 검색 확장 여지

- 문서 본문과 Node 내용은 검색 가능한 형태로 추출할 수 있어야 한다. (텍스트 추출 파이프라인)
- 메타데이터는 검색 인덱스에 필터 가능한 형태로 반영된다.
- Node 단위 분해(청크) 가능성을 저장 구조에서 배제하지 않는다.

### AI/RAG 확장 여지

- 벡터 임베딩은 원본 저장과 분리된 별도 인덱스로 관리한다.
- 임베딩 인덱스가 손상되어도 원본에서 재생성 가능해야 한다.
- AI 보조 데이터(임베딩, 요약, 추출 키워드)는 도메인 엔티티에 직접 포함하지 않는다. 별도 보조 저장소로 관리한다.

---

## 7. 저장 원칙 및 안티패턴

### 원칙 요약

| 원칙 | 내용 |
|------|------|
| 도메인 우선 | 도메인 모델을 먼저 정의하고, 저장 방식은 이를 지지하도록 선택한다. |
| 역할별 분리 | 관계형·JSON·오브젝트·검색·로그 계층의 역할을 혼합하지 않는다. |
| 현재 상태와 이력 분리 | 현재 유효 상태와 버전 이력, 감사 로그를 별도로 관리한다. |
| 검색 인덱스는 파생물 | 검색 인덱스는 원본에서 파생된 복사본이며, 원본 대체재가 아니다. |
| 메타데이터 통제 | JSON 확장 필드는 스키마 검증을 통해 무질서를 방지한다. |
| 첨부 분리 | 파일 본체와 메타 정보를 분리 저장한다. |

### 안티패턴

**안티패턴 1. 모든 것을 단일 JSON blob으로 저장**

문제: 질의 불가, 무결성 보장 불가, 메타데이터 필터링 불가, 인덱싱 불가.
위반 원칙: 도메인 우선, 역할별 분리.
대안: 정형 필드는 관계형 컬럼으로, 확장 속성만 JSON으로 분리.

**안티패턴 2. 모든 메타데이터를 관계형 컬럼으로만 강제**

문제: 유형별 속성 추가 시마다 스키마 마이그레이션 필요. 유형 확장 불가.
위반 원칙: 도메인 우선, 메타데이터 통제.
대안: 공통 속성은 컬럼, 유형별 확장 속성은 JSON 확장 필드.

**안티패턴 3. 버전과 최신 상태를 구분하지 않고 덮어쓰기**

문제: 변경 이력 소실, 복원 불가, 버전 비교 불가.
위반 원칙: 현재 상태와 이력 분리.
대안: 수정 시 새 Version 생성, Document는 최신 Version 포인터 유지.

**안티패턴 4. 검색 인덱스를 원본 데이터처럼 사용**

문제: 인덱스 손상 시 데이터 복구 불가. 인덱스와 원본 간 불일치 발생 시 원본을 신뢰할 수 없음.
위반 원칙: 검색 인덱스는 파생물.
대안: 원본은 항상 관계형 DB + 오브젝트 스토리지가 진실의 원천. 인덱스는 언제든 재구성 가능.

**안티패턴 5. 감사 로그를 비즈니스 테이블에 직접 저장**

문제: 도메인 테이블 비대화. 감사 로그 쿼리가 도메인 쿼리 성능에 영향. 보존 정책 적용 어려움.
위반 원칙: 역할별 분리.
대안: 감사 이벤트를 별도 감사 저장소(테이블 또는 로그 시스템)에 비동기로 기록.

**안티패턴 6. 첨부 파일을 DB에 BLOB으로 저장**

문제: DB 크기 폭증. 백업·복구 비용 증가. 스트리밍 서비스 불가.
위반 원칙: 첨부 분리.
대안: 오브젝트 스토리지에 저장하고, DB에는 참조 정보만 유지.
