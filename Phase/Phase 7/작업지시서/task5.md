# Task 7-5. DocumentType 관리 화면 설계

## 1. 작업 목적

문서 유형(DocumentType)을 운영자가 조회하고 관리할 수 있는 Admin UI 화면을 설계한다.

이 작업의 핵심 목표는 다음과 같다.

- DocumentType별 메타데이터 스키마 및 설정 정보를 운영자가 파악할 수 있도록 한다
- 타입 활성/비활성 제어 흐름을 정의한다
- 향후 editor / renderer / workflow / chunking 플러그인 설정 확장 포인트를 구조에 반영한다

---

## 2. 작업 범위

### 포함 범위

- DocumentType 목록 화면 설계
- DocumentType 상세 화면 설계
- 타입 활성/비활성 흐름 설계
- 메타데이터 스키마 표시 구조 설계
- 향후 확장 필드(플러그인 설정) 반영 방안 정리

### 제외 범위

- DocumentType 실제 생성/삭제 구현 (MVP 이후)
- 플러그인 시스템 구현

## 4. 주요 설계 대상

### 4-1. DocumentType 목록 화면 (`/admin/document-types`)

#### 목록 테이블 컬럼

- 타입 코드 (예: POLICY, MANUAL, REPORT)
- 표시명
- 설명
- 상태 (ACTIVE / INACTIVE)
- 메타데이터 필드 수
- 문서 수
- 생성일

#### 기능

- 키워드 검색 (타입 코드, 표시명)
- 필터 (상태)
- 행 클릭 시 상세 화면 진입

---

### 4-2. DocumentType 상세 화면 (`/admin/document-types/{type_code}`)

#### 표시 정보

**기본 정보**

- 타입 코드
- 표시명
- 설명
- 상태
- 생성일 / 최종 수정일

**메타데이터 스키마 섹션**

- 필드 목록 테이블:
  - 필드명
  - 필드 타입 (string, number, date, enum 등)
  - 필수 여부
  - 기본값
  - 설명

**플러그인 설정 섹션 (확장 포인트)**

- Editor 설정 (어떤 에디터 플러그인 사용)
- Renderer 설정
- Workflow 연결 설정 (기본 워크플로 타입)
- Chunking 설정
- 각 항목은 현재 설정값 표시 및 향후 편집 가능 구조로 Reserved

**사용 현황**

- 이 타입으로 생성된 문서 수
- 활성 문서 / 비활성 문서 수

#### 주요 액션

- 타입 비활성화 / 활성화
- (MVP 이후) 메타데이터 스키마 편집
- (MVP 이후) 플러그인 설정 편집

---

### 4-3. 타입 상태 변경 흐름

#### 비활성화 흐름

1. 비활성화 버튼 클릭
2. 영향 안내 다이얼로그 표시:
   - 현재 이 타입을 사용하는 문서 수
   - 비활성화 시 신규 문서 생성 불가 안내
   - 기존 문서에 미치는 영향 안내
3. 변경 사유 입력 (선택)
4. 확인 후 처리 결과 Toast 및 화면 갱신
5. 감사 이벤트 자동 기록 (설계 명시)

---

### 4-4. 메타데이터 스키마 표시 구조

스키마는 운영자가 쉽게 이해할 수 있도록 다음 형식으로 표시한다:

```
필드명       타입      필수    기본값    설명
──────────────────────────────────────────
title        string    Y       -         문서 제목
author       string    Y       -         작성자
version      string    N       "1.0"     버전 표기
tags         array     N       []        분류 태그
```

- JSON Schema 원본 표시는 Expand/Collapse 가능한 코드 블록으로 제공
- 필드 의미 설명이 있으면 함께 표시

---

### 4-5. 향후 확장 포인트 설계

플러그인 설정 영역은 현재 조회 전용으로 제공하고, 향후 편집 가능하도록 구조를 예약한다:

| 설정 카테고리 | 현재 | 향후 |
|--------------|------|------|
| Editor | 조회 전용 | 편집 가능 |
| Renderer | 조회 전용 | 편집 가능 |
| Default Workflow | 조회 전용 | 편집 가능 |
| Chunking | 조회 전용 | 편집 가능 |

신규 타입 추가 흐름도 구조적으로 예약:

- "새 타입 추가" 버튼 UI 위치 예약 (비활성 상태로 표시)
- 추가 시 필요한 필드 정의 (타입 코드, 표시명, 메타데이터 스키마)

- DocumentType 목록 및 상세 화면 구조가 명확히 정의되어 있다
- 메타데이터 스키마 표시 방식이 구체적으로 기술되어 있다
- 타입 상태 변경 흐름과 영향 안내가 설계에 반영되어 있다
- 향후 플러그인 설정 편집 확장을 위한 구조 여지가 반영되어 있다

---

## 7. Codex 작업 지침

- 코드 작성 금지 (설계 중심)
- DocumentType은 플랫폼의 핵심 설정이므로 운영자가 충분히 이해하고 변경할 수 있는 UX를 설계한다
- 하드코딩 금지 원칙에 따라 타입 목록은 DB에서 동적으로 조회하는 구조로 설계한다
- 메타데이터 스키마는 사람이 읽기 쉬운 형태와 원본 JSON 두 가지 모두 제공한다
- 향후 확장 포인트는 현재 구현하지 않더라도 설계에서 명시한다
