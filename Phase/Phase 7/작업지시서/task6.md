# Task 7-6. 감사 로그 조회 UI 설계

## 1. 작업 목적

운영자가 시스템 내 행위 이력을 추적하고 분석할 수 있는 감사 로그 조회 화면을 설계한다.

이 작업의 핵심 목표는 다음과 같다.

- 기간/행위자/리소스/이벤트 타입 기준 필터링이 가능한 감사 로그 UI 정의
- 이벤트 상세 패널을 통해 변경 전후 값 확인 구조 설계
- 보안 및 운영상 중요한 이벤트를 우선 가시화하는 방식 정의

---

## 4. 주요 설계 대상

### 4-1. 감사 로그 목록 화면 (`/admin/audit-logs`)

#### 목록 테이블 컬럼

- 발생 시각
- 중요도 배지 (CRITICAL / HIGH / NORMAL)
- 이벤트 타입 (USER_LOGIN, PERMISSION_CHANGED, DOCUMENT_DELETED 등)
- 행위자 (사용자명/시스템)
- 대상 리소스 (리소스 타입 + ID)
- 결과 (SUCCESS / FAILURE)
- IP 주소

#### 기능

- 기간 범위 필터 (오늘 / 최근 7일 / 최근 30일 / 직접 입력)
- 행위자 검색
- 이벤트 타입 필터 (다중 선택)
- 결과 필터 (성공/실패)
- 중요도 필터
- 리소스 타입 필터

---

### 4-2. 감사 이벤트 상세 패널

목록에서 이벤트 선택 시 우측 패널 또는 모달로 표시:

#### 표시 정보

**기본 정보**

- 이벤트 ID
- 발생 시각
- 이벤트 타입
- 중요도
- 행위자 정보 (이름, 이메일, IP, 세션 정보)
- 결과 및 오류 메시지 (실패 시)

**대상 리소스**

- 리소스 타입
- 리소스 ID
- 리소스 표시명 (가능 시)

**변경 내역 (해당 시)**

- Before / After 비교 표시
- JSON diff 형태 또는 필드별 변경값 표시

**컨텍스트 정보**

- 요청 경로 (API endpoint)
- 연관 이벤트 링크 (동일 세션/트랜잭션)

---

### 4-3. 중요 이벤트 강조 기준

중요도 분류 기준 정의:

| 중요도 | 해당 이벤트 예시 | 표시 방식 |
|--------|----------------|-----------|
| CRITICAL | 권한 정책 변경, 관리자 계정 변경, API 키 폐기 | 빨간 배지 + 행 강조 |
| HIGH | 사용자 비활성화, 역할 변경, 문서 강제 삭제 | 주황 배지 |
| NORMAL | 로그인, 문서 조회, 상태 변경 등 일반 이벤트 | 기본 표시 |

CRITICAL/HIGH 이벤트는 목록 상단 또는 별도 섹션으로 우선 표시 가능한 구조 고려.

---

### 4-4. 변경 전후 값 비교 표시 구조

정책 변경, 메타데이터 수정 등 값 변경 이벤트에 대한 비교 UI:

#### 단순 필드 변경

```
필드명: status
Before: ACTIVE
After:  INACTIVE
```

#### JSON/구조 변경

- Before / After 코드 블록 나란히 표시
- 변경된 라인 강조 (diff 스타일)

#### 주의 사항

- 민감 정보(비밀번호, 토큰)는 마스킹 처리
- 변경 내역이 없는 이벤트는 해당 섹션 숨김

## 7. Codex 작업 지침

- 코드 작성 금지 (설계 중심)
- 감사 로그는 운영 보안의 핵심이므로 조회 성능과 필터 편의성을 최우선으로 설계한다
- 민감 정보 마스킹 원칙을 설계에서 명시한다
- 이벤트 상세 보기는 별도 페이지 이동 없이 패널로 제공하여 흐름을 끊지 않는다
- 중요한 이벤트를 운영자가 놓치지 않도록 가시화 방식을 구체적으로 정의한다
