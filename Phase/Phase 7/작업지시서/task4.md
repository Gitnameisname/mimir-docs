# Task 7-4. 권한 정책 관리 화면 설계

## 1. 작업 목적

역할 기반 권한 정책을 운영 UI에서 조회하고 관리할 수 있는 화면을 설계한다.

이 작업의 핵심 목표는 다음과 같다.

- 권한 정책 목록과 상세 내용을 운영자가 파악할 수 있도록 한다
- 정책 편집 시 충돌/누락/위험성을 운영자에게 명확히 전달한다
- 정책 변경 이력을 추적 가능하게 연계한다

---

## 2. 작업 범위

### 포함 범위

- 권한 정책 목록 화면 설계
- 정책 상세 화면 설계
- 역할별 권한 매핑 확인 UI 설계
- 정책 편집 흐름 설계
- 위험한 정책 변경에 대한 확인 UX 정의
- 향후 ACL/리소스 단위 정책 확장 가능 구조 반영

### 제외 범위

- 권한 정책 엔진 구현
- 역할/사용자 기본 정보 편집 (Task 7-3에서 다룸)

---

## 3. 선행 조건

- Task 7-1 (Admin UI 정보 구조 및 접근 정책 설계) 완료
- Task 7-2 (Admin 공통 레이아웃 및 UI 패턴 설계) 완료
- Phase 2: 역할 및 권한 정책 모델 정의 완료

---

## 4. 주요 설계 대상

### 4-1. 권한 정책 목록 화면 (`/admin/permissions`)

#### 목록 테이블 컬럼

- 정책명
- 설명
- 연결된 역할 수
- 리소스 유형 (Document, WorkflowType, System 등)
- 허용 액션 (READ, WRITE, DELETE 등)
- 상태 (ACTIVE / INACTIVE)
- 최종 변경일

#### 기능

- 키워드 검색
- 필터 (역할, 리소스 유형, 상태)
- 행 클릭 시 정책 상세 진입

---

### 4-2. 정책 상세 화면 (`/admin/permissions/{policy_id}`)

#### 표시 정보

- 정책 기본 정보 (이름, 설명, 생성일)
- 허용 액션 목록
- 적용 리소스 조건 (전체/특정 조건)
- 연결된 역할 목록
- 변경 이력 요약 (최근 N건)

#### 주요 액션

- 정책 편집
- 정책 비활성화/활성화
- 역할 연결/해제

---

### 4-3. 역할별 권한 매핑 확인 UI

역할을 기준으로 어떤 정책이 적용되어 있는지 확인하는 매트릭스 뷰:

- 행: 역할(Role)
- 열: 리소스 유형 또는 주요 액션 그룹
- 셀: 허용/거부/미정의 상태 표시

목적: 운영자가 역할별 권한 현황을 한눈에 파악할 수 있도록 한다.

---

### 4-4. 정책 편집 흐름

정책 편집 시 단계별 UX:

1. 정책 상세 화면에서 편집 버튼 클릭
2. 편집 폼 표시 (현재 값 초기값으로 설정)
3. 변경 내역 미리보기 표시
4. 영향 받는 역할/사용자 수 경고 표시
5. 위험 액션 시 확인 다이얼로그 (변경 사유 입력 포함)
6. 저장 후 감사 이벤트 자동 기록 (설계 명시)

#### 편집 가능 항목

- 정책 설명
- 허용 액션 추가/제거
- 리소스 조건 변경
- 상태 변경

---

### 4-5. 위험한 정책 변경 UX

다음 변경은 위험 액션으로 분류하여 강화된 확인 절차 적용:

- 광범위 리소스에 대한 DELETE 권한 추가
- 정책 비활성화 (연결 역할이 많을수록 위험)
- 관리자 역할의 정책 변경

위험 다이얼로그 구성:

- 경고 아이콘 및 위험 레벨 표시
- 영향 받는 사용자/역할 수 명시
- 변경 사유 입력 필드 (필수)
- Danger 스타일 확인 버튼

---

### 4-6. 향후 확장 포인트

리소스 단위(ACL) 정책 확장을 위한 설계 여지:

- 정책에 리소스 인스턴스 조건 추가 가능 구조
- 조건 표현식(Condition Expression) 입력 영역 Reserved
- 정책 상속/우선순위 표시 가능 구조

---

## 5. 산출물

- 권한 정책 목록 및 상세 화면 구조가 정의되어 있다
- 역할-정책 매핑 확인 UI가 설계되어 있다
- 정책 편집 흐름과 위험 액션 확인 절차가 명확히 기술되어 있다
- 향후 ACL 확장 가능한 구조가 설계에 반영되어 있다

---

## 7. Codex 작업 지침

- 코드 작성 금지 (설계 중심)
- 권한 변경은 파급 효과가 크므로 운영자에게 충분한 정보를 제공하는 UX를 설계한다
- 변경 전 영향 범위를 명시하는 패턴을 핵심으로 한다
- 감사 이벤트 연계 포인트를 설계에서 명시한다
- 향후 정책 엔진 확장을 고려한 구조 여지를 남긴다
