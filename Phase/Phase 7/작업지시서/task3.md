# Task 7-3. 사용자 / 조직 / 역할 관리 화면 설계

## 1. 작업 목적

운영자가 플랫폼의 계정 구조와 조직 체계를 관리할 수 있는 화면을 설계한다.

이 작업의 핵심 목표는 다음과 같다.

- 사용자, 조직, 역할 각각의 목록/상세/편집 흐름 정의
- 사용자-조직-역할 매핑 관리 UX 설계
- 상태 변경(활성/비활성) 등 운영 액션 흐름 정의

---

## 2. 작업 범위

### 포함 범위

- 사용자 목록 화면 설계
- 사용자 상세 화면 설계
- 사용자 상태 변경 흐름 설계
- 조직 목록/상세 화면 설계
- 역할(Role) 관리 화면 설계
- 사용자-조직-역할 매핑 편집 흐름 설계

### 제외 범위

- 실제 API 구현
- 권한 정책 편집 (Task 7-4에서 다룸)

---

## 3. 선행 조건

- Task 7-1 (Admin UI 정보 구조 및 접근 정책 설계) 완료
- Task 7-2 (Admin 공통 레이아웃 및 UI 패턴 설계) 완료
- Phase 2: 역할/권한 모델 정의 완료

---

## 4. 주요 설계 대상

### 4-1. 사용자 목록 화면 (`/admin/users`)

#### 목록 테이블 컬럼

- 이름
- 이메일
- 소속 조직
- 역할
- 상태 (ACTIVE / INACTIVE / SUSPENDED)
- 가입일
- 최근 로그인

#### 기능

- 키워드 검색 (이름, 이메일)
- 필터 (상태, 조직, 역할)
- 정렬 (가입일, 이름)
- 행 클릭 시 상세 패널 또는 상세 페이지 진입

---

### 4-2. 사용자 상세 화면 (`/admin/users/{user_id}`)

#### 표시 정보

- 기본 정보 (이름, 이메일, 가입일)
- 소속 조직 목록
- 부여된 역할 목록
- 계정 상태 및 이력
- 최근 감사 이벤트 요약

#### 주요 액션

- 상태 변경 (활성화 / 비활성화 / 정지)
- 역할 추가/제거
- 조직 매핑 변경
- 비밀번호 초기화 (선택)

---

### 4-3. 사용자 상태 변경 UX

상태 변경 시 확인 절차:

1. 상태 변경 버튼 클릭
2. 확인 다이얼로그 표시 (변경 대상, 변경 내용, 영향 범위 안내)
3. 변경 사유 입력 필드 (선택)
4. 확인 후 처리 결과 Toast 표시
5. 감사 이벤트 자동 기록 (설계 명시)

---

### 4-4. 조직 목록 화면 (`/admin/organizations`)

#### 목록 테이블 컬럼

- 조직명
- 소속 사용자 수
- 상태
- 생성일

#### 기능

- 키워드 검색
- 필터 (상태)
- 행 클릭 시 조직 상세 진입

---

### 4-5. 조직 상세 화면 (`/admin/organizations/{org_id}`)

#### 표시 정보

- 조직 기본 정보
- 소속 사용자 목록 (역할 포함)
- 조직 레벨 설정 정보

#### 주요 액션

- 사용자 추가/제거
- 역할 매핑 변경

---

### 4-6. 역할 관리 화면 (`/admin/roles`)

#### 목록 테이블 컬럼

- 역할명
- 설명
- 부여된 사용자 수
- 연결된 권한 정책 수

#### 기능

- 역할 상세 조회
- 역할에 부여된 권한 정책 확인
- 역할 활성/비활성 (선택)

---

### 4-7. 사용자-조직-역할 매핑 편집 흐름

매핑 변경 시 UX 흐름:

1. 사용자 상세 또는 조직 상세 화면에서 매핑 편집 버튼 클릭
2. 매핑 편집 모달 표시
3. 조직/역할 선택 (드롭다운 또는 검색 가능 목록)
4. 변경 내역 미리보기
5. 확인 다이얼로그 후 저장
6. 결과 Toast 및 화면 갱신

---

## 5. 산출물

1. 사용자 관리 화면 설계서 (목록/상세/상태 변경)
2. 조직 관리 화면 설계서
3. 역할 관리 화면 설계서
4. 매핑 편집 흐름 정의서

---

## 6. 완료 기준

- 사용자/조직/역할 각각의 목록 및 상세 화면 구조가 정의되어 있다
- 상태 변경 및 매핑 편집 흐름이 명확히 기술되어 있다
- 확인 절차(다이얼로그, 감사 이벤트 연계)가 설계에 반영되어 있다
- Task 7-2에서 정의한 공통 패턴을 기준으로 설계되어 있다

---

## 7. Codex 작업 지침

- 코드 작성 금지 (설계 중심)
- 운영자가 빠르게 파악하고 조치할 수 있는 UX를 우선한다
- 각 화면의 주요 정보와 액션을 명확하게 구분한다
- 민감한 변경 액션에는 확인 절차를 반드시 포함한다
- 감사 이벤트 연계 포인트를 설계에서 명시한다
