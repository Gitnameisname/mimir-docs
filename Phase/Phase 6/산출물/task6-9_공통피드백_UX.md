# Task 6-9 산출물: 공통 피드백 및 상호작용 UX 정리

---

## 1. 공통 피드백 UX 가이드

### 1-1. 피드백 채널 분류

| 채널 | 사용 시점 | 컴포넌트 |
|------|----------|---------|
| Toast | 비동기 액션 결과, 단기 알림 | `Toast` |
| Inline Error | 폼 필드 유효성 오류 | `FormField` 내 오류 텍스트 |
| Banner | 페이지 수준 중요 상태 안내 | `Banner` |
| Modal | 위험 액션 확인, 입력 요구 | `ConfirmDialog` / `ActionModal` |
| Tooltip | 비활성 버튼 이유 설명 | `Tooltip` |
| Page State | 전체 페이지 상태 (Empty/Error/Loading) | `EmptyState`, `ErrorState`, `LoadingSpinner` |

## 3. 비동기 상태 표시 정의

### 3-1. 액션별 비동기 처리 UX

| 액션 | 처리 중 | 성공 | 실패 |
|------|--------|------|------|
| 문서 저장 | 버튼 disabled + "저장 중..." + Spinner | "저장됨 ✓" 텍스트 갱신 | Toast (error) + 재시도 |
| 검토 요청 | Modal 버튼 disabled + Spinner | Modal 닫힘 + Toast (success) + 상태 갱신 | Modal 내 오류 메시지 |
| 승인 | Modal 버튼 disabled + Spinner | Modal 닫힘 + Toast (success) + 상태 갱신 | Modal 내 오류 메시지 |
| 반려 | Modal 버튼 disabled + Spinner | Modal 닫힘 + Toast (success) + 상태 갱신 | Modal 내 오류 메시지 |
| 발행 | ConfirmDialog 버튼 disabled + Spinner | Toast (success) + 상태 갱신 | Toast (error) |
| 버전 복원 | ConfirmDialog 버튼 disabled + Spinner | Toast (success) + 편집 화면 이동 | Toast (error) |
| 페이지 로딩 | SkeletonBlock (콘텐츠 영역) | 콘텐츠 렌더링 | ErrorState |

### 3-2. 처리 중 상태 원칙

```
1. 동일 액션 중복 실행 방지: 처리 중 상태에서 동일 버튼 비활성화
2. 사용자 탈출 허용: 처리 중에도 [취소] 버튼 유지 (가능한 경우)
3. 타임아웃 처리: 30초 이상 응답 없으면 오류 처리
4. 낙관적 UI: 빠른 반응이 필요한 경우에만 선택적 사용
   (저장 등 — 단, 서버 실패 시 되돌리기 필요)
```

## 4. Confirm Dialog 정의

### 4-1. 공통 ConfirmDialog 구조

```
┌─────────────────────────────────────────┐
│  [아이콘] 제목 텍스트                    │
├─────────────────────────────────────────┤
│  설명 텍스트                             │
│  (위험 여부, 결과 안내)                   │
│                                          │
│  [추가 입력 필드] (선택 사항)             │
├─────────────────────────────────────────┤
│              [취소]  [확인/실행]           │
└─────────────────────────────────────────┘
```

### 4-2. Dialog 사용 기준

| 액션 | Dialog 필요 여부 | 이유 |
|------|----------------|------|
| 저장 | 불필요 | 가역적, 반복적 |
| 검토 요청 | 필요 (선택) | 상태 전이 — 가볍게 확인 |
| 승인 | 필요 (의견 입력 포함) | 중요한 상태 전이 |
| 반려 | 필요 (사유 입력 필수) | 중요 + 사유 필수 |
| 발행 | 필요 (ConfirmDialog) | 되돌리기 어려운 액션 |
| 보관 | 필요 (ConfirmDialog) | 비가역적 |
| 버전 복원 | 필요 (ConfirmDialog) | 새 버전 생성 안내 |
| 편집 이탈 | 필요 (Unsaved Changes) | 데이터 손실 방지 |

### 4-3. Unsaved Changes 경고 Dialog

```
발생 조건: 편집 중 (변경사항 있음) + 페이지 이탈 시도

구성:
  제목: "저장하지 않은 변경사항이 있습니다"
  설명: "이 페이지를 떠나면 변경사항이 사라집니다."
  버튼: [계속 편집]  [저장 없이 나가기]

처리 범위:
  - Sidebar 네비게이션 클릭
  - 브라우저 뒤로 가기 (router.beforePopState)
  - 브라우저 탭 닫기 / 새로고침 (window.beforeunload)
```

## 5. 폼 검증 및 오류 메시지 정책

### 5-1. 유효성 검사 시점

| 시점 | 처리 방식 |
|------|----------|
| 실시간 (onChange) | 빠른 피드백 — 필드 포커스 아웃 후 활성화 |
| 제출 시 (onSubmit) | 모든 필드 일괄 검사 |
| 서버 응답 (422) | 서버 반환 오류 메시지 인라인 표시 |

### 5-2. 오류 메시지 원칙

```
1. 인라인 우선: 폼 필드 하단에 직접 표시
2. 구체적 안내: "입력하세요" → "반려 사유를 입력해 주세요"
3. 복구 방법 제시: "최소 2자 이상 입력해 주세요"
4. 복수 오류: 모든 오류 동시 표시 (순차 표시 금지)
```

### 5-3. 필수/선택 표시

```
필수 필드: 레이블 옆 * 마크 + 미입력 시 오류
선택 필드: "(선택 사항)" 표시 또는 생략
```

## 6. 예외 상황 UX 문서

### 6-1. 접근 권한 부족

```
상황: 사용자가 접근 권한 없는 화면/기능 시도
처리:
  - 화면 전체 차단: PermissionDenied 컴포넌트
  - 버튼 비활성화: disabled + tooltip
  - API 403 응답: Toast "권한이 없습니다" + 상태 유지 (이동 없음)
```

### 6-2. 상태 불일치

```
상황: URL 직접 입력, 외부 링크 등으로 잘못된 상태 화면 진입
처리:
  - /documents/{id}/edit 진입 시 편집 불가 상태면:
    → Toast: "현재 상태에서는 편집이 불가합니다"
    → /documents/{id} 로 리다이렉트
```

### 6-3. 네트워크/서버 오류

```
상황: API 호출 실패
처리:
  - 페이지 초기 로드 실패: ErrorState (전체) + [다시 시도] 버튼
  - 액션 중 실패: Toast (error) + 버튼 재활성화 (재시도 가능)
  - 네트워크 끊김: Banner "네트워크 연결을 확인해 주세요"
```

### 6-4. 세션 만료

```
상황: 장시간 비활성 후 API 401 응답
처리:
  - Toast: "세션이 만료되었습니다. 다시 로그인해 주세요"
  - /login 리다이렉트 (현재 URL 저장하여 로그인 후 복귀)
```
