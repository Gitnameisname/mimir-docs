## 1. 버전 목록 UI 설계서 (`/documents/[id]/versions`)

### 1-1. 화면 레이아웃

```
┌─────────────────────────────────────────────────────────────┐
│  Page Header                                                 │
│  [← 문서로 돌아가기]  "버전 기록: [문서 제목]"               │
├─────────────────────────────────────────────────────────────┤
│  버전 타임라인                                               │
│                                                              │
│  ● v1.3  [현재 버전] [IN_REVIEW]                             │
│  │       홍길동 · 2026-04-08 · "1분기 검토 반영"             │
│  │       [상세 보기]                                         │
│  │                                                           │
│  ● v1.2  [PUBLISHED]                                         │
│  │       홍길동 · 2026-03-20 · "승인 후 발행"                │
│  │       [상세 보기]  [이 버전으로 복원]                      │
│  │                                                           │
│  ● v1.1  [DRAFT]                                             │
│  │       김철수 · 2026-03-10 · "초기 작성"                   │
│  │       [상세 보기]                                         │
│  │                                                           │
│  ● v1.0  [DRAFT]                                             │
│           김철수 · 2026-03-01 · "문서 최초 생성"              │
│           [상세 보기]                                         │
└─────────────────────────────────────────────────────────────┘
```

### 1-2. 버전 목록 표시 항목

| 항목 | 표시 형식 | 비고 |
|------|---------|------|
| 버전 번호 | "v1.3" | `major.minor` 형식 |
| 워크플로 상태 | `WorkflowStatusBadge` | |
| 작성자 | 사용자명 | |
| 생성 시각 | "2026-04-08" + 상대 시간 | |
| 변경 사유 | 텍스트 (없으면 "-") | `change_reason` 필드 |
| 현재 버전 표시 | "[현재 버전]" 태그 | 최신 버전 강조 |
| 발행 버전 표시 | "[발행 버전]" 태그 | `PUBLISHED` 상태 강조 |

### 1-3. 화면 상태 처리

| 상태 | 처리 방식 |
|------|----------|
| 로딩 | 타임라인 SkeletonBlock 3개 |
| 버전 있음 | 타임라인 렌더링 |
| 버전 없음 | EmptyState: "버전 기록이 없습니다" |
| 오류 | ErrorState + 재시도 |

## 2. 버전 상세 조회 UX 정의서

### 2-1. 버전 상세 화면 (`/documents/[id]/versions/[vid]`)

- Task 6-4에서 정의한 문서 상세 화면과 동일 레이아웃 사용
- 상단에 **읽기 전용 배너** 표시:

```
┌──────────────────────────────────────────────────────────────┐
│  ⚠️  v1.1 버전을 보고 있습니다. 현재 버전이 아닙니다.          │
│     [현재 버전 보기]                      [이 버전으로 복원]   │
└──────────────────────────────────────────────────────────────┘
```

- 워크플로 액션 버튼 전체 비노출 (읽기 전용)
- 편집 버튼 비노출
- 구조 트리는 동일하게 표시 (네비게이션 용도)

### 2-2. 현재 버전 / 발행 버전 구분

| 구분 | 강조 방식 |
|------|----------|
| 현재 버전 (latest) | 타임라인 최상단, 진한 텍스트, "[현재 버전]" 태그 |
| 발행 버전 (PUBLISHED) | "[발행 버전]" Teal 색상 태그 |
| 현재 + 발행 일치 | 두 태그 모두 표시 |
| 과거 버전 | 회색 처리, 접근은 가능 |

## 3. 복원 UX 정책 문서

### 3-1. 복원 가능 조건

| 조건 | 정책 |
|------|------|
| 복원 가능 역할 | Author (원작성자) 또는 Approver |
| 복원 불가 상태 | 현재 버전이 IN_REVIEW / APPROVED 상태일 때 |
| 현재 버전 자체 복원 | 버튼 비노출 ("이미 현재 버전입니다") |

### 3-2. 복원 UX 흐름

```
버전 목록 또는 버전 상세 화면에서 [이 버전으로 복원] 버튼 클릭
  ↓
ConfirmDialog:
  제목: "v1.1 버전으로 복원하시겠습니까?"
  내용: "이 버전의 내용을 기반으로 새로운 Draft 버전이 생성됩니다.
        기존 버전들은 보존됩니다."
  버튼: [취소]  [복원]
  ↓ 확인 클릭
  ↓
  POST /api/v1/documents/{id}/versions/{vid}/restore
  ↓
  새 버전 생성 (DRAFT 상태)
  ↓
  성공 Toast: "v1.2 버전이 생성되었습니다"
  → /documents/{id}/edit 로 이동 (새 Draft 편집 진입)
```

### 3-3. 복원 정책 원칙

- **비파괴 복원**: 기존 모든 버전은 변경되지 않음 — 복원 = 새 버전 생성
- **Draft만 복원 기점**: 복원 결과는 항상 DRAFT 상태
- **단순화 우선**: Phase 6 MVP에서는 선택적 머지(merge) 미지원 (Phase 9+)

## 4. API 연동 포인트

| 기능 | API | Method |
|------|-----|--------|
| 버전 목록 조회 | `GET /api/v1/documents/{id}/versions` | |
| 특정 버전 상세 | `GET /api/v1/documents/{id}/versions/{vid}` | |
| 버전 복원 | `POST /api/v1/documents/{id}/versions/{vid}/restore` | |
| 워크플로 이력 | `GET .../workflow/history` | 버전 상세에서 선택 노출 |
