## 1. 검토 대상 목록 UI 설계서 (`/reviews`)

### 1-1. 화면 레이아웃

```
┌─────────────────────────────────────────────────────────────┐
│  Page Header                                                 │
│  "검토 대기"                            검토 대기: N건        │
├─────────────────────────────────────────────────────────────┤
│  Action Bar                                                  │
│  [🔍 문서 검색...]  [유형▼]  [요청자▼]   정렬: [요청일 최신순▼] │
├─────────────────────────────────────────────────────────────┤
│  검토 목록                                                   │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ 제목         유형    상태       요청자   요청일         │   │
│  ├──────────────────────────────────────────────────────┤   │
│  │ [문서 행 1]           [IN_REVIEW]                      │   │
│  │ [문서 행 2]           [IN_REVIEW]                      │   │
│  └──────────────────────────────────────────────────────┘   │
├─────────────────────────────────────────────────────────────┤
│  Pagination                                                  │
└─────────────────────────────────────────────────────────────┘
```

### 1-2. 검토 목록 표시 항목

| 컬럼 | 데이터 소스 | 비고 |
|------|-----------|------|
| 문서 제목 | `document.title` | 클릭 시 문서 상세(검토 모드)로 이동 |
| 문서 유형 | `document.document_type` | `DocumentTypeBadge` |
| 상태 | `version.workflow_status` | `WorkflowStatusBadge` (IN_REVIEW 고정) |
| 검토 요청자 | `review_action.actor` | `submit-review` 시점의 actor |
| 검토 요청일 | `review_action.created_at` | 상대 시간 |
| 버전 | `version.version_number` | "v1.2" 형식 |

### 1-3. 화면 상태 처리

| 상태 | 처리 방식 |
|------|----------|
| 로딩 | SkeletonBlock 5행 |
| 검토 대기 있음 | 목록 렌더링 |
| 검토 대기 없음 | EmptyState: "현재 검토할 문서가 없습니다" |
| 오류 | ErrorState + 재시도 |
| 권한 없음 | PermissionDenied (Viewer 접근 시) |

## 2. 승인/반려/의견 입력 UX 정의서

### 2-1. 검토 상세 진입 흐름

```
/reviews 목록 → 문서 클릭
  ↓
/documents/{id} 문서 상세 화면
  (IN_REVIEW 상태 문서 — 검토 액션 버튼 활성화)
  ↓
Document Header 우측 액션 버튼:
  Reviewer: [승인] [반려]
  Approver: [승인] [반려]
```

### 2-2. 승인 액션 UX

```
[승인] 버튼 클릭
  ↓
ActionModal: "승인하시겠습니까?"
  ┌─────────────────────────────────────────┐
  │ 검토 의견 (선택 사항)                     │
  │ ┌──────────────────────────────────────┐│
  │ │ 텍스트 입력창 (placeholder: 의견을     ││
  │ │ 남기려면 입력하세요. 선택 사항입니다.) ││
  │ └──────────────────────────────────────┘│
  │           [취소]  [승인 완료]             │
  └─────────────────────────────────────────┘
  ↓ 승인 완료 클릭
  POST .../workflow/approve  (body: {comment?})
  ↓
  성공 Toast: "문서가 승인되었습니다"
  → 상태: IN_REVIEW → APPROVED
  → 화면: 상태 배지 갱신, 액션 버튼 변경
```

### 2-3. 반려 액션 UX

```
[반려] 버튼 클릭
  ↓
ActionModal: "반려하시겠습니까?"
  ┌─────────────────────────────────────────┐
  │ 반려 사유 (필수) *                        │
  │ ┌──────────────────────────────────────┐│
  │ │ 텍스트 입력창 (placeholder: 반려 사유 ││
  │ │ 를 입력해주세요. 필수 항목입니다.)     ││
  │ └──────────────────────────────────────┘│
  │  ⚠ 반려 시 작성자에게 반환됩니다.        │
  │           [취소]  [반려]                  │
  └─────────────────────────────────────────┘
  ↓ 반려 클릭 (사유 없으면 버튼 비활성화)
  POST .../workflow/reject  (body: {reason: "...", comment?})
  ↓
  성공 Toast: "문서가 반려되었습니다"
  → 상태: IN_REVIEW → REJECTED
  → 화면: 상태 배지 갱신, 검토 액션 버튼 제거
```

### 2-4. 발행 액션 UX (Approver 전용)

```
APPROVED 상태 문서에서 [발행] 버튼 클릭
  ↓
ConfirmDialog:
  제목: "문서를 발행하시겠습니까?"
  내용: "발행 후에는 편집이 불가합니다. 계속하시겠습니까?"
  버튼: [취소]  [발행]
  ↓
POST .../workflow/publish
  ↓
성공 Toast: "문서가 발행되었습니다"
→ 상태: APPROVED → PUBLISHED
```

## 3. 상태 전이 UI 정책 문서

### 3-1. 상태별 허용 액션 및 버튼 노출 규칙

| 현재 상태 | Author | Reviewer | Approver |
|----------|--------|----------|----------|
| DRAFT | [편집] [검토 요청] | - | - |
| IN_REVIEW | - | [승인] [반려] | [승인] [반려] |
| APPROVED | - | - | [발행] |
| PUBLISHED | - | - | [보관] |
| REJECTED | [편집] | - | - |
| ARCHIVED | - | - | - |

> 본인 작성 문서에 대한 Reviewer/Approver 역할 행사는 서버 정책에 따름

### 3-2. 상태 전이 흐름도

```
DRAFT ─[검토 요청]──→ IN_REVIEW ─[승인]──→ APPROVED ─[발행]──→ PUBLISHED
                          │                                          │
                       [반려]                                     [보관]
                          ↓                                          ↓
                       REJECTED ─[재편집]──→ DRAFT              ARCHIVED
                          
DRAFT ←─[Draft 복원]─────────────────────────────── (언제든 가능)
```

### 3-3. 버튼 노출 vs 비활성화 기준

| 상황 | 처리 방식 |
|------|----------|
| 역할 자체가 없는 액션 | 숨김 (Hidden) |
| 역할은 있지만 상태가 맞지 않음 | 비활성화 + tooltip "현재 상태에서는 불가합니다" |
| 역할 + 상태 조건 모두 충족 | 활성화 노출 |
| 서버 검증 실패 (409/403) | Toast 오류 + 상태 재조회 |

## 4. 워크플로 이력 표시 초안

### 4-1. 이력 표시 위치

- 문서 상세 화면 → 메타데이터 패널 하단 "워크플로 이력" 섹션 (접기/펼치기 가능)
- 또는 버전 상세 하단 별도 섹션

### 4-2. 이력 Timeline 구성

```
워크플로 이력
──────────────────────────────────────────────
  ● 승인  —  2026-04-08 14:30
    홍길동 (Approver)
    "내용 확인 완료. 승인합니다."

  ● 검토 요청  —  2026-04-07 10:00
    김철수 (Author)
    "1분기 검토 요청"

  ● 반려  —  2026-04-05 16:20
    이영희 (Reviewer)
    "3.2절 수정 필요"

  ● 검토 요청  —  2026-04-03 09:00
    김철수 (Author)
    —
──────────────────────────────────────────────
```

### 4-3. 이력 항목 표시 항목

| 항목 | 표시 형식 |
|------|---------|
| 액션 유형 | 아이콘 + 텍스트 (승인, 반려, 검토 요청, 발행 등) |
| 수행 시각 | "2026-04-08 14:30" |
| 수행자 | "홍길동 (Approver)" — actor_role 스냅샷 사용 |
| 코멘트/사유 | 텍스트 (없으면 "—") |

## 5. API 연동 포인트

| 기능 | API | Method |
|------|-----|--------|
| 검토 대기 목록 | `GET /api/v1/documents?status=IN_REVIEW` | |
| 승인 | `POST .../workflow/approve` | `{comment?}` |
| 반려 | `POST .../workflow/reject` | `{reason, comment?}` |
| 발행 | `POST .../workflow/publish` | |
| 워크플로 이력 | `GET .../workflow/history` | |
| 검토 액션 목록 | `GET .../workflow/review-actions` | |
