

# Task 6-8. 권한 및 상태 기반 UI 제어 체계 설계

## 1. 작업 목적

사용자 UI 전반에서 문서의 **현재 상태**와 사용자의 **역할/권한**에 따라 화면 접근과 액션 노출을 안전하게 제어할 수 있도록 UI 제어 체계를 설계한다.

이 작업의 핵심 목표는 다음과 같다.

- 사용자에게 허용된 화면과 액션만 노출되도록 한다
- 문서 상태에 따라 가능한 작업만 수행 가능하도록 한다
- 권한 부족, 상태 불일치, 정책 위반 상황을 UI에서 일관되게 처리한다

---

## 2. 작업 범위

### 포함 범위

- 역할 기반 화면 접근 정책 정의
- 상태 기반 액션 노출 정책 정의
- 숨김 / 비활성화 / 안내 메시지 기준 정의
- UI 가드 패턴 및 예외 처리 UX 정의

### 제외 범위

- 서버 권한 검증 로직 구현
- ACL 엔진 구현
- 감사 로그 수집 구현

## 4. 주요 설계 대상

### 4-1. 역할 기반 접근 정책

각 사용자 역할에 따라 접근 가능한 화면 범위를 정의한다.

#### 대상 역할 예시

- 열람자(Viewer)
- 작성자(Author)
- 검토자(Reviewer)
- 승인자(Approver)

#### 설계 요구사항

- 어떤 역할이 어떤 화면에 접근 가능한지 매트릭스로 정리
- 동일 화면에서도 역할에 따라 읽기 전용 / 편집 가능 / 승인 가능 상태를 구분
- 역할이 복수일 경우 우선 적용 규칙 정의

---

### 4-2. 상태 기반 액션 제어 정책

문서 상태에 따라 허용되는 UI 액션을 정의한다.

#### 상태 예시

- Draft
- Review
- Approved
- Published
- Rejected / Returned

#### 액션 예시

- Edit
- Save
- Request Review
- Approve
- Reject / Return
- Restore Version
- Publish

#### 설계 요구사항

- 상태별 허용 액션 표 작성
- 상태 전이에 따라 버튼 노출 기준 정의
- 현재 상태에서 불가능한 액션은 UI에서 명확히 처리

---

### 4-3. 역할 + 상태 결합 제어 매트릭스

실제 UI 제어는 역할만으로도, 상태만으로도 충분하지 않으므로 **역할 + 상태 조합** 기준의 정책을 정의한다.

#### 예시

- 작성자 + Draft → 편집 가능
- 작성자 + Review → 편집 불가
- 검토자 + Review → 승인/반려 가능
- 열람자 + Published → 읽기만 가능

#### 산출 방향

- 화면별 액션 매트릭스 작성
- 최소한 아래 화면을 포함
  - 문서 목록
  - 문서 상세
  - 문서 편집
  - 버전 관리
  - 승인 / 검토 화면

---

### 4-4. UI 처리 방식 기준

허용되지 않은 액션에 대해 UI가 어떤 방식으로 반응할지 기준을 정의한다.

#### 처리 방식

- 숨김(Hidden)
- 비활성화(Disabled)
- 노출 + 안내 메시지

#### 기준 예시

- 권한 자체가 없는 기능 → 숨김 우선
- 상태상 현재 수행 불가한 기능 → 비활성화 + 이유 안내 가능
- 정책 위반 가능성이 큰 액션 → 명시적 차단 메시지

---

### 4-5. API 실패 및 서버 판정 불일치 처리

UI에서 허용된 것처럼 보여도 서버 최종 검증에서 거부될 수 있으므로, 이에 대한 UX를 정의한다.

#### 대표 상황

- 클라이언트 상태가 오래되어 실제 권한이 바뀐 경우
- 다른 사용자가 먼저 상태를 변경한 경우
- 서버 정책상 차단된 경우

#### 요구사항

- 실패 메시지를 명확히 제공
- 가능하면 최신 상태 재조회 유도
- 낙관적 UI를 사용하더라도 실패 복구 UX 정의

---

### 4-6. 공통 Guard 패턴 정의

화면과 컴포넌트에서 재사용 가능한 UI Guard 패턴을 정의한다.

#### 예시

- Route Guard
- Page Action Guard
- Button Guard
- Section Guard

#### 설계 요구사항

- 페이지 진입 자체 제한이 필요한 경우와
- 페이지는 보이되 일부 액션만 제한하는 경우를 구분
- Guard 기준이 분산되지 않도록 공통화 방향 제시

---

### 4-7. 상태별 예외 UX

#### Permission Denied

- 접근 권한 없음 안내
- 필요한 경우 이전 화면 이동 또는 문서 목록으로 복귀 유도

#### Invalid State

- 현재 상태에서는 수행 불가 안내
- 최신 상태 확인 유도

#### Stale Data

- 문서 상태가 변경되었음을 알리고 새로고침 유도
