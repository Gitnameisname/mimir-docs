

# Task 6-9. 공통 피드백 및 상호작용 UX 정리

## 1. 작업 목적

User UI 전반에서 발생하는 모든 사용자 상호작용에 대해 **일관된 피드백 UX 체계**를 정의한다.

이 작업의 핵심 목표는 다음과 같다.

- 사용자가 시스템 상태를 항상 명확하게 이해할 수 있도록 한다
- 성공 / 실패 / 경고 / 진행 상태를 일관된 방식으로 표현한다
- 사용자 실수를 방지하고 안전한 인터랙션을 유도한다

---

## 2. 작업 범위

### 포함 범위

- 성공 / 실패 / 경고 메시지 패턴 정의
- 비동기 처리 상태 UX 정의 (Saving, Loading 등)
- Confirm / Cancel 상호작용 UX 정의
- 사용자 실수 방지 UX 정의

### 제외 범위

- 실제 메시지 로깅 시스템
- 서버 에러 처리 로직 구현

## 4. 주요 설계 대상

### 4-1. 성공 피드백 (Success Feedback)

사용자의 액션이 정상적으로 완료되었음을 알린다.

#### 대상 액션

- 저장 완료
- 승인 완료
- 문서 생성 완료

#### UI 방식

- Toast 메시지
- 짧고 명확한 메시지

#### 요구사항

- 사용자 흐름을 방해하지 않도록 비차단 방식 사용

---

### 4-2. 실패 피드백 (Error Feedback)

사용자의 요청이 실패했을 때 원인을 명확히 전달한다.

#### 대상 상황

- 저장 실패
- 권한 부족
- 상태 불일치

#### UI 방식

- Toast 또는 Inline Error

#### 요구사항

- 가능한 경우 해결 방법 안내
- 재시도 액션 제공

---

### 4-3. 경고 피드백 (Warning)

사용자가 위험한 행동을 수행하려 할 때 사전 경고를 제공한다.

#### 대상 상황

- 삭제
- 복원
- 승인/반려

#### UI 방식

- Confirm Dialog

#### 요구사항

- 명확한 결과 설명
- Cancel 기본 선택

---

### 4-4. 진행 상태 표시 (Loading / Processing)

비동기 작업 중 상태를 사용자에게 전달한다.

#### 대상 상황

- 데이터 로딩
- 저장 중
- 승인 처리 중

#### UI 방식

- Spinner
- Skeleton
- 상태 메시지

---

### 4-5. Confirm / Cancel UX

중요한 액션 수행 전 사용자 확인을 요구한다.

#### 대상 액션

- 삭제
- 복원
- 승인/반려

#### 요구사항

- 명확한 액션 설명
- 취소 옵션 항상 제공
- 기본 포커스는 안전한 선택

---

### 4-6. Unsaved Changes 처리

편집 중 저장되지 않은 변경사항이 있을 경우 사용자에게 경고한다.

#### 대상 상황

- 페이지 이동
- 브라우저 종료

#### UX

- "저장되지 않은 변경사항이 있습니다" 경고
- 저장 / 무시 선택

---

### 4-7. 권한 및 상태 오류 UX

#### Permission Denied

- 권한 없음 안내 메시지
- 가능한 경우 요청 방법 안내

#### Invalid State

- 현재 상태에서는 수행 불가 안내

#### Stale State

- 최신 상태로 갱신 필요 안내

---

### 4-8. 메시지 톤 & 스타일 가이드

#### 원칙

- 간결하고 명확하게
- 기술 용어 최소화
- 사용자 행동 중심 표현

#### 예시

- "저장이 완료되었습니다"
- "권한이 없습니다"
- "현재 상태에서는 수행할 수 없습니다"
