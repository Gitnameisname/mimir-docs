

# Task 6-5. 문서 작성/편집 UI 설계

## 1. 작업 목적

사용자가 문서를 생성하고 수정할 수 있는 **문서 작성/편집 UI(Editor)**를 설계한다.

이 작업은 Phase 6 전체에서 가장 중요한 UX 영역이며, 다음을 목표로 한다.

- 트리 구조 기반 문서를 직관적으로 편집 가능하게 한다
- Draft 중심의 안정적인 편집 경험 제공
- 저장/충돌/상태 관리까지 포함한 실제 업무용 편집 환경 설계

---

## 2. 작업 범위

### 포함 범위

- 문서 편집 화면 구조 설계
- Node 기반 구조 편집 UX 설계
- 저장 및 임시 저장 정책 정의
- 편집 상태 관리 UX 정의

### 제외 범위

- 실제 에디터 엔진 구현
- 협업 편집 (실시간 동시 편집)

## 4. 주요 설계 대상

### 4-1. 편집 화면 레이아웃

#### 기본 구조

- Header (문서 제목 + 상태)
- Edit Toolbar
- Main Editor 영역
- 구조 트리 패널 (선택)

#### 요구사항

- 읽기 모드와 편집 모드의 명확한 분리
- 편집 중 상태 시각화
- 집중 가능한 UI

---

### 4-2. Node 기반 편집 UX

문서는 Node 트리 구조로 구성된다.

#### 주요 기능

- Node 추가 (Section, Paragraph 등)
- Node 삭제
- Node 순서 변경 (Drag & Drop 또는 버튼)
- Node 타입 변경

#### UX 요구사항

- 구조가 시각적으로 명확히 드러나야 함
- 깊이(depth) 표현 가능

---

### 4-3. 편집 Toolbar

#### 주요 기능

- 텍스트 스타일 (Bold, Italic 등)
- Heading 설정
- 리스트
- 링크

#### 확장 고려

- DocumentType별 커스텀 Toolbar 가능

---

### 4-4. 저장 정책

#### 저장 방식

- 명시적 저장 (Save 버튼)
- 자동 저장 (선택)

#### 상태 정의

- Saved
- Unsaved Changes
- Saving...

#### UX 요구사항

- 사용자가 현재 저장 상태를 항상 인지 가능해야 함

---

### 4-5. Draft 상태 관리

편집은 Draft 상태에서만 가능하다.

#### 규칙

- Published 문서는 직접 편집 불가
- 편집 시 새로운 Draft 생성 또는 Draft 유지

---

### 4-6. 충돌 및 예외 처리

#### 고려 사항

- 동일 문서 동시 편집
- 오래된 Draft 수정

#### UX 방향

- 충돌 발생 시 사용자에게 명확히 안내
- overwrite 또는 merge 선택 (후속 확장)

---

### 4-7. 상태별 UI 처리

#### Loading

- 문서 로딩 상태

#### Error

- 저장 실패 메시지

#### Warning

- unsaved changes 경고 (페이지 이탈 시)
