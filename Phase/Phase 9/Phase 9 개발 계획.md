# Phase 9 개발 계획
## 변경 비교 및 이력 가시화 기능 구축

---

## 1. 단계 개요

### 단계명
Phase 9. 변경 비교 및 이력 가시화 기능 구축

### 목적
문서의 버전 간 변경 내용을 구조 단위로 추적하고, 운영자와 사용자가 무엇이 어떻게 바뀌었는지 명확히 파악할 수 있는 비교/이력 시스템을 구축한다.  
이 단계의 목표는 단순한 텍스트 diff를 제공하는 것이 아니라, **문서의 구조적 변경(노드 추가/삭제/이동/수정)을 추적하고 이를 사람이 읽기 쉬운 형태로 가시화**하는 것이다.

### 기대 결과
- 두 버전 간 구조 단위 diff 제공 (노드 추가/삭제/이동/수정)
- 노드 내 텍스트 수준 인라인 diff 표시
- 변경 요약 (어떤 섹션이 얼마나 변경되었는지)
- Phase 6 User UI에 변경 비교 화면 통합
- 워크플로 검토 단계에서 변경 내용 확인 UX 강화
- 감사 로그와 연계한 변경 이력 추적 강화

---

## 2. Phase 9의 역할

Phase 9는 이미 구축된 다음 요소들을 **변경 가시화 관점으로 연결하는 단계**다.

- Phase 1에서 정의한 Document / Version / Node 트리 구조
- Phase 4에서 구축한 버전 생성 및 관리 시스템
- Phase 5에서 구축한 워크플로 (검토자가 변경 내용을 확인해야 함)
- Phase 6에서 구축한 User UI (diff 화면 통합)
- Phase 2에서 정의한 감사 로그 (변경 사유 이력 연계)

즉, Phase 9는 데이터나 기능을 새로 만드는 단계가 아니라,  
**이미 쌓인 버전 데이터를 "무엇이 바뀌었는가"라는 질문에 답할 수 있도록 분석하고 표현하는 단계**다.

## 3. 개발 목표

### 3.1 구조 단위 변경 추적
- Node 트리 구조의 변경을 추적: 노드 추가 / 삭제 / 이동 / 수정
- 노드 ID 기반 추적으로 이동된 노드도 식별 가능
- 재귀적 트리 diff 알고리즘 설계 및 구현

### 3.2 텍스트 수준 인라인 diff
- 수정된 노드 내에서 어떤 텍스트가 추가/삭제되었는지 인라인 표시
- 단락, 문장 단위의 세밀한 변경 추적
- 마크다운/구조화 텍스트에 적합한 diff 처리

### 3.3 변경 요약 생성
- diff 결과를 사람이 읽기 쉬운 자연어 요약으로 표현
- "섹션 3개 수정, 섹션 1개 추가, 섹션 2개 삭제" 수준의 요약
- 워크플로 검토 단계에서 검토자가 변경 범위를 빠르게 파악 가능

### 3.4 비교 API 플랫폼화
- 버전 비교 전용 REST API 엔드포인트
- 외부 시스템(감사 도구, CI/CD 등)에서도 활용 가능한 API 구조
- DiffResult 응답 스키마 표준화

### 3.5 User UI 변경 비교 화면 통합
- 문서 상세 화면의 버전 선택 후 diff 보기
- 워크플로 검토 화면에서 직전 버전 대비 변경 내용 자동 표시
- Side-by-side 및 Unified diff 뷰 모두 지원

### 3.6 이력 가시화
- 버전별 변경 이력을 타임라인 형태로 표시
- 각 버전에 변경 요약, 변경 사유, 작성자 정보 포함
- 특정 버전 클릭 시 해당 버전 내용 조회 및 이전 버전과 비교 가능

## 4. 핵심 개발 범위

### 4.1 트리 Diff 알고리즘 설계 및 구현
- 두 Version의 Node 트리를 비교하는 알고리즘
- 노드 ID 기반 매칭: 동일 ID면 같은 노드로 판단
- 변경 유형 분류:
  - ADDED: 이전 버전에 없던 노드가 추가됨
  - DELETED: 이전 버전에 있던 노드가 삭제됨
  - MODIFIED: 동일 ID 노드의 내용/메타데이터가 변경됨
  - MOVED: 동일 ID 노드의 위치(순서/부모)가 변경됨
  - UNCHANGED: 변경 없음

### 4.2 텍스트 인라인 Diff 구현
- MODIFIED 노드에 대해 이전/이후 텍스트의 세밀한 diff 계산
- Myers diff 알고리즘 또는 동등한 LCS 기반 알고리즘 사용
- diff 결과를 `{type: 'added'|'deleted'|'unchanged', text: '...'}` 토큰 배열로 표현
- 마크다운 구조를 보존하면서 diff 처리

### 4.3 변경 비교 API
- `GET /documents/{id}/versions/{v1}/diff/{v2}` — 두 버전 간 diff
- `GET /documents/{id}/versions/{v}/diff` — 직전 버전 대비 diff (기본)
- DiffResult 응답 스키마 표준화
- diff 결과 캐싱 전략

### 4.4 변경 요약 생성
- DiffResult를 입력으로 받아 변경 요약 텍스트 생성
- 변경 규모 계산 (추가/삭제/수정 노드 수, 변경된 텍스트 양)
- 중요 변경 섹션 식별 (최상위 헤딩 기준)

### 4.5 변경 비교 UI (User UI 통합)
- 버전 선택 diff 뷰어 (문서 상세 화면)
- Side-by-side 뷰: 이전/이후 버전을 좌우로 나란히 표시
- Unified 뷰: 변경 사항을 하나의 뷰에 통합 표시
- 변경 유형별 색상 표시 (추가: 녹색, 삭제: 빨간색, 수정: 노란색, 이동: 파란색)
- 변경된 항목만 보기 토글

### 4.6 워크플로 검토 화면 통합
- 검토 요청된 문서에서 직전 버전 대비 변경 내용 자동 표시
- 변경 요약을 검토 화면 상단에 표시
- 검토자가 변경 내용을 확인하면서 승인/반려 결정 가능

### 4.7 버전 이력 타임라인
- 문서의 전체 버전 이력을 타임라인 형태로 표시
- 각 버전 항목: 버전 번호, 작성자, 변경 사유, 변경 요약, 날짜
- 버전 항목 클릭 시 해당 버전 조회 및 인접 버전과 비교
- 주요 이정표 버전 표시 (Published 버전 등)

### 4.8 diff 캐싱 및 성능 최적화
- 버전은 불변이므로 동일 버전 쌍의 diff 결과는 영속 캐싱 가능
- 캐시 키: `diff:{document_id}:{version_id_a}:{version_id_b}`
- 대용량 문서 diff 비동기 사전 계산 고려

## 6. 구현 원칙

### 6.1 구조 우선 diff
- 텍스트 전체를 문자열로 비교하는 방식이 아니라, Node 트리 구조를 기준으로 비교한다
- 동일 ID의 노드는 같은 노드로 추적하여 이동도 감지한다

### 6.2 버전 불변성 활용
- 이미 저장된 버전은 변경되지 않으므로, diff 결과는 영속 캐싱이 가능하다
- 동일 버전 쌍에 대한 diff는 항상 같은 결과를 반환한다

### 6.3 검토자 중심 UX
- diff 뷰어는 단순한 개발자 도구가 아니라, 일반 검토자가 변경 내용을 빠르게 파악하기 위한 도구다
- 변경 요약을 가장 먼저 표시하고, 상세 diff는 필요할 때 펼쳐볼 수 있게 한다

### 6.4 API-first diff
- diff 기능은 API로 먼저 정의되고 UI는 API 클라이언트다
- 외부 감사 도구, CI/CD 파이프라인에서도 버전 비교 API를 활용할 수 있어야 한다

### 6.5 권한 기반 diff 접근 제어
- diff 요청 시에도 두 버전 모두에 대한 접근 권한을 확인한다
- 권한 없는 버전의 내용이 diff 응답에 포함되어서는 안 된다

## 7. 기술 선택 방향

### 7.1 트리 Diff 알고리즘
- **노드 ID 기반 매칭** 우선 사용
  - Node에 고유 ID가 있으므로 Myers 알고리즘보다 단순하고 정확한 매칭 가능
  - ID가 없는 노드(레거시 데이터)의 처리 방침 별도 정의 필요

### 7.2 텍스트 인라인 Diff
- **diff-match-patch** 라이브러리 또는 동등한 구현 사용
  - Google의 diff-match-patch: Python/JavaScript 모두 지원, LCS 기반
  - 단어 단위 diff로 가독성 확보 (문자 단위는 한국어에서 가독성 저하)

### 7.3 diff 결과 저장
- **온디맨드 계산 + 캐싱** 방식
  - DB에 별도 저장하지 않고 요청 시 계산
  - 결과는 Redis 또는 애플리케이션 메모리에 캐싱 (버전 불변성 활용)
  - 대용량 문서는 비동기 사전 계산 고려

## 12. 최종 정리

Phase 9는 diff 기능을 만드는 단계이지만, 본질적으로는  
**"문서가 어떻게 변화해 왔는가"를 시스템이 추적하고 설명할 수 있게 하는 단계**다.

이 단계에서 중요한 것은 diff 뷰어를 예쁘게 만드는 것이 아니라,

- 구조적 변경(노드 추가/삭제/이동)을 정확하게 추적하는가
- 검토자가 변경 범위를 빠르게 파악할 수 있는가
- diff API가 외부 감사 도구에서도 활용 가능한가
- 버전 이력이 투명하게 가시화되는가

를 구조적으로 보장하는 것이다.

Phase 9의 산출물은 단순한 diff 뷰어가 아니라  
**문서 변경의 추적 가능성(Traceability)을 보장하는 기반**이어야 한다.
