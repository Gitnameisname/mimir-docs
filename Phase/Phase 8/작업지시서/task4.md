# Task 8-4. 권한 기반 검색 필터링 설계

## 1. 작업 목적

검색 결과가 요청자의 권한 범위를 절대로 벗어나지 않도록 필터링 구조를 설계한다.

이 작업의 목표는 다음과 같다.

- Phase 2의 ACL 모델을 검색 쿼리에 반영하는 방식 정의
- 문서 상태별 노출 정책 명확화
- 노드 단위 검색에서의 권한 상속 방식 정의
- 권한 필터링이 검색 성능에 미치는 영향 분석 및 최적화 방향 제시

## 4. 주요 설계 대상

### 4-1. 권한 기반 검색의 핵심 원칙

> 검색 결과는 해당 문서를 직접 조회했을 때 접근 가능한 경우에만 반환된다.

- 검색을 통해 접근 권한이 없는 문서의 제목, 내용, 메타데이터가 노출되어서는 안 된다
- 스니펫 포함 응답도 동일하게 권한 필터링 적용
- 클라이언트 측에서 권한 필터링을 우회하는 방법이 없어야 한다

---

### 4-2. 문서 접근 권한 필터링

#### ACL 적용 방식

Phase 2의 권한 모델을 기반으로, 검색 쿼리에 권한 조건을 추가한다:

```
검색 조건:
  FTS 매칭 조건
  AND 문서 접근 권한 조건 (ACL)
  AND 문서 상태 노출 정책 조건
```

#### 권한 확인 방식 옵션

| 방식 | 설명 | 장단점 |
|------|------|--------|
| ACL JOIN | 검색 쿼리에 직접 권한 테이블 JOIN | 정확하지만 대용량 시 성능 영향 |
| 사전 허용 문서 목록 | 사용자별 접근 가능 문서 ID 목록을 IN 조건으로 사용 | 목록 크기에 따라 성능 가변 |
| 역할 기반 간소화 | 역할(Role)만으로 접근 가능한 문서 유형 필터 | 단순하지만 세밀한 제어 제한 |

→ **권장: ACL JOIN 방식** (정확성 우선, 성능 최적화는 별도)

---

### 4-3. 문서 상태별 노출 정책

문서 상태에 따라 검색 결과에 노출 가능한 역할을 정의한다:

| 문서 상태 | 노출 가능한 역할/사용자 |
|-----------|----------------------|
| PUBLISHED | 해당 문서 열람 권한이 있는 모든 사용자 |
| DRAFT | 문서 작성자, 동일 조직 내 편집 권한자, 관리자 |
| IN_REVIEW | 작성자, 검토자로 지정된 사용자, 관리자 |
| APPROVED | 작성자, 승인자, 관리자 |
| ARCHIVED | 관리자 전용 (일반 검색에서 기본 제외) |
| DELETED | 검색 결과에서 완전 제외 |

#### 기본 검색 동작

- 일반 사용자 검색 기본값: PUBLISHED 상태 문서만
- 명시적 `status` 파라미터로 DRAFT 등 포함 가능 (권한 범위 내에서만)

---

### 4-4. 노드 단위 검색의 권한 상속

노드 검색 결과에 대한 권한 검증은 부모 문서의 권한을 상속한다:

#### 상속 원칙

```
Node 접근 가능 여부 = 부모 Document 접근 가능 여부
```

- 노드 자체에 별도 권한 설정 없이 Document ACL을 그대로 적용
- 노드 검색 쿼리에 부모 Document 권한 조건을 JOIN하여 필터링
- 삭제된 문서의 노드는 검색 결과에서 자동 제외

#### 구현 방향

```
Node 검색 결과 = FTS 매칭 노드
  JOIN versions ON nodes.version_id = versions.id
  JOIN documents ON versions.document_id = documents.id
  AND documents 접근 권한 조건
```

---

### 4-5. 조직 범위 기반 필터링

조직(Organization) 기반 권한이 있는 경우:

- 사용자의 소속 조직에 귀속된 문서만 검색 결과에 포함
- 조직 범위를 벗어난 문서는 명시적 공유 설정이 없는 한 제외
- 크로스 조직 검색: 별도 권한 있는 경우에만 허용

---

### 4-6. 성능 영향 분석 및 완화 방안

#### 권한 필터링 성능 영향 요인

| 요인 | 영향 | 완화 방안 |
|------|------|----------|
| ACL 테이블 크기 | JOIN 비용 증가 | ACL 테이블 인덱스 최적화 |
| 사용자별 권한 복잡도 | 조건 평가 비용 | 역할 기반 그룹화로 단순화 |
| 대규모 문서 수 | 필터 후 결과 수 감소로 재쿼리 필요 | FTS 먼저 → 권한 필터 순서 최적화 |

#### 단기 완화 방안 (Phase 8)

- 검색 쿼리 실행 계획(EXPLAIN ANALYZE) 분석 및 인덱스 튜닝
- ACL 테이블에 `(document_id, user_id)` 복합 인덱스 확인
- 역할이 단순한 경우 역할 기반 간소화 필터 적용 가능

#### 장기 고려 방안 (Phase 이후)

- 사용자별 접근 가능 문서 ID 캐시 레이어 (Redis)
- 권한 변경 시 캐시 무효화 전략
- 전용 검색 엔진 도입 시 권한 필터 사전 처리

---

### 4-7. 테스트 시나리오

권한 기반 검색 필터링의 정확성을 검증하기 위한 시나리오:

| 시나리오 | 기대 결과 |
|---------|----------|
| 열람 권한 없는 문서가 검색어 매칭 시 | 결과에 미포함 |
| 자신이 작성한 DRAFT 문서 검색 | 결과에 포함 |
| 타인의 DRAFT 문서 검색 (열람 권한 없음) | 결과에 미포함 |
| 조직 범위 내 PUBLISHED 문서 검색 | 결과에 포함 |
| 다른 조직 문서 검색 (공유 없음) | 결과에 미포함 |
| 관리자가 ARCHIVED 문서 검색 | 결과에 포함 |
| 스니펫에 권한 없는 내용 포함 여부 | 스니펫 생성 전 권한 검증 필수 |
| 권한 변경 후 즉시 검색 결과 반영 여부 | 즉시 반영 |
