# Phase 8 개발 계획
## 검색 및 탐색 시스템 구축

---

## 1. 단계 개요

### 단계명
Phase 8. 검색 및 탐색 시스템 구축

### 목적
플랫폼에 축적된 문서를 빠르고 정확하게 찾을 수 있는 검색 인프라를 구축한다.  
이 단계의 목표는 단순한 키워드 검색 화면을 만드는 것이 아니라, **권한 기반의 전문 검색(Full-Text Search) 엔진과 검색 API를 플랫폼의 공식 인터페이스로 확립**하는 것이다.

Phase 8의 검색 시스템은 이후 Phase 10의 벡터 검색, Phase 11의 RAG 질의응답과 연계될 수 있도록 확장 가능한 구조로 설계되어야 한다.

### 기대 결과
- 제목/본문/메타데이터 전문 검색 기능 제공
- 노드(섹션) 단위 검색으로 문서 내 특정 위치까지 탐색 가능
- 권한 기반 필터링으로 열람 권한이 있는 문서만 검색 결과에 노출
- Phase 6 User UI에 검색 기능 통합
- Phase 10 벡터 검색으로 자연스럽게 확장 가능한 검색 레이어 확보

---

## 2. Phase 8의 역할

Phase 8은 이미 구축된 다음 요소들을 **검색 가능한 형태로 연결하는 단계**다.

- Phase 1에서 정의한 Document / Version / Node 구조
- Phase 2에서 정의한 권한/역할 모델
- Phase 3에서 구축한 플랫폼 API
- Phase 4~5에서 구축한 문서/버전/워크플로 데이터
- Phase 6에서 구축한 User UI (검색 화면 통합)
- Phase 7에서 구축한 Admin UI (인덱싱 상태 관리)

즉, Phase 8은 데이터를 새로 만드는 단계가 아니라,  
**기존에 쌓인 문서 데이터를 사용자가 탐색할 수 있도록 검색 인프라를 구축하는 단계**다.

## 3. 개발 목표

### 3.1 전문 검색 인프라 구축
- PostgreSQL Full-Text Search (FTS) 기반 검색 엔진 구성
- 한국어 및 영어 형태소 분석 지원
- tsvector 인덱스 구성 및 자동 갱신 파이프라인

### 3.2 다차원 검색 대상 지원
- 문서(Document) 단위 검색: 제목, 메타데이터
- 버전(Version) 단위 검색: 특정 버전의 본문 내용
- 노드(Node) 단위 검색: 문서 내 특정 섹션/단락까지 탐색

### 3.3 권한 기반 검색 필터링
- 검색 결과는 요청자의 권한 범위 내 문서만 노출
- Phase 2의 ACL 모델을 검색 쿼리에 반영
- 문서 상태(Published / Draft 등) 기반 노출 범위 제어

### 3.4 검색 API 플랫폼화
- 검색 전용 REST API 엔드포인트 설계 및 구현
- 필터/정렬/페이지네이션/스니펫 기능 제공
- 외부 시스템 및 AI 에이전트가 활용 가능한 API 구조

### 3.5 User UI 검색 경험 통합
- Phase 6 UI에 검색 기능 통합
- 검색 결과 화면, 스니펫 표시, 결과 내 탐색 기능
- 노드 단위 검색 결과에서 문서 내 위치로 직접 이동 가능

### 3.6 Phase 10 벡터 검색 연계 대비
- 검색 레이어를 추상화하여 FTS → 하이브리드(FTS + 벡터) 전환 가능한 구조
- 검색 인터페이스(API 계약)는 Phase 10 이후에도 동일하게 유지

## 4. 핵심 개발 범위

### 4.1 검색 인덱스 설계 및 구축
- Document / Version / Node 테이블의 검색 대상 컬럼 정의
- tsvector 컬럼 추가 및 트리거 기반 자동 갱신
- 검색 가중치 설계 (제목 > 메타데이터 > 본문 > 노드)
- 언어별 검색 설정 (한국어 simple / 영어 english 딕셔너리)
- GIN 인덱스 생성 및 성능 검증

### 4.2 검색 API 설계 및 구현
- `GET /search` — 통합 검색 엔드포인트
- `GET /search/documents` — 문서 단위 검색
- `GET /search/nodes` — 노드 단위 검색
- 공통 파라미터 설계:
  - `q` (검색어)
  - `type` (DocumentType 필터)
  - `status` (문서 상태 필터)
  - `from`, `to` (날짜 범위)
  - `sort` (relevance / created_at / updated_at)
  - `page`, `limit`
- 검색 결과 응답 형식 표준화 (SearchResult 스키마)
- 스니펫(ts_headline) 포함 응답

### 4.3 권한 기반 검색 필터링
- 검색 쿼리에 ACL JOIN 적용
- 요청자의 역할/권한에 따라 열람 가능한 문서만 결과에 포함
- 문서 상태별 노출 정책:
  - PUBLISHED: 열람 권한자 모두 노출
  - DRAFT: 작성자 및 특정 역할만 노출
  - ARCHIVED: 별도 권한 필요
- 노드 단위 검색에서도 부모 문서 권한 상속 검증

### 4.4 DocumentType 기반 검색 차별화
- DocumentType별 검색 가중치 커스터마이징
- DocumentType별 메타데이터 필드 검색 지원
- 타입별 검색 결과 표시 방식 정의 (렌더링 힌트)

### 4.5 검색 결과 스니펫 처리
- PostgreSQL `ts_headline()` 활용 컨텍스트 스니펫 생성
- 매칭 키워드 하이라이팅
- 스니펫 길이 설정 (요약/상세 모드)
- 노드 단위 스니펫으로 문서 내 정확한 위치 안내

### 4.6 검색 성능 및 품질 기준 정의
- 검색 응답 시간 목표 (예: p95 < 500ms)
- 검색 결과 정렬 품질 기준 정의 (관련도 계산 방식)
- 대량 문서 환경에서의 인덱스 성능 검증 기준

### 4.7 User UI 검색 기능 통합
- 글로벌 검색 바 (헤더 상단, 단축키 지원)
- 검색 결과 페이지 (`/search?q=...`)
- 문서 목록에서의 인라인 필터/검색
- 검색 결과에서 노드 위치로 직접 이동 기능
- 최근 검색어 / 추천 검색어 (선택)

### 4.8 Admin 검색 인덱스 관리
- 인덱스 현황 조회 (Phase 7 대시보드 확장)
- 수동 재인덱싱 트리거
- 인덱스 오류 및 누락 문서 확인

## 6. 구현 원칙

### 6.1 권한 우선 (Permission-First Search)
- 검색 결과는 반드시 요청자의 권한 범위 내에서만 반환된다
- 권한 필터링은 API 레벨에서 강제 적용되며 클라이언트가 우회할 수 없다
- 성능을 이유로 권한 검증을 생략하는 구현을 허용하지 않는다

### 6.2 API-First 검색
- 검색 기능은 API로 먼저 정의되고 UI는 API의 클라이언트다
- 검색 API는 UI뿐 아니라 외부 시스템, AI 에이전트가 사용할 수 있도록 설계한다
- 검색 파라미터와 응답 스키마는 안정적으로 유지한다

### 6.3 노드 단위 탐색 지원
- 문서 전체 매칭보다 정확한 노드 단위 위치까지 안내하는 것을 목표로 한다
- 검색 결과에서 클릭 한 번으로 해당 섹션으로 이동할 수 있어야 한다

### 6.4 확장 가능한 검색 레이어
- 현재는 PostgreSQL FTS를 사용하지만, 검색 서비스 레이어는 엔진에 종속되지 않아야 한다
- Phase 10 벡터 검색 도입 시 API 계약을 변경하지 않고 내부 구현만 교체 가능해야 한다

### 6.5 DocumentType 인식 검색
- 검색 시스템은 DocumentType을 인식하며, 타입별 검색 가중치와 메타데이터 구조를 반영한다
- 타입별 검색 설정은 하드코딩하지 않고 DocumentType 설정 기반으로 처리한다

## 7. 기술 선택 방향

### 7.1 검색 엔진: PostgreSQL Full-Text Search

Phase 8에서는 별도 검색 엔진(Elasticsearch, OpenSearch 등) 없이 **PostgreSQL FTS**를 기반으로 구현한다.

**선택 근거:**
- 기존 PostgreSQL 인프라 활용으로 운영 복잡도 최소화
- tsvector/tsquery/ts_headline로 전문 검색 기본 요구사항 충족
- ACL JOIN이 동일 DB 내에서 처리되어 권한 필터링 구현이 단순
- Phase 10에서 pgvector 확장과 함께 하이브리드 검색으로 자연스럽게 확장 가능

**제약 및 대안 고려:**
- 대규모 문서(수백만 건 이상)에서의 성능 한계 인지
- 필요 시 Phase 이후 별도 검색 엔진 도입을 위한 추상화 구조 유지

### 7.2 한국어 검색
- PostgreSQL `simple` 딕셔너리 기반 기본 지원
- 형태소 분석이 필요한 경우 `pg_bigm` 확장 검토

### 7.3 검색 인덱스 갱신 전략
- 문서 저장/수정/상태 변경 시 트리거 또는 이벤트 기반 tsvector 갱신
- 초기 마이그레이션 시 일괄 인덱싱 배치 실행

## 8. MVP 범위 제안

### MVP에 포함할 항목
- Document 단위 전문 검색 (제목 + 메타데이터 + 본문)

- 고급 메타데이터 검색
- 추천 검색어 / 자동 완성
- Admin 검색 인덱스 관리 화면
- 검색 로그 및 품질 분석

## 12. 최종 정리

Phase 8은 검색 화면을 만드는 단계이지만, 본질적으로는  
**Mimir를 "저장소"에서 "탐색 가능한 지식 플랫폼"으로 전환하는 단계**다.

이 단계에서 중요한 것은 검색 기능의 수를 늘리는 것이 아니라,

- 검색 결과가 항상 권한 범위 내에서만 반환되는가
- 노드 단위까지 탐색이 가능한가
- 검색 API가 외부 시스템에서도 안전하게 사용 가능한가
- Phase 10 벡터 검색으로 자연스럽게 확장될 수 있는가

를 구조적으로 보장하는 것이다.

Phase 8의 산출물은 단순한 검색 기능이 아니라  
**권한 인식 검색 인프라**이어야 한다.
