# Phase 0 → Phase 1/2 기술 입력 요약

## 문서 목적

이 문서는 Phase 0-5에서 정의한 기술 스택 방향을 Phase 1 도메인 모델 설계와 Phase 2 API·서비스 설계에 직접 연결하는 요약서다. 각 Phase에서 기술적 전제를 공유하고 방향 논쟁 없이 설계에 들어갈 수 있도록 입력 조건을 제공한다.

## Phase 1 (도메인 모델 설계)에 주는 제약·가이드

### T-P1-1. 도메인 모델은 저장 방식에 종속되지 않아야 한다

도메인 엔티티(`Document`, `Version`, `Node`, `Metadata`, `DocumentType`)를 설계할 때 DB 테이블 구조를 먼저 생각하지 않는다. 도메인 모델이 먼저 확정되고, 저장 매핑(ORM 설정, 테이블 구조)은 그 이후에 별도로 결정한다.

**구체적 제약**:
- 엔티티에 DB 특화 필드(`rowVersion`, `partitionKey`, `dbId`)를 직접 추가하지 않는다.
- ORM 어노테이션이 도메인 클래스에 직접 붙는 구조는 지양한다. (ActiveRecord 패턴보다 Repository 패턴 선호)
- `Document.metadata`는 도메인에서 `Map<String, Any>` 수준으로 추상화하고, 물리 저장 방식(JSONB, EAV 등)은 인프라 계층에서 결정한다.

### T-P1-2. 검색·버전·상태·메타데이터는 1급 도메인 요소다

이 요소들을 "나중에 추가할 기능"으로 취급하지 않는다. 도메인 모델 초기 설계에서 이들이 독립적인 엔티티로 명확하게 자리를 잡아야 한다.

**구체적 제약**:
- `Version`은 `Document`의 부가 속성(컬럼 하나)이 아니라 독립 엔티티로 설계한다.
- `Metadata`는 고정 컬럼이 아니라 스키마 기반 확장 구조로 설계한다.
- 문서 상태는 단순 문자열 컬럼이 아니라, 상태 전이 규칙을 포함하는 상태 모델로 설계한다.
- 검색 대상이 될 필드는 도메인 모델 설계 시 미리 식별한다.

### T-P1-3. 계층 분리 구조를 전제로 설계한다

Phase 1에서 도메인 모델이 확정되면, 이 모델이 아래 계층 구조에서 어떻게 동작할지를 함께 고려해야 한다.

```
API 계층  ←  서비스/유스케이스 계층  ←  도메인 계층  ←  인프라 계층
```

**구체적 제약**:
- 도메인 엔티티는 프레임워크나 외부 라이브러리에 의존하지 않아야 한다.
- 도메인 서비스(상태 전이, 버전 생성, 권한 검사)는 도메인 계층 또는 서비스 계층에 위치하고, API 계층에 도메인 로직이 흘러들어가지 않아야 한다.

### T-P1-4. 비동기 처리 여지를 도메인 설계에 반영한다

비동기로 처리될 작업(인덱싱, 이벤트 발행 등)이 도메인 모델에서 어떻게 트리거되는지를 설계 단계에서 고려한다.

**구체적 제약**:
- 도메인 이벤트(`DocumentCreated`, `VersionCreated`, `StatusChanged`) 개념을 도메인 모델에 반영한다.
- 도메인 서비스에서 직접 검색 인덱서나 외부 시스템을 호출하지 않는다. 대신 도메인 이벤트를 발행하고, 인프라 계층이 이를 구독하여 처리한다.

## Phase 2 (API·서비스 설계)에 주는 가이드

### T-P2-1. API 설계는 도메인 모델 확정 후 시작한다

Phase 1에서 도메인 모델이 확정된 후 API 엔드포인트를 설계한다. 화면 구성에서 API를 역산하지 않는다.

**구체적 가이드**:
- 도메인 엔티티가 API 리소스로 자연스럽게 드러나야 한다. (`/documents`, `/versions`, `/nodes`, `/metadata`)
- API 응답 구조는 도메인 모델을 직렬화하되, 역할별로 다른 뷰(User/Admin)를 제공하는 직렬화 계층을 설계한다.
- API 계약(엔드포인트, 요청/응답 구조)이 확정되면 OpenAPI 스펙으로 문서화한다.

### T-P2-2. 백엔드 프레임워크 선택 시 기술 선택 기준을 적용한다

Phase 2에서 백엔드 프레임워크를 최종 선택할 때 `Task 5_tech_selection_criteria.md`의 기준을 적용한다.

**우선 확인 항목**:
1. API 친화성: REST 엔드포인트 설계를 도메인 중심으로 구성할 수 있는가?
2. 도메인 모델 수용: 계층 분리와 Repository 패턴을 자연스럽게 지원하는가?
3. 타입 안정성: 강타입 언어 또는 타입 시스템을 지원하는가?
4. 미들웨어 확장성: 인증·권한·감사 로그를 미들웨어로 처리할 수 있는가?

### T-P2-3. User/Admin API 경계를 명시적으로 설계한다

- `/api/v1/*` (사용자 도메인 API)와 `/api/v1/admin/*` (관리자 API) 경로를 분리한다.
- Admin API 경로에는 공통 미들웨어에서 관리자 권한 검사와 감사 로그 기록을 자동으로 처리한다.
- 같은 리소스라도 역할별 응답 필드가 다를 수 있도록 직렬화 전략을 설계한다.

### T-P2-4. 비동기 작업 인터페이스를 초기부터 추상화한다

비동기 처리 계층을 직접 구현(인메모리 큐, DB 기반 잡)으로 시작하더라도, 인터페이스는 추상화하여 나중에 메시지 브로커로 교체할 수 있게 한다.

**구체적 가이드**:
- 잡 큐 또는 이벤트 발행 인터페이스를 도메인·서비스 계층에서 직접 의존하지 않는다. 인프라 계층 인터페이스를 통해 의존한다.
- 비동기 작업 목록과 각 작업의 재시도 정책을 Phase 2에서 문서화한다.
