# Phase 0 — 기술 선택 기준표

## 문서 목적

이 문서는 Mimir 플랫폼의 각 기술 계층에서 기술을 선택할 때 적용할 기준을 정의한다. 특정 제품을 비교하는 것이 아니라, 무엇을 기준으로 판단해야 하는지를 명확히 한다. 기술 선택 논의에서 "익숙해서", "유행이어서" 같은 근거가 아니라 플랫폼의 목표에 맞는 기준으로 판단한다.

---

| 순위 | 기준 | 핵심 질문 |
|------|------|----------|
| 1 | API 친화성 | 이 기술이 API-first 구조를 자연스럽게 지지하는가? |
| 2 | 도메인 모델 수용 가능성 | 버전·상태·메타데이터·구조 같은 복잡한 도메인 모델을 표현할 수 있는가? |
| 3 | 타입 안정성 | 복잡한 도메인 계약을 컴파일 타임에 검증할 수 있는가? |

## 기준 상세

### 1. API 친화성

**의미**: 기술이 API 계약 중심 설계를 자연스럽게 지원하는가. 라우팅, 입력 검증, 응답 직렬화, 에러 핸들링, API 버전 관리를 일관된 방식으로 처리할 수 있는가.

**왜 중요한가**: Mimir는 UI만을 위한 시스템이 아니라 다수의 외부 소비자를 위한 API 플랫폼이다. API 설계가 프레임워크의 제약으로 왜곡되면 안 된다.

**특히 중요한 계층**: 백엔드/API 계층

**체크 포인트**:
- REST 엔드포인트 설계를 도메인 중심으로 구성할 수 있는가?
- 요청 입력 유효성 검증을 선언적으로 처리할 수 있는가?
- API 응답 구조를 역할별로 다르게 직렬화할 수 있는가?
- OpenAPI 스펙 생성 등 API 문서화 자동화가 가능한가?

---

### 2. 도메인 모델 수용 가능성

**의미**: Document·Version·Node·Metadata·DocumentType·Status 같은 복잡한 도메인 엔티티와 그 관계를 명확하게 표현할 수 있는가. 저장소나 프레임워크의 구조가 도메인 모델을 침식하지 않는가.

**왜 중요한가**: 도메인 모델이 기술 구현에 종속되면, 비즈니스 요구가 바뀔 때 데이터 구조 전체를 변경해야 하는 상황이 발생한다.

**특히 중요한 계층**: 백엔드(도메인 계층), 저장 계층

**체크 포인트**:
- 엔티티와 값 객체(Value Object)를 명확하게 구분할 수 있는가?
- 메타데이터 스키마처럼 동적으로 확장되는 구조를 수용할 수 있는가?
- 저장 방식(ORM 매핑, JSON 컬럼)이 도메인 모델에 영향을 주지 않는가?
- 상태 전이 규칙을 도메인 레이어에서 명확하게 표현할 수 있는가?

---

### 3. 타입 안정성

**의미**: 코드에서 타입 오류를 컴파일 타임 또는 정적 분석 단계에서 잡을 수 있는가. 복잡한 API 응답 구조와 도메인 계약을 타입으로 표현하고 검증할 수 있는가.

**왜 중요한가**: 메타데이터 스키마, 역할별 API 응답 필드 분기, 상태 전이 규칙 같은 복잡한 로직은 타입 오류가 런타임에서야 발견될 경우 유지보수 비용이 크다.

**특히 중요한 계층**: 백엔드(도메인·API 계층), 프론트엔드

**체크 포인트**:
- 프론트엔드와 백엔드 간 API 계약을 타입으로 공유할 수 있는가?
- 동적 메타데이터 값을 안전하게 처리할 수 있는 타입 구조를 만들 수 있는가?
- 상태 머신의 전이 규칙을 타입 수준에서 검증할 수 있는가?

---

### 4. 확장성

**의미**: 데이터 규모, 사용자 수, 기능 범위가 증가할 때 시스템이 이를 감당할 수 있는가. 수평 확장(인스턴스 추가)과 기능 확장(새 문서 유형, 새 연계) 모두를 포함한다.

**왜 중요한가**: 지식 플랫폼은 초기엔 소규모로 시작해도 장기적으로 문서 수와 사용자 수가 증가한다. 확장이 어려운 구조는 나중에 전면 재설계를 요구한다.

**특히 중요한 계층**: 백엔드, 저장 계층, 검색 계층, 비동기 처리 계층

**체크 포인트**:
- 새 문서 유형 추가 시 코어 코드 변경 없이 확장 가능한가?
- 검색 인덱스를 별도 계층으로 분리할 수 있는가?
- 비동기 잡 처리 계층을 단순 구현에서 확장 구현으로 교체할 수 있는가?
- 저장소 계층을 도메인에서 분리하여 저장소 교체가 가능한가?

---

### 5. 보안·권한 구조 수용 가능성

**의미**: 인증(Authentication)·인가(Authorization)·역할 기반 접근 제어(RBAC)·감사 로그를 플랫폼 전반에 걸쳐 일관되게 적용할 수 있는가.

**왜 중요한가**: User/Admin 분리 원칙과 API-first 구조에서 권한 통제는 선택이 아닌 필수다. 권한 처리가 각 기능에 산발적으로 구현되면 누락과 취약점이 생긴다.

**특히 중요한 계층**: 백엔드/API 계층, 프론트엔드(인증 흐름)

**체크 포인트**:
- 미들웨어 수준에서 인증·권한을 일관되게 처리할 수 있는가?
- 역할별 API 응답 필드 분리를 선언적으로 처리할 수 있는가?
- 감사 로그 기록을 도메인 코드에서 분리하여 일관되게 적용할 수 있는가?

---

### 6. 유지보수성

**의미**: 코드베이스가 시간이 지나도 이해 가능하고 변경 가능한 상태를 유지하는가. 계층 간 책임이 명확하여 특정 기능 변경이 예상치 못한 곳에 파급되지 않는가.

**왜 중요한가**: 지식 플랫폼은 장기적으로 운영되며, 도메인 요구사항이 계속 진화한다. 유지보수가 어려운 코드베이스는 기능 추가 비용이 기하급수적으로 증가한다.

**특히 중요한 계층**: 전 계층

**체크 포인트**:
- 계층 간 의존 방향이 단방향으로 유지되는가?
- 특정 저장소·프레임워크 변경이 도메인 코드에 영향을 주지 않는가?
- 코드 변경 시 영향 범위가 예측 가능한가?

---

### 7. 운영 용이성

**의미**: 배포, 환경 관리, 모니터링, 장애 대응, 로그 수집이 얼마나 체계적으로 이루어질 수 있는가.

**왜 중요한가**: Admin UI가 운영 데이터를 파악할 수 있으려면, 시스템이 운영 가시성을 기본으로 제공해야 한다.

**특히 중요한 계층**: 인프라 계층, 비동기 처리 계층

**체크 포인트**:
- 구조화 로그(JSON)를 기본으로 제공하는가?
- 비동기 작업 실패를 감지하고 재시도할 수 있는가?
- 환경 변수 기반 설정 관리가 쉬운가?
- 헬스 체크, 준비 체크 엔드포인트를 제공하는가?

---

### 8. 테스트 용이성

**의미**: 단위 테스트, 통합 테스트, E2E 테스트를 계층별로 독립적으로 작성하고 실행할 수 있는가.

**왜 중요한가**: 복잡한 도메인 규칙(상태 전이, 권한, 버전 생성 등)은 자동화 테스트 없이는 안정적으로 유지하기 어렵다.

**특히 중요한 계층**: 백엔드(도메인·서비스 계층), API 계층

**체크 포인트**:
- 도메인 계층을 외부 의존성 없이 단위 테스트할 수 있는가?
- API 계층을 실제 저장소 없이 통합 테스트할 수 있는가?
- 비동기 작업 처리를 테스트 환경에서 동기적으로 실행할 수 있는가?

---

### 9. 팀 생산성

**의미**: 러닝 커브와 개발 속도의 균형. 강력한 기술이더라도 팀이 효율적으로 사용할 수 없으면 생산성이 낮아진다.

**왜 중요한가**: 기술적 완성도와 개발 속도 모두 중요하다. 지나치게 복잡한 기술 선택은 초기 구축을 지연시키고 유지보수 인력을 제한한다.

**특히 중요한 계층**: 전 계층, 특히 초기 선택 시 균형 필요

**체크 포인트**:
- 팀의 현재 기술 수준에서 합리적인 러닝 커브인가?
- IDE 지원, 코드 자동 완성, 개발 도구 생태계가 충분한가?
- 문제 발생 시 커뮤니티나 공식 문서에서 해결 방법을 찾을 수 있는가?

---

### 10. 커뮤니티·생태계 성숙도

**의미**: 기술의 장기 지속 가능성. 활발한 커뮤니티, 안정적 릴리즈 주기, 충분한 서드파티 라이브러리 지원.

**왜 중요한가**: 지식 플랫폼은 장기 운영을 전제로 한다. 채택 시 활발했던 기술이 수년 후 유지보수 단계에 접어들거나 커뮤니티가 축소되면 기술 부채가 된다.

**특히 중요한 계층**: 전 계층

**체크 포인트**:
- 최근 2년간 활발한 개발과 릴리즈가 있는가?
- 보안 취약점 패치가 빠르게 이루어지는가?
- 필요한 기능(인증, ORM, 검색 연동 등)을 위한 성숙한 라이브러리 생태계가 있는가?
