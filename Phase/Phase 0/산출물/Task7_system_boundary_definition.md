# Phase 0 — 시스템 경계 정의서

## 문서 목적

이 문서는 Mimir 플랫폼이 직접 책임지는 영역과 외부 시스템과 연계하는 영역을 명확히 구분한다. 설계와 개발 범위가 불필요하게 팽창하거나 외부 시스템 구조에 코어 도메인이 잠식되는 것을 방지하기 위한 책임 범위 헌장이다. 이후 기능 추가 요청이나 설계 방향 논쟁 시 이 문서를 기준으로 판단한다.

---

## 1. 코어 플랫폼 책임

다음은 Mimir가 반드시 직접 책임져야 하는 영역이다. 외부 시스템에 위임하거나 나중으로 미룰 수 없다.

### 1-1. Document 관리

**코어 책임인 이유**: 플랫폼의 존재 이유 자체. 문서 생성·조회·수정·삭제, 문서 유형 분류, 문서 소유권 관리는 어떤 외부 시스템도 대신할 수 없다.

**플랫폼이 책임지는 수준**: 문서 엔티티의 생명주기 전체. 생성부터 보관·폐기까지.

**외부에 위임하지 않는 범위**: 문서 식별자, 소유권, 유형, 상태 정보. 이 데이터의 진실의 원천은 항상 플랫폼이다.

### 1-2. Version 관리

**코어 책임인 이유**: 지식 플랫폼의 핵심 가치인 변경 추적과 이력 복원은 플랫폼이 직접 보장해야 한다.

**플랫폼이 책임지는 수준**: 모든 문서 수정에 대한 버전 생성, 특정 버전 조회, 버전 간 비교 가능성.

**외부에 위임하지 않는 범위**: 버전 식별자와 불변성. 버전은 생성 후 삭제나 수정을 허용하지 않는다 (관리자 명시적 폐기 제외).

### 1-3. Node / Structure 관리

**코어 책임인 이유**: 문서 내부 구조(조항, 섹션 등)를 플랫폼이 인식하고 관리해야 Node 단위 검색·탐색·메타데이터 부착이 가능하다.

**플랫폼이 책임지는 수준**: Node 엔티티 생성·조회·수정, 트리 구조 유지, Node 단위 메타데이터.

**외부에 위임하지 않는 범위**: 문서 구조의 정의와 저장. 렌더링은 외부 클라이언트가 담당할 수 있으나, 구조 데이터 자체는 플랫폼이 소유.

### 1-4. Metadata 관리

**코어 책임인 이유**: 메타데이터는 검색·필터링·외부 연계의 기반이다. 플랫폼이 스키마를 정의하고 검증해야 일관성이 보장된다.

**플랫폼이 책임지는 수준**: 메타데이터 스키마 정의(DocumentType과 연동), 메타데이터 값 저장·검증, 검색 인덱스 반영.

**외부에 위임하지 않는 범위**: 메타데이터 원본 값과 스키마 검증. 외부 시스템이 메타데이터를 쓰더라도 플랫폼 스키마 검증을 통과해야 한다.

### 1-5. DocumentType 관리

**코어 책임인 이유**: 플랫폼의 확장성의 핵심 메커니즘. 새 문서 유형이 추가될 때 코드 변경 없이 수용하려면 플랫폼이 유형 정의를 직접 관리해야 한다.

**플랫폼이 책임지는 수준**: 유형 정의, 메타데이터 스키마 관리, 유형별 상태 정책, 구조 규칙.

### 1-6. Status / Lifecycle 관리

**코어 책임인 이유**: 문서 상태(초안·검토·승인·게시·폐기)는 플랫폼의 핵심 도메인 규칙이다. 외부 워크플로 엔진의 상태 개념이 이를 대체하면 안 된다.

**플랫폼이 책임지는 수준**: 상태 값 정의, 상태 전이 허용 규칙, 현재 상태 관리, 상태 전이 이력 기록.

**외부에 위임하지 않는 범위**: 상태의 의미와 전이 규칙. 외부 BPM 시스템이 상태를 트리거할 수 있으나, 상태 전이 유효성은 플랫폼이 검증한다.

### 1-7. Search / Query 코어 경험

**코어 책임인 이유**: 검색은 지식 플랫폼의 핵심 사용자 경험이다. 어떤 검색 인터페이스를 제공할지는 플랫폼이 정의해야 한다.

**플랫폼이 책임지는 수준**: 검색 API 정의, 검색 가능한 필드와 필터 정의, 검색 결과 구조 정의. 검색 엔진 구현은 외부 계층에 위임 가능.

**외부에 위임 가능한 범위**: 검색 엔진 내부 알고리즘(TF-IDF, BM25, 벡터 검색 등). 단, 인터페이스는 플랫폼이 소유.

### 1-8. Permission / Access Control 코어 정책

**코어 책임인 이유**: 누가 어떤 문서에 접근할 수 있는지는 플랫폼의 핵심 신뢰 영역이다.

**플랫폼이 책임지는 수준**: 권한 정책 정의, 역할(Role) 관리, 리소스 접근 제어 집행.

**외부에 위임 가능한 범위**: 사용자 인증(Authentication) 자체는 외부 Identity 시스템에 위임 가능. 단, 인가(Authorization) 정책은 플랫폼이 소유.

### 1-9. Audit / History 기록

**코어 책임인 이유**: 컴플라이언스와 운영 추적은 플랫폼이 직접 보장해야 한다.

**플랫폼이 책임지는 수준**: 주요 도메인 행위(문서 생성·수정·상태 전이·권한 변경)의 감사 이벤트 생성.

**외부에 위임 가능한 범위**: 감사 데이터의 장기 보관·분석은 외부 로그 시스템에 위임 가능. 단, 원본 이벤트 생성은 플랫폼이 책임.

### 1-10. User / Admin UI API 제공

**코어 책임인 이유**: API-first 원칙에 따라 모든 기능은 API로 노출되어야 한다.

**플랫폼이 책임지는 수준**: 모든 코어 기능의 API 계약 정의, 버전 관리, 인증·권한 적용.

## 2. 내부 서브시스템 경계

| 서브시스템 | 역할 | 코어 여부 | 인터페이스 방식 |
|-----------|------|:--------:|--------------|
| **Core Domain** | Document·Version·Node·Metadata·Status·DocumentType·Relationship 도메인 엔티티 및 규칙 | ✓ 코어 | 서비스 계층 인터페이스 |
| **API / Application Layer** | HTTP API 라우팅, 입력 검증, 직렬화, 인증 미들웨어 | ✓ 코어 | REST API (외부 소비자 접점) |
| **User UI** | 일반 사용자 지식 탐색·작성·편집 인터페이스 | 클라이언트 | API 호출 |
| **Admin UI** | 운영자 관리·정책·감사 인터페이스 | 클라이언트 | API 호출 (관리자 API) |
| **Search / Index Layer** | 검색 인덱스 관리, 색인 동기화, 검색 쿼리 처리 | 준코어 | 비동기 이벤트 + 동기 쿼리 API |
| **File / Resource Layer** | 첨부 파일 업로드·다운로드, 오브젝트 스토리지 연동 | 준코어 | API + 오브젝트 스토리지 어댑터 |
| **Auth / Permission Boundary** | 인증 토큰 검증, 역할 기반 권한 집행 | ✓ 코어 | 미들웨어 (모든 API 요청 선행) |
| **Audit / Operations Layer** | 감사 이벤트 생성·저장, 운영 로그 관리 | ✓ 코어 | 비동기 이벤트 발행 |
| **Integration / Connector Layer** | 외부 시스템 연동 어댑터, 외부 이벤트 수신·발행 | 연계 계층 | API + 이벤트 스트림 |
| **Async Job / Event Layer** | 비동기 작업 큐, 이벤트 처리, 재시도 관리 | 인프라 | 내부 이벤트 버스 |

## 3. 외부 시스템 유형 분류

### 3-1. Identity / SSO / IAM

**연계 이유**: 조직 내 통합 인증 체계(LDAP, SAML, OIDC 등)를 재구현하는 것은 비용과 보안 위험이 크다.

**플랫폼이 직접 대체하지 않는 이유**: 인증은 조직 전체 인프라의 문제다. 플랫폼이 독자 인증 시스템을 만들면 SSO 통합이 불가능해진다.

**연결 계층**: Auth/Permission Boundary (토큰 검증만 처리, 인증 자체는 위임)

### 3-2. External File / Object Storage

**연계 이유**: S3 호환 오브젝트 스토리지는 대용량 파일 저장에 특화된 인프라다. 플랫폼이 파일 저장 인프라를 직접 운영하는 것은 비효율적이다.

**연결 계층**: File/Resource Layer (어댑터를 통한 추상화, 플랫폼은 참조 경로만 소유)

### 3-3. Search Engine / Indexing Engine

**연계 이유**: 고성능 전문 검색, 다국어 형태소 분석, 벡터 검색은 전문 검색 엔진이 담당해야 한다.

**연결 계층**: Search/Index Layer (플랫폼은 인터페이스 정의, 엔진은 교체 가능)

### 3-4. Notification / Messaging System

**연계 이유**: 이메일, 슬랙, 알림 앱 등 메시징 채널은 조직마다 다르다. 플랫폼이 모든 채널을 직접 구현하면 결합도가 높아진다.

**연결 계층**: Integration/Connector Layer (이벤트 발행 → 외부 알림 시스템이 구독)

### 3-5. Workflow / Approval / BPM System

**연계 이유**: 복잡한 다단계 승인 프로세스(결재 라인, 조건부 분기 등)는 전문 BPM 시스템의 영역이다.

**플랫폼이 직접 대체하지 않는 이유**: 플랫폼의 코어는 지식 자산 관리이지 범용 워크플로 엔진이 아니다.

**경계**: 플랫폼은 상태 전이 이벤트를 발행한다. 복잡한 승인 흐름은 외부 BPM이 이 이벤트를 소비하여 처리한다.

### 3-6. External Business Applications

**연계 이유**: ERP, HR, 프로젝트 관리 시스템 등 조직 내 다른 시스템이 Mimir의 지식 자산을 참조·연동해야 할 수 있다.

**연결 계층**: API (플랫폼 API를 소비), Integration/Connector Layer (양방향 동기화 필요 시)

### 3-7. External Document Sources / Connectors

**연계 이유**: 기존 문서 시스템(SharePoint, Confluence 등)에서 문서를 마이그레이션하거나 동기화해야 할 수 있다.

**연결 계층**: Integration/Connector Layer (외부 소스 전용 커넥터, 코어 도메인에 직접 연결 금지)

### 3-8. AI / NLP / RAG / Automation Services

**연계 이유**: AI 추론·요약·임베딩 생성은 전문 AI 인프라(OpenAI, 자체 LLM 등)에 위임하는 것이 현실적이다.

**플랫폼이 직접 대체하지 않는 이유**: AI 기능 자체는 플랫폼 코어가 아니다. 플랫폼은 AI가 접근할 수 있는 구조화된 지식 인터페이스를 제공하고, AI 처리 결과는 보조 데이터로 분리 관리한다.

**연결 계층**: Integration/Connector Layer + Search/Index Layer (벡터 임베딩)

### 3-9. Observability / Logging / Monitoring Stack

**연계 이유**: 로그 집계(ELK, Datadog), APM, 알림 규칙은 플랫폼 전용이 아닌 조직 공통 인프라에 통합된다.

**연결 계층**: Audit/Operations Layer (구조화 로그 출력), Async Job Layer (잡 상태 메트릭)

## 4. API / 데이터 / 운영 경계 구분

### 4-1. API 경계

| API 유형 | 경로 | 접근 주체 | 책임 주체 |
|---------|------|---------|---------|
| **User Domain API** | `/api/v1/*` | User UI, AI Agent, External App | 플랫폼 코어 |
| **Admin API** | `/api/v1/admin/*` | Admin UI | 플랫폼 코어 (관리자 권한) |
| **Integration API** | `/api/v1/integration/*` | 외부 시스템, 서비스 계정 | 플랫폼 코어 + 커넥터 계층 |
| **Internal Ops API** | `/internal/*` | 플랫폼 운영 도구 전용 | 플랫폼 운영팀, 외부 노출 차단 |

외부 시스템이 플랫폼 내부 Admin API에 직접 접근하는 것은 허용하지 않는다.

### 4-2. 데이터 경계

| 데이터 범주 | 원본 소유자 | 성격 |
|-----------|-----------|------|
| Document·Version·Node·Metadata·Status | **플랫폼** | 원본 데이터 (Source of Truth) |
| 검색 인덱스 | **플랫폼** (인덱스 계층 관리) | 파생 데이터 (원본에서 재생성 가능) |
| 사용자 계정·인증 정보 | **외부 Identity 시스템** | 플랫폼은 참조만 |
| 첨부 파일 원본 바이너리 | **외부 오브젝트 스토리지** | 플랫폼은 메타 정보와 참조 경로 소유 |
| AI 임베딩·추론 결과 | **AI 서비스** 생성 → 플랫폼 인덱스 저장 | 보조 데이터 (원본과 분리) |
| 외부 동기화 원본 문서 | **외부 소스 시스템** | 플랫폼은 가공·변환본만 소유 |

### 4-3. 운영 경계

| 장애/문제 유형 | 책임 주체 |
|-------------|---------|
| 문서 CRUD API 오류 | 플랫폼 팀 |
| 검색 인덱스 지연·불일치 | 플랫폼 팀 (동기화 파이프라인) |
| 인증 실패 | 외부 Identity 시스템 (플랫폼은 의존성으로 기록) |
| 첨부 파일 업로드 실패 | 플랫폼 팀 (API) + 외부 스토리지 팀 (인프라) |
| AI 응답 지연·오류 | 외부 AI 서비스 (플랫폼은 timeout 처리) |
| 외부 BPM 연동 실패 | 외부 BPM 팀 (플랫폼은 이벤트 발행까지만 보장) |

## 5. 커넥터 / 어댑터 원칙

- **도메인 모델은 외부 API 형식에 종속되지 않는다.** 외부 시스템의 데이터 구조를 플랫폼 도메인 모델에 직접 매핑하지 않는다. 커넥터 계층에서 변환한다.
- **외부 식별자와 내부 식별자를 분리한다.** 외부 시스템의 ID를 플랫폼 코어 ID로 사용하지 않는다. 플랫폼 내부 UUID를 독립적으로 유지하고, 외부 ID는 별도 필드로 관리한다.
- **외부 장애가 코어 기능을 마비시키지 않도록 격리한다.** 외부 검색 엔진 장애 시 기본 조회는 가능해야 하고, 외부 알림 시스템 장애가 문서 저장에 영향을 주지 않아야 한다.
- **벤더 종속 로직을 코어 모델에 넣지 않는다.** 특정 검색 엔진의 쿼리 DSL, 특정 SSO 공급자의 클레임 구조, 특정 AI API의 응답 형식이 도메인 모델에 침투하지 않도록 어댑터가 변환한다.
- **커넥터는 교체 가능해야 한다.** 검색 엔진을 Elasticsearch에서 OpenSearch로 교체하거나, 오브젝트 스토리지를 S3에서 GCS로 교체할 때 도메인 코드 변경이 없어야 한다.

---

## 6. 위반 사례

### 사례 1. 외부 워크플로 엔진의 상태 모델이 코어 Document 모델을 지배

**상황**: BPM 시스템의 태스크 ID와 상태 코드를 Document 테이블 컬럼에 직접 추가한다.

**왜 문제인가**: BPM 시스템이 교체되면 Document 모델 전체를 수정해야 한다. 플랫폼의 상태 모델이 외부 시스템 구조에 종속된다.

**위반 원칙**: 커넥터 원칙 (외부 API 형식 종속 금지), 코어 도메인 독립성.

**개선 방향**: Document의 상태는 플랫폼 자체 상태 모델로 관리한다. BPM 시스템과의 연동은 Integration/Connector Layer에서 처리하며, 플랫폼 상태 이벤트를 BPM이 소비하는 방식으로 연결한다.

---

### 사례 2. 특정 SSO 공급자 클레임 구조가 사용자 모델을 지배

**상황**: Azure AD의 클레임 필드명(`oid`, `upn`, `groups`)을 플랫폼 User 엔티티의 컬럼명으로 사용한다.

**왜 문제인가**: SSO 공급자가 교체되면 User 모델 전체가 바뀌어야 한다. 외부 인증 시스템의 구조가 코어 도메인을 침식한다.

**위반 원칙**: 커넥터 원칙 (벤더 종속 로직 금지), 외부 식별자/내부 식별자 분리.

**개선 방향**: 플랫폼 User 엔티티는 내부 UUID와 플랫폼 관점의 속성(이메일, 이름, 역할)만 포함한다. 외부 SSO 매핑은 별도 `IdentityMapping` 엔티티로 관리한다.

---

### 사례 3. 검색 인덱스 구조가 원본 저장 모델로 사용됨

**상황**: Elasticsearch 인덱스 매핑이 도메인 모델의 기준이 되어, DB 스키마가 인덱스 구조를 따라간다.

**왜 문제인가**: 검색 엔진 교체 시 도메인 모델 전체를 변경해야 한다. 검색 인덱스 손상 시 원본 데이터도 불신뢰 상태가 된다.

**위반 원칙**: 코어 도메인 독립성, 커넥터 원칙.

**개선 방향**: 도메인 모델은 관계형 DB 기반으로 독립 설계. 검색 인덱스는 도메인 원본에서 파생되며, 언제든 재구성 가능한 보조 데이터다.

---

### 사례 4. AI 추출 결과가 검증 없이 원본 도메인 데이터로 저장됨

**상황**: AI가 문서에서 추출한 키워드·요약을 검증 없이 Metadata 필드에 직접 저장한다.

**왜 문제인가**: AI 오류나 할루시네이션이 도메인 원본 데이터를 오염시킨다. AI 서비스 교체 시 기존 데이터의 신뢰성이 불확실해진다.

**위반 원칙**: 데이터 경계 (AI 결과는 보조 데이터, 원본과 분리).

**개선 방향**: AI 생성 데이터는 별도 보조 저장소(AI Metadata 계층)에 저장하고, 원본 Metadata와 명확히 구분한다. 운영자 검토 후 원본 메타데이터로 승격하는 프로세스를 둔다.

---

### 사례 5. 외부 문서 수집기 로직이 코어 서비스에 하드코딩됨

**상황**: SharePoint 문서를 가져오는 로직이 `DocumentService` 내부에 직접 구현된다.

**왜 문제인가**: 수집 소스가 바뀌거나 추가될 때 코어 서비스를 수정해야 한다. 수집기 장애가 코어 서비스에 영향을 준다.

**위반 원칙**: 커넥터 원칙 (외부 종속 로직 분리), 코어 도메인 독립성.

**개선 방향**: 외부 문서 수집은 Integration/Connector Layer의 별도 커넥터로 분리한다. 커넥터가 수집한 데이터를 플랫폼 API를 통해 일반 문서 생성 흐름으로 투입한다.

---

## 7. 범위 통제 시사점

### 플랫폼이 반드시 직접 제공해야 하는 것

Document·Version·Node·Metadata·Status 관리, DocumentType 확장, 권한 정책, 감사 로그 기록, 검색 인터페이스, User/Admin API.

### 플랫폼이 연동만 제공하면 되는 것

사용자 인증(SSO 위임), 파일 저장(오브젝트 스토리지 어댑터), 검색 엔진(인터페이스 추상화), 알림 발송(이벤트 발행 후 외부 처리), AI 처리(보조 계층으로 분리).

### 후속 Phase로 넘겨야 하는 것

실시간 협업 편집, 복잡한 다단계 승인 워크플로 자체 구현, 벡터 검색·RAG 파이프라인 직접 구현, 외부 문서 소스 커넥터 다종 지원.

### 요구는 많지만 코어 목표와 맞지 않는 것의 예시

- "이메일/슬랙 알림도 플랫폼 안에서 보내야 한다" → 이벤트 발행 인터페이스 제공으로 충분. 알림 채널 구현은 외부 시스템 책임.
- "AI가 문서를 자동으로 분류해줘야 한다" → AI 서비스 연계 인터페이스 제공. AI 분류 로직 자체는 플랫폼 코어가 아니다.
- "외부 ERP 데이터를 가져와서 메타데이터로 써야 한다" → 커넥터 계층에서 처리. ERP 전용 로직을 코어에 내장하지 않는다.
