

# Task 4-6 작업지시서

## 작업명

버전 조회, 이력 탐색, 복원 흐름 설계

---

## 1. 작업 목적

Phase 4의 여섯 번째 작업으로, 문서 플랫폼에서 **버전 히스토리를 조회하고 탐색하며, 특정 과거 버전을 기준으로 복원 작업을 시작하는 흐름**을 상세 설계한다.

이번 작업의 목적은 다음과 같다.

- Task 4-1의 생명주기, Task 4-2의 Version 모델, Task 4-5의 Draft / Published 운영 정책을 실제 조회/복원 흐름으로 연결한다.
- 사용자가 문서의 변경 이력을 안전하게 이해하고 필요한 시점으로 돌아갈 수 있는 기준 기능을 설계한다.
- 현재 문서 조회와 특정 버전 조회를 분리하여, “현재 상태”와 “과거 이력”을 혼동하지 않도록 한다.
- 복원 기능이 이력 무결성을 훼손하지 않도록, 조회 흐름과 복원 액션의 책임 경계를 명확히 한다.
- 이후 diff 비교, 감사 추적 강화, 승인 이력 확장, 운영자 도구 확장으로 자연스럽게 이어질 수 있는 탐색 구조를 마련한다.

이 문서는 이후 Task 4-7과 구현 단계의 조회 서비스, 복원 서비스, 버전 히스토리 UI/API 테스트 설계의 기준 문서가 되어야 한다.

---

## 2. 작업 배경

앞선 작업에서 다음이 이미 정리되었다고 가정한다.

- Task 4-1: 문서 생성/개정/발행/복원의 생명주기와 상태 전이
- Task 4-2: Version 모델 상세 설계
- Task 4-4: 문서 작성/수정 API 설계
- Task 4-5: Draft / Published 상태 관리 규칙 설계
- Phase 2: 권한, 역할, 감사 추적 원칙
- Phase 3: API 버전 관리, 공통 응답 포맷, pagination/filtering/sorting 규약

이제 필요한 것은 다음을 구체화하는 일이다.

- 사용자는 어떤 기준으로 버전 목록을 볼 것인가?
- 현재 문서와 과거 버전은 어떤 방식으로 구분해서 조회할 것인가?
- 과거 버전 상세에서 어떤 정보를 보여줘야 복원 판단이 가능한가?
- 복원은 언제 허용하고, 그 결과는 무엇인가?
- 기존 Draft와 충돌할 때 복원 정책은 어떻게 동작하는가?

즉, 이번 작업은 단순한 “버전 목록 API”가 아니라, **버전 탐색과 복원 결정을 지원하는 전체 흐름을 설계하는 작업**이다.

## 4. 핵심 설계 질문

이번 작업에서는 최소한 아래 질문들에 답할 수 있어야 한다.

1. 버전 목록은 어떤 순서로 보여주는가? 최신순이 기본인가?
2. 버전 목록에는 어떤 요약 정보가 포함되어야 하는가?
3. 현재 Draft / 현재 Published / 과거 버전을 한 목록에서 함께 보여줄 것인가?
4. 사용자가 특정 버전을 선택했을 때, 어떤 상세 정보까지 볼 수 있어야 하는가?
5. 현재 문서 조회와 과거 버전 조회는 서로 다른 응답 또는 URL 구조를 가져야 하는가?
6. 복원 가능 여부는 목록 단계에서 표시하는가, 상세 단계에서 판단하는가?
7. 복원은 어떤 버전을 대상으로 허용하는가?
8. 기존 Draft가 있을 때 복원 시도는 어떻게 처리하는가?
9. 복원 후 사용자는 어디로 이동하는가? 새 Draft 상세인가, 현재 문서 편집 상태인가?
10. 일반 사용자와 편집자/관리자는 같은 버전 이력을 보아야 하는가, 일부만 보아야 하는가?
11. 복원 판단을 위해 change_summary, created_by, published_at 같은 메타데이터가 충분한가?
12. 향후 diff 기능이 추가되더라도 현재 설계가 자연스럽게 확장 가능한가?

## 5. 작업 지시

아래 항목을 순서대로 설계 문서로 정리한다.

### 5-1. 버전 탐색의 기본 사용자 흐름 정의

먼저 버전 탐색이 어떤 사용자 흐름으로 이루어지는지 정리한다.

최소한 아래 흐름을 포함한다.

1. 현재 문서 화면 진입
2. 버전 히스토리 열기
3. 버전 목록 탐색
4. 특정 버전 선택
5. 버전 상세 확인
6. 복원 가능 여부 판단
7. 복원 실행 또는 탐색 종료

각 단계에 대해 다음을 정리한다.

- 사용자 목적
- 시스템 제공 정보
- 다음 단계로 이어지는 조건
- 권한에 따른 차이 가능 여부

이 항목은 이후 API 응답 구성과 UI 흐름 정의의 공통 기준이 되어야 한다.

---

### 5-2. 버전 목록 조회 흐름 정의

버전 목록 조회에서 어떤 정보와 정렬 기준을 제공할지 정의한다.

반드시 다음을 포함한다.

- 기본 정렬 기준
- 정렬 후보(최신순, 버전 번호순 등)
- 상태 필터링 필요 여부
- pagination 필요 여부
- 기본 목록에 Draft / Published / 과거 버전 표시 방식
- “현재 공식 버전”, “현재 작업 중 Draft”, “복원 출처 버전” 등 표시 필요 여부

최소 요약 필드 후보도 정리한다.

예시:

- version_id
- version_number
- status
- created_at
- created_by
- published_at
- published_by
- restored_from_version_id
- change_summary
- is_current_published
- is_current_draft

목록은 사용자가 상세 열람 여부를 빠르게 판단할 수 있을 정도로 충분히 설명적이어야 한다.

---

### 5-3. 버전 상세 조회 흐름 정의

특정 버전 선택 시 어떤 정보를 보여줄지 정의한다.

반드시 다음을 구분해서 정리한다.

- 버전 메타 정보
- 상태 정보
- 본문 구조 또는 본문 요약 정보
- 작성/발행 이력 정보
- lineage 정보
- 복원 가능 여부 정보

다음 질문에도 답한다.

- 상세 조회는 content_snapshot 전체를 내려주는가, 일부만 내려주는가?
- 렌더링 조회와 도메인 상세 조회를 분리해야 하는가?
- 상세 응답에 related actions(restore 가능 여부 등)를 포함할지 여부

중요: 상세 조회는 복원 판단과 이력 이해를 지원해야 하므로, 단순한 ID/상태 수준을 넘는 정보를 제공해야 한다.

---

### 5-4. 현재 문서 조회와 특정 버전 조회 구분 정의

현재 상태 조회와 과거 이력 조회를 어떻게 구분할지 정의한다.

반드시 아래를 비교 검토한다.

#### 안 A. 현재 문서와 버전 조회를 별도 흐름으로 분리

- 현재 문서 조회는 current_draft 또는 current_published 기준
- 특정 버전 조회는 immutable snapshot 기준

#### 안 B. 하나의 조회 모델에서 파라미터로만 구분

- 단순해 보일 수 있음
- 의미 혼선 가능성 존재

이번 Phase 기준 권장안을 제시하고, 다음을 함께 정리한다.

- 현재 문서 조회의 기본 의미
- 특정 버전 조회의 불변성
- 두 흐름의 응답 차이
- UI/API 측면에서 혼동 방지 포인트

---

### 5-5. 버전 상태 표시 규칙 정의

버전 탐색 시 상태를 어떻게 표현할지 규칙을 정한다.

최소 검토 상태/표시값:

- current draft
- current published
- old published
- old draft 또는 discarded draft 필요 여부
- restored-derived
- archived 필요 여부

다음 내용을 포함한다.

- 상태값과 사용자 표시 라벨 관계
- 목록용 표시와 상세용 표시 차이
- 현재 활성 포인터와 상태값의 관계
- 같은 문서 안에서 현재/과거를 직관적으로 구분하는 방법

이 항목은 Task 4-5 운영 정책과 충돌하지 않도록 정리한다.

---

### 5-6. 복원 가능 여부 판단 규칙 정의

모든 버전이 복원 가능한지, 일부만 가능한지 정책을 정의한다.

반드시 다음을 검토한다.

- Published 버전만 복원 가능한지 여부
- Draft 상태 버전 복원 허용 여부
- 폐기된 Draft 복원 허용 여부
- Archived 문서의 버전 복원 허용 여부
- 현재 Draft 존재 시 복원 허용 여부
- 권한별 복원 가능 범위

다음 두 방향을 비교 검토한다.

#### 안 A. 안정성 우선

- 공식적으로 의미 있는 버전만 복원 허용
- 예: published 및 일부 명시적 snapshot만 복원 가능

#### 안 B. 유연성 우선

- 대부분의 버전에 대해 복원 허용
- 단, 예외와 충돌 정책 복잡도 증가

이번 Phase 기준 권장안을 명시한다.

---

### 5-7. 복원 실행 전 확인 정보 정의

사용자가 복원을 결정하기 전에 어떤 정보를 확인해야 하는지 정리한다.

최소 검토 항목:

- 복원 대상 버전 번호
- 생성 시각
- 생성자
- Published 여부
- change_summary
- 현재 Draft 존재 여부
- 복원 결과로 무엇이 생성되는지
- 현재 작업 중 Draft가 있는 경우 경고 필요 여부

이 항목은 나중에 UI confirm dialog, API precheck, 운영자 검토 화면으로 확장될 수 있으므로 충분히 구조화해서 적는다.

---

### 5-8. 복원 결과 흐름 정의

복원 실행 후 어떤 결과가 만들어지고, 사용자가 무엇을 보게 되는지 정의한다.

반드시 다음을 포함한다.

- 복원 후 생성되는 객체
- 복원 후 문서 상태
- 기존 current_published 유지 여부
- 기존 current_draft 처리 방식
- 새 Draft의 lineage 정보
- 복원 후 기본 이동/조회 대상
- 감사 이벤트 생성 포인트

권장 방향은, 복원 후 **새 Draft를 current_draft로 두고 기존 current_published는 유지**하는 방식이다. 다만 기존 Draft 존재 정책과 연결해 최종안을 정리한다.

---

### 5-9. 복원 충돌 및 예외 시나리오 정의

복원은 상태 충돌이 많으므로 별도로 정리한다.

최소 검토 대상:

- 기존 Draft가 있는 상태에서 Restore 요청
- 복원 대상 버전이 존재하지 않음
- 권한 없는 복원 요청
- 이미 삭제/보관된 문서에 대한 복원 요청
- 복원 가능한 상태가 아닌 버전 대상 요청
- 동일 버전에 대한 중복 복원 요청
- 복원 직후 Publish 시나리오 연계

각 시나리오에 대해 다음을 작성한다.

- 상황 설명
- 허용/금지 여부
- 권장 처리 정책
- API 에러 또는 경고 처리 포인트
- 감사 이벤트 필요 여부

---

### 5-10. 권한별 조회/복원 범위 정의

버전 이력은 민감할 수 있으므로, 역할별 접근 범위를 정리한다.

최소 검토 역할:

- viewer
- editor
- publisher
- admin

각 역할에 대해 다음을 정리한다.

- 버전 목록 조회 가능 여부
- Draft 포함 조회 가능 여부
- Published 이력 조회 가능 여부
- 상세 content_snapshot 조회 가능 범위
- 복원 가능 여부
- 복원 결과 Draft 접근 가능 여부

중요: 이번 작업은 상세 RBAC 구현이 아니라, **버전 탐색/복원 흐름에서 직접 필요한 권한 차이**만 정리한다.

---

### 5-11. 감사 추적과 이력 표시 연계 정의

버전 탐색과 복원은 감사 관점과 연결되므로, 어떤 정보가 남아야 하는지 정리한다.

최소 검토 항목:

- 버전 생성 이벤트
- Publish 이벤트
- Restore 이벤트
- Draft discard 이벤트
- 누가/언제/무엇을 기준으로 복원했는지 기록
- UI 목록/상세에서 어떤 감사 정보를 보여줄지 여부

다음도 함께 검토한다.

- change_summary와 audit event를 분리할지
- 사용자 표시용 이력과 내부 감사 로그의 차이

---

### 5-12. 권장 탐색/복원 정책안 도출

최종적으로 이번 Phase에서 채택할 탐색/복원 정책안을 요약한다.

최소한 아래 항목을 포함한다.

- 버전 목록 기본 정렬 기준
- 버전 목록 요약 필드
- 상세 조회 범위
- 현재 문서 조회와 특정 버전 조회 구분 방식
- 복원 가능 버전 범위
- 기존 Draft 존재 시 복원 정책
- 복원 결과 상태
- 역할별 기본 접근 범위

## 8. 권장 방향

아래 방향을 우선안으로 검토한다.

### 우선안 A

- 버전 목록은 **최신순**을 기본으로 하되, version_number를 함께 보여준다.
- 목록에는 current draft, current published, 과거 published를 구분할 수 있는 요약 필드를 포함한다.
- 현재 문서 조회와 특정 버전 조회는 **별도 흐름**으로 분리한다.
- 특정 버전 상세는 복원 판단이 가능할 정도의 메타데이터와 본문 구조 정보를 제공한다.
- 복원은 과거 버전을 직접 되살리는 것이 아니라 **새 Draft 생성 액션**으로 처리한다.
- 현재 Draft가 존재할 경우 복원은 제한하거나 명시적 정리 절차를 선행하도록 한다.
- 복원 후에는 새 Draft가 current_draft가 되고, 기존 current_published는 유지된다.
- 일반 사용자는 published 중심 이력을, 편집자/발행자/관리자는 Draft 포함 이력을 볼 수 있도록 권한 차등을 둔다.

이 우선안의 장점은 다음과 같다.

- 현재 상태와 과거 이력을 명확히 구분할 수 있다.
- 복원 기능이 안전하고 예측 가능하다.
- 목록/상세/복원 단계가 자연스럽게 연결된다.
- 향후 diff, 승인, 감사 강화 기능으로 확장하기 쉽다.
